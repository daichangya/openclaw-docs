# Dreaming

Dreaming هو نظام توحيد الذاكرة في الخلفية داخل `memory-core`.
وهو يساعد OpenClaw على نقل الإشارات القوية قصيرة المدى إلى ذاكرة دائمة مع
الحفاظ على العملية قابلة للشرح والمراجعة.

Dreaming هو **اشتراك اختياري** ومُعطّل افتراضيًا.

## ما الذي يكتبه Dreaming

يحتفظ Dreaming بنوعين من المخرجات:

- **حالة الآلة** في `memory/.dreams/` (مخزن الاستدعاء، وإشارات المراحل، ونقاط تحقق الإدخال، والأقفال).
- **مخرجات قابلة للقراءة البشرية** في `DREAMS.md` (أو `dreams.md` الموجود) وملفات تقارير مراحل اختيارية تحت `memory/dreaming/<phase>/YYYY-MM-DD.md`.

تظل الترقية إلى الذاكرة طويلة المدى تكتب إلى `MEMORY.md` فقط.

## نموذج المراحل

يستخدم Dreaming ثلاث مراحل تعاونية:

| المرحلة | الغرض                                   | الكتابة الدائمة     |
| ----- | ----------------------------------------- | ----------------- |
| Light | فرز وتجهيز المواد الحديثة قصيرة المدى | لا                |
| Deep  | تقييم المرشحين الدائمين وترقيتهم      | نعم (`MEMORY.md`) |
| REM   | التأمل في السمات والأفكار المتكررة     | لا                |

هذه المراحل تفاصيل تنفيذ داخلية، وليست "أوضاع" منفصلة يضبطها المستخدم.

### مرحلة Light

تستوعب مرحلة Light إشارات الذاكرة اليومية الحديثة وآثار الاستدعاء، وتزيل التكرار منها،
وتجهّز الأسطر المرشحة.

- تقرأ من حالة الاستدعاء قصيرة المدى، وملفات الذاكرة اليومية الحديثة، ونصوص الجلسات المنقحة عند توفرها.
- تكتب كتلة مُدارة `## Light Sleep` عندما يتضمن التخزين مخرجات مضمنة.
- تسجل إشارات تعزيز لاستخدامها لاحقًا في ترتيب Deep.
- لا تكتب أبدًا إلى `MEMORY.md`.

### مرحلة Deep

تقرر مرحلة Deep ما الذي يصبح ذاكرة طويلة المدى.

- ترتب المرشحين باستخدام تقييم موزون وبوابات حدود.
- تتطلب اجتياز `minScore` و`minRecallCount` و`minUniqueQueries`.
- تعيد ترطيب المقتطفات من الملفات اليومية الحية قبل الكتابة، لذلك يتم تخطي المقتطفات القديمة/المحذوفة.
- تُلحق الإدخالات المُرقّاة بـ `MEMORY.md`.
- تكتب ملخص `## Deep Sleep` داخل `DREAMS.md` وتكتب اختياريًا إلى `memory/dreaming/deep/YYYY-MM-DD.md`.

### مرحلة REM

تستخرج مرحلة REM الأنماط والإشارات التأملية.

- تبني ملخصات للسمات والتأملات من آثار قصيرة المدى الحديثة.
- تكتب كتلة مُدارة `## REM Sleep` عندما يتضمن التخزين مخرجات مضمنة.
- تسجل إشارات تعزيز REM المستخدمة في ترتيب Deep.
- لا تكتب أبدًا إلى `MEMORY.md`.

## إدخال نصوص الجلسات

يمكن لـ Dreaming استيعاب نصوص الجلسات المنقحة ضمن corpus الخاص بـ Dreaming. عند
توفر النصوص، يتم تمريرها إلى مرحلة Light إلى جانب إشارات
الذاكرة اليومية وآثار الاستدعاء. ويتم تنقيح المحتوى الشخصي والحساس
قبل الإدخال.

## Dream Diary

يحتفظ Dreaming أيضًا بسرد **Dream Diary** في `DREAMS.md`.
بعد أن تمتلك كل مرحلة مادة كافية، يشغّل `memory-core` دورة subagent في الخلفية
بأفضل جهد ممكن (باستخدام نموذج runtime الافتراضي) ويُلحق إدخال يوميات قصيرًا.

هذه اليوميات مخصصة للقراءة البشرية في واجهة Dreams، وليست مصدرًا للترقية.
تُستبعد عناصر اليوميات/التقارير التي أنشأها Dreaming من الترقية
قصيرة المدى. ولا تكون مؤهلة للترقية إلى
`MEMORY.md` إلا مقتطفات الذاكرة المرتكزة فقط.

يوجد أيضًا مسار backfill تاريخي مرتكز لأعمال المراجعة والاستعادة:

- يعرض `memory rem-harness --path ... --grounded` معاينة لمخرجات يوميات مرتكزة من ملاحظات `YYYY-MM-DD.md` التاريخية.
- يكتب `memory rem-backfill --path ...` إدخالات يوميات مرتكزة قابلة للعكس إلى `DREAMS.md`.
- يقوم `memory rem-backfill --path ... --stage-short-term` بتجهيز المرشحين الدائمين المرتكزين في مخزن الأدلة قصيرة المدى نفسه الذي تستخدمه مرحلة Deep العادية بالفعل.
- يزيل `memory rem-backfill --rollback` و`--rollback-short-term` عناصر backfill المرحلية تلك من دون المساس بإدخالات اليوميات العادية أو الاستدعاء الحي قصير المدى.

تكشف واجهة Control التدفق نفسه الخاص بـ backfill/reset لليوميات حتى تتمكن من فحص
النتائج في مشهد Dreams قبل تقرير ما إذا كان المرشحون المرتكزون
يستحقون الترقية. ويعرض المشهد أيضًا مسارًا مرتكزًا مميزًا حتى تتمكن من رؤية
أي الإدخالات المرحلية قصيرة المدى جاءت من إعادة تشغيل تاريخية، وأي العناصر
المُرقّاة كانت بقيادة grounded، ومسح الإدخالات المرحلية المرتكزة فقط من دون
المساس بحالة المدى القصير الحية العادية.

## إشارات ترتيب Deep

يستخدم ترتيب Deep ست إشارات أساسية موزونة بالإضافة إلى تعزيز المراحل:

| الإشارة              | الوزن | الوصف                                       |
| ------------------- | ------ | ----------------------------------------- |
| التكرار           | 0.24   | عدد إشارات المدى القصير التي راكمها الإدخال |
| الصلة           | 0.30   | متوسط جودة الاسترجاع للإدخال           |
| تنوع الاستعلامات     | 0.15   | سياقات الاستعلام/اليوم المختلفة التي أبرزته      |
| الحداثة             | 0.15   | درجة حداثة متناقصة زمنيًا                      |
| التوحيد       | 0.10   | قوة التكرار عبر عدة أيام                     |
| الثراء المفاهيمي | 0.06   | كثافة وسوم المفاهيم من المقتطف/المسار             |

تضيف إصابات مرحلتي Light وREM دفعة صغيرة متناقصة زمنيًا من
`memory/.dreams/phase-signals.json`.

## الجدولة

عند التمكين، يدير `memory-core` تلقائيًا مهمة Cron واحدة لاجتياح Dreaming كامل.
ويشغّل كل اجتياح المراحل بالترتيب: light -> REM -> deep.

سلوك الإيقاع الافتراضي:

| الإعداد              | الافتراضي     |
| -------------------- | ----------- |
| `dreaming.frequency` | `0 3 * * *` |

## بدء سريع

فعّل Dreaming:

```json
{
  "plugins": {
    "entries": {
      "memory-core": {
        "config": {
          "dreaming": {
            "enabled": true
          }
        }
      }
    }
  }
}
```

فعّل Dreaming مع إيقاع اجتياح مخصص:

```json
{
  "plugins": {
    "entries": {
      "memory-core": {
        "config": {
          "dreaming": {
            "enabled": true,
            "timezone": "America/Los_Angeles",
            "frequency": "0 */6 * * *"
          }
        }
      }
    }
  }
}
```

```
/dreaming status
/dreaming on
/dreaming off
/dreaming help
```

## سير عمل CLI

استخدم ترقية CLI للمعاينة أو التطبيق اليدوي:

```bash
openclaw memory promote
openclaw memory promote --apply
openclaw memory promote --limit 5
openclaw memory status --deep
```

يستخدم `memory promote` اليدوي حدود مرحلة Deep افتراضيًا ما لم يتم تجاوزها
بعلامات CLI.

اشرح لماذا سيجري أو لن يجري ترقية مرشح محدد:

```bash
openclaw memory promote-explain "router vlan"
openclaw memory promote-explain "router vlan" --json
```

عاين تأملات REM، والحقائق المرشحة، ومخرجات ترقية Deep من دون
كتابة أي شيء:

```bash
openclaw memory rem-harness
openclaw memory rem-harness --json
```

## القيم الافتراضية الأساسية

توجد جميع الإعدادات تحت `plugins.entries.memory-core.config.dreaming`.

| المفتاح         | الافتراضي     |
| ----------- | ----------- |
| `enabled`   | `false`     |
| `frequency` | `0 3 * * *` |

تُعد سياسة المراحل، والحدود، وسلوك التخزين تفاصيل تنفيذ داخلية
(وليست إعدادات موجهة للمستخدم).

راجع [مرجع تكوين الذاكرة](/ar/reference/memory-config#dreaming)
للحصول على القائمة الكاملة للمفاتيح.

## واجهة Dreams

عند التمكين، تعرض علامة التبويب **Dreams** في Gateway ما يلي:

- حالة تمكين Dreaming الحالية
- حالة على مستوى المرحلة ووجود الاجتياح المُدار
- أعداد المدى القصير، والمرتكزة، والإشارات، والمُرقّاة اليوم
- توقيت التشغيل المجدول التالي
- مسار Scene مرتكز مميز لإدخالات إعادة التشغيل التاريخية المرحلية
- قارئ Dream Diary قابل للتوسيع ومدعوم بواسطة `doctor.memory.dreamDiary`

## استكشاف الأخطاء وإصلاحها

### لا يعمل Dreaming مطلقًا (تعرض الحالة blocked)

يعتمد Cron المُدار لـ Dreaming على Heartbeat الخاص بالوكيل الافتراضي. إذا لم يكن Heartbeat يعمل لذلك الوكيل، فسيضع Cron حدث نظام في قائمة الانتظار من دون أن يستهلكه أحد، ولن يعمل Dreaming بصمت. سيعرض كل من `openclaw memory status` و`/dreaming status` الحالة `blocked` في هذه الحالة، مع ذكر الوكيل الذي يشكل Heartbeat الخاص به سبب الحظر.

سببان شائعان:

- يصرّح وكيل آخر بكتلة `heartbeat:` صريحة. عندما يحتوي أي إدخال في `agents.list` على كتلة `heartbeat` خاصة به، فإن هؤلاء الوكلاء فقط هم من يرسلون Heartbeat — وتتوقف القيم الافتراضية عن التطبيق على الجميع، لذلك قد يصمت الوكيل الافتراضي. انقل إعدادات Heartbeat إلى `agents.defaults.heartbeat`، أو أضف كتلة `heartbeat` صريحة إلى الوكيل الافتراضي. راجع [النطاق والأسبقية](/ar/gateway/heartbeat#scope-and-precedence).
- تكون `heartbeat.every` مساوية لـ `0` أو فارغة أو غير قابلة للتحليل. لا يمتلك Cron فترة ليجدول عليها، لذا يُعطَّل Heartbeat فعليًا. اضبط `every` على مدة موجبة مثل `30m`. راجع [القيم الافتراضية](/ar/gateway/heartbeat#defaults).

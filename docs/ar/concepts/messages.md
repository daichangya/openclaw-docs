# الرسائل

تجمع هذه الصفحة بين كيفية تعامل OpenClaw مع الرسائل الواردة، والجلسات، والاصطفاف،
والبثّ، ورؤية الاستدلال.

## تدفق الرسائل (نظرة عامة)

````
الرسالة الواردة
  -> التوجيه/الربط -> مفتاح الجلسة
  -> الاصطفاف (إذا كان هناك تشغيل نشط)
  -> تشغيل الوكيل (البثّ + الأدوات)
  -> الردود الصادرة (حدود القناة + التجزئة)
````

توجد مفاتيح الضبط الأساسية في الإعدادات:

- `messages.*` للبادئات، والاصطفاف، وسلوك المجموعات.
- `agents.defaults.*` للقيم الافتراضية لبثّ الكتل والتجزئة.
- تجاوزات القنوات (`channels.whatsapp.*`, `channels.telegram.*`، إلخ) للحدود القصوى ومفاتيح تبديل البثّ.

راجع [الإعدادات](/gateway/configuration) للاطلاع على المخطط الكامل.

## إزالة التكرار للرسائل الواردة

يمكن للقنوات إعادة تسليم الرسالة نفسها بعد إعادة الاتصال. يحتفظ OpenClaw بذاكرة
مؤقتة قصيرة العمر بمفتاح يعتمد على القناة/الحساب/النظير/الجلسة/معرّف الرسالة بحيث لا
تؤدي عمليات التسليم المكررة إلى تشغيل الوكيل مرة أخرى.

## إزالة الاهتزاز للرسائل الواردة

يمكن تجميع الرسائل السريعة المتتالية من **المرسل نفسه** في دورة وكيل واحدة
عبر `messages.inbound`. يتم تطبيق إزالة الاهتزاز لكل قناة + محادثة
وتستخدم أحدث رسالة لربط الرد/المعرّفات.

الإعدادات (القيمة الافتراضية العامة + تجاوزات لكل قناة):

```json5
{
  messages: {
    inbound: {
      debounceMs: 2000,
      byChannel: {
        whatsapp: 5000,
        slack: 1500,
        discord: 1500,
      },
    },
  },
}
```

ملاحظات:

- تُطبق إزالة الاهتزاز على الرسائل **النصية فقط**؛ أما الوسائط/المرفقات فتُمرَّر فورًا.
- تتجاوز أوامر التحكم إزالة الاهتزاز حتى تبقى مستقلة.

## الجلسات والأجهزة

الجلسات مملوكة لـ Gateway، وليست للعملاء.

- تُدمج الدردشات المباشرة في مفتاح الجلسة الرئيسية للوكيل.
- تحصل المجموعات/القنوات على مفاتيح جلسات خاصة بها.
- يعيش مخزن الجلسات والنصوص التفريغية على مضيف Gateway.

يمكن لعدة أجهزة/قنوات أن تُربط بالجسلة نفسها، ولكن لا تتم مزامنة السجل بالكامل
مرة أخرى إلى كل عميل. التوصية: استخدم جهازًا أساسيًا واحدًا للمحادثات الطويلة
لتجنب تباين السياق. تعرض واجهة Control UI وTUI دائمًا
النص التفريغي للجلسة المدعوم من Gateway، لذا فهما مصدر الحقيقة.

التفاصيل: [إدارة الجلسات](/ar/concepts/session).

## أجسام الرسائل الواردة وسياق السجل

يفصل OpenClaw بين **جسم prompt** و**جسم الأمر**:

- `Body`: نص prompt المرسل إلى الوكيل. وقد يتضمن هذا أغلفة القناة و
  أغلفة سجل اختيارية.
- `CommandBody`: نص المستخدم الخام لتحليل التوجيهات/الأوامر.
- `RawBody`: اسم بديل قديم لـ `CommandBody` (محتفظ به للتوافق).

عندما توفّر القناة سجلًا، فإنها تستخدم غلافًا مشتركًا:

- `[Chat messages since your last reply - for context]`
- `[Current message - respond to this]`

بالنسبة إلى **الدردشات غير المباشرة** (المجموعات/القنوات/الغرف)، يتم إلحاق **جسم الرسالة الحالية** في أوله بـ
تسمية المرسل (بالأسلوب نفسه المستخدم في إدخالات السجل). وهذا يحافظ على اتساق
الرسائل الفورية ورسائل الاصطفاف/السجل في prompt الخاص بالوكيل.

تكون مخازن السجل **pending-only**: فهي تتضمن رسائل المجموعات التي لم
تؤدِّ إلى تشغيل (على سبيل المثال، الرسائل المقيّدة بالإشارات) وتستبعد **الرسائل**
الموجودة بالفعل في النص التفريغي للجلسة.

تُطبق إزالة التوجيهات فقط على قسم **الرسالة الحالية** حتى يبقى السجل
سليمًا. يجب على القنوات التي تغلّف السجل ضبط `CommandBody` (أو
`RawBody`) على نص الرسالة الأصلية والإبقاء على `Body` كـ prompt مدمج.
يمكن ضبط مخازن السجل عبر `messages.groupChat.historyLimit` (القيمة
الافتراضية العامة) وتجاوزات لكل قناة مثل `channels.slack.historyLimit` أو
`channels.telegram.accounts.<id>.historyLimit` (اضبط القيمة على `0` للتعطيل).

## الاصطفاف والمتابعات

إذا كان هناك تشغيل نشط بالفعل، يمكن اصطفاف الرسائل الواردة، أو توجيهها إلى
التشغيل الحالي، أو جمعها لدورة متابعة.

- يتم الضبط عبر `messages.queue` (و`messages.queue.byChannel`).
- الأوضاع: `interrupt` و`steer` و`followup` و`collect`، بالإضافة إلى متغيرات backlog.

التفاصيل: [الاصطفاف](/ar/concepts/queue).

## البثّ، والتجزئة، والتجميع

يرسل بثّ الكتل الجزئية الردود الجزئية بينما ينتج النموذج كتل النص.
تحترم التجزئة حدود نص القناة وتتجنب تقسيم التعليمات البرمجية المحاطة بسياج.

الإعدادات الأساسية:

- `agents.defaults.blockStreamingDefault` (`on|off`، والافتراضي off)
- `agents.defaults.blockStreamingBreak` (`text_end|message_end`)
- `agents.defaults.blockStreamingChunk` (`minChars|maxChars|breakPreference`)
- `agents.defaults.blockStreamingCoalesce` (تجميع يعتمد على الخمول)
- `agents.defaults.humanDelay` (توقف يشبه البشر بين ردود الكتل)
- تجاوزات القنوات: `*.blockStreaming` و`*.blockStreamingCoalesce` (القنوات غير Telegram تتطلب ضبط `*.blockStreaming: true` صراحةً)

التفاصيل: [البثّ + التجزئة](/ar/concepts/streaming).

## رؤية الاستدلال والرموز

يمكن لـ OpenClaw إظهار استدلال النموذج أو إخفاؤه:

- يتحكم `/reasoning on|off|stream` في مستوى الرؤية.
- لا يزال محتوى الاستدلال يُحتسب ضمن استخدام الرموز عندما ينتجه النموذج.
- يدعم Telegram بثّ الاستدلال داخل فقاعة المسودة.

التفاصيل: [توجيهات Thinking + الاستدلال](/ar/tools/thinking) و[استخدام الرموز](/ar/reference/token-use).

## البادئات، والتسلسل الخيطي، والردود

تتم مركزة تنسيق الرسائل الصادرة في `messages`:

- `messages.responsePrefix` و`channels.<channel>.responsePrefix` و`channels.<channel>.accounts.<id>.responsePrefix` (سلسلة بادئات الرسائل الصادرة)، بالإضافة إلى `channels.whatsapp.messagePrefix` (بادئة الرسائل الواردة في WhatsApp)
- تسلسل خيطي للردود عبر `replyToMode` والقيم الافتراضية لكل قناة

التفاصيل: [الإعدادات](/ar/gateway/configuration-reference#messages) ووثائق القنوات.

## الردود الصامتة

إن الرمز الصامت المطابق تمامًا `NO_REPLY` / `no_reply` يعني “لا تُسلِّم ردًا مرئيًا للمستخدم”.
ويحل OpenClaw هذا السلوك بحسب نوع المحادثة:

- المحادثات المباشرة لا تسمح بالصمت افتراضيًا وتعيد كتابة الرد
  الصامت المجرد إلى بديل قصير مرئي.
- المجموعات/القنوات تسمح بالصمت افتراضيًا.
- التنسيق الداخلي يسمح بالصمت افتراضيًا.

توجد القيم الافتراضية تحت `agents.defaults.silentReply` و
`agents.defaults.silentReplyRewrite`، ويمكن لـ `surfaces.<id>.silentReply` و
`surfaces.<id>.silentReplyRewrite` تجاوزها لكل سطح.

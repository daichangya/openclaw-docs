تهدف حزمة QA الخاصة إلى اختبار OpenClaw بطريقة أكثر واقعية،
وبصياغة أقرب إلى القنوات، مما يمكن لاختبار وحدة واحد أن يقدمه.

المكونات الحالية:

- `extensions/qa-channel`: قناة رسائل اصطناعية بأسطح للرسائل الخاصة، والقنوات، والسلاسل،
  والتفاعلات، والتحرير، والحذف.
- `extensions/qa-lab`: واجهة مستخدم للتصحيح وناقل QA لمراقبة النص،
  وحقن الرسائل الواردة، وتصدير تقرير Markdown.
- `qa/`: أصول مزروعة ومدعومة بالمستودع لمهمة الانطلاق وسيناريوهات QA
  الأساسية.

تدفق تشغيل QA الحالي للمشغل هو موقع QA ذي لوحتين:

- اليسار: لوحة معلومات Gateway ‏(Control UI) مع الوكيل.
- اليمين: QA Lab، تعرض نصًا شبيهًا بـ Slack وخطة السيناريو.

شغّله باستخدام:

```bash
pnpm qa:lab:up
```

يبني ذلك موقع QA، ويبدأ مسار Gateway المدعوم بـ Docker، ويكشف
صفحة QA Lab حيث يمكن للمشغل أو حلقة الأتمتة إعطاء الوكيل
مهمة QA، ومراقبة السلوك الحقيقي للقناة، وتسجيل ما نجح أو فشل أو
بقي محجوبًا.

وللحصول على تكرار أسرع في واجهة QA Lab من دون إعادة بناء صورة Docker في كل مرة،
ابدأ الحزمة باستخدام حزمة QA Lab مركبة بربط bind mount:

```bash
pnpm openclaw qa docker-build-image
pnpm qa:lab:build
pnpm qa:lab:up:fast
pnpm qa:lab:watch
```

يبقي `qa:lab:up:fast` خدمات Docker على صورة مسبقة البناء ويقوم بربط
`extensions/qa-lab/web/dist` داخل حاوية `qa-lab`. ويعيد `qa:lab:watch`
بناء تلك الحزمة عند التغيير، ويُعاد تحميل المتصفح تلقائيًا عندما تتغير
قيمة hash لأصل QA Lab.

ولتشغيل مسار smoke حقيقي للنقل لـ Matrix، شغّل:

```bash
pnpm openclaw qa matrix
```

يقوم هذا المسار بتجهيز خادم Tuwunel منزلي مؤقت في Docker، ويسجل
مستخدمين مؤقتين للسائق، وSUT، والمراقب، وينشئ غرفة خاصة واحدة، ثم يشغّل
Plugin Matrix الحقيقي داخل عملية فرعية لـ QA gateway. ويحافظ مسار النقل الحي
على تهيئة العملية الفرعية ضمن نطاق النقل الجاري اختباره، لذلك يعمل Matrix
من دون `qa-channel` في تهيئة العملية الفرعية. ويكتب أصول التقرير المهيكلة
وسجل stdout/stderr المدمج في دليل خرج Matrix QA المحدد. ولالتقاط
خرج البناء/المشغّل الخارجي لـ `scripts/run-node.mjs` أيضًا، اضبط
`OPENCLAW_RUN_NODE_OUTPUT_LOG=<path>` إلى ملف سجل محلي ضمن المستودع.
يُطبع تقدم Matrix افتراضيًا. ويحد `OPENCLAW_QA_MATRIX_TIMEOUT_MS`
التشغيل الكامل، ويحد `OPENCLAW_QA_MATRIX_CLEANUP_TIMEOUT_MS` التنظيف بحيث
يعرض تفكيك Docker العالق أمر الاستعادة الدقيق بدلًا من التعليق.

ولتشغيل مسار smoke حقيقي للنقل لـ Telegram، شغّل:

```bash
pnpm openclaw qa telegram
```

يستهدف هذا المسار مجموعة Telegram خاصة وحقيقية واحدة بدلًا من تجهيز
خادم مؤقت. ويتطلب `OPENCLAW_QA_TELEGRAM_GROUP_ID`,
`OPENCLAW_QA_TELEGRAM_DRIVER_BOT_TOKEN`, و
`OPENCLAW_QA_TELEGRAM_SUT_BOT_TOKEN`، بالإضافة إلى روبوتين متميزين
في المجموعة الخاصة نفسها. ويجب أن يكون لدى روبوت SUT اسم مستخدم Telegram،
وتعمل المراقبة من روبوت إلى روبوت بأفضل شكل عندما يكون لدى كلا الروبوتين
Bot-to-Bot Communication Mode مفعّلًا في `@BotFather`.
يخرج الأمر بحالة غير صفرية عند فشل أي سيناريو. استخدم `--allow-failures` عندما
تريد الأصول من دون رمز خروج فاشل.
يتضمن تقرير Telegram وملخصه RTT لكل رد، من طلب
إرسال رسالة السائق إلى رد SUT المرصود، بدءًا من canary.

قبل استخدام بيانات اعتماد حية مجمّعة، شغّل:

```bash
pnpm openclaw qa credentials doctor
```

يفحص doctor متغيرات env لوسيط Convex، ويتحقق من إعدادات نقاط النهاية، ويتحقق
من إمكانية الوصول للإدارة/القائمة عندما يكون سر المشرف موجودًا. ولا يبلغ
إلا عن حالة الضبط/الفقدان للأسرار.

ولتشغيل مسار smoke حقيقي للنقل لـ Discord، شغّل:

```bash
pnpm openclaw qa discord
```

يستهدف هذا المسار قناة نقابة Discord خاصة وحقيقية واحدة مع روبوتين: روبوت
سائق يتحكم به الحزام، وروبوت SUT يبدأه OpenClaw gateway الفرعي عبر
Plugin Discord المضمن. ويتطلب
`OPENCLAW_QA_DISCORD_GUILD_ID`, و`OPENCLAW_QA_DISCORD_CHANNEL_ID`,
و`OPENCLAW_QA_DISCORD_DRIVER_BOT_TOKEN`, و`OPENCLAW_QA_DISCORD_SUT_BOT_TOKEN`,
و`OPENCLAW_QA_DISCORD_SUT_APPLICATION_ID` عند استخدام بيانات اعتماد env.
ويتحقق المسار من التعامل مع الإشارة في القناة ويتأكد من أن روبوت SUT قد
سجل الأمر الأصلي `/help` لدى Discord.
يخرج الأمر بحالة غير صفرية عند فشل أي سيناريو. استخدم `--allow-failures` عندما
تريد الأصول من دون رمز خروج فاشل.

تشارك مسارات النقل الحية الآن عقدًا أصغر واحدًا بدلًا من أن يخترع كل منها
شكله الخاص لقائمة السيناريوهات:

يبقى `qa-channel` مجموعة سلوك المنتج الاصطناعية الواسعة ولا يدخل ضمن
مصفوفة تغطية النقل الحي.

| -------- | ------ | -------------- | --------------- | --------------- | -------------- | ---------------- | ---------------- | -------------------- | ------------ | --------------------------- |
| Matrix   | x      | x              | x               | x               | x              | x                | x                | x                    |              |                             |
| Telegram | x      | x              |                 |                 |                |                  |                  |                      | x            |                             |
| Discord  | x      | x              |                 |                 |                |                  |                  |                      |              | x                           |

يبقي هذا `qa-channel` مجموعة سلوك المنتج الواسعة، بينما تشترك
Matrix وTelegram ووسائط النقل الحية المستقبلية في قائمة فحص صريحة واحدة
لعقد النقل.

ولتشغيل مسار VM مؤقت على Linux من دون إدخال Docker في مسار QA، شغّل:

```bash
pnpm openclaw qa suite --runner multipass --scenario channel-chat-baseline
```

يقوم هذا بتشغيل ضيف Multipass جديد، وتثبيت الاعتمادات، وبناء OpenClaw
داخل الضيف، وتشغيل `qa suite`، ثم ينسخ تقرير QA العادي
والملخص إلى `.artifacts/qa-e2e/...` على المضيف.
ويعيد استخدام سلوك اختيار السيناريو نفسه مثل `qa suite` على المضيف.
وتشغّل عمليات suite على المضيف وMultipass عدة سيناريوهات محددة بالتوازي
مع عمال Gateway معزولين افتراضيًا. ويستخدم `qa-channel` افتراضيًا توازيًا
بقيمة 4، مع حد أقصى بعدد السيناريوهات المحددة. استخدم `--concurrency <count>`
لضبط عدد العمال، أو `--concurrency 1` للتنفيذ التسلسلي.
يخرج الأمر بحالة غير صفرية عند فشل أي سيناريو. استخدم `--allow-failures` عندما
تريد الأصول من دون رمز خروج فاشل.
تمرر التشغيلات الحية مدخلات مصادقة QA المدعومة والعملية للضيف:
مفاتيح المزوّد المستندة إلى env، ومسار تهيئة مزوّد QA الحي، و
`CODEX_HOME` عند وجوده. أبقِ `--output-dir` تحت جذر المستودع كي يتمكن
الضيف من الكتابة مجددًا عبر مساحة العمل المركبة.

## البذور المدعومة بالمستودع

تعيش أصول البذور في `qa/`:

- `qa/scenarios/index.md`
- `qa/scenarios/<theme>/*.md`

وهذه موجودة في git عمدًا بحيث تكون خطة QA مرئية للبشر
وللوكيل معًا.

يجب أن يبقى `qa-lab` مشغّل Markdown عامًا. وكل ملف Markdown للسيناريو
هو مصدر الحقيقة لتشغيل اختبار واحد ويجب أن يعرّف:

- بيانات تعريف السيناريو
- بيانات تعريف اختيارية للفئة، والقدرة، والمسار، والمخاطر
- مراجع المستندات والكود
- متطلبات Plugin اختيارية
- تصحيحًا اختياريًا لتهيئة Gateway
- `qa-flow` القابل للتنفيذ

يُسمح لسطح بيئة التشغيل القابل لإعادة الاستخدام الذي يدعم `qa-flow` بأن يبقى عامًا
وعابرًا للحدود. فعلى سبيل المثال، يمكن لسيناريوهات Markdown أن تجمع
بين مساعدات جهة النقل ومساعدات جهة المتصفح التي تقود Control UI المضمنة عبر
وصلة `browser.request` في Gateway من دون إضافة مشغّل بحالة خاصة.

يجب تجميع ملفات السيناريوهات حسب قدرة المنتج لا حسب
مجلد شجرة المصدر. وأبقِ معرّفات السيناريوهات مستقرة عند نقل الملفات؛ واستخدم
`docsRefs` و`codeRefs` لتتبّع التنفيذ.

يجب أن تبقى القائمة الأساسية واسعة بما يكفي لتغطية:

- الدردشة في الرسائل الخاصة والقنوات
- سلوك السلاسل
- دورة حياة إجراءات الرسائل
- نداءات Cron الراجعة
- استرجاع الذاكرة
- تبديل النموذج
- التسليم إلى وكيل فرعي
- قراءة المستودع وقراءة المستندات
- مهمة بناء صغيرة مثل Lobster Invaders

## مسارات المحاكاة للمزوّد

يحتوي `qa suite` على مسارين محليين لمحاكاة المزوّد:

- `mock-openai` هو محاكي OpenClaw المدرك للسيناريو. ويبقى
  مسار المحاكاة الحتمي الافتراضي لـ QA المدعوم بالمستودع وبوابات التطابق.
- يبدأ `aimock` خادم مزوّد مدعومًا بـ AIMock لتغطية البروتوكول والتجهيزات
  والتسجيل/إعادة التشغيل والفوضى بشكل تجريبي. وهو إضافي ولا يستبدل
  موزع السيناريوهات `mock-openai`.

يعيش تنفيذ مسار المزوّد تحت `extensions/qa-lab/src/providers/`.
ويمتلك كل مزوّد إعداداته الافتراضية، وبدء تشغيل الخادم المحلي، وتهيئة نموذج Gateway،
واحتياجات تجهيز ملف المصادقة، وعلامات القدرات الحية/الوهمية. ويجب أن يمر
كود suite وGateway المشترك عبر سجل المزوّد بدلًا من التفريع حسب أسماء
المزوّدين.

## مهايئات النقل

يمتلك `qa-lab` وصلة نقل عامة لسيناريوهات Markdown الخاصة بـ QA.
ويُعد `qa-channel` أول مهايئ على هذه الوصلة، لكن هدف التصميم أوسع:
يجب أن تتمكن القنوات الحقيقية أو الاصطناعية المستقبلية من الاندماج في مشغّل suite نفسه
بدلًا من إضافة مشغّل QA خاص بالنقل.

على مستوى البنية، يكون التقسيم كما يلي:

- يمتلك `qa-lab` تنفيذ السيناريو العام، وتوازي العمال، وكتابة الأصول، وإعداد التقارير.
- يمتلك مهايئ النقل تهيئة Gateway، والجاهزية، ومراقبة الوارد والصادر، وإجراءات النقل، وحالة النقل المعيارية.
- تحدد ملفات سيناريو Markdown تحت `qa/scenarios/` تشغيل الاختبار؛ بينما يوفر `qa-lab` سطح بيئة التشغيل القابل لإعادة الاستخدام الذي ينفذها.

توجد إرشادات التبنّي الموجهة للمشرفين لمهايئات القنوات الجديدة في
[الاختبار](/ar/help/testing#adding-a-channel-to-qa).

## إعداد التقارير

يصدر `qa-lab` تقرير بروتوكول Markdown من الخط الزمني المرصود للناقل.
ويجب أن يجيب التقرير عن:

- ما الذي نجح
- ما الذي فشل
- ما الذي بقي محجوبًا
- ما سيناريوهات المتابعة الجديرة بالإضافة

لفحوصات الشخصية والأسلوب، شغّل السيناريو نفسه عبر عدة مراجع نماذج حية
واكتب تقرير Markdown مُحكَّمًا:

```bash
pnpm openclaw qa character-eval \
  --model openai/gpt-5.4,thinking=medium,fast \
  --model openai/gpt-5.2,thinking=xhigh \
  --model openai/gpt-5,thinking=xhigh \
  --model anthropic/claude-opus-4-6,thinking=high \
  --model anthropic/claude-sonnet-4-6,thinking=high \
  --model zai/glm-5.1,thinking=high \
  --model moonshot/kimi-k2.5,thinking=high \
  --model google/gemini-3.1-pro-preview,thinking=high \
  --judge-model openai/gpt-5.4,thinking=xhigh,fast \
  --judge-model anthropic/claude-opus-4-6,thinking=high \
  --blind-judge-models \
  --concurrency 16 \
  --judge-concurrency 16
```

يشغّل الأمر عمليات فرعية محلية لـ QA gateway، وليس Docker. ويجب أن تضبط
سيناريوهات تقييم الشخصية الشخصيةَ عبر `SOUL.md`، ثم تشغّل أدوار مستخدم عادية
مثل الدردشة، والمساعدة في مساحة العمل، ومهام الملفات الصغيرة. ولا ينبغي
إخبار النموذج المرشح بأنه قيد التقييم. ويحافظ الأمر على كل نص كامل،
ويسجل إحصاءات تشغيل أساسية، ثم يطلب من نماذج التحكيم في الوضع السريع مع
استدلال `xhigh` عند دعمه أن ترتب التشغيلات بحسب الطبيعية، والانطباع، والفكاهة.
استخدم `--blind-judge-models` عند مقارنة المزوّدين: إذ لا يزال تلقين التحكيم
يتلقى كل نص وحالة تشغيل، لكن تُستبدل مراجع المرشحين بتسميات محايدة مثل
`candidate-01`; ثم يربط التقرير الترتيب بالمراجع الحقيقية بعد التحليل.
تستخدم تشغيلات المرشحين افتراضيًا مستوى تفكير `high`، مع `medium` لـ GPT-5.4 و`xhigh`
لمراجع تقييم OpenAI الأقدم التي تدعم ذلك. ويمكنك تجاوز مرشح محدد مضمنًا عبر
`--model provider/model,thinking=<level>`. ولا يزال `--thinking <level>` يضبط
بديلًا عامًا، كما أن الصيغة الأقدم `--model-thinking <provider/model=level>` ما زالت
محتفظًا بها للتوافق.
تستخدم مراجع مرشحي OpenAI الوضع السريع افتراضيًا بحيث تُستخدم المعالجة ذات
الأولوية عندما يدعم المزوّد ذلك. أضف `,fast` أو `,no-fast` أو `,fast=false` مضمنةً عندما
يحتاج مرشح واحد أو حكم واحد إلى تجاوز. مرّر `--fast` فقط عندما تريد فرض
الوضع السريع على كل نموذج مرشح. وتُسجل مدد المرشحين والحكام في التقرير
لتحليل القياس، لكن تلقينات التحكيم تنص صراحةً على عدم الترتيب حسب السرعة.
يستخدم كل من تشغيلات نماذج المرشحين والحكام توازيًا افتراضيًا بقيمة 16. خفّض
`--concurrency` أو `--judge-concurrency` عندما تجعل حدود المزوّد أو ضغط
Gateway المحلي التشغيل صاخبًا أكثر من اللازم.
عندما لا يُمرَّر أي مرشح `--model`، يستخدم تقييم الشخصية افتراضيًا
`openai/gpt-5.4` و`openai/gpt-5.2` و`openai/gpt-5` و`anthropic/claude-opus-4-6`,
و`anthropic/claude-sonnet-4-6` و`zai/glm-5.1`,
و`moonshot/kimi-k2.5` و
`google/gemini-3.1-pro-preview` عندما لا يتم تمرير `--model`.
وعندما لا يتم تمرير `--judge-model`، تكون القيم الافتراضية للحكام هي
`openai/gpt-5.4,thinking=xhigh,fast` و
`anthropic/claude-opus-4-6,thinking=high`.

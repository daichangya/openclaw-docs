مساحة العمل هي منزل الوكيل. وهي دليل العمل الوحيد المستخدم
لأدوات الملفات ولسياق مساحة العمل. حافظ على خصوصيتها وتعامل معها كأنها ذاكرة.

وهذا منفصل عن `~/.openclaw/`، الذي يخزن الإعداد وبيانات الاعتماد
والجلسات.

**مهم:** مساحة العمل هي **cwd الافتراضي**، وليست sandbox صارمًا. تقوم
الأدوات بحل المسارات النسبية انطلاقًا من مساحة العمل، لكن المسارات المطلقة
لا يزال بإمكانها الوصول إلى أماكن أخرى على المضيف ما لم يتم تمكين sandboxing. وإذا كنت
تحتاج إلى العزل، فاستخدم
[`agents.defaults.sandbox`](/ar/gateway/sandboxing) (و/أو إعداد sandbox لكل وكيل).
عندما يكون sandboxing مفعّلًا وتكون `workspaceAccess` ليست `"rw"`، تعمل الأدوات
داخل مساحة عمل sandbox ضمن `~/.openclaw/sandboxes`، وليس داخل مساحة
عمل المضيف الخاصة بك.

## الموقع الافتراضي

- الافتراضي: `~/.openclaw/workspace`
- إذا كانت `OPENCLAW_PROFILE` معيّنة وليست `"default"`، يصبح الافتراضي
  `~/.openclaw/workspace-<profile>`.
- يمكنك التجاوز في `~/.openclaw/openclaw.json`:

```json5
{
  agents: {
    defaults: {
      workspace: "~/.openclaw/workspace",
    },
  },
}
```

سيقوم `openclaw onboard` أو `openclaw configure` أو `openclaw setup` بإنشاء
مساحة العمل وتهيئة ملفات bootstrap إذا كانت مفقودة.
ولا تقبل نسخ بذور sandbox إلا الملفات العادية داخل مساحة العمل؛ أما
الأسماء المستعارة symlink/hardlink التي تُحل خارج مساحة العمل المصدر فيتم تجاهلها.

إذا كنت تدير ملفات مساحة العمل بنفسك بالفعل، يمكنك تعطيل إنشاء ملفات
bootstrap:

```json5
{ agents: { defaults: { skipBootstrap: true } } }
```

## مجلدات مساحة العمل الإضافية

قد تكون التثبيتات الأقدم قد أنشأت `~/openclaw`. ويمكن أن يؤدي الاحتفاظ بعدة
أدلة لمساحات العمل إلى انحراف مربك في المصادقة أو الحالة، لأن مساحة
عمل واحدة فقط تكون نشطة في كل مرة.

**التوصية:** احتفظ بمساحة عمل نشطة واحدة. وإذا لم تعد تستخدم
المجلدات الإضافية، فقم بأرشفتها أو انقلها إلى سلة المهملات (مثلًا `trash ~/openclaw`).
وإذا كنت تحتفظ عمدًا بعدة مساحات عمل، فتأكد من أن
`agents.defaults.workspace` يشير إلى المساحة النشطة.

يحذّر `openclaw doctor` عندما يكتشف أدلة مساحة عمل إضافية.

## خريطة ملفات مساحة العمل (ماذا يعني كل ملف)

هذه هي الملفات القياسية التي يتوقعها OpenClaw داخل مساحة العمل:

- `AGENTS.md`
  - تعليمات التشغيل الخاصة بالوكيل وكيف ينبغي له استخدام الذاكرة.
  - يتم تحميله في بداية كل جلسة.
  - مكان جيد للقواعد والأولويات وتفاصيل "كيفية التصرف".

- `SOUL.md`
  - الشخصية والنبرة والحدود.
  - يتم تحميله في كل جلسة.
  - الدليل: [دليل شخصية SOUL.md](/ar/concepts/soul)

- `USER.md`
  - من هو المستخدم وكيفية مخاطبته.
  - يتم تحميله في كل جلسة.

- `IDENTITY.md`
  - اسم الوكيل وطابعه والرمز التعبيري الخاص به.
  - يتم إنشاؤه/تحديثه خلال طقس bootstrap.

- `TOOLS.md`
  - ملاحظات حول أدواتك المحلية واتفاقياتك.
  - لا يتحكم في توفر الأدوات؛ بل هو مجرد إرشاد.

- `HEARTBEAT.md`
  - قائمة تحقق صغيرة اختيارية لتشغيلات Heartbeat.
  - أبقِه قصيرًا لتجنب استهلاك الرموز.

- `BOOT.md`
  - قائمة تحقق اختيارية لبدء التشغيل تُنفّذ تلقائيًا عند إعادة تشغيل Gateway (عند تمكين [الخطافات الداخلية](/ar/automation/hooks)).
  - أبقِها قصيرة؛ واستخدم أداة الرسائل للإرسالات الصادرة.

- `BOOTSTRAP.md`
  - طقس تشغيل أولي لمرة واحدة.
  - لا يتم إنشاؤه إلا لمساحة عمل جديدة تمامًا.
  - احذفه بعد اكتمال الطقس.

- `memory/YYYY-MM-DD.md`
  - سجل ذاكرة يومي (ملف واحد لكل يوم).
  - يوصى بقراءة ملفي اليوم والأمس عند بدء الجلسة.

- `MEMORY.md` (اختياري)
  - ذاكرة طويلة الأجل منسقة.
  - يتم تحميلها فقط في الجلسة الرئيسية الخاصة (وليس في السياقات المشتركة/الجماعية).

راجع [الذاكرة](/ar/concepts/memory) للاطلاع على سير العمل والتفريغ التلقائي للذاكرة.

- `skills/` (اختياري)
  - Skills خاصة بمساحة العمل.
  - موقع Skills الأعلى أولوية لتلك المساحة.
  - يتجاوز Skills الوكيل الخاصة بالمشروع وSkills الوكيل الشخصية وSkills المُدارة وSkills المضمنة و`skills.load.extraDirs` عندما تتصادم الأسماء.

- `canvas/` (اختياري)
  - ملفات واجهة مستخدم Canvas لعروض العقد (مثل `canvas/index.html`).

إذا كان أي ملف bootstrap مفقودًا، يحقن OpenClaw علامة "ملف مفقود" داخل
الجلسة ويتابع. ويتم اقتطاع ملفات bootstrap الكبيرة عند حقنها؛ يمكنك
ضبط الحدود باستخدام `agents.defaults.bootstrapMaxChars` (الافتراضي: 12000) و
`agents.defaults.bootstrapTotalMaxChars` (الافتراضي: 60000).
يمكن لـ `openclaw setup` إعادة إنشاء القيم الافتراضية المفقودة من دون
الكتابة فوق الملفات الموجودة.

## ما الذي ليس داخل مساحة العمل

توجد هذه العناصر تحت `~/.openclaw/` ويجب **ألا** تُسجّل في مستودع مساحة العمل:

- `~/.openclaw/openclaw.json` (الإعداد)
- `~/.openclaw/agents/<agentId>/agent/auth-profiles.json` (ملفات تعريف مصادقة النموذج: OAuth + مفاتيح API)
- `~/.openclaw/credentials/` (حالة القناة/المزوّد بالإضافة إلى بيانات استيراد OAuth القديمة)
- `~/.openclaw/agents/<agentId>/sessions/` (نصوص الجلسات + البيانات الوصفية)
- `~/.openclaw/skills/` (Skills المُدارة)

إذا كنت بحاجة إلى ترحيل الجلسات أو الإعداد، فانقلها بشكل منفصل وأبقها
خارج التحكم في الإصدارات.

## النسخ الاحتياطي عبر Git (موصى به، خاص)

تعامل مع مساحة العمل على أنها ذاكرة خاصة. ضعها في مستودع git **خاص** حتى
تكون محفوظة وقابلة للاستعادة.

نفّذ هذه الخطوات على الجهاز الذي تعمل عليه Gateway (فهذا هو المكان الذي
توجد فيه مساحة العمل).

### 1) تهيئة المستودع

إذا كان git مثبتًا، تتم تهيئة مساحات العمل الجديدة تلقائيًا. وإذا لم تكن
مساحة العمل هذه مستودعًا بالفعل، فنفّذ:

```bash
cd ~/.openclaw/workspace
git init
git add AGENTS.md SOUL.md TOOLS.md IDENTITY.md USER.md HEARTBEAT.md memory/
git commit -m "Add agent workspace"
```

### 2) إضافة remote خاص (خيارات مناسبة للمبتدئين)

الخيار A: واجهة GitHub على الويب

1. أنشئ مستودعًا جديدًا **خاصًا** على GitHub.
2. لا تهيئه بملف README (لتجنب تعارضات الدمج).
3. انسخ عنوان URL الخاص بـ HTTPS remote.
4. أضف remote وادفع التغييرات:

```bash
git branch -M main
git remote add origin <https-url>
git push -u origin main
```

الخيار B: GitHub CLI ‏(`gh`)

```bash
gh auth login
gh repo create openclaw-workspace --private --source . --remote origin --push
```

الخيار C: واجهة GitLab على الويب

1. أنشئ مستودعًا جديدًا **خاصًا** على GitLab.
2. لا تهيئه بملف README (لتجنب تعارضات الدمج).
3. انسخ عنوان URL الخاص بـ HTTPS remote.
4. أضف remote وادفع التغييرات:

```bash
git branch -M main
git remote add origin <https-url>
git push -u origin main
```

### 3) التحديثات المستمرة

```bash
git status
git add .
git commit -m "Update memory"
git push
```

## لا تسجّل الأسرار

حتى في مستودع خاص، تجنب تخزين الأسرار في مساحة العمل:

- مفاتيح API أو رموز OAuth أو كلمات المرور أو بيانات الاعتماد الخاصة.
- أي شيء تحت `~/.openclaw/`.
- تفريغات خام للدردشات أو المرفقات الحساسة.

إذا اضطررت إلى تخزين مراجع حساسة، فاستخدم عناصر نائبة واحتفظ بالسر
الحقيقي في مكان آخر (مدير كلمات المرور أو متغيرات البيئة أو `~/.openclaw/`).

بداية مقترحة لملف `.gitignore`:

```gitignore
.DS_Store
.env
**/*.key
**/*.pem
**/secrets*
```

## نقل مساحة العمل إلى جهاز جديد

1. انسخ المستودع إلى المسار المطلوب (الافتراضي `~/.openclaw/workspace`).
2. عيّن `agents.defaults.workspace` إلى ذلك المسار في `~/.openclaw/openclaw.json`.
3. شغّل `openclaw setup --workspace <path>` لتهيئة أي ملفات مفقودة.
4. إذا كنت بحاجة إلى الجلسات، فانقل `~/.openclaw/agents/<agentId>/sessions/` من
   الجهاز القديم بشكل منفصل.

## ملاحظات متقدمة

- يمكن أن يستخدم التوجيه متعدد الوكلاء مساحات عمل مختلفة لكل وكيل. راجع
  [توجيه القنوات](/ar/channels/channel-routing) للاطلاع على إعدادات التوجيه.
- إذا كانت `agents.defaults.sandbox` مفعّلة، فقد تستخدم الجلسات غير الرئيسية
  مساحات عمل sandbox لكل جلسة ضمن `agents.defaults.sandbox.workspaceRoot`.

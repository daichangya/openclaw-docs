## نموذج القدرات العام

القدرات هي نموذج **الإضافات الأصلية** العام داخل OpenClaw. يسجّل كل Plugin أصلي في OpenClaw نفسه مقابل نوع قدرة واحد أو أكثر:

| القدرة                 | طريقة التسجيل                                   | أمثلة على الإضافات                  |
| ---------------------- | ------------------------------------------------ | ----------------------------------- |
| الاستدلال النصي        | `api.registerProvider(...)`                      | `openai`, `anthropic`               |
| الواجهة الخلفية لاستدلال CLI | `api.registerCliBackend(...)`                    | `openai`, `anthropic`               |
| الكلام                 | `api.registerSpeechProvider(...)`                | `elevenlabs`, `microsoft`           |
| النسخ الفوري           | `api.registerRealtimeTranscriptionProvider(...)` | `openai`                            |
| الصوت الفوري           | `api.registerRealtimeVoiceProvider(...)`         | `openai`                            |
| فهم الوسائط            | `api.registerMediaUnderstandingProvider(...)`    | `openai`, `google`                  |
| توليد الصور            | `api.registerImageGenerationProvider(...)`       | `openai`, `google`, `fal`, `minimax` |
| توليد الموسيقى         | `api.registerMusicGenerationProvider(...)`       | `google`, `minimax`                 |
| توليد الفيديو          | `api.registerVideoGenerationProvider(...)`       | `qwen`                              |
| جلب الويب              | `api.registerWebFetchProvider(...)`              | `firecrawl`                         |
| البحث في الويب         | `api.registerWebSearchProvider(...)`             | `google`                            |
| القناة / المراسلة      | `api.registerChannel(...)`                       | `msteams`, `matrix`                 |

الإضافة التي تسجّل صفرًا من القدرات لكنها توفّر hooks أو tools أو services هي إضافة **قديمة تعتمد على hooks فقط**. لا يزال هذا النمط مدعومًا بالكامل.

### موقف التوافق الخارجي

تم اعتماد نموذج القدرات في النواة ويُستخدم اليوم بواسطة الإضافات المضمّنة/الأصلية، لكن توافق الإضافات الخارجية لا يزال يحتاج إلى معيار أكثر صرامة من "تم تصديره، إذًا فهو ثابت."

الإرشادات الحالية:

- **الإضافات الخارجية الحالية:** حافظ على عمل عمليات التكامل القائمة على hooks؛ واعتبر هذا هو خط الأساس للتوافق
- **الإضافات المضمّنة/الأصلية الجديدة:** فضّل تسجيل القدرات الصريح بدلًا من الوصولات الخاصة بالمورّد أو التصاميم الجديدة التي تعتمد على hooks فقط
- **الإضافات الخارجية التي تعتمد تسجيل القدرات:** هذا مسموح، لكن اعتبر أسطح المساعدات الخاصة بالقدرات في حالة تطور ما لم توضح الوثائق صراحةً أن العقد مستقر

القاعدة العملية:

- واجهات API الخاصة بتسجيل القدرات هي الاتجاه المقصود
- تظل hooks القديمة المسار الأكثر أمانًا لتجنّب الكسر بالنسبة إلى الإضافات الخارجية خلال مرحلة الانتقال
- ليست كل المسارات الفرعية المساعدة المصدّرة متساوية؛ فضّل العقد الضيق الموثّق، لا تصديرات المساعدات العرضية

### أشكال Plugin

يصنّف OpenClaw كل إضافة يتم تحميلها إلى شكل معيّن بناءً على سلوك تسجيلها الفعلي (وليس فقط بياناتها الوصفية الثابتة):

- **plain-capability** -- يسجّل نوع قدرة واحدًا فقط بالضبط (مثلًا إضافة مزوّد فقط مثل `mistral`)
- **hybrid-capability** -- يسجّل أنواع قدرات متعددة (مثلًا `openai` يمتلك الاستدلال النصي والكلام وفهم الوسائط وتوليد الصور)
- **hook-only** -- يسجّل hooks فقط (مكتوبة الأنواع أو مخصّصة)، من دون قدرات أو tools أو commands أو services
- **non-capability** -- يسجّل tools أو commands أو services أو routes لكن من دون قدرات

استخدم `openclaw plugins inspect <id>` لرؤية شكل الإضافة وتفصيل قدراتها. انظر [مرجع CLI](/cli/plugins#inspect) لمزيد من التفاصيل.

### hooks القديمة

لا يزال hook `before_agent_start` مدعومًا كمسار توافق للإضافات التي تعتمد على hooks فقط. ولا تزال الإضافات الواقعية القديمة تعتمد عليه.

الاتجاه:

- أبقه يعمل
- وثّقه على أنه قديم
- فضّل `before_model_resolve` لأعمال تجاوز النموذج/المزوّد
- فضّل `before_prompt_build` لأعمال تعديل prompt
- لا تُزِلْه إلا بعد تراجع الاستخدام الفعلي وإثبات تغطية fixtures لسلامة الترحيل

### إشارات التوافق

عند تشغيل `openclaw doctor` أو `openclaw plugins inspect <id>`، قد ترى إحدى هذه التسميات:

| الإشارة                    | المعنى                                                       |
| -------------------------- | ------------------------------------------------------------ |
| **config valid**           | تم تحليل الإعداد بنجاح وتم حل الإضافات                       |
| **compatibility advisory** | تستخدم الإضافة نمطًا مدعومًا لكنه أقدم (مثل `hook-only`)    |
| **legacy warning**         | تستخدم الإضافة `before_agent_start`، وهو مهمل               |
| **hard error**             | الإعداد غير صالح أو فشل تحميل الإضافة                        |

لن يؤدّي أي من `hook-only` أو `before_agent_start` إلى كسر إضافتك اليوم -- فـ `hook-only` إرشادي، و`before_agent_start` لا يؤدي إلا إلى تحذير. تظهر هذه الإشارات أيضًا في `openclaw status --all` و `openclaw plugins doctor`.

## نظرة عامة على البنية

يحتوي نظام الإضافات في OpenClaw على أربع طبقات:

1. **البيان + الاكتشاف**
   يعثر OpenClaw على الإضافات المرشحة من المسارات المهيأة وجذور مساحات العمل وجذور الامتدادات العامة والامتدادات المضمّنة. يقرأ الاكتشاف أولًا بيانات `openclaw.plugin.json` الخاصة بالإضافة الأصلية، بالإضافة إلى بيانات الحِزم المدعومة.
2. **التمكين + التحقق**
   تقرر النواة ما إذا كانت الإضافة المكتشفة مفعّلة أو معطّلة أو محجوبة أو محددة لفتحة حصرية مثل الذاكرة.
3. **التحميل وقت التشغيل**
   يتم تحميل إضافات OpenClaw الأصلية داخل العملية عبر jiti وتسجيل القدرات في سجل مركزي. وتُطبَّع الحِزم المتوافقة إلى سجلات في السجل دون استيراد شيفرة وقت التشغيل.
4. **استهلاك السطح**
   يقرأ باقي OpenClaw السجل لعرض tools والقنوات وإعداد المزوّد وhooks وHTTP routes وأوامر CLI وservices.

وبالنسبة إلى Plugin CLI تحديدًا، ينقسم اكتشاف أوامر الجذر إلى مرحلتين:

- تأتي البيانات الوصفية وقت التحليل من `registerCli(..., { descriptors: [...] })`
- يمكن أن تبقى وحدة Plugin CLI الحقيقية كسولة وتُسجَّل عند أول استدعاء

وهذا يُبقي شيفرة CLI المملوكة للإضافة داخل الإضافة مع استمرار تمكين OpenClaw من حجز أسماء أوامر الجذر قبل التحليل.

الحدّ التصميمي المهم:

- يجب أن يعمل الاكتشاف + التحقق من الإعداد من **بيانات manifest/schema الوصفية**
  من دون تنفيذ شيفرة الإضافة
- يأتي سلوك وقت التشغيل الأصلي من مسار `register(api)` الخاص بوحدة الإضافة

يسمح هذا الفصل لـ OpenClaw بالتحقق من الإعداد وشرح الإضافات المفقودة/المعطّلة وبناء تلميحات الواجهة/المخطط قبل تنشيط وقت التشغيل الكامل.

### إضافات القنوات وmessage tool المشتركة

لا تحتاج إضافات القنوات إلى تسجيل أداة send/edit/react منفصلة لإجراءات الدردشة العادية. يحتفظ OpenClaw بأداة `message` مشتركة واحدة في النواة، وتمتلك إضافات القنوات الاكتشاف والتنفيذ الخاصين بالقناة وراءها.

الحد الحالي هو:

- تمتلك النواة مضيف أداة `message` المشتركة، وربط prompt، ومسك دفاتر الجلسة/السلسلة، وتوجيه التنفيذ
- تمتلك إضافات القنوات اكتشاف الإجراءات المقيّدة واكتشاف القدرات وأي أجزاء schema خاصة بالقناة
- تمتلك إضافات القنوات قواعد محادثة الجلسة الخاصة بالمزوّد، مثل كيفية ترميز معرّفات المحادثة لمعرّفات السلاسل أو وراثتها من المحادثات الأصلية
- تنفّذ إضافات القنوات الإجراء النهائي عبر action adapter الخاص بها

بالنسبة إلى إضافات القنوات، يكون سطح SDK هو
`ChannelMessageActionAdapter.describeMessageTool(...)`. يتيح استدعاء الاكتشاف الموحّد هذا للإضافة إرجاع إجراءاتها المرئية وقدراتها ومساهمات schema الخاصة بها معًا حتى لا تنحرف هذه الأجزاء عن بعضها.

عندما يحمل أحد معاملات message-tool الخاصة بقناة ما مصدر وسائط مثل مسار محلي أو URL لوسائط بعيدة، يجب على الإضافة أيضًا إرجاع
`mediaSourceParams` من `describeMessageTool(...)`. تستخدم النواة هذه القائمة الصريحة لتطبيق تطبيع مسار sandbox وتلميحات الوصول الصادر إلى الوسائط من دون ترميز أسماء المعاملات المملوكة للإضافة بشكل ثابت.
يفضَّل استخدام خرائط مقيّدة بالإجراء هناك، لا قائمة مسطّحة واحدة على مستوى القناة، حتى لا يُطبَّع معامل وسائط خاص بملف شخصي على إجراءات غير ذات صلة مثل
`send`.

تمرر النواة نطاق وقت التشغيل إلى خطوة الاكتشاف هذه. تتضمن الحقول المهمة:

- `accountId`
- `currentChannelId`
- `currentThreadTs`
- `currentMessageId`
- `sessionKey`
- `sessionId`
- `agentId`
- `requesterSenderId` الوارد الموثوق

وهذا مهم للإضافات الحساسة للسياق. يمكن لقناة ما إخفاء إجراءات الرسائل أو إظهارها بناءً على الحساب النشط أو الغرفة/السلسلة/الرسالة الحالية أو هوية الطالب الموثوقة، من دون ترميز تفريعات خاصة بالقنوات داخل أداة `message` الأساسية.

لهذا السبب، تظل تغييرات توجيه embedded-runner من أعمال الإضافة: إذ يتحمّل المشغّل مسؤولية تمرير هوية الدردشة/الجلسة الحالية إلى حد اكتشاف الإضافة بحيث تعرض أداة `message` المشتركة السطح الصحيح المملوك للقناة للدورة الحالية.

أما بالنسبة إلى مساعدات التنفيذ المملوكة للقنوات، فيجب على الإضافات المضمّنة إبقاء وقت تشغيل التنفيذ داخل وحدات الامتداد الخاصة بها. لم تعد النواة تمتلك أزمنة تشغيل إجراءات رسائل Discord أو Slack أو Telegram أو WhatsApp تحت `src/agents/tools`.
نحن لا ننشر مسارات فرعية منفصلة `plugin-sdk/*-action-runtime`، ويجب على الإضافات المضمّنة استيراد شيفرة وقت التشغيل المحلية الخاصة بها مباشرةً من الوحدات التي تملكها الامتدادات.

وينطبق الحد نفسه على واجهات SDK المسماة بأسماء المزوّدين عمومًا: يجب ألا تستورد النواة حاويات تسهيل خاصة بالقنوات من أجل Slack أو Discord أو Signal أو WhatsApp أو امتدادات مشابهة. إذا احتاجت النواة إلى سلوك ما، فإما أن تستهلك الحاوية `api.ts` / `runtime-api.ts` الخاصة بالإضافة المضمّنة نفسها، أو ترقّي الحاجة إلى قدرة عامة ضيقة في SDK المشتركة.

وبالنسبة إلى الاستطلاعات تحديدًا، فهناك مساران للتنفيذ:

- `outbound.sendPoll` هو خط الأساس المشترك للقنوات التي تناسب نموذج الاستطلاع العام
- `actions.handleAction("poll")` هو المسار المفضّل لدلالات الاستطلاع الخاصة بالقناة أو معاملات الاستطلاع الإضافية

تؤجّل النواة الآن تحليل الاستطلاع المشترك إلى ما بعد أن يرفض توجيه استطلاع Plugin الإجراء، بحيث يمكن لمعالِجات الاستطلاع المملوكة للإضافة قبول حقول استطلاع خاصة بالقناة من دون أن يمنعها محلّل الاستطلاع العام أولًا.

انظر [مسار التحميل](#load-pipeline) للاطلاع على تسلسل بدء التشغيل الكامل.

## نموذج ملكية القدرات

يتعامل OpenClaw مع Plugin الأصلي على أنه حدّ الملكية لـ **شركة** أو **ميزة**، وليس مجرد تجمّع من عمليات التكامل غير المرتبطة.

وهذا يعني:

- يجب أن تمتلك إضافة الشركة عادةً كل الأسطح الخاصة بـ OpenClaw لتلك الشركة
- يجب أن تمتلك إضافة الميزة عادةً السطح الكامل للميزة التي تقدمها
- يجب أن تستهلك القنوات قدرات النواة المشتركة بدلًا من إعادة تنفيذ سلوك المزوّد بشكل مخصص

أمثلة:

- الإضافة المضمّنة `openai` تمتلك سلوك مزوّد نماذج OpenAI وسلوك OpenAI
  للكلام + الصوت الفوري + فهم الوسائط + توليد الصور
- الإضافة المضمّنة `elevenlabs` تمتلك سلوك الكلام الخاص بـ ElevenLabs
- الإضافة المضمّنة `microsoft` تمتلك سلوك الكلام الخاص بـ Microsoft
- الإضافة المضمّنة `google` تمتلك سلوك مزوّد نماذج Google بالإضافة إلى سلوك
  Google لفهم الوسائط + توليد الصور + البحث في الويب
- الإضافة المضمّنة `firecrawl` تمتلك سلوك جلب الويب الخاص بـ Firecrawl
- الإضافات المضمّنة `minimax` و`mistral` و`moonshot` و`zai` تمتلك
  الواجهات الخلفية الخاصة بها لفهم الوسائط
- الإضافة المضمّنة `qwen` تمتلك سلوك مزوّد النصوص لـ Qwen بالإضافة إلى
  سلوك فهم الوسائط وتوليد الفيديو
- الإضافة `voice-call` هي إضافة ميزة: فهي تمتلك نقل المكالمات وtools وCLI
  وroutes وجسر تدفق وسائط Twilio، لكنها تستهلك القدرات المشتركة للكلام
  بالإضافة إلى النسخ الفوري والصوت الفوري بدلًا من استيراد إضافات المورّدين مباشرةً

الحالة النهائية المقصودة هي:

- يعيش OpenAI في إضافة واحدة حتى لو امتد عبر نماذج النص والكلام والصور
  والفيديو مستقبلًا
- يمكن لمورّد آخر أن يفعل الشيء نفسه ضمن نطاقه الخاص
- لا تهتم القنوات بأي إضافة مورّد تمتلك المزوّد؛ فهي تستهلك عقد القدرة
  المشتركة التي تكشفها النواة

هذا هو التمييز الأساسي:

- **plugin** = حدّ الملكية
- **capability** = عقد النواة الذي يمكن لعدة إضافات تنفيذه أو استهلاكه

لذلك إذا أضاف OpenClaw مجالًا جديدًا مثل الفيديو، فلن يكون السؤال الأول هو
"أي مزوّد يجب أن يرمّز معالجة الفيديو بشكل ثابت؟" بل السؤال الأول هو "ما هو
عقد قدرة الفيديو في النواة؟" وما إن يوجد هذا العقد، يمكن لإضافات المورّدين
التسجيل مقابله، ويمكن لإضافات القنوات/الميزات استهلاكه.

إذا لم تكن القدرة موجودة بعد، فعادةً تكون الخطوة الصحيحة هي:

1. تعريف القدرة المفقودة في النواة
2. كشفها عبر Plugin API/وقت التشغيل بطريقة مكتوبة الأنواع
3. توصيل القنوات/الميزات بهذه القدرة
4. السماح لإضافات المورّدين بتسجيل التنفيذات

وهذا يُبقي الملكية واضحة مع تجنّب سلوك في النواة يعتمد على مورّد واحد أو
على مسار شيفرة خاص بإضافة واحدة.

### طبقات القدرات

استخدم هذا النموذج الذهني عند تحديد موضع الشيفرة:

- **طبقة قدرات النواة**: التنسيق المشترك، والسياسات، وآليات التراجع، وقواعد
  دمج الإعدادات، ودلالات التسليم، والعقود مكتوبة الأنواع
- **طبقة Plugin المورّد**: واجهات API الخاصة بالمورّد، والمصادقة، وفهارس النماذج، وتركيب الكلام
  وتوليد الصور، والواجهات الخلفية المستقبلية للفيديو، ونقاط نهاية الاستخدام
- **طبقة Plugin القناة/الميزة**: تكامل Slack/Discord/voice-call/إلخ
  الذي يستهلك قدرات النواة ويعرضها على سطح معيّن

على سبيل المثال، يتبع TTS هذا الشكل:

- تمتلك النواة سياسة TTS وقت الرد، وترتيب التراجع، والتفضيلات، وتسليم القناة
- تمتلك `openai` و`elevenlabs` و`microsoft` تنفيذات التركيب
- تستهلك `voice-call` مساعد وقت تشغيل TTS الخاص بالهاتف

ويُفضَّل النمط نفسه للقدرات المستقبلية.

### مثال على Plugin شركة متعددة القدرات

يجب أن تبدو إضافة الشركة متماسكة من الخارج. إذا كان لدى OpenClaw عقود مشتركة
للنماذج والكلام والنسخ الفوري والصوت الفوري وفهم الوسائط
وتوليد الصور وتوليد الفيديو وجلب الويب والبحث في الويب،
فيمكن لمورّد أن يمتلك كل أسطحه في مكان واحد:

```ts
import type { OpenClawPluginDefinition } from "openclaw/plugin-sdk/plugin-entry";
import {
  describeImageWithModel,
  transcribeOpenAiCompatibleAudio,
} from "openclaw/plugin-sdk/media-understanding";

const plugin: OpenClawPluginDefinition = {
  id: "exampleai",
  name: "ExampleAI",
  register(api) {
    api.registerProvider({
      id: "exampleai",
      // auth/model catalog/runtime hooks
    });

    api.registerSpeechProvider({
      id: "exampleai",
      // vendor speech config — implement the SpeechProviderPlugin interface directly
    });

    api.registerMediaUnderstandingProvider({
      id: "exampleai",
      capabilities: ["image", "audio", "video"],
      async describeImage(req) {
        return describeImageWithModel({
          provider: "exampleai",
          model: req.model,
          input: req.input,
        });
      },
      async transcribeAudio(req) {
        return transcribeOpenAiCompatibleAudio({
          provider: "exampleai",
          model: req.model,
          input: req.input,
        });
      },
    });

    api.registerWebSearchProvider(
      createPluginBackedWebSearchProvider({
        id: "exampleai-search",
        // credential + fetch logic
      }),
    );
  },
};

export default plugin;
```

ما يهم ليس أسماء المساعدات الدقيقة. ما يهم هو البنية:

- إضافة واحدة تمتلك سطح المورّد
- لا تزال النواة تمتلك عقود القدرات
- تستهلك القنوات وإضافات الميزات مساعدات `api.runtime.*`، لا شيفرة المورّد
- يمكن لاختبارات العقود التأكد من أن الإضافة سجّلت القدرات التي
  تدّعي امتلاكها

### مثال على القدرة: فهم الفيديو

يعامل OpenClaw بالفعل فهم الصور/الصوت/الفيديو على أنه
قدرة مشتركة واحدة. وينطبق نموذج الملكية نفسه هنا:

1. تعرّف النواة عقد فهم الوسائط
2. تسجّل إضافات المورّدين `describeImage` و`transcribeAudio` و
   `describeVideo` بحسب الاقتضاء
3. تستهلك القنوات وإضافات الميزات السلوك المشترك في النواة بدلًا من
   التوصيل المباشر إلى شيفرة المورّد

وهذا يمنع ترسيخ افتراضات الفيديو الخاصة بمزوّد واحد داخل النواة. تمتلك الإضافة
سطح المورّد؛ وتمتلك النواة عقد القدرة وسلوك التراجع.

ويستخدم توليد الفيديو بالفعل هذا التسلسل نفسه: إذ تمتلك النواة
عقد القدرة مكتوب الأنواع ومساعد وقت التشغيل، وتسجّل إضافات المورّدين
تنفيذات `api.registerVideoGenerationProvider(...)` مقابله.

هل تحتاج إلى قائمة طرح عملية ومحددة؟ انظر
[كتاب وصفات القدرات](/ar/plugins/architecture).

## العقود وآليات الفرض

تم تصميم سطح Plugin API عمدًا ليكون مكتوب الأنواع ومتمركزًا في
`OpenClawPluginApi`. يعرّف هذا العقد نقاط التسجيل المدعومة ومساعدات وقت التشغيل
التي يمكن أن تعتمد عليها الإضافة.

لماذا هذا مهم:

- يحصل مؤلفو الإضافات على معيار داخلي ثابت واحد
- يمكن للنواة رفض تكرار الملكية مثل تسجيل إضافتين لمعرّف المزوّد نفسه
- يمكن لبدء التشغيل إظهار تشخيصات عملية لتسجيلات غير سليمة
- يمكن لاختبارات العقود فرض ملكية الإضافات المضمّنة ومنع الانحراف الصامت

هناك طبقتان من الفرض:

1. **فرض التسجيل وقت التشغيل**
   يتحقق سجل الإضافات من التسجيلات أثناء تحميل الإضافات. أمثلة:
   تؤدي معرّفات المزوّدين المكررة، ومعرّفات مزوّدي الكلام المكررة، والتسجيلات
   غير السليمة إلى تشخيصات للإضافة بدلًا من سلوك غير معرّف.
2. **اختبارات العقود**
   يتم التقاط الإضافات المضمّنة في سجلات العقود أثناء تشغيل الاختبارات حتى
   يتمكن OpenClaw من تأكيد الملكية صراحةً. ويُستخدم هذا اليوم لمزوّدي
   النماذج ومزوّدي الكلام ومزوّدي البحث في الويب وملكية التسجيل المضمّن.

والأثر العملي هو أن OpenClaw يعرف مسبقًا أي إضافة تمتلك أي سطح. وهذا يسمح
للنواة والقنوات بالتركيب بسلاسة لأن الملكية مُعلنة ومكتوبة الأنواع وقابلة
للاختبار بدلًا من أن تكون ضمنية.

### ما الذي ينبغي أن يكون ضمن عقد

العقود الجيدة للإضافات تكون:

- مكتوبة الأنواع
- صغيرة
- خاصة بالقدرة
- مملوكة للنواة
- قابلة لإعادة الاستخدام بواسطة عدة إضافات
- قابلة للاستهلاك من القنوات/الميزات من دون معرفة بالمورّد

العقود السيئة للإضافات تكون:

- سياسة خاصة بالمورّد مخفية في النواة
- منافذ هروب خاصة بإضافة واحدة تتجاوز السجل
- شيفرة قناة تصل مباشرةً إلى تنفيذ مورّد
- كائنات وقت تشغيل مخصصة ليست جزءًا من `OpenClawPluginApi` أو
  `api.runtime`

عند الشك، ارفع مستوى التجريد: عرّف القدرة أولًا، ثم اسمح للإضافات بالتوصيل بها.

## نموذج التنفيذ

تعمل إضافات OpenClaw الأصلية **داخل العملية** مع Gateway. وهي ليست
معزولة. أي إضافة أصلية يتم تحميلها لديها حد الثقة على مستوى العملية نفسه
الخاص بشيفرة النواة.

الآثار المترتبة:

- يمكن لإضافة أصلية تسجيل tools ومعالِجات الشبكة وhooks وservices
- يمكن لخلل في إضافة أصلية أن يؤدي إلى تعطّل gateway أو زعزعة استقراره
- تعادل الإضافة الأصلية الخبيثة تنفيذ شيفرة عشوائية داخل عملية OpenClaw

تكون الحِزم المتوافقة أكثر أمانًا افتراضيًا لأن OpenClaw يعاملها حاليًا
على أنها حِزم بيانات وصفية/محتوى. وفي الإصدارات الحالية، يعني ذلك في الغالب
Skills المضمّنة.

استخدم قوائم السماح ومسارات التثبيت/التحميل الصريحة للإضافات غير المضمّنة.
وتعامل مع إضافات مساحة العمل على أنها شيفرة وقت تطوير، لا إعدادات إنتاج افتراضية.

بالنسبة إلى أسماء الحِزم المضمّنة في مساحة العمل، أبقِ معرّف الإضافة مثبتًا في
اسم npm: `@openclaw/<id>` افتراضيًا، أو لاحقة مكتوبة الأنواع معتمدة مثل
`-provider` أو `-plugin` أو `-speech` أو `-sandbox` أو `-media-understanding` عندما
تكشف الحزمة عمدًا دور Plugin أضيق.

ملاحظة مهمة حول الثقة:

- `plugins.allow` يثق في **معرّفات الإضافات**، لا في مصدرها.
- تعمّد إضافة مساحة العمل التي تملك المعرّف نفسه مثل إضافة مضمّنة إلى حجب
  النسخة المضمّنة عندما تكون إضافة مساحة العمل تلك مفعّلة/ضمن قائمة السماح.
- هذا سلوك طبيعي ومفيد للتطوير المحلي واختبار الرقع والإصلاحات السريعة.

## حدّ التصدير

يصدّر OpenClaw القدرات، لا وسائل الراحة الخاصة بالتنفيذ.

أبقِ تسجيل القدرات عامًا. وقلّص تصديرات المساعدات غير التعاقدية:

- المسارات الفرعية المساعدة الخاصة بالإضافة المضمّنة
- المسارات الفرعية الخاصة بأعمال وقت التشغيل غير المقصودة كواجهة API عامة
- المساعدات الملائمة الخاصة بالمورّد
- مساعدات الإعداد/التهيئة التي تُعدّ تفاصيل تنفيذية

لا تزال بعض المسارات الفرعية المساعدة الخاصة بالإضافات المضمّنة موجودة في
خريطة تصدير SDK المُولّدة لأسباب تتعلق بالتوافق وصيانة الإضافات المضمّنة.
وتشمل الأمثلة الحالية `plugin-sdk/feishu` و`plugin-sdk/feishu-setup` و`plugin-sdk/zalo`
و`plugin-sdk/zalo-setup` وعدة حدود `plugin-sdk/matrix*`. تعامل مع هذه على أنها
تصديرات محجوزة لتفاصيل التنفيذ، لا باعتبارها نمط SDK الموصى به
للإضافات الخارجية الجديدة.

## مسار التحميل

عند بدء التشغيل، يقوم OpenClaw تقريبًا بما يلي:

1. اكتشاف جذور الإضافات المرشحة
2. قراءة بيانات manifest الأصلية أو المتوافقة مع الحِزم وبيانات الحزمة الوصفية
3. رفض المرشحين غير الآمنين
4. تطبيع إعداد Plugin (`plugins.enabled`, `allow`, `deny`, `entries`,
   `slots`, `load.paths`)
5. تقرير التمكين لكل مرشّح
6. تحميل الوحدات الأصلية المفعّلة عبر jiti
7. استدعاء hooks الأصلية `register(api)` (أو `activate(api)` — وهو اسم بديل قديم) وجمع التسجيلات في سجل Plugin
8. كشف السجل إلى أسطح الأوامر/وقت التشغيل

<Note>
`activate` هو اسم بديل قديم لـ `register` — يحلّ المُحمِّل أيهما موجود (`def.register ?? def.activate`) ويستدعيه في النقطة نفسها. تستخدم كل الإضافات المضمّنة `register`؛ ويفضَّل `register` للإضافات الجديدة.
</Note>

تحدث بوابات الأمان **قبل** التنفيذ وقت التشغيل. ويتم حظر المرشحين
عندما يخرج المدخل عن جذر الإضافة، أو يكون المسار قابلاً للكتابة من الجميع،
أو تبدو ملكية المسار مريبة بالنسبة إلى الإضافات غير المضمّنة.

### سلوك manifest-first

يُعد manifest مصدر الحقيقة لمستوى التحكّم. ويستخدمه OpenClaw من أجل:

- تحديد الإضافة
- اكتشاف القنوات/Skills/مخطط الإعداد أو قدرات الحِزم المُعلن عنها
- التحقق من `plugins.entries.<id>.config`
- تعزيز تسميات/عناصر نائبة في واجهة Control
- عرض بيانات التثبيت/الفهرس الوصفية
- الحفاظ على واصفات تفعيل وإعداد منخفضة الكلفة من دون تحميل وقت تشغيل الإضافة

أما بالنسبة إلى الإضافات الأصلية، فتمثل وحدة وقت التشغيل جزء مستوى البيانات. وهي تسجّل
السلوك الفعلي مثل hooks أو tools أو commands أو تدفقات المزوّد.

تبقى الكتلتان الاختياريتان `activation` و`setup` في manifest ضمن مستوى التحكّم.
فهما واصفات بيانات وصفية فقط لتخطيط التفعيل واكتشاف الإعداد؛
وهما لا تحلان محل تسجيل وقت التشغيل أو `register(...)` أو `setupEntry`.
ويستخدم أول مستهلكي التفعيل الحي الآن تلميحات manifest الخاصة بالأوامر والقنوات والمزوّدين
لتضييق تحميل Plugin قبل الإظهار الأوسع للسجل:

- يضيّق تحميل CLI إلى الإضافات التي تمتلك الأمر الأساسي المطلوب
- يضيّق إعداد القناة/حل Plugin إلى الإضافات التي تمتلك
  معرّف القناة المطلوب
- يضيّق حل إعداد/وقت تشغيل المزوّد الصريح إلى الإضافات التي تمتلك
  معرّف المزوّد المطلوب

يفضّل اكتشاف الإعداد الآن المعرّفات المملوكة للواصفات مثل `setup.providers` و
`setup.cliBackends` لتضييق نطاق الإضافات المرشحة قبل الرجوع إلى
`setup-api` للإضافات التي لا تزال تحتاج إلى hooks لوقت الإعداد. وإذا ادّعت
أكثر من إضافة مكتشفة واحدة ملكية معرّف مزوّد إعداد أو واجهة خلفية CLI بعد
تطبيعه، فإن البحث عن الإعداد يرفض المالك الملتبس بدلًا من الاعتماد على ترتيب
الاكتشاف.

### ما الذي يخزّنه المُحمِّل مؤقتًا

يحتفظ OpenClaw بذاكرات تخزين مؤقت قصيرة داخل العملية من أجل:

- نتائج الاكتشاف
- بيانات سجل manifest
- سجلات الإضافات المُحمّلة

تقلّل هذه الذاكرات المؤقتة من بدء التشغيل المتدفّق والعبء الناتج عن تكرار
الأوامر. ويمكن اعتبارها بأمان ذواكر تخزين مؤقت للأداء قصيرة العمر، لا
وسيلة للاستمرارية.

ملاحظة حول الأداء:

- اضبط `OPENCLAW_DISABLE_PLUGIN_DISCOVERY_CACHE=1` أو
  `OPENCLAW_DISABLE_PLUGIN_MANIFEST_CACHE=1` لتعطيل هذه الذاكرات المؤقتة.
- اضبط نوافذ التخزين المؤقت باستخدام `OPENCLAW_PLUGIN_DISCOVERY_CACHE_MS` و
  `OPENCLAW_PLUGIN_MANIFEST_CACHE_MS`.

## نموذج السجل

لا تعدّل الإضافات المُحمّلة المتغيرات العامة العشوائية في النواة مباشرةً. بل
تسجّل نفسها في سجل إضافات مركزي.

يتتبّع السجل ما يلي:

- سجلات الإضافات (الهوية، المصدر، المنشأ، الحالة، التشخيصات)
- tools
- hooks القديمة وhooks مكتوبة الأنواع
- القنوات
- المزوّدات
- معالِجات Gateway RPC
- HTTP routes
- مسجّلات CLI
- services الخلفية
- commands المملوكة للإضافة

ثم تقرأ ميزات النواة من هذا السجل بدلًا من التحدث إلى وحدات الإضافات
مباشرةً. وهذا يُبقي التحميل أحادي الاتجاه:

- وحدة Plugin -> التسجيل في السجل
- وقت تشغيل النواة -> استهلاك السجل

هذا الفصل مهم لقابلية الصيانة. فهو يعني أن معظم أسطح النواة تحتاج فقط إلى
نقطة تكامل واحدة: "اقرأ السجل"، وليس "خصّص معاملة لكل وحدة Plugin".

## callbacks ربط المحادثة

يمكن للإضافات التي تربط محادثة أن تتفاعل عند حسم موافقة ما.

استخدم `api.onConversationBindingResolved(...)` لتلقي callback بعد الموافقة
على طلب ربط أو رفضه:

```ts
export default {
  id: "my-plugin",
  register(api) {
    api.onConversationBindingResolved(async (event) => {
      if (event.status === "approved") {
        // A binding now exists for this plugin + conversation.
        console.log(event.binding?.conversationId);
        return;
      }

      // The request was denied; clear any local pending state.
      console.log(event.request.conversation.conversationId);
    });
  },
};
```

حقول حمولة callback:

- `status`: `"approved"` أو `"denied"`
- `decision`: `"allow-once"` أو `"allow-always"` أو `"deny"`
- `binding`: الربط الذي تم حسمه للطلبات الموافق عليها
- `request`: ملخص الطلب الأصلي، وتلميح الفصل، ومعرّف المرسِل، وبيانات
  المحادثة الوصفية

هذا الـ callback مخصّص للإشعارات فقط. وهو لا يغيّر من يُسمح له بربط
محادثة، ويعمل بعد انتهاء معالجة الموافقة في النواة.

## hooks وقت تشغيل المزوّد

أصبحت إضافات المزوّد الآن تمتلك طبقتين:

- بيانات manifest الوصفية: `providerAuthEnvVars` من أجل بحث رخيص عن
  مصادقة المزوّد عبر env قبل تحميل وقت التشغيل، و`providerAuthAliases`
  لمتغيرات المزوّد التي تشترك في المصادقة، و`channelEnvVars` من أجل بحث
  رخيص عن env/إعداد القناة قبل تحميل وقت التشغيل، بالإضافة إلى
  `providerAuthChoices` لتسميات رخيصة للإعداد/اختيار المصادقة وبيانات
  وصفية لأعلام CLI قبل تحميل وقت التشغيل
- hooks وقت الإعداد: `catalog` / `discovery` القديمة بالإضافة إلى `applyConfigDefaults`
- hooks وقت التشغيل: `normalizeModelId`, `normalizeTransport`,
  `normalizeConfig`,
  `applyNativeStreamingUsageCompat`, `resolveConfigApiKey`,
  `resolveSyntheticAuth`, `resolveExternalAuthProfiles`,
  `shouldDeferSyntheticProfileAuth`,
  `resolveDynamicModel`, `prepareDynamicModel`, `normalizeResolvedModel`,
  `contributeResolvedModelCompat`, `capabilities`,
  `normalizeToolSchemas`, `inspectToolSchemas`,
  `resolveReasoningOutputMode`, `prepareExtraParams`, `createStreamFn`,
  `wrapStreamFn`, `resolveTransportTurnState`,
  `resolveWebSocketSessionPolicy`, `formatApiKey`, `refreshOAuth`,
  `buildAuthDoctorHint`, `matchesContextOverflowError`,
  `classifyFailoverReason`, `isCacheTtlEligible`,
  `buildMissingAuthMessage`, `suppressBuiltInModel`, `augmentModelCatalog`,
  `resolveThinkingProfile`, `isBinaryThinking`, `supportsXHighThinking`,
  `resolveDefaultThinkingLevel`, `isModernModelRef`, `prepareRuntimeAuth`,
  `resolveUsageAuth`, `fetchUsageSnapshot`, `createEmbeddingProvider`,
  `buildReplayPolicy`,
  `sanitizeReplayHistory`, `validateReplayTurns`, `onModelSelected`

لا يزال OpenClaw يمتلك حلقة الوكيل العامة، وآليات التبديل الاحتياطي، ومعالجة
السجل النصي، وسياسة tools. وتمثل هذه hooks سطح التوسعة للسلوك الخاص
بالمزوّد من دون الحاجة إلى نقل استدلال مخصص بالكامل.

استخدم `providerAuthEnvVars` في manifest عندما يمتلك المزوّد بيانات اعتماد
مبنية على env ينبغي أن تراها مسارات المصادقة/الحالة/منتقي النماذج العامة
من دون تحميل وقت تشغيل Plugin. واستخدم `providerAuthAliases` في manifest عندما
يجب أن يعيد معرّف مزوّد ما استخدام متغيرات env الخاصة بمعرّف مزوّد آخر،
وملفات تعريف المصادقة، والمصادقة المدعومة بالإعداد، وخيار الإعداد الخاص
بمفتاح API. واستخدم `providerAuthChoices` في manifest عندما ينبغي لأسطح CLI
الخاصة بالإعداد/اختيار المصادقة أن تعرف معرّف خيار المزوّد، وتسميات
المجموعات، وربط المصادقة البسيط القائم على علامة واحدة من دون تحميل وقت
تشغيل المزوّد. واحتفظ بـ `envVars` الخاصة بوقت تشغيل المزوّد لتلميحات
المشغّل، مثل تسميات الإعداد أو متغيرات إعداد `client-id`/`client-secret`
الخاصة بـ OAuth.

استخدم `channelEnvVars` في manifest عندما تمتلك القناة مصادقة أو إعدادًا
مدفوعًا عبر env ينبغي أن تراه آليات الرجوع العامة إلى env في shell، أو
فحوصات الإعداد/الحالة، أو مطالبات الإعداد من دون تحميل وقت تشغيل القناة.

### ترتيب hooks واستخدامها

بالنسبة إلى إضافات النموذج/المزوّد، يستدعي OpenClaw hooks بهذا الترتيب
التقريبي. ويُعد عمود "متى يُستخدم" دليل القرار السريع.

| #   | Hook                              | ما الذي تفعله                                                                 | متى تُستخدم                                                                 |
| --- | --------------------------------- | ----------------------------------------------------------------------------- | ---------------------------------------------------------------------------- |
| 1   | `catalog`                         | تنشر إعداد المزوّد في `models.providers` أثناء توليد `models.json`            | عندما يمتلك المزوّد فهرسًا أو إعدادات افتراضية أساسية لـ URL                |
| 2   | `applyConfigDefaults`             | تطبّق الإعدادات الافتراضية العامة المملوكة للمزوّد أثناء تشكيل الإعداد         | عندما تعتمد الإعدادات الافتراضية على وضع المصادقة أو env أو دلالات عائلة نماذج المزوّد |
| --  | _(built-in model lookup)_         | يحاول OpenClaw أولًا مسار السجل/الفهرس العادي                                  | _(ليس hook للإضافة)_                                                        |
| 3   | `normalizeModelId`                | يطبّع الأسماء البديلة القديمة أو التجريبية لمعرّف النموذج قبل البحث            | عندما يمتلك المزوّد تنظيف الأسماء البديلة قبل حل النموذج القياسي             |
| 4   | `normalizeTransport`              | يطبّع `api` / `baseUrl` لعائلة المزوّد قبل التجميع العام للنموذج               | عندما يمتلك المزوّد تنظيف النقل لمعرّفات مزوّد مخصّصة ضمن عائلة النقل نفسها |
| 5   | `normalizeConfig`                 | يطبّع `models.providers.<id>` قبل حل وقت التشغيل/المزوّد                       | عندما يحتاج المزوّد إلى تنظيف إعداد يجب أن يعيش مع الإضافة؛ كما تدعم مساعدات عائلة Google المضمّنة إدخالات إعداد Google المدعومة كحل احتياطي |
| 6   | `applyNativeStreamingUsageCompat` | تطبّق إعادة كتابة التوافق الخاصة باستخدام البث الأصلي على مزوّدي الإعداد       | عندما يحتاج المزوّد إلى إصلاحات بيانات وصفية لاستخدام البث الأصلي مدفوعة بنقطة النهاية |
| 7   | `resolveConfigApiKey`             | يحل مصادقة env-marker لمزوّدي الإعداد قبل تحميل مصادقة وقت التشغيل             | عندما يمتلك المزوّد حلًا مملوكًا له لمفتاح API عبر env-marker؛ كما يملك `amazon-bedrock` هنا محلّلًا مضمّنًا لحل AWS env-marker |
| 8   | `resolveSyntheticAuth`            | يعرض المصادقة المحلية/المستضافة ذاتيًا أو المدعومة بالإعداد دون حفظ نص عادي    | عندما يستطيع المزوّد العمل بعلامة بيانات اعتماد اصطناعية/محلية              |
| 9   | `resolveExternalAuthProfiles`     | يضيف تراكبًا لملفات تعريف المصادقة الخارجية المملوكة للمزوّد؛ والقيمة الافتراضية لـ `persistence` هي `runtime-only` لبيانات الاعتماد المملوكة لـ CLI/التطبيق | عندما يعيد المزوّد استخدام بيانات اعتماد مصادقة خارجية دون حفظ refresh tokens المنسوخة |
| 10  | `shouldDeferSyntheticProfileAuth` | يخفض أولوية العناصر النائبة الاصطناعية المخزنة في ملفات التعريف خلف المصادقة المدعومة بـ env/الإعداد | عندما يخزّن المزوّد ملفات تعريف عناصر نائبة اصطناعية لا ينبغي أن تربح الأسبقية |
| 11  | `resolveDynamicModel`             | مزامنة التراجع لمعرّفات النماذج المملوكة للمزوّد غير الموجودة بعد في السجل المحلي | عندما يقبل المزوّد معرّفات نماذج upstream اعتباطية                           |
| 12  | `prepareDynamicModel`             | تهيئة غير متزامنة، ثم يعمل `resolveDynamicModel` مرة أخرى                       | عندما يحتاج المزوّد إلى بيانات وصفية من الشبكة قبل حل المعرّفات غير المعروفة |
| 13  | `normalizeResolvedModel`          | إعادة الكتابة النهائية قبل أن يستخدم embedded runner النموذج الذي تم حله       | عندما يحتاج المزوّد إلى إعادة كتابة النقل مع الاستمرار في استخدام نقل من النواة |
| 14  | `contributeResolvedModelCompat`   | يساهم بعلامات التوافق لنماذج المورّد خلف نقل متوافق آخر                        | عندما يتعرّف المزوّد على نماذجه الخاصة على وسائل نقل proxy من دون الاستيلاء على المزوّد |
| 15  | `capabilities`                    | بيانات وصفية مملوكة للمزوّد للسجل النصي/tools تستخدمها منطق النواة المشتركة   | عندما يحتاج المزوّد إلى خصائص خاصة بالسجل النصي/عائلة المزوّد               |

| 17  | `inspectToolSchemas`              | يعرض تشخيصات المخطط المملوكة للمزوّد بعد التطبيع                               | عندما يريد المزوّد تحذيرات كلمات مفتاحية من دون تعليم النواة قواعد خاصة بالمزوّد |
| 18  | `resolveReasoningOutputMode`      | يختار عقد مخرجات reasoning الأصلي مقابل المعلّم بعلامات                        | عندما يحتاج المزوّد إلى reasoning معلّم بعلامات/مخرجات نهائية بدلًا من الحقول الأصلية |
| 19  | `prepareExtraParams`              | تطبيع معاملات الطلب قبل أغلفة خيارات البث العامة                               | عندما يحتاج المزوّد إلى معاملات طلب افتراضية أو تنظيف معاملات لكل مزوّد      |
| 20  | `createStreamFn`                  | يستبدل بالكامل مسار البث العادي بنقل مخصّص                                     | عندما يحتاج المزوّد إلى بروتوكول wire مخصّص، وليس مجرد غلاف                  |
| 21  | `wrapStreamFn`                    | غلاف للبث بعد تطبيق الأغلفة العامة                                             | عندما يحتاج المزوّد إلى أغلفة توافق للرؤوس/الجسم/النموذج من دون نقل مخصّص   |
| 22  | `resolveTransportTurnState`       | يرفق رؤوسًا أو بيانات وصفية أصلية لكل دورة نقل                                 | عندما يريد المزوّد أن ترسل وسائل النقل العامة هوية الدورة الأصلية الخاصة به |
| 23  | `resolveWebSocketSessionPolicy`   | يرفق رؤوس WebSocket أصلية أو سياسة تهدئة للجلسة                                | عندما يريد المزوّد أن تضبط وسائل نقل WS العامة رؤوس الجلسة أو سياسة التراجع |
| 24  | `formatApiKey`                    | مُنسّق ملف تعريف المصادقة: يتحول الملف المخزّن إلى سلسلة `apiKey` لوقت التشغيل | عندما يخزّن المزوّد بيانات مصادقة وصفية إضافية ويحتاج إلى شكل token مخصّص لوقت التشغيل |
| 25  | `refreshOAuth`                    | تجاوز لتحديث OAuth لنقاط نهاية تحديث مخصصة أو سياسة فشل التحديث                | عندما لا يتوافق المزوّد مع أدوات التحديث المشتركة `pi-ai`                   |
| 26  | `buildAuthDoctorHint`             | تلميح إصلاح يُلحق عند فشل تحديث OAuth                                          | عندما يحتاج المزوّد إلى إرشاد إصلاح مصادقة مملوك له بعد فشل التحديث         |
| 27  | `matchesContextOverflowError`     | مطابِق مملوك للمزوّد لتجاوز نافذة السياق                                       | عندما تكون لدى المزوّد أخطاء تجاوز خام لا تلتقطها الاستدلالات العامة        |
| 28  | `classifyFailoverReason`          | تصنيف مملوك للمزوّد لأسباب التبديل الاحتياطي                                   | عندما يستطيع المزوّد ربط أخطاء API/النقل الخام بمثل تجاوز الحد/الحمل الزائد/إلخ |
| 29  | `isCacheTtlEligible`              | سياسة prompt-cache لمزوّدي proxy/backhaul                                      | عندما يحتاج المزوّد إلى بوابة TTL خاصة بالـ proxy                           |
| 30  | `buildMissingAuthMessage`         | بديل عن رسالة استرداد المصادقة المفقودة العامة                                  | عندما يحتاج المزوّد إلى تلميح استرداد مخصص للمصادقة المفقودة                 |

| 32  | `augmentModelCatalog`             | صفوف فهرس اصطناعية/نهائية تُضاف بعد الاكتشاف                                   | عندما يحتاج المزوّد إلى صفوف تركيبية للتوافق المستقبلي في `models list` ومنتقيات النماذج |
| 33  | `resolveThinkingProfile`          | ضبط مستوى `/think` الخاص بالنموذج، وتسميات العرض، والقيمة الافتراضية          | عندما يكشف المزوّد عن سلّم تفكير مخصص أو تسمية ثنائية لنماذج محددة          |

| 42  | `buildReplayPolicy`               | يعيد سياسة إعادة تشغيل تتحكم في معالجة السجل النصي للمزوّد | عندما يحتاج المزوّد إلى سياسة سجل نصي مخصصة (مثل إزالة كتل التفكير) |

| 45  | `onModelSelected`                 | يشغّل تأثيرات جانبية بعد الاختيار مملوكة للمزوّد | عندما يحتاج المزوّد إلى telemetry أو حالة مملوكة له عند تنشيط نموذج |

يتحقق `normalizeModelId` و`normalizeTransport` و`normalizeConfig` أولًا من
Plugin المزوّد المطابق، ثم ينتقل إلى إضافات المزوّد الأخرى القادرة على استخدام hooks
إلى أن تغيّر إحداها فعليًا معرّف النموذج أو النقل/الإعداد. وهذا يُبقي
الطبقات التوافقية وإضافات alias للمزوّدات تعمل من دون أن يحتاج المستدعي إلى معرفة
أي إضافة مضمّنة تمتلك إعادة الكتابة. وإذا لم تُعِد أي hook خاصة بمزوّد
كتابة إدخال إعداد مدعوم من عائلة Google، فسيستمر مطبّع إعداد Google المضمّن
في تطبيق هذا التنظيف التوافقي.

إذا كان المزوّد يحتاج إلى بروتوكول wire مخصص بالكامل أو منفّذ طلبات مخصص،
فهذه فئة مختلفة من الامتدادات. هذه hooks مخصّصة لسلوك المزوّد الذي لا يزال
يعمل ضمن حلقة الاستدلال العادية في OpenClaw.

### مثال على المزوّد

```ts
api.registerProvider({
  id: "example-proxy",
  label: "Example Proxy",
  auth: [],
  catalog: {
    order: "simple",
    run: async (ctx) => {
      const apiKey = ctx.resolveProviderApiKey("example-proxy").apiKey;
      if (!apiKey) {
        return null;
      }
      return {
        provider: {
          baseUrl: "https://proxy.example.com/v1",
          apiKey,
          api: "openai-completions",
          models: [{ id: "auto", name: "Auto" }],
        },
      };
    },
  },
  resolveDynamicModel: (ctx) => ({
    id: ctx.modelId,
    name: ctx.modelId,
    provider: "example-proxy",
    api: "openai-completions",
    baseUrl: "https://proxy.example.com/v1",
    reasoning: false,
    input: ["text"],
    cost: { input: 0, output: 0, cacheRead: 0, cacheWrite: 0 },
    contextWindow: 128000,
    maxTokens: 8192,
  }),
  prepareRuntimeAuth: async (ctx) => {
    const exchanged = await exchangeToken(ctx.apiKey);
    return {
      apiKey: exchanged.token,
      baseUrl: exchanged.baseUrl,
      expiresAt: exchanged.expiresAt,
    };
  },
  resolveUsageAuth: async (ctx) => {
    const auth = await ctx.resolveOAuthToken();
    return auth ? { token: auth.token } : null;
  },
  fetchUsageSnapshot: async (ctx) => {
    return await fetchExampleProxyUsage(ctx.token, ctx.timeoutMs, ctx.fetchFn);
  },
});
```

### أمثلة مضمّنة

- يستخدم Anthropic كلاً من `resolveDynamicModel` و`capabilities` و`buildAuthDoctorHint`،
  و`resolveUsageAuth` و`fetchUsageSnapshot` و`isCacheTtlEligible`،
  و`resolveThinkingProfile` و`applyConfigDefaults` و`isModernModelRef`،
  و`wrapStreamFn` لأنه يمتلك التوافق المستقبلي لـ Claude 4.6،
  وتلميحات عائلة المزوّد، وإرشادات إصلاح المصادقة، وتكامل نقطة نهاية الاستخدام،
  وأهلية prompt-cache، وإعدادات افتراضية للإعداد تراعي المصادقة، وسياسة التفكير
  الافتراضية/التكيفية لـ Claude، وتشكيل البث الخاص بـ Anthropic من أجل
  رؤوس beta و`/fast` / `serviceTier` و`context1m`.
- تبقى مساعدات البث الخاصة بـ Claude في Anthropic ضمن
  الواجهة العامة `api.ts` / `contract-api.ts` الخاصة بالإضافة المضمّنة نفسها في الوقت الحالي. يصدّر
  سطح الحزمة هذا `wrapAnthropicProviderStream` و`resolveAnthropicBetas`،
  و`resolveAnthropicFastMode` و`resolveAnthropicServiceTier`، وبُناة
  الأغلفة الخاصة بـ Anthropic على المستوى الأدنى بدلًا من توسيع SDK العامة
  حول قواعد رؤوس beta الخاصة بمزوّد واحد.
- يستخدم OpenAI كلاً من `resolveDynamicModel` و`normalizeResolvedModel` و
  `capabilities` بالإضافة إلى `buildMissingAuthMessage` و`suppressBuiltInModel`،
  و`augmentModelCatalog` و`resolveThinkingProfile` و`isModernModelRef`
  لأنه يمتلك التوافق المستقبلي لـ GPT-5.4، وتطبيع OpenAI المباشر
  من `openai-completions` إلى `openai-responses`، وتلميحات المصادقة
  المراعية لـ Codex، وإخفاء Spark، وصفوف قائمة OpenAI الاصطناعية، وسياسة
  التفكير/النموذج الحي الخاصة بـ GPT-5؛ وتمتلك عائلة البث `openai-responses-defaults`
  أغلفة OpenAI Responses الأصلية المشتركة من أجل
  رؤوس الإسناد، و`/fast`/`serviceTier`، وإسهاب النص، والبحث الأصلي في الويب لـ Codex،
  وتشكيل payload المتوافق مع reasoning، وإدارة سياق Responses.
- يستخدم OpenRouter كلًا من `catalog` و`resolveDynamicModel` و
  `prepareDynamicModel` لأن المزوّد تمريري وقد يعرّض معرّفات نماذج جديدة
  قبل تحديث الفهرس الثابت في OpenClaw؛ كما يستخدم
  `capabilities` و`wrapStreamFn` و`isCacheTtlEligible` للإبقاء على
  رؤوس الطلبات الخاصة بالمزوّد، وبيانات التوجيه الوصفية، وتصحيحات reasoning،
  وسياسة prompt-cache خارج النواة. وتأتي سياسة إعادة التشغيل الخاصة به من
  عائلة `passthrough-gemini`، بينما تمتلك عائلة البث `openrouter-thinking`
  حقن reasoning عبر proxy وتخطي النماذج غير المدعومة / `auto`.

## مساعدات وقت التشغيل

يمكن للإضافات الوصول إلى مجموعة مختارة من مساعدات النواة عبر `api.runtime`. بالنسبة إلى TTS:

```ts
const clip = await api.runtime.tts.textToSpeech({
  text: "Hello from OpenClaw",
  cfg: api.config,
});

const result = await api.runtime.tts.textToSpeechTelephony({
  text: "Hello from OpenClaw",
  cfg: api.config,
});

const voices = await api.runtime.tts.listVoices({
  provider: "elevenlabs",
  cfg: api.config,
});
```

ملاحظات:

- يعيد `textToSpeech` payload المخرجات العادي في نواة TTS لأسطح الملفات/الملاحظات الصوتية.
- يستخدم إعداد النواة `messages.tts` واختيار المزوّد.
- يعيد مخزنًا مؤقتًا للصوت PCM + معدل العيّنة. يجب على الإضافات إعادة أخذ العينات/الترميز للمزوّدات.
- `listVoices` اختياري لكل مزوّد. استخدمه لمنتقيات الأصوات أو تدفقات الإعداد المملوكة للمورّد.
- يمكن أن تتضمن قوائم الأصوات بيانات وصفية أكثر غنى مثل locale والجنس وعلامات الشخصية لمنتقيات تراعي المزوّد.
- يدعم OpenAI وElevenLabs الاتصالات الهاتفية اليوم. ولا يدعمها Microsoft.

يمكن للإضافات أيضًا تسجيل مزوّدي الكلام عبر `api.registerSpeechProvider(...)`.

```ts
api.registerSpeechProvider({
  id: "acme-speech",
  label: "Acme Speech",
  isConfigured: ({ config }) => Boolean(config.messages?.tts),
  synthesize: async (req) => {
    return {
      audioBuffer: Buffer.from([]),
      outputFormat: "mp3",
      fileExtension: ".mp3",
      voiceCompatible: false,
    };
  },
});
```

ملاحظات:

- أبقِ سياسة TTS، وآليات التراجع، وتسليم الردود في النواة.
- استخدم مزوّدي الكلام لسلوك التركيب المملوك للمورّد.
- يتم تطبيع الإدخال القديم `edge` الخاص بـ Microsoft إلى معرّف المزوّد `microsoft`.
- نموذج الملكية المفضّل موجّه نحو الشركة: يمكن لإضافة مورّد واحدة أن تمتلك
  النص والكلام والصورة ومزوّدي الوسائط المستقبليين مع إضافة OpenClaw
  عقود هذه القدرات.

بالنسبة إلى فهم الصور/الصوت/الفيديو، تسجّل الإضافات مزوّدًا واحدًا مكتوب الأنواع
لفهم الوسائط بدلًا من حقيبة عامة من أزواج المفتاح/القيمة:

```ts
api.registerMediaUnderstandingProvider({
  id: "google",
  capabilities: ["image", "audio", "video"],
  describeImage: async (req) => ({ text: "..." }),
  transcribeAudio: async (req) => ({ text: "..." }),
  describeVideo: async (req) => ({ text: "..." }),
});
```

ملاحظات:

- أبقِ التنسيق، وآليات التراجع، والإعداد، وربط القنوات في النواة.
- أبقِ سلوك المورّد في Plugin المزوّد.
- يجب أن يظل التوسع الإضافي مكتوب الأنواع: أساليب اختيارية جديدة، وحقول نتائج
  اختيارية جديدة، وقدرات اختيارية جديدة.
- يتبع توليد الفيديو بالفعل النمط نفسه:
  - تمتلك النواة عقد القدرة ومساعد وقت التشغيل
  - تسجّل إضافات المورّدين `api.registerVideoGenerationProvider(...)`
  - تستهلك إضافات الميزات/القنوات `api.runtime.videoGeneration.*`

بالنسبة إلى مساعدات وقت التشغيل الخاصة بـ media-understanding، يمكن للإضافات استدعاء:

```ts
const image = await api.runtime.mediaUnderstanding.describeImageFile({
  filePath: "/tmp/inbound-photo.jpg",
  cfg: api.config,
  agentDir: "/tmp/agent",
});

const video = await api.runtime.mediaUnderstanding.describeVideoFile({
  filePath: "/tmp/inbound-video.mp4",
  cfg: api.config,

```

بالنسبة إلى نسخ الصوت، يمكن للإضافات استخدام وقت تشغيل media-understanding
أو الاسم البديل الأقدم STT:

```ts
const { text } = await api.runtime.mediaUnderstanding.transcribeAudioFile({
  filePath: "/tmp/inbound-audio.ogg",
  cfg: api.config,
  // Optional when MIME cannot be inferred reliably:
  mime: "audio/ogg",
});
```

ملاحظات:

- `api.runtime.mediaUnderstanding.*` هو السطح المشترك المفضّل من أجل
  فهم الصور/الصوت/الفيديو.
- يستخدم إعداد الصوت الخاص بـ media-understanding في النواة (`tools.media.audio`) وترتيب التراجع بين المزوّدين.
- يعيد `{ text: undefined }` عندما لا يتم إنتاج مخرجات نسخ (مثلًا عند تخطي الإدخال أو عدم دعمه).
- يظل `api.runtime.stt.transcribeAudioFile(...)` موجودًا كاسم بديل للتوافق.

يمكن للإضافات أيضًا تشغيل عمليات subagent في الخلفية عبر `api.runtime.subagent`:

```ts
const result = await api.runtime.subagent.run({
  sessionKey: "agent:main:subagent:search-helper",
  message: "Expand this query into focused follow-up searches.",
  provider: "openai",
  model: "gpt-4.1-mini",
  deliver: false,
});
```

ملاحظات:

- `provider` و`model` هما تجاوزان اختياريان لكل تشغيل، وليسا تغييرات دائمة على الجلسة.
- لا يحترم OpenClaw حقول التجاوز هذه إلا للجهات المستدعية الموثوقة.
- بالنسبة إلى عمليات التراجع المملوكة للإضافة، يجب على المشغّلين تفعيل ذلك صراحةً عبر `plugins.entries.<id>.subagent.allowModelOverride: true`.
- استخدم `plugins.entries.<id>.subagent.allowedModels` لتقييد الإضافات الموثوقة بأهداف `provider/model` قياسية محددة، أو `"*"` للسماح بأي هدف بشكل صريح.
- لا تزال عمليات subagent الخاصة بالإضافات غير الموثوقة تعمل، لكن تُرفض طلبات التجاوز بدلًا من التراجع الصامت.

بالنسبة إلى البحث في الويب، يمكن للإضافات استهلاك مساعد وقت التشغيل المشترك بدلًا من
الوصول المباشر إلى ربط أداة الوكيل:

```ts
const providers = api.runtime.webSearch.listProviders({
  config: api.config,
});

const result = await api.runtime.webSearch.search({
  config: api.config,
  args: {
    query: "OpenClaw plugin runtime helpers",
    count: 5,
  },
});
```

يمكن للإضافات أيضًا تسجيل مزوّدي البحث في الويب عبر
`api.registerWebSearchProvider(...)`.

ملاحظات:

- أبقِ اختيار المزوّد، وحل بيانات الاعتماد، ودلالات الطلب المشتركة في النواة.
- استخدم مزوّدي البحث في الويب لوسائل نقل البحث الخاصة بالمورّد.
- `api.runtime.webSearch.*` هو السطح المشترك المفضّل لإضافات الميزات/القنوات التي تحتاج إلى سلوك بحث من دون الاعتماد على غلاف أداة الوكيل.

## HTTP routes الخاصة بـ Gateway

يمكن للإضافات كشف نقاط نهاية HTTP باستخدام `api.registerHttpRoute(...)`.

```ts
api.registerHttpRoute({
  path: "/acme/webhook",
  auth: "plugin",
  match: "exact",
  handler: async (_req, res) => {
    res.statusCode = 200;
    res.end("ok");
    return true;
  },
});
```

حقول route:

- `path`: مسار route ضمن خادم HTTP الخاص بـ Gateway.
- `auth`: مطلوب. استخدم `"gateway"` لطلب مصادقة Gateway العادية، أو `"plugin"` للمصادقة/التحقق من Webhook الذي تديره الإضافة.
- `match`: اختياري. `"exact"` (الافتراضي) أو `"prefix"`.
- `replaceExisting`: اختياري. يسمح للإضافة نفسها باستبدال تسجيل route الحالي الخاص بها.
- `handler`: أعِد `true` عندما يكون route قد عالج الطلب.

ملاحظات:

- تمت إزالة `api.registerHttpHandler(...)` وسيؤدي إلى خطأ في تحميل الإضافة. استخدم `api.registerHttpRoute(...)` بدلًا منه.
- يجب على routes الخاصة بالإضافة التصريح بـ `auth` صراحةً.
- تُرفض تعارضات `path + match` المتطابقة ما لم يكن `replaceExisting: true`، ولا يمكن لإضافة أن تستبدل route خاصًا بإضافة أخرى.
- تُرفض routes المتداخلة ذات مستويات `auth` المختلفة. أبقِ سلاسل التمرير `exact`/`prefix` ضمن مستوى auth نفسه فقط.
- لا تتلقى routes ذات `auth: "plugin"` نطاقات وقت تشغيل المشغّل تلقائيًا. فهي مخصصة لـ Webhooks/التحقق من التوقيع الذي تديره الإضافة، لا لاستدعاءات مساعدات Gateway المميّزة.
- تعمل routes ذات `auth: "gateway"` داخل نطاق وقت تشغيل طلب Gateway، لكن هذا النطاق محافظ عمدًا:
  - تبقي مصادقة bearer ذات السر المشترك (`gateway.auth.mode = "token"` / `"password"`) نطاقات وقت التشغيل الخاصة بـ route الإضافة مثبّتة على `operator.write`، حتى إذا أرسل المستدعي `x-openclaw-scopes`
  - تحترم أوضاع HTTP الموثوقة الحاملة للهوية (مثل `trusted-proxy` أو `gateway.auth.mode = "none"` على مدخل خاص) القيمة `x-openclaw-scopes` فقط عندما يكون الرأس موجودًا صراحةً
  - إذا كان `x-openclaw-scopes` غير موجود في طلبات route الإضافة الحاملة للهوية تلك، يعود نطاق وقت التشغيل إلى `operator.write`
- القاعدة العملية: لا تفترض أن route إضافة بمصادقة gateway هو سطح إدارة ضمني. إذا كان route الخاص بك يحتاج إلى سلوك خاص بالمشرف فقط، فاشترط وضع مصادقة حاملًا للهوية ووثّق عقد الرأس `x-openclaw-scopes` الصريح.

  ويُعد `channel-setup` حدّ الإعداد الضيق الخاص بالتثبيت الاختياري.

  وبدء التشغيل المؤجل، بما في ذلك مكيّفات ترقيع الإعداد الآمنة للاستيراد.
  ويُعد `setup-adapter-runtime` حدّ مكيّف إعداد الحساب المراعي لـ env.
  ويُعد `setup-tools` حدّ المساعدات الصغير الخاص بـ CLI/الأرشيف/الوثائق (`formatCliCommand`,
  `detectBinary`, `extractArchive`, `resolveBrewExecutable`, `formatDocsLink`,
  `CONFIG_DIR`).
- المسارات الفرعية الخاصة بالمجال مثل `openclaw/plugin-sdk/channel-config-helpers`,
  و`openclaw/plugin-sdk/allow-from`,
  و`openclaw/plugin-sdk/channel-config-schema`,
  و`openclaw/plugin-sdk/telegram-command-config`,
  و`openclaw/plugin-sdk/channel-policy`,
  و`openclaw/plugin-sdk/approval-gateway-runtime`,
  و`openclaw/plugin-sdk/approval-handler-adapter-runtime`,
  و`openclaw/plugin-sdk/approval-handler-runtime`,
  و`openclaw/plugin-sdk/approval-runtime`,
  و`openclaw/plugin-sdk/config-runtime`,
  و`openclaw/plugin-sdk/infra-runtime`,
  و`openclaw/plugin-sdk/agent-runtime`,
  و`openclaw/plugin-sdk/lazy-runtime`,
  و`openclaw/plugin-sdk/reply-history`,
  و`openclaw/plugin-sdk/routing`,
  و`openclaw/plugin-sdk/status-helpers`,
  و`openclaw/plugin-sdk/text-runtime`,
  و`openclaw/plugin-sdk/runtime-store`، و
  `openclaw/plugin-sdk/directory-runtime` من أجل مساعدات وقت التشغيل/الإعداد المشتركة.
  يُعد `telegram-command-config` الحدّ العام الضيق الخاص بتطبيع/التحقق من الأوامر المخصصة في Telegram، ويظل متاحًا حتى إذا
  كان سطح عقد Telegram المضمّن غير متاح مؤقتًا.
  ويُعد `text-runtime` الحدّ المشترك للنصوص/Markdown/التسجيل، بما في ذلك
  إزالة النص المرئي للمساعد، ومساعدات عرض/تجزئة Markdown، ومساعدات التنقيح،
  ومساعدات directive-tag، وأدوات النص الآمن.
- ينبغي لحدود القنوات الخاصة بالموافقات أن تفضّل عقد `approvalCapability`
  واحدًا على الإضافة. ثم تقرأ النواة مصادقة الموافقة، والتسليم، والعرض،
  والتوجيه الأصلي، وسلوك المعالج الأصلي الكسول عبر هذه القدرة الواحدة
  بدلًا من خلط سلوك الموافقة ضمن حقول إضافة غير مرتبطة.
- `openclaw/plugin-sdk/channel-runtime` مهمل ولم يبقَ إلا كطبقة توافق
  للإضافات الأقدم. ويجب على الشيفرة الجديدة استيراد البدائيات العامة الأضيق بدلًا منه،
  ويجب ألا تضيف شيفرة المستودع استيرادات جديدة لهذا الغلاف التوافقي.
- تظل الأجزاء الداخلية للامتدادات المضمّنة خاصة. ويجب على الإضافات الخارجية استخدام
  المسارات الفرعية `openclaw/plugin-sdk/*` فقط. ويمكن لشيفرة النواة/الاختبارات في OpenClaw
  استخدام نقاط الإدخال العامة في المستودع تحت جذر حزمة إضافة مثل `index.js` و`api.js`،
  و`runtime-api.js` و`setup-entry.js` وملفات ضيقة النطاق مثل
  `login-qr-api.js`. لا تستورد أبدًا `src/*` من حزمة إضافة من النواة أو من امتداد آخر.
- تقسيم نقطة الإدخال في المستودع:
  `<plugin-package-root>/api.js` هو حاوية المساعدات/الأنواع،
  و`<plugin-package-root>/runtime-api.js` هو الحاوية الخاصة بوقت التشغيل فقط،
  و`<plugin-package-root>/index.js` هو إدخال Plugin المضمّن،
  و`<plugin-package-root>/setup-entry.js` هو إدخال Plugin الخاص بالإعداد.
- أمثلة المزوّدين المضمّنين الحالية:
  - يستخدم Anthropic `api.js` / `contract-api.js` لمساعدات بث Claude مثل
    `wrapAnthropicProviderStream`، ومساعدات beta-header، وتحليل `service_tier`.
  - يستخدم OpenAI `api.js` لبُناة المزوّد، ومساعدات النموذج الافتراضي، و
    بُناة المزوّد الفوري.
  - يستخدم OpenRouter `api.js` لباني المزوّد الخاص به بالإضافة إلى مساعدات
    الإعداد/التهيئة، بينما يمكن لـ `register.runtime.js` أن يعيد تصدير مساعدات
    `plugin-sdk/provider-stream` العامة للاستخدام المحلي في المستودع.
- تفضّل نقاط الإدخال العامة المحمّلة عبر facade لقطة إعداد وقت التشغيل النشطة
  عندما تكون موجودة، ثم تعود إلى ملف الإعداد المحلول على القرص عندما لا يكون
  OpenClaw قد بدأ بعد في تقديم لقطة وقت تشغيل.
- تظل البدائيات العامة المشتركة هي عقد SDK العام المفضّل. ولا تزال هناك مجموعة توافقية صغيرة ومحجوزة من حدود المساعدات ذات العلامات الخاصة بالقنوات المضمّنة.
  تعامل مع هذه على أنها حدود صيانة/توافق للإضافات المضمّنة، لا كأهداف استيراد جديدة لأطراف ثالثة؛ ويجب أن تهبط العقود الجديدة العابرة للقنوات على
  المسارات الفرعية العامة `plugin-sdk/*` أو على حاويات `api.js` /
  `runtime-api.js` المحلية الخاصة بالإضافة.

ملاحظة التوافق:

- تجنّب الحاوية الجذرية `openclaw/plugin-sdk` في الشيفرة الجديدة.
- فضّل البدائيات المستقرة الضيقة أولًا. فالمسارات الفرعية الأحدث الخاصة بـ setup/pairing/reply/
  feedback/contract/inbound/threading/command/secret-input/webhook/infra/
  allowlist/status/message-tool هي العقد المقصود لأعمال الإضافات الجديدة
  المضمّنة والخارجية.
  ويجب أن يكون تحليل/مطابقة الأهداف على `openclaw/plugin-sdk/channel-targets`.
  أما بوابات إجراءات الرسائل ومساعدات معرّف رسالة التفاعل فتكون على
  `openclaw/plugin-sdk/channel-actions`.
- حاويات المساعدات الخاصة بالامتدادات المضمّنة ليست مستقرة افتراضيًا. وإذا كانت
  المساعدة مطلوبة فقط لامتداد مضمّن، فأبقِها خلف الحدّ المحلي
  `api.js` أو `runtime-api.js` لذلك الامتداد بدلًا من ترقيتها إلى
  `openclaw/plugin-sdk/<extension>`.
- يجب أن تكون حدود المساعدات المشتركة الجديدة عامة لا موسومة بقناة بعينها. ينبغي أن
  يكون تحليل الأهداف المشترك على `openclaw/plugin-sdk/channel-targets`؛ بينما
  تبقى الأجزاء الداخلية الخاصة بالقناة خلف الحدّ المحلي `api.js` أو `runtime-api.js`
  الخاص بالإضافة المالكة.
- توجد مسارات فرعية خاصة بالقدرات مثل `image-generation`،
  و`media-understanding`، و`speech` لأن الإضافات المضمّنة/الأصلية تستخدمها
  اليوم. لكن وجودها لا يعني بحد ذاته أن كل مساعدة مُصدّرة هي
  عقد خارجي ثابت على المدى الطويل.

## مخططات message tool

ينبغي أن تمتلك الإضافات مساهمات schema الخاصة بـ `describeMessageTool(...)`
المتعلقة بالقنوات. وأبقِ الحقول الخاصة بالمزوّد داخل الإضافة، لا في النواة المشتركة.

وبالنسبة إلى أجزاء schema المحمولة المشتركة، أعد استخدام المساعدات العامة المصدّرة عبر
`openclaw/plugin-sdk/channel-actions`:

- `createMessageToolButtonsSchema()` لحمولات بأسلوب شبكة الأزرار
- `createMessageToolCardSchema()` لحمولات البطاقات المنظمة

إذا كان شكل schema منطقيًا لمزوّد واحد فقط، فعرّفه في مصدر ذلك Plugin
بدلًا من ترقيته إلى SDK المشتركة.

## حل أهداف القنوات

ينبغي أن تمتلك إضافات القنوات دلالات الأهداف الخاصة بالقناة. وأبقِ
المضيف الصادر المشترك عامًا، واستخدم سطح مكيّف المراسلة لقواعد المزوّد:

- يقرر `messaging.inferTargetChatType({ to })` ما إذا كان يجب التعامل مع هدف
  مطبّع على أنه `direct` أو `group` أو `channel` قبل البحث في الدليل.
- يخبر `messaging.targetResolver.looksLikeId(raw, normalized)` النواة عمّا إذا كان
  ينبغي لمدخل ما أن يتجاوز مباشرةً إلى الحل الشبيه بالمعرّف بدلًا من البحث في الدليل.
- يُعد `messaging.targetResolver.resolveTarget(...)` مسار التراجع الخاص بالإضافة عندما
  تحتاج النواة إلى حل نهائي مملوك للمزوّد بعد التطبيع أو بعد
  إخفاق الدليل.
- يمتلك `messaging.resolveOutboundSessionRoute(...)` إنشاء مسار الجلسة الخاص بالمزوّد
  بمجرد حل الهدف.

التقسيم الموصى به:

- استخدم `inferTargetChatType` لقرارات التصنيف التي يجب أن تحدث قبل
  البحث في الأقران/المجموعات.
- استخدم `looksLikeId` لفحوصات "عامل هذا على أنه معرّف هدف صريح/أصلي".
- استخدم `resolveTarget` لمسار التراجع الخاص بتطبيع المزوّد، وليس للبحث
  الواسع في الدليل.
- أبقِ المعرّفات الأصلية للمزوّد مثل معرّفات الدردشة، ومعرّفات السلاسل، وJIDs، والمعرّفات اليدوية، و
  معرّفات الغرف داخل قيم `target` أو المعاملات الخاصة بالمزوّد، لا في
  حقول SDK عامة.

## الأدلة المدعومة بالإعداد

ينبغي أن تُبقي الإضافات التي تستمد إدخالات الدليل من الإعداد هذا المنطق داخل
الإضافة، وأن تعيد استخدام المساعدات المشتركة من
`openclaw/plugin-sdk/directory-runtime`.

استخدم هذا عندما تحتاج قناة إلى أقران/مجموعات مدعومة بالإعداد مثل:

- أقران DM المدفوعين بقائمة السماح
- خرائط القنوات/المجموعات المهيأة
- مسارات تراجع دليل ثابتة مقيّدة بالحساب

لا تتعامل المساعدات المشتركة في `directory-runtime` إلا مع العمليات العامة:

- ترشيح الاستعلام
- تطبيق الحدود
- مساعدات إزالة التكرار/التطبيع
- بناء `ChannelDirectoryEntry[]`

يجب أن يبقى فحص الحساب الخاص بالقناة وتطبيع المعرّفات داخل تنفيذ
الإضافة.

## فهارس المزوّدين

يمكن لإضافات المزوّد تعريف فهارس النماذج للاستدلال باستخدام
`registerProvider({ catalog: { run(...) { ... } } })`.

يعيد `catalog.run(...)` البنية نفسها التي يكتبها OpenClaw في
`models.providers`:

- `{ provider }` لإدخال مزوّد واحد
- `{ providers }` لإدخالات مزوّدين متعددين

استخدم `catalog` عندما يمتلك Plugin معرّفات نماذج خاصة بالمزوّد، أو إعدادات
افتراضية لـ URL الأساسي، أو بيانات وصفية للنماذج محمية بالمصادقة.

يتحكم `catalog.order` في وقت دمج فهرس Plugin مقارنةً بالمزوّدين الضمنيين
المضمّنين في OpenClaw:

- `simple`: المزوّدون البسيطون القائمون على مفتاح API أو env
- `profile`: المزوّدون الذين يظهرون عند وجود ملفات تعريف مصادقة
- `paired`: المزوّدون الذين يركّبون إدخالات متعددة ذات صلة
- `late`: التمريرة الأخيرة، بعد بقية المزوّدين الضمنيين

يربح المزوّدون اللاحقون عند تصادم المفاتيح، لذا يمكن للإضافات أن تتعمد
تجاوز إدخال مزوّد مضمّن يملك معرّف المزوّد نفسه.

التوافق:

- لا يزال `discovery` يعمل كاسم بديل قديم
- إذا تم تسجيل كل من `catalog` و`discovery`، يستخدم OpenClaw `catalog`

## فحص القناة للقراءة فقط

إذا كان Plugin الخاص بك يسجّل قناة، ففضّل تنفيذ
`plugin.config.inspectAccount(cfg, accountId)` إلى جانب `resolveAccount(...)`.

السبب:

- `resolveAccount(...)` هو مسار وقت التشغيل. ومن المسموح له أن يفترض أن بيانات الاعتماد
  قد شُكّلت بالكامل، ويمكنه أن يفشل سريعًا عند غياب الأسرار المطلوبة.
- يجب ألا تحتاج مسارات الأوامر للقراءة فقط مثل `openclaw status` و`openclaw status --all`،
  و`openclaw channels status` و`openclaw channels resolve`، وتدفقات إصلاح doctor/config
  إلى تشكيل بيانات اعتماد وقت التشغيل فقط لوصف الإعداد.

السلوك الموصى به لـ `inspectAccount(...)`:

- أعِد حالة وصفية للحساب فقط.
- حافظ على `enabled` و`configured`.
- ضمّن حقول مصدر/حالة بيانات الاعتماد عند الصلة، مثل:
  - `tokenSource`, `tokenStatus`
  - `botTokenSource`, `botTokenStatus`
  - `appTokenSource`, `appTokenStatus`
  - `signingSecretSource`, `signingSecretStatus`
- لا تحتاج إلى إعادة قيم tokens الخام فقط للإبلاغ عن
  التوفر للقراءة فقط. إن إعادة `tokenStatus: "available"` (ومعه حقل المصدر المطابق)
  تكفي لأوامر شبيهة بالحالة.
- استخدم `configured_unavailable` عندما تكون بيانات الاعتماد مهيأة عبر SecretRef لكنها
  غير متاحة في مسار الأوامر الحالي.

وهذا يسمح لأوامر القراءة فقط بالإبلاغ عن "مهيأ لكنه غير متاح في مسار الأوامر هذا"
بدلًا من التعطل أو الإبلاغ الخاطئ بأن الحساب غير مهيأ.

## حِزم package

يمكن أن يتضمن دليل Plugin ملف `package.json` مع `openclaw.extensions`:

```json
{
  "name": "my-pack",
  "openclaw": {
    "extensions": ["./src/safety.ts", "./src/tools.ts"],
    "setupEntry": "./src/setup-entry.ts"
  }
}
```

يصبح كل إدخال إضافة. وإذا كانت الحزمة تسرد امتدادات متعددة، يصبح معرّف Plugin
هو `name/<fileBase>`.

إذا كان Plugin الخاص بك يستورد تبعيات npm، فثبّتها في ذلك الدليل حتى
يصبح `node_modules` متاحًا (`npm install` / `pnpm install`).

حاجز أمان: يجب أن يبقى كل إدخال في `openclaw.extensions` داخل دليل Plugin
بعد حل الروابط الرمزية. وتُرفض الإدخالات التي تخرج عن دليل الحزمة.

ملاحظة أمان: يقوم `openclaw plugins install` بتثبيت تبعيات الإضافة باستخدام
`npm install --omit=dev --ignore-scripts` (من دون lifecycle scripts، ومن دون تبعيات تطوير في وقت التشغيل). أبقِ أشجار تبعيات الإضافة
"JS/TS خالصة" وتجنّب الحزم التي تتطلب بناء `postinstall`.

اختياري: يمكن أن يشير `openclaw.setupEntry` إلى وحدة خفيفة خاصة بالإعداد فقط.
وعندما يحتاج OpenClaw إلى أسطح الإعداد لإضافة قناة معطّلة، أو
عندما تكون إضافة القناة مفعّلة لكنها لا تزال غير مهيأة، فإنه يحمّل `setupEntry`
بدلًا من إدخال الإضافة الكامل. وهذا يُبقي بدء التشغيل والإعداد أخف
عندما يقوم إدخال الإضافة الرئيسي أيضًا بربط tools أو hooks أو شيفرة أخرى
مخصصة لوقت التشغيل فقط.

اختياري: يمكن أن يُدخل `openclaw.startup.deferConfiguredChannelFullLoadUntilAfterListen`
إضافة قناة في مسار `setupEntry` نفسه خلال مرحلة
بدء التشغيل السابقة للاستماع في gateway، حتى عندما تكون القناة مهيأة بالفعل.

استخدم هذا فقط عندما يغطي `setupEntry` بالكامل سطح بدء التشغيل الذي يجب أن يوجد
قبل أن يبدأ gateway بالاستماع. وعمليًا، يعني ذلك أن إدخال الإعداد
يجب أن يسجّل كل قدرة مملوكة للقناة يعتمد عليها بدء التشغيل، مثل:

- تسجيل القناة نفسها
- أي HTTP routes يجب أن تكون متاحة قبل أن يبدأ gateway بالاستماع
- أي Gateway methods أو tools أو services يجب أن توجد خلال النافذة نفسها

إذا كان إدخالك الكامل لا يزال يمتلك أي قدرة مطلوبة عند بدء التشغيل، فلا تفعّل
هذا العلم. وأبقِ الإضافة على السلوك الافتراضي ودع OpenClaw يحمّل
الإدخال الكامل أثناء بدء التشغيل.

يمكن للقنوات المضمّنة أيضًا نشر مساعدات لسطح عقد خاص بالإعداد فقط يمكن للنواة
استشارتها قبل تحميل وقت تشغيل القناة الكامل. وسطح الترقية الحالي الخاص بالإعداد
هو:

- `singleAccountKeysToMove`
- `namedAccountPromotionKeys`
- `resolveSingleAccountPromotionTarget(...)`

تستخدم النواة هذا السطح عندما تحتاج إلى ترقية إعداد قناة قديم أحادي الحساب
إلى `channels.<id>.accounts.*` من دون تحميل إدخال Plugin الكامل.
وتُعد Matrix المثال المضمّن الحالي: فهي تنقل مفاتيح المصادقة/التمهيد فقط إلى
حساب مُرقّى مسمّى عندما تكون الحسابات المسماة موجودة بالفعل، ويمكنها
الحفاظ على مفتاح حساب افتراضي غير قياسي مهيأ بدلًا من إنشاء
`accounts.default` دائمًا.

تُبقي مكيّفات ترقيع الإعداد تلك اكتشاف سطح العقد المضمّن كسولًا. إذ يبقى وقت الاستيراد خفيفًا؛
ويُحمّل سطح الترقية عند أول استخدام فقط بدلًا من إعادة دخول بدء تشغيل
القناة المضمّنة عند استيراد الوحدة.

عندما تتضمن أسطح بدء التشغيل تلك Gateway RPC methods، فأبقها على
بادئة خاصة بالإضافة. وتظل مساحات أسماء إدارة النواة (`config.*`,
`exec.approvals.*`, `wizard.*`, `update.*`) محجوزة وتُحل دائمًا إلى
`operator.admin`، حتى إذا طلبت إضافة نطاقًا أضيق.

مثال:

```json
{
  "name": "@scope/my-channel",
  "openclaw": {
    "extensions": ["./index.ts"],
    "setupEntry": "./setup-entry.ts",
    "startup": {
      "deferConfiguredChannelFullLoadUntilAfterListen": true
    }

```

### البيانات الوصفية لفهرس القنوات

يمكن لإضافات القنوات الإعلان عن بيانات وصفية للإعداد/الاكتشاف عبر `openclaw.channel` و
تلميحات التثبيت عبر `openclaw.install`. وهذا يُبقي بيانات الفهرس الأساسية خالية من البيانات.

مثال:

```json
{
  "name": "@openclaw/nextcloud-talk",
  "openclaw": {
    "extensions": ["./index.ts"],
    "channel": {
      "id": "nextcloud-talk",
      "label": "Nextcloud Talk",
      "selectionLabel": "Nextcloud Talk (self-hosted)",
      "docsPath": "/channels/nextcloud-talk",
      "docsLabel": "nextcloud-talk",
      "blurb": "دردشة مستضافة ذاتيًا عبر روبوتات Webhook الخاصة بـ Nextcloud Talk.",
      "order": 65,
      "aliases": ["nc-talk", "nc"]
    },
    "install": {
      "npmSpec": "@openclaw/nextcloud-talk",
      "localPath": "<bundled-plugin-local-path>",
      "defaultChoice": "npm"
    }
  }
}
```

حقول `openclaw.channel` المفيدة إلى جانب المثال الأدنى:

- `detailLabel`: تسمية ثانوية لأسطح الفهرس/الحالة الأكثر غنى
- `docsLabel`: تجاوز نص الرابط الخاص برابط الوثائق
- `preferOver`: معرّفات الإضافات/القنوات ذات الأولوية الأدنى التي يجب أن يتفوق عليها إدخال هذا الفهرس
- `selectionDocsPrefix`, `selectionDocsOmitLabel`, `selectionExtras`: عناصر تحكم النسخ الخاصة بسطح الاختيار
- `markdownCapable`: يحدد القناة على أنها قادرة على Markdown من أجل قرارات التنسيق الصادر
- `exposure.configured`: يخفي القناة من أسطح سرد القنوات المهيأة عندما تُضبط على `false`
- `exposure.setup`: يخفي القناة من منتقيات الإعداد/التهيئة التفاعلية عندما تُضبط على `false`
- `exposure.docs`: يحدد القناة على أنها داخلية/خاصة لأسطح التنقل في الوثائق
- `showConfigured` / `showInSetup`: أسماء بديلة قديمة لا تزال مقبولة للتوافق؛ ويفضّل `exposure`
- `quickstartAllowFrom`: يُدخل القناة في تدفق `allowFrom` القياسي الخاص بالبداية السريعة
- `forceAccountBinding`: يفرض ربط الحساب صراحةً حتى عندما يوجد حساب واحد فقط
- `preferSessionLookupForAnnounceTarget`: يفضّل البحث في الجلسة عند حل أهداف الإعلان

يمكن لـ OpenClaw أيضًا دمج **فهارس قنوات خارجية** (مثلًا تصدير سجل MPM).
ضع ملف JSON في أحد المواقع التالية:

- `~/.openclaw/mpm/plugins.json`
- `~/.openclaw/mpm/catalog.json`
- `~/.openclaw/plugins/catalog.json`

أو وجّه `OPENCLAW_PLUGIN_CATALOG_PATHS` (أو `OPENCLAW_MPM_CATALOG_PATHS`) إلى
ملف JSON واحد أو أكثر (مفصولة بفواصل/فواصل منقوطة/بصيغة `PATH`). يجب أن يحتوي كل ملف
على `{ "entries": [ { "name": "@scope/pkg", "openclaw": { "channel": {...}, "install": {...} } } ] }`. ويقبل المحلّل أيضًا
`"packages"` أو `"plugins"` كأسماء بديلة قديمة للمفتاح `"entries"`.

## إضافات محرك السياق

تمتلك إضافات محرك السياق تنسيق سياق الجلسة لعمليات الإدخال والتجميع وCompaction. سجّلها من Plugin الخاص بك باستخدام
`api.registerContextEngine(id, factory)`، ثم اختر المحرك النشط عبر
`plugins.slots.contextEngine`.

استخدم هذا عندما يحتاج Plugin الخاص بك إلى استبدال مسار السياق الافتراضي
أو توسيعه بدلًا من مجرد إضافة بحث في الذاكرة أو hooks.

```ts
import { buildMemorySystemPromptAddition } from "openclaw/plugin-sdk/core";

export default function (api) {
  api.registerContextEngine("lossless-claw", () => ({
    info: { id: "lossless-claw", name: "Lossless Claw", ownsCompaction: true },
    async ingest() {
      return { ingested: true };
    },
    async assemble({ messages, availableTools, citationsMode }) {
      return {
        messages,
        estimatedTokens: 0,
        systemPromptAddition: buildMemorySystemPromptAddition({
          availableTools: availableTools ?? new Set(),
          citationsMode,
        }),
      };
    },
    async compact() {
      return { ok: true, compacted: false };
    },
  }));
}
```

إذا كان المحرك **لا** يمتلك خوارزمية Compaction، فأبقِ `compact()`
منفذًا وفوّضه صراحةً:

```ts
import {
  buildMemorySystemPromptAddition,
  delegateCompactionToRuntime,
} from "openclaw/plugin-sdk/core";

export default function (api) {
  api.registerContextEngine("my-memory-engine", () => ({
    info: {
      id: "my-memory-engine",
      name: "My Memory Engine",
      ownsCompaction: false,
    },
    async ingest() {
      return { ingested: true };
    },
    async assemble({ messages, availableTools, citationsMode }) {
      return {
        messages,
        estimatedTokens: 0,
        systemPromptAddition: buildMemorySystemPromptAddition({
          availableTools: availableTools ?? new Set(),
          citationsMode,
        }),
      };
    },
    async compact(params) {
      return await delegateCompactionToRuntime(params);
    },
  }));
}
```

## إضافة قدرة جديدة

عندما يحتاج Plugin إلى سلوك لا يلائم API الحالية، فلا تتجاوز
نظام الإضافات عبر وصول خاص غير معلن. أضف القدرة المفقودة.

التسلسل الموصى به:

1. عرّف عقد النواة
   حدّد السلوك المشترك الذي ينبغي أن تمتلكه النواة: السياسة، والتراجع، ودمج الإعداد،
   ودورة الحياة، والدلالات المواجهة للقنوات، وشكل مساعد وقت التشغيل.
2. أضف أسطح تسجيل/وقت تشغيل مكتوبة الأنواع للإضافات
   وسّع `OpenClawPluginApi` و/أو `api.runtime` بأصغر سطح قدرة مفيد
   مكتوب الأنواع.
3. وصّل مستهلكي النواة + القنوات/الميزات
   يجب أن تستهلك القنوات وإضافات الميزات القدرة الجديدة عبر النواة،
   لا عبر استيراد تنفيذ مورّد مباشرةً.
4. سجّل تنفيذات المورّدين
   ثم تسجّل إضافات المورّدين واجهاتها الخلفية مقابل القدرة.
5. أضف تغطية للعقد
   أضف اختبارات حتى تبقى الملكية وشكل التسجيل واضحين بمرور الوقت.

هكذا يبقى OpenClaw موجّهًا من دون أن يصبح مرمّزًا بشكل ثابت
وفق رؤية مزوّد واحد. انظر [كتاب وصفات القدرات](/ar/plugins/architecture)
للاطلاع على قائمة ملفات عملية ومثال مكتمل.

### قائمة التحقق الخاصة بالقدرة

عند إضافة قدرة جديدة، ينبغي عادةً أن يلامس التنفيذ هذه الأسطح معًا:

- أنواع عقد النواة في `src/<capability>/types.ts`
- مشغّل/مساعد وقت تشغيل النواة في `src/<capability>/runtime.ts`
- سطح تسجيل Plugin API في `src/plugins/types.ts`
- ربط سجل Plugin في `src/plugins/registry.ts`
- كشف وقت تشغيل Plugin في `src/plugins/runtime/*` عندما تحتاج
  إضافات الميزات/القنوات إلى استهلاكه
- مساعدات الالتقاط/الاختبار في `src/test-utils/plugin-registration.ts`
- تأكيدات الملكية/العقد في `src/plugins/contracts/registry.ts`
- وثائق المشغّل/Plugin في `docs/`

إذا كان أحد هذه الأسطح مفقودًا، فغالبًا ما تكون هذه علامة على أن القدرة
لم تُدمَج بالكامل بعد.

### قالب القدرة

النمط الأدنى:

```ts
// core contract
export type VideoGenerationProviderPlugin = {
  id: string;
  label: string;
  generateVideo: (req: VideoGenerationRequest) => Promise<VideoGenerationResult>;
};

// plugin API
api.registerVideoGenerationProvider({
  id: "openai",
  label: "OpenAI",
  async generateVideo(req) {
    return await generateOpenAiVideo(req);
  },
});

// shared runtime helper for feature/channel plugins
const clip = await api.runtime.videoGeneration.generate({
  prompt: "Show the robot walking through the lab.",
  cfg,
});
```

نمط اختبار العقد:

```ts
expect(findVideoGenerationProviderIdsForPlugin("openai")).toEqual(["openai"]);
```

وهذا يُبقي القاعدة بسيطة:

- تمتلك النواة عقد القدرة + التنسيق
- تمتلك إضافات المورّدين تنفيذات المورّدين
- تستهلك إضافات الميزات/القنوات مساعدات وقت التشغيل
- تُبقي اختبارات العقود الملكية واضحة

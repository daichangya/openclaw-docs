## نموذج القدرات العام

القدرات هي نموذج **Plugin أصلية** العام داخل OpenClaw. كل
Plugin أصلية في OpenClaw تسجل نفسها مقابل نوع قدرة واحد أو أكثر:

| القدرة             | طريقة التسجيل                              | أمثلة على Plugins                      |
| ---------------------- | ------------------------------------------------ | ------------------------------------ |
| الاستدلال النصي         | `api.registerProvider(...)`                      | `openai`, `anthropic`                |
| الواجهة الخلفية لاستدلال CLI  | `api.registerCliBackend(...)`                    | `openai`, `anthropic`                |
| الكلام                 | `api.registerSpeechProvider(...)`                | `elevenlabs`, `microsoft`            |
| النسخ الفوري | `api.registerRealtimeTranscriptionProvider(...)` | `openai`                             |
| الصوت الفوري         | `api.registerRealtimeVoiceProvider(...)`         | `openai`                             |
| فهم الوسائط    | `api.registerMediaUnderstandingProvider(...)`    | `openai`, `google`                   |
| توليد الصور       | `api.registerImageGenerationProvider(...)`       | `openai`, `google`, `fal`, `minimax` |
| توليد الموسيقى       | `api.registerMusicGenerationProvider(...)`       | `google`, `minimax`                  |
| توليد الفيديو       | `api.registerVideoGenerationProvider(...)`       | `qwen`                               |
| جلب الويب              | `api.registerWebFetchProvider(...)`              | `firecrawl`                          |
| بحث الويب             | `api.registerWebSearchProvider(...)`             | `google`                             |
| القناة / المراسلة    | `api.registerChannel(...)`                       | `msteams`, `matrix`                  |

الـ Plugin التي تسجل صفر قدرات لكنها توفر Hooks، أو أدوات، أو
خدمات هي Plugin **قديمة تعتمد على hooks فقط**. لا يزال هذا النمط مدعومًا بالكامل.

### الموقف تجاه التوافق الخارجي

لقد تم اعتماد نموذج القدرات في النواة ويُستخدم اليوم بواسطة Plugins
المضمنة/الأصلية، لكن توافق Plugins الخارجية لا يزال يحتاج إلى معيار أكثر صرامة من "إنه
مُصدَّر، إذن فهو ثابت."

الإرشادات الحالية:

- **Plugins الخارجية الحالية:** حافظ على عمل التكاملات المعتمدة على hooks؛ واعتبر
  هذا هو خط الأساس للتوافق
- **Plugins المضمنة/الأصلية الجديدة:** فضّل تسجيل القدرات الصريح بدلًا من
  الاختراقات الخاصة بالمزوّد أو تصميمات hooks-only الجديدة
- **Plugins الخارجية التي تعتمد تسجيل القدرات:** هذا مسموح، لكن تعامل مع
  الأسطح المساعدة الخاصة بالقدرات على أنها متطورة ما لم تحدد المستندات صراحةً أن العقد مستقر

القاعدة العملية:

- APIs الخاصة بتسجيل القدرات هي الاتجاه المقصود
- تظل hooks القديمة هي المسار الأكثر أمانًا لتجنب الكسر بالنسبة إلى Plugins الخارجية خلال
  الانتقال
- ليست كل المسارات الفرعية المساعدة المُصدَّرة متساوية؛ فضّل العقد الضيق الموثق،
  وليس الصادرات المساعدة العرضية

### أشكال Plugins

يصنف OpenClaw كل Plugin محملة إلى شكل بناءً على
سلوك التسجيل الفعلي الخاص بها (وليس فقط البيانات الوصفية الثابتة):

- **plain-capability** -- تسجل نوع قدرة واحدًا فقط (على سبيل المثال
  Plugin خاصة بالمزوّد فقط مثل `mistral`)
- **hybrid-capability** -- تسجل أنواع قدرات متعددة (على سبيل المثال
  `openai` يملك الاستدلال النصي، والكلام، وفهم الوسائط، وتوليد
  الصور)
- **hook-only** -- تسجل hooks فقط (مطبوعة أو مخصصة)، من دون قدرات،
  أو أدوات، أو أوامر، أو خدمات
- **non-capability** -- تسجل أدوات، أو أوامر، أو خدمات، أو مسارات، لكن من دون
  قدرات

استخدم `openclaw plugins inspect <id>` لرؤية شكل Plugin وتفصيل
القدرات الخاصة بها. راجع [مرجع CLI](/ar/cli/plugins#inspect) للتفاصيل.

### Hooks القديمة

لا تزال Hook ‏`before_agent_start` مدعومة كمسار توافق لـ
Plugins المعتمدة على hooks فقط. ولا تزال Plugins القديمة الواقعية تعتمد عليها.

الاتجاه:

- أبقها تعمل
- وثقها على أنها قديمة
- فضّل `before_model_resolve` للأعمال الخاصة بتجاوز النموذج/المزوّد
- فضّل `before_prompt_build` للأعمال الخاصة بتعديل prompt
- أزلها فقط بعد انخفاض الاستخدام الفعلي وتأكيد تغطية العناصر الثابتة لسلامة الهجرة

### إشارات التوافق

عندما تشغّل `openclaw doctor` أو `openclaw plugins inspect <id>`، قد ترى
أحد هذه التصنيفات:

| الإشارة                     | المعنى                                                      |
| -------------------------- | ------------------------------------------------------------ |
| **config valid**           | يتم تحليل الإعدادات جيدًا ويتم حل Plugins                       |
| **compatibility advisory** | تستخدم Plugin نمطًا أقدم لكنه مدعوم (مثل `hook-only`) |
| **legacy warning**         | تستخدم Plugin القيمة `before_agent_start`، وهي مهملة        |
| **hard error**             | الإعدادات غير صالحة أو فشل تحميل Plugin                   |

لن يؤدي أي من `hook-only` أو `before_agent_start` إلى كسر Plugin الخاصة بك اليوم --
فـ `hook-only` مجرد إشارة إرشادية، و`before_agent_start` لا يؤدي إلا إلى تحذير. كما
تظهر هذه الإشارات أيضًا في `openclaw status --all` و`openclaw plugins doctor`.

## نظرة عامة على البنية

يتكون نظام Plugin في OpenClaw من أربع طبقات:

1. **Manifest + الاكتشاف**
   يعثر OpenClaw على Plugins المرشحة من المسارات المضبوطة، وجذور مساحات العمل،
   وجذور Plugins العامة، وPlugins المضمنة. يقرأ الاكتشاف
   أولًا Manifest الأصلية `openclaw.plugin.json` بالإضافة إلى manifest الحزم المدعومة.
2. **التمكين + التحقق**
   تحدد النواة ما إذا كانت Plugin المكتشفة مفعّلة، أو معطلة، أو محظورة، أو
   محددة لفتحة حصرية مثل الذاكرة.
3. **التحميل وقت التشغيل**
   يتم تحميل Plugins OpenClaw الأصلية داخل العملية عبر jiti وتسجيل
   القدرات في سجل مركزي. ويتم تطبيع الحزم المتوافقة إلى
   سجلات في السجل من دون استيراد شيفرة وقت التشغيل.
4. **استهلاك السطح**
   يقرأ باقي OpenClaw السجل لكشف الأدوات، والقنوات، وإعداد
   المزوّد، وHooks، ومسارات HTTP، وأوامر CLI، والخدمات.

وبالنسبة إلى CLI الخاصة بـ Plugin تحديدًا، ينقسم اكتشاف الأوامر الجذرية إلى مرحلتين:

- تأتي البيانات الوصفية وقت التحليل من `registerCli(..., { descriptors: [...] })`
- ويمكن أن تظل وحدة CLI الحقيقية الخاصة بـ Plugin كسولة وتُسجّل عند أول استدعاء

يبقي هذا شيفرة CLI المملوكة لـ Plugin داخل Plugin مع استمرار تمكين OpenClaw من
حجز أسماء الأوامر الجذرية قبل التحليل.

حد التصميم المهم:

- يجب أن يعمل الاكتشاف + التحقق من الإعدادات انطلاقًا من **بيانات manifest/schema الوصفية**
  من دون تنفيذ شيفرة Plugin
- يأتي السلوك الأصلي وقت التشغيل من مسار `register(api)` الخاص بوحدة Plugin

يسمح هذا الفصل لـ OpenClaw بالتحقق من الإعدادات، وشرح Plugins المفقودة/المعطلة، و
بناء تلميحات UI/schema قبل أن يصبح وقت التشغيل الكامل نشطًا.

### Plugins القنوات وأداة الرسائل المشتركة

لا تحتاج Plugins القنوات إلى تسجيل أداة إرسال/تعديل/تفاعل منفصلة من أجل
إجراءات الدردشة العادية. يحتفظ OpenClaw بأداة `message` واحدة مشتركة في النواة، و
تملك Plugins القنوات الاكتشاف والتنفيذ الخاصين بالقناة خلفها.

الحد الحالي هو:

- تملك النواة مضيف أداة `message` المشتركة، وتوصيل prompt، ودفاتر
  الجلسات/السلاسل، وتوجيه التنفيذ
- تملك Plugins القنوات اكتشاف الإجراءات محددة النطاق، واكتشاف القدرات، وأي
  أجزاء schema خاصة بالقناة
- تملك Plugins القنوات القواعد النحوية للمحادثات الخاصة بالمزوّد للجلسة، مثل
  كيفية ترميز معرّفات المحادثة لمعرّفات السلاسل أو وراثتها من المحادثات الأصلية
- تنفذ Plugins القنوات الإجراء النهائي من خلال محول الإجراءات الخاص بها

بالنسبة إلى Plugins القنوات، يكون سطح SDK هو
`ChannelMessageActionAdapter.describeMessageTool(...)`. تسمح
دعوة الاكتشاف الموحدة هذه لـ Plugin بإرجاع إجراءاتها المرئية، وقدراتها، ومساهماتها في schema
معًا حتى لا تنجرف هذه الأجزاء عن بعضها.

عندما تحمل معلمة من معلمات أداة الرسائل الخاصة بالقناة مصدر وسائط مثل
مسار محلي أو عنوان URL لوسائط بعيدة، يجب على Plugin أيضًا إرجاع
`mediaSourceParams` من `describeMessageTool(...)`. تستخدم النواة
هذه القائمة الصريحة لتطبيق تطبيع مسارات العزل وتلميحات الوصول إلى الوسائط الصادرة
من دون ترميز أسماء المعلمات المملوكة لـ Plugin بشكل صلب.
فضّل الخرائط محددة الإجراء هناك، وليس قائمة مسطحة واحدة على مستوى القناة، حتى لا يتم
تطبيع معلمة وسائط خاصة بالملف الشخصي على إجراءات غير ذات صلة مثل
`send`.

تمرر النواة نطاق وقت التشغيل إلى خطوة الاكتشاف هذه. وتشمل الحقول المهمة:

- `accountId`
- `currentChannelId`
- `currentThreadTs`
- `currentMessageId`
- `sessionKey`
- `sessionId`
- `agentId`
- `requesterSenderId` الوارد الموثوق

هذا مهم بالنسبة إلى Plugins الحساسة للسياق. إذ يمكن للقناة إخفاء أو كشف
إجراءات الرسائل بناءً على الحساب النشط، أو الغرفة/السلسلة/الرسالة الحالية، أو
هوية الطالب الموثوقة، من دون ترميز فروع خاصة بالقناة بشكل صلب في
أداة `message` الأساسية.

ولهذا السبب لا تزال تغييرات التوجيه الخاصة بالمشغل المضمن عملًا خاصًا بـ Plugin:
فالمشغل مسؤول عن تمرير هوية الدردشة/الجلسة الحالية إلى حدود
اكتشاف Plugin بحيث تكشف أداة `message` المشتركة السطح المناسب
المملوك للقناة للدور الحالي.

بالنسبة إلى مساعدات التنفيذ المملوكة للقناة، يجب أن تحتفظ Plugins المضمنة بوقت تشغيل
التنفيذ داخل وحدات الامتداد الخاصة بها. لم تعد النواة تملك
أوقات تشغيل إجراءات رسائل Discord، وSlack، وTelegram، وWhatsApp تحت `src/agents/tools`.
ولا ننشر مسارات فرعية منفصلة من نوع `plugin-sdk/*-action-runtime`، ويجب أن تقوم
Plugins المضمنة باستيراد شيفرة وقت التشغيل المحلية الخاصة بها مباشرةً من
وحدات الامتداد التي تملكها.

وينطبق الحد نفسه على مفاصل SDK المسماة على أسماء المزوّدين عمومًا: يجب ألا
تستورد النواة براميل الراحة الخاصة بالقنوات مثل Slack، وDiscord، وSignal،
وWhatsApp، أو الامتدادات المشابهة. إذا احتاجت النواة إلى سلوك ما، فإما أن تستهلك
البرميل `api.ts` / `runtime-api.ts` الخاص بـ Plugin المضمنة نفسها أو ترقي الحاجة
إلى قدرة عامة ضيقة ضمن SDK المشتركة.

وبالنسبة إلى استطلاعات الرأي تحديدًا، توجد مسارا تنفيذ:

- `outbound.sendPoll` هو خط الأساس المشترك للقنوات التي تناسب النموذج
  الشائع للاستطلاع
- `actions.handleAction("poll")` هو المسار المفضل للدلالات الخاصة
  بالقناة أو لمعلمات الاستطلاع الإضافية

تؤجل النواة الآن تحليل الاستطلاع المشترك إلى ما بعد رفض إرسال الاستطلاع
الخاص بـ Plugin للإجراء، بحيث يمكن لمعالجات الاستطلاع المملوكة لـ Plugin قبول
حقول استطلاع خاصة بالقناة من دون أن يمنعها محلل الاستطلاع العام أولًا.

راجع [مسار التحميل](#load-pipeline) للتسلسل الكامل لبدء التشغيل.

## نموذج ملكية القدرات

يتعامل OpenClaw مع Plugin الأصلية على أنها حد الملكية لـ **شركة** أو **ميزة**،
وليس على أنها مجموعة عشوائية من التكاملات غير المرتبطة.

وهذا يعني:

- يجب أن تملك Plugin الشركة عادةً جميع الأسطح المواجهة لـ OpenClaw الخاصة بتلك الشركة
- يجب أن تملك Plugin الميزة عادةً سطح الميزة الكامل الذي تقدمه
- يجب أن تستهلك القنوات القدرات المشتركة في النواة بدلًا من إعادة تنفيذ سلوك
  المزوّد بشكل مخصص

أمثلة:

- تملك Plugin المضمنة `openai` سلوك مزوّد نماذج OpenAI وسلوك OpenAI
  الخاص بالكلام + الصوت الفوري + فهم الوسائط + توليد الصور
- تملك Plugin المضمنة `elevenlabs` سلوك الكلام الخاص بـ ElevenLabs
- تملك Plugin المضمنة `microsoft` سلوك الكلام الخاص بـ Microsoft
- تملك Plugin المضمنة `google` سلوك مزوّد نماذج Google بالإضافة إلى سلوك Google
  الخاص بفهم الوسائط + توليد الصور + بحث الويب
- تملك Plugin المضمنة `firecrawl` سلوك جلب الويب الخاص بـ Firecrawl
- تملك Plugins المضمنة `minimax` و`mistral` و`moonshot` و`zai`
  الواجهات الخلفية الخاصة بها لفهم الوسائط
- تملك Plugin المضمنة `qwen` سلوك مزوّد النصوص الخاص بـ Qwen بالإضافة إلى
  سلوك فهم الوسائط وتوليد الفيديو
- تُعد Plugin ‏`voice-call` Plugin ميزة: فهي تملك نقل المكالمات، والأدوات،
  وCLI، والمسارات، والربط بجريان وسائط Twilio، لكنها تستهلك القدرات المشتركة الخاصة بالكلام
  بالإضافة إلى النسخ الفوري والصوت الفوري بدلًا من
  استيراد Plugins المزوّدين مباشرةً

الحالة النهائية المقصودة هي:

- تعيش OpenAI في Plugin واحدة حتى لو كانت تمتد عبر النماذج النصية، والكلام، والصور،
  والفيديو مستقبلًا
- يمكن لمزوّد آخر أن يفعل الشيء نفسه لمساحة السطح الخاصة به
- لا تهتم القنوات بأي Plugin مزوّد تملك المزوّد؛ فهي تستهلك عقد
  القدرة المشتركة الذي تكشفه النواة

وهذا هو التمييز الأساسي:

- **plugin** = حد الملكية
- **capability** = عقد النواة الذي يمكن لعدة Plugins تنفيذه أو استهلاكه

لذلك إذا أضاف OpenClaw مجالًا جديدًا مثل الفيديو، فإن السؤال الأول ليس
"أي مزوّد يجب أن يرمّز معالجة الفيديو بشكل صلب؟" بل السؤال الأول هو "ما هو
عقد القدرة الأساسية للفيديو؟" وبمجرد وجود هذا العقد، يمكن لـ Plugins المزوّدين
التسجيل مقابله ويمكن لـ Plugins القنوات/الميزات استهلاكه.

إذا لم تكن القدرة موجودة بعد، فغالبًا ما تكون الخطوة الصحيحة هي:

1. تعريف القدرة المفقودة في النواة
2. كشفها عبر API/وقت تشغيل Plugin بطريقة مطبوعة
3. توصيل القنوات/الميزات بتلك القدرة
4. السماح لـ Plugins المزوّدين بتسجيل التنفيذات

يبقي هذا الملكية صريحة مع تجنب سلوك في النواة يعتمد على
مزوّد واحد أو مسار شيفرة خاص بـ Plugin واحدة.

### طبقات القدرات

استخدم هذا النموذج الذهني عند تقرير مكان انتماء الشيفرة:

- **طبقة القدرة الأساسية**: التنسيق المشترك، والسياسة، والبدائل، وقواعد دمج
  الإعدادات، ودلالات التسليم، والعقود المطبوعة
- **طبقة Plugin المزوّد**: APIs الخاصة بالمزوّد، والمصادقة، وفهارس النماذج، و
  توليف الكلام، وتوليد الصور، والواجهات الخلفية المستقبلية للفيديو، ونقاط نهاية الاستخدام
- **طبقة Plugin القناة/الميزة**: تكامل Slack/Discord/voice-call/إلخ
  الذي يستهلك القدرات الأساسية ويعرضها على سطح ما

على سبيل المثال، يتبع TTS هذا الشكل:

- تملك النواة سياسة TTS وقت الرد، وترتيب البدائل، والتفضيلات، وتسليم القناة
- تملك `openai` و`elevenlabs` و`microsoft` تنفيذات التوليف
- تستهلك `voice-call` مساعد وقت تشغيل TTS الخاص بالهاتف

ويجب تفضيل هذا النمط نفسه للقدرات المستقبلية.

### مثال على Plugin شركة متعددة القدرات

يجب أن تبدو Plugin الشركة متماسكة من الخارج. إذا كان لدى OpenClaw
عقود مشتركة للنماذج، والكلام، والنسخ الفوري، والصوت الفوري، وفهم الوسائط،
وتوليد الصور، وتوليد الفيديو، وجلب الويب، وبحث الويب،
فيمكن للمزوّد أن يملك جميع أسطحه في مكان واحد:

```ts
import type { OpenClawPluginDefinition } from "openclaw/plugin-sdk/plugin-entry";
import {
  describeImageWithModel,
  transcribeOpenAiCompatibleAudio,
} from "openclaw/plugin-sdk/media-understanding";

const plugin: OpenClawPluginDefinition = {
  id: "exampleai",
  name: "ExampleAI",
  register(api) {
    api.registerProvider({
      id: "exampleai",
      // auth/model catalog/runtime hooks
    });

    api.registerSpeechProvider({
      id: "exampleai",
      // vendor speech config — implement the SpeechProviderPlugin interface directly
    });

    api.registerMediaUnderstandingProvider({
      id: "exampleai",
      capabilities: ["image", "audio", "video"],
      async describeImage(req) {
        return describeImageWithModel({
          provider: "exampleai",
          model: req.model,
          input: req.input,
        });
      },
      async transcribeAudio(req) {
        return transcribeOpenAiCompatibleAudio({
          provider: "exampleai",
          model: req.model,
          input: req.input,
        });
      },
    });

    api.registerWebSearchProvider(
      createPluginBackedWebSearchProvider({
        id: "exampleai-search",
        // credential + fetch logic
      }),
    );
  },
};

export default plugin;
```

المهم ليس أسماء المساعدات الدقيقة. بل الشكل هو المهم:

- Plugin واحدة تملك سطح المزوّد
- النواة لا تزال تملك عقود القدرات
- تستهلك Plugins القنوات والميزات مساعدات `api.runtime.*`، لا شيفرة المزوّد
- يمكن لاختبارات العقود أن تؤكد أن Plugin سجلت القدرات التي
  تدعي ملكيتها

### مثال على قدرة: فهم الفيديو

يعامل OpenClaw بالفعل فهم الصور/الصوت/الفيديو على أنه
قدرة مشتركة واحدة. وينطبق نموذج الملكية نفسه هنا:

1. تعرّف النواة عقد فهم الوسائط
2. تسجل Plugins المزوّدين `describeImage` و`transcribeAudio` و
   `describeVideo` حسب الاقتضاء
3. تستهلك Plugins القنوات والميزات السلوك الأساسي المشترك بدلًا من
   التوصيل المباشر بشيفرة المزوّد

هذا يتجنب ترسيخ افتراضات الفيديو الخاصة بمزوّد واحد داخل النواة. فالـ Plugin تملك
سطح المزوّد؛ وتملك النواة عقد القدرة وسلوك البدائل.

ويستخدم توليد الفيديو بالفعل التسلسل نفسه: إذ تملك النواة عقد
القدرة المطبوعة ومساعد وقت التشغيل، وتسجل Plugins المزوّدين
تنفيذات `api.registerVideoGenerationProvider(...)` مقابله.

هل تحتاج إلى قائمة تحقق عملية لطرح هذه الميزة؟ راجع
[كتاب طبخ القدرات](/ar/plugins/architecture).

## العقود والفرض

إن سطح API الخاص بـ Plugin مطبوع ومركزي عمدًا في
`OpenClawPluginApi`. يحدد هذا العقد نقاط التسجيل المدعومة و
مساعدات وقت التشغيل التي يمكن أن تعتمد عليها Plugin.

لماذا هذا مهم:

- يحصل مؤلفو Plugins على معيار داخلي ثابت واحد
- يمكن للنواة رفض ازدواجية الملكية مثل تسجيل Pluginين لمعرّف
  مزوّد واحد
- يمكن لبدء التشغيل كشف تشخيصات عملية للتسجيلات غير الصحيحة
- يمكن لاختبارات العقود فرض ملكية Plugins المضمنة ومنع الانجراف الصامت

هناك طبقتان من الفرض:

1. **فرض التسجيل وقت التشغيل**
   يتحقق سجل Plugin من التسجيلات أثناء تحميل Plugins. أمثلة:
   معرّفات المزوّد المكررة، ومعرّفات مزوّد الكلام المكررة، والتسجيلات
   غير الصحيحة تنتج تشخيصات Plugin بدلًا من السلوك غير المعرّف.
2. **اختبارات العقود**
   يتم التقاط Plugins المضمنة في سجلات العقود أثناء تشغيل الاختبارات بحيث
   يمكن لـ OpenClaw تأكيد الملكية صراحةً. ويُستخدم هذا اليوم لمزوّدي
   النماذج، ومزوّدي الكلام، ومزوّدي بحث الويب، وملكية التسجيل
   المضمنة.

التأثير العملي هو أن OpenClaw يعرف مسبقًا أي Plugin تملك أي
سطح. وهذا يسمح للنواة والقنوات بالتركيب بسلاسة لأن الملكية
معلنة، ومطبوعة، وقابلة للاختبار بدلًا من أن تكون ضمنية.

### ما الذي يجب أن ينتمي إلى العقد

العقود الجيدة لـ Plugin تكون:

- مطبوعة
- صغيرة
- خاصة بالقدرة
- مملوكة للنواة
- قابلة لإعادة الاستخدام من قبل عدة Plugins
- قابلة للاستهلاك من قبل القنوات/الميزات من دون معرفة بالمزوّد

العقود السيئة لـ Plugin تكون:

- سياسة خاصة بالمزوّد مخفية داخل النواة
- مسارات هروب لمرة واحدة خاصة بـ Plugin تتجاوز السجل
- شيفرة القناة التي تصل مباشرة إلى تنفيذ مزوّد
- كائنات وقت تشغيل مخصصة ليست جزءًا من `OpenClawPluginApi` أو
  `api.runtime`

عند الشك، ارفع مستوى التجريد: عرّف القدرة أولًا، ثم
دع Plugins تتصل بها.

## نموذج التنفيذ

تعمل Plugins OpenClaw الأصلية **داخل العملية** مع Gateway. وهي غير
معزولة. وأي Plugin أصلية محمّلة لها حد الثقة على مستوى العملية نفسه الذي تملكه
شيفرة النواة.

الآثار المترتبة:

- يمكن لـ Plugin أصلية تسجيل أدوات، ومعالجات شبكة، وHooks، وخدمات
- يمكن لخطأ في Plugin أصلية أن يتسبب في تعطل gateway أو زعزعة استقرارها
- تعادل Plugin أصلية خبيثة تنفيذ شيفرة عشوائية داخل
  عملية OpenClaw

تكون الحزم المتوافقة أكثر أمانًا افتراضيًا لأن OpenClaw يتعامل معها حاليًا
كحزم بيانات/محتوى. وهذا يعني في الإصدارات الحالية، في الغالب،
Skills المضمنة.

استخدم قوائم السماح ومسارات التثبيت/التحميل الصريحة لـ Plugins غير المضمنة. وتعامل مع
Plugins مساحة العمل على أنها شيفرة وقت تطوير، وليست إعدادات افتراضية للإنتاج.

بالنسبة إلى أسماء الحزم المضمنة في مساحة العمل، أبقِ معرّف Plugin مرتكزًا في اسم npm:
`@openclaw/<id>` افتراضيًا، أو لاحقة مطبوعة معتمدة مثل
`-provider` أو `-plugin` أو `-speech` أو `-sandbox` أو `-media-understanding` عندما
تكشف الحزمة عمدًا عن دور Plugin أضيق.

ملاحظة ثقة مهمة:

- تثق `plugins.allow` في **معرّفات Plugin**، وليس في مصدر المنشأ.
- تقوم Plugin مساحة العمل التي تحمل المعرّف نفسه الذي تحمله Plugin مضمنة بتظليل
  النسخة المضمنة عن قصد عندما تكون تلك Plugin في مساحة العمل مفعّلة/مدرجة في قائمة السماح.
- وهذا أمر طبيعي ومفيد للتطوير المحلي، واختبار التصحيحات، والإصلاحات السريعة.
- تُحل الثقة في Plugin المضمنة من لقطة المصدر — أي manifest و
  الشيفرة الموجودة على القرص وقت التحميل — وليس من بيانات التثبيت الوصفية. وبالتالي لا يمكن
  لسجل تثبيت تالف أو مستبدل أن يوسّع بصمت سطح الثقة الخاص
  بـ Plugin مضمنة إلى ما يتجاوز ما يدعيه المصدر الفعلي.

## حد التصدير

يصدّر OpenClaw القدرات، وليس وسائل الراحة التنفيذية.

أبقِ تسجيل القدرات عامًا. وقلّم صادرات المساعدات غير التعاقدية:

- المسارات الفرعية الخاصة بمساعدات Plugins المضمنة
- المسارات الفرعية الخاصة بتوصيلات وقت التشغيل غير المقصودة كـ API عامة
- المساعدات الملائمة الخاصة بالمزوّد
- مساعدات الإعداد/التهيئة الأولية التي تُعد تفاصيل تنفيذ

لا تزال بعض المسارات الفرعية للمساعدات الخاصة بـ Plugins المضمنة موجودة في خريطة تصدير SDK
المولدة من أجل التوافق وصيانة Plugins المضمنة. وتشمل الأمثلة الحالية
`plugin-sdk/feishu` و`plugin-sdk/feishu-setup` و`plugin-sdk/zalo`,
و`plugin-sdk/zalo-setup`، وعدة مفاصل `plugin-sdk/matrix*`. تعامل مع هذه
على أنها صادرات محفوظة كتفاصيل تنفيذ، وليست النمط الموصى به في SDK لـ
Plugins الخارجية الجديدة.

## مسار التحميل

عند بدء التشغيل، يقوم OpenClaw تقريبًا بما يلي:

1. يكتشف جذور Plugins المرشحة
2. يقرأ manifest الأصلية أو Manifest الحزم المتوافقة وبيانات الحزم الوصفية
3. يرفض المرشحين غير الآمنين
4. يطبع إعدادات Plugin (`plugins.enabled`, `allow`, `deny`, `entries`,
   `slots`, `load.paths`)
5. يقرر التمكين لكل مرشح
6. يحمّل الوحدات الأصلية المفعّلة — تمر الوحدات المضمنة المبنية `dist/*` عبر
   مسار تحميل أصلي، بينما يتم تحميل وحدات Plugin الأصلية غير المبنية عبر
   jiti
7. يستدعي Hooks الأصلية `register(api)` ‏(أو `activate(api)` — وهو اسم بديل قديم) ويجمع التسجيلات في سجل Plugin
8. يكشف السجل إلى أسطح الأوامر/وقت التشغيل

<Note>
`activate` هو اسم بديل قديم لـ `register` — يقوم المحمّل بحل أي منهما موجود (`def.register ?? def.activate`) ويستدعيه في النقطة نفسها. تستخدم جميع Plugins المضمنة `register`؛ ويفضّل استخدام `register` في Plugins الجديدة.
</Note>

تحدث بوابات الأمان **قبل** تنفيذ وقت التشغيل. ويتم حظر المرشحين
عندما يفلت المدخل من جذر Plugin، أو يكون المسار قابلاً للكتابة من الجميع، أو
تبدو ملكية المسار مريبة بالنسبة إلى Plugins غير المضمنة.

### سلوك manifest-first

تُعد manifest مصدر الحقيقة الخاص بمستوى التحكم. يستخدمها OpenClaw من أجل:

- تحديد Plugin
- اكتشاف القنوات/‏Skills/‏schema الخاصة بالإعدادات المعلنة أو قدرات الحزمة
- التحقق من `plugins.entries.<id>.config`
- تعزيز التسميات/العناصر النائبة في Control UI
- عرض بيانات التثبيت/الفهرس الوصفية
- الحفاظ على واصفات التنشيط والإعداد الرخيصة من دون تحميل وقت تشغيل Plugin

بالنسبة إلى Plugins الأصلية، تكون وحدة وقت التشغيل هي جزء مستوى البيانات. فهي تسجل
السلوك الفعلي مثل Hooks، أو الأدوات، أو الأوامر، أو تدفقات المزوّد.

تظل كتلتا `activation` و`setup` الاختياريتان في manifest ضمن مستوى التحكم.
وهما واصفات تعتمد على البيانات الوصفية فقط من أجل تخطيط التنشيط واكتشاف الإعداد؛
ولا تستبدلان التسجيل وقت التشغيل، أو `register(...)`، أو `setupEntry`.
ويستخدم أول المستهلكين الفعليين للتنشيط الآن تلميحات manifest الخاصة بالأوامر، والقنوات، والمزوّدين
لتضييق تحميل Plugin قبل تكوين السجل الأوسع:

- يضيّق تحميل CLI إلى Plugins التي تملك الأمر الأساسي المطلوب
- يضيّق حل إعداد/Plugin القناة إلى Plugins التي تملك
  معرّف القناة المطلوب
- يضيّق حل إعداد/وقت تشغيل المزوّد الصريح إلى Plugins التي تملك
  معرّف المزوّد المطلوب

يفضّل اكتشاف الإعداد الآن المعرفات التي تملكها الواصفات مثل `setup.providers` و
`setup.cliBackends` لتضييق Plugins المرشحة قبل الرجوع إلى
`setup-api` بالنسبة إلى Plugins التي لا تزال تحتاج إلى Hooks وقت تشغيل للإعداد. وإذا ادعت أكثر من Plugin
مكتشفة ملكية معرّف مزوّد إعداد أو CLI backend موحّدًا نفسه، فإن
البحث عن الإعداد يرفض المالك الملتبس بدلًا من الاعتماد على ترتيب الاكتشاف.

### ما الذي يخزنه المحمّل مؤقتًا

يحتفظ OpenClaw بمخازن مؤقتة قصيرة داخل العملية من أجل:

- نتائج الاكتشاف
- بيانات سجل manifest
- سجلات Plugins المحمّلة

تقلل هذه المخازن المؤقتة من بدء التشغيل المتفجر ومن العبء المتكرر للأوامر. ومن الآمن
التفكير فيها على أنها مخازن أداء مؤقتة قصيرة العمر، وليست استمرارية.

ملاحظة أداء:

- اضبط `OPENCLAW_DISABLE_PLUGIN_DISCOVERY_CACHE=1` أو
  `OPENCLAW_DISABLE_PLUGIN_MANIFEST_CACHE=1` لتعطيل هذه المخازن المؤقتة.
- اضبط نوافذ التخزين المؤقت باستخدام `OPENCLAW_PLUGIN_DISCOVERY_CACHE_MS` و
  `OPENCLAW_PLUGIN_MANIFEST_CACHE_MS`.

## نموذج السجل

لا تقوم Plugins المحمّلة بتعديل متغيرات عامة عشوائية في النواة مباشرةً. بل تسجل نفسها في
سجل Plugin مركزي.

يتتبع السجل:

- سجلات Plugins ‏(الهوية، والمصدر، والمنشأ، والحالة، والتشخيصات)
- الأدوات
- Hooks القديمة وHooks المطبوعة
- القنوات
- المزوّدين
- معالجات Gateway RPC
- مسارات HTTP
- مسجلات CLI
- الخدمات الخلفية
- الأوامر المملوكة لـ Plugin

ثم تقرأ ميزات النواة من هذا السجل بدلًا من التحدث إلى وحدات Plugin
مباشرةً. وهذا يبقي التحميل في اتجاه واحد:

- وحدة Plugin -> التسجيل في السجل
- وقت تشغيل النواة -> استهلاك السجل

هذا الفصل مهم من أجل سهولة الصيانة. إذ يعني أن معظم أسطح النواة تحتاج فقط
إلى نقطة تكامل واحدة: "اقرأ السجل"، وليس "ضع حالة خاصة لكل وحدة Plugin".

## نداءات ربط المحادثة

يمكن لـ Plugins التي تربط محادثة أن تتفاعل عند حل موافقة ما.

استخدم `api.onConversationBindingResolved(...)` لتلقي نداء
بعد اعتماد أو رفض طلب ربط:

```ts
export default {
  id: "my-plugin",
  register(api) {
    api.onConversationBindingResolved(async (event) => {
      if (event.status === "approved") {
        // A binding now exists for this plugin + conversation.
        console.log(event.binding?.conversationId);
        return;
      }

      // The request was denied; clear any local pending state.
      console.log(event.request.conversation.conversationId);
    });
  },
};
```

حقول حمولة النداء:

- `status`: ‏`"approved"` أو `"denied"`
- `decision`: ‏`"allow-once"` أو `"allow-always"` أو `"deny"`
- `binding`: الربط المحلول للطلبات المعتمدة
- `request`: ملخص الطلب الأصلي، وتلميح الفصل، ومعرّف المرسل، و
  البيانات الوصفية للمحادثة

هذا النداء مخصص للإشعارات فقط. ولا يغيّر من يُسمح له بربط
محادثة، كما أنه يعمل بعد انتهاء معالجة الموافقة الأساسية.

## Hooks وقت تشغيل المزوّد

أصبحت Plugins المزوّدين الآن تحتوي على طبقتين:

- بيانات manifest الوصفية: `providerAuthEnvVars` للبحث الرخيص عن مصادقة المزوّد عبر env
  قبل تحميل وقت التشغيل، و`providerAuthAliases` لمتغيرات المزوّد التي تشترك في
  المصادقة، و`channelEnvVars` للبحث الرخيص عن env/setup الخاص بالقناة قبل تحميل وقت التشغيل،
  بالإضافة إلى `providerAuthChoices` للتسميات الرخيصة الخاصة بالتهيئة الأولية/اختيار المصادقة و
  بيانات أعلام CLI الوصفية قبل تحميل وقت التشغيل
- Hooks وقت الإعداد: `catalog` / `discovery` القديمة بالإضافة إلى `applyConfigDefaults`
- Hooks وقت التشغيل: `normalizeModelId`, `normalizeTransport`,
  `normalizeConfig`,
  `applyNativeStreamingUsageCompat`, `resolveConfigApiKey`,
  `resolveSyntheticAuth`, `resolveExternalAuthProfiles`,
  `shouldDeferSyntheticProfileAuth`,
  `resolveDynamicModel`, `prepareDynamicModel`, `normalizeResolvedModel`,
  `contributeResolvedModelCompat`, `capabilities`,
  `normalizeToolSchemas`, `inspectToolSchemas`,
  `resolveReasoningOutputMode`, `prepareExtraParams`, `createStreamFn`,
  `wrapStreamFn`, `resolveTransportTurnState`,
  `resolveWebSocketSessionPolicy`, `formatApiKey`, `refreshOAuth`,
  `buildAuthDoctorHint`, `matchesContextOverflowError`,
  `classifyFailoverReason`, `isCacheTtlEligible`,
  `buildMissingAuthMessage`, `suppressBuiltInModel`, `augmentModelCatalog`,
  `resolveThinkingProfile`, `isBinaryThinking`, `supportsXHighThinking`,
  `resolveDefaultThinkingLevel`, `isModernModelRef`, `prepareRuntimeAuth`,
  `resolveUsageAuth`, `fetchUsageSnapshot`, `createEmbeddingProvider`,
  `buildReplayPolicy`,
  `sanitizeReplayHistory`, `validateReplayTurns`, `onModelSelected`

لا يزال OpenClaw يملك حلقة الوكيل العامة، وعمليات التبديل الاحتياطي، ومعالجة النصوص، و
سياسة الأدوات. وتمثل هذه Hooks سطح الامتداد للسلوك الخاص بالمزوّد من دون
الحاجة إلى نقل استدلال مخصص كامل.

استخدم `providerAuthEnvVars` في manifest عندما يمتلك المزوّد بيانات اعتماد معتمدة على env
يجب أن تراها مسارات المصادقة/الحالة/منتقي النماذج العامة من دون تحميل وقت تشغيل Plugin.
واستخدم `providerAuthAliases` في manifest عندما يجب أن يعيد معرّف مزوّد ما استخدام
متغيرات env الخاصة بمعرّف مزوّد آخر، وملفات تعريف المصادقة، والمصادقة المدعومة بالإعدادات، وخيار
التهيئة الأولية المرتبط بمفتاح API. واستخدم `providerAuthChoices` في manifest عندما تكون
أسطح CLI الخاصة بالتهيئة الأولية/اختيار المصادقة بحاجة إلى معرفة معرّف اختيار المزوّد، وتسميات المجموعات،
وتوصيل المصادقة البسيط أحادي العلم من دون تحميل وقت تشغيل المزوّد. وأبقِ `envVars` الخاصة بوقت تشغيل المزوّد
للتلميحات الموجهة إلى المشغل مثل تسميات التهيئة الأولية أو
متغيرات إعداد client-id/client-secret الخاصة بـ OAuth.

استخدم `channelEnvVars` في manifest عندما تكون لدى القناة مصادقة أو إعداد مدفوع بـ env
يجب أن تراه المسارات العامة الخاصة بالاحتياط من shell-env، أو فحوصات config/status، أو مطالبات الإعداد
من دون تحميل وقت تشغيل القناة.

### ترتيب Hooks واستخدامها

بالنسبة إلى Plugins النموذج/المزوّد، يستدعي OpenClaw Hooks تقريبًا بهذا الترتيب.

تتحقق `normalizeModelId` و`normalizeTransport` و`normalizeConfig` أولًا من
Plugin المزوّد المطابقة، ثم تمر إلى Plugins المزوّدين الأخرى القادرة على استخدام hooks
حتى تقوم إحداها فعليًا بتغيير معرّف النموذج أو النقل/الإعدادات. وهذا يُبقي
التسويات الخاصة بالأسماء المستعارة/التوافق الخاصة بالمزوّدين تعمل من دون الحاجة إلى أن يعرف
المستدعي أي Plugin مضمنة تملك عملية إعادة الكتابة. وإذا لم تُعد أي Hook خاصة
بالمزوّد كتابة إدخال إعدادات مدعوم من عائلة Google،
فسيستمر مطبّع إعدادات Google المضمن في تطبيق تنظيف التوافق هذا.

إذا كان المزوّد يحتاج إلى بروتوكول سلكي مخصص بالكامل أو منفّذ طلبات مخصص،
فهذه فئة مختلفة من الامتداد. فهذه Hooks مخصصة لسلوك المزوّد الذي لا يزال
يعمل على حلقة الاستدلال العادية في OpenClaw.

### مثال على مزوّد

```ts
api.registerProvider({
  id: "example-proxy",
  label: "Example Proxy",
  auth: [],
  catalog: {
    order: "simple",
    run: async (ctx) => {
      const apiKey = ctx.resolveProviderApiKey("example-proxy").apiKey;
      if (!apiKey) {
        return null;
      }
      return {
        provider: {
          baseUrl: "https://proxy.example.com/v1",
          apiKey,
          api: "openai-completions",
          models: [{ id: "auto", name: "Auto" }],
        },
      };
    },
  },
  resolveDynamicModel: (ctx) => ({
    id: ctx.modelId,
    name: ctx.modelId,
    provider: "example-proxy",
    api: "openai-completions",
    baseUrl: "https://proxy.example.com/v1",
    reasoning: false,
    input: ["text"],
    cost: { input: 0, output: 0, cacheRead: 0, cacheWrite: 0 },
    contextWindow: 128000,
    maxTokens: 8192,
  }),
  prepareRuntimeAuth: async (ctx) => {
    const exchanged = await exchangeToken(ctx.apiKey);
    return {
      apiKey: exchanged.token,
      baseUrl: exchanged.baseUrl,
      expiresAt: exchanged.expiresAt,
    };
  },
  resolveUsageAuth: async (ctx) => {
    const auth = await ctx.resolveOAuthToken();
    return auth ? { token: auth.token } : null;
  },
  fetchUsageSnapshot: async (ctx) => {
    return await fetchExampleProxyUsage(ctx.token, ctx.timeoutMs, ctx.fetchFn);
  },
});
```

### أمثلة مدمجة

- يستخدم Anthropic كلًا من `resolveDynamicModel` و`capabilities` و`buildAuthDoctorHint`,
  و`resolveUsageAuth` و`fetchUsageSnapshot` و`isCacheTtlEligible`,
  و`resolveThinkingProfile` و`applyConfigDefaults` و`isModernModelRef`,
  و`wrapStreamFn` لأنه يملك التوافق الأمامي لـ Claude 4.6،
  وتلميحات عائلة المزوّد، وإرشادات إصلاح المصادقة، وتكامل نقطة نهاية الاستخدام،
  وأهلية prompt-cache، والإعدادات الافتراضية الواعية بالمصادقة،
  وسياسة التفكير الافتراضية/التكيفية الخاصة بـ Claude، وتشكيل البث الخاص بـ Anthropic من أجل
  ترويسات beta، و`/fast` / `serviceTier`، و`context1m`.
- تظل مساعدات البث الخاصة بـ Claude في Anthropic ضمن
  الوصلة العامة `api.ts` / `contract-api.ts` الخاصة بـ Plugin المضمنة نفسها في الوقت الحالي. ويُصدّر
  سطح الحزمة هذا `wrapAnthropicProviderStream` و`resolveAnthropicBetas`,
  و`resolveAnthropicFastMode` و`resolveAnthropicServiceTier` وبناة
  أغلفة Anthropic ذات المستوى الأدنى بدلًا من توسيع SDK العامة حول
  قواعد ترويسات beta الخاصة بمزوّد واحد.
- يستخدم OpenAI كلًا من `resolveDynamicModel` و`normalizeResolvedModel` و
  `capabilities` بالإضافة إلى `buildMissingAuthMessage` و`suppressBuiltInModel`,
  و`augmentModelCatalog` و`resolveThinkingProfile` و`isModernModelRef`
  لأنه يملك التوافق الأمامي لـ GPT-5.4، وتطبيع
  النقل المباشر `openai-completions` -> `openai-responses` الخاص بـ OpenAI،
  وتلميحات المصادقة الواعية بـ Codex، وإخفاء Spark، وصفوف OpenAI التركيبية،
  وسياسة التفكير/النموذج الحي الخاصة بـ GPT-5؛ كما تملك عائلة البث `openai-responses-defaults`
  أغلفة OpenAI Responses الأصلية المشتركة لترويسات الإسناد،
  و`/fast`/`serviceTier`، وإسهاب النص، والبحث الأصلي على الويب في Codex،
  وتشكيل حمولات توافق الاستدلال، وإدارة سياق Responses.
- يستخدم OpenRouter كلًا من `catalog` بالإضافة إلى `resolveDynamicModel` و
  `prepareDynamicModel` لأن المزوّد تمريري وقد يكشف عن معرّفات نماذج
  جديدة قبل تحديث الفهرس الثابت في OpenClaw؛ كما يستخدم أيضًا
  `capabilities` و`wrapStreamFn` و`isCacheTtlEligible` لإبقاء

## مساعدات وقت التشغيل

يمكن لـ Plugins الوصول إلى مجموعة محددة من مساعدات النواة عبر `api.runtime`. وبالنسبة إلى TTS:

```ts
const clip = await api.runtime.tts.textToSpeech({
  text: "Hello from OpenClaw",
  cfg: api.config,
});

const result = await api.runtime.tts.textToSpeechTelephony({
  text: "Hello from OpenClaw",
  cfg: api.config,
});

const voices = await api.runtime.tts.listVoices({
  provider: "elevenlabs",
  cfg: api.config,
});
```

ملاحظات:

- تعيد `textToSpeech` حمولة خرج TTS الأساسية العادية لأسطح الملفات/الملاحظات الصوتية.
- تستخدم إعدادات `messages.tts` الأساسية واختيار المزوّد.
- تعيد PCM audio buffer + sample rate. ويجب على Plugins إعادة أخذ العينات/الترميز للمزوّدين.
- تكون `listVoices` اختيارية لكل مزوّد. استخدمها لمنتقيات الأصوات أو تدفقات الإعداد المملوكة للمزوّد.
- يمكن أن تتضمن قوائم الأصوات بيانات وصفية أغنى مثل locale، والجنس، ووسوم الشخصية لمنتقيات تدرك المزوّد.
- يدعم كل من OpenAI وElevenLabs الهاتف اليوم. أما Microsoft فلا.

يمكن لـ Plugins أيضًا تسجيل مزوّدي الكلام عبر `api.registerSpeechProvider(...)`.

```ts
api.registerSpeechProvider({
  id: "acme-speech",
  label: "Acme Speech",
  isConfigured: ({ config }) => Boolean(config.messages?.tts),
  synthesize: async (req) => {
    return {
      audioBuffer: Buffer.from([]),
      outputFormat: "mp3",
      fileExtension: ".mp3",
      voiceCompatible: false,
    };
  },
});
```

ملاحظات:

- أبقِ سياسة TTS، والبدائل، وتسليم الردود في النواة.
- استخدم مزوّدي الكلام لسلوك التوليف الذي يملكه المزوّد.
- يتم تطبيع الإدخال القديم `edge` الخاص بـ Microsoft إلى معرّف المزوّد `microsoft`.
- نموذج الملكية المفضل موجّه نحو الشركة: يمكن لـ Plugin مزوّد واحدة أن تملك
  مزوّدي النصوص، والكلام، والصور، والوسائط مستقبلًا مع إضافة OpenClaw
  لعقود هذه القدرات.

بالنسبة إلى فهم الصور/الصوت/الفيديو، تسجل Plugins
مزوّدًا مطبوعًا واحدًا لفهم الوسائط بدلًا من حاوية عامة للمفاتيح/القيم:

```ts
api.registerMediaUnderstandingProvider({
  id: "google",
  capabilities: ["image", "audio", "video"],
  describeImage: async (req) => ({ text: "..." }),
  transcribeAudio: async (req) => ({ text: "..." }),
  describeVideo: async (req) => ({ text: "..." }),
});
```

ملاحظات:

- أبقِ التنسيق، والبدائل، والإعدادات، وتوصيل القنوات في النواة.
- أبقِ سلوك المزوّد داخل Plugin المزوّد.
- يجب أن يبقى التوسع الإضافي مطبوعًا: أساليب اختيارية جديدة، وحقول نتائج اختيارية
  جديدة، وقدرات اختيارية جديدة.
- يتبع توليد الفيديو بالفعل النمط نفسه:
  - تملك النواة عقد القدرة ومساعد وقت التشغيل
  - تسجل Plugins المزوّدين `api.registerVideoGenerationProvider(...)`
  - تستهلك Plugins الميزات/القنوات `api.runtime.videoGeneration.*`

بالنسبة إلى مساعدات وقت التشغيل الخاصة بـ media-understanding، يمكن لـ Plugins استدعاء:

```ts
const image = await api.runtime.mediaUnderstanding.describeImageFile({
  filePath: "/tmp/inbound-photo.jpg",
  cfg: api.config,
  agentDir: "/tmp/agent",
});

const video = await api.runtime.mediaUnderstanding.describeVideoFile({
  filePath: "/tmp/inbound-video.mp4",
  cfg: api.config,
});
```

وبالنسبة إلى نسخ الصوت، يمكن لـ Plugins استخدام وقت تشغيل media-understanding
أو الاسم البديل القديم STT:

```ts
const { text } = await api.runtime.mediaUnderstanding.transcribeAudioFile({
  filePath: "/tmp/inbound-audio.ogg",
  cfg: api.config,
  // اختياري عندما يتعذر استنتاج MIME بشكل موثوق:
  mime: "audio/ogg",
});
```

ملاحظات:

- يُعد `api.runtime.mediaUnderstanding.*` السطح المشترك المفضل لـ
  فهم الصور/الصوت/الفيديو.
- يستخدم إعدادات الصوت الأساسية الخاصة بـ media-understanding (`tools.media.audio`) وترتيب بدائل المزوّد.
- يعيد `{ text: undefined }` عندما لا يتم إنتاج خرج نسخ (على سبيل المثال، عند تخطي الإدخال/عدم دعمه).
- يظل `api.runtime.stt.transcribeAudioFile(...)` اسمًا بديلًا للتوافق.

يمكن لـ Plugins أيضًا إطلاق تشغيلات Subagent خلفية عبر `api.runtime.subagent`:

```ts
const result = await api.runtime.subagent.run({
  sessionKey: "agent:main:subagent:search-helper",
  message: "Expand this query into focused follow-up searches.",
  provider: "openai",
  model: "gpt-4.1-mini",
  deliver: false,
});
```

ملاحظات:

- يُعد كل من `provider` و`model` تجاوزين اختياريين لكل تشغيل، وليس تغييرات مستمرة على الجلسة.
- لا يحترم OpenClaw حقول التجاوز هذه إلا للمستدعين الموثوقين.
- بالنسبة إلى تشغيلات الاحتياط المملوكة لـ Plugin، يجب على المشغّلين التفعيل الصريح عبر `plugins.entries.<id>.subagent.allowModelOverride: true`.
- استخدم `plugins.entries.<id>.subagent.allowedModels` لتقييد Plugins الموثوقة بأهداف معيارية محددة من نوع `provider/model`، أو `"*"` للسماح الصريح بأي هدف.
- تظل تشغيلات Subagent الخاصة بـ Plugin غير الموثوقة تعمل، لكن تُرفض طلبات التجاوز بدلًا من الرجوع بصمت إلى الاحتياط.

وبالنسبة إلى بحث الويب، يمكن لـ Plugins استهلاك مساعد وقت التشغيل المشترك بدلًا من
الوصول إلى توصيل أداة الوكيل:

```ts
const providers = api.runtime.webSearch.listProviders({
  config: api.config,
});

const result = await api.runtime.webSearch.search({
  config: api.config,
  args: {
    query: "OpenClaw plugin runtime helpers",
    count: 5,
  },
});
```

يمكن لـ Plugins أيضًا تسجيل مزوّدي بحث الويب عبر
`api.registerWebSearchProvider(...)`.

ملاحظات:

- أبقِ اختيار المزوّد، وحل بيانات الاعتماد، ودلالات الطلب المشتركة في النواة.
- استخدم مزوّدي بحث الويب لوسائل النقل الخاصة بالبحث المملوكة للمزوّد.
- يُعد `api.runtime.webSearch.*` السطح المشترك المفضل لـ Plugins الميزات/القنوات التي تحتاج إلى سلوك بحث من دون الاعتماد على غلاف أداة الوكيل.

### `api.runtime.imageGeneration`

```ts
const result = await api.runtime.imageGeneration.generate({
```

## مسارات HTTP الخاصة بـ Gateway

يمكن لـ Plugins كشف نقاط نهاية HTTP باستخدام `api.registerHttpRoute(...)`.

```ts
api.registerHttpRoute({
  path: "/acme/webhook",
  auth: "plugin",
  match: "exact",
  handler: async (_req, res) => {
    res.statusCode = 200;
    res.end("ok");
    return true;
  },
});
```

حقول المسار:

- `path`: مسار التوجيه تحت خادم HTTP الخاص بـ gateway.
- `auth`: مطلوب. استخدم `"gateway"` لطلب مصادقة gateway العادية، أو `"plugin"` لمصادقة/تحقق Webhook يُدار بواسطة Plugin.
- `match`: اختياري. `"exact"` ‏(الافتراضي) أو `"prefix"`.
- `replaceExisting`: اختياري. يسمح لـ Plugin نفسها باستبدال تسجيل المسار الموجود الخاص بها.
- `handler`: أعد `true` عندما يكون المسار قد عالج الطلب.

ملاحظات:

- تمت إزالة `api.registerHttpHandler(...)` وستتسبب في خطأ تحميل لـ Plugin. استخدم `api.registerHttpRoute(...)` بدلًا منها.
- يجب أن تعلن مسارات Plugin عن `auth` صراحةً.
- يتم رفض تعارضات `path + match` الدقيقة ما لم تكن `replaceExisting: true`، ولا يمكن لـ Plugin واحدة استبدال مسار Plugin أخرى.
- يتم رفض المسارات المتداخلة ذات مستويات `auth` المختلفة. أبقِ سلاسل السقوط `exact`/`prefix` ضمن مستوى المصادقة نفسه فقط.
- لا تتلقى مسارات `auth: "plugin"` تلقائيًا نطاقات وقت تشغيل operator. فهي مخصصة للـ webhooks/التحقق من التواقيع الذي تديره Plugin، وليست لاستدعاءات مساعدات Gateway ذات الامتيازات.
- تعمل مسارات `auth: "gateway"` داخل نطاق وقت تشغيل طلب Gateway، لكن هذا النطاق محافظ عمدًا:
  - تُبقي مصادقة bearer ذات السر المشترك (`gateway.auth.mode = "token"` / `"password"`) نطاقات وقت تشغيل مسارات Plugin مثبتة على `operator.write`، حتى لو أرسل المستدعي `x-openclaw-scopes`
  - تحترم أوضاع HTTP الموثوقة الحاملة للهوية (مثل `trusted-proxy` أو `gateway.auth.mode = "none"` على إدخال خاص) القيمة `x-openclaw-scopes` فقط عندما يكون الترويس حاضرًا صراحةً
  - إذا كانت `x-openclaw-scopes` غائبة في طلبات مسارات Plugin الحاملة للهوية تلك، فيعود نطاق وقت التشغيل إلى `operator.write`
- القاعدة العملية: لا تفترض أن مسار Plugin بمصادقة gateway هو سطح إدارة ضمني. إذا كان مسارك يحتاج إلى سلوك خاص بالإدارة فقط، فاطلب وضع مصادقة حاملًا للهوية ووثّق عقد الترويسة `x-openclaw-scopes` الصريح.

## مسارات استيراد Plugin SDK

استخدم المسارات الفرعية لـ SDK بدلًا من الاستيراد الأحادي `openclaw/plugin-sdk`
عند تأليف Plugins:

- `openclaw/plugin-sdk/plugin-entry` للعناصر الأولية الخاصة بتسجيل Plugin.
- `openclaw/plugin-sdk/core` للعقد العامة المشتركة الموجهة إلى Plugin.
- `openclaw/plugin-sdk/config-schema` لتصدير مخطط Zod الجذري الخاص بـ `openclaw.json`
  (`OpenClawSchema`).
- العناصر الأولية المستقرة للقنوات مثل `openclaw/plugin-sdk/channel-setup`,
  و`openclaw/plugin-sdk/setup-runtime`,
  و`openclaw/plugin-sdk/setup-adapter-runtime`,
  و`openclaw/plugin-sdk/setup-tools`,
  و`openclaw/plugin-sdk/channel-pairing`,
  و`openclaw/plugin-sdk/channel-contract`,
  و`openclaw/plugin-sdk/channel-feedback`,
  و`openclaw/plugin-sdk/channel-inbound`,
  و`openclaw/plugin-sdk/channel-lifecycle`,
  و`openclaw/plugin-sdk/channel-reply-pipeline`,
  و`openclaw/plugin-sdk/command-auth`,
  و`openclaw/plugin-sdk/secret-input`, و
  `openclaw/plugin-sdk/webhook-ingress` لتوصيل الإعداد/المصادقة/الرد/الـ webhook
  المشترك. ويُعد `channel-inbound` الموطن المشترك لآليات debounce، ومطابقة الإشارات،
  ومساعدات سياسة الإشارة الواردة، وتنسيق الأغلفة، ومساعدات سياق
  الأغلفة الواردة.
  ويُعد `channel-setup` وصلة الإعداد الضيقة الخاصة بالتثبيت الاختياري.

  ويُعد `setup-adapter-runtime` وصلة محولات إعداد الحساب الواعية بـ env.
  ويُعد `setup-tools` الوصلة الصغيرة الخاصة بمساعدات CLI/الأرشيف/التوثيق (`formatCliCommand`,
  و`detectBinary`, و`extractArchive`, و`resolveBrewExecutable`, و`formatDocsLink`,

- المسارات الفرعية الخاصة بالمجالات مثل `openclaw/plugin-sdk/channel-config-helpers`,
  و`openclaw/plugin-sdk/allow-from`,

  و`openclaw/plugin-sdk/lazy-runtime`,

  و`openclaw/plugin-sdk/text-runtime`,
  و`openclaw/plugin-sdk/runtime-store`, و

  ويُعد `telegram-command-config` الوصلة العامة الضيقة الخاصة
  بتطبيع/التحقق من الأوامر المخصصة في Telegram، وتظل متاحة حتى إذا كان سطح العقد المضمن الخاص بـ
  Telegram غير متاح مؤقتًا.
  ويُعد `text-runtime` الوصلة المشتركة للنص/Markdown/السجل، بما في ذلك
  إزالة النص المرئي للمساعد، ومساعدات عرض/تقطيع Markdown، ومساعدات التنقيح،
  ومساعدات directive-tag، وأدوات النص الآمن.
- بالنسبة إلى الوصلات الخاصة بالقنوات المعنية بالموافقات، يجب أن يُفضَّل عقد
  `approvalCapability` واحد على Plugin. ثم تقرأ النواة مصادقة الموافقة، والتسليم، والعرض،
  والتوجيه الأصلي، وسلوك المعالج الأصلي الكسول من خلال هذه القدرة الواحدة
  بدلًا من خلط سلوك الموافقة داخل حقول Plugin غير المرتبطة.
- تم إهمال `openclaw/plugin-sdk/channel-runtime` ولم يعد موجودًا إلا كغلاف
  توافق للPlugins الأقدم. يجب أن تستورد الشيفرة الجديدة العناصر العامة الأضيق بدلًا من ذلك، ويجب ألا تضيف شيفرة المستودع استيرادات جديدة لهذا الغلاف.
- تظل داخليات الامتدادات المضمنة خاصة. يجب أن تستخدم Plugins الخارجية فقط المسارات الفرعية `openclaw/plugin-sdk/*`. ويمكن لشيفرة OpenClaw الأساسية/الاختبارية استخدام
  نقاط الدخول العامة في المستودع تحت جذر حزمة Plugin مثل `index.js` و`api.js`,
  و`runtime-api.js`, و`setup-entry.js`، والملفات الضيقة النطاق مثل
  `login-qr-api.js`. ولا تستورد مطلقًا `src/*` الخاصة بحزمة Plugin من النواة أو من
  امتداد آخر.
- تقسيم نقطة الدخول في المستودع:
  `<plugin-package-root>/api.js` هو برميل المساعدات/الأنواع،
  و`<plugin-package-root>/runtime-api.js` هو البرميل الخاص بوقت التشغيل فقط،
  و`<plugin-package-root>/index.js` هو مدخل Plugin المضمنة،
  و`<plugin-package-root>/setup-entry.js` هو مدخل إعداد Plugin.
- أمثلة المزوّدين المضمنين الحالية:
  - يستخدم Anthropic `api.js` / `contract-api.js` لمساعدات بث Claude مثل
    `wrapAnthropicProviderStream`، ومساعدات beta-header، وتحليل `service_tier`.
  - يستخدم OpenAI `api.js` لبناة المزوّد، ومساعدات النموذج الافتراضي، وبناة
    المزوّد الفوري.
  - يستخدم OpenRouter `api.js` لباني المزوّد الخاص به بالإضافة إلى مساعدات
    التهيئة الأولية/الإعدادات، بينما لا يزال بإمكان `register.runtime.js` إعادة تصدير
    مساعدات `plugin-sdk/provider-stream` العامة للاستخدام المحلي في المستودع.
- تفضّل نقاط الدخول العامة المحمّلة عبر الواجهة الإعدادات النشطة الخاصة بوقت التشغيل
  عندما تكون موجودة، ثم تعود إلى ملف الإعدادات المحلول على القرص عندما لا يكون
  OpenClaw يقدّم بعد لقطة وقت تشغيل.
- تظل العناصر العامة المشتركة هي عقد SDK العامة المفضلة. لا تزال توجد مجموعة توافق صغيرة محفوظة من الوصلات المساعدة ذات العلامات الخاصة بالقنوات المضمنة. تعامل مع هذه على أنها وصلات صيانة/توافق للPlugins المضمنة، وليست أهداف استيراد جديدة لطرف ثالث؛ ويجب أن تهبط العقود الجديدة متعددة القنوات على المسارات الفرعية العامة `plugin-sdk/*` أو على البراميل المحلية الخاصة بـ Plugin مثل `api.js` /
  `runtime-api.js`.

ملاحظة التوافق:

- تجنب برميل الجذر `openclaw/plugin-sdk` في الشيفرة الجديدة.
- فضّل أولًا العناصر الأولية الضيقة والمستقرة. فالمسارات الفرعية الأحدث الخاصة بـ setup/pairing/reply/
  feedback/contract/inbound/threading/command/secret-input/webhook/infra/
  allowlist/status/message-tool هي العقد المقصود للأعمال الجديدة الخاصة بالPlugins
  المضمنة والخارجية.
  وينتمي تحليل/مطابقة الهدف إلى `openclaw/plugin-sdk/channel-targets`.
  وتنتمي بوابات إجراءات الرسائل ومساعدات message-id الخاصة بالتفاعلات إلى
  `openclaw/plugin-sdk/channel-actions`.
- لا تكون براميل المساعدات الخاصة بالامتدادات المضمنة مستقرة افتراضيًا. وإذا كان
  المساعد مطلوبًا فقط لامتداد مضمن، فأبقِه خلف الوصلة المحلية الخاصة بالامتداد `api.js` أو `runtime-api.js`
  بدلًا من ترقيته إلى
  `openclaw/plugin-sdk/<extension>`.
- يجب أن تكون وصلات المساعدات المشتركة الجديدة عامة، لا تحمل علامة قناة معينة. فتحليل
  الأهداف المشتركة ينتمي إلى `openclaw/plugin-sdk/channel-targets`؛ أما
  الداخلية الخاصة بالقنوات فتبقى خلف الوصلة المحلية `api.js` أو `runtime-api.js`
  الخاصة بالPlugin المالكة لها.
- توجد المسارات الفرعية الخاصة بالقدرات مثل `image-generation`,
  و`media-understanding`, و`speech` لأن Plugins المضمنة/الأصلية تستخدمها
  اليوم. ووجودها لا يعني بحد ذاته أن كل مساعد مُصدَّر هو
  عقد خارجي مجمّد طويل الأمد.

## مخططات أداة الرسائل

يجب أن تملك Plugins مساهمات schema الخاصة بـ `describeMessageTool(...)` للقنوات
في العناصر غير الرسائلية مثل التفاعلات، والقراءات، واستطلاعات الرأي.
كما يجب أن يستخدم عرض الإرسال المشترك عقد `MessagePresentation` العام
بدلًا من الحقول الأصلية الخاصة بالمزوّد مثل الأزرار، أو المكونات، أو الكتل، أو البطاقات.
راجع [عرض الرسائل](/ar/plugins/message-presentation) لمعرفة العقد،
وقواعد الاحتياط، وربط المزوّد، وقائمة التحقق الخاصة بمؤلفي Plugins.

تعلن Plugins القادرة على الإرسال عما يمكنها عرضه من خلال قدرات الرسائل:

- `presentation` لكتل العرض الدلالية (`text`, `context`, `divider`, `buttons`, `select`)
- `delivery-pin` لطلبات التسليم المثبتة

تقرر النواة ما إذا كانت ستعرض العرض بشكل أصلي أم ستخفضه إلى نص.
ولا تكشف مخارج هروب لواجهة مستخدم أصلية خاصة بالمزوّد من أداة الرسائل العامة.
ولا تزال مساعدات SDK المهملة الخاصة بالمخططات الأصلية القديمة مُصدَّرة من أجل
Plugins الخارجية الحالية، لكن يجب ألا تستخدمها Plugins الجديدة.

## حل هدف القناة

يجب أن تملك Plugins القنوات دلالات الأهداف الخاصة بالقناة. أبقِ
مضيف الإرسال المشترك عامًا واستخدم سطح محول الرسائل لقواعد المزوّد:

- يقرر `messaging.inferTargetChatType({ to })` ما إذا كان الهدف المطبع
  يجب أن يُعامل على أنه `direct` أو `group` أو `channel` قبل البحث في الدليل.
- يخبر `messaging.targetResolver.looksLikeId(raw, normalized)` النواة بما إذا كان
  الإدخال يجب أن يتجاوز مباشرةً إلى حل شبيه بالمعرّف بدلًا من البحث في الدليل.
- يُعد `messaging.targetResolver.resolveTarget(...)` الاحتياط الخاص بـ Plugin عندما
  تحتاج النواة إلى حل نهائي يملكه المزوّد بعد التطبيع أو بعد
  عدم العثور على نتيجة في الدليل.
- يملك `messaging.resolveOutboundSessionRoute(...)` بناء مسار الجلسة الخاص
  بالمزوّد بمجرد حل الهدف.

التقسيم الموصى به:

- استخدم `inferTargetChatType` لقرارات التصنيف التي يجب أن تحدث قبل
  البحث في الأقران/المجموعات.
- استخدم `looksLikeId` لفحوصات "عامِل هذا كمعرّف هدف صريح/أصلي".
- استخدم `resolveTarget` كاحتياط للتطبيع الخاص بالمزوّد، وليس من أجل
  البحث الواسع في الدليل.
- أبقِ المعرفات الأصلية الخاصة بالمزوّد مثل chat ids، وthread ids، وJIDs، والمقابض، وroom
  ids داخل قيم `target` أو داخل المعلمات الخاصة بالمزوّد، وليس داخل الحقول العامة لـ SDK.

## الأدلة المدعومة بالإعدادات

يجب أن تُبقي Plugins التي تستمد إدخالات الدليل من الإعدادات هذه المنطق داخل
Plugin وأن تعيد استخدام المساعدات المشتركة من
`openclaw/plugin-sdk/directory-runtime`.

استخدم هذا عندما تحتاج قناة إلى أقران/مجموعات مدعومة بالإعدادات مثل:

- أقران الرسائل الخاصة المدفوعين بقائمة السماح
- خرائط القنوات/المجموعات المضبوطة
- احتياطات الدليل الثابتة المحددة بالنطاق لكل حساب

لا تتعامل المساعدات المشتركة في `directory-runtime` إلا مع العمليات العامة:

- تصفية الاستعلام
- تطبيق الحدود
- مساعدات إزالة التكرار/التطبيع
- بناء `ChannelDirectoryEntry[]`

أما فحص الحساب الخاص بالقناة وتطبيع المعرّف فيجب أن يبقيا داخل
تنفيذ Plugin.

## فهارس المزوّدين

يمكن لـ Plugins المزوّدين تعريف فهارس النماذج للاستدلال باستخدام
`registerProvider({ catalog: { run(...) { ... } } })`.

تعيد `catalog.run(...)` الشكل نفسه الذي يكتبه OpenClaw داخل
`models.providers`:

- `{ provider }` لإدخال مزوّد واحد
- `{ providers }` لإدخالات عدة مزوّدين

استخدم `catalog` عندما تملك Plugin معرّفات نماذج خاصة بالمزوّد، أو إعدادات
افتراضية لـ base URL، أو بيانات وصفية للنماذج محكومة بالمصادقة.

يتحكم `catalog.order` في توقيت دمج فهرس Plugin نسبةً إلى المزوّدين
الضمنيين المدمجين في OpenClaw:

- `simple`: مزوّدون عاديون معتمدون على API-key أو env
- `profile`: مزوّدون يظهرون عند وجود ملفات تعريف مصادقة
- `paired`: مزوّدون يصنعون عدة إدخالات مزوّدين مترابطة
- `late`: تمريرة أخيرة، بعد بقية المزوّدين الضمنيين

يفوز المزوّدون اللاحقون عند تصادم المفاتيح، بحيث يمكن لـ Plugins
أن تتجاوز عمدًا إدخال مزوّد مدمجًا يحمل معرّف المزوّد نفسه.

التوافق:

- لا يزال `discovery` يعمل كاسم بديل قديم
- إذا تم تسجيل كل من `catalog` و`discovery`، يستخدم OpenClaw القيمة `catalog`

## فحص القناة للقراءة فقط

إذا سجّلت Plugin الخاصة بك قناة، ففضّل تنفيذ
`plugin.config.inspectAccount(cfg, accountId)` إلى جانب `resolveAccount(...)`.

السبب:

- يُعد `resolveAccount(...)` مسار وقت التشغيل. ويُسمح له بأن يفترض أن بيانات الاعتماد
  قد تبلورت بالكامل، ويمكنه الفشل سريعًا عند غياب الأسرار المطلوبة.
- لا ينبغي لمسارات الأوامر المخصصة للقراءة فقط مثل `openclaw status`, و`openclaw status --all`,
  و`openclaw channels status`, و`openclaw channels resolve`، وتدفقات doctor/config
  repair أن تحتاج إلى تبلور بيانات اعتماد وقت التشغيل لمجرد
  وصف الإعدادات.

السلوك الموصى به لـ `inspectAccount(...)`:

- أعد فقط حالة وصفية للحساب.
- حافظ على `enabled` و`configured`.
- ضمّن حقول مصدر/حالة بيانات الاعتماد عند الاقتضاء، مثل:
  - `tokenSource`, `tokenStatus`
  - `botTokenSource`, `botTokenStatus`
  - `appTokenSource`, `appTokenStatus`
  - `signingSecretSource`, `signingSecretStatus`
- لا تحتاج إلى إرجاع قيم الرموز الخام لمجرد الإبلاغ عن
  التوفر المخصص للقراءة فقط. يكفي إرجاع `tokenStatus: "available"` (وحقل المصدر المطابق
  له) للأوامر ذات نمط الحالة.
- استخدم `configured_unavailable` عندما يتم تكوين بيانات الاعتماد عبر SecretRef لكنها
  غير متاحة في مسار الأمر الحالي.

وهذا يسمح للأوامر المخصصة للقراءة فقط بالإبلاغ عن "مكوّن لكنه غير متاح في مسار هذا الأمر"
بدلًا من التعطل أو الإبلاغ الخاطئ عن أن الحساب غير مكوّن.

## حزم Package

يمكن أن يتضمن دليل Plugin ملف `package.json` مع `openclaw.extensions`:

```json
{
  "name": "my-pack",
  "openclaw": {
    "extensions": ["./src/safety.ts", "./src/tools.ts"],
    "setupEntry": "./src/setup-entry.ts"
  }
}
```

يصبح كل إدخال Plugin. وإذا كانت الحزمة تسرد عدة امتدادات، فيصبح معرّف Plugin
هو `name/<fileBase>`.

إذا كانت Plugin الخاصة بك تستورد تبعيات npm، فثبّتها في ذلك الدليل حتى
يكون `node_modules` متاحًا (`npm install` / `pnpm install`).

حاجز أمان: يجب أن يبقى كل إدخال من `openclaw.extensions` داخل دليل Plugin
بعد حل الروابط الرمزية. ويتم رفض الإدخالات التي تخرج من دليل الحزمة.

ملاحظة أمان: يقوم `openclaw plugins install` بتثبيت تبعيات Plugin باستخدام
`npm install --omit=dev --ignore-scripts` (من دون نصوص دورة حياة، ومن دون تبعيات تطوير وقت التشغيل). أبقِ أشجار تبعيات Plugin
"نقية JS/TS" وتجنب الحزم التي تتطلب بناء `postinstall`.

اختياري: يمكن أن يشير `openclaw.setupEntry` إلى وحدة خفيفة خاصة بالإعداد فقط.
عندما يحتاج OpenClaw إلى أسطح إعداد لقناة Plugin معطلة، أو
عندما تكون قناة Plugin مفعّلة لكنها لا تزال غير مكوّنة، فإنه يحمّل `setupEntry`
بدلًا من مدخل Plugin الكامل. وهذا يُبقي بدء التشغيل والإعداد أخف
عندما يوصّل مدخل Plugin الرئيسي أيضًا الأدوات، أو Hooks، أو أي شيفرة أخرى
خاصة بوقت التشغيل فقط.

اختياري: يمكن أن يُدخل `openclaw.startup.deferConfiguredChannelFullLoadUntilAfterListen`
قناة Plugin إلى مسار `setupEntry` نفسه خلال مرحلة
بدء التشغيل قبل الاستماع في gateway، حتى عندما تكون القناة مكوّنة بالفعل.

استخدم هذا فقط عندما يغطي `setupEntry` بالكامل سطح بدء التشغيل الذي يجب أن يوجد
قبل أن يبدأ gateway بالاستماع. وعمليًا، يعني هذا أن مدخل الإعداد
يجب أن يسجل كل قدرة مملوكة للقناة يعتمد عليها بدء التشغيل، مثل:

- تسجيل القناة نفسها
- أي مسارات HTTP يجب أن تكون متاحة قبل أن يبدأ gateway بالاستماع
- أي أساليب Gateway، أو أدوات، أو خدمات يجب أن توجد خلال النافذة الزمنية نفسها

إذا كان المدخل الكامل الخاص بك لا يزال يملك أي قدرة مطلوبة عند بدء التشغيل، فلا تفعّل
هذا العلم. أبقِ Plugin على السلوك الافتراضي ودَع OpenClaw يحمّل
المدخل الكامل أثناء بدء التشغيل.

يمكن للقنوات المضمنة أيضًا نشر مساعدات سطح عقد خاصة بالإعداد فقط يمكن للنواة
استشارتها قبل تحميل وقت تشغيل القناة الكامل. وسطح الترقية الحالي للإعداد
هو:

- `singleAccountKeysToMove`
- `namedAccountPromotionKeys`
- `resolveSingleAccountPromotionTarget(...)`

تستخدم النواة هذا السطح عندما تحتاج إلى ترقية إعدادات قناة قديمة ذات حساب واحد
إلى `channels.<id>.accounts.*` من دون تحميل مدخل Plugin الكامل.
ويُعد Matrix المثال المضمن الحالي: فهو ينقل فقط مفاتيح
المصادقة/التمهيد إلى حساب مُرقّى مسمّى عندما توجد حسابات مسمّاة بالفعل، ويمكنه
الاحتفاظ بمفتاح حساب افتراضي غير معياري مكوّن بدلًا من إنشاء
`accounts.default` دائمًا.

تُبقي محولات ترقيعات الإعداد تلك اكتشاف سطح العقد الخاص بالحزمة المضمنة كسولًا. ويبقى
وقت الاستيراد خفيفًا؛ ويتم تحميل سطح الترقية فقط عند أول استخدام بدلًا من
إعادة الدخول إلى بدء تشغيل القناة المضمنة عند استيراد الوحدة.

وعندما تتضمن أسطح بدء التشغيل تلك أساليب Gateway RPC، فأبقها على
بادئة خاصة بـ Plugin. تظل مساحات أسماء الإدارة الأساسية (`config.*`,
`exec.approvals.*`, `wizard.*`, `update.*`) محجوزة وتُحل دائمًا
إلى `operator.admin`، حتى إذا طلبت Plugin نطاقًا أضيق.

مثال:

```json
{
  "name": "@scope/my-channel",
  "openclaw": {
    "extensions": ["./index.ts"],
    "setupEntry": "./setup-entry.ts",
    "startup": {
      "deferConfiguredChannelFullLoadUntilAfterListen": true
    }
  }
}
```

### البيانات الوصفية لفهرس القناة

يمكن لـ Plugins القنوات الإعلان عن بيانات وصفية للإعداد/الاكتشاف عبر `openclaw.channel` وعن
تلميحات التثبيت عبر `openclaw.install`. وهذا يُبقي بيانات الفهرس الأساسية خالية من البيانات الثابتة.

مثال:

```json
{
  "name": "@openclaw/nextcloud-talk",
  "openclaw": {
    "extensions": ["./index.ts"],
    "channel": {
      "id": "nextcloud-talk",
      "label": "Nextcloud Talk",
      "selectionLabel": "Nextcloud Talk (self-hosted)",
      "docsPath": "/channels/nextcloud-talk",
      "docsLabel": "nextcloud-talk",
      "blurb": "Self-hosted chat via Nextcloud Talk webhook bots.",
      "order": 65,
      "aliases": ["nc-talk", "nc"]
    },
    "install": {
      "npmSpec": "@openclaw/nextcloud-talk",
      "localPath": "<bundled-plugin-local-path>",
      "defaultChoice": "npm"
    }
  }
}
```

الحقول المفيدة في `openclaw.channel` إلى جانب المثال الأدنى:

- `detailLabel`: تسمية ثانوية لأسطح الفهرس/الحالة الأكثر غنى
- `docsLabel`: تجاوز نص الرابط الخاص برابط المستندات
- `preferOver`: معرّفات Plugin/قنوات ذات أولوية أقل يجب أن يتفوق عليها إدخال هذا الفهرس
- `selectionDocsPrefix`, `selectionDocsOmitLabel`, `selectionExtras`: عناصر تحكم خاصة بنسخة سطح الاختيار
- `markdownCapable`: يميز القناة على أنها قادرة على Markdown من أجل قرارات التنسيق الصادر
- `exposure.configured`: إخفاء القناة من أسطح سرد القنوات المكوّنة عند ضبطها على `false`
- `exposure.setup`: إخفاء القناة من أدوات اختيار الإعداد/التهيئة التفاعلية عند ضبطها على `false`
- `exposure.docs`: تعليم القناة على أنها داخلية/خاصة لأسطح التنقل في المستندات
- `showConfigured` / `showInSetup`: أسماء بديلة قديمة لا تزال مقبولة للتوافق؛ ويفضل `exposure`
- `quickstartAllowFrom`: إدخال القناة ضمن تدفق `allowFrom` القياسي للبدء السريع
- `forceAccountBinding`: طلب ربط حساب صريح حتى عند وجود حساب واحد فقط
- `preferSessionLookupForAnnounceTarget`: تفضيل البحث في الجلسة عند حل أهداف الإعلان

يمكن لـ OpenClaw أيضًا دمج **فهارس قنوات خارجية** (على سبيل المثال، تصدير من
سجل MPM). ضع ملف JSON في أحد المسارات التالية:

- `~/.openclaw/mpm/plugins.json`
- `~/.openclaw/mpm/catalog.json`
- `~/.openclaw/plugins/catalog.json`

أو وجّه `OPENCLAW_PLUGIN_CATALOG_PATHS` (أو `OPENCLAW_MPM_CATALOG_PATHS`) إلى
ملف JSON واحد أو أكثر (محددة بفواصل/فواصل منقوطة/تنسيق `PATH`). يجب أن يحتوي كل ملف
على `{ "entries": [ { "name": "@scope/pkg", "openclaw": { "channel": {...}, "install": {...} } } ] }`. كما يقبل المحلل أيضًا
`"packages"` أو `"plugins"` كأسماء بديلة قديمة للمفتاح `"entries"`.

## Plugins محرك السياق

تملك Plugins محرك السياق تنسيق سياق الجلسة من أجل الإدخال، والتجميع،
وCompaction. قم بتسجيلها من Plugin الخاصة بك باستخدام
`api.registerContextEngine(id, factory)`، ثم اختر المحرك النشط باستخدام
`plugins.slots.contextEngine`.

استخدم هذا عندما تحتاج Plugin الخاصة بك إلى استبدال أو توسيع مسار
السياق الافتراضي بدلًا من مجرد إضافة بحث في الذاكرة أو Hooks.

```ts
import { buildMemorySystemPromptAddition } from "openclaw/plugin-sdk/core";

export default function (api) {
  api.registerContextEngine("lossless-claw", () => ({
    info: { id: "lossless-claw", name: "Lossless Claw", ownsCompaction: true },
    async ingest() {
      return { ingested: true };
    },
    async assemble({ messages, availableTools, citationsMode }) {
      return {
        messages,
        estimatedTokens: 0,
        systemPromptAddition: buildMemorySystemPromptAddition({
          availableTools: availableTools ?? new Set(),
          citationsMode,
        }),
      };
    },
    async compact() {
      return { ok: true, compacted: false };
    },
  }));
}
```

إذا كان محركك **لا** يملك خوارزمية Compaction، فأبقِ `compact()`
منفذة وفوّضها صراحةً:

```ts
import {
  buildMemorySystemPromptAddition,
  delegateCompactionToRuntime,
} from "openclaw/plugin-sdk/core";

export default function (api) {
  api.registerContextEngine("my-memory-engine", () => ({
    info: {
      id: "my-memory-engine",
      name: "My Memory Engine",
      ownsCompaction: false,
    },
    async ingest() {
      return { ingested: true };
    },
    async assemble({ messages, availableTools, citationsMode }) {
      return {
        messages,
        estimatedTokens: 0,
        systemPromptAddition: buildMemorySystemPromptAddition({
          availableTools: availableTools ?? new Set(),
          citationsMode,
        }),
      };
    },
    async compact(params) {
      return await delegateCompactionToRuntime(params);
    },
  }));
}
```

## إضافة قدرة جديدة

عندما تحتاج Plugin إلى سلوك لا يناسب API الحالية، فلا تتجاوز
نظام Plugin باختراق خاص. بل أضف القدرة المفقودة.

التسلسل الموصى به:

1. عرّف العقد الأساسي
   حدّد ما السلوك المشترك الذي يجب أن تملكه النواة: السياسة، والبدائل، ودمج الإعدادات،
   ودورة الحياة، والدلالات المواجهة للقنوات، وشكل مساعد وقت التشغيل.
2. أضف أسطح تسجيل/وقت تشغيل Plugin مطبوعة
   وسّع `OpenClawPluginApi` و/أو `api.runtime` بأصغر
   سطح قدرة مطبوع مفيد.
3. وصّل مستهلكي النواة + القنوات/الميزات
   يجب أن تستهلك القنوات وPlugins الميزات القدرة الجديدة عبر النواة،
   وليس عبر استيراد تنفيذ مزوّد مباشرةً.
4. سجّل تنفيذات المزوّدين
   بعد ذلك تسجل Plugins المزوّدين واجهاتها الخلفية مقابل القدرة.
5. أضف تغطية للعقد
   أضف اختبارات حتى تبقى الملكية وشكل التسجيل صريحين مع مرور الوقت.

هذه هي الطريقة التي يبقى بها OpenClaw ذا رأي من دون أن يصبح مشفرًا بشكل صلب وفق
رؤية مزوّد واحد للعالم. راجع [كتاب طبخ القدرات](/ar/plugins/architecture)
للحصول على قائمة تحقق عملية للملفات ومثال مفصل.

### قائمة تحقق القدرة

عندما تضيف قدرة جديدة، يجب أن يلمس التنفيذ عادةً
هذه الأسطح معًا:

- أنواع العقد الأساسية في `src/<capability>/types.ts`
- مشغّل/مساعد وقت التشغيل الأساسي في `src/<capability>/runtime.ts`
- سطح تسجيل Plugin API في `src/plugins/types.ts`
- توصيل سجل Plugin في `src/plugins/registry.ts`
- كشف وقت تشغيل Plugin في `src/plugins/runtime/*` عندما تحتاج
  Plugins الميزات/القنوات إلى استهلاكه
- مساعدات الالتقاط/الاختبار في `src/test-utils/plugin-registration.ts`
- تأكيدات الملكية/العقد في `src/plugins/contracts/registry.ts`
- مستندات المشغّل/Plugin في `docs/`

إذا كان أحد هذه الأسطح مفقودًا، فهذه عادةً علامة على أن القدرة
لم تُدمج بالكامل بعد.

### قالب القدرة

النمط الأدنى:

```ts
// العقد الأساسي
export type VideoGenerationProviderPlugin = {
  id: string;
  label: string;
  generateVideo: (req: VideoGenerationRequest) => Promise<VideoGenerationResult>;
};

// Plugin API
api.registerVideoGenerationProvider({
  id: "openai",
  label: "OpenAI",
  async generateVideo(req) {
    return await generateOpenAiVideo(req);
  },
});

// مساعد وقت تشغيل مشترك لPlugins الميزات/القنوات
const clip = await api.runtime.videoGeneration.generate({
  prompt: "Show the robot walking through the lab.",
  cfg,
});
```

نمط اختبار العقد:

```ts
expect(findVideoGenerationProviderIdsForPlugin("openai")).toEqual(["openai"]);
```

ويبقي ذلك القاعدة بسيطة:

- تملك النواة عقد القدرة + التنسيق
- تملك Plugins المزوّدين تنفيذات المزوّد
- تستهلك Plugins الميزات/القنوات مساعدات وقت التشغيل
- تُبقي اختبارات العقود الملكية صريحة

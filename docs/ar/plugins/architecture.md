## نموذج القدرات العام

القدرات هي النموذج العام لـ **Plugins الأصلية** داخل OpenClaw. كل
Plugin أصلية في OpenClaw تسجّل نفسها مقابل نوع قدرة واحد أو أكثر:

| القدرة                | أسلوب التسجيل                                   | أمثلة على Plugins                    |
| --------------------- | ------------------------------------------------ | ------------------------------------ |
| الاستدلال النصي       | `api.registerProvider(...)`                      | `openai`، `anthropic`                |
| واجهة استدلال خلفية CLI | `api.registerCliBackend(...)`                  | `openai`، `anthropic`                |
| الكلام                | `api.registerSpeechProvider(...)`                | `elevenlabs`، `microsoft`            |
| النسخ الفوري          | `api.registerRealtimeTranscriptionProvider(...)` | `openai`                             |
| الصوت الفوري          | `api.registerRealtimeVoiceProvider(...)`         | `openai`                             |
| فهم الوسائط           | `api.registerMediaUnderstandingProvider(...)`    | `openai`، `google`                   |
| توليد الصور           | `api.registerImageGenerationProvider(...)`       | `openai`، `google`، `fal`، `minimax` |
| توليد الموسيقى        | `api.registerMusicGenerationProvider(...)`       | `google`، `minimax`                  |
| توليد الفيديو         | `api.registerVideoGenerationProvider(...)`       | `qwen`                               |
| جلب الويب             | `api.registerWebFetchProvider(...)`              | `firecrawl`                          |
| البحث على الويب       | `api.registerWebSearchProvider(...)`             | `google`                             |
| القناة / المراسلة     | `api.registerChannel(...)`                       | `msteams`، `matrix`                  |
| اكتشاف Gateway        | `api.registerGatewayDiscoveryService(...)`       | `bonjour`                            |

أي Plugin تسجل صفر قدرات لكنها توفر hooks أو tools أو خدمات اكتشاف
أو خدمات خلفية تُعد Plugin **قديمة تعتمد على hooks فقط**. ولا يزال هذا النمط
مدعومًا بالكامل.

### الموقف من التوافق الخارجي

تم اعتماد نموذج القدرات في core ويُستخدم من Plugins المضمّنة/الأصلية
اليوم، لكن توافق Plugins الخارجية لا يزال يحتاج إلى معيار أوضح من مجرد
"تم تصديره، إذًا فهو مجمّد."

| وضع Plugin                                      | التوجيه                                                                                         |
| ------------------------------------------------ | ------------------------------------------------------------------------------------------------ |
| Plugins الخارجية الحالية                         | أبقِ التكاملات القائمة على hooks تعمل؛ فهذا هو خط الأساس للتوافق.                              |
| Plugins المضمّنة/الأصلية الجديدة                 | فضّل تسجيل القدرات الصريح على التوغلات الخاصة بالمزوّد أو تصميمات hook-only الجديدة.           |
| Plugins الخارجية التي تعتمد تسجيل القدرات        | مسموح، لكن اعتبر الأسطح المساعدة الخاصة بكل قدرة قابلة للتطور ما لم تُعلِّمها المستندات بأنها مستقرة. |

يُعد تسجيل القدرات هو الاتجاه المقصود. وتبقى hooks القديمة
أكثر مسارات عدم الكسر أمانًا للـ Plugins الخارجية أثناء الانتقال. وليست جميع
المسارات الفرعية المساعدة المصدّرة متساوية — ففضّل العقود الضيقة الموثقة على
عمليات التصدير المساعدة العرضية.

### أشكال Plugin

يصنّف OpenClaw كل Plugin محملة إلى شكل بناءً على سلوك التسجيل الفعلي
(وليس فقط البيانات الوصفية الساكنة):

- **plain-capability**: تسجل نوع قدرة واحدًا فقط (مثل
  Plugin مزوّد فقط مثل `mistral`).
- **hybrid-capability**: تسجل أنواع قدرات متعددة (مثل
  `openai` التي تملك الاستدلال النصي والكلام وفهم الوسائط وتوليد
  الصور).
- **hook-only**: تسجل hooks فقط (مكتوبة النوع أو مخصصة)، من دون قدرات،
  أو tools، أو commands، أو services.
- **non-capability**: تسجل tools أو commands أو services أو routes لكن من دون
  قدرات.

استخدم `openclaw plugins inspect <id>` لرؤية شكل Plugin وتفصيل
قدراتها. راجع [مرجع CLI](/ar/cli/plugins#inspect) للتفاصيل.

### hooks القديمة

لا يزال hook ‏`before_agent_start` مدعومًا كمسار توافق للـ Plugins
التي تعتمد على hooks فقط. ولا تزال Plugins حقيقية قديمة تعتمد عليه.

الاتجاه:

- أبقِه يعمل
- وثّقه بوصفه قديمًا
- فضّل `before_model_resolve` لعمل تجاوزات النموذج/المزوّد
- فضّل `before_prompt_build` لعمل تعديل الموجّه
- أزله فقط بعد انخفاض الاستخدام الفعلي وإثبات تغطية fixtures لأمان الترحيل

### إشارات التوافق

عند تشغيل `openclaw doctor` أو `openclaw plugins inspect <id>`، قد ترى
أحد هذه الوسوم:

| الإشارة                    | المعنى                                                      |
| -------------------------- | ------------------------------------------------------------ |
| **config valid**           | يجري تحليل Config بشكل سليم ويتم حل Plugins                |
| **compatibility advisory** | تستخدم Plugin نمطًا مدعومًا لكنه أقدم (مثل `hook-only`)    |
| **legacy warning**         | تستخدم Plugin القيمة `before_agent_start`، وهي مهجورة      |
| **hard error**             | Config غير صالحة أو فشل تحميل Plugin                        |

لن يكسر كل من `hook-only` و`before_agent_start` Plugin الخاصة بك اليوم:
فـ `hook-only` مجرد إشارة استشارية، و`before_agent_start` لا يؤدي إلا إلى تحذير. وتظهر
هذه الإشارات أيضًا في `openclaw status --all` و`openclaw plugins doctor`.

## نظرة عامة على البنية

يحتوي نظام Plugin في OpenClaw على أربع طبقات:

1. **Manifest + الاكتشاف**
   يعثر OpenClaw على Plugins مرشحة من المسارات المكوّنة، وجذور مساحة العمل،
   وجذور Plugins العامة، وPlugins المضمّنة. ويقرأ الاكتشاف ملفات manifest الأصلية
   `openclaw.plugin.json` بالإضافة إلى ملفات manifest المجمّعة المدعومة أولًا.
2. **التمكين + التحقق**
   يقرر core ما إذا كانت Plugin المكتشفة ممكّنة أو معطلة أو محظورة أو
   مختارة لفتحة حصرية مثل الذاكرة.
3. **التحميل وقت التشغيل**
   تُحمَّل Plugins OpenClaw الأصلية داخل العملية عبر jiti وتُسجل
   القدرات في سجل مركزي. وتُطبَّع الحزم المتوافقة إلى سجلات داخل
   السجل من دون استيراد شيفرة وقت التشغيل.
4. **استهلاك السطح**
   يقرأ بقية OpenClaw السجل لعرض tools والقنوات وإعداد
   المزوّد وhooks وHTTP routes وCLI commands والخدمات.

وبالنسبة إلى CLI الخاصة بالـ Plugin تحديدًا، فإن اكتشاف أوامر الجذر مقسّم إلى مرحلتين:

- تأتي البيانات الوصفية وقت التحليل من `registerCli(..., { descriptors: [...] })`
- ويمكن أن تظل وحدة CLI الحقيقية للـ Plugin مؤجلة وتُسجل عند أول استدعاء

يُبقي ذلك شيفرة CLI المملوكة للـ Plugin داخل Plugin نفسها مع السماح لـ OpenClaw
بحجز أسماء أوامر الجذر قبل التحليل.

الحد الفاصل التصميمي المهم:

- يجب أن يعمل التحقق من manifest/config من **manifest/schema metadata**
  من دون تنفيذ شيفرة Plugin
- قد يؤدي اكتشاف القدرات الأصلية إلى تحميل شيفرة إدخال Plugin موثوقة لبناء
  لقطة سجل غير مفعّلة
- يأتي سلوك وقت التشغيل الأصلي من مسار وحدة Plugin ‏`register(api)` مع
  `api.registrationMode === "full"`

يسمح هذا الفصل لـ OpenClaw بالتحقق من config، وشرح Plugins المفقودة/المعطلة،
وبناء تلميحات UI/schema قبل أن يصبح وقت التشغيل الكامل نشطًا.

### تخطيط التفعيل

يُعد تخطيط التفعيل جزءًا من مستوى التحكم. ويمكن للمستدعين أن يسألوا عن
Plugins المرتبطة بأمر أو مزوّد أو قناة أو route أو harness وكيل أو
قدرة معيّنة قبل تحميل سجلات وقت تشغيل أوسع.

يبقي المخطط سلوك manifest الحالي متوافقًا:

- حقول `activation.*` هي تلميحات صريحة للمخطط
- تبقى `providers` و`channels` و`commandAliases` و`setup.providers`،
  و`contracts.tools`، وhooks مسار الاحتياط لملكية manifest
- تظل API الخاصة بالمخطط التي تعيد المعرّفات فقط متاحة للمستدعين الحاليين
- تعرض API الخاصة بالخطة وسوم الأسباب حتى تتمكن التشخيصات من التمييز بين التلميحات الصريحة وبين احتياط الملكية

لا تتعامل مع `activation` على أنه lifecycle hook أو بديل عن
`register(...)`. فهو بيانات وصفية تُستخدم لتضييق التحميل. وفضّل حقول الملكية
عندما تكون تصف العلاقة بالفعل؛ واستخدم `activation` فقط لتلميحات إضافية
للمخطط.

### Plugins القنوات وأداة الرسائل المشتركة

لا تحتاج Plugins القنوات إلى تسجيل أداة send/edit/react منفصلة لإجراءات
الدردشة العادية. إذ يحتفظ OpenClaw بأداة `message` مشتركة واحدة في core،
وتتولى Plugins القنوات الاكتشاف والتنفيذ الخاصين بالقناة خلفها.

الحد الفاصل الحالي هو:

- يملك core مضيف أداة `message` المشتركة، وربط الموجّه، وحفظ الجلسة/الخيط،
  وتوجيه التنفيذ
- تملك Plugins القنوات اكتشاف الإجراءات ذات النطاق، واكتشاف القدرات، وأي
  أجزاء schema خاصة بالقناة
- تملك Plugins القنوات قواعد محادثة الجلسات الخاصة بالمزوّد، مثل
  كيفية ترميز معرّفات المحادثات لمعرّفات الخيوط أو وراثتها من المحادثات الأصلية
- تنفّذ Plugins القنوات الإجراء النهائي عبر محول الإجراءات الخاص بها

بالنسبة إلى Plugins القنوات، يكون سطح SDK هو
`ChannelMessageActionAdapter.describeMessageTool(...)`. وتسمح عملية الاكتشاف الموحدة
هذه للـ Plugin بإرجاع الإجراءات المرئية والقدرات ومساهمات schema
معًا حتى لا تنحرف هذه الأجزاء عن بعضها.

عندما يحمل معامل في أداة الرسائل الخاصة بالقناة مصدر وسائط مثل
مسار محلي أو URL لوسائط بعيدة، فيجب على Plugin أيضًا إرجاع
`mediaSourceParams` من `describeMessageTool(...)`. ويستخدم core هذه
القائمة الصريحة لتطبيق تطبيع مسارات sandbox وتلميحات الوصول إلى الوسائط الصادرة
من دون تثبيت أسماء المعاملات المملوكة للـ Plugin في الشيفرة.
فضّل الخرائط ذات النطاق الإجرائي هناك، وليس قائمة مسطحة واحدة على مستوى القناة،
حتى لا يتم تطبيع معامل وسائط خاص بملف تعريف فقط على إجراءات غير مرتبطة مثل
`send`.

يمرّر core نطاق وقت التشغيل إلى خطوة الاكتشاف هذه. وتشمل الحقول المهمة:

- `accountId`
- `currentChannelId`
- `currentThreadTs`
- `currentMessageId`
- `sessionKey`
- `sessionId`
- `agentId`
- `requesterSenderId` الوارد الموثوق

وهذا مهم للـ Plugins الحساسة للسياق. إذ يمكن للقناة إخفاء إجراءات
الرسائل أو إظهارها استنادًا إلى الحساب النشط أو الغرفة/الخيط/الرسالة الحالية أو
هوية الطالب الموثوقة من دون تثبيت فروع خاصة بالقناة داخل
أداة `message` الأساسية.

ولهذا السبب تظل تغييرات التوجيه في embedded-runner من أعمال Plugin:
فالمشغّل مسؤول عن تمرير هوية الدردشة/الجلسة الحالية إلى حد
اكتشاف Plugin بحيث تعرض أداة `message` المشتركة السطح الصحيح المملوك للقناة
للدور الحالي.

بالنسبة إلى مساعدات التنفيذ المملوكة للقناة، يجب أن تُبقي Plugins المضمّنة وقت تشغيل التنفيذ
داخل وحدات الامتداد الخاصة بها. فلم يعد core يملك أوقات تشغيل إجراءات الرسائل
الخاصة بـ Discord أو Slack أو Telegram أو WhatsApp ضمن `src/agents/tools`.
ولا ننشر مسارات فرعية منفصلة من نوع `plugin-sdk/*-action-runtime`، ويجب أن
تستورد Plugins المضمّنة شيفرة وقت التشغيل المحلية الخاصة بها مباشرةً من
وحداتها المملوكة للامتداد.

وينطبق الحد الفاصل نفسه على مسارات SDK المسماة باسم المزوّد عمومًا: يجب ألا
يستورد core حاويات convenience خاصة بقنوات مثل Slack أو Discord أو Signal أو
WhatsApp أو امتدادات مشابهة. وإذا احتاج core إلى سلوك ما، فإما أن يستهلك
الحاوية `api.ts` / `runtime-api.ts` الخاصة بـ Plugin المضمّنة نفسها، أو يرفع
الحاجة إلى قدرة عامة ضيقة في SDK المشتركة.

بالنسبة إلى الاستطلاعات تحديدًا، يوجد مساران للتنفيذ:

- `outbound.sendPoll` هو خط الأساس المشترك للقنوات التي تناسب
  نموذج الاستطلاع العام
- `actions.handleAction("poll")` هو المسار المفضل لدلالات الاستطلاع الخاصة بالقناة أو لمعلمات استطلاع إضافية

يؤجل core الآن تحليل الاستطلاع المشترك إلى ما بعد رفض توجيه استطلاع الـ Plugin
للإجراء، بحيث يمكن لمعالجات الاستطلاع المملوكة للـ Plugin قبول حقول استطلاع
خاصة بالقناة من دون أن يحظرها محلل الاستطلاع العام أولًا.

راجع [البنية الداخلية لمعمارية Plugin](/ar/plugins/architecture-internals) للحصول على تسلسل بدء التشغيل الكامل.

## نموذج ملكية القدرات

يتعامل OpenClaw مع Plugin الأصلية على أنها حد الملكية الخاص بـ **شركة** أو
**ميزة**، وليس كمجموعة عشوائية من تكاملات غير مترابطة.

هذا يعني:

- يجب أن تملك Plugin الخاصة بالشركة عادةً جميع الأسطح المواجهة لـ OpenClaw
  الخاصة بتلك الشركة
- يجب أن تملك Plugin الخاصة بالميزة عادةً السطح الكامل للميزة التي تقدمها
- يجب أن تستهلك القنوات قدرات core المشتركة بدلًا من إعادة تنفيذ سلوك
  المزوّد بشكل مخصص

<Accordion title="أمثلة على أنماط الملكية عبر Plugins المضمّنة">
  - **مزوّد متعدد القدرات**: تملك `openai` الاستدلال النصي، والكلام، والصوت
    الفوري، وفهم الوسائط، وتوليد الصور. وتملك `google` الاستدلال
    النصي بالإضافة إلى فهم الوسائط وتوليد الصور والبحث على الويب.
    وتملك `qwen` الاستدلال النصي بالإضافة إلى فهم الوسائط وتوليد الفيديو.
  - **مزوّد بقدرة واحدة**: تملك `elevenlabs` و`microsoft` الكلام؛
    وتملك `firecrawl` جلب الويب؛ وتمتلك `minimax` / `mistral` / `moonshot` / `zai`
    واجهات فهم الوسائط الخلفية.
  - **Plugin ميزة**: تملك `voice-call` نقل المكالمات وtools وCLI وroutes
    وجسر تدفق وسائط Twilio، لكنها تستهلك القدرات المشتركة للكلام
    والنسخ الفوري والصوت الفوري بدلًا من استيراد Plugins المزودين مباشرةً.
</Accordion>

الحالة النهائية المقصودة هي:

- تعيش OpenAI في Plugin واحدة حتى إذا امتدت عبر نماذج النص والكلام والصور
  والفيديو مستقبلًا
- يمكن لمزوّد آخر أن يفعل الشيء نفسه لمساحته الخاصة
- لا تهتم القنوات بأي Plugin مزوّد تملك المزوّد؛ فهي تستهلك عقد
  القدرة المشتركة التي يكشفها core

وهذا هو التمييز الأساسي:

- **plugin** = حد الملكية
- **capability** = عقد core يمكن لعدة Plugins تنفيذه أو استهلاكه

لذلك إذا أضاف OpenClaw مجالًا جديدًا مثل الفيديو، فالسؤال الأول ليس
"أي مزوّد يجب أن يضمّن معالجة الفيديو بشكل صلب؟" بل السؤال الأول هو "ما
هو عقد القدرة الأساسي للفيديو؟" وبعد وجود هذا العقد، يمكن لـ Plugins المزوّدين
التسجيل ضده ويمكن لـ Plugins القنوات/الميزات استهلاكه.

إذا لم تكن القدرة موجودة بعد، فإن الخطوة الصحيحة عادةً هي:

1. تعريف القدرة المفقودة في core
2. كشفها عبر API/وقت تشغيل Plugin بشكل مكتوب النوع
3. توصيل القنوات/الميزات بهذه القدرة
4. ترك Plugins المزوّدين تسجّل التنفيذات

يبقي هذا الملكية صريحة مع تجنب سلوك core الذي يعتمد على مزوّد واحد
أو مسار شيفرة خاص بـ Plugin واحدة.

### تدرّج القدرات

استخدم هذا النموذج الذهني عند تحديد مكان الشيفرة:

- **طبقة القدرة الأساسية**: التنسيق المشترك، والسياسة، والاحتياط، وقواعد
  دمج الإعداد، ودلالات التسليم، والعقود المكتوبة النوع
- **طبقة Plugin المزوّد**: واجهات API الخاصة بالمزوّد، والمصادقة، وفهارس النماذج، وتركيب
  الكلام، وتوليد الصور، وواجهات الفيديو الخلفية المستقبلية، ونقاط نهاية الاستخدام
- **طبقة Plugin القناة/الميزة**: تكامل Slack/Discord/voice-call/إلخ
  الذي يستهلك القدرات الأساسية ويعرضها على سطح ما

على سبيل المثال، يتبع TTS هذا الشكل:

- يملك core سياسة TTS وقت الرد، وترتيب الاحتياط، والتفضيلات، وتسليم القنوات
- تملك `openai` و`elevenlabs` و`microsoft` تنفيذات التركيب
- تستهلك `voice-call` مساعد وقت تشغيل TTS الخاص بالهاتف

ويجب تفضيل هذا النمط نفسه للقدرات المستقبلية.

### مثال على Plugin شركة متعددة القدرات

يجب أن تبدو Plugin الشركة مترابطة من الخارج. وإذا كانت لدى OpenClaw
عقود مشتركة للنماذج، والكلام، والنسخ الفوري، والصوت الفوري، وفهم الوسائط،
وتوليد الصور، وتوليد الفيديو، وجلب الويب، والبحث على الويب،
فيمكن لمزوّد أن يملك كل أسطحه في مكان واحد:

```ts
import type { OpenClawPluginDefinition } from "openclaw/plugin-sdk/plugin-entry";
import {
  describeImageWithModel,
  transcribeOpenAiCompatibleAudio,
} from "openclaw/plugin-sdk/media-understanding";

const plugin: OpenClawPluginDefinition = {
  id: "exampleai",
  name: "ExampleAI",
  register(api) {
    api.registerProvider({
      id: "exampleai",
      // خطافات وقت التشغيل الخاصة بالمصادقة/فهرس النماذج
    });

    api.registerSpeechProvider({
      id: "exampleai",
      // إعدادات الكلام الخاصة بالمزوّد — نفّذ واجهة SpeechProviderPlugin مباشرةً
    });

    api.registerMediaUnderstandingProvider({
      id: "exampleai",
      capabilities: ["image", "audio", "video"],
      async describeImage(req) {
        return describeImageWithModel({
          provider: "exampleai",
          model: req.model,
          input: req.input,
        });
      },
      async transcribeAudio(req) {
        return transcribeOpenAiCompatibleAudio({
          provider: "exampleai",
          model: req.model,
          input: req.input,
        });
      },
    });

    api.registerWebSearchProvider(
      createPluginBackedWebSearchProvider({
        id: "exampleai-search",
        // منطق بيانات الاعتماد + الجلب
      }),
    );
  },
};

export default plugin;
```

المهم ليس أسماء المساعدات الدقيقة. بل الشكل هو المهم:

- Plugin واحدة تملك سطح المزوّد
- لا يزال core يملك عقود القدرات
- تستهلك Plugins القنوات والميزات مساعدات `api.runtime.*`، وليس شيفرة المزوّد
- يمكن لاختبارات العقود أن تؤكد أن Plugin قد سجّلت القدرات التي
  تدعي أنها تملكها

### مثال قدرة: فهم الفيديو

يتعامل OpenClaw بالفعل مع فهم الصور/الصوت/الفيديو كقدرة مشتركة واحدة.
وينطبق نموذج الملكية نفسه هناك:

1. يعرّف core عقد فهم الوسائط
2. تسجل Plugins المزوّدين القيم `describeImage` و`transcribeAudio` و
   `describeVideo` حسب الاقتضاء
3. تستهلك Plugins القنوات والميزات السلوك الأساسي المشترك بدلًا من
   التوصيل المباشر إلى شيفرة المزوّد

وهذا يتجنب تضمين افتراضات فيديو خاصة بمزوّد واحد داخل core. إذ تملك Plugin
سطح المزوّد؛ بينما يملك core عقد القدرة وسلوك الاحتياط.

ويستخدم توليد الفيديو بالفعل التسلسل نفسه: يملك core عقد
القدرة المكتوب النوع ومساعد وقت التشغيل، وتسجّل Plugins المزوّدين
تنفيذات `api.registerVideoGenerationProvider(...)` ضده.

هل تحتاج إلى قائمة تحقق عملية للإطلاق؟ راجع
[دليل القدرة العملي](/ar/plugins/architecture).

## العقود والفرض

سطح API الخاص بالـ Plugin مكتوب النوع ومركزي عمدًا في
`OpenClawPluginApi`. ويعرّف هذا العقد نقاط التسجيل المدعومة
ومساعدات وقت التشغيل التي يمكن أن تعتمد عليها Plugin.

لماذا يهم ذلك:

- يحصل مؤلفو Plugins على معيار داخلي واحد مستقر
- يمكن لـ core رفض الملكية المكررة مثل تسجيل Pluginين لمعرّف مزوّد واحد
- يمكن لبدء التشغيل عرض تشخيصات عملية للتسجيل المشوّه
- يمكن لاختبارات العقود فرض ملكية Plugins المضمّنة ومنع الانحراف الصامت

هناك طبقتان من الفرض:

1. **فرض التسجيل وقت التشغيل**
   يتحقق سجل Plugins من التسجيلات أثناء تحميل Plugins. ومن الأمثلة:
   معرّفات المزوّدين المكررة، ومعرّفات مزوّد الكلام المكررة، والتسجيلات
   المشوهة التي تنتج تشخيصات Plugin بدلًا من سلوك غير معرّف.
2. **اختبارات العقود**
   يتم التقاط Plugins المضمّنة في سجلات العقود أثناء تشغيل الاختبارات بحيث
   يمكن لـ OpenClaw تأكيد الملكية صراحةً. واليوم يُستخدم هذا
   لمزوّدي النماذج، ومزوّدي الكلام، ومزوّدي البحث على الويب، وملكية
   التسجيل المضمّن.

والأثر العملي هو أن OpenClaw يعرف مسبقًا أي Plugin تملك أي
سطح. ويسمح ذلك لـ core والقنوات بالتركيب السلس لأن الملكية
معلنة، ومكتوبة النوع، وقابلة للاختبار بدلًا من أن تكون ضمنية.

### ما الذي يجب أن يكون في العقد

العقود الجيدة للـ Plugin تكون:

- مكتوبة النوع
- صغيرة
- خاصة بالقدرة
- مملوكة للـ core
- قابلة لإعادة الاستخدام بواسطة عدة Plugins
- قابلة للاستهلاك من القنوات/الميزات من دون معرفة بالمزوّد

العقود السيئة للـ Plugin تكون:

- سياسة خاصة بالمزوّد مخفية داخل core
- منافذ هروب خاصة بـ Plugin واحدة تتجاوز السجل
- شيفرة قناة تصل مباشرةً إلى تنفيذ مزوّد
- كائنات وقت تشغيل مخصصة ليست جزءًا من `OpenClawPluginApi` أو
  `api.runtime`

عند الشك، ارفع مستوى التجريد: عرّف القدرة أولًا، ثم دع
Plugins تتصل بها.

## نموذج التنفيذ

تعمل Plugins OpenClaw الأصلية **داخل العملية** مع Gateway. وهي غير
معزولة. فالـ Plugin الأصلية المحمّلة تملك حد الثقة على مستوى العملية نفسه الذي
تملكه شيفرة core.

النتائج المترتبة:

- يمكن لـ Plugin أصلية تسجيل tools ومعالجات شبكة وhooks وخدمات
- يمكن لعلة في Plugin أصلية أن تتسبب في تعطل gateway أو زعزعتها
- تعادل Plugin أصلية خبيثة تنفيذ شيفرة عشوائية داخل عملية OpenClaw

تكون الحزم المتوافقة أكثر أمانًا افتراضيًا لأن OpenClaw يتعامل معها حاليًا
كحزم بيانات وصفية/محتوى. وفي الإصدارات الحالية، يعني هذا غالبًا
Skills المضمنة.

استخدم قوائم السماح ومسارات التثبيت/التحميل الصريحة للـ Plugins غير المضمنة. وتعامل
مع Plugins مساحة العمل على أنها شيفرة وقت تطوير، لا إعدادات إنتاج افتراضية.

بالنسبة إلى أسماء حزم مساحة العمل المضمنة، أبقِ معرّف Plugin مثبتًا في اسم npm:
`@openclaw/<id>` افتراضيًا، أو لاحقة مكتوبة النوع معتمدة مثل
`-provider` أو `-plugin` أو `-speech` أو `-sandbox` أو `-media-understanding` عندما
تكشف الحزمة عمدًا دورًا أضيق للـ Plugin.

ملاحظة ثقة مهمة:

- تثق `plugins.allow` في **معرّفات Plugins**، لا في مصدرها.
- تتعمد Plugin في مساحة العمل لها المعرّف نفسه مثل Plugin مضمّنة أن تظلل
  النسخة المضمّنة عندما تكون Plugin مساحة العمل تلك مفعّلة/موجودة في قائمة السماح.
- هذا سلوك طبيعي ومفيد للتطوير المحلي، واختبار التصحيحات، والإصلاحات السريعة.
- يتم حل ثقة Plugins المضمّنة من لقطة المصدر — manifest و
  الشيفرة الموجودة على القرص وقت التحميل — وليس من بيانات تعريف التثبيت. فلا يمكن
  لسجل تثبيت تالف أو مستبدَل أن يوسّع سطح الثقة الخاص بـ Plugin مضمّنة بصمت
  إلى ما يتجاوز ما يدّعيه المصدر الفعلي.

## حد التصدير

تصدّر OpenClaw القدرات، لا وسائل التنفيذ المريحة.

أبقِ تسجيل القدرات عامًا. وقلّص عمليات تصدير المساعدات غير العقدية:

- المسارات الفرعية المساعدة الخاصة بالـ Plugins المضمّنة
- المسارات الفرعية الخاصة ببنية وقت التشغيل غير المقصودة كواجهة API عامة
- المساعدات المريحة الخاصة بالمزوّد
- مساعدات الإعداد/التهيئة الأولية التي تُعد تفاصيل تنفيذ

لا تزال بعض المسارات الفرعية المساعدة للـ Plugins المضمّنة موجودة في خريطة
تصدير SDK المولدة لأغراض التوافق وصيانة Plugins المضمّنة. وتشمل الأمثلة الحالية
`plugin-sdk/feishu` و`plugin-sdk/feishu-setup` و`plugin-sdk/zalo` و
`plugin-sdk/zalo-setup` وعدة مفاصل `plugin-sdk/matrix*`. تعامل مع هذه
على أنها عمليات تصدير محجوزة لتفاصيل التنفيذ، وليست النمط الموصى به في SDK
للـ Plugins الخارجية الجديدة.

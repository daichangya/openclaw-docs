تغطي هذه الصفحة البنية الداخلية لنظام Plugins في OpenClaw.

## نموذج القدرات العام

القدرات هي النموذج العام لـ **Plugins الأصلية** داخل OpenClaw. كل
Plugin أصلية في OpenClaw تسجّل نفسها مقابل نوع قدرة واحد أو أكثر:

| القدرة | طريقة التسجيل | أمثلة على Plugins |
| ---------------------- | ------------------------------------------------ | ------------------------------------ |
| الاستدلال النصي | `api.registerProvider(...)` | `openai`, `anthropic` |
| الواجهة الخلفية للاستدلال عبر CLI | `api.registerCliBackend(...)` | `openai`, `anthropic` |
| الكلام | `api.registerSpeechProvider(...)` | `elevenlabs`, `microsoft` |
| النسخ الفوري | `api.registerRealtimeTranscriptionProvider(...)` | `openai` |
| الصوت الفوري | `api.registerRealtimeVoiceProvider(...)` | `openai` |
| فهم الوسائط | `api.registerMediaUnderstandingProvider(...)` | `openai`, `google` |
| توليد الصور | `api.registerImageGenerationProvider(...)` | `openai`, `google`, `fal`, `minimax` |
| توليد الموسيقى | `api.registerMusicGenerationProvider(...)` | `google`, `minimax` |
| توليد الفيديو | `api.registerVideoGenerationProvider(...)` | `qwen` |
| جلب الويب | `api.registerWebFetchProvider(...)` | `firecrawl` |
| البحث على الويب | `api.registerWebSearchProvider(...)` | `google` |
| القنوات / المراسلة | `api.registerChannel(...)` | `msteams`, `matrix` |

الـ Plugin التي تسجّل صفر قدرات لكنها توفّر خطافات أو أدوات أو
خدمات تُعد Plugin **قديمة تعتمد على الخطافات فقط**. ولا يزال هذا النمط
مدعومًا بالكامل.

### موقف التوافق الخارجي

تم اعتماد نموذج القدرات في القلب ويُستخدم اليوم بواسطة Plugins
المضمّنة/الأصلية، لكن توافق Plugins الخارجية لا يزال يحتاج إلى معيار
أشد إحكامًا من مجرد "تم تصديره، إذن فهو ثابت."

الإرشادات الحالية:

- **Plugins الخارجية الحالية:** حافظ على عمل التكاملات القائمة على الخطافات؛ واعتبر
  هذا هو خط الأساس للتوافق
- **Plugins المضمّنة/الأصلية الجديدة:** فضّل تسجيل القدرات الصريح بدل
  الاختراقات الخاصة بالمورّد أو التصاميم الجديدة المعتمدة على الخطافات فقط
- **Plugins الخارجية التي تعتمد تسجيل القدرات:** هذا مسموح، لكن اعتبر
  أسطح المساعدة الخاصة بكل قدرة في طور التطور ما لم تُعلّم المستندات
  عقدًا ما على أنه مستقر صراحةً

قاعدة عملية:

- واجهات API الخاصة بتسجيل القدرات هي الاتجاه المقصود
- تظل الخطافات القديمة المسار الأكثر أمانًا لتجنب الكسر بالنسبة إلى Plugins الخارجية أثناء
  الانتقال
- ليست كل المسارات الفرعية المصدَّرة متساوية؛ ففضّل العقد الضيق الموثق،
  وليس صادرات المساعدة العرضية

### أشكال Plugin

يصنّف OpenClaw كل Plugin محمّلة إلى شكل بناءً على سلوكها الفعلي
في التسجيل (وليس فقط البيانات الوصفية الثابتة):

- **plain-capability** -- تسجّل نوع قدرة واحدًا بالضبط (مثل
  Plugin لموفّر فقط مثل `mistral`)
- **hybrid-capability** -- تسجّل أنواع قدرات متعددة (مثلًا
  `openai` تملك الاستدلال النصي والكلام وفهم الوسائط وتوليد
  الصور)
- **hook-only** -- تسجّل الخطافات فقط (مكتوبة النوع أو مخصصة)، من دون قدرات،
  أو أدوات، أو أوامر، أو خدمات
- **non-capability** -- تسجّل أدوات أو أوامر أو خدمات أو مسارات، لكن من دون
  قدرات

استخدم `openclaw plugins inspect <id>` لرؤية شكل Plugin وتفصيل
القدرات فيها. راجع [مرجع CLI](/cli/plugins#inspect) لمزيد من التفاصيل.

### الخطافات القديمة

لا يزال الخطاف `before_agent_start` مدعومًا كمسار توافق للـ Plugins
المعتمدة على الخطافات فقط. ولا تزال Plugins القديمة في العالم الحقيقي تعتمد عليه.

الاتجاه:

- أبقه عاملًا
- وثّقه على أنه قديم
- فضّل `before_model_resolve` لأعمال تجاوز النموذج/الموفّر
- فضّل `before_prompt_build` لأعمال تعديل المطالبة
- لا تزلْه إلا بعد انخفاض الاستخدام الفعلي وتثبت تغطية الـ fixtures أمان الترحيل

### إشارات التوافق

عند تشغيل `openclaw doctor` أو `openclaw plugins inspect <id>`، قد ترى
إحدى هذه العلامات:

| الإشارة | المعنى |
| -------------------------- | ------------------------------------------------------------ |
| **config valid** | يتم تحليل الإعدادات بنجاح ويتم حل Plugins |
| **compatibility advisory** | تستخدم Plugin نمطًا مدعومًا لكنه أقدم (مثل `hook-only`) |
| **legacy warning** | تستخدم Plugin الخطاف `before_agent_start`، وهو مهجور |
| **hard error** | الإعدادات غير صالحة أو فشل تحميل Plugin |

لن تؤدي `hook-only` ولا `before_agent_start` إلى كسر Plugin الخاصة بك اليوم --
فـ `hook-only` مجرد إرشاد، و`before_agent_start` لا يطلق إلا تحذيرًا. وتظهر
هذه الإشارات أيضًا في `openclaw status --all` و`openclaw plugins doctor`.

## نظرة عامة على البنية

يحتوي نظام Plugins في OpenClaw على أربع طبقات:

1. **البيان + الاكتشاف**
   يعثر OpenClaw على Plugins المرشحة من المسارات المهيأة، وجذور مساحات العمل،
   وجذور الإضافات العامة، والإضافات المضمّنة. ويقرأ الاكتشاف أولًا
   بيانات `openclaw.plugin.json` الأصلية بالإضافة إلى بيانات الحِزم المدعومة.
2. **التمكين + التحقق**
   يقرر القلب ما إذا كانت Plugin المكتشفة مفعّلة أو معطّلة أو محجوبة أو
   مختارة لفتحة حصرية مثل الذاكرة.
3. **التحميل في وقت التشغيل**
   يتم تحميل Plugins OpenClaw الأصلية داخل العملية عبر jiti وتسجّل
   القدرات في سجل مركزي. أما الحِزم المتوافقة فتُطبّع إلى
   سجلات داخل السجل من دون استيراد كود وقت التشغيل.
4. **استهلاك السطح**
   يقرأ بقية OpenClaw السجل لعرض الأدوات، والقنوات، وإعداد
   الموفّرين، والخطافات، ومسارات HTTP، وأوامر CLI، والخدمات.

وبالنسبة إلى CLI الخاص بالـ Plugin تحديدًا، يُقسَّم اكتشاف الأمر الجذري إلى مرحلتين:

- تأتي البيانات الوصفية وقت التحليل من `registerCli(..., { descriptors: [...] })`
- ويمكن أن تبقى وحدة CLI الحقيقية الخاصة بالـ Plugin كسولة وتُسجَّل عند أول استدعاء

وهذا يبقي كود CLI المملوك للـ Plugin داخل Plugin مع السماح لـ OpenClaw
بحجز أسماء الأوامر الجذرية قبل التحليل.

الحد التصميمي المهم:

- يجب أن يعمل الاكتشاف + التحقق من الإعدادات انطلاقًا من **بيانات البيان/المخطط**
  من دون تنفيذ كود Plugin
- يأتي السلوك الأصلي في وقت التشغيل من مسار `register(api)` الخاص بوحدة Plugin

يسمح هذا الفصل لـ OpenClaw بالتحقق من الإعدادات، وشرح Plugins المفقودة/المعطلة،
وبناء تلميحات الواجهة/المخطط قبل تنشيط وقت التشغيل الكامل.

### Plugins القنوات وأداة الرسائل المشتركة

لا تحتاج Plugins القنوات إلى تسجيل أداة مستقلة للإرسال/التعديل/التفاعل من أجل
إجراءات الدردشة العادية. إذ يحتفظ OpenClaw بأداة `message` مشتركة واحدة في القلب،
وتملك Plugins القنوات الاكتشاف والتنفيذ الخاصين بالقناة خلفها.

الحد الحالي هو:

- يملك القلب مضيف أداة `message` المشتركة، وتوصيل المطالبات، ومسك دفاتر
  الجلسة/السلسلة، وإرسال التنفيذ
- تملك Plugins القنوات اكتشاف الإجراءات المحددة النطاق، واكتشاف القدرات،
  وأي أجزاء مخطط خاصة بالقناة
- تملك Plugins القنوات قواعد المحادثة الخاصة بالمزوّد على مستوى الجلسة، مثل
  كيفية ترميز معرّفات المحادثة لمعرّفات السلاسل أو وراثتها من المحادثات الأصل
- تنفذ Plugins القنوات الإجراء النهائي عبر مكيّف الإجراءات الخاص بها

وبالنسبة إلى Plugins القنوات، فإن سطح SDK هو
`ChannelMessageActionAdapter.describeMessageTool(...)`. يتيح نداء الاكتشاف
الموحّد هذا للـ Plugin إرجاع إجراءاتها المرئية، وقدراتها، ومساهماتها
في المخطط معًا حتى لا تنحرف هذه الأجزاء عن بعضها.

وعندما يحمل أحد معاملات أداة الرسائل الخاصة بالقناة مصدر وسائط مثل
مسار محلي أو عنوان URL لوسائط بعيدة، ينبغي أيضًا أن تعيد Plugin
`mediaSourceParams` من `describeMessageTool(...)`. يستخدم القلب هذه القائمة
الصريحة لتطبيق تطبيع مسارات العزل وتلميحات الوصول إلى الوسائط الصادرة
من دون ترميز أسماء معاملات تملكها Plugin داخل القلب. وفضّل
الخرائط المحددة الإجراء هناك، لا قائمة مسطحة واحدة على مستوى القناة،
حتى لا يتم تطبيع معامل وسائط خاص بالملف الشخصي على إجراءات غير مرتبطة مثل
`send`.

يمرر القلب نطاق وقت التشغيل إلى خطوة الاكتشاف هذه. وتتضمن الحقول المهمة:

- `accountId`
- `currentChannelId`
- `currentThreadTs`
- `currentMessageId`
- `sessionKey`
- `sessionId`
- `agentId`
- `requesterSenderId` الوارد الموثوق

وهذا مهم بالنسبة إلى Plugins الحساسة للسياق. إذ يمكن للقناة إخفاء أو إظهار
إجراءات الرسائل بناءً على الحساب النشط أو الغرفة/السلسلة/الرسالة الحالية أو
هوية الطالب الموثوق بها، من دون ترميز فروع خاصة بالقناة داخل
أداة `message` في القلب.

ولهذا السبب لا تزال تغييرات توجيه embedded-runner عملًا خاصًا بالـ Plugin: إذ يكون
الـ runner مسؤولًا عن تمرير هوية الدردشة/الجلسة الحالية إلى حد
اكتشاف Plugin بحيث تعرض أداة `message` المشتركة السطح الصحيح المملوك للقناة
في الدورة الحالية.

أما بالنسبة إلى مساعدات التنفيذ المملوكة للقناة، فينبغي أن تُبقي Plugins المضمّنة
وقت تشغيل التنفيذ داخل وحدات الإضافة الخاصة بها. لم يعد القلب يملك
أوقات تشغيل إجراءات رسائل Discord أو Slack أو Telegram أو WhatsApp تحت `src/agents/tools`.
ولا ننشر مسارات فرعية مستقلة من نوع `plugin-sdk/*-action-runtime`، وينبغي أن
تستورد Plugins المضمّنة كود وقت التشغيل المحلي الخاص بها مباشرةً من
الوحدات التي تملكها الإضافة نفسها.

وينطبق الحد نفسه على مسارات SDK المسماة باسم الموفّر عمومًا: لا ينبغي
للقلب أن يستورد براميل convenience خاصة بالقناة لـ Slack أو Discord أو Signal أو
WhatsApp أو إضافات مشابهة. وإذا احتاج القلب إلى سلوك ما، فإما أن يستهلك
البرميل `api.ts` / `runtime-api.ts` الخاص بالـ Plugin المضمّنة نفسها، أو يرفع
الحاجة إلى قدرة عامة ضيقة داخل SDK المشتركة.

وبالنسبة إلى الاستطلاعات تحديدًا، يوجد مسارا تنفيذ:

- `outbound.sendPoll` هو خط الأساس المشترك للقنوات التي تناسب
  نموذج الاستطلاع العام
- `actions.handleAction("poll")` هو المسار المفضّل لدلالات الاستطلاع الخاصة بالقناة
  أو لمعاملات استطلاع إضافية

يؤجل القلب الآن تحليل الاستطلاع المشترك إلى ما بعد رفض إرسال
الاستطلاع الخاص بالـ Plugin للإجراء، بحيث يمكن لمعالجات الاستطلاع المملوكة للـ Plugin
قبول حقول استطلاع خاصة بالقناة من دون أن يحظرها محلل الاستطلاع العام أولًا.

راجع [مسار التحميل](#load-pipeline) للاطلاع على تسلسل بدء التشغيل الكامل.

## نموذج ملكية القدرات

يتعامل OpenClaw مع الـ Plugin الأصلية على أنها حد الملكية لـ **شركة** أو
**ميزة**، وليس كحقيبة تجمع تكاملات غير مترابطة.

وهذا يعني:

- عادةً ما ينبغي أن تملك Plugin الشركة جميع الأسطح المواجهة لـ OpenClaw الخاصة بتلك الشركة
- عادةً ما ينبغي أن تملك Plugin الميزة السطح الكامل للميزة التي تقدمها
- ينبغي أن تستهلك القنوات القدرات المشتركة في القلب بدل إعادة تنفيذ
  سلوك الموفّرين على نحو مخصص

أمثلة:

- تملك Plugin المضمّنة `openai` سلوك موفّر نماذج OpenAI وسلوك OpenAI الخاص
  بالكلام + الصوت الفوري + فهم الوسائط + توليد الصور
- تملك Plugin المضمّنة `elevenlabs` سلوك الكلام في ElevenLabs
- تملك Plugin المضمّنة `microsoft` سلوك الكلام في Microsoft
- تملك Plugin المضمّنة `google` سلوك موفّر نماذج Google بالإضافة إلى
  سلوك Google الخاص بفهم الوسائط + توليد الصور + البحث على الويب
- تملك Plugin المضمّنة `firecrawl` سلوك جلب الويب في Firecrawl
- تملك Plugins المضمّنة `minimax` و`mistral` و`moonshot` و`zai`
  واجهاتها الخلفية الخاصة بفهم الوسائط
- تملك Plugin المضمّنة `qwen` سلوك موفّر النصوص في Qwen بالإضافة إلى
  سلوك فهم الوسائط وتوليد الفيديو
- تُعد Plugin `voice-call` Plugin ميزة: فهي تملك نقل المكالمات، والأدوات،
  وCLI، والمسارات، وجسر تدفق الوسائط مع Twilio، لكنها تستهلك قدرات الكلام المشتركة
  بالإضافة إلى النسخ الفوري والصوت الفوري بدلًا من
  استيراد Plugins المورّد مباشرةً

الحالة النهائية المقصودة هي:

- تعيش OpenAI في Plugin واحدة حتى لو امتدت عبر النماذج النصية والكلام والصور
  والفيديو مستقبلًا
- ويمكن لمورّد آخر أن يفعل الشيء نفسه لمساحة السطح الخاصة به
- ولا تهتم القنوات بأي Plugin مورّد تملك الموفّر؛ إذ تستهلك عقد القدرة
  المشترك الذي يعرّضه القلب

وهذا هو التمييز الرئيسي:

- **plugin** = حد الملكية
- **capability** = عقد القلب الذي يمكن أن تنفّذه أو تستهلكه Plugins متعددة

لذا إذا أضاف OpenClaw مجالًا جديدًا مثل الفيديو، فالسؤال الأول ليس
"أي موفّر يجب أن يضمّن معالجة الفيديو مباشرة؟" بل السؤال الأول هو "ما هو
عقد قدرة الفيديو في القلب؟" وبمجرد وجود هذا العقد، يمكن لـ Plugins المورّدين
التسجيل مقابله ويمكن لـ Plugins القنوات/الميزات استهلاكه.

إذا لم تكن القدرة موجودة بعد، فعادةً ما تكون الخطوة الصحيحة هي:

1. تعريف القدرة الناقصة في القلب
2. تعريضها عبر API/وقت تشغيل Plugin بطريقة مكتوبة النوع
3. توصيل القنوات/الميزات بهذه القدرة
4. ترك Plugins المورّدين تسجّل التنفيذات

يبقي هذا الملكية صريحة مع تجنب سلوك في القلب يعتمد على
مورّد واحد أو مسار كود خاص بـ Plugin واحدة.

### طبقات القدرات

استخدم هذا النموذج الذهني عند تقرير مكان الكود:

- **طبقة القدرة في القلب**: التنسيق المشترك، والسياسة، والرجوع الاحتياطي، وقواعد
  دمج الإعدادات، ودلالات التسليم، والعقود مكتوبة النوع
- **طبقة Plugin المورّد**: واجهات API الخاصة بالمورّد، والمصادقة، وكتالوجات النماذج، وSynthesizing الكلام،
  وتوليد الصور، وواجهات الفيديو الخلفية المستقبلية، ونقاط استخدام الخدمة
- **طبقة Plugin القناة/الميزة**: تكاملات Slack/Discord/voice-call/إلخ
  التي تستهلك قدرات القلب وتعرضها على سطح ما

على سبيل المثال، يتبع TTS هذا الشكل:

- يملك القلب سياسة TTS وقت الرد، وترتيب الرجوع الاحتياطي، والتفضيلات، وتسليم القناة
- تملك `openai` و`elevenlabs` و`microsoft` تنفيذات synthesis
- تستهلك `voice-call` مساعد وقت تشغيل TTS الخاص بالهاتف

ويُفضّل النمط نفسه للقدرات المستقبلية.

### مثال على Plugin شركة متعددة القدرات

يجب أن تبدو Plugin الشركة متماسكة من الخارج. إذا كان لدى OpenClaw
عقود مشتركة للنماذج، والكلام، والنسخ الفوري، والصوت الفوري، وفهم
الوسائط، وتوليد الصور، وتوليد الفيديو، وجلب الويب، والبحث على الويب،
فيمكن لمورّد أن يملك جميع أسطحه في مكان واحد:

```ts
import type { OpenClawPluginDefinition } from "openclaw/plugin-sdk/plugin-entry";
import {
  describeImageWithModel,
  transcribeOpenAiCompatibleAudio,
} from "openclaw/plugin-sdk/media-understanding";

const plugin: OpenClawPluginDefinition = {
  id: "exampleai",
  name: "ExampleAI",
  register(api) {
    api.registerProvider({
      id: "exampleai",
      // auth/model catalog/runtime hooks
    });

    api.registerSpeechProvider({
      id: "exampleai",
      // vendor speech config — implement the SpeechProviderPlugin interface directly
    });

    api.registerMediaUnderstandingProvider({
      id: "exampleai",
      capabilities: ["image", "audio", "video"],
      async describeImage(req) {
        return describeImageWithModel({
          provider: "exampleai",
          model: req.model,
          input: req.input,
        });
      },
      async transcribeAudio(req) {
        return transcribeOpenAiCompatibleAudio({
          provider: "exampleai",
          model: req.model,
          input: req.input,
        });
      },
    });

    api.registerWebSearchProvider(
      createPluginBackedWebSearchProvider({
        id: "exampleai-search",
        // credential + fetch logic
      }),
    );
  },
};

export default plugin;
```

المهم ليس أسماء المساعدات الدقيقة. بل المهم هو الشكل:

- Plugin واحدة تملك سطح المورّد
- لا يزال القلب يملك عقود القدرات
- تستهلك Plugins القنوات والميزات مساعدات `api.runtime.*`، وليس كود المورّد
- يمكن لاختبارات العقد أن تؤكد أن Plugin سجلت القدرات التي
  تزعم أنها تملكها

### مثال قدرة: فهم الفيديو

يعامل OpenClaw بالفعل فهم الصور/الصوت/الفيديو كقدرة مشتركة
واحدة. وينطبق نموذج الملكية نفسه هناك:

1. يعرّف القلب عقد فهم الوسائط
2. تسجل Plugins المورّدين `describeImage` و`transcribeAudio` و
   `describeVideo` بحسب الاقتضاء
3. تستهلك Plugins القنوات والميزات سلوك القلب المشترك بدل
   التوصيل مباشرة إلى كود المورّد

وهذا يتجنب غرس افتراضات فيديو موفّر واحد داخل القلب. فالـ Plugin تملك
سطح المورّد؛ والقلب يملك عقد القدرة وسلوك الرجوع الاحتياطي.

ويستخدم توليد الفيديو بالفعل التسلسل نفسه: يملك القلب
عقد القدرة مكتوب النوع ومساعد وقت التشغيل، وتسجّل Plugins المورّدين
تنفيذات `api.registerVideoGenerationProvider(...)` مقابله.

هل تحتاج إلى قائمة تحقق عملية للإطلاق؟ راجع
[كتاب وصفات القدرات](/ar/plugins/architecture).

## العقود والإنفاذ

سطح API الخاص بالـ Plugin مكتوب النوع عمدًا ومركزي داخل
`OpenClawPluginApi`. ويعرّف هذا العقد نقاط التسجيل المدعومة
ومساعدات وقت التشغيل التي قد تعتمد عليها Plugin.

سبب أهمية ذلك:

- يحصل مؤلفو Plugins على معيار داخلي مستقر واحد
- يستطيع القلب رفض الملكية المكررة مثل تسجيل Pluginين للمعرّف نفسه
  لموفّر ما
- يستطيع بدء التشغيل إظهار تشخيصات عملية للتسجيلات المشوهة
- يمكن لاختبارات العقد فرض ملكية Plugins المضمّنة ومنع الانجراف الصامت

هناك طبقتان من الإنفاذ:

1. **إنفاذ التسجيل في وقت التشغيل**
   يتحقق سجل Plugins من التسجيلات أثناء تحميل Plugins. من الأمثلة:
   معرّفات الموفّرين المكررة، ومعرّفات موفّري الكلام المكررة، والتسجيلات
   المشوهة التي تنتج تشخيصات للـ Plugin بدلًا من سلوك غير معرّف.
2. **اختبارات العقد**
   يتم التقاط Plugins المضمّنة في سجلات عقود أثناء تشغيل الاختبارات حتى
   يتمكن OpenClaw من تأكيد الملكية صراحةً. ويُستخدم هذا اليوم بالنسبة إلى
   موفّري النماذج، وموفّري الكلام، وموفّري البحث على الويب، وملكية التسجيل المضمّنة.

والأثر العملي هو أن OpenClaw يعرف مسبقًا أي Plugin تملك أي
سطح. وهذا يتيح للقلب والقنوات التركيب بسلاسة لأن الملكية
معلنة ومكتوبة النوع وقابلة للاختبار بدلًا من أن تكون ضمنية.

### ما الذي ينتمي إلى عقد

عقود Plugins الجيدة تكون:

- مكتوبة النوع
- صغيرة
- خاصة بقدرة محددة
- مملوكة من القلب
- قابلة لإعادة الاستخدام من Plugins متعددة
- قابلة للاستهلاك من القنوات/الميزات من دون معرفة بالمورّد

أما عقود Plugins السيئة فهي:

- سياسة خاصة بالمورّد مخفية في القلب
- منافذ هروب خاصة بPlugin واحدة تتجاوز السجل
- كود القناة الذي يصل مباشرةً إلى تنفيذ خاص بمورّد
- كائنات وقت تشغيل مخصصة ليست جزءًا من `OpenClawPluginApi` أو
  `api.runtime`

عند الشك، ارفع مستوى التجريد: عرّف القدرة أولًا، ثم دع Plugins تتصل بها.

## نموذج التنفيذ

تعمل Plugins الأصلية في OpenClaw **داخل العملية نفسها** مع Gateway. وهي ليست
معزولة. وتمتلك Plugin الأصلية المحمّلة حد ثقة على مستوى العملية مماثلًا
لكود القلب.

النتائج:

- يمكن للـ Plugin الأصلية تسجيل أدوات، ومعالجات شبكية، وخطافات، وخدمات
- يمكن لخلل في Plugin أصلية أن يُسقط الـ gateway أو يزعزع استقراره
- تعادل Plugin أصلية خبيثة تنفيذ كود عشوائي داخل عملية OpenClaw

تكون الحِزم المتوافقة أكثر أمانًا افتراضيًا لأن OpenClaw يعاملها حاليًا
كحِزم بيانات/محتوى. وفي الإصدارات الحالية، يعني ذلك في الغالب
Skills المضمّنة.

استخدم قوائم السماح ومسارات التثبيت/التحميل الصريحة بالنسبة إلى Plugins غير
المضمّنة. وتعامل مع Plugins مساحة العمل على أنها كود وقت تطوير، لا
إعدادات افتراضية للإنتاج.

بالنسبة إلى أسماء الحزم المضمّنة في مساحة العمل، أبقِ معرّف Plugin مرتبطًا باسم
npm: ‏`@openclaw/<id>` افتراضيًا، أو لاحقة مكتوبة النوع معتمدة مثل
`-provider` أو `-plugin` أو `-speech` أو `-sandbox` أو `-media-understanding`
عندما تكشف الحزمة عمدًا دور Plugin أضيق.

ملاحظة ثقة مهمة:

- يثق `plugins.allow` في **معرّفات Plugins**، لا في مصدرها.
- تتعمد Plugin مساحة العمل التي تحمل المعرّف نفسه لPlugin مضمّنة أن تحجب
  النسخة المضمّنة عندما تكون Plugin مساحة العمل تلك مفعّلة/ضمن قائمة السماح.
- وهذا طبيعي ومفيد للتطوير المحلي، واختبار التصحيحات، والإصلاحات السريعة.

## حد التصدير

يُصدّر OpenClaw القدرات، لا وسائل الراحة التنفيذية.

أبقِ تسجيل القدرات عامًا. وقلّص صادرات المساعدة غير العقدية:

- المسارات الفرعية الخاصة بمساعدات Plugins المضمّنة
- المسارات الفرعية الخاصة ببنية وقت التشغيل غير المقصودة كواجهة API عامة
- وسائل الراحة الخاصة بالمورّد
- مساعدات الإعداد/التهيئة الأولية التي تُعد تفاصيل تنفيذ

لا تزال بعض المسارات الفرعية لمساعدات Plugins المضمّنة موجودة في خريطة تصدير SDK المولّدة من أجل التوافق وصيانة Plugins المضمّنة. ومن الأمثلة الحالية
`plugin-sdk/feishu` و`plugin-sdk/feishu-setup` و`plugin-sdk/zalo`,
و`plugin-sdk/zalo-setup`، وعدة وصلات `plugin-sdk/matrix*`. تعامل مع هذه
الصادرات على أنها صادرات تفاصيل تنفيذ محجوزة، لا كنمط SDK موصى به
لـ Plugins الخارجية الجديدة.

## مسار التحميل

عند بدء التشغيل، ينفّذ OpenClaw تقريبًا ما يلي:

1. يكتشف جذور Plugins المرشحة
2. يقرأ بيانات Plugins الأصلية أو بيانات الحِزم المتوافقة والبيانات الوصفية للحزم
3. يرفض المرشحين غير الآمنين
4. يطبّع إعدادات Plugin (`plugins.enabled` و`allow` و`deny` و`entries`،
   و`slots` و`load.paths`)
5. يقرر التمكين لكل مرشح
6. يحمّل الوحدات الأصلية المفعّلة عبر jiti
7. يستدعي خطافات `register(api)` الأصلية (أو `activate(api)` — اسمًا مستعارًا قديمًا) ويجمع التسجيلات في سجل Plugins
8. يعرّض السجل إلى أسطح الأوامر/وقت التشغيل

<Note>
`activate` هو اسم مستعار قديم لـ `register` — إذ يحلّ اللودر أيهما موجود (`def.register ?? def.activate`) ويستدعيه في النقطة نفسها. تستخدم جميع Plugins المضمّنة `register`؛ ويفضّل `register` للـ Plugins الجديدة.
</Note>

تحدث بوابات الأمان **قبل** تنفيذ وقت التشغيل. ويتم حظر المرشحين
عندما يخرج المدخل من جذر Plugin، أو يكون المسار قابلاً للكتابة من الجميع، أو
تبدو ملكية المسار مريبة بالنسبة إلى Plugins غير المضمّنة.

### سلوك manifest-first

يُعد manifest مصدر الحقيقة على مستوى control-plane. ويستخدمه OpenClaw من أجل:

- تحديد Plugin
- اكتشاف القنوات/Skills/مخطط الإعدادات المعلنة أو قدرات الحِزم
- التحقق من `plugins.entries.<id>.config`
- تعزيز تسميات/عناصر نائبة في واجهة التحكم
- إظهار بيانات التثبيت/الكتالوج
- الحفاظ على واصفات تفعيل وإعداد رخيصة من دون تحميل وقت تشغيل Plugin

وبالنسبة إلى Plugins الأصلية، تكون وحدة وقت التشغيل هي الجزء الخاص بـ data-plane. فهي تسجل
السلوك الفعلي مثل الخطافات، والأدوات، والأوامر، أو تدفقات الموفّر.

تبقى كتلتا `activation` و`setup` الاختياريتان في manifest على control-plane.
فهما واصفات بيانات فقط لتخطيط التفعيل واكتشاف الإعداد؛
ولا تحلان محل التسجيل في وقت التشغيل، أو `register(...)`، أو `setupEntry`.
ويستخدم أول مستهلكي التفعيل الحي الآن تلميحات الأوامر والقنوات والموفّرين في manifest
لتضييق تحميل Plugins قبل تجسيد السجل الأوسع:

- يضيَّق تحميل CLI إلى Plugins التي تملك الأمر الأساسي المطلوب
- يضيَّق حل إعداد/Plugin القناة إلى Plugins التي تملك
  معرّف القناة المطلوب
- يضيَّق حل إعداد/وقت تشغيل الموفّر الصريح إلى Plugins التي تملك
  معرّف الموفّر المطلوب

يفضّل اكتشاف الإعداد الآن المعرّفات المملوكة للواصفات مثل `setup.providers` و
`setup.cliBackends` لتضييق نطاق Plugins المرشحة قبل الرجوع إلى
`setup-api` بالنسبة إلى Plugins التي لا تزال تحتاج إلى خطافات وقت تشغيل أثناء الإعداد. وإذا ادعت أكثر من Plugin مكتشفة واحدة معرّف موفّر إعداد أو واجهة CLI خلفية
مطبعًا نفسه، فإن البحث عن الإعداد يرفض هذا المالك الملتبس بدلًا من الاعتماد على ترتيب الاكتشاف.

### ما الذي يخزّنه اللودر مؤقتًا

يحتفظ OpenClaw بذاكرات مؤقتة قصيرة داخل العملية من أجل:

- نتائج الاكتشاف
- بيانات سجل manifest
- سجلات Plugins المحمّلة

تقلّل هذه الذاكرات المؤقتة من كلفة بدء التشغيل المتكرر على دفعات ومن عبء
الأوامر المتكررة. ويمكن اعتبارها ذواكر مؤقتة قصيرة العمر للأداء، لا
وسيلة استمرارية.

ملاحظة أداء:

- اضبط `OPENCLAW_DISABLE_PLUGIN_DISCOVERY_CACHE=1` أو
  `OPENCLAW_DISABLE_PLUGIN_MANIFEST_CACHE=1` لتعطيل هذه الذاكرات المؤقتة.
- اضبط نوافذ الذاكرة المؤقتة عبر `OPENCLAW_PLUGIN_DISCOVERY_CACHE_MS` و
  `OPENCLAW_PLUGIN_MANIFEST_CACHE_MS`.

## نموذج السجل

لا تقوم Plugins المحمّلة بتعديل متغيرات القلب العشوائية مباشرةً. بل
تسجّل نفسها في سجل Plugins مركزي.

يتتبع السجل:

- سجلات Plugins ‏(الهوية، والمصدر، والمنشأ، والحالة، والتشخيصات)
- الأدوات
- الخطافات القديمة والخطافات مكتوبة النوع
- القنوات
- الموفّرين
- معالجات Gateway RPC
- مسارات HTTP
- مسجّلات CLI
- خدمات الخلفية
- الأوامر المملوكة للـ Plugin

ثم تقرأ ميزات القلب من هذا السجل بدل التحدث إلى وحدات Plugins مباشرةً.
وهذا يبقي التحميل أحادي الاتجاه:

- وحدة Plugin -> التسجيل في السجل
- وقت تشغيل القلب -> استهلاك السجل

ويهم هذا الفصل من أجل قابلية الصيانة. فهو يعني أن معظم أسطح القلب
تحتاج فقط إلى نقطة تكامل واحدة: "اقرأ السجل"، وليس "أنشئ حالة خاصة لكل وحدة Plugin".

## استدعاءات ربط المحادثة

يمكن للـ Plugins التي تربط محادثةً ما أن تتفاعل عند حسم الموافقة.

استخدم `api.onConversationBindingResolved(...)` لتلقي استدعاء بعد الموافقة
على طلب الربط أو رفضه:

```ts
export default {
  id: "my-plugin",
  register(api) {
    api.onConversationBindingResolved(async (event) => {
      if (event.status === "approved") {
        // A binding now exists for this plugin + conversation.
        console.log(event.binding?.conversationId);
        return;
      }

      // The request was denied; clear any local pending state.
      console.log(event.request.conversation.conversationId);
    });
  },
};
```

حقول حمولة الاستدعاء:

- `status`: ‏`"approved"` أو `"denied"`
- `decision`: ‏`"allow-once"` أو `"allow-always"` أو `"deny"`
- `binding`: الربط الذي تم حسمه للطلبات الموافق عليها
- `request`: ملخص الطلب الأصلي، وتلميح الفصل، ومعرّف المُرسل،
  وبيانات المحادثة الوصفية

هذا الاستدعاء مخصص للإشعار فقط. وهو لا يغيّر من يُسمح له بربط
محادثة، ويعمل بعد انتهاء معالجة الموافقة في القلب.

## خطافات وقت تشغيل الموفّر

أصبحت Plugins الموفّر الآن ذات طبقتين:

- بيانات manifest الوصفية: `providerAuthEnvVars` من أجل البحث الرخيص عن مصادقة الموفّر عبر المتغيرات البيئية
  قبل تحميل وقت التشغيل، و`providerAuthAliases` من أجل متغيرات الموفّر التي تشترك
  في المصادقة، و`channelEnvVars` من أجل البحث الرخيص عن إعداد/بيئة القناة قبل تحميل
  وقت التشغيل، بالإضافة إلى `providerAuthChoices` من أجل تسميات رخيصة للإعداد الأولي/اختيار المصادقة
  وبيانات وصفية لأعلام CLI قبل تحميل وقت التشغيل
- خطافات وقت الإعداد: `catalog` / و`discovery` القديمة بالإضافة إلى `applyConfigDefaults`
- خطافات وقت التشغيل: `normalizeModelId` و`normalizeTransport`,
  و`normalizeConfig`,
  و`applyNativeStreamingUsageCompat` و`resolveConfigApiKey`,
  و`resolveSyntheticAuth` و`resolveExternalAuthProfiles`,
  و`shouldDeferSyntheticProfileAuth`,
  و`resolveDynamicModel` و`prepareDynamicModel` و`normalizeResolvedModel`,
  و`contributeResolvedModelCompat` و`capabilities`,
  و`normalizeToolSchemas` و`inspectToolSchemas`,
  و`resolveReasoningOutputMode` و`prepareExtraParams` و`createStreamFn`,
  و`wrapStreamFn` و`resolveTransportTurnState`,
  و`resolveWebSocketSessionPolicy` و`formatApiKey` و`refreshOAuth`,
  و`buildAuthDoctorHint` و`matchesContextOverflowError`,
  و`classifyFailoverReason` و`isCacheTtlEligible`,
  و`buildMissingAuthMessage` و`suppressBuiltInModel` و`augmentModelCatalog`,
  و`isBinaryThinking` و`supportsXHighThinking` و`supportsAdaptiveThinking`,
  و`supportsMaxThinking`,
  و`resolveDefaultThinkingLevel` و`isModernModelRef` و`prepareRuntimeAuth`,
  و`resolveUsageAuth` و`fetchUsageSnapshot` و`createEmbeddingProvider`,
  و`buildReplayPolicy`,
  و`sanitizeReplayHistory` و`validateReplayTurns` و`onModelSelected`

لا يزال OpenClaw يملك حلقة الوكيل العامة، والرجوع الاحتياطي، ومعالجة النصوص،
وسياسة الأدوات. وتمثّل هذه الخطافات سطح التوسعة للسلوك الخاص بالموفّر من دون
الحاجة إلى نقل استدلال مخصص بالكامل.

استخدم manifest `providerAuthEnvVars` عندما يكون لدى الموفّر بيانات اعتماد قائمة على المتغيرات البيئية
ويجب أن تراها مسارات المصادقة/الحالة/منتقي النماذج العامة من دون تحميل وقت تشغيل Plugin. واستخدم manifest `providerAuthAliases` عندما يجب أن يعيد معرّف موفّر واحد استخدام
المتغيرات البيئية، وملفات تعريف المصادقة، والمصادقة المعتمدة على الإعدادات، وخيار الإعداد الأولي
المعتمد على مفتاح API الخاص بمعرّف موفّر آخر. واستخدم manifest `providerAuthChoices` عندما ينبغي لأسطح CLI الخاصة بالإعداد الأولي/اختيار المصادقة
أن تعرف معرّف اختيار الموفّر، وتسميات المجموعات، وتوصيل المصادقة البسيط بعلم واحد
من دون تحميل وقت تشغيل الموفّر. وأبقِ `envVars` في وقت تشغيل الموفّر للتلميحات المواجهة للمشغّل
مثل تسميات الإعداد الأولي أو متغيرات إعداد معرّف/سر عميل OAuth.

استخدم manifest `channelEnvVars` عندما تكون لدى قناة مصادقة أو إعداد مدفوع بالمتغيرات البيئية ويجب أن تراه
آليات الرجوع إلى بيئة shell العامة، أو فحوصات الإعداد/الحالة، أو مطالبات الإعداد
من دون تحميل وقت تشغيل القناة.

### ترتيب الخطافات واستخدامها

بالنسبة إلى Plugins النماذج/الموفّرين، يستدعي OpenClaw الخطافات تقريبًا بهذا الترتيب.
ويمثل عمود "متى يُستخدم" دليل القرار السريع.

| #   | الخطاف | ما الذي يفعله | متى يُستخدم |
| --- | --------------------------------- | -------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------- |
| 1   | `catalog` | ينشر إعدادات الموفّر داخل `models.providers` أثناء توليد `models.json` | عندما يملك الموفّر كتالوجًا أو إعدادات افتراضية لـ base URL |
| 2   | `applyConfigDefaults` | يطبّق الإعدادات الافتراضية العامة المملوكة للموفّر أثناء تجسيد الإعدادات | عندما تعتمد القيم الافتراضية على وضع المصادقة أو البيئة أو دلالات عائلة نماذج الموفّر |
| --  | _(البحث المضمّن عن النماذج)_ | يحاول OpenClaw أولًا المسار العادي للسجل/الكتالوج | _(ليس خطاف Plugin)_ |
| 3   | `normalizeModelId` | يطبّع الأسماء المستعارة القديمة أو الخاصة بالمعاينة لمعرّف النموذج قبل البحث | عندما يملك الموفّر تنظيف الأسماء المستعارة قبل تحليل النموذج القياسي |
| 4   | `normalizeTransport` | يطبّع `api` / `baseUrl` الخاصين بعائلة الموفّر قبل التجميع العام للنموذج | عندما يملك الموفّر تنظيف النقل لمعرّفات موفّرين مخصصة ضمن عائلة النقل نفسها |
| 5   | `normalizeConfig` | يطبّع `models.providers.<id>` قبل تحليل وقت التشغيل/الموفّر | عندما يحتاج الموفّر إلى تنظيف إعدادات يجب أن يعيش مع Plugin؛ كما أن مساعدات عائلة Google المضمّنة تدعم أيضًا إدخالات إعداد Google المدعومة |
| 6   | `applyNativeStreamingUsageCompat` | يطبّق عمليات إعادة كتابة توافق native streaming-usage على موفّري الإعدادات | عندما يحتاج الموفّر إلى إصلاحات بيانات وصفية خاصة بـ native streaming usage مدفوعة بنقطة النهاية |
| 7   | `resolveConfigApiKey` | يحل مصادقة env-marker لموفّري الإعدادات قبل تحميل مصادقة وقت التشغيل | عندما يكون لدى الموفّر تحليل مملوك لحل مفتاح API من نوع env-marker؛ كما يملك `amazon-bedrock` هنا أيضًا محللًا مضمّنًا لمؤشرات بيئة AWS |
| 8   | `resolveSyntheticAuth` | يعرّض مصادقة محلية/مستضافة ذاتيًا أو معتمدة على الإعدادات من دون الإبقاء على نص واضح | عندما يستطيع الموفّر العمل باستخدام علامة بيانات اعتماد synthetic/local |
| 9   | `resolveExternalAuthProfiles` | يراكب ملفات تعريف المصادقة الخارجية المملوكة للموفّر؛ وتكون القيمة الافتراضية لـ `persistence` هي `runtime-only` لبيانات الاعتماد المملوكة للـ CLI/التطبيق | عندما يعيد الموفّر استخدام بيانات اعتماد المصادقة الخارجية من دون حفظ رموز التحديث المنسوخة |
| 10  | `shouldDeferSyntheticProfileAuth` | يخفض أولوية العناصر النائبة لملفات التعريف synthetic المخزنة خلف المصادقة المدعومة بالبيئة/الإعدادات | عندما يخزّن الموفّر ملفات تعريف synthetic نائبة لا ينبغي أن تتقدّم في الأولوية |
| 11  | `resolveDynamicModel` | تزامن رجوع احتياطي لمعرّفات النماذج المملوكة للموفّر غير الموجودة بعد في السجل المحلي | عندما يقبل الموفّر معرّفات نماذج عشوائية من المصدر العلوي |
| 12  | `prepareDynamicModel` | تهيئة غير متزامنة، ثم يعمل `resolveDynamicModel` مرة أخرى | عندما يحتاج الموفّر إلى بيانات وصفية شبكية قبل تحليل المعرّفات غير المعروفة |
| 13  | `normalizeResolvedModel` | إعادة الكتابة النهائية قبل أن يستخدم embedded runner النموذج المحلل | عندما يحتاج الموفّر إلى إعادة كتابة على مستوى النقل لكنه لا يزال يستخدم نقلًا من القلب |
| 14  | `contributeResolvedModelCompat` | يساهم بعلامات توافق لنماذج المورّد الموجودة خلف نقل متوافق آخر | عندما يتعرف الموفّر على نماذجه الخاصة على عمليات نقل وكيلة من دون الاستيلاء على الموفّر |
| 15  | `capabilities` | بيانات وصفية خاصة بالنصوص/الأدوات يملكها الموفّر وتستخدمها منطقيات القلب المشتركة | عندما يحتاج الموفّر إلى خصائص خاصة بالنصوص أو بعائلة الموفّر |
| 16  | `normalizeToolSchemas` | يطبّع مخططات الأدوات قبل أن يراها embedded runner | عندما يحتاج الموفّر إلى تنظيف مخططات خاص بعائلة النقل |
| 17  | `inspectToolSchemas` | يعرّض تشخيصات مخططات مملوكة للموفّر بعد التطبيع | عندما يريد الموفّر تحذيرات كلمات مفتاحية من دون تعليم القلب قواعد خاصة بالموفّر |
| 18  | `resolveReasoningOutputMode` | يحدد عقد مخرجات الاستدلال: native أو tagged | عندما يحتاج الموفّر إلى reasoning/final output موسوم بدل الحقول الأصلية |
| 19  | `prepareExtraParams` | تطبيع معاملات الطلب قبل مغلفات خيارات البث العامة | عندما يحتاج الموفّر إلى معاملات طلب افتراضية أو تنظيف معاملات خاص بكل موفّر |
| 20  | `createStreamFn` | يستبدل مسار البث العادي بالكامل بنقل مخصص | عندما يحتاج الموفّر إلى بروتوكول نقل مخصص، لا مجرد غلاف |
| 21  | `wrapStreamFn` | غلاف للبث بعد تطبيق الأغلفة العامة | عندما يحتاج الموفّر إلى أغلفة توافق خاصة بالرؤوس/الجسم/النموذج من دون نقل مخصص |
| 22  | `resolveTransportTurnState` | يرفق رؤوسًا أو بيانات وصفية أصلية لكل دورة نقل | عندما يريد الموفّر أن ترسل عمليات النقل العامة هوية الدورة الأصلية الخاصة به |
| 23  | `resolveWebSocketSessionPolicy` | يرفق رؤوس WebSocket أصلية أو سياسة تهدئة للجلسة | عندما يريد الموفّر أن تضبط عمليات WS العامة رؤوس الجلسة أو سياسة الرجوع الاحتياطي |
| 24  | `formatApiKey` | مُنسّق ملف تعريف المصادقة: يتحول الملف المخزّن إلى سلسلة `apiKey` في وقت التشغيل | عندما يخزن الموفّر بيانات مصادقة وصفية إضافية ويحتاج إلى شكل رمز مخصص في وقت التشغيل |
| 25  | `refreshOAuth` | تجاوز تحديث OAuth لنقاط تحديث مخصصة أو سياسة فشل التحديث | عندما لا يناسب الموفّر آليات التحديث المشتركة لـ `pi-ai` |
| 26  | `buildAuthDoctorHint` | تلميح إصلاح يُلحق عند فشل تحديث OAuth | عندما يحتاج الموفّر إلى إرشادات إصلاح مصادقة يملكها هو بعد فشل التحديث |
| 27  | `matchesContextOverflowError` | مطابقة مملوكة للموفّر لأخطاء تجاوز نافذة السياق | عندما تكون لدى الموفّر أخطاء overflow خام قد تفوتها الإرشادات العامة |
| 28  | `classifyFailoverReason` | تصنيف مملوك للموفّر لأسباب الرجوع الاحتياطي | عندما يستطيع الموفّر ربط أخطاء API/النقل الخام بأسباب مثل rate-limit أو overload وغيرها |
| 29  | `isCacheTtlEligible` | سياسة prompt-cache لموفّري proxy/backhaul | عندما يحتاج الموفّر إلى بوابة TTL خاصة بالوكيل |
| 30  | `buildMissingAuthMessage` | بديل للرسالة العامة الخاصة باسترداد المصادقة المفقودة | عندما يحتاج الموفّر إلى تلميح خاص به لاسترداد المصادقة المفقودة |
| 31  | `suppressBuiltInModel` | إخفاء النماذج القديمة من المصدر العلوي مع تلميح خطأ اختياري موجّه للمستخدم | عندما يحتاج الموفّر إلى إخفاء صفوف قديمة من المصدر العلوي أو استبدالها بتلميح خاص بالمورّد |
| 32  | `augmentModelCatalog` | صفوف كتالوج synthetic/نهائية تُلحق بعد الاكتشاف | عندما يحتاج الموفّر إلى صفوف synthetic للتوافق المستقبلي في `models list` ومنتقيات النماذج |
| 33  | `isBinaryThinking` | تبديل reasoning تشغيل/إيقاف لموفّري binary-thinking | عندما يعرّض الموفّر فقط thinking ثنائيًا تشغيل/إيقاف |
| 34  | `supportsXHighThinking` | دعم reasoning من نوع `xhigh` لنماذج محددة | عندما يريد الموفّر إظهار `xhigh` فقط لمجموعة فرعية من النماذج |
| 35  | `supportsAdaptiveThinking` | دعم thinking من نوع `adaptive` لنماذج محددة | عندما يريد الموفّر إظهار `adaptive` فقط للنماذج ذات thinking التكيفي المُدار من الموفّر |
| 36  | `supportsMaxThinking` | دعم reasoning من نوع `max` لنماذج محددة | عندما يريد الموفّر إظهار `max` فقط للنماذج التي تدعم max thinking لديه |
| 37  | `resolveDefaultThinkingLevel` | مستوى `/think` الافتراضي لعائلة نماذج محددة | عندما يملك الموفّر سياسة `/think` الافتراضية لعائلة نماذج محددة |
| 38  | `isModernModelRef` | مطابق modern-model لمرشحات الملفات الحية واختيار smoke | عندما يملك الموفّر مطابقة النموذج المفضّل للاختبارات الحية/‏smoke |
| 39  | `prepareRuntimeAuth` | يبادل بيانات اعتماد مهيأة إلى الرمز/المفتاح الفعلي في وقت التشغيل مباشرةً قبل الاستدلال | عندما يحتاج الموفّر إلى تبادل رمز أو بيانات اعتماد طلب قصيرة العمر |
| 40  | `resolveUsageAuth` | يحل بيانات اعتماد الاستخدام/الفوترة لأجل `/usage` والأسطح ذات الصلة بالحالة | عندما يحتاج الموفّر إلى تحليل مخصص لرمز الاستخدام/الحصة أو إلى بيانات اعتماد استخدام مختلفة |
| 41  | `fetchUsageSnapshot` | يجلب ويطبّع لقطات الاستخدام/الحصة الخاصة بالموفّر بعد حل المصادقة | عندما يحتاج الموفّر إلى نقطة استخدام أو محلل حمولة خاص بالموفّر |
| 42  | `createEmbeddingProvider` | يبني مكيّف تضمين مملوكًا للموفّر من أجل الذاكرة/البحث | عندما يكون سلوك تضمين الذاكرة تابعًا لـ Plugin الموفّر |
| 43  | `buildReplayPolicy` | يعيد سياسة replay تتحكم في معالجة النصوص الخاصة بالموفّر | عندما يحتاج الموفّر إلى سياسة نصوص مخصصة (مثل إزالة كتل thinking) |
| 44  | `sanitizeReplayHistory` | يعيد كتابة سجل replay بعد تنظيف النصوص العام | عندما يحتاج الموفّر إلى إعادة كتابة خاصة به لسجل replay تتجاوز مساعدات Compaction المشتركة |
| 45  | `validateReplayTurns` | التحقق النهائي من جولات replay أو إعادة تشكيلها قبل embedded runner | عندما يحتاج نقل الموفّر إلى تحقق أشد من الجولات بعد التنظيف العام |
| 46  | `onModelSelected` | يشغّل آثارًا جانبية بعد الاختيار يملكها الموفّر | عندما يحتاج الموفّر إلى telemetry أو حالة مملوكة له عند تنشيط نموذج ما |

يتحقق `normalizeModelId` و`normalizeTransport` و`normalizeConfig` أولًا من
Plugin الموفّر المطابقة، ثم ينتقل عبر Plugins الموفّرين الأخرى القادرة على استخدام الخطافات
حتى تقوم إحداها فعلًا بتغيير معرّف النموذج أو النقل/الإعدادات. وهذا يبقي
طبقات alias/compat الخاصة بموفّرين بديلين تعمل من دون الحاجة إلى أن يعرف
المستدعي أي Plugin مضمّنة تملك إعادة الكتابة. وإذا لم يُعد أي خطاف موفّر
كتابة إدخال إعدادات مدعوم من عائلة Google، فإن مُطبّع إعدادات Google المضمّن
يطبّق ذلك التنظيف التوافقي أيضًا.

إذا كان الموفّر يحتاج إلى بروتوكول نقل مخصص بالكامل أو منفّذ طلبات مخصص،
فهذه فئة مختلفة من التوسعة. فهذه الخطافات مخصّصة لسلوك الموفّر الذي لا يزال
يعمل على حلقة الاستدلال العادية في OpenClaw.

### مثال على موفّر

```ts
api.registerProvider({
  id: "example-proxy",
  label: "Example Proxy",
  auth: [],
  catalog: {
    order: "simple",
    run: async (ctx) => {
      const apiKey = ctx.resolveProviderApiKey("example-proxy").apiKey;
      if (!apiKey) {
        return null;
      }
      return {
        provider: {
          baseUrl: "https://proxy.example.com/v1",
          apiKey,
          api: "openai-completions",
          models: [{ id: "auto", name: "Auto" }],
        },
      };
    },
  },
  resolveDynamicModel: (ctx) => ({
    id: ctx.modelId,
    name: ctx.modelId,
    provider: "example-proxy",
    api: "openai-completions",
    baseUrl: "https://proxy.example.com/v1",
    reasoning: false,
    input: ["text"],
    cost: { input: 0, output: 0, cacheRead: 0, cacheWrite: 0 },
    contextWindow: 128000,
    maxTokens: 8192,
  }),
  prepareRuntimeAuth: async (ctx) => {
    const exchanged = await exchangeToken(ctx.apiKey);
    return {
      apiKey: exchanged.token,
      baseUrl: exchanged.baseUrl,
      expiresAt: exchanged.expiresAt,
    };
  },
  resolveUsageAuth: async (ctx) => {
    const auth = await ctx.resolveOAuthToken();
    return auth ? { token: auth.token } : null;
  },
  fetchUsageSnapshot: async (ctx) => {
    return await fetchExampleProxyUsage(ctx.token, ctx.timeoutMs, ctx.fetchFn);
  },
});
```

### أمثلة مضمّنة

- تستخدم Anthropic كلًا من `resolveDynamicModel` و`capabilities` و`buildAuthDoctorHint`,
  و`resolveUsageAuth` و`fetchUsageSnapshot` و`isCacheTtlEligible`,
  و`supportsAdaptiveThinking` و`supportsMaxThinking` و`resolveDefaultThinkingLevel` و`applyConfigDefaults` و`isModernModelRef`,
  و`wrapStreamFn` لأنها تملك توافق Claude 4.6 المستقبلي،
  وتلميحات عائلة الموفّر، وإرشادات إصلاح المصادقة، وتكامل نقطة الاستخدام،
  وأهلية prompt-cache، والإعدادات الافتراضية الواعية بالمصادقة، وسياسة
  Claude الافتراضية/التكيفية للتفكير، وتشكيل البث الخاص بـ Anthropic من أجل
  رؤوس beta، و`/fast` / `serviceTier`، و`context1m`.
- تبقى مساعدات البث الخاصة بـ Claude في Anthropic في الوصلة العامة
  `api.ts` / `contract-api.ts` الخاصة بـ Plugin المضمّنة نفسها في الوقت الحالي. ويصدّر سطح
  هذه الحزمة `wrapAnthropicProviderStream` و`resolveAnthropicBetas`,
  و`resolveAnthropicFastMode` و`resolveAnthropicServiceTier` والمُنشئات
  الأدنى مستوى لأغلفة Anthropic بدل توسيع SDK العامة حول
  قواعد رؤوس beta الخاصة بموفّر واحد.
- تستخدم OpenAI كلًا من `resolveDynamicModel` و`normalizeResolvedModel` و
  `capabilities` بالإضافة إلى `buildMissingAuthMessage` و`suppressBuiltInModel`,
  و`augmentModelCatalog` و`supportsXHighThinking` و`isModernModelRef`
  لأنها تملك توافق GPT-5.4 المستقبلي، وتطبيع OpenAI المباشر
  من `openai-completions` إلى `openai-responses`، وتلميحات المصادقة الواعية بـ Codex،
  وكبت Spark، وصفوف OpenAI synthetic، وسياسة التفكير /
  النموذج الحي لـ GPT-5؛ كما تملك عائلة البث `openai-responses-defaults`
  أغلفة OpenAI Responses الأصلية المشتركة لرؤوس الإسناد،
  و`/fast`/`serviceTier`، ودرجة تفصيل النص، والبحث الأصلي على الويب في Codex،
  وتشكيل حمولة توافق reasoning، وإدارة سياق Responses.
- تستخدم OpenRouter `catalog` بالإضافة إلى `resolveDynamicModel` و
  `prepareDynamicModel` لأن الموفّر يمرر الطلبات وقد يعرّض
  معرّفات نماذج جديدة قبل أن يتحدث كتالوج OpenClaw الثابت؛ كما تستخدم
  `capabilities` و`wrapStreamFn` و`isCacheTtlEligible` لإبقاء
  رؤوس الطلبات الخاصة بالموفّر، وبيانات التوجيه، ورُقَع reasoning، وسياسة
  prompt-cache خارج القلب. وتأتي سياسة replay الخاصة بها من
  عائلة `passthrough-gemini`، بينما تملك عائلة البث `openrouter-thinking`
  حقن reasoning الوكيل وتخطّي `auto` / والنماذج غير المدعومة.
- تستخدم GitHub Copilot `catalog` و`auth` و`resolveDynamicModel` و
  `capabilities` بالإضافة إلى `prepareRuntimeAuth` و`fetchUsageSnapshot` لأنها
  تحتاج إلى تسجيل دخول الجهاز المملوك للموفّر، وسلوك الرجوع الاحتياطي للنموذج، وخصائص
  نص Claude، وتبادل رمز GitHub إلى رمز Copilot، ونقطة استخدام
  مملوكة للموفّر.
- تستخدم OpenAI Codex `catalog` و`resolveDynamicModel`,
  و`normalizeResolvedModel` و`refreshOAuth` و`augmentModelCatalog` بالإضافة

## مساعدات وقت التشغيل

يمكن للـ Plugins الوصول إلى مجموعة محددة من مساعدات القلب عبر `api.runtime`. بالنسبة إلى TTS:

```ts
const clip = await api.runtime.tts.textToSpeech({
  text: "Hello from OpenClaw",
  cfg: api.config,
});

const result = await api.runtime.tts.textToSpeechTelephony({
  text: "Hello from OpenClaw",
  cfg: api.config,
});

const voices = await api.runtime.tts.listVoices({
  provider: "elevenlabs",
  cfg: api.config,
});
```

ملاحظات:

- يعيد `textToSpeech` حمولة خرج TTS العادية في القلب لأسطح الملفات/الملاحظات الصوتية.
- يستخدم إعداد `messages.tts` في القلب واختيار الموفّر.
- يعيد مخزنًا مؤقتًا لصوت PCM + معدل العينة. وعلى Plugins إعادة أخذ العينات/الترميز للمزوّدين.
- يكون `listVoices` اختياريًا لكل موفّر. استخدمه لمنتقيات الأصوات المملوكة للمورّد أو لتدفقات الإعداد.
- يمكن أن تتضمن قوائم الأصوات بيانات وصفية أغنى مثل اللغة والمنطقة، والجنس، وعلامات الشخصية لمنتقيات واعية بالموفّر.
- يدعم OpenAI وElevenLabs الهاتف اليوم. أما Microsoft فلا يدعم ذلك.

يمكن للـ Plugins أيضًا تسجيل موفّري الكلام عبر `api.registerSpeechProvider(...)`.

```ts
api.registerSpeechProvider({
  id: "acme-speech",
  label: "Acme Speech",
  isConfigured: ({ config }) => Boolean(config.messages?.tts),
  synthesize: async (req) => {
    return {
      audioBuffer: Buffer.from([]),
      outputFormat: "mp3",
      fileExtension: ".mp3",
      voiceCompatible: false,
    };
  },
});
```

ملاحظات:

- أبقِ سياسة TTS، والرجوع الاحتياطي، وتسليم الرد في القلب.
- استخدم موفّري الكلام لسلوك synthesis المملوك للمورّد.
- يُطبّع الإدخال القديم `edge` من Microsoft إلى معرّف الموفّر `microsoft`.
- نموذج الملكية المفضّل موجّه إلى الشركة: إذ يمكن لـ Plugin مورّد واحدة أن تملك
  النص، والكلام، والصورة، وموفّري الوسائط مستقبلًا كلما أضاف OpenClaw
  عقود تلك القدرات.

بالنسبة إلى فهم الصور/الصوت/الفيديو، تسجّل Plugins موفّرًا واحدًا مكتوب النوع
لفهم الوسائط بدل حقيبة مفاتيح/قيم عامة:

```ts
api.registerMediaUnderstandingProvider({
  id: "google",
  capabilities: ["image", "audio", "video"],
  describeImage: async (req) => ({ text: "..." }),
  transcribeAudio: async (req) => ({ text: "..." }),
  describeVideo: async (req) => ({ text: "..." }),
});
```

ملاحظات:

- أبقِ التنسيق، والرجوع الاحتياطي، والإعدادات، وتوصيل القنوات في القلب.
- أبقِ سلوك المورّد في Plugin الموفّر.
- يجب أن يبقى التوسّع الإضافي مكتوب النوع: أساليب اختيارية جديدة، وحقول نتائج
  اختيارية جديدة، وقدرات اختيارية جديدة.
- يتبع توليد الفيديو بالفعل النمط نفسه:
  - يملك القلب عقد القدرة ومساعد وقت التشغيل
  - تسجّل Plugins المورّدين `api.registerVideoGenerationProvider(...)`
  - تستهلك Plugins الميزات/القنوات `api.runtime.videoGeneration.*`

بالنسبة إلى مساعدات وقت التشغيل الخاصة بفهم الوسائط، يمكن للـ Plugins استدعاء:

```ts
const image = await api.runtime.mediaUnderstanding.describeImageFile({
  filePath: "/tmp/inbound-photo.jpg",
  cfg: api.config,
  agentDir: "/tmp/agent",
});

const video = await api.runtime.mediaUnderstanding.describeVideoFile({
  filePath: "/tmp/inbound-video.mp4",
  cfg: api.config,
});
```

وبالنسبة إلى نسخ الصوت، يمكن للـ Plugins استخدام وقت تشغيل فهم الوسائط أو
الاسم المستعار الأقدم STT:

```ts
const { text } = await api.runtime.mediaUnderstanding.transcribeAudioFile({
  filePath: "/tmp/inbound-audio.ogg",
  cfg: api.config,
  // Optional when MIME cannot be inferred reliably:
  mime: "audio/ogg",
});
```

ملاحظات:

- `api.runtime.mediaUnderstanding.*` هو السطح المشترك المفضل
  لفهم الصور/الصوت/الفيديو.
- يستخدم إعدادات الصوت الخاصة بفهم الوسائط في القلب (`tools.media.audio`) وترتيب الرجوع الاحتياطي للمزوّد.
- يعيد `{ text: undefined }` عندما لا يتم إنتاج خرج نسخ (على سبيل المثال عند تخطي إدخال غير مدعوم).
- يبقى `api.runtime.stt.transcribeAudioFile(...)` اسمًا مستعارًا للتوافق.

يمكن للـ Plugins أيضًا إطلاق عمليات وكيل فرعي في الخلفية عبر `api.runtime.subagent`:

```ts
const result = await api.runtime.subagent.run({
  sessionKey: "agent:main:subagent:search-helper",
  message: "Expand this query into focused follow-up searches.",
  provider: "openai",
  model: "gpt-4.1-mini",
  deliver: false,
});
```

ملاحظات:

- `provider` و`model` تجاوزات اختيارية لكل تشغيل، وليست تغييرات دائمة على الجلسة.
- لا يحترم OpenClaw حقول التجاوز هذه إلا للمستدعين الموثوقين.
- بالنسبة إلى عمليات الرجوع الاحتياطي المملوكة للـ Plugin، يجب على المشغلين الاشتراك صراحةً عبر `plugins.entries.<id>.subagent.allowModelOverride: true`.
- استخدم `plugins.entries.<id>.subagent.allowedModels` لتقييد Plugins الموثوقة بأهداف `provider/model` قياسية محددة، أو `"*"` للسماح صراحةً بأي هدف.
- لا تزال عمليات الوكيل الفرعي غير الموثوقة الخاصة بالـ Plugin تعمل، لكن يتم رفض طلبات التجاوز بدل الرجوع بصمت.

بالنسبة إلى البحث على الويب، يمكن للـ Plugins استهلاك مساعد وقت التشغيل المشترك بدل
الوصول إلى توصيل أداة الوكيل:

```ts
const providers = api.runtime.webSearch.listProviders({
  config: api.config,
});

const result = await api.runtime.webSearch.search({
  config: api.config,
  args: {
    query: "OpenClaw plugin runtime helpers",
    count: 5,
  },
});
```

يمكن للـ Plugins أيضًا تسجيل موفّري البحث على الويب عبر
`api.registerWebSearchProvider(...)`.

ملاحظات:

- أبقِ اختيار الموفّر، وتحليل بيانات الاعتماد، ودلالات الطلب المشتركة في القلب.
- استخدم موفّري البحث على الويب لعمليات النقل الخاصة بالمورّد.
- `api.runtime.webSearch.*` هو السطح المشترك المفضّل لـ Plugins الميزات/القنوات التي تحتاج إلى سلوك البحث من دون الاعتماد على غلاف أداة الوكيل.

### `api.runtime.imageGeneration`

```ts
const result = await api.runtime.imageGeneration.generate({

  args: { prompt: "A friendly lobster mascot", size: "1024x1024" },

const providers = api.runtime.imageGeneration.listProviders({
```

## مسارات Gateway HTTP

يمكن للـ Plugins كشف نقاط نهاية HTTP عبر `api.registerHttpRoute(...)`.

```ts
api.registerHttpRoute({
  path: "/acme/webhook",
  auth: "plugin",
  match: "exact",
  handler: async (_req, res) => {
    res.statusCode = 200;
    res.end("ok");
    return true;
  },
});
```

حقول المسار:

- `path`: مسار الطريق تحت خادم HTTP الخاص بـ Gateway.
- `auth`: مطلوب. استخدم `"gateway"` لطلب مصادقة Gateway العادية، أو `"plugin"` للمصادقة/التحقق من Webhook الذي تديره Plugin.
- `match`: اختياري. `"exact"` (الافتراضي) أو `"prefix"`.
- `replaceExisting`: اختياري. يسمح للـ Plugin نفسها باستبدال تسجيل مسار موجود خاص بها.
- `handler`: أعد `true` عندما يكون المسار قد عالج الطلب.

ملاحظات:

- تمت إزالة `api.registerHttpHandler(...)` وسيتسبب في خطأ تحميل Plugin. استخدم `api.registerHttpRoute(...)` بدلًا منه.
- يجب أن تعلن مسارات Plugins عن `auth` صراحةً.
- يتم رفض تعارضات `path + match` الدقيقة ما لم يكن `replaceExisting: true`، ولا يمكن لـ Plugin واحدة استبدال مسار Plugin أخرى.
- يتم رفض المسارات المتداخلة ذات مستويات `auth` المختلفة. أبقِ سلاسل fallthrough من نوع `exact`/`prefix` على مستوى المصادقة نفسه فقط.
- لا تتلقى مسارات `auth: "plugin"` نطاقات وقت تشغيل المشغّل تلقائيًا. فهي مخصّصة لـ Webhooks/التحقق من التواقيع الذي تديره Plugin، وليست لاستدعاءات مساعدات Gateway ذات الامتيازات.
- تعمل مسارات `auth: "gateway"` داخل نطاق وقت تشغيل طلب Gateway، لكن هذا النطاق متحفّظ عمدًا:
  - تُبقي مصادقة bearer ذات السر المشترك (`gateway.auth.mode = "token"` / `"password"`) نطاقات وقت تشغيل مسارات Plugins مثبتة على `operator.write`، حتى إذا أرسل المستدعي `x-openclaw-scopes`
  - تحترم أوضاع HTTP الموثوقة الحاملة للهوية (مثل `trusted-proxy` أو `gateway.auth.mode = "none"` على مدخل خاص) القيمة `x-openclaw-scopes` فقط عندما يكون هذا الرأس موجودًا صراحةً
  - إذا كان `x-openclaw-scopes` غائبًا في طلبات مسارات Plugins الحاملة للهوية، يعود نطاق وقت التشغيل إلى `operator.write`
- قاعدة عملية: لا تفترض أن مسار Plugin المصادق عبر gateway هو سطح إدارة ضمني. إذا كان مسارك يحتاج إلى سلوك خاص بالمشرف فقط، فاطلب وضع مصادقة حاملًا للهوية ووثّق عقد الرأس الصريح `x-openclaw-scopes`.

  و`openclaw/plugin-sdk/runtime-store`، و
  `openclaw/plugin-sdk/directory-runtime` من أجل مساعدات وقت التشغيل/الإعدادات المشتركة.
  ويُعد `telegram-command-config` الوصلة العامة الضيقة لتطبيع/التحقق من الأوامر
  المخصصة في Telegram، ويظل متاحًا حتى إذا كان سطح عقد Telegram المضمّن
  غير متاح مؤقتًا.
  و`text-runtime` هو وصلة النص/Markdown/التسجيل المشتركة، بما في ذلك
  إزالة النص الظاهر للمساعد، ومساعدات عرض/تقطيع Markdown، ومساعدات التنقيح،
  ومساعدات directive-tag، وأدوات النص الآمن.
- ينبغي للوصلات الخاصة بالقنوات المتعلقة بالموافقات أن تفضّل عقدًا واحدًا `approvalCapability`
  على Plugin. ثم يقرأ القلب مصادقة الموافقات، والتسليم، والعرض،
  والتوجيه الأصلي، وسلوك المعالج الأصلي الكسول عبر هذه القدرة الواحدة
  بدل خلط سلوك الموافقات في حقول Plugin غير ذات الصلة.
- `openclaw/plugin-sdk/channel-runtime` مهجور ويبقى فقط كطبقة توافق
  للـ Plugins الأقدم. ينبغي للكود الجديد استيراد البدائيات العامة الأضيق بدلًا منه، ولا ينبغي لكود المستودع إضافة واردات جديدة من هذه الطبقة.
- تظل الآليات الداخلية للإضافات المضمّنة خاصة. يجب على Plugins الخارجية استخدام مسارات `openclaw/plugin-sdk/*` الفرعية فقط. ويمكن لكود القلب/الاختبار في OpenClaw استخدام
  نقاط الدخول العامة في المستودع تحت جذر حزمة Plugin مثل `index.js` و`api.js`,
  و`runtime-api.js` و`setup-entry.js`، والملفات الضيقة النطاق مثل
  `login-qr-api.js`. ولا تستورد أبدًا `src/*` الخاصة بحزمة Plugin من القلب أو من
  إضافة أخرى.
- تقسيم نقطة الدخول في المستودع:
  `<plugin-package-root>/api.js` هو برميل المساعدات/الأنواع،
  و`<plugin-package-root>/runtime-api.js` هو برميل وقت التشغيل فقط،
  و`<plugin-package-root>/index.js` هو مدخل Plugin المضمّنة،
  و`<plugin-package-root>/setup-entry.js` هو مدخل Plugin الخاصة بالإعداد.
- أمثلة موفّرين مضمّنين حاليًا:
  - تستخدم Anthropic `api.js` / `contract-api.js` لمساعدات بث Claude مثل
    `wrapAnthropicProviderStream`، ومساعدات رؤوس beta، وتحليل `service_tier`.
  - تستخدم OpenAI `api.js` لبناة الموفّرين، ومساعدات النماذج الافتراضية، وبناة الموفّرين الفوريين.
  - تستخدم OpenRouter `api.js` لباني الموفّر الخاص بها بالإضافة إلى مساعدات
    الإعداد/التهيئة الأولية، بينما يمكن لـ `register.runtime.js` الاستمرار في إعادة تصدير مساعدات
    `plugin-sdk/provider-stream` العامة للاستخدام المحلي في المستودع.
- تفضّل نقاط الدخول العامة المحمّلة عبر facade اللقطة الفعالة الحالية لإعدادات وقت التشغيل عندما تكون موجودة، ثم تعود إلى ملف الإعدادات المحلل على القرص عندما لا يكون OpenClaw
  قد بدأ بعد في تقديم لقطة وقت تشغيل.
- تظل البدائيات العامة المشتركة هي عقد SDK العامة المفضلة. ولا تزال توجد مجموعة توافق صغيرة محجوزة من وصلات المساعدة ذات العلامات الخاصة بالقنوات المضمّنة. تعامل مع هذه الوصلات على أنها وصلات صيانة/توافق للـ Plugins المضمّنة، لا أهداف استيراد جديدة لجهات خارجية؛ ويجب أن تظل العقود الجديدة العابرة للقنوات تهبط على المسارات العامة `plugin-sdk/*` أو براميل `api.js` /
  `runtime-api.js` المحلية الخاصة بالـ Plugin.

ملاحظة توافق:

- تجنّب البرميل الجذري `openclaw/plugin-sdk` في الكود الجديد.
- فضّل أولًا البدائيات المستقرة الضيقة. فالمسارات الفرعية الأحدث الخاصة بالإعداد/الاقتران/الرد/
  والتغذية الراجعة/العقد/الوارد/الترابط/الأوامر/إدخال الأسرار/Webhook/البنية التحتية/
  وقائمة السماح/الحالة/أداة الرسائل هي العقد المقصود للأعمال الجديدة
  للـ Plugins المضمّنة والخارجية.
  يجب أن يكون تحليل/مطابقة الهدف على `openclaw/plugin-sdk/channel-targets`.
  كما يجب أن تكون بوابات إجراءات الرسائل ومساعدات معرّف رسالة التفاعل على
  `openclaw/plugin-sdk/channel-actions`.
- لا تُعد براميل المساعدة الخاصة بالإضافة المضمّنة مستقرة افتراضيًا. إذا كانت
  هناك مساعدة مطلوبة فقط من إضافة مضمّنة، فأبقها خلف
  الوصلة المحلية `api.js` أو `runtime-api.js` الخاصة بالإضافة بدل ترقيتها إلى
  `openclaw/plugin-sdk/<extension>`.
- يجب أن تكون وصلات المساعدة المشتركة الجديدة عامة، لا موسومة باسم قناة. وينتمي
  تحليل الأهداف المشترك إلى `openclaw/plugin-sdk/channel-targets`؛ أما
  الآليات الداخلية الخاصة بالقناة فتبقى خلف وصلة `api.js` أو `runtime-api.js`
  المحلية الخاصة بالـ Plugin المالكة.
- توجد مسارات فرعية خاصة بالقدرات مثل `image-generation`,
  و`media-understanding`، و`speech` لأن Plugins المضمّنة/الأصلية تستخدمها
  اليوم. لكن وجودها لا يعني بحد ذاته أن كل مساعدة مُصدّرة هي عقد خارجي
  مجمّد على المدى الطويل.

## مخططات أداة الرسائل

يجب أن تملك Plugins المساهمات الخاصة بالقناة في
مخطط `describeMessageTool(...)`. أبقِ الحقول الخاصة بالمزوّد في Plugin، لا في القلب المشترك.

وبالنسبة إلى أجزاء المخطط المحمولة المشتركة، أعد استخدام المساعدات العامة
المُصدّرة عبر `openclaw/plugin-sdk/channel-actions`:

- `createMessageToolButtonsSchema()` لحمولات نمط شبكة الأزرار
- `createMessageToolCardSchema()` لحمولات البطاقات المهيكلة

إذا كان شكل مخطط ما لا يناسب إلا مزودًا واحدًا، فعرّفه في
مصدر Plugin نفسه بدل ترقيته إلى SDK المشتركة.

## تحليل هدف القناة

يجب أن تملك Plugins القنوات دلالات الأهداف الخاصة بالقناة. أبقِ مضيف
الصادر المشترك عامًا واستخدم سطح مكيّف المراسلة لقواعد المزوّد:

- يقرر `messaging.inferTargetChatType({ to })` ما إذا كان الهدف المطبع
  يجب أن يُعامل كـ `direct` أو `group` أو `channel` قبل البحث في الدليل.
- يخبر `messaging.targetResolver.looksLikeId(raw, normalized)` القلب عما إذا كان
  ينبغي للإدخال أن يتجاوز مباشرةً إلى تحليل يشبه المعرّف بدل البحث في الدليل.
- يمثّل `messaging.targetResolver.resolveTarget(...)` الرجوع الاحتياطي للـ Plugin عندما
  يحتاج القلب إلى تحليل نهائي مملوك للمزوّد بعد التطبيع أو بعد
  فشل في الدليل.
- يملك `messaging.resolveOutboundSessionRoute(...)` إنشاء
  مسار الجلسة الصادر الخاص بالمزوّد بعد تحليل الهدف.

التقسيم الموصى به:

- استخدم `inferTargetChatType` لقرارات الفئة التي يجب أن تحدث قبل
  البحث في الأقران/المجموعات.
- استخدم `looksLikeId` لفحوصات "عامل هذا كمعرّف هدف صريح/أصلي".
- استخدم `resolveTarget` للرجوع الاحتياطي في التطبيع الخاص بالمزوّد، وليس
  للبحث الواسع في الدليل.
- أبقِ المعرّفات الأصلية للمزوّد مثل chat ids وthread ids وJIDs وhandles وroom
  ids داخل قيم `target` أو المعاملات الخاصة بالمزوّد، وليس في حقول SDK عامة.

## أدلة مدعومة بالإعدادات

يجب أن تُبقي Plugins التي تستمد إدخالات الدليل من الإعدادات هذا المنطق داخل
الـ Plugin وأن تعيد استخدام المساعدات المشتركة من
`openclaw/plugin-sdk/directory-runtime`.

استخدم هذا عندما تحتاج قناة إلى أقران/مجموعات مدعومة بالإعدادات مثل:

- أقران الرسائل المباشرة المدفوعين بقائمة السماح
- خرائط القنوات/المجموعات المهيأة
- بدائل أدلة ثابتة ضمن نطاق الحساب

تتعامل المساعدات المشتركة في `directory-runtime` فقط مع العمليات العامة:

- تصفية الاستعلام
- تطبيق الحدود
- مساعدات إزالة التكرار/التطبيع
- بناء `ChannelDirectoryEntry[]`

أما فحص الحساب الخاص بالقناة وتطبيع المعرّفات فيجب أن يبقيا داخل
تنفيذ Plugin.

## كتالوجات الموفّرين

يمكن لـ Plugins الموفّرين تعريف كتالوجات نماذج للاستدلال باستخدام
`registerProvider({ catalog: { run(...) { ... } } })`.

يعيد `catalog.run(...)` الشكل نفسه الذي يكتبه OpenClaw داخل
`models.providers`:

- `{ provider }` لإدخال موفّر واحد
- `{ providers }` لإدخالات موفّرين متعددة

استخدم `catalog` عندما تملك Plugin معرّفات نماذج خاصة بالمزوّد، أو إعدادات
افتراضية لـ base URL، أو بيانات وصفية للنماذج محكومة بالمصادقة.

يتحكم `catalog.order` في توقيت دمج كتالوج Plugin نسبةً إلى
الموفّرين الضمنيين المضمّنين في OpenClaw:

- `simple`: موفّرون عاديون يعتمدون على API-key أو المتغيرات البيئية
- `profile`: موفّرون يظهرون عند وجود ملفات تعريف مصادقة
- `paired`: موفّرون يركّبون إدخالات موفّرين متعددة ذات صلة
- `late`: المرور الأخير، بعد الموفّرين الضمنيين الآخرين

يفوز الموفّرون المتأخرون عند تعارض المفاتيح، لذا يمكن للـ Plugins أن
تتجاوز عمدًا إدخال موفّر مضمّنًا يملك معرّف الموفّر نفسه.

التوافق:

- لا يزال `discovery` يعمل كاسم مستعار قديم
- إذا تم تسجيل كل من `catalog` و`discovery`، يستخدم OpenClaw `catalog`

## فحص القنوات للقراءة فقط

إذا كانت Plugin الخاصة بك تسجّل قناة، ففضّل تنفيذ
`plugin.config.inspectAccount(cfg, accountId)` إلى جانب `resolveAccount(...)`.

السبب:

- يمثّل `resolveAccount(...)` مسار وقت التشغيل. ويُسمح له بافتراض أن بيانات الاعتماد
  قد جُسّدت بالكامل وأن يفشل سريعًا عند غياب الأسرار المطلوبة.
- يجب ألا تحتاج مسارات الأوامر للقراءة فقط مثل `openclaw status` و`openclaw status --all`,
  و`openclaw channels status` و`openclaw channels resolve`، وتدفقات doctor/إصلاح الإعدادات
  إلى تجسيد بيانات اعتماد وقت التشغيل لمجرد
  وصف الإعدادات.

السلوك الموصى به لـ `inspectAccount(...)`:

- أعد فقط حالة الحساب الوصفية.
- حافظ على `enabled` و`configured`.
- ضمّن حقول مصدر/حالة بيانات الاعتماد عند الحاجة، مثل:
  - `tokenSource`, `tokenStatus`
  - `botTokenSource`, `botTokenStatus`
  - `appTokenSource`, `appTokenStatus`
  - `signingSecretSource`, `signingSecretStatus`
- لست بحاجة إلى إرجاع قيم الرموز الخام فقط من أجل الإبلاغ عن التوفر
  للقراءة فقط. يكفي إرجاع `tokenStatus: "available"` (ومعه حقل المصدر
  المطابق) لأوامر نمط الحالة.
- استخدم `configured_unavailable` عندما تكون بيانات الاعتماد مهيأة عبر SecretRef لكن
  غير متاحة في مسار الأوامر الحالي.

يسمح هذا لأوامر القراءة فقط بالإبلاغ عن "مهيأ لكنه غير متاح في مسار هذا الأمر"
بدل التعطل أو الإبلاغ الخاطئ عن الحساب على أنه غير مهيأ.

## حِزم Package

قد يتضمن دليل Plugin ملف `package.json` يحتوي على `openclaw.extensions`:

```json
{
  "name": "my-pack",
  "openclaw": {
    "extensions": ["./src/safety.ts", "./src/tools.ts"],
    "setupEntry": "./src/setup-entry.ts"
  }
}
```

يصبح كل إدخال Plugin. وإذا كانت الحزمة تسرد إضافات متعددة، يصبح معرّف Plugin
هو `name/<fileBase>`.

إذا كانت Plugin تستورد تبعيات npm، فقم بتثبيتها في ذلك الدليل بحيث
يتوفر `node_modules` (`npm install` / `pnpm install`).

حاجز أمان: يجب أن يبقى كل إدخال في `openclaw.extensions` داخل دليل Plugin
بعد تحليل الروابط الرمزية. وترفَض الإدخالات التي تهرب خارج دليل الحزمة.

ملاحظة أمنية: يقوم `openclaw plugins install` بتثبيت تبعيات Plugin باستخدام
`npm install --omit=dev --ignore-scripts` (من دون نصوص دورة الحياة، ومن دون تبعيات تطوير وقت التشغيل). أبقِ شجرات تبعيات Plugin
"JS/TS خالصة" وتجنّب الحزم التي تتطلب بناءات `postinstall`.

اختياري: يمكن لـ `openclaw.setupEntry` أن يشير إلى وحدة خفيفة مخصصة للإعداد فقط.
وعندما يحتاج OpenClaw إلى أسطح إعداد لـ Plugin قناة معطلة، أو
عندما تكون Plugin قناة مفعّلة لكنها لا تزال غير مهيأة، فإنه يحمّل `setupEntry`
بدل مدخل Plugin الكامل. وهذا يبقي بدء التشغيل والإعداد أخف
عندما كان مدخل Plugin الرئيسي لديك يوصّل أيضًا أدوات أو خطافات أو كودًا
آخر خاصًا بوقت التشغيل فقط.

اختياري: يمكن لـ `openclaw.startup.deferConfiguredChannelFullLoadUntilAfterListen`
أن يُدرج Plugin قناة في مسار `setupEntry` نفسه خلال
مرحلة بدء التشغيل السابقة للاستماع في Gateway، حتى عندما تكون القناة مهيأة بالفعل.

استخدم هذا فقط عندما يغطي `setupEntry` بالكامل سطح بدء التشغيل الذي يجب أن يوجد
قبل أن يبدأ Gateway في الاستماع. وعمليًا، يعني ذلك أن مدخل الإعداد
يجب أن يسجّل كل قدرة مملوكة للقناة يعتمد عليها بدء التشغيل، مثل:

- تسجيل القناة نفسها
- أي مسارات HTTP يجب أن تكون متاحة قبل أن يبدأ Gateway في الاستماع
- أي أساليب Gateway أو أدوات أو خدمات يجب أن توجد خلال هذه النافذة نفسها

إذا كان مدخلك الكامل لا يزال يملك أي قدرة بدء تشغيل مطلوبة، فلا تمكّن
هذا العلم. أبقِ Plugin على السلوك الافتراضي ودَع OpenClaw يحمّل
المدخل الكامل أثناء بدء التشغيل.

يمكن للقنوات المضمّنة أيضًا نشر مساعدات سطح عقد للإعداد فقط بحيث يستطيع القلب
الرجوع إليها قبل تحميل وقت تشغيل القناة الكامل. وسطح ترقية الإعداد الحالي
هو:

- `singleAccountKeysToMove`
- `namedAccountPromotionKeys`
- `resolveSingleAccountPromotionTarget(...)`

يستخدم القلب هذا السطح عندما يحتاج إلى ترقية إعداد قناة قديم أحادي الحساب
إلى `channels.<id>.accounts.*` من دون تحميل مدخل Plugin الكامل.
وتُعد Matrix المثال المضمّن الحالي: فهي تنقل فقط مفاتيح المصادقة/bootstrap إلى
حساب مسمى مُرقّى عندما تكون الحسابات المسماة موجودة بالفعل، ويمكنها
الحفاظ على مفتاح حساب افتراضي غير قياسي مهيأ بدل إنشاء
`accounts.default` دائمًا.

تُبقي مكيّفات رقعة الإعداد هذه اكتشاف سطح العقد المضمّن كسولًا.
فيبقى وقت الاستيراد خفيفًا؛ ولا يتم تحميل سطح الترقية إلا عند أول استخدام بدل
إعادة الدخول إلى بدء تشغيل القناة المضمّنة عند استيراد الوحدة.

وعندما تتضمن أسطح بدء التشغيل تلك أساليب Gateway RPC، فأبقها على
بادئة خاصة بالـ Plugin. وتظل مساحات أسماء الإدارة الأساسية (`config.*`,
و`exec.approvals.*`, و`wizard.*`, و`update.*`) محجوزة وتُحل دائمًا
إلى `operator.admin`، حتى لو طلبت Plugin نطاقًا أضيق.

مثال:

```json
{
  "name": "@scope/my-channel",
  "openclaw": {
    "extensions": ["./index.ts"],
    "setupEntry": "./setup-entry.ts",
    "startup": {
      "deferConfiguredChannelFullLoadUntilAfterListen": true
    }
  }
}
```

### بيانات وصفية لكتالوج القنوات

يمكن لـ Plugins القنوات الإعلان عن بيانات وصفية للإعداد/الاكتشاف عبر `openclaw.channel` و
تلميحات التثبيت عبر `openclaw.install`. وهذا يبقي بيانات كتالوج القلب خالية من البيانات الثابتة.

مثال:

```json
{
  "name": "@openclaw/nextcloud-talk",
  "openclaw": {
    "extensions": ["./index.ts"],
    "channel": {
      "id": "nextcloud-talk",
      "label": "Nextcloud Talk",
      "selectionLabel": "Nextcloud Talk (self-hosted)",
      "docsPath": "/channels/nextcloud-talk",
      "docsLabel": "nextcloud-talk",
      "blurb": "Self-hosted chat via Nextcloud Talk webhook bots.",
      "order": 65,
      "aliases": ["nc-talk", "nc"]
    },
    "install": {
      "npmSpec": "@openclaw/nextcloud-talk",
      "localPath": "<bundled-plugin-local-path>",
      "defaultChoice": "npm"
    }
  }
}
```

حقول `openclaw.channel` المفيدة إلى جانب المثال الأدنى:

- `detailLabel`: تسمية ثانوية لأسطح الكتالوج/الحالة الأكثر غنى
- `docsLabel`: تجاوز نص رابط المستندات
- `preferOver`: معرّفات Plugin/قنوات أقل أولوية يجب أن يتفوق عليها هذا الإدخال في الكتالوج
- `selectionDocsPrefix`, `selectionDocsOmitLabel`, `selectionExtras`: عناصر تحكم النص على سطح الاختيار
- `markdownCapable`: يعلّم القناة على أنها قادرة على Markdown لقرارات التنسيق الصادر
- `exposure.configured`: يخفي القناة من أسطح سرد القنوات المهيأة عند تعيينه إلى `false`
- `exposure.setup`: يخفي القناة من منتقيات الإعداد/التهيئة التفاعلية عند تعيينه إلى `false`
- `exposure.docs`: يعلّم القناة على أنها داخلية/خاصة لأسطح التنقل في المستندات
- `showConfigured` / `showInSetup`: أسماء مستعارة قديمة لا تزال مقبولة للتوافق؛ ويفضَّل `exposure`
- `quickstartAllowFrom`: يُدرج القناة في تدفق `allowFrom` القياسي للبدء السريع
- `forceAccountBinding`: يفرض ربط حساب صريحًا حتى عندما يوجد حساب واحد فقط
- `preferSessionLookupForAnnounceTarget`: يفضّل البحث في الجلسة عند تحليل أهداف الإعلان

يمكن لـ OpenClaw أيضًا دمج **كتالوجات قنوات خارجية** (مثل
تصدير سجل MPM). ضع ملف JSON في أحد المواقع التالية:

- `~/.openclaw/mpm/plugins.json`
- `~/.openclaw/mpm/catalog.json`
- `~/.openclaw/plugins/catalog.json`

أو وجّه `OPENCLAW_PLUGIN_CATALOG_PATHS` (أو `OPENCLAW_MPM_CATALOG_PATHS`) إلى
ملف JSON واحد أو أكثر (مفصولًا بفواصل أو فاصلة منقوطة أو وفق `PATH`). يجب أن يحتوي كل ملف
على `{ "entries": [ { "name": "@scope/pkg", "openclaw": { "channel": {...}, "install": {...} } } ] }`. ويقبل المحلل أيضًا `"packages"` أو `"plugins"` كأسماء مستعارة قديمة للمفتاح `"entries"`.

## Plugins محرك السياق

تملك Plugins محرك السياق تنسيق سياق الجلسة لعمليات الإدخال، والتجميع،
وCompaction. سجّلها من Plugin الخاصة بك باستخدام
`api.registerContextEngine(id, factory)`، ثم اختر المحرك النشط عبر
`plugins.slots.contextEngine`.

استخدم هذا عندما تحتاج Plugin الخاصة بك إلى استبدال مسار
السياق الافتراضي أو توسيعه بدل الاكتفاء بإضافة البحث في الذاكرة أو الخطافات.

```ts
import { buildMemorySystemPromptAddition } from "openclaw/plugin-sdk/core";

export default function (api) {
  api.registerContextEngine("lossless-claw", () => ({
    info: { id: "lossless-claw", name: "Lossless Claw", ownsCompaction: true },
    async ingest() {
      return { ingested: true };
    },
    async assemble({ messages, availableTools, citationsMode }) {
      return {
        messages,
        estimatedTokens: 0,
        systemPromptAddition: buildMemorySystemPromptAddition({
          availableTools: availableTools ?? new Set(),
          citationsMode,
        }),
      };
    },
    async compact() {
      return { ok: true, compacted: false };
    },
  }));
}
```

إذا كان محركك **لا** يملك خوارزمية Compaction، فأبقِ `compact()`
منفذة وفوّضها صراحةً:

```ts
import {
  buildMemorySystemPromptAddition,
  delegateCompactionToRuntime,
} from "openclaw/plugin-sdk/core";

export default function (api) {
  api.registerContextEngine("my-memory-engine", () => ({
    info: {
      id: "my-memory-engine",
      name: "My Memory Engine",
      ownsCompaction: false,
    },
    async ingest() {
      return { ingested: true };
    },
    async assemble({ messages, availableTools, citationsMode }) {
      return {
        messages,
        estimatedTokens: 0,
        systemPromptAddition: buildMemorySystemPromptAddition({
          availableTools: availableTools ?? new Set(),
          citationsMode,
        }),
      };
    },
    async compact(params) {
      return await delegateCompactionToRuntime(params);
    },
  }));
}
```

## إضافة قدرة جديدة

عندما تحتاج Plugin إلى سلوك لا يلائم API الحالية، فلا تتجاوز
نظام Plugins عبر اختراق خاص. أضف القدرة الناقصة.

التسلسل الموصى به:

1. عرّف عقد القلب
   حدّد ما السلوك المشترك الذي يجب أن يملكه القلب: السياسة، والرجوع الاحتياطي، ودمج الإعدادات،
   ودورة الحياة، والدلالات المواجهة للقنوات، وشكل مساعد وقت التشغيل.
2. أضف أسطح تسجيل/وقت تشغيل مكتوبة النوع للـ Plugin
   وسّع `OpenClawPluginApi` و/أو `api.runtime` بأصغر
   سطح قدرة مكتوب النوع ومفيد.
3. وصّل مستهلكي القلب + القنوات/الميزات
   يجب أن تستهلك القنوات وPlugins الميزات القدرة الجديدة عبر القلب،
   لا عبر استيراد تنفيذ خاص بمورّد مباشرةً.
4. سجّل تنفيذات المورّد
   ثم تقوم Plugins المورّدين بتسجيل واجهاتها الخلفية مقابل القدرة.
5. أضف تغطية العقد
   أضف اختبارات حتى تبقى الملكية وشكل التسجيل صريحين مع مرور الوقت.

بهذه الطريقة يبقى OpenClaw صاحب رأي من دون أن يصبح مضمّنًا
في رؤية موفّر واحد للعالم. راجع [كتاب وصفات القدرات](/ar/plugins/architecture)
للحصول على قائمة ملفات عملية ومثال تطبيقي.

### قائمة التحقق الخاصة بالقدرة

عندما تضيف قدرة جديدة، يجب أن يلمس التنفيذ عادةً هذه
الأسطح معًا:

- أنواع عقد القلب في `src/<capability>/types.ts`
- مساعد وقت التشغيل/التشغيل في القلب داخل `src/<capability>/runtime.ts`
- سطح تسجيل API الخاص بالـ Plugin في `src/plugins/types.ts`
- توصيل سجل Plugins في `src/plugins/registry.ts`
- تعريض وقت تشغيل Plugin في `src/plugins/runtime/*` عندما تحتاج
  Plugins الميزات/القنوات إلى استهلاكها
- مساعدات الالتقاط/الاختبار في `src/test-utils/plugin-registration.ts`
- تأكيدات الملكية/العقد في `src/plugins/contracts/registry.ts`
- مستندات المشغّل/الـ Plugin في `docs/`

إذا كان أحد هذه الأسطح مفقودًا، فهذه عادةً علامة على أن القدرة
لم تُدمج بالكامل بعد.

### قالب القدرة

النمط الأدنى:

```ts
// core contract
export type VideoGenerationProviderPlugin = {
  id: string;
  label: string;
  generateVideo: (req: VideoGenerationRequest) => Promise<VideoGenerationResult>;
};

// plugin API
api.registerVideoGenerationProvider({
  id: "openai",
  label: "OpenAI",
  async generateVideo(req) {
    return await generateOpenAiVideo(req);
  },
});

// shared runtime helper for feature/channel plugins
const clip = await api.runtime.videoGeneration.generate({
  prompt: "Show the robot walking through the lab.",
  cfg,
});
```

نمط اختبار العقد:

```ts
expect(findVideoGenerationProviderIdsForPlugin("openai")).toEqual(["openai"]);
```

وهذا يبقي القاعدة بسيطة:

- يملك القلب عقد القدرة + التنسيق
- تملك Plugins المورّدين تنفيذات المورّد
- تستهلك Plugins الميزات/القنوات مساعدات وقت التشغيل
- تُبقي اختبارات العقد الملكية صريحة

## نموذج الإمكانات العام

الإمكانات هي النموذج العام لـ **Plugins الأصلية** داخل OpenClaw. يسجّل كل
Plugin أصلي في OpenClaw نفسه مقابل نوع أو أكثر من أنواع الإمكانات:

| الإمكانية             | طريقة التسجيل                                   | أمثلة على Plugins                    |
| --------------------- | ------------------------------------------------ | ------------------------------------ |
| استدلال نصي           | `api.registerProvider(...)`                      | `openai`، `anthropic`                |
| واجهة خلفية لاستدلال CLI | `api.registerCliBackend(...)`                 | `openai`، `anthropic`                |
| الكلام                | `api.registerSpeechProvider(...)`                | `elevenlabs`، `microsoft`            |
| النسخ الفوري           | `api.registerRealtimeTranscriptionProvider(...)` | `openai`                             |
| الصوت الفوري          | `api.registerRealtimeVoiceProvider(...)`         | `openai`                             |
| فهم الوسائط           | `api.registerMediaUnderstandingProvider(...)`    | `openai`، `google`                   |
| توليد الصور           | `api.registerImageGenerationProvider(...)`       | `openai`، `google`، `fal`، `minimax` |
| توليد الموسيقى        | `api.registerMusicGenerationProvider(...)`       | `google`، `minimax`                  |
| توليد الفيديو         | `api.registerVideoGenerationProvider(...)`       | `qwen`                               |
| جلب الويب             | `api.registerWebFetchProvider(...)`              | `firecrawl`                          |
| البحث على الويب       | `api.registerWebSearchProvider(...)`             | `google`                             |
| القناة / المراسلة     | `api.registerChannel(...)`                       | `msteams`، `matrix`                  |

أما الـ Plugin الذي لا يسجّل أي إمكانة لكنه يوفّر hooks أو tools أو
services، فيُعد **Plugin قديمًا يعتمد على hooks فقط**. وما يزال هذا النمط
مدعومًا بالكامل.

### موقف التوافق الخارجي

تم اعتماد نموذج الإمكانات داخل الأساس ويُستخدم اليوم من قِبل Plugins
المضمّنة/الأصلية، لكن توافق Plugins الخارجية ما يزال يحتاج إلى معيار أوضح من "بما أنه
مصدَّر، فهو إذًا ثابت."

الإرشادات الحالية:

- **Plugins الخارجية الحالية:** حافظ على عمل التكاملات المعتمدة على hooks؛ واعتبر
  هذا خط الأساس للتوافق
- **Plugins المضمّنة/الأصلية الجديدة:** فضّل تسجيل الإمكانات الصريح بدلًا من
  التوغلات الخاصة بالبائع أو التصميمات الجديدة المعتمدة على hooks فقط
- **Plugins الخارجية التي تعتمد تسجيل الإمكانات:** هذا مسموح، لكن اعتبر
  الأسطح المساعدة الخاصة بالإمكانات قابلة للتطور ما لم تنص الوثائق صراحةً على أن
  العقد مستقر

القاعدة العملية:

- APIs تسجيل الإمكانات هي الاتجاه المقصود
- تظل hooks القديمة المسار الأكثر أمانًا لتفادي الكسر بالنسبة إلى Plugins الخارجية أثناء
  الانتقال
- ليست كل المسارات الفرعية المصدّرة للمساعدات متساوية؛ فافضّل العقد
  الضيق الموثق، لا المساعدات المصدّرة عرضًا

### أشكال Plugin

يصنّف OpenClaw كل Plugin محمّل إلى شكل معين استنادًا إلى سلوك
التسجيل الفعلي الخاص به (وليس فقط البيانات الثابتة):

- **plain-capability** -- يسجّل نوع إمكانة واحدًا فقط (مثلًا
  Plugin مزود فقط مثل `mistral`)
- **hybrid-capability** -- يسجّل عدة أنواع من الإمكانات (مثل
  `openai` الذي يمتلك الاستدلال النصي والكلام وفهم الوسائط وتوليد
  الصور)
- **hook-only** -- يسجّل hooks فقط (مكتوبة النوع أو مخصصة)، بدون إمكانات،
  أو tools، أو commands، أو services
- **non-capability** -- يسجّل tools أو commands أو services أو routes لكن دون
  إمكانات

استخدم `openclaw plugins inspect <id>` لرؤية شكل Plugin وتفصيل
الإمكانات. راجع [مرجع CLI](/cli/plugins#inspect) للتفاصيل.

### hooks القديمة

ما يزال hook `before_agent_start` مدعومًا كمسار توافق بالنسبة إلى
Plugins المعتمدة على hooks فقط. وما تزال Plugins القديمة الواقعية تعتمد عليه.

الاتجاه:

- استمر في دعمه
- وثّقه باعتباره قديمًا
- فضّل `before_model_resolve` لأعمال تجاوز النموذج/المزود
- فضّل `before_prompt_build` لأعمال تعديل prompt
- لا تزلْه إلا بعد انخفاض الاستخدام الفعلي وتأكد تغطية fixtures من أمان الترحيل

### إشارات التوافق

عند تشغيل `openclaw doctor` أو `openclaw plugins inspect <id>`، قد ترى
إحدى هذه التسميات:

| الإشارة                    | المعنى                                                       |
| -------------------------- | ------------------------------------------------------------ |
| **config valid**           | يتم تحليل التهيئة بشكل صحيح وتُحل Plugins                    |
| **compatibility advisory** | تستخدم Plugin نمطًا مدعومًا لكنه أقدم (مثل `hook-only`)     |
| **legacy warning**         | تستخدم Plugin `before_agent_start`، وهو مهمل                |
| **hard error**             | التهيئة غير صالحة أو فشل تحميل Plugin                        |

لن يؤدي `hook-only` ولا `before_agent_start` إلى كسر Plugin لديك اليوم --
فـ `hook-only` مجرد تنبيه، و`before_agent_start` لا يؤدي إلا إلى تحذير. كما
تظهر هذه الإشارات أيضًا في `openclaw status --all` و`openclaw plugins doctor`.

## نظرة عامة على البنية

يحتوي نظام Plugins في OpenClaw على أربع طبقات:

1. **البيان + الاكتشاف**
   يجد OpenClaw Plugins المرشحة من المسارات المهيأة، وجذور workspace،
   وجذور extension العامة، وextensions المضمّنة. ويقرأ الاكتشاف بيانات
   `openclaw.plugin.json` الأصلية وبيانات bundle المدعومة أولًا.
2. **التمكين + التحقق**
   يقرر الأساس ما إذا كانت Plugin المكتشفة مفعلة أو معطلة أو محظورة أو
   محددة لفتحة حصرية مثل الذاكرة.
3. **التحميل في runtime**
   تُحمّل Plugins OpenClaw الأصلية داخل العملية عبر jiti وتُسجّل
   الإمكانات في سجل مركزي. أما bundles المتوافقة فتُطبَّع إلى
   سجلات ضمن السجل دون استيراد كود runtime.
4. **استهلاك السطح**
   يقرأ باقي OpenClaw السجل لكشف tools والقنوات وإعداد
   المزود وhooks وHTTP routes وأوامر CLI وservices.

وبالنسبة إلى Plugin CLI تحديدًا، ينقسم اكتشاف الأمر الجذري إلى مرحلتين:

- تأتي بيانات وقت التحليل من `registerCli(..., { descriptors: [...] })`
- ويمكن لوحدة CLI الحقيقية الخاصة بـ Plugin أن تظل lazy وتُسجَّل عند أول استدعاء

وهذا يُبقي كود CLI المملوك لـ Plugin داخل Plugin نفسها مع السماح لـ OpenClaw
بحجز أسماء الأوامر الجذرية قبل التحليل.

الحد التصميمي المهم:

- يجب أن يعمل الاكتشاف + التحقق من التهيئة من **بيانات البيان/المخطط**
  من دون تنفيذ كود Plugin
- يأتي سلوك runtime الأصلي من مسار `register(api)` داخل وحدة Plugin

يتيح هذا الفصل لـ OpenClaw التحقق من التهيئة وشرح Plugins المفقودة/المعطلة
وبناء تلميحات UI/schema قبل أن يصبح runtime الكامل نشطًا.

### Plugins القنوات وأداة الرسائل المشتركة

لا تحتاج Plugins القنوات إلى تسجيل أداة منفصلة للإرسال/التعديل/التفاعل من أجل
إجراءات الدردشة العادية. يحتفظ OpenClaw بأداة `message` مشتركة واحدة في الأساس،
وتمتلك Plugins القنوات الاكتشاف والتنفيذ الخاصين بالقناة وراءها.

الحد الحالي هو:

- يمتلك الأساس مضيف أداة `message` المشتركة، وربط prompt، وحفظ الجلسات/الخيوط،
  وإرسال التنفيذ
- تمتلك Plugins القنوات اكتشاف الإجراءات المحددة النطاق، واكتشاف الإمكانات، وأي أجزاء schema خاصة بالقناة
- تمتلك Plugins القنوات قواعد المحادثة الخاصة بالجلسة على مستوى المزود، مثل
  كيفية ترميز معرّفات المحادثات لمعرّفات الخيوط أو وراثتها من المحادثات الأصلية
- تنفذ Plugins القنوات الإجراء النهائي عبر action adapter الخاص بها

بالنسبة إلى Plugins القنوات، يكون سطح SDK هو
`ChannelMessageActionAdapter.describeMessageTool(...)`. تسمح مكالمة الاكتشاف
الموحدة هذه لـ Plugin بإرجاع إجراءاتها المرئية وإمكاناتها ومساهمات
schema الخاصة بها معًا حتى لا تنحرف هذه الأجزاء عن بعضها.

عندما يحمل أحد معلمات أداة الرسائل الخاصة بالقناة مصدر وسائط مثل
مسار محلي أو عنوان URL لوسائط بعيدة، ينبغي على Plugin أيضًا أن تُرجع
`mediaSourceParams` من `describeMessageTool(...)`. يستخدم الأساس هذه
القائمة الصريحة لتطبيق تطبيع مسارات sandbox وتلميحات الوصول إلى الوسائط الصادرة
من دون ترميز أسماء معلمات تملكها Plugin بشكل صلب.
ويُفضّل هناك الخرائط المحددة بالإجراء، وليس قائمة مسطحة واحدة على مستوى القناة،
بحيث لا يُطبَّع معلم وسائط خاص بملف تعريفي على إجراءات غير مرتبطة مثل
`send`.

يمرر الأساس نطاق runtime إلى خطوة الاكتشاف تلك. وتشمل الحقول المهمة:

- `accountId`
- `currentChannelId`
- `currentThreadTs`
- `currentMessageId`
- `sessionKey`
- `sessionId`
- `agentId`
- `requesterSenderId` الوارد الموثوق

وهذا مهم بالنسبة إلى Plugins الحساسة للسياق. إذ يمكن للقناة إخفاء أو إظهار
إجراءات الرسائل بناءً على الحساب النشط، أو الغرفة/الخيط/الرسالة الحالية، أو
هوية الطالب الموثوقة، من دون ترميز تفريعات خاصة بالقناة داخل أداة `message`
الأساسية.

ولهذا السبب ما تزال تغييرات توجيه embedded-runner من عمل Plugin: إذ إن runner
مسؤول عن تمرير هوية الدردشة/الجلسة الحالية إلى حد اكتشاف Plugin بحيث
تكشف أداة `message` المشتركة السطح الصحيح المملوك للقناة للدورة الحالية.

أما بالنسبة إلى مساعدات التنفيذ المملوكة للقناة، فينبغي على Plugins المضمّنة أن تُبقي
runtime التنفيذ داخل وحدات extension الخاصة بها. لم يعد الأساس يمتلك
runtimes إجراءات رسائل Discord وSlack وTelegram وWhatsApp ضمن `src/agents/tools`.
كما أننا لا ننشر مسارات فرعية منفصلة `plugin-sdk/*-action-runtime`، وينبغي على
Plugins المضمّنة استيراد كود runtime المحلي الخاص بها مباشرة من
الوحدات المملوكة لـ extension الخاصة بها.

وينطبق الحد نفسه على واجهات SDK المسماة بحسب المزود عمومًا: يجب ألا
يستورد الأساس حِزمًا مريحة خاصة بقنوات Slack أو Discord أو Signal أو
WhatsApp أو extensions مشابهة. وإذا احتاج الأساس إلى سلوك ما، فإما أن يستهلك
الحزمة `api.ts` / `runtime-api.ts` الخاصة بـ Plugin المضمّنة نفسها أو
يرفع الحاجة إلى إمكانة عامة ضيقة داخل SDK المشتركة.

وبالنسبة إلى الاستطلاعات تحديدًا، توجد مسارا تنفيذ:

- `outbound.sendPoll` هو الأساس المشترك للقنوات التي تتوافق مع نموذج
  الاستطلاع العام
- `actions.handleAction("poll")` هو المسار المفضل للدلالات الخاصة بالقناة
  أو لمعلمات الاستطلاع الإضافية

ويؤجل الأساس الآن تحليل الاستطلاع المشترك إلى ما بعد رفض إرسال الاستطلاع من Plugin
للإجراء، بحيث يمكن لمعالجات الاستطلاع المملوكة لـ Plugin قبول حقول
استطلاع خاصة بالقناة من دون أن يمنعها محلل الاستطلاع العام أولًا.

راجع [مسار التحميل](#load-pipeline) للتسلسل الكامل لبدء التشغيل.

## نموذج ملكية الإمكانات

يتعامل OpenClaw مع Plugin الأصلية على أنها حد الملكية لـ **شركة** أو
**ميزة**، وليس كحقيبة مختلطة من عمليات التكامل غير المرتبطة.

وهذا يعني:

- ينبغي عادةً أن تمتلك Plugin الشركة جميع الأسطح
  المواجهة لـ OpenClaw الخاصة بتلك الشركة
- ينبغي عادةً أن تمتلك Plugin الميزة كامل سطح الميزة التي تقدمها
- ينبغي أن تستهلك القنوات إمكانات الأساس المشتركة بدلًا من إعادة تنفيذ سلوك
  المزود بشكل مخصص

أمثلة:

- يمتلك Plugin المضمّن `openai` سلوك مزود نماذج OpenAI وسلوك OpenAI
  للكلام + الصوت الفوري + فهم الوسائط + توليد الصور
- يمتلك Plugin المضمّن `elevenlabs` سلوك الكلام الخاص بـ ElevenLabs
- يمتلك Plugin المضمّن `microsoft` سلوك الكلام الخاص بـ Microsoft
- يمتلك Plugin المضمّن `google` سلوك مزود نماذج Google بالإضافة إلى سلوك Google
  لفهم الوسائط + توليد الصور + البحث على الويب
- يمتلك Plugin المضمّن `firecrawl` سلوك جلب الويب الخاص بـ Firecrawl
- تمتلك Plugins المضمّنة `minimax` و`mistral` و`moonshot` و`zai`
  واجهاتها الخلفية الخاصة بفهم الوسائط
- تُعد Plugin ‏`voice-call` Plugin ميزة: فهي تمتلك نقل المكالمات وtools وCLI وroutes
  وجسر Twilio media-stream، لكنها تستهلك إمكانات الكلام المشتركة
  بالإضافة إلى النسخ الفوري والصوت الفوري بدلًا من استيراد Plugins البائع مباشرةً

الحالة النهائية المقصودة هي:

- تعيش OpenAI في Plugin واحدة حتى لو امتدت إلى النماذج النصية والكلام والصور و
  الفيديو مستقبلًا
- ويمكن لبائع آخر أن يفعل الشيء نفسه لمساحته الخاصة
- ولا تهتم القنوات بأي Plugin بائع يمتلك المزود؛ فهي تستهلك عقد الإمكانات
  المشترك الذي يكشفه الأساس

وهذا هو التمييز الأساسي:

- **plugin** = حد الملكية
- **capability** = عقد أساسي يمكن لعدة Plugins تنفيذه أو استهلاكه

لذلك إذا أضاف OpenClaw مجالًا جديدًا مثل الفيديو، فليس السؤال الأول هو
"أي مزود يجب أن يرمّز معالجة الفيديو بشكل صلب؟" بل السؤال الأول هو "ما هو
عقد إمكانات الفيديو الأساسي؟" وبمجرد وجود هذا العقد، يمكن لـ Plugins البائع
التسجيل مقابله، ويمكن لـ Plugins القنوات/الميزات استهلاكه.

إذا لم تكن الإمكانية موجودة بعد، فعادةً ما تكون الخطوة الصحيحة هي:

1. تعريف الإمكانية المفقودة في الأساس
2. كشفها عبر Plugin API/runtime بطريقة مكتوبة النوع
3. ربط القنوات/الميزات بهذه الإمكانية
4. وترك Plugins البائع تسجّل تنفيذاتها

وهذا يُبقي الملكية صريحة مع تجنب سلوك أساسي يعتمد على بائع واحد
أو على مسار كود خاص بـ Plugin واحدة.

### طبقات الإمكانات

استخدم هذا النموذج الذهني عند تقرير مكان الكود:

- **طبقة الإمكانات الأساسية**: التنسيق المشترك، والسياسات، والرجوع الاحتياطي، وقواعد
  دمج التهيئة، ودلالات التسليم، والعقود المكتوبة النوع
- **طبقة Plugin البائع**: APIs الخاصة بالبائع، والمصادقة، وفهارس النماذج، وتركيب الكلام،
  وتوليد الصور، وواجهات الفيديو الخلفية مستقبلًا، ونقاط نهاية الاستخدام
- **طبقة Plugin القناة/الميزة**: تكامل Slack/Discord/voice-call/إلخ
  الذي يستهلك الإمكانات الأساسية ويعرضها على سطح ما

على سبيل المثال، يتبع TTS هذا الشكل:

- يمتلك الأساس سياسة TTS وقت الرد، وترتيب الرجوع الاحتياطي، والتفضيلات، وتسليم القناة
- وتمتلك `openai` و`elevenlabs` و`microsoft` تنفيذات التركيب
- وتستهلك `voice-call` مساعد runtime الخاص بـ TTS الهاتفي

ويجب تفضيل النمط نفسه للإمكانات المستقبلية.

### مثال على Plugin شركة متعددة الإمكانات

ينبغي أن تبدو Plugin الشركة متماسكة من الخارج. فإذا كان لدى OpenClaw عقود مشتركة
للنماذج والكلام والنسخ الفوري والصوت الفوري وفهم الوسائط
وتوليد الصور وتوليد الفيديو وجلب الويب والبحث على الويب،
فيمكن لبائع ما أن يمتلك جميع أسطحه في مكان واحد:

```ts
import type { OpenClawPluginDefinition } from "openclaw/plugin-sdk/plugin-entry";
import {
  describeImageWithModel,
  transcribeOpenAiCompatibleAudio,
} from "openclaw/plugin-sdk/media-understanding";

const plugin: OpenClawPluginDefinition = {
  id: "exampleai",
  name: "ExampleAI",
  register(api) {
    api.registerProvider({
      id: "exampleai",
      // auth/model catalog/runtime hooks
    });

    api.registerSpeechProvider({
      id: "exampleai",
      // vendor speech config — implement the SpeechProviderPlugin interface directly
    });

    api.registerMediaUnderstandingProvider({
      id: "exampleai",
      capabilities: ["image", "audio", "video"],
      async describeImage(req) {
        return describeImageWithModel({
          provider: "exampleai",
          model: req.model,
          input: req.input,
        });
      },
      async transcribeAudio(req) {
        return transcribeOpenAiCompatibleAudio({
          provider: "exampleai",
          model: req.model,
          input: req.input,
        });
      },
    });

    api.registerWebSearchProvider(
      createPluginBackedWebSearchProvider({
        id: "exampleai-search",
        // credential + fetch logic
      }),
    );
  },
};

export default plugin;
```

ما يهم ليس أسماء المساعدات الدقيقة. بل المهم هو الشكل:

- Plugin واحدة تمتلك سطح البائع
- ولا يزال الأساس يمتلك عقود الإمكانات
- وتستهلك القنوات وPlugins الميزات مساعدات `api.runtime.*`، لا كود البائع
- ويمكن لاختبارات العقد التأكد من أن Plugin سجّلت الإمكانات التي
  تدّعي امتلاكها

### مثال على الإمكانية: فهم الفيديو

يعامل OpenClaw بالفعل فهم الصور/الصوت/الفيديو كإمكانية مشتركة
واحدة. وينطبق نموذج الملكية نفسه هنا:

1. يعرّف الأساس عقد فهم الوسائط
2. تسجّل Plugins البائع `describeImage` و`transcribeAudio` و
   `describeVideo` حسب الاقتضاء
3. وتستهلك القنوات وPlugins الميزات السلوك الأساسي المشترك بدلًا من
   الربط المباشر بكود البائع

وهذا يتجنب ترسيخ افتراضات الفيديو الخاصة بمزود واحد داخل الأساس. إذ تمتلك Plugin
سطح البائع؛ بينما يمتلك الأساس عقد الإمكانية وسلوك الرجوع الاحتياطي.

ويستخدم توليد الفيديو بالفعل التسلسل نفسه: يمتلك الأساس عقد
الإمكانية المكتوب النوع ومساعد runtime، وتقوم Plugins البائع بتسجيل
تنفيذات `api.registerVideoGenerationProvider(...)` مقابله.

هل تحتاج إلى قائمة تنفيذ عملية؟ راجع
[دليل الإمكانات العملي](/ar/plugins/architecture).

## العقود وآليات الإنفاذ

إن سطح Plugin API مقصود أن يكون مكتوب النوع ومركزيًا داخل
`OpenClawPluginApi`. ويعرّف هذا العقد نقاط التسجيل المدعومة
ومساعدات runtime التي يمكن لـ Plugin الاعتماد عليها.

سبب أهمية ذلك:

- يحصل مؤلفو Plugins على معيار داخلي مستقر واحد
- يمكن للأساس رفض الملكية المكررة مثل تسجيل Pluginَين للمعرّف نفسه
  الخاص بالمزود
- يمكن لبدء التشغيل عرض تشخيصات عملية للتسجيلات المشوهة
- يمكن لاختبارات العقد فرض ملكية Plugins المضمّنة ومنع الانحراف الصامت

توجد طبقتان من الإنفاذ:

1. **إنفاذ التسجيل في runtime**
   يتحقق سجل Plugins من التسجيلات أثناء تحميل Plugins. ومن الأمثلة:
   معرّفات المزود المكررة، ومعرّفات مزود الكلام المكررة، والتسجيلات
   المشوهة؛ حيث تنتج تشخيصات Plugin بدلًا من سلوك غير معرّف.
2. **اختبارات العقد**
   تُلتقط Plugins المضمّنة داخل سجلات عقود أثناء تشغيل الاختبارات حتى
   يتمكن OpenClaw من التأكيد الصريح على الملكية. ويُستخدم هذا اليوم من أجل
   مزودي النماذج، ومزودي الكلام، ومزودي البحث على الويب، وملكية التسجيلات المضمّنة.

والنتيجة العملية هي أن OpenClaw يعرف مسبقًا أي Plugin يمتلك أي
سطح. وهذا يتيح للأساس والقنوات التركيب بسلاسة لأن الملكية
معلنة، ومكتوبة النوع، وقابلة للاختبار بدلًا من أن تكون ضمنية.

### ما الذي يجب أن يكون ضمن عقد

العقود الجيدة لـ Plugin تكون:

- مكتوبة النوع
- صغيرة
- خاصة بإمكانية معينة
- مملوكة للأساس
- قابلة لإعادة الاستخدام من قِبل عدة Plugins
- قابلة للاستهلاك من القنوات/الميزات دون معرفة بالبائع

أما العقود السيئة لـ Plugin فهي:

- سياسة خاصة بالبائع مخفية داخل الأساس
- منافذ هروب خاصة بـ Plugin واحدة تتجاوز السجل
- كود قناة يصل مباشرة إلى تنفيذ خاص ببائع
- كائنات runtime مخصصة ليست جزءًا من `OpenClawPluginApi` أو
  `api.runtime`

عند التردد، ارفع مستوى التجريد: عرّف الإمكانية أولًا، ثم
اسمح لـ Plugins بالتوصيل بها.

## نموذج التنفيذ

تعمل Plugins OpenClaw الأصلية **داخل العملية** مع Gateway. وهي ليست
معزولة داخل sandbox. وتملك Plugin الأصلية المحمّلة حد الثقة على مستوى العملية نفسه
الذي يملكه كود الأساس.

الآثار المترتبة:

- يمكن لـ Plugin الأصلية تسجيل tools ومعالجات الشبكة وhooks وservices
- يمكن لخلل في Plugin الأصلية أن يؤدي إلى تعطل gateway أو زعزعة استقرارها
- تعادل Plugin الأصلية الخبيثة تنفيذ تعليمات برمجية عشوائية داخل عملية OpenClaw

أما bundles المتوافقة فهي أكثر أمانًا افتراضيًا لأن OpenClaw يعاملها حاليًا
كحزم بيانات تعريف/محتوى. وهذا يعني في الإصدارات الحالية غالبًا
Skills المضمّنة.

استخدم قوائم السماح ومسارات التثبيت/التحميل الصريحة بالنسبة إلى Plugins غير المضمّنة. وتعامل مع
Plugins الخاصة بـ workspace باعتبارها كودًا وقت التطوير، لا إعدادات إنتاج افتراضية.

وبالنسبة إلى أسماء حزم workspace المضمّنة، أبقِ معرّف Plugin مثبتًا في اسم npm:
`@openclaw/<id>` افتراضيًا، أو لاحقة مكتوبة النوع معتمدة مثل
`-provider` أو `-plugin` أو `-speech` أو `-sandbox` أو `-media-understanding` عندما
تكشف الحزمة عمدًا دور Plugin أضيق.

ملاحظة ثقة مهمة:

- يثق `plugins.allow` في **معرّفات Plugins**، لا في مصدرها.
- تتعمد Plugin workspace التي تحمل المعرف نفسه لـ Plugin مضمّنة أن تظلّل
  النسخة المضمّنة عندما تكون Plugin workspace هذه مفعّلة/موجودة في قائمة السماح.
- وهذا سلوك طبيعي ومفيد للتطوير المحلي، واختبار التصحيحات، والإصلاحات العاجلة.

## حد التصدير

يصدّر OpenClaw الإمكانات، لا وسائل التنفيذ المريحة.

أبقِ تسجيل الإمكانات عامًا. وقلّص تصدير المساعدات غير التعاقدية:

- المسارات الفرعية الخاصة بمساعدات Plugins المضمّنة
- المسارات الفرعية لتمديدات runtime غير المقصودة كـ API عامة
- المساعدات المريحة الخاصة بالبائع
- مساعدات الإعداد/Onboarding التي تُعد تفاصيل تنفيذ

ما تزال بعض المسارات الفرعية المساعدة الخاصة بـ Plugins المضمّنة موجودة في خريطة تصدير SDK
المولدة لأغراض التوافق وصيانة Plugins المضمّنة. وتشمل الأمثلة الحالية
`plugin-sdk/feishu` و`plugin-sdk/feishu-setup` و`plugin-sdk/zalo`،
و`plugin-sdk/zalo-setup`، وعدة واجهات `plugin-sdk/matrix*`. تعامل مع هذه
على أنها تصديرات محجوزة لتفاصيل التنفيذ، لا بوصفها النمط الموصى به في SDK من أجل
Plugins الجديدة التابعة لجهات خارجية.

## مسار التحميل

عند بدء التشغيل، يقوم OpenClaw تقريبًا بما يلي:

1. اكتشاف جذور Plugins المرشحة
2. قراءة بيانات البيان الأصلية أو بيانات bundles المتوافقة وبيانات الحزم
3. رفض المرشحين غير الآمنين
4. تطبيع تهيئة Plugin (`plugins.enabled` و`allow` و`deny` و`entries`،
   و`slots` و`load.paths`)
5. تقرير التمكين لكل مرشح
6. تحميل الوحدات الأصلية المفعلة عبر jiti
7. استدعاء hooks الأصلية `register(api)` (أو `activate(api)` — وهو اسم مستعار قديم) وجمع التسجيلات داخل سجل Plugins
8. كشف السجل لأسطح الأوامر/runtime

<Note>
`activate` اسم مستعار قديم لـ `register` — يقوم المحمّل بحل أيهما موجود (`def.register ?? def.activate`) واستدعائه في النقطة نفسها. تستخدم جميع Plugins المضمّنة `register`؛ لذا فضّل `register` بالنسبة إلى Plugins الجديدة.
</Note>

تحدث بوابات الأمان **قبل** تنفيذ runtime. ويُحظر المرشحون
عندما يخرج الإدخال عن جذر Plugin، أو يكون المسار قابلاً للكتابة من الجميع، أو
تبدو ملكية المسار مشبوهة بالنسبة إلى Plugins غير المضمّنة.

### السلوك القائم على البيان أولًا

البيان هو مصدر الحقيقة في طبقة التحكم. يستخدمه OpenClaw من أجل:

- تعريف Plugin
- اكتشاف القنوات/Skills/مخطط التهيئة المعلنة أو إمكانات bundle
- التحقق من `plugins.entries.<id>.config`
- تعزيز تسميات Control UI والعناصر النائبة
- عرض بيانات التثبيت/الفهرس
- الحفاظ على واصفات التفعيل والإعداد الرخيصة من دون تحميل runtime الـ Plugin

أما بالنسبة إلى Plugins الأصلية، فإن وحدة runtime هي جزء طبقة البيانات. فهي تسجّل
السلوك الفعلي مثل hooks أو tools أو commands أو تدفقات المزود.

تظل كتل `activation` و`setup` الاختيارية في البيان ضمن طبقة التحكم.
فهي واصفات بيانات تعريف فقط لتخطيط التفعيل واكتشاف الإعداد؛
ولا تستبدل تسجيل runtime أو `register(...)` أو `setupEntry`.
أما أول المستهلكين الحيين للتفعيل فيستخدمون الآن تلميحات البيان الخاصة بالأوامر والقنوات والمزودات
لتضييق تحميل Plugins قبل تجسيد السجل الأوسع:

- يضيّق تحميل CLI إلى Plugins التي تمتلك الأمر الأساسي المطلوب
- ويضيّق إعداد القناة/حل Plugin إلى Plugins التي تمتلك معرّف
  القناة المطلوب
- ويضيّق حل إعداد/Runtime المزود الصريح إلى Plugins التي تمتلك معرّف
  المزود المطلوب

يُفضّل اكتشاف الإعداد الآن المعرّفات المملوكة للواصفات مثل `setup.providers` و
`setup.cliBackends` لتضييق Plugins المرشحة قبل أن يعود إلى
`setup-api` بالنسبة إلى Plugins التي ما تزال تحتاج إلى hooks runtime وقت الإعداد. وإذا ادعت
أكثر من Plugin مكتشفة واحدة ملكية معرّف مزود إعداد مُطبّع أو معرّف
واجهة CLI خلفية مُطبّعة نفسه، فإن البحث عن الإعداد يرفض المالك
الملتبس بدلًا من الاعتماد على ترتيب الاكتشاف.

### ما الذي يخزنه المحمّل مؤقتًا

يحتفظ OpenClaw بذاكرات مؤقتة قصيرة داخل العملية من أجل:

- نتائج الاكتشاف
- بيانات سجل البيان
- سجلات Plugins المحمّلة

تقلل هذه الذاكرات المؤقتة من ضغط بدء التشغيل المتكرر وتكرار الأوامر. ومن الآمن
اعتبارها ذواكر مؤقتة للأداء قصيرة العمر، لا آلية حفظ دائم.

ملاحظة أداء:

- اضبط `OPENCLAW_DISABLE_PLUGIN_DISCOVERY_CACHE=1` أو
  `OPENCLAW_DISABLE_PLUGIN_MANIFEST_CACHE=1` لتعطيل هذه الذواكر المؤقتة.
- اضبط نوافذ الذاكرة المؤقتة عبر `OPENCLAW_PLUGIN_DISCOVERY_CACHE_MS` و
  `OPENCLAW_PLUGIN_MANIFEST_CACHE_MS`.

## نموذج السجل

لا تقوم Plugins المحمّلة بتعديل متغيرات عامة عشوائية في الأساس مباشرةً. بل
تسجّل نفسها ضمن سجل Plugins مركزي.

يتتبع السجل ما يلي:

- سجلات Plugins (الهوية، والمصدر، والأصل، والحالة، والتشخيصات)
- tools
- hooks القديمة وhooks المكتوبة النوع
- القنوات
- المزودات
- معالجات Gateway RPC
- HTTP routes
- مسجلات CLI
- services الخلفية
- commands المملوكة لـ Plugin

ثم تقرأ ميزات الأساس من هذا السجل بدلًا من التحدث مباشرةً إلى وحدات Plugin.
وهذا يُبقي التحميل أحادي الاتجاه:

- وحدة Plugin -> التسجيل في السجل
- runtime الأساسي -> الاستهلاك من السجل

ويهم هذا الفصل من أجل قابلية الصيانة. إذ يعني أن معظم أسطح الأساس لا
تحتاج إلا إلى نقطة تكامل واحدة: "اقرأ السجل"، لا "ضع حالة خاصة لكل
وحدة Plugin".

## استدعاءات ربط المحادثات

يمكن لـ Plugins التي تربط محادثة ما أن تتفاعل عند حسم الموافقة.

استخدم `api.onConversationBindingResolved(...)` لتلقي استدعاء بعد
الموافقة على طلب الربط أو رفضه:

```ts
export default {
  id: "my-plugin",
  register(api) {
    api.onConversationBindingResolved(async (event) => {
      if (event.status === "approved") {
        // A binding now exists for this plugin + conversation.
        console.log(event.binding?.conversationId);
        return;
      }

      // The request was denied; clear any local pending state.
      console.log(event.request.conversation.conversationId);
    });
  },
};
```

حقول حمولة الاستدعاء:

- `status`: ‏`"approved"` أو `"denied"`
- `decision`: ‏`"allow-once"` أو `"allow-always"` أو `"deny"`
- `binding`: الربط الناتج للطلبات الموافق عليها
- `request`: ملخص الطلب الأصلي، وتلميح الفصل، ومعرّف المرسل، و
  بيانات تعريف المحادثة

هذا الاستدعاء خاص بالإشعارات فقط. ولا يغير من المسموح له بربط
محادثة، ويعمل بعد انتهاء معالجة الموافقة الأساسية.

## hooks runtime الخاصة بالمزود

تملك Plugins المزود الآن طبقتين:

- بيانات تعريف البيان: `providerAuthEnvVars` للبحث الرخيص عن مصادقة المزود عبر env
  قبل تحميل runtime، و`providerAuthAliases` لمتغيرات المزود التي تشترك في
  المصادقة، و`channelEnvVars` للبحث الرخيص عن إعداد/مصادقة القناة عبر env قبل
  تحميل runtime، بالإضافة إلى `providerAuthChoices` لتسميات Onboarding/خيارات المصادقة
  الرخيصة وبيانات تعريف أعلام CLI قبل تحميل runtime
- hooks وقت التهيئة: `catalog` / `discovery` القديم بالإضافة إلى `applyConfigDefaults`
- hooks runtime: ‏`normalizeModelId` و`normalizeTransport`،
  و`normalizeConfig`،
  و`applyNativeStreamingUsageCompat`، و`resolveConfigApiKey`،
  و`resolveSyntheticAuth`، و`resolveExternalAuthProfiles`،
  و`shouldDeferSyntheticProfileAuth`،
  و`resolveDynamicModel`، و`prepareDynamicModel`، و`normalizeResolvedModel`،
  و`contributeResolvedModelCompat`، و`capabilities`،
  و`normalizeToolSchemas`، و`inspectToolSchemas`،
  و`resolveReasoningOutputMode`، و`prepareExtraParams`، و`createStreamFn`،
  و`wrapStreamFn`، و`resolveTransportTurnState`،
  و`resolveWebSocketSessionPolicy`، و`formatApiKey`، و`refreshOAuth`،
  و`buildAuthDoctorHint`، و`matchesContextOverflowError`،
  و`classifyFailoverReason`، و`isCacheTtlEligible`،
  و`buildMissingAuthMessage`، و`suppressBuiltInModel`، و`augmentModelCatalog`،
  و`resolveThinkingProfile`، و`isBinaryThinking`، و`supportsXHighThinking`،
  و`resolveDefaultThinkingLevel`، و`isModernModelRef`، و`prepareRuntimeAuth`،
  و`resolveUsageAuth`، و`fetchUsageSnapshot`، و`createEmbeddingProvider`،
  و`buildReplayPolicy`،
  و`sanitizeReplayHistory`، و`validateReplayTurns`، و`onModelSelected`

ما يزال OpenClaw يمتلك حلقة الوكيل العامة، والرجوع الاحتياطي، ومعالجة transcript،
وسياسة الأدوات. وتمثل هذه hooks سطح التوسعة للسلوك الخاص بالمزود من دون
الحاجة إلى نقل استدلال مخصص بالكامل.

استخدم `providerAuthEnvVars` في البيان عندما يملك المزود بيانات اعتماد قائمة على env
ينبغي أن تراها مسارات المصادقة/الحالة/اختيار النموذج العامة من دون تحميل runtime الـ Plugin.
واستخدم `providerAuthAliases` في البيان عندما ينبغي أن يعيد أحد معرّفات المزود استخدام
متغيرات env الخاصة بمعرّف مزود آخر، وملفات تعريف المصادقة، والمصادقة المدعومة بالتهيئة،
وخيار Onboarding القائم على مفتاح API. واستخدم `providerAuthChoices` في البيان عندما
ينبغي لأسطح CLI الخاصة بـ Onboarding/خيارات المصادقة أن تعرف معرّف اختيار المزود،
وتسميات المجموعات، وربط المصادقة البسيط أحادي العلم، من دون تحميل runtime الخاص بالمزود.
وأبقِ `envVars` الخاصة بـ runtime المزود للتلميحات المواجهة للمشغل مثل تسميات Onboarding أو
متغيرات إعداد `client-id`/`client-secret` الخاصة بـ OAuth.

استخدم `channelEnvVars` في البيان عندما تكون للقناة مصادقة أو إعداد مدفوع بـ env
ينبغي أن تراه آليات shell-env fallback العامة، أو فحوصات التهيئة/الحالة، أو مطالبات الإعداد،
من دون تحميل runtime الخاص بالقناة.

يفحص `normalizeModelId` و`normalizeTransport` و`normalizeConfig` أولًا
Plugin المزود المطابقة، ثم ينتقل إلى Plugins المزود الأخرى القادرة على hook
حتى يغيّر أحدها معرّف النموذج أو النقل/التهيئة فعليًا. وهذا يُبقي
طبقات alias/compat الخاصة بالمزود عاملة من دون أن يضطر المستدعي إلى معرفة أي
Plugin مضمّنة تمتلك إعادة الكتابة. وإذا لم تُعِد أي hook خاصة بالمزود كتابة
إدخال تهيئة مدعوم من Google-family، فسيظل مُطبِّع تهيئة Google المضمّن يطبق
تنظيف التوافق هذا.

إذا احتاج المزود إلى بروتوكول wire مخصص بالكامل أو منفذ طلبات مخصص،
فهذه فئة مختلفة من التوسعة. وهذه hooks مخصصة لسلوك المزود الذي ما يزال
يعمل ضمن حلقة الاستدلال العادية في OpenClaw.

### مثال على مزود

```ts
api.registerProvider({
  id: "example-proxy",
  label: "Example Proxy",
  auth: [],
  catalog: {
    order: "simple",
    run: async (ctx) => {
      const apiKey = ctx.resolveProviderApiKey("example-proxy").apiKey;
      if (!apiKey) {
        return null;
      }
      return {
        provider: {
          baseUrl: "https://proxy.example.com/v1",
          apiKey,
          api: "openai-completions",
          models: [{ id: "auto", name: "Auto" }],
        },
      };
    },
  },
  resolveDynamicModel: (ctx) => ({
    id: ctx.modelId,
    name: ctx.modelId,
    provider: "example-proxy",
    api: "openai-completions",
    baseUrl: "https://proxy.example.com/v1",
    reasoning: false,
    input: ["text"],
    cost: { input: 0, output: 0, cacheRead: 0, cacheWrite: 0 },
    contextWindow: 128000,
    maxTokens: 8192,
  }),
  prepareRuntimeAuth: async (ctx) => {
    const exchanged = await exchangeToken(ctx.apiKey);
    return {
      apiKey: exchanged.token,
      baseUrl: exchanged.baseUrl,
      expiresAt: exchanged.expiresAt,
    };
  },
  resolveUsageAuth: async (ctx) => {
    const auth = await ctx.resolveOAuthToken();
    return auth ? { token: auth.token } : null;
  },
  fetchUsageSnapshot: async (ctx) => {
    return await fetchExampleProxyUsage(ctx.token, ctx.timeoutMs, ctx.fetchFn);
  },
});
```

### أمثلة مضمّنة

- يستخدم Anthropic كلاً من `resolveDynamicModel` و`capabilities` و`buildAuthDoctorHint`،
  و`resolveUsageAuth`، و`fetchUsageSnapshot`، و`isCacheTtlEligible`،
  و`resolveThinkingProfile`، و`applyConfigDefaults`، و`isModernModelRef`،
  و`wrapStreamFn` لأنه يمتلك التوافق المستقبلي لـ Claude 4.6،
  وتلميحات provider-family، وإرشادات إصلاح المصادقة، وتكامل نقطة نهاية الاستخدام،
  وأهلية prompt-cache، والإعدادات الافتراضية الواعية بالمصادقة، وسياسة Claude
  الافتراضية/التكيفية للتفكير، وتشكيل التدفق الخاص بـ Anthropic من أجل
  رؤوس beta، و`/fast` / `serviceTier`، و`context1m`.
- تبقى مساعدات التدفق الخاصة بـ Claude ضمن Anthropic في
  الواجهة العامة `api.ts` / `contract-api.ts` الخاصة بـ Plugin المضمّنة في الوقت الحالي.
  ويصدّر سطح الحزمة هذا `wrapAnthropicProviderStream`،
  و`resolveAnthropicBetas`، و`resolveAnthropicFastMode`، و`resolveAnthropicServiceTier`، والمستوى الأدنى
  من أدوات بناء أغلفة Anthropic بدلًا من توسيع SDK العامة حول
  قواعد رؤوس beta الخاصة بمزود واحد.
- يستخدم OpenAI كلاً من `resolveDynamicModel` و`normalizeResolvedModel` و
  `capabilities` بالإضافة إلى `buildMissingAuthMessage` و`suppressBuiltInModel`،
  و`augmentModelCatalog`، و`resolveThinkingProfile`، و`isModernModelRef`
  لأنه يمتلك التوافق المستقبلي لـ GPT-5.4، والتطبيع المباشر في OpenAI
  من `openai-completions` إلى `openai-responses`، وتلميحات المصادقة
  الواعية بـ Codex، وكبت Spark، وصفوف قوائم OpenAI الاصطناعية، وسياسة
  GPT-5 للتفكير / live-model؛ وتمتلك عائلة التدفق `openai-responses-defaults`
  أغلفة OpenAI Responses الأصلية المشتركة من أجل رؤوس الإسناد،
  و`/fast`/`serviceTier`، وكثافة النص، والبحث الأصلي على الويب في Codex،
  وتشكيل حمولة توافق الاستدلال، وإدارة سياق Responses.
- يستخدم OpenRouter كلاً من `catalog` بالإضافة إلى `resolveDynamicModel` و
  `prepareDynamicModel` لأن المزود تمريري وقد يكشف معرّفات
  نماذج جديدة قبل تحديث الفهرس الثابت في OpenClaw؛ كما يستخدم أيضًا
  `capabilities` و`wrapStreamFn` و`isCacheTtlEligible` لإبقاء
  رؤوس الطلبات الخاصة بالمزود، وبيانات تعريف التوجيه، وتصحيحات الاستدلال، و
  سياسة prompt-cache خارج الأساس. وتأتي سياسة إعادة التشغيل الخاصة به من
  عائلة `passthrough-gemini`، بينما تمتلك عائلة التدفق `openrouter-thinking`
  حقن الاستدلال عبر proxy وتجاوزات النماذج غير المدعومة / `auto`.
- يستخدم GitHub Copilot كلاً من `catalog` و`auth` و`resolveDynamicModel` و
  `capabilities` بالإضافة إلى `prepareRuntimeAuth` و`fetchUsageSnapshot` لأنه
  يحتاج إلى تسجيل دخول الجهاز مملوك للمزود، وسلوك رجوع احتياطي للنماذج، وخصائص شاذة في
  transcript الخاص بـ Claude، وتبادل رمز GitHub إلى رمز Copilot،
  ونقطة نهاية استخدام مملوكة للمزود.
- يستخدم OpenAI Codex كلاً من `catalog` و`resolveDynamicModel`،
  و`normalizeResolvedModel`، و`refreshOAuth`، و`augmentModelCatalog` بالإضافة
  إلى `prepareExtraParams` و`resolveUsageAuth` و`fetchUsageSnapshot` لأنه
  ما يزال يعمل على وسائل نقل OpenAI الأساسية لكنه يمتلك تطبيع

  وخيار النقل الافتراضي، وصفوف فهرس Codex الاصطناعية،
  وتكامل نقطة نهاية استخدام ChatGPT؛ وهو يشترك في عائلة التدفق

  ووضع مخرجات الاستدلال الموسوم، ومطابقة النماذج الحديثة، بينما
  تمتلك عائلة التدفق `google-thinking` تطبيع حمولة التفكير في Gemini؛
  كما يستخدم Gemini CLI OAuth أيضًا `formatApiKey` و`resolveUsageAuth` و

## مساعدات runtime

يمكن لـ Plugins الوصول إلى مساعدات أساسية محددة عبر `api.runtime`. بالنسبة إلى TTS:

```ts
const clip = await api.runtime.tts.textToSpeech({
  text: "Hello from OpenClaw",
  cfg: api.config,
});

const result = await api.runtime.tts.textToSpeechTelephony({
  text: "Hello from OpenClaw",
  cfg: api.config,
});

const voices = await api.runtime.tts.listVoices({
  provider: "elevenlabs",
  cfg: api.config,
});
```

ملاحظات:

- تعيد `textToSpeech` حمولة خرج TTS الأساسية العادية لأسطح الملفات/الملاحظات الصوتية.
- تستخدم تهيئة `messages.tts` الأساسية واختيار المزود.
- تعيد مخزنًا مؤقتًا للصوت PCM + معدل العينة. ويجب على Plugins إعادة أخذ العينات/الترميز من أجل المزودات.
- تكون `listVoices` اختيارية لكل مزود. استخدمها لمنتقيات الأصوات أو تدفقات الإعداد المملوكة للبائع.
- يمكن أن تتضمن قوائم الأصوات بيانات تعريف أغنى مثل locale والنوع الاجتماعي وعلامات الشخصية لمنتقيات واعية بالمزود.
- يدعم OpenAI وElevenLabs الاتصالات الهاتفية اليوم. أما Microsoft فلا يدعمها.

يمكن لـ Plugins أيضًا تسجيل مزودي كلام عبر `api.registerSpeechProvider(...)`.

```ts
api.registerSpeechProvider({
  id: "acme-speech",
  label: "Acme Speech",
  isConfigured: ({ config }) => Boolean(config.messages?.tts),
  synthesize: async (req) => {
    return {
      audioBuffer: Buffer.from([]),
      outputFormat: "mp3",
      fileExtension: ".mp3",
      voiceCompatible: false,
    };
  },
});
```

ملاحظات:

- أبقِ سياسة TTS والرجوع الاحتياطي وتسليم الردود في الأساس.
- استخدم مزودي الكلام لسلوك التركيب المملوك للبائع.
- يُطبَّع إدخال Microsoft `edge` القديم إلى معرّف المزود `microsoft`.
- نموذج الملكية المفضل موجّه نحو الشركة: يمكن لـ Plugin بائع واحدة أن تمتلك
  مزودي النص والكلام والصورة والوسائط المستقبلية مع إضافة OpenClaw لتلك
  العقود الخاصة بالإمكانات.

بالنسبة إلى فهم الصور/الصوت/الفيديو، تسجّل Plugins مزودًا واحدًا مكتوب النوع
لفهم الوسائط بدلًا من حقيبة عامة من المفتاح/القيمة:

```ts
api.registerMediaUnderstandingProvider({
  id: "google",
  capabilities: ["image", "audio", "video"],
  describeImage: async (req) => ({ text: "..." }),
  transcribeAudio: async (req) => ({ text: "..." }),
  describeVideo: async (req) => ({ text: "..." }),
});
```

ملاحظات:

- أبقِ التنسيق، والرجوع الاحتياطي، والتهيئة، وربط القنوات في الأساس.
- أبقِ سلوك البائع داخل Plugin المزود.
- يجب أن يظل التوسع الإضافي مكتوب النوع: أساليب اختيارية جديدة، وحقول
  نتائج اختيارية جديدة، وإمكانات اختيارية جديدة.
- يتبع توليد الفيديو بالفعل النمط نفسه:
  - يمتلك الأساس عقد الإمكانية ومساعد runtime
  - تسجّل Plugins البائع `api.registerVideoGenerationProvider(...)`
  - وتستهلك Plugins الميزات/القنوات `api.runtime.videoGeneration.*`

وبالنسبة إلى مساعدات runtime الخاصة بفهم الوسائط، يمكن لـ Plugins استدعاء:

```ts
const image = await api.runtime.mediaUnderstanding.describeImageFile({
  filePath: "/tmp/inbound-photo.jpg",
  cfg: api.config,
  agentDir: "/tmp/agent",
});

const video = await api.runtime.mediaUnderstanding.describeVideoFile({
  filePath: "/tmp/inbound-video.mp4",
  cfg: api.config,
});
```

وبالنسبة إلى نسخ الصوت، يمكن لـ Plugins استخدام runtime الخاصة
بفهم الوسائط أو الاسم المستعار الأقدم لـ STT:

```ts
const { text } = await api.runtime.mediaUnderstanding.transcribeAudioFile({
  filePath: "/tmp/inbound-audio.ogg",
  cfg: api.config,
  // Optional when MIME cannot be inferred reliably:
  mime: "audio/ogg",
});
```

ملاحظات:

- يُعد `api.runtime.mediaUnderstanding.*` السطح المشترك المفضل من أجل
  فهم الصور/الصوت/الفيديو.
- يستخدم تهيئة الصوت الأساسية الخاصة بفهم الوسائط (`tools.media.audio`) وترتيب الرجوع الاحتياطي للمزود.
- يعيد `{ text: undefined }` عندما لا يتم إنتاج ناتج نسخ (مثلًا عند تخطي الإدخال/عدم دعمه).
- يبقى `api.runtime.stt.transcribeAudioFile(...)` اسمًا مستعارًا للتوافق.

يمكن لـ Plugins أيضًا تشغيل عمليات subagent في الخلفية عبر `api.runtime.subagent`:

```ts
const result = await api.runtime.subagent.run({
  sessionKey: "agent:main:subagent:search-helper",
  message: "Expand this query into focused follow-up searches.",
  provider: "openai",
  model: "gpt-4.1-mini",
  deliver: false,
});
```

ملاحظات:

- إن `provider` و`model` تجاوزات لكل تشغيل بشكل اختياري، وليسا تغييرات دائمة للجلسة.
- لا يلتزم OpenClaw بحقول التجاوز هذه إلا للمستدعين الموثوقين.
- بالنسبة إلى عمليات الرجوع الاحتياطي المملوكة لـ Plugin، يجب على المشغلين التفعيل صراحةً عبر `plugins.entries.<id>.subagent.allowModelOverride: true`.
- استخدم `plugins.entries.<id>.subagent.allowedModels` لتقييد Plugins الموثوقة بأهداف `provider/model` القياسية المحددة، أو `"*"` للسماح الصريح بأي هدف.
- ما تزال عمليات subagent الخاصة بـ Plugin غير الموثوقة تعمل، لكن طلبات التجاوز تُرفض بدلًا من الرجوع الاحتياطي بصمت.

وبالنسبة إلى البحث على الويب، يمكن لـ Plugins استهلاك مساعد runtime المشترك بدلًا من
الوصول المباشر إلى ربط أداة الوكيل:

```ts
const providers = api.runtime.webSearch.listProviders({
  config: api.config,
});

const result = await api.runtime.webSearch.search({
  config: api.config,
  args: {
    query: "OpenClaw plugin runtime helpers",
    count: 5,
  },
});
```

يمكن لـ Plugins أيضًا تسجيل مزودي بحث على الويب عبر
`api.registerWebSearchProvider(...)`.

ملاحظات:

- أبقِ اختيار المزود، وحل بيانات الاعتماد، ودلالات الطلب المشتركة في الأساس.
- استخدم مزودي البحث على الويب لوسائل النقل الخاصة بالبحث المملوك للبائع.
- يُعد `api.runtime.webSearch.*` السطح المشترك المفضل لـ Plugins الميزات/القنوات التي تحتاج إلى سلوك البحث دون الاعتماد على غلاف أداة الوكيل.

const result = await api.runtime.imageGeneration.generate({

## HTTP routes الخاصة بـ Gateway

يمكن لـ Plugins كشف نقاط نهاية HTTP عبر `api.registerHttpRoute(...)`.

```ts
api.registerHttpRoute({
  path: "/acme/webhook",
  auth: "plugin",
  match: "exact",
  handler: async (_req, res) => {
    res.statusCode = 200;
    res.end("ok");
    return true;
  },
});
```

حقول route:

- `path`: مسار route تحت خادم HTTP الخاص بـ Gateway.
- `auth`: مطلوب. استخدم `"gateway"` لطلب مصادقة Gateway العادية، أو `"plugin"` من أجل مصادقة/تحقق Webhook تُديرها Plugin.
- `match`: اختياري. `"exact"` (الافتراضي) أو `"prefix"`.
- `replaceExisting`: اختياري. يسمح لـ Plugin نفسها باستبدال تسجيل route موجود خاص بها.
- `handler`: أعد `true` عندما تكون route قد عالجت الطلب.

ملاحظات:

- تمت إزالة `api.registerHttpHandler(...)` وستؤدي إلى خطأ في تحميل Plugin. استخدم `api.registerHttpRoute(...)` بدلًا منها.
- يجب أن تعلن routes الخاصة بـ Plugin عن `auth` صراحةً.
- تُرفض تعارضات `path + match` التامة ما لم تكن `replaceExisting: true`، ولا يمكن لـ Plugin واحدة أن تستبدل route خاصة بـ Plugin أخرى.
- تُرفض routes المتداخلة ذات مستويات `auth` المختلفة. وأبقِ سلاسل fallthrough الخاصة بـ `exact`/`prefix` على مستوى auth نفسه فقط.
- لا تتلقى routes ذات `auth: "plugin"` **نطاقات runtime للمشغّل** تلقائيًا. فهي مخصصة لـ Webhook/التحقق من التوقيع الذي تديره Plugin، لا لاستدعاءات مساعدات Gateway ذات الامتيازات.
- تعمل routes ذات `auth: "gateway"` داخل نطاق runtime لطلب Gateway، لكن هذا النطاق محافظ عمدًا:
  - إن مصادقة bearer بالمفتاح السري المشترك (`gateway.auth.mode = "token"` / `"password"`) تُبقي نطاقات runtime الخاصة بـ plugin-route مثبتة على `operator.write`، حتى إذا أرسل المستدعي `x-openclaw-scopes`
  - أوضاع HTTP الموثوقة الحاملة للهوية (مثل `trusted-proxy` أو `gateway.auth.mode = "none"` على ingress خاص) لا تلتزم بـ `x-openclaw-scopes` إلا عندما يكون الرأس موجودًا صراحةً
  - إذا كان `x-openclaw-scopes` غائبًا في طلبات plugin-route الحاملة للهوية هذه، فإن نطاق runtime يعود إلى `operator.write`
- القاعدة العملية: لا تفترض أن plugin route المصادق عليها عبر gateway هي سطح إدارة ضمني. وإذا احتاجت route الخاصة بك إلى سلوك إداري فقط، فاطلب وضع مصادقة حاملًا للهوية ووثّق عقد الرأس الصريح `x-openclaw-scopes`.

  توافقية لـ Plugins الأقدم. ويجب على الشيفرة الجديدة استيراد البدائيات العامة الأضيق بدلًا منه، كما يجب ألا تضيف شيفرة المستودع استيرادات جديدة من
  هذه الطبقة.
- تظل الأجزاء الداخلية للـ extensions المضمّنة خاصة. ويجب على Plugins الخارجية استخدام
  المسارات الفرعية `openclaw/plugin-sdk/*` فقط. ويمكن لشيفرة OpenClaw الأساسية/الاختبارية استخدام
  نقاط الإدخال العامة الخاصة بالمستودع تحت جذر حزمة Plugin مثل `index.js` و`api.js`،
  و`runtime-api.js`، و`setup-entry.js`، وملفات ضيقة النطاق مثل
  `login-qr-api.js`. ولا تستورد أبدًا `src/*` من حزمة Plugin من داخل الأساس أو من
  extension أخرى.
- تقسيم نقطة إدخال المستودع:
  `<plugin-package-root>/api.js` هو حزمة المساعدات/الأنواع،
  و`<plugin-package-root>/runtime-api.js` هو الحزمة الخاصة بـ runtime فقط،
  و`<plugin-package-root>/index.js` هي نقطة إدخال Plugin المضمّنة،
  و`<plugin-package-root>/setup-entry.js` هي نقطة إدخال Plugin الخاصة بالإعداد.
- أمثلة مزودات مضمّنة حالية:
  - يستخدم Anthropic ‏`api.js` / `contract-api.js` لمساعدات تدفق Claude مثل
    `wrapAnthropicProviderStream`، ومساعدات رؤوس beta، وتحليل `service_tier`.
  - يستخدم OpenAI ‏`api.js` لبناة المزود، ومساعدات النموذج الافتراضي، وبناة مزودات realtime.
  - يستخدم OpenRouter ‏`api.js` لباني المزود الخاص به بالإضافة إلى مساعدات
    Onboarding/التهيئة، بينما يمكن لـ `register.runtime.js` أن يعيد تصدير مساعدات
    `plugin-sdk/provider-stream` العامة للاستخدام المحلي داخل المستودع.
- تفضّل نقاط الإدخال العامة المحمّلة عبر Facade لقطة تهيئة runtime النشطة
  عندما تكون موجودة، ثم تعود إلى ملف التهيئة المحلول على القرص عندما لا يكون
  OpenClaw قد بدأ بعد في تقديم لقطة runtime.
- تظل البدائيات العامة المشتركة هي عقد SDK العامة المفضل. وما تزال مجموعة توافقية محجوزة صغيرة من الواجهات المساعدة ذات العلامة الخاصة بالقنوات المضمّنة موجودة.
  تعامل مع هذه على أنها واجهات خاصة بصيانة/توافق الحزم المضمّنة، لا كأهداف استيراد جديدة لجهات خارجية؛ إذ يجب أن تظل العقود الجديدة العابرة للقنوات تُضاف إلى
  المسارات الفرعية العامة `plugin-sdk/*` أو إلى الحزم المحلية الخاصة بـ Plugin مثل `api.js` /
  `runtime-api.js`.

ملاحظة توافق:

- تجنب الحزمة الجذرية `openclaw/plugin-sdk` في الشيفرة الجديدة.
- فضّل البدائيات المستقرة والضيقة أولًا. فالمسارات الفرعية الأحدث الخاصة بـ setup/pairing/reply/
  feedback/contract/inbound/threading/command/secret-input/webhook/infra/
  allowlist/status/message-tool هي العقد المقصود لأعمال Plugins الجديدة
  المضمّنة والخارجية.
  وينتمي تحليل/مطابقة الهدف إلى `openclaw/plugin-sdk/channel-targets`.
  أما بوابات إجراءات الرسائل ومساعدات معرّف رسالة التفاعل فتنتمي إلى
  `openclaw/plugin-sdk/channel-actions`.
- إن الحزم المساعدة الخاصة بالـ extensions المضمّنة ليست مستقرة افتراضيًا. وإذا
  كانت المساعدة مطلوبة فقط من قِبل extension مضمّنة، فأبقها خلف
  الواجهة المحلية `api.js` أو `runtime-api.js` الخاصة بالـ extension بدلًا من ترقيتها إلى
  `openclaw/plugin-sdk/<extension>`.
- يجب أن تكون الواجهات المساعدة المشتركة الجديدة عامة، لا موسومة باسم قناة. ويجب أن يكون
  تحليل الهدف المشترك على `openclaw/plugin-sdk/channel-targets`؛ بينما تبقى
  الأجزاء الداخلية الخاصة بالقناة خلف الواجهة المحلية `api.js` أو `runtime-api.js`
  الخاصة بالـ Plugin المالكة.
- توجد المسارات الفرعية الخاصة بالإمكانات مثل `image-generation`،
  و`media-understanding`، و`speech` لأن Plugins المضمّنة/الأصلية تستخدمها
  اليوم. ووجودها لا يعني بحد ذاته أن كل مساعد مُصدَّر هو عقد خارجي مجمّد على المدى الطويل.

## مخططات أداة الرسائل

يجب أن تمتلك Plugins مساهمات schema الخاصة بـ `describeMessageTool(...)` المحددة بالقناة
بالنسبة إلى البدائيات غير الرسائلية مثل التفاعلات وعمليات القراءة والاستطلاعات.
وينبغي لعرض الإرسال المشترك أن يستخدم عقد `MessagePresentation` العام
بدلًا من حقول الأزرار أو المكونات أو الكتل أو البطاقات الأصلية للمزود.
راجع [عرض الرسائل](/ar/plugins/message-presentation) لمعرفة العقد،
وقواعد الرجوع الاحتياطي، وربط المزود، وقائمة التحقق الخاصة بمؤلفي Plugins.

تُعلن Plugins القادرة على الإرسال ما الذي يمكنها عرضه من خلال إمكانات الرسائل:

- `presentation` لكتل العرض الدلالية (`text` و`context` و`divider` و`buttons` و`select`)
- `delivery-pin` لطلبات التسليم المثبتة

ويقرر الأساس ما إذا كان سيعرض العرض بشكل أصلي أم سيخفضه إلى نص.
ولا تكشف منافذ هروب UI الأصلية للمزود من أداة الرسائل العامة.
وتظل مساعدات SDK المهملة الخاصة بالمخططات الأصلية القديمة مُصدَّرة لصالح
Plugins الجهات الخارجية الحالية، لكن Plugins الجديدة يجب ألا تستخدمها.

## حل أهداف القنوات

يجب أن تمتلك Plugins القنوات دلالات الأهداف الخاصة بالقناة. وأبقِ
مضيف الإرسال الخارجي عامًا، واستخدم سطح مهايئ المراسلة لقواعد المزود:

- `messaging.inferTargetChatType({ to })` يقرر ما إذا كان الهدف المُطبّع
  ينبغي اعتباره `direct` أو `group` أو `channel` قبل البحث في الدليل.
- `messaging.targetResolver.looksLikeId(raw, normalized)` يخبر الأساس ما إذا كان
  الإدخال يجب أن يتجاوز مباشرة إلى الحل الشبيه بالمعرّفات بدلًا من البحث في الدليل.
- `messaging.targetResolver.resolveTarget(...)` هو الرجوع الاحتياطي الخاص بـ Plugin عندما
  يحتاج الأساس إلى حل نهائي مملوك للمزود بعد التطبيع أو بعد فشل
  البحث في الدليل.
- `messaging.resolveOutboundSessionRoute(...)` يمتلك بناء مسار
  الجلسة الصادر الخاص بالمزود بمجرد حل الهدف.

التقسيم الموصى به:

- استخدم `inferTargetChatType` لقرارات الفئات التي ينبغي أن تحدث قبل
  البحث في النظراء/المجموعات.
- استخدم `looksLikeId` لفحوصات "عامل هذا على أنه معرّف هدف صريح/أصلي".
- استخدم `resolveTarget` للرجوع الاحتياطي في التطبيع الخاص بالمزود، وليس
  لبحث دليل واسع.
- أبقِ المعرّفات الأصلية للمزود مثل chat ids وthread ids وJIDs وhandles وroom
  ids داخل قيم `target` أو المعلمات الخاصة بالمزود، لا في حقول SDK العامة.

## الأدلة المدعومة بالتهيئة

يجب أن تُبقي Plugins التي تستمد إدخالات الدليل من التهيئة هذا المنطق داخل
Plugin وأن تعيد استخدام المساعدات المشتركة من
`openclaw/plugin-sdk/directory-runtime`.

استخدم هذا عندما تحتاج القناة إلى نظراء/مجموعات مدعومة بالتهيئة مثل:

- نظراء الرسائل الخاصة المدفوعين بقائمة السماح
- خرائط القناة/المجموعة المهيأة
- بدائل الدليل الثابتة ذات النطاق المحاسبي

أما المساعدات المشتركة في `directory-runtime` فلا تتولى إلا العمليات العامة:

- تصفية الاستعلام
- تطبيق الحدود
- مساعدات إزالة التكرار/التطبيع
- بناء `ChannelDirectoryEntry[]`

ويجب أن يبقى فحص الحسابات الخاص بالقناة وتطبيع المعرّفات داخل
تنفيذ Plugin.

## فهارس المزودات

يمكن لـ Plugins المزود تعريف فهارس نماذج للاستدلال عبر
`registerProvider({ catalog: { run(...) { ... } } })`.

يعيد `catalog.run(...)` الشكل نفسه الذي يكتبه OpenClaw داخل
`models.providers`:

- `{ provider }` لإدخال مزود واحد
- `{ providers }` لعدة إدخالات مزود

استخدم `catalog` عندما تمتلك Plugin معرّفات نماذج خاصة بالمزود، أو إعدادات
افتراضية لعنوان URL الأساسي، أو بيانات تعريف نماذج محمية بالمصادقة.

يتحكم `catalog.order` في وقت دمج فهرس Plugin بالنسبة إلى
المزودات الضمنية المدمجة داخل OpenClaw:

- `simple`: مزودات عادية تعتمد على مفتاح API أو env
- `profile`: مزودات تظهر عند وجود ملفات تعريف للمصادقة
- `paired`: مزودات تُركّب عدة إدخالات مزود مترابطة
- `late`: تمريرة أخيرة بعد المزودات الضمنية الأخرى

وتفوز المزودات اللاحقة عند تعارض المفاتيح، لذا يمكن لـ Plugins
أن تتعمد تجاوز إدخال مزود مدمج يحمل معرّف المزود نفسه.

التوافق:

- ما يزال `discovery` يعمل بوصفه اسمًا مستعارًا قديمًا
- إذا سُجّل كلٌّ من `catalog` و`discovery`، يستخدم OpenClaw `catalog`

## فحص القنوات للقراءة فقط

إذا كانت Plugin الخاصة بك تسجّل قناة، ففضّل تنفيذ
`plugin.config.inspectAccount(cfg, accountId)` إلى جانب `resolveAccount(...)`.

السبب:

- إن `resolveAccount(...)` هو مسار runtime. ويُسمح له بأن يفترض أن بيانات الاعتماد
  قد جُسِّدت بالكامل، ويمكنه الإخفاق سريعًا عند غياب الأسرار المطلوبة.
- أما مسارات الأوامر للقراءة فقط مثل `openclaw status` و`openclaw status --all`،
  و`openclaw channels status`، و`openclaw channels resolve`، وتدفقات doctor/config
  repair، فلا ينبغي أن تحتاج إلى تجسيد بيانات اعتماد runtime فقط
  لوصف التهيئة.

السلوك الموصى به لـ `inspectAccount(...)`:

- أعد حالة الحساب الوصفية فقط.
- حافظ على `enabled` و`configured`.
- ضمّن حقول مصدر/حالة بيانات الاعتماد عند الصلة، مثل:
  - `tokenSource`، `tokenStatus`
  - `botTokenSource`، `botTokenStatus`
  - `appTokenSource`، `appTokenStatus`
  - `signingSecretSource`، `signingSecretStatus`
- لست بحاجة إلى إعادة قيم tokens الخام فقط للإبلاغ عن
  التوافر للقراءة فقط. فإعادة `tokenStatus: "available"` (وحقل المصدر المطابق
  له) تكفي لأوامر الحالة.
- استخدم `configured_unavailable` عندما تكون بيانات الاعتماد مهيأة عبر SecretRef لكنها
  غير متاحة في مسار الأمر الحالي.

ويتيح هذا لأوامر القراءة فقط أن تُبلغ "مهيأ ولكنه غير متاح في مسار هذا الأمر"
بدلًا من التعطل أو الإبلاغ الخاطئ عن الحساب باعتباره غير مهيأ.

## حِزم Package

يمكن أن يتضمن دليل Plugin ملف `package.json` يحتوي على `openclaw.extensions`:

```json
{
  "name": "my-pack",
  "openclaw": {
    "extensions": ["./src/safety.ts", "./src/tools.ts"],
    "setupEntry": "./src/setup-entry.ts"
  }
}
```

يصبح كل إدخال Plugin. وإذا أدرجت الحزمة عدة extensions، فإن معرّف Plugin
يصبح `name/<fileBase>`.

إذا كانت Plugin الخاصة بك تستورد تبعيات npm، فثبّتها داخل ذلك الدليل حتى
تتوفر `node_modules` (`npm install` / `pnpm install`).

حاجز أمان: يجب أن يبقى كل إدخال في `openclaw.extensions` داخل دليل Plugin
بعد حل الروابط الرمزية. وتُرفض الإدخالات التي تخرج من دليل الحزمة.

ملاحظة أمان: يقوم `openclaw plugins install` بتثبيت تبعيات Plugin باستخدام
`npm install --omit=dev --ignore-scripts` (من دون lifecycle scripts، ومن دون تبعيات dev في runtime). لذا أبقِ أشجار تبعيات Plugin
"JS/TS خالصة" وتجنب الحزم التي تتطلب بناء `postinstall`.

اختياري: يمكن لـ `openclaw.setupEntry` أن يشير إلى وحدة خفيفة خاصة بالإعداد فقط.
وعندما يحتاج OpenClaw إلى أسطح إعداد لقناة Plugin معطلة، أو
عندما تكون قناة Plugin مفعلة لكنها ما تزال غير مهيأة، فإنه يحمّل `setupEntry`
بدلًا من إدخال Plugin الكامل. وهذا يُبقي بدء التشغيل والإعداد أخف
عندما يكون إدخال Plugin الرئيسي لديك يربط أيضًا tools أو hooks أو شيفرة أخرى خاصة بـ runtime فقط.

اختياري: يمكن لـ `openclaw.startup.deferConfiguredChannelFullLoadUntilAfterListen`
أن يُدخل قناة Plugin في مسار `setupEntry` نفسه أثناء مرحلة بدء التشغيل السابقة للاستماع في Gateway،
حتى عندما تكون القناة مهيأة بالفعل.

استخدم هذا فقط عندما يغطي `setupEntry` كامل سطح بدء التشغيل الذي يجب أن يوجد
قبل أن تبدأ Gateway في الاستماع. ومن الناحية العملية، يعني ذلك أن إدخال الإعداد
يجب أن يسجّل كل إمكانة مملوكة للقناة يعتمد عليها بدء التشغيل، مثل:

- تسجيل القناة نفسها
- أي HTTP routes يجب أن تكون متاحة قبل أن تبدأ Gateway في الاستماع
- أي Gateway methods أو tools أو services يجب أن توجد خلال تلك النافذة نفسها

إذا كان إدخالك الكامل ما يزال يمتلك أي إمكانة مطلوبة لبدء التشغيل، فلا تفعّل
هذا الخيار. وأبقِ Plugin على السلوك الافتراضي ودع OpenClaw يحمّل
الإدخال الكامل أثناء بدء التشغيل.

يمكن للقنوات المضمّنة أيضًا نشر مساعدات سطح عقد خاصة بالإعداد فقط بحيث يستطيع الأساس
استشارتها قبل تحميل runtime القناة الكامل. وسطح ترقية الإعداد الحالي هو:

- `singleAccountKeysToMove`
- `namedAccountPromotionKeys`
- `resolveSingleAccountPromotionTarget(...)`

ويستخدم الأساس هذا السطح عندما يحتاج إلى ترقية تهيئة قناة قديمة
بحساب واحد إلى `channels.<id>.accounts.*` من دون تحميل إدخال Plugin الكامل.
وتُعد Matrix المثال المضمّن الحالي: فهي تنقل فقط مفاتيح المصادقة/التمهيد
إلى حساب مسمى مُرقّى عندما تكون الحسابات المسماة موجودة بالفعل، ويمكنها
الحفاظ على مفتاح حساب افتراضي غير قياسي مهيأ بدلًا من إنشاء
`accounts.default` دائمًا.

وتُبقي مهايئات تصحيح الإعداد هذه اكتشاف سطح العقد الخاص بالحزم المضمّنة كسولًا. إذ يبقى
وقت الاستيراد خفيفًا؛ ولا يُحمَّل سطح الترقية إلا عند أول استخدام بدلًا من
إعادة الدخول في بدء تشغيل القناة المضمّنة عند استيراد الوحدة.

وعندما تتضمن أسطح بدء التشغيل تلك أساليب Gateway RPC، فأبقها على
بادئة خاصة بـ Plugin. وتظل مساحات أسماء الإدارة الأساسية (`config.*`،
و`exec.approvals.*`، و`wizard.*`، و`update.*`) محجوزة وتُحل دائمًا
إلى `operator.admin`، حتى لو طلبت Plugin نطاقًا أضيق.

مثال:

```json
{
  "name": "@scope/my-channel",
  "openclaw": {
    "extensions": ["./index.ts"],
    "setupEntry": "./setup-entry.ts",
    "startup": {
      "deferConfiguredChannelFullLoadUntilAfterListen": true
    }
  }
}
```

### بيانات تعريف فهرس القنوات

يمكن لـ Plugins القنوات الإعلان عن بيانات تعريف الإعداد/الاكتشاف عبر `openclaw.channel` و
تلميحات التثبيت عبر `openclaw.install`. وهذا يُبقي بيانات الفهرس الأساسية خالية من البيانات.

مثال:

```json
{
  "name": "@openclaw/nextcloud-talk",
  "openclaw": {
    "extensions": ["./index.ts"],
    "channel": {
      "id": "nextcloud-talk",
      "label": "Nextcloud Talk",
      "selectionLabel": "Nextcloud Talk (self-hosted)",
      "docsPath": "/channels/nextcloud-talk",
      "docsLabel": "nextcloud-talk",
      "blurb": "دردشة مستضافة ذاتيًا عبر Nextcloud Talk webhook bots.",
      "order": 65,
      "aliases": ["nc-talk", "nc"]
    },
    "install": {
      "npmSpec": "@openclaw/nextcloud-talk",
      "localPath": "<bundled-plugin-local-path>",
      "defaultChoice": "npm"
    }
  }
}
```

حقول `openclaw.channel` المفيدة خارج المثال الأدنى:

- `detailLabel`: تسمية ثانوية لأسطح الفهرس/الحالة الأكثر غنى
- `docsLabel`: تجاوز نص الرابط الخاص برابط الوثائق
- `preferOver`: معرّفات Plugin/قنوات ذات أولوية أقل يجب أن يتفوق عليها هذا الإدخال في الفهرس
- `selectionDocsPrefix` و`selectionDocsOmitLabel` و`selectionExtras`: عناصر تحكم في النص الخاص بسطح الاختيار
- `markdownCapable`: يعلّم القناة بأنها قادرة على Markdown لقرارات التنسيق الصادر
- `exposure.configured`: إخفاء القناة من أسطح سرد القنوات المهيأة عندما تُضبط على `false`
- `exposure.setup`: إخفاء القناة من أدوات اختيار الإعداد/التهيئة التفاعلية عندما تُضبط على `false`
- `exposure.docs`: تعليم القناة على أنها داخلية/خاصة لأسطح التنقل في الوثائق
- `showConfigured` / `showInSetup`: أسماء مستعارة قديمة ما تزال مقبولة للتوافق؛ ويفضّل `exposure`
- `quickstartAllowFrom`: إدخال القناة في تدفق `allowFrom` القياسي في البدء السريع
- `forceAccountBinding`: فرض ربط حساب صريح حتى عند وجود حساب واحد فقط
- `preferSessionLookupForAnnounceTarget`: تفضيل البحث في الجلسة عند حل أهداف الإعلان

يمكن لـ OpenClaw أيضًا دمج **فهارس القنوات الخارجية** (مثلًا، تصدير سجل
MPM). ضع ملف JSON في أحد المواقع التالية:

- `~/.openclaw/mpm/plugins.json`
- `~/.openclaw/mpm/catalog.json`
- `~/.openclaw/plugins/catalog.json`

أو وجّه `OPENCLAW_PLUGIN_CATALOG_PATHS` (أو `OPENCLAW_MPM_CATALOG_PATHS`) إلى
ملف JSON واحد أو أكثر (مفصولة بفواصل/فواصل منقوطة/على نمط `PATH`). ويجب أن
يحتوي كل ملف على `{ "entries": [ { "name": "@scope/pkg", "openclaw": { "channel": {...}, "install": {...} } } ] }`. كما يقبل المحلل
`"packages"` أو `"plugins"` بوصفهما اسمين مستعارين قديمين للمفتاح `"entries"`.

## Plugins محرك السياق

تمتلك Plugins محرك السياق تنسيق سياق الجلسة من أجل الإدخال والتجميع
وCompaction. سجّلها من Plugin الخاصة بك عبر
`api.registerContextEngine(id, factory)`، ثم اختر المحرك النشط بواسطة
`plugins.slots.contextEngine`.

استخدم هذا عندما تحتاج Plugin الخاصة بك إلى استبدال مسار السياق الافتراضي
أو توسيعه بدلًا من مجرد إضافة بحث في الذاكرة أو hooks.

```ts
import { buildMemorySystemPromptAddition } from "openclaw/plugin-sdk/core";

export default function (api) {
  api.registerContextEngine("lossless-claw", () => ({
    info: { id: "lossless-claw", name: "Lossless Claw", ownsCompaction: true },
    async ingest() {
      return { ingested: true };
    },
    async assemble({ messages, availableTools, citationsMode }) {
      return {
        messages,
        estimatedTokens: 0,
        systemPromptAddition: buildMemorySystemPromptAddition({
          availableTools: availableTools ?? new Set(),
          citationsMode,
        }),
      };
    },
    async compact() {
      return { ok: true, compacted: false };
    },
  }));
}
```

إذا كان محركك **لا** يمتلك خوارزمية Compaction، فأبقِ `compact()`
منفذة وفوّضها صراحةً:

```ts
import {
  buildMemorySystemPromptAddition,
  delegateCompactionToRuntime,
} from "openclaw/plugin-sdk/core";

export default function (api) {
  api.registerContextEngine("my-memory-engine", () => ({
    info: {
      id: "my-memory-engine",
      name: "My Memory Engine",
      ownsCompaction: false,
    },
    async ingest() {
      return { ingested: true };
    },
    async assemble({ messages, availableTools, citationsMode }) {
      return {
        messages,
        estimatedTokens: 0,
        systemPromptAddition: buildMemorySystemPromptAddition({
          availableTools: availableTools ?? new Set(),
          citationsMode,
        }),
      };
    },
    async compact(params) {
      return await delegateCompactionToRuntime(params);
    },
  }));
}
```

## إضافة إمكانية جديدة

عندما تحتاج Plugin إلى سلوك لا يتوافق مع API الحالية، فلا تتجاوز
نظام Plugins عبر وصول خاص. بل أضف الإمكانية المفقودة.

التسلسل الموصى به:

1. تعريف العقد الأساسي
   قرر ما السلوك المشترك الذي ينبغي أن يمتلكه الأساس: السياسة، والرجوع الاحتياطي، ودمج التهيئة،
   ودورة الحياة، والدلالات المواجهة للقناة، وشكل مساعد runtime.
2. إضافة أسطح تسجيل/runtime مكتوبة النوع للـ Plugin
   وسّع `OpenClawPluginApi` و/أو `api.runtime` بأصغر
   سطح إمكانات مكتوب النوع ومفيد.
3. ربط مستهلكي الأساس + القنوات/الميزات
   ينبغي أن تستهلك القنوات وPlugins الميزات الإمكانية الجديدة عبر الأساس،
   لا عبر استيراد تنفيذ بائع مباشرةً.
4. تسجيل تنفيذات البائع
   ثم تسجّل Plugins البائع واجهاتها الخلفية مقابل الإمكانية.
5. إضافة تغطية للعقد
   أضف اختبارات حتى تظل الملكية وشكل التسجيل صريحين مع الزمن.

هذه هي الطريقة التي يبقى بها OpenClaw ذا رأي واضح دون أن يصبح مرمّزًا
بشكل صلب وفق رؤية مزود واحد. راجع [دليل الإمكانات العملي](/ar/plugins/architecture)
للحصول على قائمة ملفات عملية ومثال مكتمل.

### قائمة التحقق الخاصة بالإمكانية

عندما تضيف إمكانية جديدة، فعادةً ما يجب أن يلمس التنفيذ هذه
الأسطح معًا:

- أنواع العقود الأساسية في `src/<capability>/types.ts`
- runner/runtime helper الأساسي في `src/<capability>/runtime.ts`
- سطح تسجيل Plugin API في `src/plugins/types.ts`
- ربط سجل Plugins في `src/plugins/registry.ts`
- كشف runtime الخاصة بـ Plugin في `src/plugins/runtime/*` عندما تحتاج
  Plugins الميزات/القنوات إلى استهلاكها
- مساعدات الالتقاط/الاختبار في `src/test-utils/plugin-registration.ts`
- تأكيدات الملكية/العقود في `src/plugins/contracts/registry.ts`
- وثائق المشغل/Plugin في `docs/`

إذا كان أحد هذه الأسطح مفقودًا، فعادةً ما تكون هذه علامة على أن الإمكانية
لم تُدمج بالكامل بعد.

### قالب الإمكانية

النمط الأدنى:

```ts
// core contract
export type VideoGenerationProviderPlugin = {
  id: string;
  label: string;
  generateVideo: (req: VideoGenerationRequest) => Promise<VideoGenerationResult>;
};

// plugin API
api.registerVideoGenerationProvider({
  id: "openai",
  label: "OpenAI",
  async generateVideo(req) {
    return await generateOpenAiVideo(req);
  },
});

// shared runtime helper for feature/channel plugins
const clip = await api.runtime.videoGeneration.generate({
  prompt: "Show the robot walking through the lab.",
  cfg,
});
```

نمط اختبار العقد:

```ts
expect(findVideoGenerationProviderIdsForPlugin("openai")).toEqual(["openai"]);
```

وهذا يُبقي القاعدة بسيطة:

- يمتلك الأساس عقد الإمكانية + التنسيق
- تمتلك Plugins البائع التنفيذات الخاصة بالبائع
- تستهلك Plugins الميزات/القنوات مساعدات runtime
- تُبقي اختبارات العقد الملكية صريحة

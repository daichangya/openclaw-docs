# مستويات التفكير (توجيهات `/think`)

## ما الذي يفعله

- توجيه مضمّن في أي نص وارد: `/t <level>` أو `/think:<level>` أو `/thinking <level>`.
- المستويات (الأسماء البديلة): `off | minimal | low | medium | high | xhigh | adaptive | max`
  - minimal ← "think"
  - low ← "think hard"
  - medium ← "think harder"
  - high ← "ultrathink" ‏(أقصى ميزانية)
  - xhigh ← "ultrathink+" ‏(جهد GPT-5.2 + نماذج Codex وAnthropic Claude Opus 4.7)
  - adaptive ← تفكير تكيفي يديره المزوّد (مدعوم في Claude 4.6 على Anthropic/Bedrock وAnthropic Claude Opus 4.7)
  - max ← أقصى استدلال لدى المزوّد (حاليًا Anthropic Claude Opus 4.7)
  - يتم ربط `x-high` و`x_high` و`extra-high` و`extra high` و`extra_high` بـ `xhigh`.
  - يتم ربط `highest` بـ `high`.
- ملاحظات المزوّد:
  - تعتمد قوائم ومحدّدات التفكير على ملف تعريف المزوّد. وتعلن Plugins الخاصة بالمزوّد عن مجموعة المستويات الدقيقة للنموذج المحدد، بما في ذلك تسميات مثل `on` الثنائية.
  - لا يتم الإعلان عن `adaptive` و`xhigh` و`max` إلا لملفات تعريف المزوّد/النموذج التي تدعمها. وتُرفض التوجيهات المكتوبة لمستويات غير مدعومة مع عرض الخيارات الصالحة لذلك النموذج.
  - تُعاد مواءمة المستويات المخزنة غير المدعومة الموجودة، بما في ذلك قيم `max` القديمة بعد تبديل النماذج، إلى أكبر مستوى مدعوم للنموذج المحدد.
  - تستخدم نماذج Anthropic Claude 4.6 المستوى الافتراضي `adaptive` عندما لا يتم ضبط مستوى تفكير صريح.
  - لا يستخدم Anthropic Claude Opus 4.7 التفكير التكيفي افتراضيًا. تبقى قيمة الجهد الافتراضية في API مملوكة للمزوّد ما لم تضبط مستوى تفكير صريحًا.
  - يربط Anthropic Claude Opus 4.7 الأمر `/think xhigh` بالتفكير التكيفي مع `output_config.effort: "xhigh"`، لأن `/think` هو توجيه تفكير و`xhigh` هو إعداد الجهد في Opus 4.7.
  - يوفّر Anthropic Claude Opus 4.7 أيضًا `/think max`؛ ويرتبط بالمسار نفسه لأقصى جهد مملوك للمزوّد.
  - تربط نماذج OpenAI GPT الأمر `/think` عبر دعم الجهد الخاص بـ Responses API لكل نموذج. يرسل `/think off` القيمة `reasoning.effort: "none"` فقط عندما يدعمها النموذج الهدف؛ وإلا فإن OpenClaw يحذف حمولة تعطيل الاستدلال بدلًا من إرسال قيمة غير مدعومة.
  - يستخدم MiniMax ‏(`minimax/*`) على مسار التدفق المتوافق مع Anthropic القيمة الافتراضية `thinking: { type: "disabled" }` ما لم تضبط التفكير صراحة في params الخاصة بالنموذج أو الطلب. وهذا يمنع تسرب فروق `reasoning_content` من تنسيق تدفق Anthropic غير الأصلي لدى MiniMax.
  - يدعم Z.AI ‏(`zai/*`) التفكير الثنائي فقط (`on`/`off`). ويُعامل أي مستوى غير `off` على أنه `on` (ويرتبط بـ `low`).
  - يربط Moonshot ‏(`moonshot/*`) الأمر `/think off` بـ `thinking: { type: "disabled" }` وأي مستوى غير `off` بـ `thinking: { type: "enabled" }`. وعند تمكين التفكير، لا يقبل Moonshot إلا `tool_choice` بالقيم `auto|none`؛ ويقوم OpenClaw بتطبيع القيم غير المتوافقة إلى `auto`.

## ترتيب الحل

1. التوجيه المضمّن في الرسالة (ينطبق على تلك الرسالة فقط).
2. تجاوز الجلسة (يُضبط بإرسال رسالة تحتوي على التوجيه فقط).
3. الافتراضي لكل وكيل (`agents.list[].thinkingDefault` في الإعدادات).
4. الافتراضي العام (`agents.defaults.thinkingDefault` في الإعدادات).
5. الاحتياطي: الافتراضي المعلن من المزوّد عند توفره، و`low` لنماذج الفهرس الأخرى المعلّمة على أنها قادرة على الاستدلال، و`off` بخلاف ذلك.

## ضبط افتراضي للجلسة

- أرسل رسالة تكون **فقط** التوجيه (يُسمح بالمسافات البيضاء)، مثل `/think:medium` أو `/t high`.
- يظل هذا ساريًا للجلسة الحالية (لكل مرسل افتراضيًا)؛ ويُمسح بواسطة `/think:off` أو بإعادة ضبط خمول الجلسة.
- يتم إرسال رد تأكيد (`Thinking level set to high.` / `Thinking disabled.`). وإذا كان المستوى غير صالح (مثل `/thinking big`)، يُرفض الأمر مع تلميح وتبقى حالة الجلسة دون تغيير.
- أرسل `/think` (أو `/think:`) بدون وسيطة لمعرفة مستوى التفكير الحالي.

## التطبيق حسب الوكيل

- **Pi المضمّن**: يتم تمرير المستوى المحسوم إلى runtime الخاص بوكيل Pi داخل العملية.

## وضع السرعة (/fast)

- المستويات: `on|off`.
- تبدّل الرسالة التي تحتوي على التوجيه فقط تجاوز وضع السرعة للجلسة وترد بـ `Fast mode enabled.` / `Fast mode disabled.`.
- أرسل `/fast` (أو `/fast status`) بدون وضع لمعرفة حالة وضع السرعة الفعلية الحالية.
- يحسم OpenClaw وضع السرعة بهذا الترتيب:
  1. `/fast on|off` المضمّن/الخاص بالتوجيه فقط
  2. تجاوز الجلسة
  3. الافتراضي لكل وكيل (`agents.list[].fastModeDefault`)
  4. إعداد كل نموذج: `agents.defaults.models["<provider>/<model>"].params.fastMode`
  5. الاحتياطي: `off`
- بالنسبة إلى `openai/*`، يرتبط وضع السرعة بالمعالجة ذات الأولوية في OpenAI عن طريق إرسال `service_tier=priority` في طلبات Responses المدعومة.
- بالنسبة إلى `openai-codex/*`، يرسل وضع السرعة العلامة نفسها `service_tier=priority` في Codex Responses. ويحافظ OpenClaw على مفتاح تبديل `/fast` مشترك واحد عبر مساري المصادقة.
- بالنسبة إلى طلبات `anthropic/*` العامة المباشرة، بما في ذلك الحركة الموثقة عبر OAuth والمرسلة إلى `api.anthropic.com`، يرتبط وضع السرعة بطبقات خدمة Anthropic: يضبط `/fast on` القيمة `service_tier=auto`، ويضبط `/fast off` القيمة `service_tier=standard_only`.
- بالنسبة إلى `minimax/*` على المسار المتوافق مع Anthropic، يعيد `/fast on` (أو `params.fastMode: true`) كتابة `MiniMax-M2.7` إلى `MiniMax-M2.7-highspeed`.
- تتجاوز params الصريحة للنموذج `serviceTier` / `service_tier` في Anthropic افتراضي وضع السرعة عند ضبطهما معًا. وما يزال OpenClaw يتخطى حقن طبقة خدمة Anthropic لعناوين URL الأساسية الوكيلة غير التابعة لـ Anthropic.

## التوجيهات المفصّلة (/verbose أو /v)

- المستويات: `on` ‏(الحد الأدنى) | `full` | `off` ‏(الافتراضي).
- تبدّل الرسالة التي تحتوي على التوجيه فقط حالة verbose للجلسة وترد بـ `Verbose logging enabled.` / `Verbose logging disabled.`؛ وتعيد المستويات غير الصالحة تلميحًا بدون تغيير الحالة.
- يخزن `/verbose off` تجاوزًا صريحًا للجلسة؛ ويمكنك مسحه عبر واجهة Sessions باختيار `inherit`.
- يؤثر التوجيه المضمّن على تلك الرسالة فقط؛ وتُطبق افتراضيات الجلسة/العالمية بخلاف ذلك.
- أرسل `/verbose` (أو `/verbose:`) بدون وسيطة لمعرفة مستوى verbose الحالي.
- عندما تكون verbose مفعّلة، ترسل الوكلاء التي تُخرج نتائج أدوات منظّمة (Pi، ووكلاء JSON الآخرين) كل استدعاء أداة مرة أخرى كرسالة مستقلة تحتوي على بيانات وصفية فقط، مسبوقة بـ `<emoji> <tool-name>: <arg>` عند توفرها (المسار/الأمر). وتُرسل ملخصات الأدوات هذه بمجرد بدء كل أداة (فقاعات منفصلة)، وليس كتدفقات دلتا.
- تبقى ملخصات فشل الأدوات مرئية في الوضع العادي، لكن لواحق تفاصيل الأخطاء الخام تُخفى ما لم تكن verbose على `on` أو `full`.
- عندما تكون verbose على `full`، تُمرر أيضًا مخرجات الأدوات بعد اكتمالها (فقاعة منفصلة، مقتطعة إلى طول آمن). وإذا بدّلت `/verbose on|full|off` أثناء تشغيلٍ جارٍ، فإن فقاعات الأدوات اللاحقة تحترم الإعداد الجديد.

## توجيهات تتبع Plugin ‏(/trace)

- المستويات: `on` | `off` ‏(الافتراضي).
- تبدّل الرسالة التي تحتوي على التوجيه فقط مخرجات تتبع Plugin للجلسة وترد بـ `Plugin trace enabled.` / `Plugin trace disabled.`.
- يؤثر التوجيه المضمّن على تلك الرسالة فقط؛ وتُطبق افتراضيات الجلسة/العالمية بخلاف ذلك.
- أرسل `/trace` (أو `/trace:`) بدون وسيطة لمعرفة مستوى التتبع الحالي.
- `/trace` أضيق من `/verbose`: فهو يكشف فقط أسطر التتبع/التصحيح المملوكة للـ Plugin مثل ملخصات تصحيح Active Memory.
- قد تظهر أسطر التتبع في `/status` وكَرسالة تشخيصية لاحقة بعد رد المساعد العادي.

## إظهار الاستدلال (/reasoning)

- المستويات: `on|off|stream`.
- تبدّل الرسالة التي تحتوي على التوجيه فقط ما إذا كانت كتل التفكير تظهر في الردود.
- عند التمكين، يُرسل الاستدلال كـ **رسالة منفصلة** مسبوقة بـ `Reasoning:`.
- `stream` ‏(Telegram فقط): يبث الاستدلال في فقاعة المسودة في Telegram أثناء إنشاء الرد، ثم يرسل الإجابة النهائية بدون الاستدلال.
- الاسم البديل: `/reason`.
- أرسل `/reasoning` (أو `/reasoning:`) بدون وسيطة لمعرفة مستوى الاستدلال الحالي.
- ترتيب الحل: التوجيه المضمّن، ثم تجاوز الجلسة، ثم الافتراضي لكل وكيل (`agents.list[].reasoningDefault`)، ثم الاحتياطي (`off`).

## Heartbeat

- نص فحص Heartbeat هو prompt الخاص بـ heartbeat كما هو مُهيأ (الافتراضي: `Read HEARTBEAT.md if it exists (workspace context). Follow it strictly. Do not infer or repeat old tasks from prior chats. If nothing needs attention, reply HEARTBEAT_OK.`). وتنطبق التوجيهات المضمّنة في رسالة heartbeat كالمعتاد (لكن تجنب تغيير افتراضيات الجلسة من رسائل heartbeat).
- يكون تسليم Heartbeat افتراضيًا إلى الحمولة النهائية فقط. ولإرسال رسالة `Reasoning:` المنفصلة أيضًا (عند توفرها)، اضبط `agents.defaults.heartbeat.includeReasoning: true` أو لكل وكيل `agents.list[].heartbeat.includeReasoning: true`.

## واجهة دردشة الويب

- يعكس محدد التفكير في دردشة الويب المستوى المخزن للجلسة من مخزن الجلسات/الإعدادات الواردة عند تحميل الصفحة.
- يؤدي اختيار مستوى آخر إلى كتابة تجاوز الجلسة فورًا عبر `sessions.patch`؛ ولا ينتظر الإرسال التالي، وليس تجاوزًا أحادي الاستخدام من نوع `thinkingOnce`.
- يكون الخيار الأول دائمًا `Default (<resolved level>)`، حيث يأتي المستوى الافتراضي المحسوم من ملف تعريف التفكير الخاص بمزوّد النموذج النشط في الجلسة.
- يستخدم المحدد `thinkingOptions` المُعادة من صف جلسة Gateway. ولا تحتفظ واجهة المتصفح بقائمة regex خاصة بالمزوّد؛ إذ تملك Plugins مجموعات المستويات الخاصة بالنماذج.
- ما يزال `/think:<level>` يعمل ويحدّث المستوى المخزن نفسه للجلسة، لذلك تبقى توجيهات الدردشة والمحدد متزامنين.

## ملفات تعريف المزوّد

- يمكن لـ Plugins الخاصة بالمزوّد كشف `resolveThinkingProfile(ctx)` لتعريف المستويات المدعومة والافتراضي للنموذج.
- يحتوي كل مستوى في ملف التعريف على `id` قانوني مخزن (`off` أو `minimal` أو `low` أو `medium` أو `high` أو `xhigh` أو `adaptive` أو `max`) وقد يتضمن `label` للعرض. ويستخدم المزوّدون الثنائيون `{ id: "low", label: "on" }`.
- تبقى الخطافات المنشورة القديمة (`supportsXHighThinking` و`isBinaryThinking` و`resolveDefaultThinkingLevel`) كمهايئات توافقية، لكن مجموعات المستويات المخصصة الجديدة يجب أن تستخدم `resolveThinkingProfile`.
- تكشف صفوف Gateway القيمتين `thinkingOptions` و`thinkingDefault` لكي تعرض عملاء ACP/الدردشة ملف التعريف نفسه الذي يستخدمه التحقق أثناء runtime.

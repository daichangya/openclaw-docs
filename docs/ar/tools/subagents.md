تكون الوكلاء الفرعيون تشغيلات وكلاء في الخلفية يتم إنشاؤها من تشغيل وكيل موجود. وهي تعمل في جلستها الخاصة (`agent:<agentId>:subagent:<uuid>`) وعند الانتهاء **تعلن** نتيجتها مرة أخرى إلى قناة دردشة الطالب. ويتم تتبع كل تشغيل وكيل فرعي على أنه [مهمة في الخلفية](/ar/automation/tasks).

## أمر الشرطة المائلة

استخدم `/subagents` لفحص تشغيلات الوكلاء الفرعيين أو التحكم فيها للجلسة **الحالية**:

- `/subagents list`
- `/subagents kill <id|#|all>`
- `/subagents log <id|#> [limit] [tools]`
- `/subagents info <id|#>`
- `/subagents send <id|#> <message>`
- `/subagents steer <id|#> <message>`
- `/subagents spawn <agentId> <task> [--model <model>] [--thinking <level>]`

عناصر تحكم ربط السلاسل:

تعمل هذه الأوامر على القنوات التي تدعم ارتباطات السلاسل الدائمة. راجع **القنوات التي تدعم السلاسل** أدناه.

- `/focus <subagent-label|session-key|session-id|session-label>`
- `/unfocus`
- `/agents`
- `/session idle <duration|off>`
- `/session max-age <duration|off>`

يعرض `/subagents info` بيانات تعريف التشغيل (الحالة، والطوابع الزمنية، ومعرّف الجلسة، ومسار النص، والتنظيف).
استخدم `sessions_history` لعرض استدعاء محدود ومفلتر من ناحية السلامة؛ وافحص
مسار النص على القرص عندما تحتاج إلى النص الخام الكامل.

### سلوك الإنشاء

يبدأ `/subagents spawn` وكيلًا فرعيًا في الخلفية كأمر مستخدم، وليس كترحيل داخلي، ويرسل تحديث إكمال نهائيًا واحدًا مرة أخرى إلى دردشة الطالب عند انتهاء التشغيل.

- أمر الإنشاء غير حاجب؛ فهو يعيد معرّف تشغيل فورًا.
- عند الإكمال، يعلن الوكيل الفرعي رسالة ملخص/نتيجة مرة أخرى إلى قناة دردشة الطالب.
- يعتمد تسليم الإكمال على الدفع. بعد الإنشاء، لا تستعلم عن `/subagents list`
  أو `sessions_list` أو `sessions_history` بشكل متكرر في حلقة فقط لانتظار
  الانتهاء؛ وافحص الحالة عند الطلب فقط لأغراض التصحيح أو التدخل.
- عند الإكمال، يغلق OpenClaw على أساس أفضل جهد علامات تبويب/عمليات المتصفح المتعقبة التي فتحها ذلك الوكيل الفرعي قبل متابعة تدفق تنظيف الإعلان.
- بالنسبة إلى الإنشاءات اليدوية، يكون التسليم مرنًا:
  - يحاول OpenClaw أولًا التسليم المباشر إلى `agent` باستخدام مفتاح idempotency ثابت.
  - إذا فشل التسليم المباشر، يعود إلى التوجيه عبر الطابور.
  - وإذا ظل التوجيه عبر الطابور غير متاح، يُعاد الإعلان مع تراجع أسي قصير قبل الاستسلام النهائي.
- يحتفظ تسليم الإكمال بالمسار المحلول للطالب:
  - تفوز مسارات الإكمال المرتبطة بالسلاسل أو بالمحادثات عند توفرها
  - إذا كان أصل الإكمال يوفّر قناة فقط، يملأ OpenClaw الهدف/الحساب المفقودين من المسار المحلول لجلسة الطالب (`lastChannel` / `lastTo` / `lastAccountId`) بحيث يظل التسليم المباشر يعمل
- إن عملية تسليم الإكمال إلى جلسة الطالب هي سياق داخلي مولد في وقت التشغيل (وليست نصًا كتبه المستخدم) وتتضمن:
  - `Result` (أحدث نص رد ظاهر من `assistant`، وإلا أحدث نص منظف من `tool`/`toolResult`؛ ولا تعيد التشغيلات الفاشلة نهائيًا استخدام نص الرد الملتقط)
  - `Status` (`completed successfully` / `failed` / `timed out` / `unknown`)
  - إحصاءات مضغوطة لوقت التشغيل/الرموز
  - تعليمات تسليم تطلب من وكيل الطالب إعادة الصياغة بصوت المساعد العادي (وليس تمرير بيانات التعريف الداخلية الخام)
- يتجاوز `--model` و`--thinking` الإعدادات الافتراضية لذلك التشغيل المحدد.
- استخدم `info`/`log` لفحص التفاصيل والمخرجات بعد الإكمال.
- يعمل `/subagents spawn` في وضع الطلقة الواحدة (`mode: "run"`). وبالنسبة إلى الجلسات الدائمة المرتبطة بالسلاسل، استخدم `sessions_spawn` مع `thread: true` و`mode: "session"`.
- بالنسبة إلى جلسات حزام ACP (Codex وClaude Code وGemini CLI)، استخدم `sessions_spawn` مع `runtime: "acp"` وراجع [وكلاء ACP](/ar/tools/acp-agents)، وخاصة [نموذج تسليم ACP](/ar/tools/acp-agents#delivery-model) عند تصحيح عمليات الإكمال أو حلقات وكيل إلى وكيل.

الأهداف الأساسية:

- موازاة أعمال "البحث / المهمة الطويلة / الأداة البطيئة" من دون حجب التشغيل الرئيسي.
- إبقاء الوكلاء الفرعيين معزولين افتراضيًا (فصل الجلسات + Sandbox اختياري).
- إبقاء سطح الأداة صعب الاستخدام الخاطئ: لا تحصل الوكلاء الفرعيون على أدوات الجلسات افتراضيًا.
- دعم عمق تعشيش قابل للتهيئة لأنماط المنسّق.

ملاحظة حول التكلفة: لكل وكيل فرعي سياقه **الخاص** واستخدام الرموز الخاص به افتراضيًا. بالنسبة إلى المهام الثقيلة أو
المتكررة، اضبط نموذجًا أرخص للوكلاء الفرعيين واحتفظ بالوكيل الرئيسي على نموذج
أعلى جودة. يمكنك تهيئة ذلك عبر `agents.defaults.subagents.model` أو
التجاوزات لكل وكيل. وعندما يحتاج الابن فعلًا إلى النص الحالي للطالب، يمكن للوكيل طلب
`context: "fork"` في ذلك الإنشاء فقط.

## أوضاع السياق

تبدأ الوكلاء الفرعيون الأصليون معزولين ما لم يطلب المستدعي صراحةً تفريع
النص الحالي.

| الوضع       | متى تستخدمه                                                                                                                               | السلوك                                                                          |
| ---------- | ------------------------------------------------------------------------------------------------------------------------------------------ | --------------------------------------------------------------------------------- |
| `isolated` | بحث جديد، أو تنفيذ مستقل، أو عمل بأداة بطيئة، أو أي شيء يمكن شرحه بإيجاز في نص المهمة                                                   | ينشئ نصًا فرعيًا نظيفًا. وهذا هو الافتراضي ويحافظ على استخدام الرموز أقل.        |
| `fork`     | عمل يعتمد على المحادثة الحالية، أو نتائج أدوات سابقة، أو تعليمات دقيقة موجودة بالفعل في نص الطالب                                       | يفرّع نص الطالب إلى الجلسة الفرعية قبل بدء الجلسة الفرعية.                       |

استخدم `fork` باعتدال. فهو مخصص للتفويض الحساس للسياق، وليس بديلًا
عن كتابة مطالبة مهمة واضحة.

## الأداة

استخدم `sessions_spawn`:

- يبدأ تشغيل وكيل فرعي (`deliver: false`، المسار العام: `subagent`)
- ثم يشغّل خطوة إعلان وينشر رد الإعلان إلى قناة دردشة الطالب
- النموذج الافتراضي: يرث من المستدعي ما لم تضبط `agents.defaults.subagents.model` (أو `agents.list[].subagents.model` لكل وكيل)؛ ولا يزال `sessions_spawn.model` الصريح يتغلب.
- التفكير الافتراضي: يرث من المستدعي ما لم تضبط `agents.defaults.subagents.thinking` (أو `agents.list[].subagents.thinking` لكل وكيل)؛ ولا يزال `sessions_spawn.thinking` الصريح يتغلب.
- المهلة الافتراضية للتشغيل: إذا تم حذف `sessions_spawn.runTimeoutSeconds`، يستخدم OpenClaw القيمة `agents.defaults.subagents.runTimeoutSeconds` عند ضبطها؛ وإلا يعود إلى `0` (بدون مهلة).

معلمات الأداة:

- `task` (مطلوب)
- `label?` (اختياري)
- `agentId?` (اختياري؛ الإنشاء تحت معرّف وكيل آخر إذا كان مسموحًا)
- `model?` (اختياري؛ يتجاوز نموذج الوكيل الفرعي؛ ويتم تخطي القيم غير الصالحة ويعمل الوكيل الفرعي على النموذج الافتراضي مع تحذير في نتيجة الأداة)
- `thinking?` (اختياري؛ يتجاوز مستوى التفكير لتشغيل الوكيل الفرعي)
- `runTimeoutSeconds?` (القيمة الافتراضية هي `agents.defaults.subagents.runTimeoutSeconds` عند ضبطها، وإلا `0`؛ وعند ضبطها، يتم إجهاض تشغيل الوكيل الفرعي بعد N ثانية)
- `thread?` (الافتراضي `false`؛ عندما تكون `true`، تطلب تدفق ربط سلسلة لهذه الجلسة الفرعية)
- `mode?` (`run|session`)
  - الافتراضي هو `run`
  - إذا كان `thread: true` وتم حذف `mode`، تصبح القيمة الافتراضية `session`
  - تتطلب `mode: "session"` وجود `thread: true`
- `cleanup?` (`delete|keep`، الافتراضي `keep`)
- `sandbox?` (`inherit|require`، الافتراضي `inherit`؛ ترفض `require` الإنشاء ما لم يكن وقت تشغيل الابن الهدف داخل Sandbox)
- `context?` (`isolated|fork`، الافتراضي `isolated`؛ للوكلاء الفرعيين الأصليين فقط)
  - تنشئ `isolated` نصًا فرعيًا نظيفًا وهي الافتراضية.
  - تفرّع `fork` نص الطالب الحالي إلى الجلسة الفرعية بحيث يبدأ الابن بالسياق نفسه للمحادثة.
  - استخدم `fork` فقط عندما يحتاج الابن إلى النص الحالي. وبالنسبة إلى العمل المحدد النطاق، احذف `context`.
- لا يقبل `sessions_spawn` معلمات تسليم القنوات (`target` أو `channel` أو `to` أو `threadId` أو `replyTo` أو `transport`). وبالنسبة إلى التسليم، استخدم `message`/`sessions_send` من التشغيل الذي تم إنشاؤه.

## الجلسات المرتبطة بالسلاسل

عندما تكون ارتباطات السلاسل مفعّلة لقناة، يمكن للوكيل الفرعي أن يظل مرتبطًا بسلسلة بحيث تستمر رسائل المستخدم اللاحقة في تلك السلسلة في التوجيه إلى جلسة الوكيل الفرعي نفسها.

### القنوات التي تدعم السلاسل

- Discord (القناة الوحيدة المدعومة حاليًا): تدعم جلسات الوكلاء الفرعيين الدائمة المرتبطة بالسلاسل (`sessions_spawn` مع `thread: true`)، وعناصر التحكم اليدوية في السلاسل (`/focus` و`/unfocus` و`/agents` و`/session idle` و`/session max-age`)، ومفاتيح المحول `channels.discord.threadBindings.enabled` و`channels.discord.threadBindings.idleHours` و`channels.discord.threadBindings.maxAgeHours` و`channels.discord.threadBindings.spawnSubagentSessions`.

التدفق السريع:

1. أنشئ باستخدام `sessions_spawn` مع `thread: true` (واختياريًا `mode: "session"`).
2. ينشئ OpenClaw سلسلة أو يربطها بهدف تلك الجلسة في القناة النشطة.
3. تُوجَّه الردود ورسائل المتابعة في تلك السلسلة إلى الجلسة المرتبطة.
4. استخدم `/session idle` لفحص/تحديث إلغاء التركيز التلقائي بسبب عدم النشاط، واستخدم `/session max-age` للتحكم في الحد الأقصى الصارم.
5. استخدم `/unfocus` لفك الارتباط يدويًا.

عناصر التحكم اليدوية:

- يربط `/focus <target>` السلسلة الحالية (أو ينشئ واحدة) بهدف وكيل فرعي/جلسة.
- يزيل `/unfocus` الارتباط للسلسلة المرتبطة الحالية.
- يسرد `/agents` التشغيلات النشطة وحالة الارتباط (`thread:<id>` أو `unbound`).
- يعمل `/session idle` و`/session max-age` فقط للسلاسل المرتبطة التي عليها تركيز.

مفاتيح الإعداد:

- الإعدادات العامة الافتراضية: `session.threadBindings.enabled` و`session.threadBindings.idleHours` و`session.threadBindings.maxAgeHours`
- مفاتيح تجاوز القنوات والربط التلقائي عند الإنشاء خاصة بالمحول. راجع **القنوات التي تدعم السلاسل** أعلاه.

راجع [مرجع الإعداد](/ar/gateway/configuration-reference) و[أوامر الشرطة المائلة](/ar/tools/slash-commands) للحصول على تفاصيل المحول الحالية.

قائمة السماح:

- `agents.list[].subagents.allowAgents`: قائمة بمعرّفات الوكلاء التي يمكن استهدافها عبر `agentId` (`["*"]` للسماح بأي وكيل). الافتراضي: وكيل الطالب فقط.
- `agents.defaults.subagents.allowAgents`: قائمة السماح الافتراضية للوكيل الهدف المستخدمة عندما لا يضبط وكيل الطالب `subagents.allowAgents` الخاص به.
- حارس وراثة Sandbox: إذا كانت جلسة الطالب داخل Sandbox، يرفض `sessions_spawn` الأهداف التي ستعمل خارج Sandbox.
- `agents.defaults.subagents.requireAgentId` / `agents.list[].subagents.requireAgentId`: عند تفعيلها، تُحظر استدعاءات `sessions_spawn` التي تحذف `agentId` (لفرض اختيار ملف تعريف صريح). الافتراضي: false.

الاكتشاف:

- استخدم `agents_list` لمعرفة معرّفات الوكلاء المسموح بها حاليًا لـ `sessions_spawn`.

الأرشفة التلقائية:

- تتم أرشفة جلسات الوكلاء الفرعيين تلقائيًا بعد `agents.defaults.subagents.archiveAfterMinutes` (الافتراضي: 60).
- تستخدم الأرشفة `sessions.delete` وتعيد تسمية النص إلى `*.deleted.<timestamp>` (في المجلد نفسه).
- يؤدي `cleanup: "delete"` إلى الأرشفة فورًا بعد الإعلان (مع الاحتفاظ بالنص عبر إعادة التسمية).
- الأرشفة التلقائية تعتمد أفضل جهد؛ وتُفقد المؤقتات المعلقة إذا أُعيد تشغيل Gateway.
- لا يؤدي `runTimeoutSeconds` إلى الأرشفة التلقائية؛ فهو يوقف التشغيل فقط. وتبقى الجلسة حتى الأرشفة التلقائية.
- تنطبق الأرشفة التلقائية بالتساوي على جلسات العمق 1 والعمق 2.
- تنظيف المتصفح منفصل عن تنظيف الأرشفة: يتم إغلاق علامات تبويب/عمليات المتصفح المتعقبة على أساس أفضل جهد عند انتهاء التشغيل، حتى إذا تم الاحتفاظ بسجل الجلسة/النص.

## الوكلاء الفرعيون المتداخلون

افتراضيًا، لا يمكن للوكلاء الفرعيين إنشاء وكلاء فرعيين خاصين بهم (`maxSpawnDepth: 1`). يمكنك تمكين مستوى واحد من التداخل بضبط `maxSpawnDepth: 2`، مما يسمح **بنمط المنسّق**: الرئيسي → وكيل فرعي منسّق → وكلاء فرعيون فرعيون عاملون.

### كيفية التمكين

```json5
{
  agents: {
    defaults: {
      subagents: {
        maxSpawnDepth: 2, // السماح للوكلاء الفرعيين بإنشاء أبناء (الافتراضي: 1)
        maxChildrenPerAgent: 5, // الحد الأقصى للأبناء النشطين لكل جلسة وكيل (الافتراضي: 5)
        maxConcurrent: 8, // الحد الأقصى العالمي للتزامن في المسار (الافتراضي: 8)
        runTimeoutSeconds: 900, // المهلة الافتراضية لـ sessions_spawn عند الحذف (0 = بلا مهلة)
      },
    },
  },
}
```

### مستويات العمق

| العمق | شكل مفتاح الجلسة                            | الدور                                         | هل يمكنه الإنشاء؟              |
| ----- | ------------------------------------------ | --------------------------------------------- | ------------------------------ |
| 0     | `agent:<id>:main`                          | الوكيل الرئيسي                                | دائمًا                         |
| 1     | `agent:<id>:subagent:<uuid>`               | وكيل فرعي (منسّق عند السماح بالعمق 2)         | فقط إذا كان `maxSpawnDepth >= 2` |
| 2     | `agent:<id>:subagent:<uuid>:subagent:<uuid>` | وكيل فرعي فرعي (عامل طرفي)                   | أبدًا                          |

### سلسلة الإعلان

تتدفق النتائج عائدةً صعودًا عبر السلسلة:

1. ينتهي عامل العمق 2 ← ويعلن إلى أصله (منسّق العمق 1)
2. يتلقى منسّق العمق 1 الإعلان، ويجمّع النتائج، وينتهي ← ثم يعلن إلى الرئيسي
3. يتلقى الوكيل الرئيسي الإعلان ويسلّمه إلى المستخدم

يرى كل مستوى الإعلانات القادمة من أبنائه المباشرين فقط.

إرشادات تشغيلية:

- ابدأ عمل الابن مرة واحدة وانتظر أحداث الإكمال بدلًا من بناء حلقات
  استعلام حول `sessions_list` أو `sessions_history` أو `/subagents list` أو
  أوامر السكون `exec`.
- يُبقي `sessions_list` و`/subagents list` علاقات جلسات الأبناء مركّزة
  على العمل الحي: يظل الأبناء الأحياء مرفقين، وتبقى الأبناء المنتهية مرئية
  لنافذة حديثة قصيرة، ويتم تجاهل روابط الأبناء القديمة الموجودة في المخزن فقط بعد
  نافذة حداثتها. وهذا يمنع بيانات التعريف القديمة `spawnedBy` / `parentSessionKey`
  من إحياء أبناء أشباح بعد إعادة التشغيل.
- إذا وصل حدث إكمال ابن بعد أن تكون قد أرسلت الإجابة النهائية بالفعل،
  فالمتابعة الصحيحة هي الرمز الصامت الدقيق `NO_REPLY` / `no_reply`.

### سياسة الأدوات حسب العمق

- يُكتب الدور ونطاق التحكم في بيانات تعريف الجلسة وقت الإنشاء. وهذا يمنع مفاتيح الجلسات المسطحة أو المستعادة من استعادة صلاحيات المنسّق بالخطأ.
- **العمق 1 (منسّق، عندما `maxSpawnDepth >= 2`)**: يحصل على `sessions_spawn` و`subagents` و`sessions_list` و`sessions_history` حتى يتمكن من إدارة أبنائه. وتظل أدوات الجلسة/النظام الأخرى مرفوضة.
- **العمق 1 (طرفي، عندما `maxSpawnDepth == 1`)**: لا توجد أدوات جلسة (السلوك الافتراضي الحالي).
- **العمق 2 (عامل طرفي)**: لا توجد أدوات جلسة — يكون `sessions_spawn` مرفوضًا دائمًا عند العمق 2. ولا يمكنه إنشاء أبناء إضافيين.

### حد الإنشاء لكل وكيل

يمكن لكل جلسة وكيل (على أي عمق) أن تمتلك بحد أقصى `maxChildrenPerAgent` (الافتراضي: 5) أبناء نشطين في الوقت نفسه. وهذا يمنع التفرع غير المنضبط من منسّق واحد.

### الإيقاف المتسلسل

يؤدي إيقاف منسّق العمق 1 تلقائيًا إلى إيقاف جميع أبنائه من العمق 2:

- يؤدي `/stop` في الدردشة الرئيسية إلى إيقاف جميع وكلاء العمق 1 ويتسلسل إلى أبنائهم من العمق 2.
- يؤدي `/subagents kill <id>` إلى إيقاف وكيل فرعي محدد ويتسلسل إلى أبنائه.
- يؤدي `/subagents kill all` إلى إيقاف جميع الوكلاء الفرعيين للطالب ويتسلسل.

## المصادقة

يتم حل مصادقة الوكيل الفرعي حسب **معرّف الوكيل**، وليس حسب نوع الجلسة:

- مفتاح جلسة الوكيل الفرعي هو `agent:<agentId>:subagent:<uuid>`.
- يتم تحميل مخزن المصادقة من `agentDir` الخاص بهذا الوكيل.
- يتم دمج ملفات تعريف مصادقة الوكيل الرئيسي كخيار **احتياطي**؛ وتتغلب ملفات تعريف الوكيل على ملفات تعريف الرئيسي عند التعارض.

ملاحظة: الدمج إضافي، لذا تكون ملفات تعريف الرئيسي متاحة دائمًا كبدائل احتياطية. ولا تزال المصادقة المعزولة بالكامل لكل وكيل غير مدعومة بعد.

## الإعلان

تبلغ الوكلاء الفرعيون النتائج مرة أخرى عبر خطوة إعلان:

- تعمل خطوة الإعلان داخل جلسة الوكيل الفرعي (وليس جلسة الطالب).
- إذا رد الوكيل الفرعي بالنص المطابق تمامًا `ANNOUNCE_SKIP`، فلن يتم نشر شيء.
- إذا كان أحدث نص من المساعد هو الرمز الصامت المطابق تمامًا `NO_REPLY` / `no_reply`،
  فسيتم منع مخرجات الإعلان حتى لو وُجد تقدم مرئي سابق.
- وبخلاف ذلك يعتمد التسليم على عمق الطالب:
  - تستخدم جلسات الطالب ذات المستوى الأعلى استدعاء `agent` للمتابعة مع تسليم خارجي (`deliver=true`)
  - تتلقى جلسات الوكيل الفرعي المتداخلة التابعة للطالب حقن متابعة داخليًا (`deliver=false`) حتى يتمكن المنسّق من تجميع نتائج الأبناء داخل الجلسة
  - إذا اختفت جلسة وكيل فرعي متداخلة تابعة للطالب، يعود OpenClaw إلى طالب تلك الجلسة عند توفره
- بالنسبة إلى جلسات الطالب ذات المستوى الأعلى، يحل التسليم المباشر في وضع الإكمال أولًا أي مسار محادثة/سلسلة مرتبط وأي تجاوز للخطافات، ثم يملأ حقول هدف القناة المفقودة من المسار المخزن لجلسة الطالب. وهذا يُبقي الإكمالات على الدردشة/الموضوع الصحيحين حتى عندما يحدد أصل الإكمال القناة فقط.
- يقتصر تجميع إكمالات الأبناء على تشغيل الطالب الحالي عند بناء نتائج الإكمال المتداخلة، مما يمنع تسرب مخرجات الأبناء القديمة من تشغيلات سابقة إلى الإعلان الحالي.
- تحافظ ردود الإعلان على توجيه السلسلة/الموضوع عند توفره في محولات القنوات.
- يُطبَّع سياق الإعلان إلى كتلة حدث داخلية مستقرة:
  - المصدر (`subagent` أو `cron`)
  - مفتاح/معرّف جلسة الابن
  - نوع الإعلان + تسمية المهمة
  - سطر حالة مشتق من نتيجة وقت التشغيل (`success` أو `error` أو `timeout` أو `unknown`)
  - محتوى النتيجة المختار من أحدث نص ظاهر للمساعد، وإلا أحدث نص منظف من `tool`/`toolResult`؛ وتبلغ التشغيلات الفاشلة نهائيًا عن حالة الفشل من دون إعادة تشغيل نص الرد الملتقط
  - تعليمات متابعة تصف متى يجب الرد ومتى يجب البقاء صامتًا
- لا يُستنتج `Status` من مخرجات النموذج؛ بل يأتي من إشارات نتيجة وقت التشغيل.
- عند انتهاء المهلة، إذا لم يصل الابن إلا إلى استدعاءات الأدوات، يمكن للإعلان أن يختصر هذا السجل في ملخص قصير للتقدم الجزئي بدلًا من إعادة تشغيل مخرجات الأدوات الخام.

تتضمن حمولات الإعلان سطر إحصاءات في النهاية (حتى عند التفافها):

- وقت التشغيل (مثل `runtime 5m12s`)
- استخدام الرموز (الإدخال/الإخراج/الإجمالي)
- التكلفة التقديرية عند تهيئة تسعير النموذج (`models.providers.*.models[].cost`)
- `sessionKey` و`sessionId` ومسار النص (حتى يتمكن الوكيل الرئيسي من جلب السجل عبر `sessions_history` أو فحص الملف على القرص)
- بيانات التعريف الداخلية مخصصة للتنسيق فقط؛ ويجب إعادة كتابة الردود الموجهة للمستخدم بصوت المساعد العادي.

يمثل `sessions_history` مسار التنسيق الأكثر أمانًا:

- يتم أولًا تطبيع استدعاء المساعد:
  - تتم إزالة وسوم التفكير
  - تتم إزالة كتل البنية `relevant-memories` / `<relevant_memories>`
  - تتم إزالة كتل حمولة XML النصية العادية لاستدعاءات الأدوات مثل `<tool_call>...</tool_call>`،
    و`<function_call>...</function_call>`، و`<tool_calls>...</tool_calls>`، و
    `<function_calls>...</function_calls>`، بما في ذلك الحمولات المقتطعة
    التي لا تُغلق بشكل نظيف
  - تتم إزالة البنية المخفّضة لاستدعاءات/نتائج الأدوات وعلامات السياق التاريخي
  - تتم إزالة رموز التحكم بالنموذج المتسربة مثل `<|assistant|>`، ورموز ASCII
    الأخرى من الشكل `<|...|>`، والمتغيرات كاملة العرض `<｜...｜>`
  - تتم إزالة XML استدعاءات الأدوات المشوه الخاص بـ MiniMax
- تتم تنقيح النصوص المشابهة لبيانات الاعتماد/الرموز
- يمكن اقتطاع الكتل الطويلة
- قد تؤدي السجلات الكبيرة جدًا إلى إسقاط الصفوف الأقدم أو استبدال صف ضخم بـ
  `[sessions_history omitted: message too large]`
- يكون فحص النص الخام على القرص هو الحل الاحتياطي عندما تحتاج إلى النص الكامل كما هو بايتًا ببايت

## سياسة الأدوات (أدوات الوكيل الفرعي)

افتراضيًا، تحصل الوكلاء الفرعيون على **كل الأدوات باستثناء أدوات الجلسات** وأدوات النظام:

- `sessions_list`
- `sessions_history`
- `sessions_send`
- `sessions_spawn`

يظل `sessions_history` هنا أيضًا عرض استدعاء محدودًا ومنظفًا؛ وليس
تفريغًا خامًا للنص.

عندما يكون `maxSpawnDepth >= 2`، تتلقى الوكلاء الفرعيون المنسّقون من العمق 1 بالإضافة إلى ذلك `sessions_spawn` و`subagents` و`sessions_list` و`sessions_history` حتى يتمكنوا من إدارة أبنائهم.

يمكن التجاوز عبر الإعداد:

```json5
{
  agents: {
    defaults: {
      subagents: {
        maxConcurrent: 1,
      },
    },
  },
  tools: {
    subagents: {
      tools: {
        // الرفض يتغلب
        deny: ["gateway", "cron"],
        // إذا تم ضبط allow، يصبح السماح فقط (ويظل الرفض متغلبًا)
        // allow: ["read", "exec", "process"]
      },
    },
  },
}
```

## التزامن

تستخدم الوكلاء الفرعيون مسار طابور مخصصًا داخل العملية:

- اسم المسار: `subagent`
- التزامن: `agents.defaults.subagents.maxConcurrent` (الافتراضي `8`)

## الحيوية والاسترداد

لا يتعامل OpenClaw مع غياب `endedAt` على أنه دليل دائم على أن وكيلًا فرعيًا
لا يزال حيًا. وتتوقف التشغيلات غير المنتهية الأقدم من نافذة التشغيلات الراكدة عن
الاحتساب على أنها نشطة/معلقة في `/subagents list`، وملخصات الحالة، وحظر إكمال
الأبناء التنازلي، وفحوصات التزامن لكل جلسة.

بعد إعادة تشغيل Gateway، يتم تقليم التشغيلات المستعادة الراكدة غير المنتهية ما لم تكن
جلسة الابن معلمة بالقيمة `abortedLastRun: true`. وتظل جلسات الأبناء التي أُجهضت بسبب
إعادة التشغيل قابلة للاسترداد عبر تدفق استرداد أيتام الوكيل الفرعي، الذي
يرسل رسالة استئناف اصطناعية قبل مسح علامة الإجهاض.

## الإيقاف

- يؤدي إرسال `/stop` في دردشة الطالب إلى إجهاض جلسة الطالب وإيقاف أي تشغيلات وكلاء فرعيين نشطة أُنشئت منها، مع التسلسل إلى الأبناء المتداخلين.
- يؤدي `/subagents kill <id>` إلى إيقاف وكيل فرعي محدد ويتسلسل إلى أبنائه.

## القيود

- إعلان الوكيل الفرعي يعتمد **أفضل جهد**. إذا أُعيد تشغيل Gateway، فسيتم فقدان أعمال "الإعلان عودةً" المعلقة.
- لا تزال الوكلاء الفرعيون تتشارك موارد عملية Gateway نفسها؛ لذا تعامل مع `maxConcurrent` على أنه صمام أمان.
- يكون `sessions_spawn` دائمًا غير حاجب: فهو يعيد `{ status: "accepted", runId, childSessionKey }` فورًا.
- يحقن سياق الوكيل الفرعي فقط `AGENTS.md` + `TOOLS.md` (ولا يحقن `SOUL.md` أو `IDENTITY.md` أو `USER.md` أو `HEARTBEAT.md` أو `BOOTSTRAP.md`).
- الحد الأقصى لعمق التداخل هو 5 (نطاق `maxSpawnDepth`: من 1 إلى 5). ويُنصح بالعمق 2 لمعظم حالات الاستخدام.
- يحد `maxChildrenPerAgent` عدد الأبناء النشطين لكل جلسة (الافتراضي: 5، النطاق: 1–20).

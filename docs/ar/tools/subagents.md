# الوكلاء الفرعيون

الوكلاء الفرعيون هم عمليات وكيل في الخلفية يتم تشغيلها من عملية وكيل موجودة. تعمل في جلستها الخاصة (`agent:<agentId>:subagent:<uuid>`) وعند الانتهاء، **تعلن** نتيجتها مرة أخرى في قناة دردشة الطالب. يتم تتبّع كل عملية وكيل فرعي على أنها [مهمة في الخلفية](/ar/automation/tasks).

## أمر الشرطة المائلة

استخدم `/subagents` لفحص أو التحكم في عمليات الوكيل الفرعي الخاصة بـ **الجلسة الحالية**:

- `/subagents list`
- `/subagents kill <id|#|all>`
- `/subagents log <id|#> [limit] [tools]`
- `/subagents info <id|#>`
- `/subagents send <id|#> <message>`
- `/subagents steer <id|#> <message>`
- `/subagents spawn <agentId> <task> [--model <model>] [--thinking <level>]`

عناصر التحكم في ربط المحادثات:

تعمل هذه الأوامر على القنوات التي تدعم ربط المحادثات المستمر. راجع **القنوات الداعمة للمحادثات** أدناه.

- `/focus <subagent-label|session-key|session-id|session-label>`
- `/unfocus`
- `/agents`
- `/session idle <duration|off>`
- `/session max-age <duration|off>`

يعرض `/subagents info` بيانات تعريف العملية (الحالة، والطوابع الزمنية، ومعرّف الجلسة، ومسار النص التفريغي، والتنظيف).
استخدم `sessions_history` للحصول على عرض استدعاء مقيّد ومفلتر للسلامة؛ وافحص
مسار النص التفريغي على القرص عندما تحتاج إلى النص التفريغي الخام الكامل.

### سلوك التشغيل

يبدأ `/subagents spawn` وكيلاً فرعيًا في الخلفية بوصفه أمر مستخدم، وليس ترحيلاً داخليًا، ويرسل تحديثًا نهائيًا واحدًا عند الاكتمال مرة أخرى إلى دردشة الطالب عند انتهاء العملية.

- أمر التشغيل غير حاجب؛ إذ يعيد معرّف العملية فورًا.
- عند الاكتمال، يعلن الوكيل الفرعي رسالة ملخص/نتيجة مرة أخرى إلى قناة دردشة الطالب.
- تسليم الاكتمال قائم على الدفع. بعد التشغيل، لا تستطلع `/subagents list` أو
  `sessions_list` أو `sessions_history` في حلقة فقط لانتظار اكتماله؛ افحص الحالة فقط عند الطلب لأغراض التصحيح أو التدخل.
- عند الاكتمال، يحاول OpenClaw بأفضل جهد إغلاق علامات تبويب/عمليات المتصفح المتتبعة التي فتحها ذلك الوكيل الفرعي قبل أن يستمر تدفق تنظيف الإعلان.
- بالنسبة لعمليات التشغيل اليدوية، يكون التسليم متينًا:
  - يحاول OpenClaw أولاً التسليم المباشر إلى `agent` باستخدام مفتاح idempotency ثابت.
  - إذا فشل التسليم المباشر، فإنه يعود إلى التوجيه عبر الطابور.
  - إذا ظل التوجيه عبر الطابور غير متاح، يُعاد محاولة الإعلان مع تراجع أسي قصير قبل الاستسلام النهائي.
- يحتفظ تسليم الاكتمال بمسار الطالب المحلول:
  - مسارات الاكتمال المرتبطة بالمحادثة أو بالمحادثة المستمرة لها الأولوية عندما تكون متاحة
  - إذا كان أصل الاكتمال يوفّر قناة فقط، يملأ OpenClaw الهدف/الحساب المفقود من المسار المحلول لجلسة الطالب (`lastChannel` / `lastTo` / `lastAccountId`) بحيث يظل التسليم المباشر ممكنًا
- إن تسليم الاكتمال إلى جلسة الطالب هو سياق داخلي مُنشأ وقت التشغيل (وليس نصًا كتبه المستخدم) ويتضمن:
  - `Result` (أحدث نص رد `assistant` مرئي، وإلا فأحدث نص `tool`/`toolResult` مُنقّى؛ العمليات الفاشلة النهائية لا تعيد استخدام نص الرد الملتقط)
  - `Status` (`completed successfully` / `failed` / `timed out` / `unknown`)
  - إحصاءات وقت تشغيل/رموز مدمجة
  - تعليمة تسليم تُخبر وكيل الطالب بإعادة الصياغة بصوت مساعد عادي (وليس إعادة توجيه بيانات تعريف داخلية خام)
- يتجاوز `--model` و`--thinking` الإعدادات الافتراضية لتلك العملية المحددة.
- استخدم `info`/`log` لفحص التفاصيل والمخرجات بعد الاكتمال.
- يعمل `/subagents spawn` بوضع اللقطة الواحدة (`mode: "run"`). بالنسبة للجلسات المستمرة المرتبطة بالمحادثات، استخدم `sessions_spawn` مع `thread: true` و`mode: "session"`.
- بالنسبة إلى جلسات ACP harness ‏(Codex وClaude Code وGemini CLI)، استخدم `sessions_spawn` مع `runtime: "acp"` وراجع [وكلاء ACP](/ar/tools/acp-agents).

الأهداف الأساسية:

- موازاة أعمال "البحث / المهام الطويلة / الأدوات البطيئة" دون حجب العملية الرئيسية.
- إبقاء الوكلاء الفرعيين معزولين افتراضيًا (فصل الجلسات + العزل الاختياري).
- إبقاء سطح الأدوات صعب الإساءة في استخدامه: لا يحصل الوكلاء الفرعيون على أدوات الجلسة افتراضيًا.
- دعم عمق تداخل قابل للتهيئة لأنماط المنسّق.

ملاحظة حول التكلفة: لكل وكيل فرعي **سياقه الخاص** واستخدامه الخاص للرموز. بالنسبة إلى المهام
الثقيلة أو المتكررة، اضبط نموذجًا أقل تكلفة للوكلاء الفرعيين وأبقِ وكيلك الرئيسي على نموذج أعلى جودة.
يمكنك تهيئة ذلك عبر `agents.defaults.subagents.model` أو تجاوزات لكل وكيل.

## الأداة

استخدم `sessions_spawn`:

- يبدأ عملية وكيل فرعي (`deliver: false`، ومسار عام: `subagent`)
- ثم يشغّل خطوة إعلان وينشر رد الإعلان إلى قناة دردشة الطالب
- النموذج الافتراضي: يرث من المستدعي ما لم تضبط `agents.defaults.subagents.model` (أو `agents.list[].subagents.model` لكل وكيل)؛ ولا يزال `sessions_spawn.model` الصريح له الأولوية.
- التفكير الافتراضي: يرث من المستدعي ما لم تضبط `agents.defaults.subagents.thinking` (أو `agents.list[].subagents.thinking` لكل وكيل)؛ ولا يزال `sessions_spawn.thinking` الصريح له الأولوية.
- المهلة الافتراضية للتشغيل: إذا تم حذف `sessions_spawn.runTimeoutSeconds`، يستخدم OpenClaw القيمة `agents.defaults.subagents.runTimeoutSeconds` عند ضبطها؛ وإلا يعود إلى `0` (من دون مهلة).

معاملات الأداة:

- `task` (مطلوب)
- `label?` (اختياري)
- `agentId?` (اختياري؛ شغّل تحت معرّف وكيل آخر إذا كان مسموحًا)
- `model?` (اختياري؛ يتجاوز نموذج الوكيل الفرعي؛ يتم تخطي القيم غير الصالحة ويعمل الوكيل الفرعي على النموذج الافتراضي مع تحذير في نتيجة الأداة)
- `thinking?` (اختياري؛ يتجاوز مستوى التفكير لعملية الوكيل الفرعي)
- `runTimeoutSeconds?` (يفترَض إلى `agents.defaults.subagents.runTimeoutSeconds` عند ضبطها، وإلا `0`؛ وعند ضبطه، تُجهض عملية الوكيل الفرعي بعد N ثانية)
- `thread?` (الافتراضي `false`؛ عندما تكون `true`، يطلب ربط محادثة القناة لجلسة هذا الوكيل الفرعي)
- `mode?` (`run|session`)
  - الافتراضي هو `run`
  - إذا كانت `thread: true` وتم حذف `mode`، يصبح الافتراضي `session`
  - يتطلب `mode: "session"` القيمة `thread: true`
- `cleanup?` (`delete|keep`، الافتراضي `keep`)
- `sandbox?` (`inherit|require`، الافتراضي `inherit`؛ ترفض القيمة `require` التشغيل ما لم يكن وقت تشغيل الطفل الهدف معزولًا)
- لا يقبل `sessions_spawn` معاملات تسليم القناة (`target`، `channel`، `to`، `threadId`، `replyTo`، `transport`). للتسليم، استخدم `message`/`sessions_send` من العملية التي تم تشغيلها.

## الجلسات المرتبطة بالمحادثات

عندما تكون روابط المحادثات مفعّلة لقناة ما، يمكن لوكيل فرعي أن يظل مرتبطًا بمحادثة بحيث تستمر رسائل المستخدم اللاحقة في تلك المحادثة بالتوجيه إلى جلسة الوكيل الفرعي نفسها.

### القنوات الداعمة للمحادثات

- Discord (وهي حاليًا القناة الوحيدة المدعومة): تدعم جلسات الوكيل الفرعي المستمرة المرتبطة بالمحادثات (`sessions_spawn` مع `thread: true`)، وعناصر التحكم اليدوية في المحادثات (`/focus` و`/unfocus` و`/agents` و`/session idle` و`/session max-age`)، ومفاتيح المهايئ `channels.discord.threadBindings.enabled` و`channels.discord.threadBindings.idleHours` و`channels.discord.threadBindings.maxAgeHours` و`channels.discord.threadBindings.spawnSubagentSessions`.

تدفق سريع:

1. شغّل باستخدام `sessions_spawn` مع `thread: true` (واختياريًا `mode: "session"`).
2. ينشئ OpenClaw محادثة أو يربطها بهدف تلك الجلسة في القناة النشطة.
3. تُوجَّه الردود ورسائل المتابعة في تلك المحادثة إلى الجلسة المرتبطة.
4. استخدم `/session idle` لفحص/تحديث إلغاء التركيز التلقائي بسبب عدم النشاط، واستخدم `/session max-age` للتحكم في الحد الأقصى الصارم.
5. استخدم `/unfocus` لفصل الربط يدويًا.

عناصر التحكم اليدوية:

- يربط `/focus <target>` المحادثة الحالية (أو ينشئ واحدة) بهدف وكيل فرعي/جلسة.
- يزيل `/unfocus` الربط للمحادثة الحالية المرتبطة.
- يسرد `/agents` العمليات النشطة وحالة الربط (`thread:<id>` أو `unbound`).
- لا يعمل `/session idle` و`/session max-age` إلا مع المحادثات المرتبطة التي عليها تركيز.

مفاتيح التهيئة:

- الافتراضي العام: `session.threadBindings.enabled` و`session.threadBindings.idleHours` و`session.threadBindings.maxAgeHours`
- تجاوز القناة ومفاتيح الربط التلقائي عند التشغيل خاصة بكل مهايئ. راجع **القنوات الداعمة للمحادثات** أعلاه.

راجع [مرجع التهيئة](/ar/gateway/configuration-reference) و[أوامر الشرطة المائلة](/ar/tools/slash-commands) للحصول على تفاصيل المهايئات الحالية.

قائمة السماح:

- `agents.list[].subagents.allowAgents`: قائمة بمعرّفات الوكلاء التي يمكن استهدافها عبر `agentId` (`["*"]` للسماح بأي وكيل). الافتراضي: وكيل الطالب فقط.
- `agents.defaults.subagents.allowAgents`: قائمة السماح الافتراضية للوكيل الهدف المستخدمة عندما لا يضبط وكيل الطالب `subagents.allowAgents` الخاص به.
- حاجز وراثة العزل: إذا كانت جلسة الطالب معزولة، يرفض `sessions_spawn` الأهداف التي ستعمل من دون عزل.
- `agents.defaults.subagents.requireAgentId` / `agents.list[].subagents.requireAgentId`: عندما تكون true، تُحظر استدعاءات `sessions_spawn` التي تحذف `agentId` (يفرض اختيار ملف تعريف صريح). الافتراضي: false.

الاكتشاف:

- استخدم `agents_list` لمعرفة معرّفات الوكلاء المسموح بها حاليًا لـ `sessions_spawn`.

الأرشفة التلقائية:

- تتم أرشفة جلسات الوكلاء الفرعيين تلقائيًا بعد `agents.defaults.subagents.archiveAfterMinutes` (الافتراضي: 60).
- تستخدم الأرشفة `sessions.delete` وتعيد تسمية النص التفريغي إلى `*.deleted.<timestamp>` (في المجلد نفسه).
- تؤدي `cleanup: "delete"` إلى الأرشفة فورًا بعد الإعلان (مع الاحتفاظ بالنص التفريغي عبر إعادة التسمية).
- الأرشفة التلقائية بأفضل جهد؛ وتُفقد المؤقتات المعلقة إذا أُعيد تشغيل Gateway.
- لا تؤدي `runTimeoutSeconds` إلى الأرشفة التلقائية؛ فهي توقف العملية فقط. تبقى الجلسة حتى الأرشفة التلقائية.
- تنطبق الأرشفة التلقائية بالتساوي على الجلسات ذات العمق 1 والجلسات ذات العمق 2.
- تنظيف المتصفح منفصل عن تنظيف الأرشفة: تُغلق علامات تبويب/عمليات المتصفح المتتبعة بأفضل جهد عند انتهاء العملية، حتى إذا تم الاحتفاظ بسجل الجلسة/النص التفريغي.

## الوكلاء الفرعيون المتداخلون

افتراضيًا، لا يمكن للوكلاء الفرعيين تشغيل وكلاء فرعيين خاصين بهم (`maxSpawnDepth: 1`). يمكنك تمكين مستوى واحد من التداخل بضبط `maxSpawnDepth: 2`، ما يسمح **بنمط المنسّق**: الرئيسي ← وكيل فرعي منسّق ← وكلاء فرعيون فرعيون عاملون.

### كيفية التمكين

```json5
{
  agents: {
    defaults: {
      subagents: {
        maxSpawnDepth: 2, // السماح للوكلاء الفرعيين بتشغيل أطفال (الافتراضي: 1)
        maxChildrenPerAgent: 5, // الحد الأقصى للأطفال النشطين لكل جلسة وكيل (الافتراضي: 5)
        maxConcurrent: 8, // الحد الأقصى العام للتزامن في المسار (الافتراضي: 8)
        runTimeoutSeconds: 900, // المهلة الافتراضية لـ sessions_spawn عند الحذف (0 = بلا مهلة)
      },
    },
  },
}
```

### مستويات العمق

| العمق | شكل مفتاح الجلسة                             | الدور                                         | هل يمكنه التشغيل؟            |
| ----- | -------------------------------------------- | --------------------------------------------- | ---------------------------- |
| 0     | `agent:<id>:main`                            | الوكيل الرئيسي                                 | دائمًا                       |
| 1     | `agent:<id>:subagent:<uuid>`                 | وكيل فرعي (منسّق عندما يكون العمق 2 مسموحًا) | فقط إذا كان `maxSpawnDepth >= 2` |
| 2     | `agent:<id>:subagent:<uuid>:subagent:<uuid>` | وكيل فرعي فرعي (عامل طرفي)                    | أبدًا                        |

### سلسلة الإعلان

تتدفق النتائج عودةً إلى أعلى السلسلة:

1. ينتهي العامل ذو العمق 2 ← ويعلن إلى والده (المنسّق ذو العمق 1)
2. يتلقى المنسّق ذو العمق 1 الإعلان، ويُركّب النتائج، وينتهي ← ثم يعلن إلى الرئيسي
3. يتلقى الوكيل الرئيسي الإعلان ويسلّمه إلى المستخدم

لا يرى كل مستوى إلا الإعلانات القادمة من أطفاله المباشرين.

إرشادات تشغيلية:

- ابدأ عمل الطفل مرة واحدة وانتظر أحداث الاكتمال بدلًا من بناء حلقات استطلاع
  حول `sessions_list` أو `sessions_history` أو `/subagents list` أو
  أوامر `exec` الخاصة بالنوم.
- إذا وصل حدث اكتمال طفل بعد أن تكون قد أرسلت الإجابة النهائية بالفعل،
  فالمتابعة الصحيحة هي الرمز الصامت الدقيق `NO_REPLY` / `no_reply`.

### سياسة الأدوات حسب العمق

- يُكتب الدور ونطاق التحكم في بيانات تعريف الجلسة وقت التشغيل. وهذا يمنع مفاتيح الجلسات المسطّحة أو المستعادة من استعادة امتيازات المنسّق بالخطأ.
- **العمق 1 (منسّق، عندما يكون `maxSpawnDepth >= 2`)**: يحصل على `sessions_spawn` و`subagents` و`sessions_list` و`sessions_history` حتى يتمكن من إدارة أطفاله. وتظل أدوات الجلسة/النظام الأخرى مرفوضة.
- **العمق 1 (طرفي، عندما يكون `maxSpawnDepth == 1`)**: لا توجد أدوات جلسة (السلوك الافتراضي الحالي).
- **العمق 2 (عامل طرفي)**: لا توجد أدوات جلسة — يكون `sessions_spawn` مرفوضًا دائمًا عند العمق 2. ولا يمكنه تشغيل أطفال إضافيين.

### حد التشغيل لكل وكيل

يمكن لكل جلسة وكيل (عند أي عمق) أن يكون لديها بحد أقصى `maxChildrenPerAgent` (الافتراضي: 5) من الأطفال النشطين في الوقت نفسه. وهذا يمنع التفرّع الجامح من منسّق واحد.

### الإيقاف المتسلسل

يؤدي إيقاف منسّق بالعمق 1 تلقائيًا إلى إيقاف جميع أطفاله من العمق 2:

- يؤدي `/stop` في الدردشة الرئيسية إلى إيقاف جميع الوكلاء من العمق 1 وينتقل بالتسلسل إلى أطفالهم من العمق 2.
- يؤدي `/subagents kill <id>` إلى إيقاف وكيل فرعي محدد وينتقل بالتسلسل إلى أطفاله.
- يؤدي `/subagents kill all` إلى إيقاف جميع الوكلاء الفرعيين للطالب وينتقل بالتسلسل.

## المصادقة

تُحل مصادقة الوكيل الفرعي حسب **معرّف الوكيل**، وليس حسب نوع الجلسة:

- مفتاح جلسة الوكيل الفرعي هو `agent:<agentId>:subagent:<uuid>`.
- يُحمَّل مخزن المصادقة من `agentDir` الخاص بذلك الوكيل.
- تُدمج ملفات تعريف مصادقة الوكيل الرئيسي باعتبارها **احتياطيًا**؛ وتلغي ملفات تعريف الوكيل ملفات تعريف الرئيسي عند التعارض.

ملاحظة: الدمج إضافي، لذا تكون ملفات تعريف الرئيسي متاحة دائمًا كاحتياطي. ولا تزال المصادقة المعزولة بالكامل لكل وكيل غير مدعومة.

## الإعلان

تُبلغ الوكلاء الفرعيون مرة أخرى عبر خطوة إعلان:

- تُنفَّذ خطوة الإعلان داخل جلسة الوكيل الفرعي (وليس جلسة الطالب).
- إذا رد الوكيل الفرعي بالنص المطابق تمامًا `ANNOUNCE_SKIP`، فلن يتم نشر أي شيء.
- إذا كان أحدث نص للمساعد هو الرمز الصامت المطابق تمامًا `NO_REPLY` / `no_reply`،
  فسيتم كتم مخرجات الإعلان حتى لو وُجد تقدم مرئي سابق.
- بخلاف ذلك يعتمد التسليم على عمق الطالب:
  - تستخدم جلسات الطالب ذات المستوى الأعلى استدعاء `agent` لاحقًا مع تسليم خارجي (`deliver=true`)
  - تتلقى جلسات الوكيل الفرعي المتداخلة الخاصة بالطالب إدخالًا لاحقًا داخليًا (`deliver=false`) حتى يتمكن المنسّق من تركيب نتائج الأطفال داخل الجلسة
  - إذا كانت جلسة الوكيل الفرعي المتداخلة الخاصة بالطالب قد اختفت، يعود OpenClaw إلى الطالب الخاص بتلك الجلسة عندما يكون متاحًا
- بالنسبة إلى جلسات الطالب ذات المستوى الأعلى، يحل التسليم المباشر في وضع الاكتمال أولاً أي مسار محادثة/محادثة مستمرة مرتبط وأي تجاوز hook، ثم يملأ حقول هدف القناة الناقصة من المسار المخزن لجلسة الطالب. وهذا يُبقي عمليات الاكتمال في الدردشة/الموضوع الصحيحين حتى عندما يحدد أصل الاكتمال القناة فقط.
- يكون تجميع اكتمال الأطفال مقيّدًا بعملية الطالب الحالية عند بناء نتائج الاكتمال المتداخلة، ما يمنع تسرّب مخرجات أطفال قديمة من عمليات سابقة إلى الإعلان الحالي.
- تحافظ ردود الإعلان على توجيه المحادثة/الموضوع عندما يكون متاحًا في مهايئات القنوات.
- يُطبَّع سياق الإعلان إلى كتلة حدث داخلية ثابتة:
  - المصدر (`subagent` أو `cron`)
  - مفتاح/معرّف جلسة الطفل
  - نوع الإعلان + تسمية المهمة
  - سطر حالة مشتق من نتيجة وقت التشغيل (`success` أو `error` أو `timeout` أو `unknown`)
  - محتوى النتيجة المحدد من أحدث نص مرئي للمساعد، وإلا أحدث نص `tool`/`toolResult` مُنقّى؛ العمليات الفاشلة النهائية تُبلغ عن حالة الفشل من دون إعادة تشغيل نص الرد الملتقط
  - تعليمة لاحقة توضّح متى يجب الرد ومتى يجب التزام الصمت
- لا يُستنتج `Status` من مخرجات النموذج؛ بل يأتي من إشارات نتائج وقت التشغيل.
- عند حدوث مهلة، إذا كان الطفل قد أنجز استدعاءات أدوات فقط، يمكن للإعلان اختزال ذلك السجل إلى ملخص قصير عن التقدم الجزئي بدلًا من إعادة تشغيل مخرجات الأدوات الخام.

تتضمن حمولات الإعلان سطر إحصاءات في النهاية (حتى عند الالتفاف):

- وقت التشغيل (مثل `runtime 5m12s`)
- استخدام الرموز (إدخال/إخراج/إجمالي)
- التكلفة المقدّرة عند تهيئة تسعير النموذج (`models.providers.*.models[].cost`)
- `sessionKey` و`sessionId` ومسار النص التفريغي (حتى يتمكن الوكيل الرئيسي من جلب السجل عبر `sessions_history` أو فحص الملف على القرص)
- الغرض من بيانات التعريف الداخلية هو التنسيق فقط؛ ويجب إعادة كتابة الردود الموجهة للمستخدم بصوت مساعد عادي.

يُعد `sessions_history` مسار التنسيق الأكثر أمانًا:

- يُطبَّع استدعاء المساعد أولاً:
  - تُزال وسوم التفكير
  - تُزال كتل السقالات `<relevant-memories>` / `<relevant_memories>`
  - تُزال كتل حمولة XML الخاصة باستدعاءات الأدوات النصية العادية مثل `<tool_call>...</tool_call>`،
    و`<function_call>...</function_call>`، و`<tool_calls>...</tool_calls>`، و
    `<function_calls>...</function_calls>`، بما في ذلك الحمولات المقتطعة
    التي لا تُغلق بشكل صحيح
  - تُزال سقالات استدعاء/نتيجة الأداة المخفّضة وعلامات السياق التاريخي
  - تُزال رموز تحكم النموذج المتسربة مثل `<|assistant|>`، وغيرها من رموز ASCII
    بصيغة `<|...|>`، ومتغيرات العرض الكامل بصيغة `<｜...｜>`
  - يُزال XML المشوّه لاستدعاء أدوات MiniMax
- يُحجَب النص الشبيه ببيانات الاعتماد/الرموز
- يمكن اقتطاع الكتل الطويلة
- يمكن أن تُسقط السجلات الكبيرة جدًا الصفوف الأقدم أو تستبدل صفًا كبيرًا جدًا بـ
  `[sessions_history omitted: message too large]`
- فحص النص التفريغي الخام على القرص هو البديل عندما تحتاج إلى النص التفريغي الكامل بايتًا ببايت

## سياسة الأدوات (أدوات الوكيل الفرعي)

افتراضيًا، يحصل الوكلاء الفرعيون على **جميع الأدوات باستثناء أدوات الجلسة** وأدوات النظام:

- `sessions_list`
- `sessions_history`
- `sessions_send`
- `sessions_spawn`

يظل `sessions_history` هنا أيضًا عرض استدعاء مقيّدًا ومنقّى؛ وليس
تفريغًا خامًا للنص التفريغي.

عندما يكون `maxSpawnDepth >= 2`، تحصل الوكلاء الفرعيون المنسّقون من العمق 1 بالإضافة إلى ذلك على `sessions_spawn` و`subagents` و`sessions_list` و`sessions_history` حتى يتمكنوا من إدارة أطفالهم.

يمكن التجاوز عبر التهيئة:

```json5
{
  agents: {
    defaults: {
      subagents: {
        maxConcurrent: 1,
      },
    },
  },
  tools: {
    subagents: {
      tools: {
        // الرفض له الأولوية
        deny: ["gateway", "cron"],
        // إذا تم ضبط allow، فإنه يصبح سماحًا فقط (ويظل deny له الأولوية)
        // allow: ["read", "exec", "process"]
      },
    },
  },
}
```

## التزامن

تستخدم الوكلاء الفرعيون مسار طابور مخصصًا داخل العملية:

- اسم المسار: `subagent`
- التزامن: `agents.defaults.subagents.maxConcurrent` (الافتراضي `8`)

## الإيقاف

- يؤدي إرسال `/stop` في دردشة الطالب إلى إجهاض جلسة الطالب وإيقاف أي عمليات وكيل فرعي نشطة تم تشغيلها منها، مع الانتقال بالتسلسل إلى الأطفال المتداخلين.
- يؤدي `/subagents kill <id>` إلى إيقاف وكيل فرعي محدد وينتقل بالتسلسل إلى أطفاله.

## القيود

- إعلان الوكيل الفرعي يتم **بأفضل جهد**. إذا أُعيد تشغيل Gateway، فسيضيع عمل "الإعلان مرة أخرى" المعلق.
- لا تزال الوكلاء الفرعيون تتشارك موارد عملية Gateway نفسها؛ لذا تعامل مع `maxConcurrent` باعتباره صمام أمان.
- يكون `sessions_spawn` دائمًا غير حاجب: إذ يعيد `{ status: "accepted", runId, childSessionKey }` فورًا.
- لا يحقن سياق الوكيل الفرعي إلا `AGENTS.md` و`TOOLS.md` (ولا يحقن `SOUL.md` أو `IDENTITY.md` أو `USER.md` أو `HEARTBEAT.md` أو `BOOTSTRAP.md`).
- الحد الأقصى لعمق التداخل هو 5 (نطاق `maxSpawnDepth`: من 1 إلى 5). ويوصى بالعمق 2 لمعظم حالات الاستخدام.
- يحدد `maxChildrenPerAgent` سقف الأطفال النشطين لكل جلسة (الافتراضي: 5، النطاق: من 1 إلى 20).

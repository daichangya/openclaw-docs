يمثل "الاقتران" في OpenClaw خطوة **موافقة صريحة من المالك**.
ويُستخدم في موضعين:

1. **اقتران الرسائل الخاصة** (من يُسمح له بالتحدث إلى الروبوت)
2. **اقتران Node** (أي الأجهزة/العُقد يُسمح لها بالانضمام إلى شبكة Gateway)

سياق الأمان: [الأمان](/ar/gateway/security)

## 1) اقتران الرسائل الخاصة (الوصول إلى الدردشة الواردة)

عندما تُهيأ قناة بسياسة رسائل خاصة `pairing`، يحصل المرسلون غير المعروفين على رمز قصير ولا **تُعالج** رسالتهم حتى توافق أنت عليها.

سياسات الرسائل الخاصة الافتراضية موثقة في: [الأمان](/ar/gateway/security)

رموز الاقتران:

- 8 أحرف، كبيرة، من دون أحرف ملتبسة (`0O1I`).
- **تنتهي صلاحيتها بعد ساعة واحدة**. لا يرسل الروبوت رسالة الاقتران إلا عند إنشاء طلب جديد (تقريبًا مرة كل ساعة لكل مرسل).
- يُحدَّد الحد الأقصى لطلبات اقتران الرسائل الخاصة المعلقة إلى **3 لكل قناة** افتراضيًا؛ وتُتجاهل الطلبات الإضافية حتى تنتهي صلاحية أحدها أو تتم الموافقة عليه.

### الموافقة على مرسل

```bash
openclaw pairing list telegram
openclaw pairing approve telegram <CODE>
```

القنوات المدعومة: `bluebubbles`, `discord`, `feishu`, `googlechat`, `imessage`, `irc`, `line`, `matrix`, `mattermost`, `msteams`, `nextcloud-talk`, `nostr`, `openclaw-weixin`, `signal`, `slack`, `synology-chat`, `telegram`, `twitch`, `whatsapp`, `zalo`, `zalouser`.

### مكان وجود الحالة

تُخزن ضمن `~/.openclaw/credentials/`:

- الطلبات المعلقة: `<channel>-pairing.json`
- مخزن قائمة السماح المعتمدة:
  - الحساب الافتراضي: `<channel>-allowFrom.json`
  - الحساب غير الافتراضي: `<channel>-<accountId>-allowFrom.json`

سلوك نطاق الحساب:

- تقرأ الحسابات غير الافتراضية وتكتب فقط في ملف قائمة السماح الخاص بنطاقها.
- يستخدم الحساب الافتراضي ملف قائمة السماح غير المقيّد بنطاق على مستوى القناة.

تعامل مع هذه الملفات على أنها حساسة (فهي تتحكم في الوصول إلى مساعدك).

مهم: هذا المخزن مخصص للوصول إلى الرسائل الخاصة. أما تفويض المجموعات فهو منفصل.
فالموافقة على رمز اقتران رسالة خاصة لا تسمح تلقائيًا لذلك المرسل بتشغيل أوامر المجموعات أو التحكم في الروبوت داخل المجموعات. للوصول إلى المجموعات، هيّئ قوائم السماح الصريحة للمجموعات في القناة (مثل `groupAllowFrom` أو `groups` أو تجاوزات كل مجموعة/كل موضوع بحسب القناة).

## 2) اقتران أجهزة Node (عُقد iOS/Android/macOS/عديمة الواجهة)

تتصل Node بـ Gateway على أنها **أجهزة** بدور `role: node`. وينشئ Gateway
طلب اقتران جهاز يجب الموافقة عليه.

### الاقتران عبر Telegram (موصى به لنظام iOS)

إذا كنت تستخدم Plugin ‏`device-pair`، يمكنك إجراء اقتران الجهاز لأول مرة بالكامل من خلال Telegram:

1. في Telegram، أرسل إلى الروبوت: `/pair`
2. يرد الروبوت برسالتين: رسالة تعليمات ورسالة منفصلة تحتوي على **رمز إعداد** (ليسهل نسخه/لصقه في Telegram).
3. على هاتفك، افتح تطبيق OpenClaw على iOS ← الإعدادات ← Gateway.
4. الصق رمز الإعداد واتصل.
5. ارجع إلى Telegram: `/pair pending` (راجع معرّفات الطلبات، والدور، والنطاقات)، ثم وافق.

رمز الإعداد هو حمولة JSON بترميز base64 تحتوي على:

- `url`: عنوان WebSocket الخاص بـ Gateway (`ws://...` أو `wss://...`)
- `bootstrapToken`: رمز bootstrap قصير العمر لجهاز واحد يُستخدم في المصافحة الأولية للاقتران

يحمل رمز bootstrap هذا ملف bootstrap المضمّن للاقتران:

- يظل رمز `node` الأساسي المُسلَّم `scopes: []`
- يظل أي رمز `operator` مُسلَّم مقيدًا بقائمة السماح الخاصة بـ bootstrap:
  `operator.approvals`, `operator.read`, `operator.talk.secrets`, `operator.write`
- تكون عمليات التحقق من نطاق bootstrap مسبوقة بالدور، وليست تجمع نطاقات مسطحًا واحدًا:
  فإدخالات نطاق `operator` لا تلبّي إلا طلبات `operator`، ويجب على الأدوار غير `operator`
  أن تطلب النطاقات تحت بادئة دورها الخاص

تعامل مع رمز الإعداد مثل كلمة مرور ما دام صالحًا.

### الموافقة على جهاز Node

```bash
openclaw devices list
openclaw devices approve <requestId>
openclaw devices reject <requestId>
```

إذا أعاد الجهاز نفسه المحاولة بتفاصيل مصادقة مختلفة (مثل
دور/نطاقات/مفتاح عام مختلف)، يُستبدل الطلب المعلق السابق ويُنشأ
`requestId` جديد.

مهم: الجهاز المقترن بالفعل لا يحصل على وصول أوسع بصمت. إذا
أعاد الاتصال طالبًا مزيدًا من النطاقات أو دورًا أوسع، فإن OpenClaw يُبقي
الموافقة الحالية كما هي وينشئ طلب ترقية معلقًا جديدًا. استخدم
`openclaw devices list` لمقارنة الوصول المعتمد حاليًا مع الوصول
المطلوب حديثًا قبل أن توافق.

### الموافقة التلقائية الاختيارية لـ Node ضمن CIDR موثوق

يبقى اقتران الأجهزة يدويًا افتراضيًا. وبالنسبة إلى شبكات Node المحكمة التحكم،
يمكنك تفعيل الموافقة التلقائية لأول اقتران Node بشكل صريح باستخدام CIDR أو عناوين IP مطابقة تمامًا:

```json5
{
  gateway: {
    nodes: {
      pairing: {
        autoApproveCidrs: ["192.168.1.0/24"],
      },
    },
  },
}
```

ينطبق هذا فقط على طلبات الاقتران الجديدة `role: node` التي لا تطلب
أي نطاقات. ولا يزال يتطلب عملاء operator وbrowser وControl UI وWebChat
موافقة يدوية. كما تتطلب تغييرات الدور والنطاق والبيانات الوصفية والمفتاح العام
موافقة يدوية أيضًا.

### تخزين حالة اقتران Node

تُخزن ضمن `~/.openclaw/devices/`:

- `pending.json` (قصير العمر؛ تنتهي صلاحية الطلبات المعلقة)
- `paired.json` (الأجهزة المقترنة + الرموز)

### ملاحظات

- تُعد واجهة API القديمة `node.pair.*` ‏(CLI: `openclaw nodes pending|approve|reject|rename`) مخزن اقتران منفصلًا مملوكًا لـ Gateway. وما زالت Node عبر WS تتطلب اقتران الأجهزة.
- يمثل سجل الاقتران مصدر الحقيقة الدائم للأدوار المعتمدة. وتظل
  رموز الأجهزة النشطة مقيدة بمجموعة الأدوار المعتمدة تلك؛ ولا يؤدي وجود
  إدخال رمز شارد خارج الأدوار المعتمدة إلى إنشاء وصول جديد.

# الأمان

<Warning>
**نموذج الثقة للمساعد الشخصي:** يفترض هذا الإرشاد وجود حد مشغّل موثوق واحد لكل Gateway (نموذج المستخدم الواحد/المساعد الشخصي).
لا يُعد OpenClaw **حدًا أمنيًا متعدد المستأجرين ومعاديًا** لعدة مستخدمين عدائيين يشاركون وكيلاً/Gateway واحدًا.
إذا كنت تحتاج إلى تشغيل بثقة مختلطة أو مع مستخدمين عدائيين، فقسّم حدود الثقة (Gateway + بيانات اعتماد منفصلة، ومن الأفضل أيضًا مستخدمو/مضيفو نظام تشغيل منفصلون).
</Warning>

## أولًا تحديد النطاق: نموذج أمان المساعد الشخصي

يفترض إرشاد الأمان في OpenClaw نشرًا على هيئة **مساعد شخصي**: حد مشغّل موثوق واحد، مع إمكانية وجود عدة وكلاء.

- وضعية الأمان المدعومة: مستخدم واحد/حد ثقة واحد لكل Gateway (ويُفضل مستخدم نظام تشغيل/مضيف/VPS واحد لكل حد).
- ما لا يُعد حدًا أمنيًا مدعومًا: Gateway/وكيل واحد مشترك يستخدمه مستخدمون غير موثوقين أو عدائيون بشكل متبادل.
- إذا كان عزل المستخدمين العدائيين مطلوبًا، فاقسم حسب حدود الثقة (Gateway + بيانات اعتماد منفصلة، ويفضل أيضًا مستخدمو/مضيفو نظام تشغيل منفصلون).
- إذا كان عدة مستخدمين غير موثوقين قادرين على مراسلة وكيل مفعّل بالأدوات، فاعتبرهم يشاركون سلطة الأدوات المفوضة نفسها لذلك الوكيل.

تشرح هذه الصفحة أساليب التقوية **ضمن هذا النموذج**. وهي لا تدّعي وجود عزل عدائي متعدد المستأجرين على Gateway مشترك واحد.

## فحص سريع: `openclaw security audit`

راجع أيضًا: [التحقق الصوري (نماذج الأمان)](/ar/security/formal-verification)

شغّل هذا بانتظام (خصوصًا بعد تغيير التكوين أو كشف أسطح الشبكة):

```bash
openclaw security audit
openclaw security audit --deep
openclaw security audit --fix
openclaw security audit --json
```

يبقى `security audit --fix` محدودًا عمدًا: فهو يحول سياسات المجموعات المفتوحة الشائعة
إلى قوائم سماح، ويستعيد `logging.redactSensitive: "tools"`، ويشدد
أذونات الحالة/التكوين/ملفات التضمين، ويستخدم إعادة تعيين ACL في Windows بدل
POSIX `chmod` عند التشغيل على Windows.

ويكشف عن مواطن الخطأ الشائعة (كشف مصادقة Gateway، وكشف التحكم في المتصفح، وقوائم السماح المرتفعة، وأذونات نظام الملفات، وموافقات التنفيذ المتساهلة، وكشف أدوات القنوات المفتوحة).

OpenClaw منتج وتجربة في آن واحد: فأنت تصل سلوك النماذج المتقدمة بأسطح مراسلة حقيقية وأدوات حقيقية. **لا توجد بيئة “آمنة بالكامل”.** الهدف هو أن تكون متعمدًا بشأن:

- من يمكنه التحدث إلى روبوتك
- أين يُسمح للروبوت بالتصرف
- ما الذي يمكن للروبوت الوصول إليه

ابدأ بأقل قدر من الوصول الذي لا يزال ينجح، ثم وسّعه مع ازدياد ثقتك.

### النشر والثقة في المضيف

يفترض OpenClaw أن المضيف وحد التكوين موثوقان:

- إذا كان بإمكان شخص ما تعديل حالة/تكوين مضيف Gateway (`~/.openclaw`، بما في ذلك `openclaw.json`) فاعتبره مشغّلًا موثوقًا.
- تشغيل Gateway واحد لعدة مشغّلين غير موثوقين/عدائيين بشكل متبادل **ليس إعدادًا موصى به**.
- بالنسبة إلى الفرق ذات الثقة المختلطة، قسّم حدود الثقة باستخدام Gateways منفصلة (أو على الأقل مستخدمي/مضيفي نظام تشغيل منفصلين).
- الإعداد الافتراضي الموصى به: مستخدم واحد لكل جهاز/مضيف (أو VPS)، وGateway واحد لذلك المستخدم، ووكيل واحد أو أكثر داخل ذلك Gateway.
- داخل مثيل Gateway واحد، يكون وصول المشغّل الموثق دورًا موثوقًا في مستوى التحكم، وليس دور مستأجر لكل مستخدم.
- معرّفات الجلسات (`sessionKey`، ومعرّفات الجلسات، والملصقات) هي محددات توجيه وليست رموز تخويل.
- إذا كان بإمكان عدة أشخاص مراسلة وكيل مفعّل بالأدوات، فيمكن لكل واحد منهم توجيه مجموعة الأذونات نفسها. يساعد عزل الجلسة/الذاكرة لكل مستخدم في الخصوصية، لكنه لا يحوّل الوكيل المشترك إلى تخويل مضيف لكل مستخدم.

### مساحة عمل Slack مشتركة: خطر حقيقي

إذا كان "بإمكان الجميع في Slack مراسلة الروبوت"، فإن الخطر الأساسي هو سلطة الأدوات المفوضة:

- يمكن لأي مرسل مسموح له تحفيز استدعاءات الأدوات (`exec`، والمتصفح، وأدوات الشبكة/الملفات) ضمن سياسة الوكيل؛
- يمكن لحقن prompt/المحتوى من مرسل واحد أن يسبب إجراءات تؤثر في الحالة المشتركة أو الأجهزة أو المخرجات؛
- إذا كان لدى وكيل مشترك واحد بيانات اعتماد/ملفات حساسة، فقد يتمكن أي مرسل مسموح له من توجيه تسريبها عبر استخدام الأدوات.

استخدم وكلاء/Gateways منفصلين مع أقل عدد من الأدوات لتدفقات عمل الفرق؛ واحتفظ بوكلاء البيانات الشخصية بشكل خاص.

### وكيل مشترك على مستوى الشركة: نمط مقبول

يُعد هذا مقبولًا عندما يكون كل من يستخدم ذلك الوكيل ضمن حد الثقة نفسه (على سبيل المثال فريق واحد داخل شركة) ويكون نطاق الوكيل تجاريًا فقط بشكل صارم.

- شغّله على جهاز/VM/حاوية مخصصة؛
- استخدم مستخدم نظام تشغيل مخصصًا + متصفحًا/ملفًا شخصيًا/حسابات مخصصة لذلك وقت التشغيل؛
- لا تسجّل دخول ذلك وقت التشغيل إلى حسابات Apple/Google الشخصية أو ملفات المتصفح/مدير كلمات المرور الشخصية.

إذا خلطت بين الهويات الشخصية وهوية الشركة في وقت التشغيل نفسه، فإنك تلغي الفصل وتزيد خطر كشف البيانات الشخصية.

## مفهوم الثقة بين Gateway وNode

تعامل مع Gateway وNode باعتبارهما نطاق ثقة واحدًا للمشغّل، ولكن بأدوار مختلفة:

- **Gateway** هو مستوى التحكم وسطح السياسة (`gateway.auth`، وسياسة الأدوات، والتوجيه).
- **Node** هو سطح التنفيذ البعيد المقترن بذلك Gateway (الأوامر، وإجراءات الأجهزة، والقدرات المحلية على المضيف).
- المتصل الذي تتم مصادقته لدى Gateway يكون موثوقًا على مستوى Gateway. وبعد الاقتران، تصبح إجراءات node إجراءات مشغّل موثوقة على ذلك node.
- `sessionKey` هو اختيار للتوجيه/السياق، وليس مصادقة لكل مستخدم.
- موافقات التنفيذ (قائمة السماح + السؤال) هي حواجز حماية لنية المشغّل، وليست عزلًا عدائيًا متعدد المستأجرين.
- الإعداد الافتراضي للمنتج في OpenClaw لبيئات المشغّل الواحد الموثوق هو أن تنفيذ المضيف على `gateway`/`node` مسموح من دون مطالبات موافقة (`security="full"` و`ask="off"` ما لم تشدده). هذا الإعداد الافتراضي مقصود من ناحية تجربة الاستخدام، وليس ثغرة بحد ذاته.
- تربط موافقات التنفيذ سياق الطلب الدقيق ومعاملات الملفات المحلية المباشرة بأفضل جهد. وهي لا تمثل دلاليًا كل مسارات المحمّل الخاصة بوقت التشغيل/المفسر. استخدم Sandbox وعزل المضيف لحدود قوية.

إذا كنت تحتاج إلى عزل مستخدمين عدائيين، فقسّم حدود الثقة حسب مستخدم/مضيف نظام التشغيل وشغّل Gateways منفصلة.

## مصفوفة حدود الثقة

استخدم هذا كنموذج سريع عند تقييم المخاطر:

| الحد أو آلية التحكم                                         | ما الذي يعنيه                                     | سوء الفهم الشائع                                                                |
| ----------------------------------------------------------- | ------------------------------------------------- | ------------------------------------------------------------------------------- |
| `gateway.auth` (token/password/trusted-proxy/device auth)   | يصادق المتصلين على واجهات Gateway API             | "يحتاج إلى تواقيع لكل رسالة في كل إطار ليكون آمنًا"                            |
| `sessionKey`                                                | مفتاح توجيه لاختيار السياق/الجلسة                | "مفتاح الجلسة هو حد مصادقة للمستخدم"                                            |
| حواجز حماية prompt/المحتوى                                  | تقلل خطر إساءة استخدام النموذج                   | "حقن prompt وحده يثبت تجاوز المصادقة"                                           |
| `canvas.eval` / browser evaluate                            | قدرة مقصودة للمشغّل عند تفعيلها                  | "أي بدائية JS eval هي تلقائيًا ثغرة ضمن نموذج الثقة هذا"                        |
| `!` shell المحلي في TUI                                     | تنفيذ محلي يُحفَّز صراحةً من قبل المشغّل         | "أمر shell المحلي المريح هو حقن عن بُعد"                                        |
| اقتران Node وأوامر Node                                     | تنفيذ بعيد على مستوى المشغّل على أجهزة مقترنة    | "يجب التعامل مع التحكم البعيد في الجهاز على أنه وصول مستخدم غير موثوق افتراضيًا" |

## ليست ثغرات بحكم التصميم

تُبلَّغ هذه الأنماط كثيرًا وعادةً ما تُغلق من دون إجراء ما لم يظهر تجاوز حقيقي لحدود الثقة:

- سلاسل تعتمد فقط على حقن prompt من دون تجاوز للسياسة/المصادقة/Sandbox.
- الادعاءات التي تفترض تشغيلًا عدائيًا متعدد المستأجرين على مضيف/تكوين مشترك واحد.
- الادعاءات التي تصنف وصول القراءة الطبيعي للمشغّل (مثل `sessions.list`/`sessions.preview`/`chat.history`) على أنه IDOR في إعداد Gateway مشترك.
- النتائج الخاصة بالنشر المحلي فقط على localhost (مثل HSTS على Gateway يعمل على loopback فقط).
- نتائج توقيع Webhook الواردة في Discord لمسارات واردة غير موجودة في هذا المستودع.
- التقارير التي تتعامل مع بيانات اقتران node الوصفية كطبقة موافقة ثانية مخفية لكل أمر لـ `system.run`، بينما يظل حد التنفيذ الحقيقي هو سياسة الأوامر العامة الخاصة بـ node في Gateway بالإضافة إلى موافقات التنفيذ الخاصة بـ node نفسه.
- نتائج "غياب التخويل لكل مستخدم" التي تتعامل مع `sessionKey` على أنه token مصادقة.

## خط أساس محصّن خلال 60 ثانية

استخدم هذا الخط الأساسي أولًا، ثم أعد تمكين الأدوات بشكل انتقائي لكل وكيل موثوق:

```json5
{
  gateway: {
    mode: "local",
    bind: "loopback",
    auth: { mode: "token", token: "replace-with-long-random-token" },
  },
  session: {
    dmScope: "per-channel-peer",
  },
  tools: {
    profile: "messaging",
    deny: ["group:automation", "group:runtime", "group:fs", "sessions_spawn", "sessions_send"],
    fs: { workspaceOnly: true },
    exec: { security: "deny", ask: "always" },
    elevated: { enabled: false },
  },
  channels: {
    whatsapp: { dmPolicy: "pairing", groups: { "*": { requireMention: true } } },
  },
}
```

يبقي هذا Gateway محليًا فقط، ويعزل الرسائل المباشرة، ويعطّل أدوات مستوى التحكم/وقت التشغيل افتراضيًا.

## قاعدة سريعة للبريد المشترك

إذا كان بإمكان أكثر من شخص واحد إرسال رسائل مباشرة إلى الروبوت:

- اضبط `session.dmScope: "per-channel-peer"` (أو `"per-account-channel-peer"` للقنوات متعددة الحسابات).
- أبقِ `dmPolicy: "pairing"` أو قوائم السماح الصارمة.
- لا تجمع أبدًا بين الرسائل المباشرة المشتركة والوصول الواسع إلى الأدوات.
- يعزز هذا صناديق البريد المشتركة/التعاونية، لكنه ليس مصممًا كعزل عدائي للمستأجرين المشاركين في وصول الكتابة إلى المضيف/التكوين.

## نموذج رؤية السياق

يفصل OpenClaw بين مفهومين:

- **تخويل التحفيز**: من يمكنه تحفيز الوكيل (`dmPolicy` و`groupPolicy` وقوائم السماح وبوابات الإشارات).
- **رؤية السياق**: ما السياق الإضافي الذي يُحقن في إدخال النموذج (نص الرد، والنص المقتبس، وسجل الخيط، وبيانات إعادة التوجيه الوصفية).

تتحكم قوائم السماح في التحفيز وتخويل الأوامر. أما الإعداد `contextVisibility` فيتحكم في كيفية تصفية السياق الإضافي (الردود المقتبسة، وجذور الخيوط، والسجل الذي تم جلبه):

- `contextVisibility: "all"` (الافتراضي) يحتفظ بالسياق الإضافي كما تم استلامه.
- `contextVisibility: "allowlist"` يصفّي السياق الإضافي إلى المرسلين المسموح لهم بواسطة فحوصات قائمة السماح النشطة.
- `contextVisibility: "allowlist_quote"` يعمل مثل `allowlist`، لكنه يحتفظ مع ذلك برد مقتبس صريح واحد.

اضبط `contextVisibility` لكل قناة أو لكل غرفة/محادثة. راجع [الدردشات الجماعية](/ar/channels/groups#context-visibility-and-allowlists) للحصول على تفاصيل الإعداد.

إرشادات تقييم التنبيهات:

- الادعاءات التي تُظهر فقط أن "النموذج يمكنه رؤية نص مقتبس أو تاريخي من مرسلين غير موجودين في قائمة السماح" هي نتائج تقوية يمكن معالجتها عبر `contextVisibility`، وليست بحد ذاتها تجاوزًا لحدود المصادقة أو Sandbox.
- لكي تكون مؤثرة أمنيًا، لا تزال التقارير تحتاج إلى تجاوز مثبت لحدود الثقة (المصادقة، أو السياسة، أو Sandbox، أو الموافقة، أو حد موثق آخر).

## ما الذي يتحقق منه التدقيق (بشكل عام)

- **الوصول الوارد** (سياسات الرسائل المباشرة، وسياسات المجموعات، وقوائم السماح): هل يمكن للغرباء تحفيز الروبوت؟
- **نطاق تأثير الأدوات** (الأدوات المرتفعة + الغرف المفتوحة): هل يمكن أن يتحول حقن prompt إلى إجراءات shell/ملفات/شبكة؟
- **انحراف موافقات التنفيذ** (`security=full`، و`autoAllowSkills`، وقوائم سماح المفسرات من دون `strictInlineEval`): هل لا تزال حواجز حماية تنفيذ المضيف تقوم بما تعتقده؟
  - `security="full"` هو تحذير عام متعلق بالوضعية، وليس دليلًا على وجود خطأ. إنه الإعداد الافتراضي المختار لبيئات المساعد الشخصي الموثوقة؛ شدده فقط عندما يتطلب نموذج التهديد لديك حواجز حماية قائمة على الموافقة أو قائمة السماح.
- **كشف الشبكة** (ربط Gateway/المصادقة، وTailscale Serve/Funnel، ورموز المصادقة الضعيفة/القصيرة).
- **كشف التحكم في المتصفح** (العُقد البعيدة، ومنافذ relay، ونقاط نهاية CDP البعيدة).
- **نظافة القرص المحلي** (الأذونات، والروابط الرمزية، وتضمينات التكوين، ومسارات “المجلد المتزامن”).
- **Plugins** (يتم تحميل Plugins من دون قائمة سماح صريحة).
- **انحراف السياسة/سوء التهيئة** (إعدادات Sandbox Docker مضبوطة لكن وضع Sandbox متوقف؛ أنماط `gateway.nodes.denyCommands` غير الفعالة لأن المطابقة تكون على اسم الأمر الدقيق فقط مثل `system.run` ولا تفحص نص shell؛ وإدخالات `gateway.nodes.allowCommands` الخطرة؛ و`tools.profile="minimal"` العام الذي تتجاوزه ملفات التعريف لكل وكيل؛ وأدوات مملوكة لـ Plugin يمكن الوصول إليها تحت سياسة أدوات متساهلة).
- **انحراف توقعات وقت التشغيل** (على سبيل المثال افتراض أن التنفيذ الضمني لا يزال يعني `sandbox` عندما أصبح الافتراضي لـ `tools.exec.host` هو `auto`، أو ضبط `tools.exec.host="sandbox"` صراحةً بينما وضع Sandbox متوقف).
- **نظافة النماذج** (تحذير عندما تبدو النماذج المكوّنة قديمة؛ وليس حظرًا صارمًا).

إذا شغّلت `--deep`، يحاول OpenClaw أيضًا إجراء فحص حي لـ Gateway بأفضل جهد.

## خريطة تخزين بيانات الاعتماد

استخدم هذه الخريطة عند تدقيق الوصول أو اتخاذ قرار بشأن ما يجب نسخه احتياطيًا:

- **WhatsApp**: `~/.openclaw/credentials/whatsapp/<accountId>/creds.json`
- **رمز Telegram bot**: التكوين/env أو `channels.telegram.tokenFile` (ملف عادي فقط؛ تُرفض الروابط الرمزية)
- **رمز Discord bot**: التكوين/env أو SecretRef (موفّرات env/file/exec)
- **رموز Slack**: التكوين/env (`channels.slack.*`)
- **قوائم سماح الاقتران**:
  - `~/.openclaw/credentials/<channel>-allowFrom.json` (الحساب الافتراضي)
  - `~/.openclaw/credentials/<channel>-<accountId>-allowFrom.json` (الحسابات غير الافتراضية)
- **ملفات تعريف مصادقة النموذج**: `~/.openclaw/agents/<agentId>/agent/auth-profiles.json`
- **حمولة الأسرار المعتمدة على الملفات (اختيارية)**: `~/.openclaw/secrets.json`
- **استيراد OAuth القديم**: `~/.openclaw/credentials/oauth.json`

## قائمة مراجعة تدقيق الأمان

عندما يطبع التدقيق نتائج، تعامل مع هذا باعتباره ترتيب الأولويات:

1. **أي شيء “مفتوح” + أدوات مفعلة**: أغلق الرسائل المباشرة/المجموعات أولًا (الاقتران/قوائم السماح)، ثم شدد سياسة الأدوات/Sandbox.
2. **كشف الشبكة العامة** (ربط LAN، Funnel، غياب المصادقة): أصلحه فورًا.
3. **كشف التحكم البعيد في المتصفح**: تعامل معه كأنه وصول مشغّل (tailnet فقط، وأقرِن العُقد عمدًا، وتجنب الكشف العام).
4. **الأذونات**: تأكد من أن الحالة/التكوين/بيانات الاعتماد/ملفات المصادقة ليست قابلة للقراءة من المجموعة أو العموم.
5. **Plugins**: حمّل فقط ما تثق به صراحةً.
6. **اختيار النموذج**: فضّل النماذج الحديثة المقواة ضد التعليمات لأي روبوت مزود بالأدوات.

## مسرد تدقيق الأمان

يُفهرس كل اكتشاف في التدقيق بواسطة `checkId` منظم (على سبيل المثال
`gateway.bind_no_auth` أو `tools.exec.security_full_configured`). الفئات الشائعة ذات الخطورة الحرجة:

- `fs.*` — أذونات نظام الملفات على الحالة، والتكوين، وبيانات الاعتماد، وملفات تعريف المصادقة.
- `gateway.*` — وضع الربط، والمصادقة، وTailscale، وControl UI، وإعداد trusted-proxy.
- `hooks.*` و`browser.*` و`sandbox.*` و`tools.exec.*` — التقوية الخاصة بكل سطح.
- `plugins.*` و`skills.*` — سلسلة توريد Plugin/Skill ونتائج الفحص.
- `security.exposure.*` — فحوصات عابرة للجوانب حيث تلتقي سياسة الوصول مع نطاق تأثير الأدوات.

راجع الفهرس الكامل مع مستويات الخطورة ومفاتيح الإصلاح ودعم الإصلاح التلقائي في
[فحوصات تدقيق الأمان](/ar/gateway/security/audit-checks).

## Control UI عبر HTTP

تحتاج Control UI إلى **سياق آمن** (HTTPS أو localhost) لإنشاء هوية الجهاز.
`gateway.controlUi.allowInsecureAuth` هو مفتاح توافق محلي:

- على localhost، يسمح بمصادقة Control UI من دون هوية جهاز عندما تكون الصفحة
  محمّلة عبر HTTP غير آمن.
- لا يتجاوز فحوصات الاقتران.
- لا يخفف متطلبات هوية الجهاز البعيدة (غير localhost).

يفضل استخدام HTTPS (Tailscale Serve) أو فتح الواجهة على `127.0.0.1`.

في حالات الطوارئ فقط، يعطّل `gateway.controlUi.dangerouslyDisableDeviceAuth`
فحوصات هوية الجهاز بالكامل. وهذا خفض شديد في مستوى الأمان؛
أبقِه معطلًا إلا إذا كنت تصحح خطأً بنشاط ويمكنك التراجع بسرعة.

وبشكل منفصل عن هذه العلامات الخطرة، يمكن لنجاح `gateway.auth.mode: "trusted-proxy"`
أن يسمح بجلسات Control UI الخاصة بـ **المشغّل** من دون هوية جهاز. هذا
سلوك مقصود لوضع المصادقة، وليس اختصارًا من `allowInsecureAuth`، كما أنه
لا يمتد إلى جلسات Control UI الخاصة بدور node.

يحذر `openclaw security audit` عند تفعيل هذا الإعداد.

## تكوين Reverse Proxy

إذا كنت تشغّل Gateway خلف reverse proxy (مثل nginx أو Caddy أو Traefik أو غيرها)، فاضبط
`gateway.trustedProxies` للتعامل الصحيح مع عنوان IP الخاص بالعميل المُمرَّر.

عندما يكتشف Gateway ترويسات proxy من عنوان **غير** موجود في `trustedProxies`، فلن
يعامل الاتصالات على أنها عملاء محليون. وإذا كانت مصادقة gateway معطلة، فسيتم رفض
تلك الاتصالات. يمنع هذا تجاوز المصادقة حيث كانت الاتصالات الممررة عبر proxy ستبدو
وكأنها قادمة من localhost وتحصل على ثقة تلقائية.

يغذي `gateway.trustedProxies` أيضًا `gateway.auth.mode: "trusted-proxy"`، لكن هذا
الوضع للمصادقة أكثر صرامة:

- تفشل مصادقة trusted-proxy **بشكل مغلق على proxyات ذات مصدر loopback**
- لا يزال بإمكان reverse proxyات ذات loopback على المضيف نفسه استخدام `gateway.trustedProxies` لاكتشاف العملاء المحليين والتعامل مع عناوين IP المُمرَّرة
- بالنسبة إلى reverse proxyات ذات loopback على المضيف نفسه، استخدم مصادقة token/password بدلًا من `gateway.auth.mode: "trusted-proxy"`

```yaml
gateway:
  trustedProxies:
    - "10.0.0.1" # عنوان IP الخاص بـ reverse proxy
  # اختياري. الافتراضي false.
  # فعّله فقط إذا كان proxy لديك لا يستطيع توفير X-Forwarded-For.
  allowRealIpFallback: false
  auth:
    mode: password
    password: ${OPENCLAW_GATEWAY_PASSWORD}
```

عندما يكون `trustedProxies` مضبوطًا، يستخدم Gateway الترويسة `X-Forwarded-For` لتحديد عنوان IP الخاص بالعميل. ويتم تجاهل `X-Real-IP` افتراضيًا ما لم يتم ضبط `gateway.allowRealIpFallback: true` صراحةً.

سلوك reverse proxy الجيد (الكتابة فوق ترويسات التمرير الواردة):

```nginx
proxy_set_header X-Forwarded-For $remote_addr;
proxy_set_header X-Real-IP $remote_addr;
```

سلوك reverse proxy السيئ (إلحاق/الحفاظ على ترويسات تمرير غير موثوقة):

```nginx
proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
```

## ملاحظات HSTS وorigin

- يعمل OpenClaw gateway محليًا/على loopback أولًا. إذا أنهيت TLS عند reverse proxy، فاضبط HSTS على نطاق HTTPS المواجه لـ proxy هناك.
- إذا كان gateway نفسه هو من ينهي HTTPS، فيمكنك ضبط `gateway.http.securityHeaders.strictTransportSecurity` لإخراج ترويسة HSTS من استجابات OpenClaw.
- يوجد إرشاد نشر مفصل في [مصادقة Trusted Proxy](/ar/gateway/trusted-proxy-auth#tls-termination-and-hsts).
- في عمليات نشر Control UI غير المعتمدة على loopback، تكون `gateway.controlUi.allowedOrigins` مطلوبة افتراضيًا.
- تمثل `gateway.controlUi.allowedOrigins: ["*"]` سياسة صريحة تسمح بجميع browser-origin، وليست إعدادًا افتراضيًا محصنًا. تجنبها خارج الاختبارات المحلية المحكمة.
- لا تزال حالات فشل مصادقة browser-origin على loopback محددة المعدل حتى عند
  تفعيل الاستثناء العام لـ loopback، لكن مفتاح الحظر يكون محدد النطاق لكل
  قيمة `Origin` مطبّعة بدلًا من سلة localhost مشتركة واحدة.
- يفعّل `gateway.controlUi.dangerouslyAllowHostHeaderOriginFallback=true` وضع fallback الخاص بـ Host-header origin؛ تعامل معه باعتباره سياسة خطرة اختارها المشغّل.
- تعامل مع DNS rebinding وسلوك ترويسة مضيف proxy كمسائل تقوية للنشر؛ أبقِ `trustedProxies` مضيقًا وتجنب كشف gateway مباشرة إلى الإنترنت العام.

## سجلات الجلسات المحلية موجودة على القرص

يخزن OpenClaw السجلات النصية للجلسات على القرص تحت `~/.openclaw/agents/<agentId>/sessions/*.jsonl`.
وهذا مطلوب لاستمرارية الجلسة و(اختياريًا) لفهرسة ذاكرة الجلسة، لكنه يعني أيضًا أن
**أي عملية/مستخدم لديه وصول إلى نظام الملفات يمكنه قراءة تلك السجلات**. تعامل مع الوصول إلى القرص باعتباره
حد الثقة وأحكم أذونات `~/.openclaw` (راجع قسم التدقيق أدناه). وإذا كنت تحتاج إلى
عزل أقوى بين الوكلاء، فشغّلهم تحت مستخدمي نظام تشغيل منفصلين أو على مضيفين منفصلين.

## تنفيذ Node (`system.run`)

إذا كان macOS node مقترنًا، فيمكن لـ Gateway استدعاء `system.run` على ذلك node. وهذا يُعد **تنفيذًا برمجيًا عن بُعد** على جهاز Mac:

- يتطلب اقتران node (موافقة + token).
- اقتران Gateway مع node ليس سطح موافقة لكل أمر. بل يثبت هوية node/الثقة ويصدر token.
- يطبق Gateway سياسة عامة خشنة لأوامر node عبر `gateway.nodes.allowCommands` / `denyCommands`.
- يتم التحكم فيه على جهاز Mac عبر **Settings → Exec approvals** (الأمان + السؤال + قائمة السماح).
- تكون سياسة `system.run` لكل node هي ملف موافقات التنفيذ الخاص بذلك node نفسه (`exec.approvals.node.*`)، وقد تكون أكثر صرامة أو أكثر تساهلًا من سياسة معرّف الأوامر العامة الخاصة بـ gateway.
- يتبع node الذي يعمل بالإعداد `security="full"` و`ask="off"` نموذج المشغّل الموثوق الافتراضي. تعامل مع ذلك كسلوك متوقع ما لم يكن النشر لديك يتطلب صراحةً وضعًا أشد يعتمد على الموافقة أو قائمة السماح.
- يربط وضع الموافقة سياق الطلب الدقيق، وعند الإمكان، معامل script/file محلي واحد محدد. وإذا لم يتمكن OpenClaw من تحديد ملف محلي مباشر واحد بالضبط لأمر مفسر/وقت تشغيل، فسيُرفض التنفيذ المدعوم بالموافقة بدلًا من الادعاء بتغطية دلالية كاملة.
- بالنسبة إلى `host=node`، تخزن أيضًا التشغيلات المدعومة بالموافقة
  `systemRunPlan` جاهزة وقياسية؛ وتعيد عمليات التمرير الموافق عليها لاحقًا استخدام تلك الخطة المخزنة، كما
  يرفض تحقق gateway تعديلات المتصل على command/cwd/session context بعد إنشاء
  طلب الموافقة.
- إذا كنت لا تريد تنفيذًا عن بُعد، فاضبط الأمان على **deny** وأزل اقتران node لذلك الجهاز Mac.

هذا التمييز مهم عند التقييم:

- لا يُعد node مقترن يعيد الاتصال ويعلن قائمة أوامر مختلفة ثغرة بحد ذاته، إذا كانت السياسة العامة لـ Gateway وموافقات التنفيذ المحلية الخاصة بـ node لا تزال تفرض حد التنفيذ الفعلي.
- التقارير التي تتعامل مع بيانات اقتران node الوصفية كطبقة موافقة ثانية مخفية لكل أمر تكون عادةً التباسًا في السياسة/تجربة الاستخدام، وليست تجاوزًا لحدود الأمان.

## Skills الديناميكية (المراقِب / العُقد البعيدة)

يمكن لـ OpenClaw تحديث قائمة Skills أثناء الجلسة:

- **مراقِب Skills**: يمكن أن تؤدي التغييرات على `SKILL.md` إلى تحديث لقطة Skills في دور الوكيل التالي.
- **العُقد البعيدة**: يمكن أن يؤدي اتصال macOS node إلى جعل Skills الخاصة بـ macOS فقط مؤهلة (استنادًا إلى فحص bin).

تعامل مع مجلدات Skill على أنها **شفرة موثوقة** وقيّد من يمكنه تعديلها.

## نموذج التهديد

يمكن لمساعدك المعتمد على الذكاء الاصطناعي أن:

- ينفذ أوامر shell اعتباطية
- يقرأ الملفات ويكتبها
- يصل إلى خدمات الشبكة
- يرسل رسائل إلى أي شخص (إذا منحته وصول WhatsApp)

يمكن للأشخاص الذين يراسلونك أن:

- يحاولوا خداع الذكاء الاصطناعي لديك ليفعل أشياء سيئة
- يمارسوا هندسة اجتماعية للوصول إلى بياناتك
- يستكشفوا تفاصيل البنية التحتية

## المفهوم الأساسي: التحكم في الوصول قبل الذكاء

معظم حالات الفشل هنا ليست استغلالات معقدة — بل هي "أحدهم راسل الروبوت والروبوت فعل ما طلبه".

موقف OpenClaw:

- **الهوية أولًا:** قرر من يمكنه التحدث إلى الروبوت (اقتران الرسائل المباشرة / قوائم السماح / وضع “open” الصريح).
- **ثم النطاق:** قرر أين يُسمح للروبوت بالتصرف (قوائم سماح المجموعات + بوابة الإشارة، والأدوات، وSandbox، وأذونات الجهاز).
- **وأخيرًا النموذج:** افترض أن النموذج يمكن التلاعب به؛ وصمّم بحيث يكون نطاق الضرر محدودًا.

## نموذج تخويل الأوامر

لا يتم احترام أوامر slash والتوجيهات إلا من **المرسلين المخولين**. ويُشتق التخويل من
قوائم سماح القنوات/الاقتران بالإضافة إلى `commands.useAccessGroups` (راجع [التكوين](/ar/gateway/configuration)
و[أوامر slash](/ar/tools/slash-commands)). وإذا كانت قائمة سماح القناة فارغة أو تتضمن `"*"`,
فإن الأوامر تكون فعليًا مفتوحة لتلك القناة.

يُعد `/exec` وسيلة مريحة خاصة بالجلسة فقط للمشغلين المخولين. وهو **لا** يكتب التكوين ولا
يغيّر الجلسات الأخرى.

## مخاطر أدوات مستوى التحكم

يمكن لأداتين مدمجتين إجراء تغييرات دائمة على مستوى التحكم:

- يستطيع `gateway` فحص التكوين باستخدام `config.schema.lookup` / `config.get`، ويمكنه إجراء تغييرات دائمة باستخدام `config.apply` و`config.patch` و`update.run`.
- يمكن لـ `cron` إنشاء وظائف مجدولة تستمر في العمل بعد انتهاء الدردشة/المهمة الأصلية.

لا تزال أداة وقت التشغيل `gateway` الخاصة بالمالك فقط ترفض إعادة كتابة
`tools.exec.ask` أو `tools.exec.security`؛ كما تُطبّع الأسماء المستعارة القديمة `tools.bash.*`
إلى مسارات التنفيذ المحمية نفسها قبل الكتابة.

بالنسبة إلى أي وكيل/سطح يتعامل مع محتوى غير موثوق، ارفض هذه الأدوات افتراضيًا:

```json5
{
  tools: {
    deny: ["gateway", "cron", "sessions_spawn", "sessions_send"],
  },
}
```

لا يؤدي `commands.restart=false` إلا إلى حظر إجراءات إعادة التشغيل. وهو لا يعطّل إجراءات `gateway` الخاصة بالتكوين/التحديث.

## Plugins

تعمل Plugins **داخل العملية نفسها** مع Gateway. تعامل معها على أنها شفرة موثوقة:

- ثبّت Plugins فقط من مصادر تثق بها.
- فضّل قوائم السماح الصريحة `plugins.allow`.
- راجع تكوين Plugin قبل التمكين.
- أعد تشغيل Gateway بعد تغييرات Plugin.
- إذا قمت بتثبيت Plugins أو تحديثها (`openclaw plugins install <package>`، و`openclaw plugins update <id>`)، فتعامل مع ذلك كما لو كنت تشغّل شفرة غير موثوقة:
  - مسار التثبيت هو دليل Plugin الفردي تحت جذر تثبيت Plugin النشط.
  - يشغّل OpenClaw فحصًا مدمجًا للشفرة الخطرة قبل التثبيت/التحديث. وتؤدي النتائج `critical` إلى الحظر افتراضيًا.
  - يستخدم OpenClaw `npm pack` ثم يشغّل `npm install --omit=dev` في ذلك الدليل (يمكن لبرامج npm lifecycle النصية تنفيذ شفرة أثناء التثبيت).
  - فضّل الإصدارات المثبتة والدقيقة (`@scope/pkg@1.2.3`) وافحص الشفرة المفكوكة على القرص قبل التمكين.
  - يُستخدم `--dangerously-force-unsafe-install` لحالات الطوارئ فقط عندما تكون هناك نتائج إيجابية كاذبة من الفحص المدمج في تدفقات تثبيت/تحديث Plugin. وهو لا يتجاوز حظر سياسة hook `before_install` الخاصة بـ Plugin ولا يتجاوز حالات فشل الفحص.
  - تتبع عمليات تثبيت تبعيات Skill المدعومة من Gateway الانقسام نفسه بين الخطر/الاشتباه: تؤدي النتائج `critical` المدمجة إلى الحظر ما لم يضبط المتصل صراحةً `dangerouslyForceUnsafeInstall`، بينما تبقى النتائج المشتبه بها تحذيرات فقط. ويظل `openclaw skills install` هو تدفق تنزيل/تثبيت Skill المنفصل الخاص بـ ClawHub.

التفاصيل: [Plugins](/ar/tools/plugin)

## نموذج الوصول إلى الرسائل المباشرة (pairing / allowlist / open / disabled)

تدعم جميع القنوات الحالية القادرة على الرسائل المباشرة سياسة رسائل مباشرة (`dmPolicy` أو `*.dm.policy`) تتحكم في الرسائل المباشرة الواردة **قبل** معالجة الرسالة:

- `pairing` (الافتراضي): يتلقى المرسلون غير المعروفين رمز اقتران قصيرًا ويتجاهل الروبوت رسالتهم حتى تتم الموافقة عليهم. تنتهي صلاحية الرموز بعد ساعة واحدة؛ ولن تؤدي الرسائل المباشرة المتكررة إلى إعادة إرسال الرمز حتى يُنشأ طلب جديد. وتُحدد الطلبات المعلقة افتراضيًا عند **3 لكل قناة**.
- `allowlist`: يُحظر المرسلون غير المعروفين (من دون مصافحة اقتران).
- `open`: السماح لأي شخص بإرسال رسالة مباشرة (عام). **يتطلب** أن تتضمن قائمة سماح القناة `"*"` (اشتراك صريح).
- `disabled`: تجاهل الرسائل المباشرة الواردة بالكامل.

الموافقة عبر CLI:

```bash
openclaw pairing list <channel>
openclaw pairing approve <channel> <code>
```

التفاصيل + الملفات على القرص: [الاقتران](/ar/channels/pairing)

## عزل جلسات الرسائل المباشرة (وضع تعدد المستخدمين)

بشكل افتراضي، يوجّه OpenClaw **جميع الرسائل المباشرة إلى الجلسة الرئيسية** حتى يحافظ مساعدك على الاستمرارية عبر الأجهزة والقنوات. وإذا كان **عدة أشخاص** يستطيعون إرسال رسائل مباشرة إلى الروبوت (رسائل مباشرة مفتوحة أو قائمة سماح متعددة الأشخاص)، ففكر في عزل جلسات الرسائل المباشرة:

```json5
{
  session: { dmScope: "per-channel-peer" },
}
```

يمنع هذا تسرب السياق بين المستخدمين مع الإبقاء على عزل الدردشات الجماعية.

هذا حد لسياق المراسلة، وليس حد إدارة للمضيف. وإذا كان المستخدمون عدائيين بشكل متبادل ويشتركون في مضيف/تكوين Gateway نفسه، فشغّل Gateways منفصلة لكل حد ثقة.

### وضع الرسائل المباشرة الآمن (موصى به)

تعامل مع المقتطف أعلاه على أنه **وضع الرسائل المباشرة الآمن**:

- الافتراضي: `session.dmScope: "main"` (تشترك جميع الرسائل المباشرة في جلسة واحدة للاستمرارية).
- افتراضي الإعداد الأولي المحلي عبر CLI: يكتب `session.dmScope: "per-channel-peer"` عندما لا يكون مضبوطًا (ويُبقي القيم الصريحة الموجودة).
- وضع الرسائل المباشرة الآمن: `session.dmScope: "per-channel-peer"` (يحصل كل زوج قناة+مرسل على سياق رسائل مباشرة معزول).
- العزل بين القنوات لنفس النظير: `session.dmScope: "per-peer"` (يحصل كل مرسل على جلسة واحدة عبر جميع القنوات من النوع نفسه).

إذا كنت تشغّل عدة حسابات على القناة نفسها، فاستخدم `per-account-channel-peer` بدلًا من ذلك. وإذا كان الشخص نفسه يتواصل معك عبر عدة قنوات، فاستخدم `session.identityLinks` لدمج جلسات الرسائل المباشرة تلك في هوية قانونية واحدة. راجع [إدارة الجلسات](/ar/concepts/session) و[التكوين](/ar/gateway/configuration).

## قوائم السماح (الرسائل المباشرة + المجموعات) - المصطلحات

يحتوي OpenClaw على طبقتين منفصلتين لسؤال "من يمكنه تحفيزي؟":

- **قائمة سماح الرسائل المباشرة** (`allowFrom` / `channels.discord.allowFrom` / `channels.slack.allowFrom`؛ القديم: `channels.discord.dm.allowFrom`، و`channels.slack.dm.allowFrom`): من المسموح له التحدث إلى الروبوت في الرسائل المباشرة.
  - عندما يكون `dmPolicy="pairing"`، تُكتب الموافقات إلى مخزن قائمة سماح الاقتران المحدد النطاق بالحساب تحت `~/.openclaw/credentials/` (`<channel>-allowFrom.json` للحساب الافتراضي، و`<channel>-<accountId>-allowFrom.json` للحسابات غير الافتراضية)، وتُدمج مع قوائم سماح التكوين.
- **قائمة سماح المجموعات** (خاصة بالقناة): أي مجموعات/قنوات/خوادم guilds سيقبل الروبوت الرسائل منها أصلًا.
  - الأنماط الشائعة:
    - `channels.whatsapp.groups` و`channels.telegram.groups` و`channels.imessage.groups`: إعدادات افتراضية لكل مجموعة مثل `requireMention`؛ وعند ضبطها، تعمل أيضًا كقائمة سماح للمجموعات (ضمّن `"*"` للإبقاء على سلوك السماح للجميع).
    - `groupPolicy="allowlist"` + `groupAllowFrom`: تقييد من يمكنه تحفيز الروبوت _داخل_ جلسة المجموعة (WhatsApp/Telegram/Signal/iMessage/Microsoft Teams).
    - `channels.discord.guilds` / `channels.slack.channels`: قوائم سماح لكل سطح + إعدادات الإشارة الافتراضية.
  - تُنفذ فحوصات المجموعات بهذا الترتيب: `groupPolicy`/قوائم سماح المجموعات أولًا، ثم تفعيل الإشارة/الرد ثانيًا.
  - لا يؤدي الرد على رسالة الروبوت (إشارة ضمنية) إلى تجاوز قوائم سماح المرسل مثل `groupAllowFrom`.
  - **ملاحظة أمان:** تعامل مع `dmPolicy="open"` و`groupPolicy="open"` باعتبارهما إعدادين للملاذ الأخير. يجب أن يكون استخدامهما نادرًا جدًا؛ وفضّل الاقتران + قوائم السماح ما لم تكن تثق تمامًا بكل أعضاء الغرفة.

التفاصيل: [التكوين](/ar/gateway/configuration) و[المجموعات](/ar/channels/groups)

## حقن prompt (ما هو ولماذا يهم)

يحدث حقن prompt عندما يصوغ المهاجم رسالة تتلاعب بالنموذج ليفعل شيئًا غير آمنًا ("تجاهل تعليماتك"، "افرغ نظام ملفاتك"، "اتبع هذا الرابط وشغّل أوامر"، إلخ).

حتى مع prompts نظام قوية، **لم يُحلّ حقن prompt**. تمثل حواجز حماية prompt النظامية إرشادات مرنة فقط؛ أما الفرض الصارم فيأتي من سياسة الأدوات، وموافقات التنفيذ، وSandbox، وقوائم سماح القنوات (ويمكن للمشغلين تعطيل هذه الأشياء بحكم التصميم). ما يساعد عمليًا:

- أبقِ الرسائل المباشرة الواردة مغلقة بإحكام (الاقتران/قوائم السماح).
- فضّل بوابة الإشارة في المجموعات؛ وتجنب الروبوتات "الدائمة التشغيل" في الغرف العامة.
- تعامل مع الروابط والمرفقات والتعليمات الملصقة على أنها عدائية افتراضيًا.
- شغّل تنفيذ الأدوات الحساسة داخل Sandbox؛ وأبقِ الأسرار خارج نظام الملفات الذي يمكن للوكيل الوصول إليه.
- ملاحظة: Sandbox اختيارية. إذا كان وضع Sandbox متوقفًا، فإن `host=auto` الضمني يُحل إلى مضيف gateway. ولا يزال `host=sandbox` الصريح يفشل بشكل مغلق لأنه لا يتوفر وقت تشغيل Sandbox. اضبط `host=gateway` إذا كنت تريد أن يكون هذا السلوك صريحًا في التكوين.
- قلل الأدوات عالية الخطورة (`exec` و`browser` و`web_fetch` و`web_search`) إلى الوكلاء الموثوقين أو قوائم السماح الصريحة.
- إذا استخدمت قائمة سماح للمفسرات (`python` و`node` و`ruby` و`perl` و`php` و`lua` و`osascript`)، ففعّل `tools.exec.strictInlineEval` حتى تظل صيغ eval المضمنة تتطلب موافقة صريحة.
- يرفض تحليل موافقة shell أيضًا صيغ توسيع معلمات POSIX (`$VAR` و`$?` و`$$` و`$1` و`$@` و`${…}`) داخل **heredoc غير المقتبسة**، بحيث لا يمكن لجسم heredoc الموجود في قائمة السماح تمرير توسيع shell خلسةً متجاوزًا مراجعة قائمة السماح على أنه نص عادي. اقتبس محدد heredoc (مثل `<<'EOF'`) لاختيار دلالات الجسم الحرفية؛ وتُرفض heredoc غير المقتبسة التي كانت ستوسع المتغيرات.
- **اختيار النموذج مهم:** النماذج الأقدم/الأصغر/القديمة أقل صمودًا بشكل ملحوظ أمام حقن prompt وإساءة استخدام الأدوات. بالنسبة إلى الوكلاء المفعّلين بالأدوات، استخدم أقوى نموذج متاح من الجيل الأحدث والمقوّى ضد التعليمات.

علامات التحذير التي يجب اعتبارها غير موثوقة:

- "اقرأ هذا الملف/عنوان URL وافعل بالضبط ما يقوله."
- "تجاهل prompt النظام أو قواعد الأمان الخاصة بك."
- "اكشف تعليماتك المخفية أو مخرجات أدواتك."
- "الصق المحتويات الكاملة لـ ~/.openclaw أو سجلاتك."

## تنقية الرموز الخاصة للمحتوى الخارجي

يقوم OpenClaw بإزالة الرموز الحرفية الشائعة الخاصة بقوالب الدردشة في LLM المستضافة ذاتيًا من المحتوى الخارجي المغلف وبياناته الوصفية قبل أن تصل إلى النموذج. وتشمل عائلات العلامات المغطاة رموز الأدوار/الأدوار-الدورات الخاصة بـ Qwen/ChatML وLlama وGemma وMistral وPhi وGPT-OSS.

السبب:

- أحيانًا تحتفظ الواجهات الخلفية المتوافقة مع OpenAI والتي تعمل أمام نماذج مستضافة ذاتيًا بالرموز الخاصة التي تظهر في نص المستخدم بدلًا من إخفائها. ويمكن لمهاجم قادر على الكتابة في محتوى خارجي وارد (صفحة تم جلبها، أو نص رسالة بريد، أو مخرجات أداة محتويات ملف) أن يحقن حدًا اصطناعيًا لدور `assistant` أو `system` ويتجاوز حواجز حماية المحتوى المغلف.
- تتم التنقية في طبقة تغليف المحتوى الخارجي، لذلك تُطبق بشكل موحد عبر أدوات الجلب/القراءة ومحتوى القنوات الواردة بدلًا من أن تكون خاصة بكل موفّر.
- تحتوي استجابات النموذج الصادرة بالفعل على أداة تنقية منفصلة تزيل البنية المتسربة مثل `<tool_call>` و`<function_calls>` وما شابه من الردود المرئية للمستخدم. وتمثل أداة تنقية المحتوى الخارجي النظير الوارد لهذه الآلية.

لا يحل هذا محل أساليب التقوية الأخرى في هذه الصفحة — فما زالت `dmPolicy` وقوائم السماح وموافقات التنفيذ وSandbox و`contextVisibility` هي التي تقوم بالعمل الأساسي. ولكنه يغلق مسار تجاوز واحدًا محددًا على مستوى tokenizer ضد الحزم المستضافة ذاتيًا التي تمرر نص المستخدم مع بقاء الرموز الخاصة سليمة.

## علامات تجاوز المحتوى الخارجي غير الآمن

يتضمن OpenClaw علامات تجاوز صريحة تعطل تغليف أمان المحتوى الخارجي:

- `hooks.mappings[].allowUnsafeExternalContent`
- `hooks.gmail.allowUnsafeExternalContent`
- حقل الحمولة `allowUnsafeExternalContent` الخاص بـ Cron

الإرشادات:

- أبقِ هذه القيم غير مضبوطة/`false` في الإنتاج.
- فعّلها مؤقتًا فقط لتصحيح أخطاء محدد بإحكام.
- إذا تم تفعيلها، فاعزل ذلك الوكيل (Sandbox + أقل قدر من الأدوات + مساحة أسماء جلسة مخصصة).

ملاحظة بشأن مخاطر Hooks:

- حمولة Hook هي محتوى غير موثوق، حتى عندما يأتي التسليم من أنظمة تتحكم بها (يمكن للبريد/الوثائق/محتوى الويب أن يحمل حقن prompt).
- تزيد طبقات النماذج الضعيفة من هذا الخطر. بالنسبة إلى الأتمتة المدفوعة بـ hook، فضّل طبقات النماذج الحديثة والقوية وأبقِ سياسة الأدوات محكمة (`tools.profile: "messaging"` أو أشد)، بالإضافة إلى Sandbox حيثما أمكن.

### لا يتطلب حقن prompt رسائل مباشرة عامة

حتى إذا كان **أنت فقط** من يمكنه مراسلة الروبوت، فلا يزال من الممكن حدوث حقن prompt عبر
أي **محتوى غير موثوق** يقرأه الروبوت (نتائج البحث/الجلب على الويب، وصفحات المتصفح،
ورسائل البريد الإلكتروني، والوثائق، والمرفقات، والسجلات/الشفرة الملصقة). وبعبارة أخرى: ليس المرسل
هو سطح التهديد الوحيد؛ بل إن **المحتوى نفسه** يمكن أن يحمل تعليمات عدائية.

عند تفعيل الأدوات، يكون الخطر المعتاد هو تسريب السياق أو تحفيز
استدعاءات الأدوات. قلّل نطاق الضرر من خلال:

- استخدام **وكيل قارئ** للقراءة فقط أو بدون أدوات لتلخيص المحتوى غير الموثوق،
  ثم تمرير الملخص إلى وكيلك الرئيسي.
- إبقاء `web_search` / `web_fetch` / `browser` معطلة للوكلاء المفعّلين بالأدوات ما لم تكن هناك حاجة إليها.
- بالنسبة إلى مدخلات URL الخاصة بـ OpenResponses (`input_file` / `input_image`)، اضبط
  `gateway.http.endpoints.responses.files.urlAllowlist` و
  `gateway.http.endpoints.responses.images.urlAllowlist` بإحكام، وأبقِ `maxUrlParts` منخفضًا.
  تُعامل قوائم السماح الفارغة على أنها غير مضبوطة؛ استخدم `files.allowUrl: false` / `images.allowUrl: false`
  إذا كنت تريد تعطيل جلب URL بالكامل.
- بالنسبة إلى مدخلات الملفات في OpenResponses، لا يزال النص المفكوك من `input_file` يُحقن
  باعتباره **محتوى خارجيًا غير موثوق**. لا تعتمد على اعتبار نص الملف موثوقًا لمجرد
  أن Gateway فكّه محليًا. فما زالت الكتلة المحقونة تحمل علامات حدود
  `<<<EXTERNAL_UNTRUSTED_CONTENT ...>>>` صريحة بالإضافة إلى بيانات وصفية `Source: External`
  حتى وإن كان هذا المسار يحذف لافتة `SECURITY NOTICE:` الأطول.
- يُطبق التغليف نفسه المعتمد على العلامات عندما يستخرج فهم الوسائط نصًا
  من المستندات المرفقة قبل إلحاق ذلك النص بـ prompt الوسائط.
- تفعيل Sandbox وقوائم سماح صارمة للأدوات لأي وكيل يلامس مدخلات غير موثوقة.
- إبقاء الأسرار خارج prompts؛ ومرّرها عبر env/التكوين على مضيف gateway بدلًا من ذلك.

### الواجهات الخلفية لـ LLM المستضافة ذاتيًا

يمكن أن تختلف الواجهات الخلفية المتوافقة مع OpenAI والمستضافة ذاتيًا مثل vLLM وSGLang وTGI وLM Studio
أو حزم tokenizer المخصصة من Hugging Face عن الموفّرين المستضافين في كيفية
التعامل مع الرموز الخاصة بقوالب الدردشة. وإذا كان backend يجزئ سلاسل حرفية
مثل `<|im_start|>` أو `<|start_header_id|>` أو `<start_of_turn>` إلى
رموز هيكلية لقوالب الدردشة داخل محتوى المستخدم، فقد يحاول النص غير الموثوق
تزوير حدود الأدوار على مستوى tokenizer.

يقوم OpenClaw بإزالة الرموز الحرفية الشائعة الخاصة بعائلات النماذج من
المحتوى الخارجي المغلف قبل إرساله إلى النموذج. أبقِ تغليف المحتوى الخارجي
مفعّلًا، وفضّل إعدادات backend التي تقسّم الرموز الخاصة أو تهربها في
المحتوى المقدم من المستخدم عند توفر ذلك. أما الموفّرون المستضافون مثل OpenAI
وAnthropic فيطبقون بالفعل تنقية خاصة بهم على جانب الطلب.

### قوة النموذج (ملاحظة أمنية)

إن مقاومة حقن prompt **ليست موحدة** عبر طبقات النماذج. فالنماذج الأصغر/الأرخص تكون عمومًا أكثر عرضة لإساءة استخدام الأدوات واختطاف التعليمات، خصوصًا مع prompts عدائية.

<Warning>
بالنسبة إلى الوكلاء المفعّلين بالأدوات أو الوكلاء الذين يقرؤون محتوى غير موثوق، يكون خطر حقن prompt مع النماذج الأقدم/الأصغر مرتفعًا جدًا غالبًا. لا تشغّل هذه الأحمال على طبقات نماذج ضعيفة.
</Warning>

التوصيات:

- **استخدم أحدث جيل وأفضل طبقة نموذج** لأي روبوت يمكنه تشغيل الأدوات أو الوصول إلى الملفات/الشبكات.
- **لا تستخدم الطبقات الأقدم/الأضعف/الأصغر** للوكلاء المفعّلين بالأدوات أو صناديق الوارد غير الموثوقة؛ فخطر حقن prompt مرتفع جدًا.
- إذا اضطررت إلى استخدام نموذج أصغر، **فقلّل نطاق الضرر** (أدوات للقراءة فقط، وSandbox قوية، وأقل قدر من الوصول إلى نظام الملفات، وقوائم سماح صارمة).
- عند تشغيل نماذج صغيرة، **فعّل Sandbox لجميع الجلسات** و**عطّل `web_search`/`web_fetch`/`browser`** ما لم تكن المدخلات محكومة بإحكام.
- بالنسبة إلى المساعدين الشخصيين للدردشة فقط مع مدخلات موثوقة ومن دون أدوات، تكون النماذج الأصغر مناسبة عادةً.

## `reasoning` والمخرجات المفصلة في المجموعات

يمكن لـ `/reasoning` و`/verbose` و`/trace` كشف reasoning الداخلي أو مخرجات
الأدوات أو تشخيصات Plugin التي
لم يكن المقصود عرضها في قناة عامة. وفي إعدادات المجموعات، تعامل معها على أنها **للتصحيح فقط**
وأبقها معطلة ما لم تكن تحتاجها صراحةً.

الإرشادات:

- أبقِ `/reasoning` و`/verbose` و`/trace` معطلة في الغرف العامة.
- إذا فعّلتها، فافعل ذلك فقط في الرسائل المباشرة الموثوقة أو الغرف المحكمة بإحكام.
- تذكر: يمكن أن تتضمن المخرجات المفصلة ومخرجات التتبع معاملات الأدوات وعناوين URL وتشخيصات Plugin والبيانات التي رآها النموذج.

## تقوية التكوين (أمثلة)

### أذونات الملفات

أبقِ التكوين + الحالة خاصة على مضيف gateway:

- `~/.openclaw/openclaw.json`: `600` (قراءة/كتابة للمستخدم فقط)
- `~/.openclaw`: `700` (المستخدم فقط)

يمكن لـ `openclaw doctor` التحذير وعرض تشديد هذه الأذونات.

### كشف الشبكة (الربط، المنفذ، الجدار الناري)

يضاعف Gateway **WebSocket + HTTP** على منفذ واحد:

- الافتراضي: `18789`
- التكوين/الخيارات/env: `gateway.port`، و`--port`، و`OPENCLAW_GATEWAY_PORT`

يتضمن سطح HTTP هذا Control UI ومضيف canvas:

- Control UI (أصول SPA) (المسار الأساسي الافتراضي `/`)
- مضيف Canvas: `/__openclaw__/canvas/` و`/__openclaw__/a2ui/` (HTML/JS اعتباطي؛ تعامل معه على أنه محتوى غير موثوق)

إذا حمّلت محتوى canvas في متصفح عادي، فتعامل معه كما تتعامل مع أي صفحة ويب غير موثوقة:

- لا تكشف مضيف canvas لشبكات/مستخدمين غير موثوقين.
- لا تجعل محتوى canvas يشارك origin نفسه مع أسطح الويب المميزة ما لم تكن تفهم الآثار بالكامل.

يتحكم وضع الربط في المكان الذي يستمع فيه Gateway:

- `gateway.bind: "loopback"` (الافتراضي): يمكن للعملاء المحليين فقط الاتصال.
- توسّع عمليات الربط غير المعتمدة على loopback (`"lan"`، و`"tailnet"`، و`"custom"`) سطح الهجوم. لا تستخدمها إلا مع مصادقة gateway (token/password مشترك أو trusted proxy غير معتمد على loopback ومضبوط بشكل صحيح) وجدار ناري حقيقي.

قواعد عامة:

- فضّل Tailscale Serve على عمليات ربط LAN (يبقي Serve الـ Gateway على loopback، ويتولى Tailscale الوصول).
- إذا اضطررت إلى الربط على LAN، فاحصر المنفذ في الجدار الناري على قائمة سماح ضيقة من عناوين IP المصدر؛ ولا تقم بعمل port-forward له على نطاق واسع.
- لا تكشف Gateway مطلقًا من دون مصادقة على `0.0.0.0`.

### نشر منافذ Docker مع UFW

إذا كنت تشغّل OpenClaw باستخدام Docker على VPS، فتذكر أن منافذ الحاويات المنشورة
(`-p HOST:CONTAINER` أو Compose `ports:`) تُوجَّه عبر سلاسل إعادة التوجيه الخاصة بـ Docker،
وليس فقط قواعد `INPUT` الخاصة بالمضيف.

لإبقاء حركة Docker متوافقة مع سياسة الجدار الناري لديك، افرض القواعد في
`DOCKER-USER` (تُقيَّم هذه السلسلة قبل قواعد القبول الخاصة بـ Docker نفسها).
وفي كثير من التوزيعات الحديثة، تستخدم `iptables`/`ip6tables` الواجهة الأمامية `iptables-nft`
ولا تزال تطبق هذه القواعد على backend الخاصة بـ nftables.

مثال حد أدنى لقائمة سماح (IPv4):

```bash
# /etc/ufw/after.rules (ألحِقه كقسم *filter مستقل)
*filter
:DOCKER-USER - [0:0]
-A DOCKER-USER -m conntrack --ctstate ESTABLISHED,RELATED -j RETURN
-A DOCKER-USER -s 127.0.0.0/8 -j RETURN
-A DOCKER-USER -s 10.0.0.0/8 -j RETURN
-A DOCKER-USER -s 172.16.0.0/12 -j RETURN
-A DOCKER-USER -s 192.168.0.0/16 -j RETURN
-A DOCKER-USER -s 100.64.0.0/10 -j RETURN
-A DOCKER-USER -p tcp --dport 80 -j RETURN
-A DOCKER-USER -p tcp --dport 443 -j RETURN
-A DOCKER-USER -m conntrack --ctstate NEW -j DROP
-A DOCKER-USER -j RETURN
COMMIT
```

لـ IPv6 جداول منفصلة. أضف سياسة مطابقة في `/etc/ufw/after6.rules` إذا
كان Docker IPv6 مفعّلًا.

تجنب ترميز أسماء الواجهات مثل `eth0` مباشرةً في مقتطفات الوثائق. تختلف أسماء الواجهات
بين صور VPS (`ens3` و`enp*` وغيرها) ويمكن أن تؤدي حالات عدم التطابق إلى
تجاوز قاعدة الرفض لديك عن غير قصد.

تحقق سريع بعد إعادة التحميل:

```bash
ufw reload
iptables -S DOCKER-USER
ip6tables -S DOCKER-USER
nmap -sT -p 1-65535 <public-ip> --open
```

يجب أن تكون المنافذ الخارجية المتوقعة فقط هي ما تكشفه عمدًا (في معظم
الإعدادات: SSH + منافذ reverse proxy الخاصة بك).

### اكتشاف mDNS/Bonjour

يبث Gateway وجوده عبر mDNS (`_openclaw-gw._tcp` على المنفذ 5353) لاكتشاف الأجهزة المحلية. وفي الوضع الكامل، يتضمن هذا سجلات TXT قد تكشف تفاصيل تشغيلية:

- `cliPath`: المسار الكامل في نظام الملفات إلى ثنائي CLI (يكشف اسم المستخدم وموقع التثبيت)
- `sshPort`: يعلن عن توفر SSH على المضيف
- `displayName` و`lanHost`: معلومات اسم المضيف

**اعتبار أمان تشغيلي:** يجعل بث تفاصيل البنية التحتية الاستطلاع أسهل لأي شخص على الشبكة المحلية. وحتى المعلومات "غير الضارة" مثل مسارات نظام الملفات وتوفر SSH تساعد المهاجمين على رسم خريطة لبيئتك.

**التوصيات:**

1. **الوضع الأدنى** (الافتراضي، وموصى به للـ Gateways المكشوفة): يحذف الحقول الحساسة من بث mDNS:

   ```json5
   {
     discovery: {
       mdns: { mode: "minimal" },
     },
   }
   ```

2. **عطّله بالكامل** إذا لم تكن بحاجة إلى اكتشاف الأجهزة المحلية:

   ```json5
   {
     discovery: {
       mdns: { mode: "off" },
     },
   }
   ```

3. **الوضع الكامل** (اشتراك صريح): يتضمن `cliPath` + `sshPort` في سجلات TXT:

   ```json5
   {
     discovery: {
       mdns: { mode: "full" },
     },
   }
   ```

4. **متغير البيئة** (بديل): اضبط `OPENCLAW_DISABLE_BONJOUR=1` لتعطيل mDNS من دون تغييرات في التكوين.

في الوضع الأدنى، لا يزال Gateway يبث ما يكفي لاكتشاف الأجهزة (`role` و`gatewayPort` و`transport`) لكنه يحذف `cliPath` و`sshPort`. ويمكن للتطبيقات التي تحتاج إلى معلومات مسار CLI جلبها عبر اتصال WebSocket الموثق بدلًا من ذلك.

### أحكم WebSocket الخاص بـ Gateway (المصادقة المحلية)

تكون مصادقة Gateway **مطلوبة افتراضيًا**. وإذا لم يتم تكوين
مسار مصادقة صالح لـ gateway، فسيرفض Gateway اتصالات WebSocket (فشل مغلق).

يولّد الإعداد الأولي token افتراضيًا (حتى بالنسبة إلى loopback) بحيث
يجب على العملاء المحليين المصادقة.

اضبط token حتى **يجب على جميع** عملاء WS المصادقة:

```json5
{
  gateway: {
    auth: { mode: "token", token: "your-token" },
  },
}
```

يمكن لـ Doctor توليد واحد لك: `openclaw doctor --generate-gateway-token`.

ملاحظة: يُعد `gateway.remote.token` / `.password` مصدرَي بيانات اعتماد للعميل. وهما
لا يحميان وصول WS المحلي بمفردهما.
يمكن لمسارات الاستدعاء المحلية استخدام `gateway.remote.*` كخيار احتياطي فقط عندما لا يكون `gateway.auth.*`
مضبوطًا.
إذا تم تكوين `gateway.auth.token` / `gateway.auth.password` صراحةً عبر
SecretRef وكان غير قابل للحل، فإن الحل يفشل بشكل مغلق (من دون إخفاء عبر fallback بعيد).
اختياري: ثبّت TLS البعيد باستخدام `gateway.remote.tlsFingerprint` عند استخدام `wss://`.
يكون `ws://` النصي الصريح مخصصًا لـ loopback فقط افتراضيًا. أما لمسارات
الشبكات الخاصة الموثوقة، فاضبط `OPENCLAW_ALLOW_INSECURE_PRIVATE_WS=1` على عملية العميل كخيار طوارئ.

اقتران الجهاز المحلي:

- تتم الموافقة تلقائيًا على اقتران الجهاز لاتصالات loopback المحلية المباشرة للحفاظ
  على سلاسة عمل العملاء على المضيف نفسه.
- يمتلك OpenClaw أيضًا مسار self-connect ضيقًا محليًا للـ backend/الحاوية من أجل
  تدفقات المساعدات الموثوقة ذات السر المشترك.
- تُعامل اتصالات tailnet وLAN، بما في ذلك عمليات ربط tailnet على المضيف نفسه، على أنها
  بعيدة لأغراض الاقتران وما زالت تحتاج إلى موافقة.
- يؤدي وجود دليل من الترويسات المُمرَّرة على طلب loopback إلى إلغاء صفة المحلية الخاصة بـ loopback.
  ويكون نطاق الموافقة التلقائية لترقية البيانات الوصفية محدودًا بشكل ضيق. راجع
  [اقتران Gateway](/ar/gateway/pairing) لكلا القاعدتين.

أوضاع المصادقة:

- `gateway.auth.mode: "token"`: token حامل مشترك (موصى به لمعظم الإعدادات).
- `gateway.auth.mode: "password"`: مصادقة بكلمة مرور (يُفضل ضبطها عبر env: `OPENCLAW_GATEWAY_PASSWORD`).
- `gateway.auth.mode: "trusted-proxy"`: الثقة في reverse proxy واعٍ بالهوية لمصادقة المستخدمين وتمرير الهوية عبر الترويسات (راجع [مصادقة Trusted Proxy](/ar/gateway/trusted-proxy-auth)).

قائمة مراجعة التدوير (token/password):

1. ولّد/اضبط سرًا جديدًا (`gateway.auth.token` أو `OPENCLAW_GATEWAY_PASSWORD`).
2. أعد تشغيل Gateway (أو أعد تشغيل تطبيق macOS إذا كان هو من يشرف على Gateway).
3. حدّث أي عملاء بعيدين (`gateway.remote.token` / `.password` على الأجهزة التي تستدعي Gateway).
4. تحقق من أنك لم تعد قادرًا على الاتصال باستخدام بيانات الاعتماد القديمة.

### ترويسات هوية Tailscale Serve

عندما تكون `gateway.auth.allowTailscale` بقيمة `true` (الافتراضي لـ Serve)، فإن OpenClaw
يقبل ترويسات هوية Tailscale Serve (`tailscale-user-login`) لمصادقة Control
UI/WebSocket. ويتحقق OpenClaw من الهوية عن طريق حل عنوان
`x-forwarded-for` عبر daemon المحلي لـ Tailscale (`tailscale whois`)
ومطابقته مع الترويسة. ولا يُفعَّل هذا إلا للطلبات التي تصل إلى loopback
وتتضمن `x-forwarded-for` و`x-forwarded-proto` و`x-forwarded-host` كما
يحقنها Tailscale.
بالنسبة إلى مسار فحص الهوية غير المتزامن هذا، تتم تسلسل المحاولات الفاشلة
للعنوان نفسه `{scope, ip}` قبل أن يسجل limiter الفشل. لذلك يمكن أن تؤدي
إعادات المحاولة السيئة المتزامنة من عميل Serve واحد إلى حظر المحاولة الثانية
فورًا بدلًا من مرور المحاولتين على شكل حالتي عدم تطابق عاديتين.
أما نقاط نهاية HTTP API (مثل `/v1/*` و`/tools/invoke` و`/api/channels/*`)
فلا تستخدم مصادقة ترويسة هوية Tailscale. بل ما زالت تتبع
وضع مصادقة HTTP المضبوط في gateway.

ملاحظة مهمة حول حدود الثقة:

- تكون مصادقة حامل HTTP الخاصة بـ Gateway فعليًا وصول مشغّل شاملًا أو لا شيء.
- تعامل مع بيانات الاعتماد القادرة على استدعاء `/v1/chat/completions` أو `/v1/responses` أو `/api/channels/*` على أنها أسرار مشغّل كاملة الوصول لذلك gateway.
- على سطح HTTP المتوافق مع OpenAI، تعيد مصادقة الحامل ذات السر المشترك جميع نطاقات المشغّل الافتراضية الكاملة (`operator.admin` و`operator.approvals` و`operator.pairing` و`operator.read` و`operator.talk.secrets` و`operator.write`) ودلالات المالك لأدوار الوكيل؛ ولا تقلل القيم الأضيق لـ `x-openclaw-scopes` من هذا المسار المعتمد على السر المشترك.
- لا تنطبق دلالات النطاق لكل طلب على HTTP إلا عندما يأتي الطلب من وضع يحمل هوية مثل مصادقة trusted proxy أو `gateway.auth.mode="none"` على ingress خاص.
- في تلك الأوضاع الحاملة للهوية، يؤدي حذف `x-openclaw-scopes` إلى الرجوع إلى مجموعة نطاقات المشغّل الافتراضية المعتادة؛ أرسل الترويسة صراحةً عندما تريد مجموعة نطاقات أضيق.
- يتبع `/tools/invoke` القاعدة نفسها الخاصة بالسر المشترك: تُعامل مصادقة token/password bearer هناك أيضًا على أنها وصول مشغّل كامل، بينما تظل الأوضاع الحاملة للهوية تحترم النطاقات المعلنة.
- لا تشارك بيانات الاعتماد هذه مع متصلين غير موثوقين؛ وفضّل Gateways منفصلة لكل حد ثقة.

**افتراض الثقة:** تفترض مصادقة Serve من دون token أن مضيف gateway موثوق.
لا تتعامل مع هذا كحماية من العمليات العدائية على المضيف نفسه. وإذا كان من الممكن
تشغيل شفرة محلية غير موثوقة على مضيف gateway، فعطّل `gateway.auth.allowTailscale`
واشترط مصادقة صريحة بسر مشترك باستخدام `gateway.auth.mode: "token"` أو
`"password"`.

**قاعدة أمان:** لا تمرر هذه الترويسات من reverse proxy الخاص بك. وإذا
أنهيت TLS أو استخدمت proxy أمام gateway، فعطّل
`gateway.auth.allowTailscale` واستخدم مصادقة السر المشترك (`gateway.auth.mode:
"token"` أو `"password"`) أو [مصادقة Trusted Proxy](/ar/gateway/trusted-proxy-auth)
بدلًا من ذلك.

الـ trusted proxies:

- إذا أنهيت TLS أمام Gateway، فاضبط `gateway.trustedProxies` على عناوين IP الخاصة بـ proxy لديك.
- سيثق OpenClaw في `x-forwarded-for` (أو `x-real-ip`) القادم من تلك العناوين لتحديد عنوان IP الخاص بالعميل من أجل فحوصات الاقتران المحلي ومصادقة HTTP/الفحوصات المحلية.
- تأكد من أن proxy لديك **يكتب فوق** `x-forwarded-for` ويمنع الوصول المباشر إلى منفذ Gateway.

راجع [Tailscale](/ar/gateway/tailscale) و[نظرة عامة على الويب](/ar/web).

### التحكم في المتصفح عبر مضيف node (موصى به)

إذا كان Gateway لديك بعيدًا لكن المتصفح يعمل على جهاز آخر، فشغّل **مضيف node**
على جهاز المتصفح ودع Gateway يمرر إجراءات المتصفح (راجع [أداة Browser](/ar/tools/browser)).
تعامل مع اقتران node على أنه وصول إداري.

النمط الموصى به:

- أبقِ Gateway ومضيف node على tailnet نفسها (Tailscale).
- أقرِن node عمدًا؛ وعطّل توجيه proxy الخاص بالمتصفح إذا لم تكن بحاجة إليه.

تجنب:

- كشف منافذ relay/control عبر LAN أو الإنترنت العام.
- استخدام Tailscale Funnel لنقاط نهاية التحكم في المتصفح (كشف عام).

### الأسرار على القرص

افترض أن أي شيء تحت `~/.openclaw/` (أو `$OPENCLAW_STATE_DIR/`) قد يحتوي على أسرار أو بيانات خاصة:

- `openclaw.json`: قد يتضمن التكوين tokens (gateway، وgateway البعيدة)، وإعدادات الموفّرين، وقوائم السماح.
- `credentials/**`: بيانات اعتماد القنوات (مثال: بيانات اعتماد WhatsApp)، وقوائم سماح الاقتران، واستيرادات OAuth القديمة.
- `agents/<agentId>/agent/auth-profiles.json`: مفاتيح API، وملفات تعريف token، وtokens OAuth، و`keyRef`/`tokenRef` الاختياريان.
- `secrets.json` (اختياري): حمولة أسرار معتمدة على الملفات يستخدمها موفرو SecretRef من نوع `file` (`secrets.providers`).
- `agents/<agentId>/agent/auth.json`: ملف توافق قديم. وتُنظف إدخالات `api_key` الثابتة عند اكتشافها.
- `agents/<agentId>/sessions/**`: السجلات النصية للجلسات (`*.jsonl`) + بيانات التوجيه الوصفية (`sessions.json`) التي قد تحتوي على رسائل خاصة ومخرجات الأدوات.
- حزم Plugins المضمّنة: Plugins المثبتة (بالإضافة إلى `node_modules/` الخاصة بها).
- `sandboxes/**`: مساحات عمل Sandbox للأدوات؛ ويمكن أن تتراكم فيها نسخ من الملفات التي تقرؤها/تكتبها داخل Sandbox.

نصائح للتقوية:

- أبقِ الأذونات محكمة (`700` على الأدلة، و`600` على الملفات).
- استخدم تشفير القرص الكامل على مضيف gateway.
- فضّل حساب مستخدم نظام تشغيل مخصصًا لـ Gateway إذا كان المضيف مشتركًا.

### ملفات `.env` الخاصة بمساحة العمل

يحمّل OpenClaw ملفات `.env` المحلية الخاصة بمساحة العمل للوكلاء والأدوات، لكنه لا يسمح مطلقًا لتلك الملفات بتجاوز ضوابط وقت تشغيل gateway بصمت.

- يُحظر أي مفتاح يبدأ بـ `OPENCLAW_*` من ملفات `.env` غير الموثوقة الخاصة بمساحة العمل.
- تُحظر أيضًا إعدادات نقطة نهاية القناة الخاصة بـ Matrix وMattermost وIRC وSynology Chat من تجاوزات `.env` الخاصة بمساحة العمل، بحيث لا تستطيع مساحات العمل المستنسخة إعادة توجيه حركة موصلات الحزم المضمّنة عبر تكوين نقطة نهاية محلي. يجب أن تأتي مفاتيح env الخاصة بنقطة النهاية (مثل `MATRIX_HOMESERVER` و`MATTERMOST_URL` و`IRC_HOST` و`SYNOLOGY_CHAT_INCOMING_URL`) من بيئة عملية gateway أو من `env.shellEnv`، وليس من `.env` المحمّلة من مساحة العمل.
- يكون الحظر بفشل مغلق: فلا يمكن لمتغير جديد للتحكم في وقت التشغيل يُضاف في إصدار مستقبلي أن يُورث من `.env` مُضمّنة في المستودع أو مقدمة من مهاجم؛ إذ يُتجاهل المفتاح ويحتفظ gateway بقيمته الخاصة.
- لا تزال متغيرات بيئة العملية/نظام التشغيل الموثوقة (shell الخاصة بـ gateway، أو وحدة launchd/systemd، أو حزمة التطبيق) تُطبق — فهذا يقيد فقط تحميل ملفات `.env`.

السبب: غالبًا ما تعيش ملفات `.env` الخاصة بمساحة العمل بجانب شفرة الوكيل، أو تُرتكب في المستودع بالخطأ، أو تكتبها الأدوات. ويعني حظر البادئة الكاملة `OPENCLAW_*` أن إضافة علامة `OPENCLAW_*` جديدة لاحقًا لن ترتد أبدًا إلى وراثة صامتة من حالة مساحة العمل.

### السجلات والسجلات النصية (الإخفاء والاحتفاظ)

يمكن أن تسرّب السجلات والسجلات النصية معلومات حساسة حتى عندما تكون ضوابط الوصول صحيحة:

- قد تتضمن سجلات Gateway ملخصات الأدوات، والأخطاء، وعناوين URL.
- قد تتضمن السجلات النصية للجلسات أسرارًا ملصقة، ومحتويات ملفات، ومخرجات أوامر، وروابط.

التوصيات:

- أبقِ إخفاء ملخصات الأدوات مفعّلًا (`logging.redactSensitive: "tools"`؛ وهو الافتراضي).
- أضف أنماطًا مخصصة لبيئتك عبر `logging.redactPatterns` (tokens، وأسماء المضيفين، وعناوين URL الداخلية).
- عند مشاركة التشخيصات، فضّل `openclaw status --all` (قابل للصق، مع إخفاء الأسرار) على السجلات الخام.
- احذف السجلات النصية القديمة للجلسات وملفات السجل إذا لم تكن بحاجة إلى احتفاظ طويل.

التفاصيل: [التسجيل](/ar/gateway/logging)

### أرقام منفصلة (WhatsApp وSignal وTelegram)

بالنسبة إلى القنوات المعتمدة على أرقام الهواتف، فكّر في تشغيل الذكاء الاصطناعي لديك على رقم هاتف منفصل عن رقمك الشخصي:

- الرقم الشخصي: تبقى محادثاتك خاصة
- رقم الروبوت: يتعامل الذكاء الاصطناعي مع هذه المحادثات، ضمن حدود مناسبة

### وضع القراءة فقط (عبر Sandbox والأدوات)

يمكنك إنشاء ملف تعريف للقراءة فقط بدمج ما يلي:

- `agents.defaults.sandbox.workspaceAccess: "ro"` (أو `"none"` لعدم الوصول إلى مساحة العمل)
- قوائم سماح/رفض الأدوات التي تحظر `write` و`edit` و`apply_patch` و`exec` و`process` وغير ذلك.

خيارات تقوية إضافية:

- `tools.exec.applyPatch.workspaceOnly: true` (الافتراضي): يضمن أن `apply_patch` لا يمكنه الكتابة/الحذف خارج دليل مساحة العمل حتى عند تعطيل Sandbox. اضبطه على `false` فقط إذا كنت تريد عمدًا أن يلمس `apply_patch` ملفات خارج مساحة العمل.
- `tools.fs.workspaceOnly: true` (اختياري): يقيّد مسارات `read`/`write`/`edit`/`apply_patch` ومسارات التحميل التلقائي الأصلية لصور prompt إلى دليل مساحة العمل (وهو مفيد إذا كنت تسمح اليوم بالمسارات المطلقة وتريد حاجز حماية واحدًا).
- أبقِ جذور نظام الملفات ضيقة: تجنب الجذور الواسعة مثل دليلك المنزلي لمساحات عمل الوكيل/مساحات عمل Sandbox. فقد تكشف الجذور الواسعة ملفات محلية حساسة (مثل الحالة/التكوين تحت `~/.openclaw`) لأدوات نظام الملفات.

### خط أساس آمن (نسخ/لصق)

تكوين "افتراضي آمن" يبقي Gateway خاصًا، ويشترط اقتران الرسائل المباشرة، ويتجنب روبوتات المجموعات الدائمة التشغيل:

```json5
{
  gateway: {
    mode: "local",
    bind: "loopback",
    port: 18789,
    auth: { mode: "token", token: "your-long-random-token" },
  },
  channels: {
    whatsapp: {
      dmPolicy: "pairing",
      groups: { "*": { requireMention: true } },
    },
  },
}
```

إذا كنت تريد أيضًا تنفيذ أدوات "أكثر أمانًا افتراضيًا"، فأضف Sandbox + ارفض الأدوات الخطرة لأي وكيل غير مالك (المثال أدناه تحت "ملفات تعريف الوصول لكل وكيل").

خط الأساس المدمج لأدوار الوكيل المدفوعة بالدردشة: لا يمكن للمرسلين غير المالكين استخدام أداتي `cron` أو `gateway`.

## Sandbox (موصى به)

وثيقة مخصصة: [Sandboxing](/ar/gateway/sandboxing)

نهجان متكاملان:

- **شغّل Gateway الكامل داخل Docker** (حد الحاوية): [Docker](/ar/install/docker)
- **Sandbox الأدوات** (`agents.defaults.sandbox`، مضيف gateway + أدوات معزولة داخل Sandbox؛ وDocker هو backend الافتراضي): [Sandboxing](/ar/gateway/sandboxing)

ملاحظة: لمنع الوصول المتبادل بين الوكلاء، أبقِ `agents.defaults.sandbox.scope` على `"agent"` (الافتراضي)
أو `"session"` لعزل أشد لكل جلسة. أما `scope: "shared"` فيستخدم
حاوية/مساحة عمل واحدة.

وفكّر أيضًا في وصول مساحة عمل الوكيل داخل Sandbox:

- `agents.defaults.sandbox.workspaceAccess: "none"` (الافتراضي) يُبقي مساحة عمل الوكيل خارج الحدود؛ وتعمل الأدوات على مساحة عمل Sandbox تحت `~/.openclaw/sandboxes`
- `agents.defaults.sandbox.workspaceAccess: "ro"` يركب مساحة عمل الوكيل للقراءة فقط عند `/agent` (ويعطل `write`/`edit`/`apply_patch`)
- `agents.defaults.sandbox.workspaceAccess: "rw"` يركب مساحة عمل الوكيل للقراءة والكتابة عند `/workspace`
- تُتحقق عمليات الربط الإضافية `sandbox.docker.binds` في مقابل مسارات المصدر المطبّعة والمقنّنة. ولا تنجح حيل الروابط الرمزية على الآباء ولا الأسماء المستعارة المنزلية المقنّنة إذا حُلّت إلى جذور محظورة مثل `/etc` أو `/var/run` أو أدلة بيانات الاعتماد تحت المنزل الخاص بنظام التشغيل، بل تفشل بشكل مغلق.

مهم: يُعد `tools.elevated` فتحة الهروب الأساسية العامة التي تشغّل التنفيذ خارج Sandbox. ويكون المضيف الفعال هو `gateway` افتراضيًا، أو `node` عندما يكون هدف التنفيذ مضبوطًا على `node`. أبقِ `tools.elevated.allowFrom` ضيقًا ولا تفعّله للغرباء. ويمكنك تقييد elevated أكثر لكل وكيل عبر `agents.list[].tools.elevated`. راجع [الوضع المرتفع](/ar/tools/elevated).

### حاجز حماية تفويض الوكيل الفرعي

إذا سمحت بأدوات الجلسة، فتعامل مع تشغيلات الوكيل الفرعي المفوّضة باعتبارها قرارًا آخر متعلقًا بالحدود:

- ارفض `sessions_spawn` ما لم يكن الوكيل يحتاج التفويض فعلًا.
- أبقِ `agents.defaults.subagents.allowAgents` وأي تجاوزات لكل وكيل في `agents.list[].subagents.allowAgents` مقيدة بوكلاء مستهدفين معروفين وآمنين.
- لأي تدفق عمل يجب أن يبقى داخل Sandbox، استدعِ `sessions_spawn` باستخدام `sandbox: "require"` (الافتراضي هو `inherit`).
- يفشل `sandbox: "require"` سريعًا عندما لا يكون وقت تشغيل الابن الهدف داخل Sandbox.

## مخاطر التحكم في المتصفح

يمنح تفعيل التحكم في المتصفح النموذج القدرة على قيادة متصفح حقيقي.
وإذا كان ملف تعريف ذلك المتصفح يحتوي بالفعل على جلسات مسجّل الدخول،
فيمكن للنموذج الوصول إلى تلك الحسابات والبيانات. تعامل مع ملفات تعريف المتصفح باعتبارها **حالة حساسة**:

- فضّل ملف تعريف مخصصًا للوكيل (الملف الافتراضي `openclaw`).
- تجنب توجيه الوكيل إلى ملفك الشخصي اليومي.
- أبقِ التحكم في متصفح المضيف معطلًا للوكلاء الموجودين داخل Sandbox ما لم تكن تثق بهم.
- لا تحترم واجهة API المستقلة للتحكم في المتصفح على loopback إلا المصادقة
  ذات السر المشترك (مصادقة bearer الخاصة بـ gateway token أو كلمة مرور gateway). وهي لا تستهلك
  ترويسات الهوية الخاصة بـ trusted-proxy أو Tailscale Serve.
- تعامل مع تنزيلات المتصفح على أنها مدخلات غير موثوقة؛ وفضّل دليل تنزيلات معزولًا.
- عطّل مزامنة المتصفح/مديري كلمات المرور في ملف تعريف الوكيل إن أمكن (لتقليل نطاق الضرر).
- بالنسبة إلى Gateways البعيدة، افترض أن "التحكم في المتصفح" يعادل "وصول المشغّل" إلى كل ما يمكن لذلك الملف الشخصي الوصول إليه.
- أبقِ Gateway ومضيفات node على tailnet فقط؛ وتجنب كشف منافذ التحكم في المتصفح إلى LAN أو الإنترنت العام.
- عطّل توجيه proxy الخاص بالمتصفح عندما لا تحتاج إليه (`gateway.nodes.browser.mode="off"`).
- لا يُعد وضع existing-session الخاص بـ Chrome MCP "أكثر أمانًا"؛ فهو يمكن أن يتصرف نيابة عنك ضمن كل ما يمكن لملف تعريف Chrome الخاص بذلك المضيف الوصول إليه.

### سياسة SSRF الخاصة بالمتصفح (صارمة افتراضيًا)

سياسة التنقل في المتصفح لدى OpenClaw صارمة افتراضيًا: تبقى الوجهات الخاصة/الداخلية محجوبة ما لم تشترك صراحةً.

- الافتراضي: لا تكون `browser.ssrfPolicy.dangerouslyAllowPrivateNetwork` مضبوطة، لذلك يبقي التنقل عبر المتصفح الوجهات الخاصة/الداخلية/ذات الاستخدام الخاص محجوبة.
- الاسم المستعار القديم: لا يزال `browser.ssrfPolicy.allowPrivateNetwork` مقبولًا للتوافق.
- وضع الاشتراك الصريح: اضبط `browser.ssrfPolicy.dangerouslyAllowPrivateNetwork: true` للسماح بالوجهات الخاصة/الداخلية/ذات الاستخدام الخاص.
- في الوضع الصارم، استخدم `hostnameAllowlist` (أنماط مثل `*.example.com`) و`allowedHostnames` (استثناءات مضيفات دقيقة، بما فيها الأسماء المحجوبة مثل `localhost`) للحصول على استثناءات صريحة.
- يتم التحقق من التنقل قبل الطلب، ثم يُعاد التحقق بأفضل جهد على عنوان URL النهائي من نوع `http(s)` بعد التنقل لتقليل التحولات المعتمدة على إعادة التوجيه.

مثال لسياسة صارمة:

```json5
{
  browser: {
    ssrfPolicy: {
      dangerouslyAllowPrivateNetwork: false,
      hostnameAllowlist: ["*.example.com", "example.com"],
      allowedHostnames: ["localhost"],
    },
  },
}
```

## ملفات تعريف الوصول لكل وكيل (متعدد الوكلاء)

مع التوجيه متعدد الوكلاء، يمكن أن يمتلك كل وكيل سياسة Sandbox + أدوات خاصة به:
استخدم هذا لمنح **وصول كامل** أو **قراءة فقط** أو **دون وصول** لكل وكيل.
راجع [Sandbox والأدوات متعددة الوكلاء](/ar/tools/multi-agent-sandbox-tools) للحصول على التفاصيل الكاملة
وقواعد الأولوية.

حالات استخدام شائعة:

- وكيل شخصي: وصول كامل، بدون Sandbox
- وكيل عائلي/عمل: داخل Sandbox + أدوات للقراءة فقط
- وكيل عام: داخل Sandbox + بدون أدوات نظام ملفات/shell

### مثال: وصول كامل (بدون Sandbox)

```json5
{
  agents: {
    list: [
      {
        id: "personal",
        workspace: "~/.openclaw/workspace-personal",
        sandbox: { mode: "off" },
      },
    ],
  },
}
```

### مثال: أدوات للقراءة فقط + مساحة عمل للقراءة فقط

```json5
{
  agents: {
    list: [
      {
        id: "family",
        workspace: "~/.openclaw/workspace-family",
        sandbox: {
          mode: "all",
          scope: "agent",
          workspaceAccess: "ro",
        },
        tools: {
          allow: ["read"],
          deny: ["write", "edit", "apply_patch", "exec", "process", "browser"],
        },
      },
    ],
  },
}
```

### مثال: بدون وصول إلى نظام الملفات/shell (مع السماح بمراسلة الموفر)

```json5
{
  agents: {
    list: [
      {
        id: "public",
        workspace: "~/.openclaw/workspace-public",
        sandbox: {
          mode: "all",
          scope: "agent",
          workspaceAccess: "none",
        },
        // يمكن أن تكشف أدوات الجلسة بيانات حساسة من السجلات النصية. افتراضيًا يقيّد OpenClaw هذه الأدوات
        // بالجلسة الحالية + جلسات الوكيل الفرعي الناتجة، لكن يمكنك التشديد أكثر عند الحاجة.
        // راجع `tools.sessions.visibility` في مرجع التكوين.
        tools: {
          sessions: { visibility: "tree" }, // self | tree | agent | all
          allow: [
            "sessions_list",
            "sessions_history",
            "sessions_send",
            "sessions_spawn",
            "session_status",
            "whatsapp",
            "telegram",
            "slack",
            "discord",
          ],
          deny: [
            "read",
            "write",
            "edit",
            "apply_patch",
            "exec",
            "process",
            "browser",
            "canvas",
            "nodes",
            "cron",
            "gateway",
            "image",
          ],
        },
      },
    ],
  },
}
```

## الاستجابة للحوادث

إذا قام الذكاء الاصطناعي لديك بشيء سيئ:

### الاحتواء

1. **أوقفه:** أوقف تطبيق macOS (إذا كان هو من يشرف على Gateway) أو أنهِ عملية `openclaw gateway` لديك.
2. **أغلق الكشف:** اضبط `gateway.bind: "loopback"` (أو عطّل Tailscale Funnel/Serve) حتى تفهم ما حدث.
3. **جمّد الوصول:** بدّل الرسائل المباشرة/المجموعات عالية الخطورة إلى `dmPolicy: "disabled"` / اشترط الإشارات، وأزل إدخالات السماح للجميع `"*"` إذا كنت قد استخدمتها.

### التدوير (افترض حدوث اختراق إذا تسربت أسرار)

1. دوّر مصادقة Gateway (`gateway.auth.token` / `OPENCLAW_GATEWAY_PASSWORD`) وأعد التشغيل.
2. دوّر أسرار العملاء البعيدين (`gateway.remote.token` / `.password`) على أي جهاز يمكنه استدعاء Gateway.
3. دوّر بيانات اعتماد الموفّرين/API (بيانات اعتماد WhatsApp، ورموز Slack/Discord، ومفاتيح النموذج/API في `auth-profiles.json`، وقيم حمولة الأسرار المشفرة عند استخدامها).

### التدقيق

1. تحقق من سجلات Gateway: `/tmp/openclaw/openclaw-YYYY-MM-DD.log` (أو `logging.file`).
2. راجع السجلات النصية ذات الصلة: `~/.openclaw/agents/<agentId>/sessions/*.jsonl`.
3. راجع تغييرات التكوين الأخيرة (أي شيء قد يكون وسّع الوصول: `gateway.bind`، و`gateway.auth`، وسياسات الرسائل المباشرة/المجموعات، و`tools.elevated`، وتغييرات Plugin).
4. أعد تشغيل `openclaw security audit --deep` وتأكد من حل النتائج الحرجة.

### ما يجب جمعه لتقرير

- الطابع الزمني، ونظام تشغيل مضيف gateway + إصدار OpenClaw
- السجلات النصية للجلسة + مقتطف قصير من السجل (بعد الإخفاء)
- ما الذي أرسله المهاجم + ما الذي فعله الوكيل
- ما إذا كان Gateway مكشوفًا خارج loopback (LAN/Tailscale Funnel/Serve)

## فحص الأسرار (detect-secrets)

يشغّل CI hook `detect-secrets` الخاص بـ pre-commit في مهمة `secrets`.
وتشغّل عمليات الدفع إلى `main` دائمًا فحصًا لكل الملفات. وتستخدم طلبات السحب مسارًا سريعًا
للملفات المتغيرة عندما يكون commit أساسي متاحًا، وتعود إلى فحص كل الملفات
خلاف ذلك. وإذا فشل، فهناك مرشحون جدد لم يكونوا بعد في baseline.

### إذا فشل CI

1. أعد إنتاج المشكلة محليًا:

   ```bash
   pre-commit run --all-files detect-secrets
   ```

2. افهم الأدوات:
   - يقوم `detect-secrets` في pre-commit بتشغيل `detect-secrets-hook` مع baseline
     والاستثناءات الخاصة بالمستودع.
   - يفتح `detect-secrets audit` مراجعة تفاعلية لوضع علامة على كل عنصر في baseline
     باعتباره حقيقيًا أو إيجابيًا كاذبًا.
3. بالنسبة إلى الأسرار الحقيقية: دوّرها/أزلها، ثم أعد تشغيل الفحص لتحديث baseline.
4. بالنسبة إلى الإيجابيات الكاذبة: شغّل التدقيق التفاعلي وضع علامة عليها على أنها كاذبة:

   ```bash
   detect-secrets audit .secrets.baseline
   ```

5. إذا كنت تحتاج إلى استثناءات جديدة، فأضفها إلى `.detect-secrets.cfg` وأعد توليد
   baseline مع أعلام `--exclude-files` / `--exclude-lines` المطابقة (ملف
   التكوين مرجعي فقط؛ إذ لا يقرأه detect-secrets تلقائيًا).

التزم بملف `.secrets.baseline` المحدث عندما يعكس الحالة المقصودة.

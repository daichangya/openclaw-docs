واجهة Control UI هي تطبيق صفحة واحدة صغير **Vite + Lit** يقدّمه Gateway:

- الافتراضي: `http://<host>:18789/`
- بادئة اختيارية: اضبط `gateway.controlUi.basePath` (مثل `/openclaw`)

وهي تتحدث **مباشرة إلى Gateway WebSocket** على المنفذ نفسه.

## فتح سريع (محلي)

إذا كان Gateway يعمل على الجهاز نفسه، فافتح:

- [http://127.0.0.1:18789/](http://127.0.0.1:18789/) (أو [http://localhost:18789/](http://localhost:18789/))

إذا فشل تحميل الصفحة، فابدأ Gateway أولًا: `openclaw gateway`.

تُمرَّر المصادقة أثناء مصافحة WebSocket عبر:

- `connect.params.auth.token`
- `connect.params.auth.password`
- رؤوس هوية Tailscale Serve عندما تكون `gateway.auth.allowTailscale: true`
- رؤوس هوية trusted-proxy عندما تكون `gateway.auth.mode: "trusted-proxy"`

تحتفظ لوحة إعدادات لوحة التحكم برمز مميز لجلسة علامة التبويب الحالية في المتصفح
ولعنوان gateway المحدد؛ أما كلمات المرور فلا تُحفَظ. ويولّد onboarding عادةً
رمز gateway مميزًا لمصادقة السر المشترك عند أول اتصال، لكن مصادقة كلمة المرور
تعمل أيضًا عندما تكون `gateway.auth.mode` مساوية لـ `"password"`.

## اقتران الأجهزة (الاتصال الأول)

عندما تتصل بواجهة Control UI من متصفح أو جهاز جديد، يتطلب Gateway
**موافقة اقتران لمرة واحدة** — حتى إذا كنت على Tailnet نفسه
مع `gateway.auth.allowTailscale: true`. وهذا إجراء أمني لمنع
الوصول غير المصرح به.

**ما الذي ستراه:** `"disconnected (1008): pairing required"`

**للموافقة على الجهاز:**

```bash
# List pending requests
openclaw devices list

# Approve by request ID
openclaw devices approve <requestId>
```

إذا أعاد المتصفح محاولة الاقتران مع تفاصيل مصادقة متغيرة (الدور/النطاقات/المفتاح
العام)، فسيُستبدل الطلب المعلق السابق ويُنشأ `requestId` جديد.
أعد تشغيل `openclaw devices list` قبل الموافقة.

إذا كان المتصفح مقترنًا بالفعل وقمت بتغييره من وصول القراءة إلى
وصول الكتابة/الإدارة، فسيُعامل هذا على أنه ترقية موافقة، وليس إعادة
اتصال صامتة. ويُبقي OpenClaw الموافقة القديمة نشطة، ويحظر إعادة الاتصال الأوسع،
ويطلب منك الموافقة على مجموعة النطاقات الجديدة صراحةً.

وبمجرد الموافقة، يُتذكَّر الجهاز ولن يتطلب إعادة موافقة إلا إذا
ألغيتَه باستخدام `openclaw devices revoke --device <id> --role <role>`. راجع
[CLI الخاص بالأجهزة](/ar/cli/devices) لمعرفة تدوير الرموز المميزة والإلغاء.

**ملاحظات:**

- اتصالات المتصفح المحلية المباشرة عبر loopback (`127.0.0.1` / `localhost`) تتم
  الموافقة عليها تلقائيًا.
- تظل اتصالات المتصفح عبر Tailnet وLAN تتطلب موافقة صريحة، حتى عندما
  تنشأ من الجهاز نفسه.
- يولد كل ملف تعريف متصفح معرّف جهاز فريدًا، لذلك فإن تبديل المتصفحات أو
  مسح بيانات المتصفح سيتطلب إعادة الاقتران.

## الهوية الشخصية (محلية في المتصفح)

تدعم واجهة Control UI هوية شخصية لكل متصفح (اسم عرض
وصورة رمزية) تُرفق بالرسائل الصادرة لإسناد الملكية في الجلسات المشتركة. وهي
تعيش في تخزين المتصفح، ومحصورة بملف تعريف المتصفح الحالي، ولا تتم
مزامنتها إلى أجهزة أخرى أو حفظها على الخادم خارج بيانات
نسبة التأليف العادية في transcript على الرسائل التي ترسلها فعليًا. ويؤدي مسح بيانات
الموقع أو تبديل المتصفحات إلى إعادتها إلى حالة فارغة.

## نقطة نهاية إعدادات وقت التشغيل

تجلب واجهة Control UI إعدادات وقت التشغيل الخاصة بها من
`/__openclaw/control-ui-config.json`. وتخضع نقطة النهاية هذه لمصادقة
gateway نفسها المطبقة على بقية سطح HTTP: فلا يمكن للمتصفحات غير
المصادَق عليها جلبها، ويتطلب الجلب الناجح إما رمزًا/كلمة مرور
صالحة بالفعل للـ gateway، أو هوية Tailscale Serve، أو هوية trusted-proxy.

## دعم اللغة

يمكن لواجهة Control UI أن تترجم نفسها عند أول تحميل بناءً على لغة متصفحك.
وللتجاوز لاحقًا، افتح **Overview -> Gateway Access -> Language**. ويوجد
منتقي اللغة في بطاقة Gateway Access، وليس ضمن Appearance.

- اللغات المدعومة: `en`، و`zh-CN`، و`zh-TW`، و`pt-BR`، و`de`، و`es`، و`ja-JP`، و`ko`، و`fr`، و`tr`، و`uk`، و`id`، و`pl`، و`th`
- تُحمَّل الترجمات غير الإنجليزية في المتصفح عند الطلب.
- تُحفَظ اللغة المحددة في تخزين المتصفح ويُعاد استخدامها في الزيارات المستقبلية.
- تعود مفاتيح الترجمة المفقودة إلى الإنجليزية.

## ما الذي يمكنها فعله (اليوم)

- الدردشة مع النموذج عبر Gateway WS (`chat.history`، و`chat.send`، و`chat.abort`، و`chat.inject`)
- التحدث إلى OpenAI Realtime مباشرة من المتصفح عبر WebRTC. إذ
  ينشئ Gateway سر عميل Realtime قصير الأجل باستخدام `talk.realtime.session`؛ بينما
  يرسل المتصفح صوت الميكروفون مباشرة إلى OpenAI ويعيد توجيه
  استدعاءات أداة `openclaw_agent_consult` عبر `chat.send` إلى
  نموذج OpenClaw الأكبر المضبوط.
- بث استدعاءات الأدوات + بطاقات خرج الأدوات الحي في الدردشة (أحداث الوكيل)
- القنوات: حالة القنوات المضمّنة بالإضافة إلى قنوات Plugins المضمّنة/الخارجية، وتسجيل الدخول عبر QR، وإعدادات كل قناة (`channels.status`، و`web.login.*`، و`config.patch`)
- المثيلات: قائمة الحضور + التحديث (`system-presence`)
- الجلسات: السرد + تجاوزات كل جلسة للنموذج/التفكير/السريع/المطوّل/التتبع/reasoning (`sessions.list`، و`sessions.patch`)
- Dreams: حالة Dreaming، ومفتاح التفعيل/التعطيل، وقارئ Dream Diary (`doctor.memory.status`، و`doctor.memory.dreamDiary`، و`config.patch`)
- مهام Cron: السرد/الإضافة/التحرير/التشغيل/التمكين/التعطيل + سجل التشغيل (`cron.*`)
- Skills: الحالة، والتمكين/التعطيل، والتثبيت، وتحديثات مفاتيح API (`skills.*`)
- العُقد: السرد + الإمكانات (`node.list`)
- موافقات Exec: تحرير قوائم السماح وسياسة الاستئذان الخاصة بالـ gateway أو العقدة لأمر `exec host=gateway/node` (`exec.approvals.*`)
- الإعدادات: عرض/تحرير `~/.openclaw/openclaw.json` (`config.get`، و`config.set`)
- الإعدادات: التطبيق + إعادة التشغيل مع التحقق (`config.apply`) وإيقاظ آخر جلسة نشطة
- تتضمن عمليات كتابة الإعدادات حارس base-hash لمنع الكتابة فوق التعديلات المتزامنة
- كما تُجري عمليات كتابة الإعدادات (`config.set`/`config.apply`/`config.patch`) فحصًا مسبقًا لتحليل SecretRef النشط للمراجع الموجودة في حمولة config المقدمة؛ وتُرفض المراجع النشطة غير المحلولة في الحمولة المقدمة قبل الكتابة
- مخطط الإعدادات + عرض النموذج (`config.schema` / `config.schema.lookup`,
  بما في ذلك `title` / `description` الخاصين بالحقل، وتلميحات UI المطابقة، وملخصات الأبناء المباشرين، وبيانات المستندات التعريفية على العُقد المتداخلة من نوع object/wildcard/array/composition،
  بالإضافة إلى مخططات Plugins والقنوات عند توفرها)؛ ولا يكون محرر Raw JSON
  متاحًا إلا عندما تكون اللقطة قابلة لرحلة خام آمنة ذهابًا وإيابًا
- إذا لم تتمكن اللقطة من إجراء رحلة خام آمنة ذهابًا وإيابًا، فإن Control UI تفرض وضع Form وتعطّل وضع Raw لتلك اللقطة
- يقوم خيار "Reset to saved" في محرر Raw JSON بالحفاظ على الشكل الذي كُتب به raw (التنسيق، والتعليقات، وتخطيط `$include`) بدلًا من إعادة عرض لقطة مسطحة، بحيث تبقى التعديلات الخارجية بعد إعادة التعيين عندما تكون اللقطة قابلة لرحلة خام آمنة ذهابًا وإيابًا
- تُعرض قيم كائنات Structured SecretRef للقراءة فقط في حقول النص الخاصة بالنماذج لمنع الإفساد العرضي من كائن إلى سلسلة
- التصحيح: لقطات الحالة/الصحة/النماذج + سجل الأحداث + استدعاءات RPC يدوية (`status`، و`health`، و`models.list`)
- السجلات: تتبع حي لسجلات ملفات gateway مع التصفية/التصدير (`logs.tail`)
- التحديث: تشغيل تحديث حزمة/‏git + إعادة تشغيل (`update.run`) مع تقرير إعادة تشغيل

ملاحظات لوحة مهام Cron:

- بالنسبة إلى المهام المعزولة، يكون التسليم الافتراضي هو ملخص announce. ويمكنك التبديل إلى none إذا كنت تريد تشغيلات داخلية فقط.
- تظهر حقول القناة/الهدف عند اختيار announce.
- يستخدم وضع Webhook القيمة `delivery.mode = "webhook"` مع ضبط `delivery.to` على عنوان URL صالح لـ HTTP(S) Webhook.
- بالنسبة إلى مهام الجلسة الرئيسية، تكون أوضاع التسليم webhook وnone متاحة.
- تتضمن عناصر التحكم المتقدمة للتحرير خيار الحذف بعد التشغيل، ومسح تجاوز الوكيل، وخيارات cron exact/stagger،
  وتجاوزات نموذج/تفكير الوكيل، ومفاتيح التسليم بأفضل جهد.
- يكون التحقق من صحة النموذج مضمّنًا مع أخطاء على مستوى الحقول؛ والقيم غير الصالحة تعطل زر الحفظ حتى يتم إصلاحها.
- اضبط `cron.webhookToken` لإرسال bearer token مخصص؛ وإذا حُذف، يُرسل webhook من دون ترويسة مصادقة.
- الرجوع الاحتياطي القديم المهمل: لا تزال المهام القديمة المخزنة ذات `notify: true` قادرة على استخدام `cron.webhook` إلى أن يتم ترحيلها.

## سلوك الدردشة

- يكون `chat.send` **غير حاجز**: إذ يقرّ فورًا بـ `{ runId, status: "started" }` ويتدفق الرد عبر أحداث `chat`.
- تؤدي إعادة الإرسال باستخدام `idempotencyKey` نفسه إلى إرجاع `{ status: "in_flight" }` أثناء التشغيل، و`{ status: "ok" }` بعد الاكتمال.
- تكون استجابات `chat.history` مقيّدة الحجم من أجل أمان UI. وعندما تكون إدخالات transcript كبيرة جدًا، قد يقوم Gateway باقتطاع حقول النص الطويلة، وحذف كتل بيانات التعريف الثقيلة، واستبدال الرسائل كبيرة الحجم جدًا بعنصر نائب (`[chat.history omitted: message too large]`).
- تُحفَظ الصور التي أنشأها المساعد/المولَّدة كمراجع وسائط مُدارة وتُعاد خدمتها عبر عناوين URL وسائط مصادَق عليها خاصة بـ Gateway، بحيث لا تعتمد عمليات إعادة التحميل على بقاء حمولات الصور raw base64 في استجابة سجل الدردشة.
- يقوم `chat.history` أيضًا بإزالة وسوم التوجيه المضمّنة الخاصة بالعرض فقط من النص الظاهر للمساعد (مثل `[[reply_to_*]]` و`[[audio_as_voice]]`)، وحمولات XML الخاصة باستدعاءات الأدوات بالنص الصريح (بما في ذلك `<tool_call>...</tool_call>`، و`<function_call>...</function_call>`، و`<tool_calls>...</tool_calls>`، و`<function_calls>...</function_calls>`، وكتل استدعاء الأدوات المبتورة)، وكذلك بحذف رموز تحكم النموذج المسرّبة من نوع ASCII/العرض الكامل، كما يحذف إدخالات المساعد التي يكون نصها الظاهر الكامل هو الرمز الصامت الدقيق `NO_REPLY` / `no_reply` فقط.
- أثناء إرسال نشط وأثناء التحديث النهائي للسجل، يحتفظ عرض الدردشة
  برسائل المستخدم/المساعد المحلية التفاؤلية مرئية إذا أعاد `chat.history`
  مؤقتًا لقطة أقدم؛ ويستبدل transcript القياسي تلك الرسائل المحلية بمجرد
  أن يلحق سجل Gateway بالركب.
- يقوم `chat.inject` بإلحاق ملاحظة مساعد إلى transcript الجلسة ويبث حدث `chat` لتحديثات UI فقط (من دون تشغيل وكيل، ومن دون تسليم إلى القناة).
- تقوم منتقيات النموذج والتفكير في رأس الدردشة بترقيع الجلسة النشطة فورًا عبر `sessions.patch`؛ فهي تجاوزات جلسة دائمة، وليست خيارات إرسال لدورة واحدة فقط.
- عندما تُظهر تقارير استخدام جلسة Gateway الحديثة ضغطًا عاليًا على السياق، فإن
  منطقة محرر الدردشة تعرض إشعارًا بالسياق، وعند مستويات Compaction الموصى بها، زر
  compact يشغّل مسار compaction العادي للجلسة. وتُخفى
  لقطات token القديمة إلى أن يبلغ Gateway عن استخدام جديد مرة أخرى.
- يستخدم وضع Talk مزوّد صوت فوري مسجلًا يدعم جلسات WebRTC في المتصفح. اضبط OpenAI باستخدام `talk.provider: "openai"` بالإضافة إلى
  `talk.providers.openai.apiKey`، أو أعد استخدام إعداد مزوّد Voice Call الفوري.
  ولا يتلقى المتصفح مفتاح OpenAI API القياسي أبدًا؛ بل يتلقى
  فقط سر عميل Realtime المؤقت. ويدعم الصوت الفوري Google Live
  Voice Call الخلفي وجسور Google Meet، لكن ليس مسار WebRTC هذا في المتصفح
  بعد. ويتم تجميع مطالبة جلسة Realtime بواسطة Gateway؛ ولا يقبل
  `talk.realtime.session` تجاوزات تعليمات مقدمة من المتصل.
- في محرر الدردشة، يكون عنصر التحكم Talk هو زر الأمواج المجاور لزر
  الإملاء عبر الميكروفون. وعندما يبدأ Talk، يعرض صف حالة المحرر
  `Connecting Talk...`، ثم `Talk live` أثناء اتصال الصوت، أو
  `Asking OpenClaw...` بينما يستشير استدعاء أداة فورية النموذج
  الأكبر المضبوط عبر `chat.send`.
- الإيقاف:
  - انقر **Stop** (يستدعي `chat.abort`)
  - أثناء نشاط تشغيل ما، تُوضَع المتابعات العادية في الطابور. انقر **Steer** على رسالة موضوعة في الطابور لحقن تلك المتابعة داخل الدورة الجارية.
  - اكتب `/stop` (أو عبارات الإيقاف المستقلة مثل `stop`، أو `stop action`، أو `stop run`، أو `stop openclaw`، أو `please stop`) للإيقاف خارج النطاق
  - يدعم `chat.abort` القيمة `{ sessionKey }` (من دون `runId`) لإيقاف جميع التشغيلات النشطة لتلك الجلسة
- الاحتفاظ بالجزئيات عند الإلغاء:
  - عند إلغاء تشغيل، قد يظل نص المساعد الجزئي معروضًا في UI
  - يحفظ Gateway نص المساعد الجزئي الملغى في سجل transcript عندما يكون هناك خرج مخزَّن مؤقتًا
  - تتضمن الإدخالات المحفوظة بيانات تعريف الإلغاء حتى تتمكن المستهلكات الخاصة بالـ transcript من التمييز بين الجزئيات الملغاة ومخرجات الإكمال العادية

## تثبيت PWA وWeb Push

تشحن واجهة Control UI الملف `manifest.webmanifest` وعامل خدمة، لذلك
يمكن للمتصفحات الحديثة تثبيتها كتطبيق PWA مستقل. ويسمح Web Push للـ
Gateway بإيقاظ PWA المثبّتة عبر الإشعارات حتى عندما لا تكون علامة التبويب أو
نافذة المتصفح مفتوحة.

| السطح                                               | ما الذي يفعله                                                       |
| ----------------------------------------------------- | ------------------------------------------------------------------ |
| `ui/public/manifest.webmanifest`                      | ملف manifest الخاص بـ PWA. تعرض المتصفحات خيار "تثبيت التطبيق" بمجرد أن يصبح قابلاً للوصول.   |
| `ui/public/sw.js`                                     | عامل الخدمة الذي يتعامل مع أحداث `push` ونقرات الإشعارات. |
| `push/vapid-keys.json` (ضمن دليل حالة OpenClaw) | زوج مفاتيح VAPID مُولَّد تلقائيًا ويُستخدم لتوقيع حمولات Web Push.       |
| `push/web-push-subscriptions.json`                    | نقاط نهاية اشتراك المتصفح المحفوظة.                          |

تجاوز زوج مفاتيح VAPID عبر متغيرات البيئة في عملية Gateway عندما
تريد تثبيت المفاتيح (من أجل النشر متعدد المضيفين، أو تدوير الأسرار، أو
الاختبارات):

- `OPENCLAW_VAPID_PUBLIC_KEY`
- `OPENCLAW_VAPID_PRIVATE_KEY`
- `OPENCLAW_VAPID_SUBJECT` (الافتراضي `mailto:openclaw@localhost`)

تستخدم واجهة Control UI أساليب Gateway هذه المقيدة بالنطاق من أجل التسجيل
واختبار اشتراكات المتصفح:

- `push.web.vapidPublicKey` — يجلب مفتاح VAPID العام النشط.
- `push.web.subscribe` — يسجل `endpoint` بالإضافة إلى `keys.p256dh`/`keys.auth`.
- `push.web.unsubscribe` — يزيل نقطة نهاية مسجلة.
- `push.web.test` — يرسل إشعار اختبار إلى اشتراك المتصل.

إن Web Push مستقل عن مسار ترحيل APNS الخاص بـ iOS
(راجع [الإعدادات](/ar/gateway/configuration) بالنسبة إلى push المدعوم بالترحيل) وعن
الطريقة الحالية `push.test`، اللذين يستهدفان الاقتران الأصلي للأجهزة المحمولة.

## التضمينات المستضافة

يمكن لرسائل المساعد عرض محتوى ويب مستضاف داخلها باستخدام
الرمز المختصر `[embed ...]`. وتتحكم السياسة الرملية لـ iframe عبر
`gateway.controlUi.embedSandbox`:

- `strict`: يعطل تنفيذ scripts داخل التضمينات المستضافة
- `scripts`: يتيح التضمينات التفاعلية مع الحفاظ على عزل الأصل؛ وهذا هو
  الافتراضي ويكفي عادةً للألعاب/العناصر المصغّرة المستقلة في المتصفح
- `trusted`: يضيف `allow-same-origin` فوق `allow-scripts` للمستندات
  من الموقع نفسه التي تحتاج عمدًا إلى صلاحيات أقوى

مثال:

```json5
{
  gateway: {
    controlUi: {
      embedSandbox: "scripts",
    },
  },
}
```

استخدم `trusted` فقط عندما يحتاج المستند المضمَّن فعلًا إلى
سلوك same-origin. فبالنسبة إلى معظم الألعاب واللوحات التفاعلية التي ينشئها الوكيل، يكون
`scripts` هو الخيار الأكثر أمانًا.

تظل عناوين URL المطلقة الخارجية للتضمين من نوع `http(s)` محظورة افتراضيًا. وإذا كنت
تريد عمدًا أن يقوم `[embed url="https://..."]` بتحميل صفحات طرف ثالث، فاضبط
`gateway.controlUi.allowExternalEmbedUrls: true`.

## الوصول عبر tailnet (موصى به)

### Tailscale Serve المدمج (المفضل)

أبقِ Gateway على loopback ودَع Tailscale Serve يمرّره عبر proxy باستخدام HTTPS:

```bash
openclaw gateway --tailscale serve
```

افتح:

- `https://<magicdns>/` (أو `gateway.controlUi.basePath` الذي ضبطته)

افتراضيًا، يمكن لمصادقة طلبات Serve الخاصة بـ Control UI/WebSocket أن تتم عبر رؤوس هوية Tailscale
(`tailscale-user-login`) عندما تكون `gateway.auth.allowTailscale` مساوية لـ `true`. ويتحقق OpenClaw
من الهوية عبر تحليل عنوان `x-forwarded-for` بواسطة
`tailscale whois` ومطابقته مع الترويسة، ولا يقبل هذه الآلية إلا عندما يصل
الطلب إلى loopback مع رؤوس `x-forwarded-*` الخاصة بـ Tailscale. اضبط
`gateway.auth.allowTailscale: false` إذا كنت تريد فرض بيانات اعتماد صريحة بسر مشترك
حتى على حركة Serve. ثم استخدم `gateway.auth.mode: "token"` أو
`"password"`.
وفي هذا المسار غير المتزامن لهوية Serve، تُسلسَل محاولات المصادقة الفاشلة الخاصة بعنوان IP
العميل نفسه ونطاق المصادقة نفسه قبل عمليات كتابة حد المعدل. ولذلك يمكن أن
تُظهر عمليات إعادة المحاولة السيئة المتزامنة من المتصفح نفسه الرسالة `retry later`
في الطلب الثاني بدلًا من سباق حالتي عدم تطابق عاديتين بالتوازي.
وتفترض مصادقة Serve من دون token أن مضيف gateway موثوق. وإذا كان من الممكن
تشغيل كود محلي غير موثوق على ذلك المضيف، فاطلب مصادقة token/كلمة مرور.

### الربط بـ tailnet + token

```bash
openclaw gateway --bind tailnet --token "$(openssl rand -hex 32)"
```

ثم افتح:

- `http://<tailscale-ip>:18789/` (أو `gateway.controlUi.basePath` الذي ضبطته)

الصق السر المشترك المطابق في إعدادات واجهة المستخدم (ويُرسل على شكل
`connect.params.auth.token` أو `connect.params.auth.password`).

## HTTP غير الآمن

إذا فتحت لوحة التحكم عبر HTTP غير المشفّر (`http://<lan-ip>` أو `http://<tailscale-ip>`)،
فإن المتصفح يعمل في **سياق غير آمن** ويمنع WebCrypto. وافتراضيًا،
يقوم OpenClaw **بحظر** اتصالات Control UI من دون هوية جهاز.

الاستثناءات الموثقة:

- توافق HTTP غير الآمن الخاص بـ localhost فقط مع `gateway.controlUi.allowInsecureAuth=true`
- نجاح مصادقة operator في Control UI عبر `gateway.auth.mode: "trusted-proxy"`
- وضع كسر الزجاج `gateway.controlUi.dangerouslyDisableDeviceAuth=true`

**الإصلاح الموصى به:** استخدم HTTPS (Tailscale Serve) أو افتح الواجهة محليًا:

- `https://<magicdns>/` ‏(Serve)
- `http://127.0.0.1:18789/` ‏(على مضيف gateway)

**سلوك مفتاح المصادقة غير الآمنة:**

```json5
{
  gateway: {
    controlUi: { allowInsecureAuth: true },
    bind: "tailnet",
    auth: { mode: "token", token: "replace-me" },
  },
}
```

إن `allowInsecureAuth` هو مفتاح توافق محلي فقط:

- يتيح لجلسات Control UI المحلية على localhost أن تستمر من دون هوية جهاز في
  سياقات HTTP غير الآمنة.
- وهو لا يتجاوز فحوصات الاقتران.
- ولا يخفف متطلبات هوية الجهاز البعيدة (غير localhost).

**للكسر الزجاجي فقط:**

```json5
{
  gateway: {
    controlUi: { dangerouslyDisableDeviceAuth: true },
    bind: "tailnet",
    auth: { mode: "token", token: "replace-me" },
  },
}
```

يقوم `dangerouslyDisableDeviceAuth` بتعطيل فحوصات هوية جهاز Control UI وهو
تخفيض أمني شديد. أعده بسرعة بعد الاستخدام الطارئ.

ملاحظة trusted-proxy:

- يمكن أن تسمح مصادقة trusted-proxy الناجحة بجلسات Control UI **operator**
  من دون هوية جهاز
- لا يمتد هذا إلى جلسات Control UI ذات دور العقدة
- كما أن reverse proxies على loopback في المضيف نفسه لا تحقق مصادقة trusted-proxy؛ راجع
  [مصادقة trusted proxy](/ar/gateway/trusted-proxy-auth)

راجع [Tailscale](/ar/gateway/tailscale) للحصول على إرشادات إعداد HTTPS.

## سياسة أمن المحتوى

تشحن واجهة Control UI مع سياسة `img-src` محكمة: لا يُسمح إلا بالأصول **من الأصل نفسه**، وعناوين URL من نوع `data:`، وعناوين `blob:` المولَّدة محليًا. وتُرفض عناوين الصور البعيدة من نوع `http(s)` والعناوين النسبية للبروتوكول من قبل المتصفح ولا تصدر طلبات جلب للشبكة.

ما الذي يعنيه هذا عمليًا:

- لا تزال الصور الرمزية والصور المخدومة تحت مسارات نسبية (مثل `/avatars/<id>`) تُعرض، بما في ذلك مسارات الصور الرمزية المصادَق عليها التي تجلبها الواجهة وتحولها إلى عناوين `blob:` محلية.
- لا تزال عناوين `data:image/...` المضمّنة تُعرض (وهو مفيد للحمولات داخل البروتوكول).
- لا تزال عناوين `blob:` المحلية التي تنشئها واجهة Control UI تُعرض.
- تُزال عناوين الصور الرمزية البعيدة التي تصدرها بيانات تعريف القنوات في مساعدات الصور الرمزية الخاصة بواجهة Control UI وتُستبدل بالشعار/الشارة المضمّنة، بحيث لا تتمكن قناة مخترقة أو خبيثة من فرض عمليات جلب صور بعيدة اعتباطية من متصفح المشغّل.

لا تحتاج إلى تغيير أي شيء للحصول على هذا السلوك — فهو مفعّل دائمًا وغير قابل للتهيئة.

## مصادقة مسار الصورة الرمزية

عندما تكون مصادقة gateway مضبوطة، تتطلب نقطة نهاية الصورة الرمزية في Control UI الرمز المميز نفسه الخاص بالـ gateway كما في بقية API:

- يعيد `GET /avatar/<agentId>` صورة الصورة الرمزية فقط إلى المستدعين المصادَق عليهم. ويعيد `GET /avatar/<agentId>?meta=1` بيانات تعريف الصورة الرمزية تحت القاعدة نفسها.
- تُرفض الطلبات غير المصادَق عليها إلى أي من المسارين (بما يطابق مسار وسائط المساعد المجاور). ويمنع ذلك مسار الصورة الرمزية من تسريب هوية الوكيل على المضيفين المحميين.
- تقوم واجهة Control UI نفسها بتمرير رمز gateway المميز كترويسة bearer عند جلب الصور الرمزية، وتستخدم عناوين blob مصادَق عليها بحيث تظل الصورة تُعرض داخل لوحات التحكم.

إذا عطلت مصادقة gateway (وهو غير موصى به على المضيفين المشتركين)، فإن مسار الصورة الرمزية يصبح أيضًا غير مصادَق عليه، بما يتماشى مع بقية gateway.

## بناء الواجهة

يقدّم Gateway الملفات الثابتة من `dist/control-ui`. وابنِها باستخدام:

```bash
pnpm ui:build
```

قاعدة مطلقة اختيارية (عندما تريد عناوين URL ثابتة للأصول):

```bash
OPENCLAW_CONTROL_UI_BASE_PATH=/openclaw/ pnpm ui:build
```

للتطوير المحلي (خادم تطوير منفصل):

```bash
pnpm ui:dev
```

ثم وجّه الواجهة إلى عنوان Gateway WS URL الخاص بك (مثل `ws://127.0.0.1:18789`).

## التصحيح/الاختبار: خادم التطوير + Gateway بعيد

واجهة Control UI هي ملفات ثابتة؛ وهدف WebSocket قابل للتهيئة ويمكن أن
يكون مختلفًا عن أصل HTTP. وهذا مفيد عندما تريد خادم Vite للتطوير
محليًا بينما يعمل Gateway في مكان آخر.

1. ابدأ خادم تطوير الواجهة: `pnpm ui:dev`
2. افتح عنوان URL مثل:

```text
http://localhost:5173/?gatewayUrl=ws://<gateway-host>:18789
```

مصادقة اختيارية لمرة واحدة (عند الحاجة):

```text
http://localhost:5173/?gatewayUrl=wss://<gateway-host>:18789#token=<gateway-token>
```

ملاحظات:

- يُخزَّن `gatewayUrl` في localStorage بعد التحميل ويُزال من عنوان URL.
- يجب تمرير `token` عبر جزء URL (`#token=...`) كلما أمكن. فالأجزاء لا تُرسل إلى الخادم، مما يتجنب تسربها في سجلات الطلبات وReferer. ولا تزال معاملات الاستعلام القديمة `?token=` تُستورد مرة واحدة من أجل التوافق، ولكن كرجوع احتياطي فقط، وتُزال فورًا بعد التهيئة.
- تبقى `password` في الذاكرة فقط.
- عندما يكون `gatewayUrl` مضبوطًا، لا تعود الواجهة إلى بيانات اعتماد الإعدادات أو البيئة.
  قدّم `token` (أو `password`) صراحةً. وغياب بيانات اعتماد صريحة يعد خطأ.
- استخدم `wss://` عندما يكون Gateway خلف TLS (مثل Tailscale Serve، أو HTTPS proxy، وما إلى ذلك).
- لا يُقبل `gatewayUrl` إلا في نافذة من المستوى الأعلى (غير مضمّنة) لمنع clickjacking.
- يجب أن تضبط عمليات نشر Control UI غير المعتمدة على loopback القيمة `gateway.controlUi.allowedOrigins`
  صراحةً (أصول كاملة). ويشمل ذلك إعدادات التطوير البعيدة.
- لا تستخدم `gateway.controlUi.allowedOrigins: ["*"]` إلا في
  الاختبارات المحلية المضبوطة بإحكام. فهي تعني السماح لأي أصل متصفح، وليس “مطابقة أي مضيف
  أستخدمه”.
- يؤدي `gateway.controlUi.dangerouslyAllowHostHeaderOriginFallback=true` إلى تفعيل
  وضع الرجوع الاحتياطي إلى أصل ترويسة Host، لكنه وضع أمني خطير.

مثال:

```json5
{
  gateway: {
    controlUi: {
      allowedOrigins: ["http://localhost:5173"],
    },
  },
}
```

تفاصيل إعداد الوصول البعيد: [الوصول عن بُعد](/ar/gateway/remote).

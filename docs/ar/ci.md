# مسار CI

يعمل CI عند كل عملية push إلى `main` وكل طلب سحب. ويستخدم تحديد نطاق ذكيًا لتخطي المهام المكلفة عندما تكون التغييرات محصورة في مناطق غير مرتبطة.

## ترتيب الإخفاق السريع

تُرتَّب المهام بحيث تفشل الفحوصات الرخيصة قبل تشغيل المهام المكلفة:

1. يقرر `preflight` أي المسارات موجودة أصلًا. ويكون منطق `docs-scope` و`changed-scope` خطوات داخل هذه المهمة، وليس مهامًا مستقلة.
2. تفشل `security-scm-fast` و`security-dependency-audit` و`security-fast` و`check` و`check-additional` و`check-docs` و`skills-python` بسرعة من دون انتظار مهام المخرجات ومصفوفات المنصات الأثقل.
3. يتداخل `build-artifacts` مع مسارات Linux السريعة حتى تتمكن الجهات اللاحقة من البدء بمجرد جاهزية البناء المشترك.
4. بعد ذلك تتفرع مسارات المنصات وبيئات التشغيل الأثقل: `checks-fast-core` و`checks-fast-contracts-channels` و`checks-node-extensions` و`checks-node-core-test` و`extension-fast` و`checks` و`checks-windows` و`macos-node` و`macos-swift` و`android`.

يوجد منطق النطاق في `scripts/ci-changed-scope.mjs` وتغطيه اختبارات الوحدة في `src/scripts/ci-changed-scope.test.ts`.
ويعيد سير العمل المنفصل `install-smoke` استخدام سكربت النطاق نفسه عبر مهمة `preflight` الخاصة به. وهو يحسب `run_install_smoke` من إشارة changed-smoke الأضيق، لذلك لا يعمل Docker/install smoke إلا للتغييرات المرتبطة بالتثبيت والتغليف والحاويات.

يوجد منطق المسارات المتغيرة المحلي في `scripts/changed-lanes.mjs` ويُنفَّذ بواسطة `scripts/check-changed.mjs`. وهذه البوابة المحلية أكثر صرامة بشأن حدود البنية من نطاق منصة CI العام: تغييرات إنتاج core تشغّل فحص أنواع إنتاج core بالإضافة إلى اختبارات core، وتغييرات اختبارات core فقط تشغّل فقط فحص أنواع اختبارات core واختباراتها، وتغييرات إنتاج extension تشغّل فحص أنواع إنتاج extension بالإضافة إلى اختبارات extension، وتغييرات اختبارات extension فقط تشغّل فقط فحص أنواع اختبارات extension واختباراتها. أما تغييرات Plugin SDK العامة أو عقود Plugin فتوسّع التحقق ليشمل extensions لأن extensions تعتمد على تلك العقود الأساسية. وتفشل تغييرات الجذر/الإعدادات غير المعروفة بشكل آمن إلى جميع المسارات.

في عمليات push، تضيف مصفوفة `checks` مسار `compat-node22` الخاص بعمليات push فقط. وفي طلبات السحب، يُتخطى هذا المسار وتبقى المصفوفة مركزة على مسارات الاختبار/القنوات العادية.

تُقسَّم أبطأ عائلات اختبارات Node إلى أجزاء عبر include-file بحيث تبقى كل مهمة صغيرة: تُقسَّم عقود القنوات إلى تغطية السجل والتغطية الأساسية إلى ثمانية أجزاء موزونة لكل منها، وتُقسَّم اختبارات أوامر الرد auto-reply إلى أربعة أجزاء بأنماط include، وتُقسَّم مجموعات بادئات الرد auto-reply الكبيرة الأخرى إلى جزأين لكل مجموعة. كما يفصل `check-additional` بين أعمال التجميع/الاختبار التجريبي الخاصة بحدود الحزم وبين أعمال طوبولوجيا التشغيل الخاصة بـ Gateway/البنية.

قد يضع GitHub علامة `cancelled` على المهام المستبدلة عندما يصل push أحدث إلى طلب السحب نفسه أو إلى مرجع `main`. تعامل مع ذلك على أنه ضجيج CI ما لم يكن أحدث تشغيل للمرجع نفسه فاشلًا أيضًا. وتشير فحوصات الأجزاء المجمعة إلى حالة الإلغاء هذه صراحةً لتسهيل تمييزها عن فشل الاختبار.

## المكافئات المحلية

```bash
pnpm changed:lanes   # افحص مصنّف المسارات المتغيرة المحلي لـ origin/main...HEAD
pnpm check:changed   # بوابة محلية ذكية: typecheck/lint/tests المتغيرة حسب مسار الحدود
pnpm check          # بوابة محلية سريعة: tsgo للإنتاج + lint موزع على أجزاء + حواجز سريعة متوازية
pnpm check:test-types
pnpm check:timed    # البوابة نفسها مع توقيتات لكل مرحلة
pnpm build:strict-smoke
pnpm check:architecture
pnpm test:gateway:watch-regression
pnpm test           # اختبارات vitest
pnpm test:channels
pnpm test:contracts:channels
pnpm check:docs     # تنسيق الوثائق + lint + الروابط المعطلة
pnpm build          # ابنِ dist عندما تكون مسارات CI الخاصة بالمخرجات/‏build-smoke مهمة
```

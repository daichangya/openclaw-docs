يعمل CI عند كل push إلى `main` وكل pull request. ويستخدم تحديد نطاق ذكيًا لتخطي المهام المكلفة عندما تكون التغييرات مقتصرة على مناطق غير ذات صلة.

يمتلك QA Lab مسارات CI مخصصة خارج سير العمل الرئيسي ذي النطاق الذكي. يعمل
سير العمل `Parity gate` عند تغييرات PR المطابقة وعند التشغيل اليدوي؛ إذ
يبني بيئة تشغيل QA الخاصة ويقارن بين حزم الوكلاء mock GPT-5.4 وOpus 4.6
الوكيلية. يعمل سير العمل `QA-Lab - All Lanes` ليلًا على `main` وعند
التشغيل اليدوي؛ وهو يوزّع mock parity gate، ومسار Matrix الحي، ومسار
Telegram الحي كمهام متوازية. تستخدم المهام الحية البيئة `qa-live-shared`،
ويستخدم مسار Telegram عقود Convex. كما يشغّل `OpenClaw Release
Checks` مسارات QA Lab نفسها قبل الموافقة على الإصدار.

سير العمل `Duplicate PRs After Merge` هو سير عمل يدوي للمشرفين من أجل
تنظيف PRs المكررة بعد الدمج. ويكون افتراضيًا في وضع التشغيل التجريبي
ويغلق فقط PRs المدرجة صراحة عند `apply=true`. قبل تعديل GitHub، يتحقق من أن
PR الذي تم إنزاله مدموج، وأن لكل عنصر مكرر إما issue مرجعية مشتركة
أو مقاطع تغييرات متداخلة.

سير العمل `Docs Agent` هو مسار صيانة Codex معتمد على الأحداث للحفاظ على
اتساق المستندات الحالية مع التغييرات التي نُزلت مؤخرًا. لا يملك جدولًا
خالصًا: إذ يمكن لتشغيل CI ناجح غير صادر عن روبوت على `main` أن يفعّله، كما
يمكن للتشغيل اليدوي أن يشغّله مباشرة. تتخطى استدعاءات workflow-run التنفيذ
إذا كان `main` قد تقدم أو إذا تم إنشاء تشغيل `Docs Agent` آخر غير متخطى
خلال الساعة الماضية. وعندما يعمل، فإنه يراجع نطاق الالتزامات من SHA
المصدري السابق لآخر `Docs Agent` غير متخطى إلى `main` الحالي، لذلك يمكن
لتشغيل واحد كل ساعة أن يغطي كل تغييرات main المتراكمة منذ آخر مرور على
المستندات.

سير العمل `Test Performance Agent` هو مسار صيانة Codex معتمد على الأحداث
للاختبارات البطيئة. لا يملك جدولًا خالصًا: إذ يمكن لتشغيل CI ناجح غير صادر
عن روبوت على `main` أن يفعّله، لكنه يتخطى التنفيذ إذا كان هناك استدعاء
workflow-run آخر قد عمل بالفعل أو ما يزال يعمل في ذلك اليوم حسب UTC.
يتجاوز التشغيل اليدوي بوابة النشاط اليومية هذه. يبني هذا المسار تقرير
أداء Vitest مجمعًا للمجموعة الكاملة، ويسمح لـ Codex بإجراء إصلاحات صغيرة
فقط لأداء الاختبارات مع الحفاظ على التغطية بدلًا من إعادة هيكلة واسعة، ثم
يعيد تشغيل تقرير المجموعة الكاملة ويرفض التغييرات التي تقلل عدد اختبارات
خط الأساس الناجحة. إذا كان خط الأساس يحتوي على اختبارات فاشلة، فيجوز لـ
Codex إصلاح الإخفاقات الواضحة فقط، ويجب أن ينجح تقرير المجموعة الكاملة بعد
الوكيل قبل تثبيت أي شيء. وعندما يتقدم `main` قبل وصول push الروبوت،
يعيد المسار تأسيس الرقعة التي تم التحقق منها، ويعيد تشغيل `pnpm check:changed`،
ويحاول push مرة أخرى؛ أما الرقع القديمة المتعارضة فيتم تخطيها. ويستخدم
Ubuntu المستضاف من GitHub حتى يتمكن إجراء Codex من الحفاظ على وضع
السلامة drop-sudo نفسه الذي يستخدمه وكيل المستندات.

```bash
gh workflow run duplicate-after-merge.yml \
  -f landed_pr=70532 \
  -f duplicate_prs='70530,70592' \
  -f apply=true
```

## ترتيب الإخفاق السريع

تُرتَّب المهام بحيث تفشل الفحوصات الرخيصة قبل تشغيل المهام المكلفة:

1. يقرر `preflight` أي المسارات توجد أصلًا. منطق `docs-scope` و`changed-scope` هو خطوات داخل هذه المهمة، وليس مهام مستقلة.
2. تفشل `security-scm-fast` و`security-dependency-audit` و`security-fast` و`check` و`check-additional` و`check-docs` و`skills-python` بسرعة من دون انتظار مهام العناصر المبنية والمنصات الأثقل.
3. يتداخل `build-artifacts` مع مسارات Linux السريعة حتى يتمكن المستهلكون downstream من البدء بمجرد أن يصبح البناء المشترك جاهزًا.
4. بعد ذلك تتشعب مسارات المنصات ووقت التشغيل الأثقل: `checks-fast-core` و`checks-fast-contracts-channels` و`checks-node-extensions` و`checks-node-core-test` و`extension-fast` الخاص بطلبات السحب فقط و`checks` و`checks-windows` و`macos-node` و`macos-swift` و`android`.

يوجد منطق النطاق في `scripts/ci-changed-scope.mjs` وتغطيه اختبارات وحدات في `src/scripts/ci-changed-scope.test.ts`.
تتحقق تعديلات سير عمل CI من رسم CI البياني لـ Node بالإضافة إلى lint لسير العمل، لكنها لا تفرض بمفردها عمليات بناء أصلية لـ Windows أو Android أو macOS؛ إذ تبقى مسارات تلك المنصات محصورة في تغييرات مصدر المنصة.
تستخدم التعديلات الخاصة بتوجيه CI فقط، وتعديلات محددة ورخيصة على تركيبات اختبارات core، والتعديلات الضيقة على مساعدات/توجيه اختبارات عقود Plugin مسار بيان Node-only سريعًا: preflight، والأمان، ومهمة `checks-fast-core` واحدة. يتجنب هذا المسار عناصر البناء، وتوافق Node 22، وعقود القنوات، والأجزاء الكاملة لـ core، وأجزاء Plugins المضمنة، ومصفوفات الحواجز الإضافية عندما تقتصر الملفات المتغيرة على أسطح التوجيه أو المساعدات التي تمارسها المهمة السريعة مباشرة.
تقتصر فحوصات Windows Node على الأغلفة الخاصة بالعمليات/المسارات في Windows، ومساعدات مشغل npm/pnpm/UI، وإعداد مدير الحزم، وأسطح سير عمل CI التي تنفذ ذلك المسار؛ أما تغييرات المصدر أو Plugin أو install-smoke أو الاختبارات فقط غير ذات الصلة فتبقى على مسارات Linux Node حتى لا تحجز عامل Windows ‏16-vCPU من أجل تغطية تتم ممارستها بالفعل عبر أجزاء الاختبار العادية.
يعيد سير العمل المنفصل `install-smoke` استخدام نص النطاق نفسه عبر مهمة `preflight` الخاصة به. ويقسّم تغطية smoke إلى `run_fast_install_smoke` و`run_full_install_smoke`. تشغّل طلبات السحب المسار السريع لأسطح Docker/package، وتغييرات package/manifest الخاصة بـ Plugin المضمن، وأسطح core plugin/channel/gateway/Plugin SDK التي تمارسها مهام Docker smoke. أما تغييرات Plugin المضمنة المصدرية فقط، والتعديلات الخاصة بالاختبارات فقط، والتعديلات الخاصة بالمستندات فقط، فلا تحجز عمال Docker. يبني المسار السريع صورة Dockerfile الجذر مرة واحدة، ويفحص CLI، ويشغّل smoke لـ CLI الخاص بحذف agents من مساحة العمل المشتركة، ويشغّل e2e لشبكة gateway داخل الحاوية، ويتحقق من وسيط build arg لامتداد مضمن، ويشغّل ملف تعريف Docker المحدود لـ Plugin المضمن تحت مهلة إجمالية للأوامر قدرها 240 ثانية مع تحديد سقف منفصل لكل سيناريو لتشغيل Docker. ويحافظ المسار الكامل على تغطية تثبيت حزمة QR وتغطية Docker/update الخاصة بالمثبّت لعمليات التشغيل الليلية المجدولة، وعمليات التشغيل اليدوية، وفحوصات الإصدارات عبر workflow-call، وطلبات السحب التي تمس فعلًا أسطح المثبّت/package/Docker. إن عمليات push إلى `main`، بما في ذلك merge commits، لا تفرض المسار الكامل؛ وعندما يطلب منطق changed-scope تغطية كاملة على push، يحتفظ سير العمل بـ Docker smoke السريع ويترك install smoke الكامل للتحقق الليلي أو تحقق الإصدار. يتم أيضًا تقييد smoke البطيء لموفر الصور في تثبيت Bun العمومي بشكل منفصل بواسطة `run_bun_global_install_smoke`; فهو يعمل في الجدول الليلي ومن سير عمل فحوصات الإصدار، ويمكن لعمليات dispatch اليدوية لـ `install-smoke` اختياره، لكن طلبات السحب وعمليات push إلى `main` لا تشغّله. تحتفظ اختبارات QR وDocker الخاصة بالمثبّت بملفات Dockerfiles الخاصة بها والمركزة على التثبيت. يقوم `test:docker:all` المحلي بالبناء المسبق لصورة live-test مشتركة واحدة وصورة built-app مشتركة واحدة من `scripts/e2e/Dockerfile`، ثم يشغّل مسارات smoke الحية/E2E باستخدام مجدول مرجّح ومع `OPENCLAW_SKIP_DOCKER_BUILD=1`; اضبط العدد الافتراضي لفتحات التجمع الرئيسي وهو 10 عبر `OPENCLAW_DOCKER_ALL_PARALLELISM`، واضبط عدد فتحات تجمع الذيل الحساس للموفر وهو 10 عبر `OPENCLAW_DOCKER_ALL_TAIL_PARALLELISM`. الحدود القصوى للمسارات الثقيلة تكون افتراضيًا `OPENCLAW_DOCKER_ALL_LIVE_LIMIT=6` و`OPENCLAW_DOCKER_ALL_NPM_LIMIT=8` و`OPENCLAW_DOCKER_ALL_SERVICE_LIMIT=7` حتى لا تبالغ مسارات npm install ومتعددة الخدمات في حجز Docker بينما تملأ المسارات الأخف الفتحات المتاحة. يتم ترتيب بدايات المسارات بفاصل ثانيتين افتراضيًا لتجنب عواصف الإنشاء المحلية في Docker daemon؛ ويمكن تجاوز ذلك عبر `OPENCLAW_DOCKER_ALL_START_STAGGER_MS=0` أو قيمة أخرى بالمللي ثانية. يجري المجمّع المحلي المسبق فحوصات مسبقة لـ Docker، ويزيل حاويات OpenClaw E2E القديمة، ويصدر حالة المسارات النشطة، ويحفظ توقيتات المسارات لترتيب الأطول أولًا، ويدعم `OPENCLAW_DOCKER_ALL_DRY_RUN=1` لفحص المجدول. ويتوقف افتراضيًا عن جدولة مسارات مجمعة جديدة بعد أول فشل، ولكل مسار مهلة احتياطية قدرها 120 دقيقة يمكن تجاوزها عبر `OPENCLAW_DOCKER_ALL_LANE_TIMEOUT_MS`; وتستخدم مسارات حية/ذيل محددة حدودًا أشد لكل مسار. ويعكس سير العمل الحي/E2E القابل لإعادة الاستخدام نمط الصورة المشتركة عبر بناء ودفع صورة Docker E2E واحدة معلمة بـ SHA إلى GHCR قبل مصفوفة Docker، ثم تشغيل المصفوفة مع `OPENCLAW_SKIP_DOCKER_BUILD=1`. يشغّل سير العمل الحي/E2E المجدول مجموعة Docker الكاملة لمسار الإصدار يوميًا. ويتم تقسيم مصفوفة التحديث المضمن حسب هدف التحديث حتى تتمكن عمليات npm update وdoctor repair المتكررة من التجزئة مع فحوصات مضمنة أخرى.

يوجد منطق changed-lane المحلي في `scripts/changed-lanes.mjs` ويتم تنفيذه بواسطة `scripts/check-changed.mjs`. هذه البوابة المحلية أكثر صرامة بشأن حدود البنية المعمارية من نطاق منصة CI العام: تشغّل تغييرات إنتاج core فحص أنواع prod لـ core بالإضافة إلى اختبارات core، وتشغّل تغييرات اختبارات core فقط فحص أنواع/اختبارات core test فقط، وتشغّل تغييرات إنتاج extension فحص أنواع prod لـ extension بالإضافة إلى اختبارات extension، وتشغّل تغييرات اختبارات extension فقط فحص أنواع/اختبارات extension test فقط. توسّع تغييرات Plugin SDK العامة أو عقود Plugin التحقق إلى extensions لأن extensions تعتمد على تلك العقود الأساسية. وتشغّل زيادات الإصدارات الخاصة ببيانات الإصدار الوصفية فقط فحوصات موجهة للإصدار/الإعداد/التبعيات الجذرية. أما تغييرات الجذر/الإعداد غير المعروفة فتفشل بأمان إلى جميع المسارات.

في عمليات push، تضيف مصفوفة `checks` مسار `compat-node22` الخاص بعمليات push فقط. وفي طلبات السحب، يتم تخطي ذلك المسار وتبقى المصفوفة مركزة على مسارات الاختبار/القنوات المعتادة.

يتم تقسيم أو موازنة أبطأ عائلات اختبارات Node بحيث تبقى كل مهمة صغيرة من دون المبالغة في حجز المشغلات: تعمل عقود القنوات كثلاثة أجزاء مرجّحة، وتتم موازنة اختبارات Plugins المضمنة عبر ستة عمال extension، ويتم إقران مسارات وحدات core الصغيرة، ويعمل auto-reply عبر ثلاثة عمال متوازنين بدلًا من ستة عمال صغار، ويتم توزيع إعدادات الوكيلية gateway/plugin عبر مهام agentic Node المصدرية فقط الموجودة بدلًا من انتظار العناصر المبنية. وتستخدم اختبارات المتصفح الواسعة وQA والوسائط وPlugins المتنوعة إعدادات Vitest المخصصة لها بدلًا من catch-all المشترك للـ Plugin. تشغّل مهام تجزئة extension حتى مجموعتي إعداد Plugin في الوقت نفسه مع عامل Vitest واحد لكل مجموعة وذاكرة Node heap أكبر حتى لا تنشئ دفعات Plugins الثقيلة من حيث الاستيراد مهام CI إضافية. ويستخدم مسار agents الواسع مجدول Vitest المشترك المتوازي على الملفات لأنه يهيمن عليه الاستيراد/الجدولة بدلًا من أن يملكه ملف اختبار بطيء واحد. يعمل `runtime-config` مع جزء infra core-runtime حتى لا يملك جزء وقت التشغيل المشترك الذيل. ويحافظ `check-additional` على أعمال الترجمة/التحقق الحدودية للحزم مجتمعة ويفصل بين بنية طوبولوجيا وقت التشغيل وتغطية gateway watch؛ ويشغّل جزء حارس الحدود حواجزه الصغيرة المستقلة بالتوازي داخل مهمة واحدة. وتعمل gateway watch واختبارات القنوات وجزء حدود دعم core بالتوازي داخل `build-artifacts` بعد أن يكون `dist/` و`dist-runtime/` قد بُنيا بالفعل، مع الإبقاء على أسماء الفحوصات القديمة كمهام تحقق خفيفة مع تجنب عاملين إضافيين من Blacksmith وطابور ثانٍ لمستهلكي العناصر المبنية.
يشغّل Android CI كلاً من `testPlayDebugUnitTest` و`testThirdPartyDebugUnitTest`، ثم يبني Play debug APK. لا تحتوي نكهة الطرف الثالث على source set أو manifest منفصل؛ لكن مسار اختبار الوحدات الخاص بها ما يزال يترجم تلك النكهة باستخدام علامات BuildConfig الخاصة بـ SMS/call-log، مع تجنب مهمة تعبئة debug APK مكررة في كل push ذي صلة بـ Android.
يكون `extension-fast` خاصًا بطلبات السحب فقط لأن عمليات push تشغّل بالفعل الأجزاء الكاملة لـ Plugin المضمن. وهذا يحافظ على تغذية راجعة لتغييرات Plugin أثناء المراجعات من دون حجز عامل Blacksmith إضافي على `main` لتغطية موجودة بالفعل في `checks-node-extensions`.

قد يضع GitHub علامة `cancelled` على المهام المستبدلة عندما يصل push أحدث إلى PR نفسه أو إلى مرجع `main` نفسه. تعامل مع ذلك كضوضاء CI ما لم يكن أحدث تشغيل للمرجع نفسه فاشلًا أيضًا. تستخدم فحوصات الأجزاء المجمعة `!cancelled() && always()` بحيث تستمر في الإبلاغ عن إخفاقات الأجزاء العادية لكنها لا تدخل في قائمة الانتظار بعد أن يكون سير العمل بأكمله قد استُبدل بالفعل.
تمت مراجعة مفتاح التزامن في CI (`CI-v7-*`) حتى لا يتمكن zombie على جهة GitHub ضمن مجموعة انتظار قديمة من حظر عمليات main الأحدث إلى أجل غير مسمى.

## المكافئات المحلية

```bash
pnpm changed:lanes   # inspect the local changed-lane classifier for origin/main...HEAD
pnpm check:changed   # smart local gate: changed typecheck/lint/tests by boundary lane
pnpm check          # fast local gate: production tsgo + sharded lint + parallel fast guards
pnpm check:test-types
pnpm check:timed    # same gate with per-stage timings
pnpm build:strict-smoke
pnpm check:architecture
pnpm test:gateway:watch-regression
pnpm test           # vitest tests
pnpm test:channels
pnpm test:contracts:channels
pnpm check:docs     # docs format + lint + broken links
pnpm build          # build dist when CI artifact/build-smoke lanes matter
node scripts/ci-run-timings.mjs <run-id>      # summarize wall time, queue time, and slowest jobs
node scripts/ci-run-timings.mjs --recent 10   # compare recent successful main CI runs
pnpm test:perf:groups --full-suite --allow-failures --output .artifacts/test-perf/baseline-before.json
pnpm test:perf:groups:compare .artifacts/test-perf/baseline-before.json .artifacts/test-perf/after-agent.json
```

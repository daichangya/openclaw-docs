La CI se ejecuta en cada push a `main` y en cada pull request. Usa alcance inteligente para omitir trabajos costosos cuando solo cambiaron áreas no relacionadas.

QA Lab tiene lanes de CI dedicadas fuera del flujo principal con alcance inteligente. El flujo de trabajo
`Parity gate` se ejecuta en cambios de PR coincidentes y mediante dispatch manual; compila el runtime privado de QA y compara los paquetes agentic simulados de GPT-5.4 y Opus 4.6. El flujo de trabajo `QA-Lab - All Lanes` se ejecuta cada noche en `main` y mediante dispatch manual; despliega en paralelo el parity gate simulado, la lane live de Matrix y la lane live de Telegram. Los trabajos live usan el entorno `qa-live-shared`, y la lane de Telegram usa leases de Convex. `OpenClaw Release Checks` también ejecuta las mismas lanes de QA Lab antes de la aprobación de lanzamiento.

El flujo de trabajo `Duplicate PRs After Merge` es un flujo manual para mantenedores orientado a limpiar duplicados después de hacer land. Por defecto usa dry-run y solo cierra PR listados explícitamente cuando `apply=true`. Antes de modificar GitHub, verifica que el PR aterrizado esté fusionado y que cada duplicado tenga un issue referenciado compartido o hunks cambiados superpuestos.

El flujo de trabajo `Docs Agent` es una lane de mantenimiento de Codex impulsada por eventos para mantener la documentación existente alineada con cambios aterrizados recientemente. No tiene una programación pura: una ejecución de CI exitosa en `main` desde un push no hecho por bot puede activarlo, y el dispatch manual puede ejecutarlo directamente. Las invocaciones desde workflow-run se omiten cuando `main` ya avanzó o cuando se creó otra ejecución no omitida de Docs Agent en la última hora. Cuando se ejecuta, revisa el rango de commits desde el SHA de origen del Docs Agent no omitido anterior hasta el `main` actual, de modo que una ejecución por hora puede cubrir todos los cambios acumulados en main desde la última pasada de documentación.

El flujo de trabajo `Test Performance Agent` es una lane de mantenimiento de Codex impulsada por eventos para pruebas lentas. No tiene una programación pura: una ejecución de CI exitosa en `main` desde un push no hecho por bot puede activarlo, pero se omite si otra invocación por workflow-run ya se ejecutó o se está ejecutando ese día UTC. El dispatch manual omite esa compuerta diaria de actividad. La lane compila un informe completo de rendimiento de Vitest agrupado para toda la suite, permite que Codex haga solo pequeñas mejoras de rendimiento de pruebas que preserven cobertura en lugar de refactorizaciones amplias, luego vuelve a ejecutar el informe completo y rechaza cambios que reduzcan la cantidad base de pruebas aprobadas. Si la base tiene pruebas fallidas, Codex puede corregir solo fallos evidentes y el informe completo posterior al agente debe aprobar antes de que se haga cualquier commit. Cuando `main` avanza antes de que llegue el push del bot, la lane hace rebase del parche validado, vuelve a ejecutar `pnpm check:changed` y reintenta el push; los parches obsoletos con conflictos se omiten. Usa Ubuntu alojado por GitHub para que la acción de Codex pueda mantener la misma postura de seguridad sin sudo que el agente de documentación.

```bash
gh workflow run duplicate-after-merge.yml \
  -f landed_pr=70532 \
  -f duplicate_prs='70530,70592' \
  -f apply=true
```

## Orden de fail-fast

Los trabajos están ordenados para que las comprobaciones baratas fallen antes de que se ejecuten las costosas:

1. `preflight` decide qué lanes existen en absoluto. La lógica `docs-scope` y `changed-scope` son pasos dentro de este trabajo, no trabajos independientes.
2. `security-scm-fast`, `security-dependency-audit`, `security-fast`, `check`, `check-additional`, `check-docs` y `skills-python` fallan rápido sin esperar los trabajos más pesados de artefactos y matrices de plataforma.
3. `build-artifacts` se superpone con las lanes rápidas de Linux para que los consumidores aguas abajo puedan comenzar en cuanto la compilación compartida esté lista.
4. Después se despliegan las lanes más pesadas de plataforma y runtime: `checks-fast-core`, `checks-fast-contracts-channels`, `checks-node-extensions`, `checks-node-core-test`, `extension-fast` solo para PR, `checks`, `checks-windows`, `macos-node`, `macos-swift` y `android`.

La lógica de alcance vive en `scripts/ci-changed-scope.mjs` y está cubierta por pruebas unitarias en `src/scripts/ci-changed-scope.test.ts`.
Las ediciones del flujo de trabajo de CI validan el grafo de CI de Node más el lint de workflows, pero por sí solas no fuerzan compilaciones nativas de Windows, Android o macOS; esas lanes de plataforma siguen limitadas a cambios en código fuente de la plataforma.
Las ediciones solo de enrutamiento de CI, ciertas ediciones baratas de fixtures de pruebas core y ediciones estrechas de helpers de contratos de plugins/enrutamiento de pruebas usan una ruta rápida de manifiesto solo de Node: preflight, seguridad y una sola tarea `checks-fast-core`. Esa ruta evita artefactos de compilación, compatibilidad con Node 22, contratos de canales, fragmentos core completos, fragmentos de plugins incluidos y matrices adicionales de guards cuando los archivos cambiados se limitan a las superficies de enrutamiento o helpers que la tarea rápida ejercita directamente.
Las comprobaciones de Node en Windows se limitan a wrappers de procesos/rutas específicos de Windows, helpers de ejecución npm/pnpm/UI, configuración del gestor de paquetes y superficies del flujo de trabajo de CI que ejecutan esa lane; los cambios no relacionados en código fuente, plugins, install-smoke y solo de pruebas permanecen en las lanes de Node para Linux para no reservar un worker de Windows de 16 vCPU para una cobertura que ya ejercitan los fragmentos normales de prueba.
El flujo de trabajo separado `install-smoke` reutiliza el mismo script de alcance mediante su propio trabajo `preflight`. Divide la cobertura smoke en `run_fast_install_smoke` y `run_full_install_smoke`. Los pull requests ejecutan la ruta rápida para superficies Docker/paquetes, cambios en paquetes/manifiestos de plugins incluidos y superficies core de plugins/canales/Gateway/SDK de Plugin que ejercitan los trabajos smoke de Docker. Los cambios solo de código fuente en plugins incluidos, las ediciones solo de pruebas y las ediciones solo de documentación no reservan workers de Docker. La ruta rápida compila una vez la imagen del Dockerfile raíz, comprueba la CLI, ejecuta el smoke de CLI de eliminación de agentes en espacio de trabajo compartido, ejecuta el e2e de gateway-network del contenedor, verifica un argumento de compilación de extensión incluida y ejecuta el perfil acotado de plugin incluido en Docker bajo un tiempo máximo agregado de 240 segundos, con el `docker run` de cada escenario limitado por separado. La ruta completa mantiene la instalación de paquetes por QR y la cobertura de Docker/actualización de instaladores para ejecuciones nocturnas programadas, dispatches manuales, comprobaciones de lanzamiento por workflow-call y pull requests que realmente tocan superficies de instalador/paquete/Docker. Los pushes a `main`, incluidos los merge commits, no fuerzan la ruta completa; cuando la lógica de changed-scope solicitaría cobertura completa en un push, el flujo mantiene el smoke rápido de Docker y deja el install smoke completo para la validación nocturna o de lanzamiento. El smoke lento del proveedor de imágenes con instalación global de Bun está controlado por separado mediante `run_bun_global_install_smoke`; se ejecuta en la programación nocturna y desde el flujo de comprobaciones de lanzamiento, y los dispatches manuales de `install-smoke` pueden incluirlo opcionalmente, pero los pull requests y pushes a `main` no lo ejecutan. Las pruebas Docker de QR e instalador conservan sus propios Dockerfiles centrados en instalación. El `test:docker:all` local precompila una imagen compartida de live-test y una imagen compartida de app compilada `scripts/e2e/Dockerfile`, luego ejecuta las lanes smoke live/E2E con un programador ponderado y `OPENCLAW_SKIP_DOCKER_BUILD=1`; ajusta el recuento predeterminado de 10 slots del pool principal con `OPENCLAW_DOCKER_ALL_PARALLELISM` y el recuento de 10 slots del pool final sensible al proveedor con `OPENCLAW_DOCKER_ALL_TAIL_PARALLELISM`. Los límites de lanes pesadas usan por defecto `OPENCLAW_DOCKER_ALL_LIVE_LIMIT=6`, `OPENCLAW_DOCKER_ALL_NPM_LIMIT=8` y `OPENCLAW_DOCKER_ALL_SERVICE_LIMIT=7` para que las lanes de instalación npm y múltiples servicios no sobrecomprometan Docker mientras las lanes ligeras siguen ocupando slots disponibles. Los inicios de lane se escalonan 2 segundos por defecto para evitar tormentas de creación en el daemon local de Docker; sobrescribe con `OPENCLAW_DOCKER_ALL_START_STAGGER_MS=0` u otro valor en milisegundos. El agregado local hace preflight de Docker, elimina contenedores E2E obsoletos de OpenClaw, emite el estado de las lanes activas, conserva los tiempos de las lanes para ordenarlas por mayor duración primero y admite `OPENCLAW_DOCKER_ALL_DRY_RUN=1` para inspeccionar el programador. Deja de programar nuevas lanes agrupadas tras el primer fallo por defecto, y cada lane tiene un tiempo máximo de respaldo de 120 minutos que puede sobrescribirse con `OPENCLAW_DOCKER_ALL_LANE_TIMEOUT_MS`; algunas lanes live/tail seleccionadas usan límites más ajustados por lane. El flujo reutilizable live/E2E refleja el patrón de imagen compartida compilando y publicando una imagen Docker E2E de GHCR etiquetada con SHA antes de la matriz Docker, y luego ejecuta la matriz con `OPENCLAW_SKIP_DOCKER_BUILD=1`. El flujo programado live/E2E ejecuta diariamente la suite Docker completa de la ruta de lanzamiento. La matriz de actualización incluida se divide por objetivo de actualización para que los pasos repetidos de npm update y doctor repair puedan fragmentarse junto con otras comprobaciones incluidas.

La lógica local de lanes cambiadas vive en `scripts/changed-lanes.mjs` y se ejecuta mediante `scripts/check-changed.mjs`. Esa compuerta local es más estricta con los límites de arquitectura que el alcance amplio de plataformas de CI: los cambios de producción core ejecutan typecheck de producción core más pruebas core, los cambios solo de pruebas core ejecutan solo typecheck/pruebas de pruebas core, los cambios de producción de extensiones ejecutan typecheck de producción de extensiones más pruebas de extensiones y los cambios solo de pruebas de extensiones ejecutan solo typecheck/pruebas de pruebas de extensiones. Los cambios públicos del SDK de Plugin o de contratos de plugins amplían la validación a extensiones porque las extensiones dependen de esos contratos core. Los incrementos de versión solo de metadatos de lanzamiento ejecutan comprobaciones dirigidas de versión/configuración/dependencias raíz. Los cambios desconocidos en raíz/configuración fallan de forma segura hacia todas las lanes.

En los pushes, la matriz `checks` agrega la lane solo para push `compat-node22`. En los pull requests, esa lane se omite y la matriz se mantiene enfocada en las lanes normales de pruebas/canales.

Las familias de pruebas Node más lentas están divididas o equilibradas para que cada trabajo siga siendo pequeño sin sobrerreservar runners: los contratos de canales se ejecutan como tres fragmentos ponderados, las pruebas de plugins incluidos se equilibran entre seis workers de extensiones, las lanes pequeñas de unidades core se emparejan, auto-reply se ejecuta como tres workers equilibrados en lugar de seis workers pequeños, y las configuraciones agentic de gateway/plugin se distribuyen entre los trabajos Node agentic existentes solo de código fuente en lugar de esperar artefactos compilados. Las pruebas amplias de navegador, QA, multimedia y plugins diversos usan sus configuraciones Vitest dedicadas en lugar del comodín compartido de plugins. Los trabajos de fragmentos de extensiones ejecutan hasta dos grupos de configuración de plugins a la vez con un worker de Vitest por grupo y un heap de Node más grande para que los lotes de plugins con muchas importaciones no creen trabajos extra de CI. La lane amplia de agentes usa el programador compartido de paralelismo por archivo de Vitest porque está dominada por importaciones/programación en vez de por un único archivo de prueba lento. `runtime-config` se ejecuta con el fragmento infra core-runtime para evitar que el fragmento compartido de runtime posea la cola final. `check-additional` mantiene juntos el trabajo de compile/canary de límites de paquetes y separa la arquitectura de topología de runtime de la cobertura de gateway watch; el fragmento de guards de límites ejecuta sus pequeños guards independientes de forma concurrente dentro de un trabajo. Gateway watch, pruebas de canales y el fragmento core de límites de soporte se ejecutan concurrentemente dentro de `build-artifacts` después de que `dist/` y `dist-runtime/` ya estén compilados, manteniendo sus nombres antiguos de comprobación como trabajos ligeros de verificación mientras evita dos workers extra de Blacksmith y una segunda cola de consumidores de artefactos.
La CI de Android ejecuta tanto `testPlayDebugUnitTest` como `testThirdPartyDebugUnitTest`, y luego compila el APK debug de Play. El sabor third-party no tiene un source set o manifiesto separado; su lane de pruebas unitarias aun así compila ese sabor con los flags BuildConfig de SMS/registro de llamadas, mientras evita un trabajo duplicado de empaquetado de APK debug en cada push relevante para Android.
`extension-fast` es solo para PR porque las ejecuciones en push ya ejecutan los fragmentos completos de plugins incluidos. Eso mantiene la retroalimentación sobre plugins modificados para revisiones sin reservar un worker extra de Blacksmith en `main` para una cobertura ya presente en `checks-node-extensions`.

GitHub puede marcar trabajos reemplazados como `cancelled` cuando llega un push más reciente en el mismo PR o ref `main`. Trátalo como ruido de CI salvo que la ejecución más reciente para la misma ref también esté fallando. Las comprobaciones agregadas de fragmentos usan `!cancelled() && always()` para que sigan informando fallos normales de fragmentos pero no se encolen después de que todo el flujo ya haya sido reemplazado.
La clave de concurrencia de CI está versionada (`CI-v7-*`) para que un zombie del lado de GitHub en un grupo de cola antiguo no pueda bloquear indefinidamente ejecuciones nuevas de main.

## Equivalentes locales

```bash
pnpm changed:lanes   # inspeccionar el clasificador local de lanes cambiadas para origin/main...HEAD
pnpm check:changed   # compuerta local inteligente: typecheck/lint/pruebas cambiados por lane de límites
pnpm check          # compuerta local rápida: producción tsgo + lint fragmentado + guards rápidos en paralelo
pnpm check:test-types
pnpm check:timed    # misma compuerta con tiempos por etapa
pnpm build:strict-smoke
pnpm check:architecture
pnpm test:gateway:watch-regression
pnpm test           # pruebas vitest
pnpm test:channels
pnpm test:contracts:channels
pnpm check:docs     # formato de docs + lint + enlaces rotos
pnpm build          # compilar dist cuando importan las lanes de artefactos/ build-smoke de CI
node scripts/ci-run-timings.mjs <run-id>      # resumir tiempo total, tiempo de cola y trabajos más lentos
node scripts/ci-run-timings.mjs --recent 10   # comparar ejecuciones recientes exitosas de CI en main
pnpm test:perf:groups --full-suite --allow-failures --output .artifacts/test-perf/baseline-before.json
pnpm test:perf:groups:compare .artifacts/test-perf/baseline-before.json .artifacts/test-perf/after-agent.json
```

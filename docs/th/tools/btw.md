`/btw` ช่วยให้คุณถามคำถามแทรกแบบรวดเร็วเกี่ยวกับ **เซสชันปัจจุบัน** ได้โดย
ไม่ทำให้คำถามนั้นกลายเป็นส่วนหนึ่งของประวัติการสนทนาปกติ

มันมีแนวคิดมาจากพฤติกรรม `/btw` ของ Claude Code แต่ปรับให้เข้ากับ
Gateway และสถาปัตยกรรมหลายช่องทางของ OpenClaw

## สิ่งที่มันทำ

เมื่อคุณส่ง:

```text
/btw what changed?
```

OpenClaw จะ:

1. สแนปช็อตบริบทของเซสชันปัจจุบัน
2. รันการเรียกโมเดลแยกต่างหากแบบ **ไม่มีเครื่องมือ**
3. ตอบเฉพาะคำถามแทรกนั้น
4. ปล่อยให้การรันหลักดำเนินต่อไปตามเดิม
5. **ไม่** เขียนคำถามหรือคำตอบของ BTW ลงในประวัติเซสชัน
6. ส่งคำตอบออกมาเป็น **ผลลัพธ์แทรกแบบสด** แทนที่จะเป็นข้อความผู้ช่วยปกติ

แบบจำลองทางความคิดที่สำคัญคือ:

- ใช้บริบทของเซสชันเดียวกัน
- เป็น side query แบบครั้งเดียวแยกต่างหาก
- ไม่มีการเรียกใช้เครื่องมือ
- ไม่ทำให้บริบทในอนาคตปนเปื้อน
- ไม่มีการเก็บ transcript แบบถาวร

## สิ่งที่มันไม่ทำ

`/btw` **ไม่**:

- สร้างเซสชันถาวรใหม่
- ดำเนินงานหลักที่ค้างอยู่ต่อ
- รันเครื่องมือหรือ agent tool loops
- เขียนข้อมูลคำถาม/คำตอบ BTW ลงในประวัติ transcript
- ปรากฏใน `chat.history`
- อยู่ต่อหลังการรีโหลด

มันถูกออกแบบให้เป็นสิ่งที่ **ชั่วคราว**

## การทำงานของบริบท

BTW ใช้เซสชันปัจจุบันเป็น **บริบทเบื้องหลังเท่านั้น**

หากการรันหลักกำลังทำงานอยู่ OpenClaw จะสแนปช็อตสถานะข้อความปัจจุบัน
และรวม prompt หลักที่กำลังอยู่ระหว่างรันเข้าไปเป็นบริบทเบื้องหลัง พร้อมทั้งระบุชัดเจนกับโมเดลว่า:

- ให้ตอบเฉพาะคำถามแทรก
- ห้ามกลับไปทำต่อหรือทำงานหลักที่ยังไม่เสร็จให้สมบูรณ์
- ห้ามส่งการเรียกใช้เครื่องมือหรือการเรียกใช้เครื่องมือจำลอง

สิ่งนี้ช่วยให้ BTW แยกออกจากการรันหลัก ขณะเดียวกันก็ยังรับรู้ว่าเซสชันนั้นเกี่ยวกับอะไร

## โมเดลการส่งมอบ

BTW **ไม่** ถูกส่งมอบเป็นข้อความ transcript ของผู้ช่วยแบบปกติ

ในระดับโปรโตคอลของ Gateway:

- แชตของผู้ช่วยแบบปกติใช้เหตุการณ์ `chat`
- BTW ใช้เหตุการณ์ `chat.side_result`

การแยกนี้เป็นสิ่งตั้งใจ หาก BTW ใช้เส้นทางเหตุการณ์ `chat` แบบปกติซ้ำ
ไคลเอนต์จะปฏิบัติกับมันเหมือนเป็นประวัติการสนทนาปกติ

เนื่องจาก BTW ใช้เหตุการณ์สดแยกต่างหาก และไม่ถูกเล่นซ้ำจาก
`chat.history` มันจึงหายไปหลังการรีโหลด

## พฤติกรรมของพื้นผิว

### TUI

ใน TUI, BTW จะถูกแสดงแบบอินไลน์ในมุมมองของเซสชันปัจจุบัน แต่ยังคงเป็นแบบชั่วคราว:

- มองเห็นได้ชัดว่าแตกต่างจากคำตอบผู้ช่วยแบบปกติ
- ปิดได้ด้วย `Enter` หรือ `Esc`
- ไม่ถูกเล่นซ้ำหลังรีโหลด

### ช่องทางภายนอก

ในช่องทางอย่าง Telegram, WhatsApp และ Discord, BTW จะถูกส่งเป็น
คำตอบแบบครั้งเดียวที่มีป้ายกำกับชัดเจน เพราะพื้นผิวเหล่านั้นไม่มีแนวคิดเรื่องโอเวอร์เลย์ชั่วคราวในเครื่อง

คำตอบยังคงถูกปฏิบัติเป็นผลลัพธ์แทรก ไม่ใช่ประวัติเซสชันแบบปกติ

### Control UI / เว็บ

Gateway ส่ง BTW ออกมาอย่างถูกต้องในรูปแบบ `chat.side_result` และ BTW ก็ไม่ถูกใส่รวมใน
`chat.history` ดังนั้นสัญญาด้าน persistence จึงถูกต้องแล้วสำหรับเว็บ

Control UI ปัจจุบันยังต้องมีตัวรับ `chat.side_result` โดยเฉพาะเพื่อ
แสดง BTW แบบสดในเบราว์เซอร์ จนกว่าการรองรับฝั่งไคลเอนต์นี้จะมาถึง BTW จะยังเป็นฟีเจอร์ระดับ
Gateway ที่มีพฤติกรรมสมบูรณ์ใน TUI และช่องทางภายนอก แต่ยังไม่ใช่ UX บนเบราว์เซอร์ที่สมบูรณ์

## เมื่อใดควรใช้ BTW

ใช้ `/btw` เมื่อคุณต้องการ:

- คำชี้แจงแบบรวดเร็วเกี่ยวกับงานปัจจุบัน
- คำตอบแทรกเชิงข้อเท็จจริงขณะที่การรันยาวกำลังดำเนินอยู่
- คำตอบชั่วคราวที่ไม่ควรกลายเป็นส่วนหนึ่งของบริบทเซสชันในอนาคต

ตัวอย่าง:

```text
/btw what file are we editing?
/btw what does this error mean?
/btw summarize the current task in one sentence
/btw what is 17 * 19?
```

## เมื่อใดไม่ควรใช้ BTW

อย่าใช้ `/btw` หากคุณต้องการให้คำตอบกลายเป็นส่วนหนึ่งของ
บริบทการทำงานในอนาคตของเซสชัน

ในกรณีนั้น ให้ถามตามปกติในเซสชันหลักแทนการใช้ BTW

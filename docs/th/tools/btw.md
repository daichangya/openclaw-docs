# คำถามแทรก BTW

`/btw` ช่วยให้คุณถามคำถามแทรกอย่างรวดเร็วเกี่ยวกับ **เซสชันปัจจุบัน** โดยไม่ทำให้คำถามนั้นกลายเป็นส่วนหนึ่งของประวัติการสนทนาปกติ

มันมีพื้นฐานมาจากพฤติกรรม `/btw` ของ Claude Code แต่ปรับให้เข้ากับสถาปัตยกรรม Gateway และหลายช่องทางของ OpenClaw

## สิ่งที่มันทำ

เมื่อคุณส่ง:

```text
/btw what changed?
```

OpenClaw จะ:

1. ทำ snapshot ของบริบทเซสชันปัจจุบัน
2. รันการเรียกโมเดลแบบแยกต่างหากที่ **ไม่มีเครื่องมือ**
3. ตอบเฉพาะคำถามแทรกเท่านั้น
4. ปล่อยให้การรันหลักไม่ถูกรบกวน
5. **ไม่** เขียนคำถามหรือคำตอบของ BTW ลงในประวัติเซสชัน
6. ส่งคำตอบออกมาเป็น **ผลลัพธ์แทรกแบบสด** แทนข้อความผู้ช่วยแบบปกติ

โมเดลความคิดที่สำคัญคือ:

- ใช้บริบทของเซสชันเดียวกัน
- เป็น side query แบบ one-shot แยกต่างหาก
- ไม่มีการเรียกใช้เครื่องมือ
- ไม่ทำให้บริบทในอนาคตปนเปื้อน
- ไม่มีการคงไว้ในทรานสคริปต์

## สิ่งที่มันไม่ทำ

`/btw` **ไม่** ทำสิ่งต่อไปนี้:

- สร้างเซสชันถาวรใหม่
- ดำเนินงานหลักที่ยังไม่เสร็จต่อ
- รันเครื่องมือหรือ agent tool loop
- เขียนข้อมูลคำถาม/คำตอบของ BTW ลงในประวัติทรานสคริปต์
- ปรากฏใน `chat.history`
- อยู่รอดหลังการรีโหลด

มันถูกออกแบบให้เป็นแบบ **ชั่วคราว** โดยตั้งใจ

## วิธีการทำงานของบริบท

BTW ใช้เซสชันปัจจุบันเป็น **บริบทพื้นหลังเท่านั้น**

หากการรันหลักกำลังทำงานอยู่ OpenClaw จะทำ snapshot ของสถานะข้อความปัจจุบัน
และรวมพรอมป์หลักที่กำลังรันอยู่เป็นบริบทพื้นหลัง พร้อมทั้งบอกโมเดลอย่างชัดเจนว่า:

- ให้ตอบเฉพาะคำถามแทรกเท่านั้น
- อย่ากลับไปทำงานหลักที่ยังไม่เสร็จต่อหรือทำให้เสร็จ
- อย่าส่งการเรียกใช้เครื่องมือหรือการเรียกใช้เครื่องมือเทียม

สิ่งนี้ทำให้ BTW แยกจากการรันหลัก แต่ยังคงรับรู้ได้ว่าเซสชันกำลังเกี่ยวกับอะไร

## โมเดลการส่งมอบ

BTW **ไม่ได้** ถูกส่งมอบเป็นข้อความทรานสคริปต์ของผู้ช่วยแบบปกติ

ในระดับโปรโตคอลของ Gateway:

- แชตผู้ช่วยปกติใช้เหตุการณ์ `chat`
- BTW ใช้เหตุการณ์ `chat.side_result`

การแยกนี้เกิดขึ้นโดยตั้งใจ หาก BTW ใช้พาธเหตุการณ์ `chat` แบบปกติซ้ำ
ไคลเอนต์จะปฏิบัติต่อมันเหมือนเป็นประวัติการสนทนาปกติ

เนื่องจาก BTW ใช้เหตุการณ์สดแยกต่างหาก และไม่ถูก replay จาก
`chat.history`, มันจึงหายไปหลังการรีโหลด

## พฤติกรรมของพื้นผิว

### TUI

ใน TUI, BTW จะถูกเรนเดอร์แบบอินไลน์ในมุมมองเซสชันปัจจุบัน แต่ยังคง
เป็นแบบชั่วคราว:

- มองเห็นความแตกต่างจากคำตอบปกติของผู้ช่วยอย่างชัดเจน
- ปิดได้ด้วย `Enter` หรือ `Esc`
- ไม่ถูก replay หลังการรีโหลด

### ช่องทางภายนอก

บนช่องทางอย่าง Telegram, WhatsApp และ Discord, BTW จะถูกส่งเป็นคำตอบแบบ one-off
ที่ติดป้ายชัดเจน เพราะพื้นผิวเหล่านั้นไม่มีแนวคิด overlay แบบชั่วคราวในเครื่อง

คำตอบยังคงถูกปฏิบัติเป็น side result ไม่ใช่ประวัติเซสชันปกติ

### Control UI / web

Gateway ส่ง BTW ได้ถูกต้องเป็น `chat.side_result` และ BTW ไม่ถูกรวมอยู่ใน
`chat.history` ดังนั้นสัญญาเรื่องการคงอยู่จึงถูกต้องสำหรับเว็บอยู่แล้ว

Control UI ปัจจุบันยังต้องมี consumer สำหรับ `chat.side_result` โดยเฉพาะเพื่อเรนเดอร์ BTW แบบสดในเบราว์เซอร์ จนกว่าการรองรับฝั่งไคลเอนต์นี้จะพร้อม BTW จึงยังเป็นฟีเจอร์ระดับ Gateway ที่มีพฤติกรรมครบถ้วนใน TUI และช่องทางภายนอก แต่ยังไม่ใช่ UX บนเบราว์เซอร์ที่สมบูรณ์

## เมื่อใดควรใช้ BTW

ใช้ `/btw` เมื่อคุณต้องการ:

- คำชี้แจงแบบรวดเร็วเกี่ยวกับงานปัจจุบัน
- คำตอบเชิงข้อเท็จจริงแทรกระหว่างที่การรันยาวกำลังดำเนินอยู่
- คำตอบชั่วคราวที่ไม่ควรกลายเป็นส่วนหนึ่งของบริบทเซสชันในอนาคต

ตัวอย่าง:

```text
/btw what file are we editing?
/btw what does this error mean?
/btw summarize the current task in one sentence
/btw what is 17 * 19?
```

## เมื่อใดไม่ควรใช้ BTW

อย่าใช้ `/btw` เมื่อคุณต้องการให้คำตอบนั้นกลายเป็นส่วนหนึ่งของ
บริบทการทำงานของเซสชันในอนาคต

ในกรณีนั้น ให้ถามตามปกติในเซสชันหลักแทนการใช้ BTW

# การล็อกอินเบราว์เซอร์ + การโพสต์บน X/Twitter

## การล็อกอินด้วยตนเอง (แนะนำ)

เมื่อเว็บไซต์ต้องการการล็อกอิน ให้ **ลงชื่อเข้าใช้ด้วยตนเอง** ในโปรไฟล์เบราว์เซอร์บน **โฮสต์** (เบราว์เซอร์ openclaw)

อย่าให้ข้อมูลยืนยันตัวตนของคุณกับโมเดล การล็อกอินแบบอัตโนมัติมักกระตุ้นระบบป้องกันบอต และอาจทำให้บัญชีถูกล็อกได้

กลับไปยังเอกสารหลักของเบราว์เซอร์: [Browser](/th/tools/browser)

## ใช้โปรไฟล์ Chrome ไหน?

OpenClaw ควบคุม **โปรไฟล์ Chrome เฉพาะ** (ชื่อ `openclaw`, UI สีส้ม) ซึ่งแยกจากโปรไฟล์เบราว์เซอร์ที่คุณใช้ประจำ

สำหรับการเรียก browser tool ของเอเจนต์:

- ตัวเลือกเริ่มต้น: เอเจนต์ควรใช้เบราว์เซอร์ `openclaw` แบบแยกของมัน
- ใช้ `profile="user"` เฉพาะเมื่อเซสชันที่ล็อกอินอยู่แล้วมีความสำคัญ และผู้ใช้อยู่ที่เครื่องเพื่อคลิก/อนุมัติ prompt การ attach ที่อาจเกิดขึ้น
- หากคุณมีหลายโปรไฟล์เบราว์เซอร์ของผู้ใช้ ให้ระบุโปรไฟล์อย่างชัดเจนแทนการเดา

มีสองวิธีง่าย ๆ ในการเข้าถึง:

1. **บอกให้เอเจนต์เปิดเบราว์เซอร์** แล้วคุณค่อยล็อกอินเอง
2. **เปิดผ่าน CLI**:

```bash
openclaw browser start
openclaw browser open https://x.com
```

หากคุณมีหลายโปรไฟล์ ให้ส่ง `--browser-profile <name>` (ค่าเริ่มต้นคือ `openclaw`)

## X/Twitter: โฟลว์ที่แนะนำ

- **อ่าน/ค้นหา/เธรด:** ใช้เบราว์เซอร์บน **โฮสต์** (ล็อกอินด้วยตนเอง)
- **โพสต์อัปเดต:** ใช้เบราว์เซอร์บน **โฮสต์** (ล็อกอินด้วยตนเอง)

## Sandboxing + การเข้าถึงเบราว์เซอร์บนโฮสต์

เซสชันเบราว์เซอร์ใน sandbox มีแนวโน้มที่จะกระตุ้นการตรวจจับบอตได้มากกว่า สำหรับ X/Twitter (และเว็บไซต์อื่นที่เข้มงวด) ให้เลือกใช้เบราว์เซอร์บน **โฮสต์**

หากเอเจนต์ถูก sandboxed browser tool จะใช้ sandbox เป็นค่าเริ่มต้น หากต้องการอนุญาตให้ควบคุมเบราว์เซอร์บนโฮสต์ได้:

```json5
{
  agents: {
    defaults: {
      sandbox: {
        mode: "non-main",
        browser: {
          allowHostControl: true,
        },
      },
    },
  },
}
```

จากนั้นระบุให้ใช้เบราว์เซอร์บนโฮสต์:

```bash
openclaw browser open https://x.com --browser-profile openclaw --target host
```

หรือปิด sandboxing สำหรับเอเจนต์ที่ใช้โพสต์อัปเดต

# TaskFlow

TaskFlow คือชั้นโครงสร้างพื้นฐานสำหรับการจัดประสานโฟลว์ที่อยู่เหนือ [แบ็กกราวด์ทาสก์](/th/automation/tasks) โดยจัดการโฟลว์หลายขั้นตอนแบบคงทนซึ่งมีสถานะของตนเอง การติดตาม revision และความหมายของการซิงก์ ในขณะที่แต่ละทาสก์ยังคงเป็นหน่วยของงานแบบแยกตัว

## เมื่อใดควรใช้ TaskFlow

ใช้ TaskFlow เมื่องานครอบคลุมหลายขั้นตอนแบบลำดับต่อเนื่องหรือแตกแขนง และคุณต้องการการติดตามความคืบหน้าแบบคงทนข้ามการรีสตาร์ต Gateway สำหรับการดำเนินการเบื้องหลังเพียงรายการเดียว [ทาสก์](/th/automation/tasks) ปกติก็เพียงพอ

| สถานการณ์                              | ใช้                  |
| -------------------------------------- | -------------------- |
| งานเบื้องหลังรายการเดียว               | ทาสก์ปกติ            |
| ไปป์ไลน์หลายขั้นตอน (A แล้ว B แล้ว C) | TaskFlow (จัดการเอง) |
| สังเกตทาสก์ที่ถูกสร้างจากภายนอก       | TaskFlow (มิเรอร์)   |
| การเตือนครั้งเดียว                     | งาน Cron             |

## โหมดการซิงก์

### โหมดที่จัดการเอง

TaskFlow เป็นเจ้าของวงจรชีวิตทั้งหมดแบบต้นทางถึงปลายทาง โดยจะสร้างทาสก์เป็นขั้นตอนของโฟลว์ ขับเคลื่อนจนเสร็จสมบูรณ์ และเลื่อนสถานะโฟลว์โดยอัตโนมัติ

ตัวอย่าง: โฟลว์รายงานประจำสัปดาห์ที่ (1) รวบรวมข้อมูล (2) สร้างรายงาน และ (3) ส่งมอบรายงาน TaskFlow จะสร้างแต่ละขั้นตอนเป็นแบ็กกราวด์ทาสก์ รอให้เสร็จสิ้น แล้วจึงไปยังขั้นตอนถัดไป

```bash
Flow: weekly-report
  Step 1: gather-data     → task created → succeeded
  Step 2: generate-report → task created → succeeded
  Step 3: deliver         → task created → running
```

### โหมดมิเรอร์

TaskFlow สังเกตทาสก์ที่ถูกสร้างจากภายนอกและคงสถานะโฟลว์ให้ซิงก์กัน โดยไม่รับสิทธิ์เป็นเจ้าของการสร้างทาสก์ ซึ่งมีประโยชน์เมื่อทาสก์มีต้นทางมาจากงาน Cron, คำสั่ง CLI หรือแหล่งอื่น และคุณต้องการมุมมองรวมของความคืบหน้าในรูปแบบโฟลว์

ตัวอย่าง: งาน Cron อิสระสามงานที่รวมกันเป็นรูทีน "ปฏิบัติการตอนเช้า" โฟลว์แบบมิเรอร์จะติดตามความคืบหน้าโดยรวมของงานเหล่านั้นโดยไม่ควบคุมว่าจะรันเมื่อใดหรืออย่างไร

## สถานะแบบคงทนและการติดตาม revision

แต่ละโฟลว์จะเก็บสถานะของตนเองแบบถาวรและติดตาม revision เพื่อให้ความคืบหน้ายังคงอยู่แม้ Gateway จะรีสตาร์ต การติดตาม revision ช่วยให้ตรวจพบความขัดแย้งเมื่อมีหลายแหล่งพยายามเลื่อนโฟลว์เดียวกันพร้อมกัน

## พฤติกรรมการยกเลิก

`openclaw tasks flow cancel` จะตั้งเจตนาการยกเลิกแบบคงค้างบนโฟลว์ ทาสก์ที่กำลังทำงานอยู่ภายในโฟลว์จะถูกยกเลิก และจะไม่มีการเริ่มขั้นตอนใหม่ใด ๆ เจตนาการยกเลิกนี้คงอยู่ข้ามการรีสตาร์ต ดังนั้นโฟลว์ที่ถูกยกเลิกจะยังคงถูกยกเลิก แม้ Gateway จะรีสตาร์ตก่อนที่ทาสก์ลูกทั้งหมดจะสิ้นสุดลงก็ตาม

## คำสั่ง CLI

```bash
# แสดงรายการโฟลว์ที่กำลังทำงานและล่าสุด
openclaw tasks flow list

# แสดงรายละเอียดของโฟลว์ที่ระบุ
openclaw tasks flow show <lookup>

# ยกเลิกโฟลว์ที่กำลังทำงานและทาสก์ที่ยังทำงานอยู่ของมัน
openclaw tasks flow cancel <lookup>
```

| คำสั่ง                            | คำอธิบาย                                        |
| --------------------------------- | ----------------------------------------------- |
| `openclaw tasks flow list`        | แสดงโฟลว์ที่ติดตามอยู่พร้อมสถานะและโหมดการซิงก์ |
| `openclaw tasks flow show <id>`   | ตรวจสอบโฟลว์หนึ่งรายการด้วย flow id หรือ lookup key |
| `openclaw tasks flow cancel <id>` | ยกเลิกโฟลว์ที่กำลังทำงานและทาสก์ที่ยังทำงานอยู่ของมัน |

## โฟลว์เกี่ยวข้องกับทาสก์อย่างไร

โฟลว์ทำหน้าที่ประสานทาสก์ ไม่ได้มาแทนที่ทาสก์ โฟลว์หนึ่งรายการอาจขับเคลื่อนแบ็กกราวด์ทาสก์หลายรายการตลอดอายุของมัน ใช้ `openclaw tasks` เพื่อตรวจสอบบันทึกของแต่ละทาสก์ และใช้ `openclaw tasks flow` เพื่อตรวจสอบโฟลว์ที่ทำหน้าที่จัดประสาน

# โฟลว์งาน

TaskFlow คือชั้นรองรับการประสานโฟลว์งานที่อยู่เหนือ [งานเบื้องหลัง](/th/automation/tasks) โดยจะจัดการโฟลว์หลายขั้นตอนแบบคงทนซึ่งมีสถานะของตัวเอง การติดตามรีวิชัน และกลไกการซิงก์ ขณะที่แต่ละ task ยังคงเป็นหน่วยของงานที่ทำงานแยกออกไป

## ควรใช้ TaskFlow เมื่อใด

ใช้ TaskFlow เมื่องานครอบคลุมหลายขั้นตอนแบบลำดับหรือแบบแตกแขนง และคุณต้องการการติดตามความคืบหน้าแบบคงทนข้ามการรีสตาร์ตของ Gateway สำหรับการดำเนินการเบื้องหลังเพียงรายการเดียว [task](/th/automation/tasks) แบบปกติก็เพียงพอแล้ว

| สถานการณ์                            | การใช้งาน             |
| ------------------------------------ | --------------------- |
| งานเบื้องหลังเดี่ยว                  | task แบบปกติ         |
| ไปป์ไลน์หลายขั้นตอน (A แล้ว B แล้ว C) | TaskFlow (แบบ managed) |
| เฝ้าสังเกต task ที่สร้างจากภายนอก    | TaskFlow (แบบ mirrored) |
| การเตือนแบบครั้งเดียว                | งาน Cron              |

## โหมดการซิงก์

### โหมด managed

TaskFlow เป็นเจ้าของวงจรชีวิตทั้งหมดแบบครบวงจร โดยจะสร้าง task เป็นขั้นตอนของโฟลว์ ขับเคลื่อนให้เสร็จสิ้น และเลื่อนสถานะโฟลว์โดยอัตโนมัติ

ตัวอย่าง: โฟลว์รายงานประจำสัปดาห์ที่ (1) รวบรวมข้อมูล (2) สร้างรายงาน และ (3) ส่งรายงาน TaskFlow จะสร้างแต่ละขั้นตอนเป็นงานเบื้องหลัง รอให้เสร็จสิ้น แล้วจึงไปยังขั้นตอนถัดไป

```text
Flow: weekly-report
  Step 1: gather-data     → task created → succeeded
  Step 2: generate-report → task created → succeeded
  Step 3: deliver         → task created → running
```

### โหมด mirrored

TaskFlow เฝ้าสังเกต task ที่สร้างจากภายนอกและคงสถานะโฟลว์ให้ซิงก์กันโดยไม่รับหน้าที่สร้าง task เอง ซึ่งมีประโยชน์เมื่อ task มีต้นทางจากงาน Cron, คำสั่ง CLI หรือแหล่งอื่น และคุณต้องการมุมมองรวมของความคืบหน้าในรูปแบบโฟลว์

ตัวอย่าง: งาน Cron อิสระ 3 งานที่รวมกันเป็นกระบวนการ "งานปฏิบัติการตอนเช้า" โฟลว์แบบ mirrored จะติดตามความคืบหน้ารวมของงานเหล่านั้นโดยไม่ควบคุมว่าเริ่มทำงานเมื่อใดหรืออย่างไร

## สถานะแบบคงทนและการติดตามรีวิชัน

แต่ละโฟลว์จะบันทึกสถานะของตัวเองแบบคงทนและติดตามรีวิชัน เพื่อให้ความคืบหน้ายังคงอยู่แม้ Gateway จะรีสตาร์ต การติดตามรีวิชันช่วยให้ตรวจจับความขัดแย้งได้เมื่อมีหลายแหล่งพยายามเลื่อนโฟลว์เดียวกันพร้อมกัน

## พฤติกรรมการยกเลิก

`openclaw tasks flow cancel` จะตั้งเจตนาการยกเลิกแบบคงค้างไว้บนโฟลว์ task ที่กำลังทำงานอยู่ภายในโฟลว์จะถูกยกเลิก และจะไม่มีการเริ่มขั้นตอนใหม่ เจตนาการยกเลิกนี้จะคงอยู่ข้ามการรีสตาร์ต ดังนั้นโฟลว์ที่ถูกยกเลิกจะยังคงถูกยกเลิก แม้ Gateway จะรีสตาร์ตก่อนที่ task ลูกทั้งหมดจะสิ้นสุดลง

## คำสั่ง CLI

```bash
# List active and recent flows
openclaw tasks flow list

# Show details for a specific flow
openclaw tasks flow show <lookup>

# Cancel a running flow and its active tasks
openclaw tasks flow cancel <lookup>
```

| คำสั่ง                            | คำอธิบาย                                      |
| --------------------------------- | --------------------------------------------- |
| `openclaw tasks flow list`        | แสดงโฟลว์ที่ติดตามอยู่พร้อมสถานะและโหมดการซิงก์ |
| `openclaw tasks flow show <id>`   | ตรวจสอบโฟลว์หนึ่งรายการด้วย flow id หรือ lookup key |
| `openclaw tasks flow cancel <id>` | ยกเลิกโฟลว์ที่กำลังทำงานและ task ที่ยังทำงานอยู่ |

## โฟลว์เกี่ยวข้องกับ task อย่างไร

โฟลว์ทำหน้าที่ประสานงาน task ไม่ได้มาแทนที่ task โฟลว์หนึ่งรายการอาจขับเคลื่อนงานเบื้องหลังหลาย task ตลอดอายุการทำงาน ใช้ `openclaw tasks` เพื่อตรวจสอบระเบียน task แต่ละรายการ และใช้ `openclaw tasks flow` เพื่อตรวจสอบโฟลว์ผู้ประสานงาน

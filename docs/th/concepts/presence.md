  # Presence

  “Presence” ใน OpenClaw คือมุมมองแบบเบาและ best-effort ของ:

  - **Gateway** เอง และ
  - **ไคลเอนต์ที่เชื่อมต่อกับ Gateway** (แอป Mac, WebChat, CLI ฯลฯ)

  Presence ใช้เป็นหลักเพื่อเรนเดอร์แท็บ **Instances** ในแอป macOS และเพื่อ
  ให้ผู้ปฏิบัติงานมองเห็นสถานะได้อย่างรวดเร็ว

  ## ฟิลด์ของ Presence (สิ่งที่แสดงขึ้นมา)

  รายการ Presence เป็นอ็อบเจ็กต์แบบมีโครงสร้าง โดยมีฟิลด์เช่น:

  - `instanceId` (ไม่บังคับแต่แนะนำอย่างยิ่ง): ตัวตนไคลเอนต์ที่เสถียร (โดยทั่วไปคือ `connect.client.instanceId`)
  - `host`: ชื่อโฮสต์แบบอ่านเข้าใจง่าย
  - `ip`: ที่อยู่ IP แบบ best-effort
  - `version`: สตริงเวอร์ชันของไคลเอนต์
  - `deviceFamily` / `modelIdentifier`: คำใบ้เกี่ยวกับฮาร์ดแวร์
  - `mode`: `ui`, `webchat`, `cli`, `backend`, `probe`, `test`, `node`, ...
  - `lastInputSeconds`: “จำนวนวินาทีนับจากอินพุตผู้ใช้ล่าสุด” (หากทราบ)
  - `reason`: `self`, `connect`, `node-connected`, `periodic`, ...
  - `ts`: timestamp ของการอัปเดตล่าสุด (มิลลิวินาทีนับจาก epoch)

  ## ผู้สร้าง (Presence มาจากที่ใด)

  รายการ Presence ถูกสร้างจากหลายแหล่งและ **ถูกรวมเข้าด้วยกัน**

  ### 1) รายการ self ของ Gateway

  Gateway จะ seed รายการ “self” เสมอเมื่อเริ่มต้น เพื่อให้ UI แสดงโฮสต์ของ gateway
  ได้แม้ก่อนที่จะมีไคลเอนต์เชื่อมต่อ

  ### 2) การเชื่อมต่อ WebSocket

  ไคลเอนต์ WS ทุกตัวเริ่มต้นด้วยคำขอ `connect` เมื่อ handshake สำเร็จ
  Gateway จะ upsert รายการ Presence สำหรับการเชื่อมต่อนั้น

  #### ทำไมคำสั่ง CLI แบบครั้งเดียวจึงไม่แสดงขึ้นมา

  CLI มักเชื่อมต่อเพื่อรันคำสั่งสั้น ๆ แบบครั้งเดียว เพื่อหลีกเลี่ยงการสแปม
  รายการ Instances, `client.mode === "cli"` จะ **ไม่** ถูกเปลี่ยนเป็นรายการ Presence

  ### 3) บีคอน `system-event`

  ไคลเอนต์สามารถส่งบีคอนเป็นระยะที่มีรายละเอียดมากขึ้นผ่านเมธอด `system-event` แอป Mac
  ใช้วิธีนี้ในการรายงานชื่อโฮสต์ IP และ `lastInputSeconds`

  ### 4) การเชื่อมต่อของ Node (role: node)

  เมื่อ Node เชื่อมต่อผ่าน Gateway WebSocket ด้วย `role: node`, Gateway
  จะ upsert รายการ Presence สำหรับ Node นั้น (โฟลว์เดียวกับไคลเอนต์ WS อื่น)

  ## กฎการรวม + การลบข้อมูลซ้ำ (ทำไม `instanceId` จึงสำคัญ)

  รายการ Presence ถูกเก็บไว้ใน map ในหน่วยความจำเพียงตัวเดียว:

  - รายการต่าง ๆ ถูกคีย์ด้วย **presence key**
  - คีย์ที่ดีที่สุดคือ `instanceId` ที่เสถียร (จาก `connect.client.instanceId`) ซึ่งอยู่รอดข้ามการรีสตาร์ตได้
  - คีย์ไม่สนตัวพิมพ์เล็ก/ใหญ่

  หากไคลเอนต์เชื่อมต่อใหม่โดยไม่มี `instanceId` ที่เสถียร มันอาจแสดงเป็นแถว
  **ซ้ำกัน**

  ## TTL และขนาดที่มีขอบเขต

  Presence ถูกออกแบบให้เป็นข้อมูลชั่วคราวโดยตั้งใจ:

  - **TTL:** รายการที่เก่ากว่า 5 นาทีจะถูกตัดออก
  - **จำนวนรายการสูงสุด:** 200 (ลบรายการเก่าสุดก่อน)

  สิ่งนี้ทำให้รายการสดใหม่และหลีกเลี่ยงการเติบโตของหน่วยความจำแบบไร้ขอบเขต

  ## ข้อควรระวังสำหรับรีโมต/ทันเนล (IP แบบ loopback)

  เมื่อไคลเอนต์เชื่อมต่อผ่าน SSH tunnel / local port forward, Gateway อาจ
  มองเห็น remote address เป็น `127.0.0.1` เพื่อหลีกเลี่ยงการเขียนทับ IP ที่ไคลเอนต์รายงานมาอย่างถูกต้อง
  ระบบจะละเลย remote address แบบ loopback

  ## ผู้ใช้ข้อมูล

  ### แท็บ Instances บน macOS

  แอป macOS จะเรนเดอร์เอาต์พุตของ `system-presence` และใช้ตัวบ่งชี้สถานะขนาดเล็ก
  (Active/Idle/Stale) โดยอิงตามอายุของการอัปเดตล่าสุด

  ## เคล็ดลับการแก้ไขปัญหา

  - หากต้องการดูรายการดิบ ให้เรียก `system-presence` กับ Gateway
  - หากคุณเห็นข้อมูลซ้ำ:
  - ยืนยันว่าไคลเอนต์ส่ง `client.instanceId` ที่เสถียรใน handshake
  - ยืนยันว่าบีคอนแบบเป็นระยะใช้ `instanceId` เดียวกัน
  - ตรวจสอบว่ารายการที่ได้มาจากการเชื่อมต่อไม่มี `instanceId` หรือไม่ (หากซ้ำกันถือว่าเป็นสิ่งที่คาดไว้)

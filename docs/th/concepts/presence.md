สถานะการแสดงตัวของ OpenClaw คือมุมมองแบบเบาและแบบ best-effort ของ:

- **Gateway** เอง และ
- **ไคลเอนต์ที่เชื่อมต่อกับ Gateway** (แอป mac, WebChat, CLI เป็นต้น)

Presence ใช้เป็นหลักเพื่อเรนเดอร์แท็บ **Instances** ของแอป macOS และเพื่อให้ผู้ปฏิบัติการมองเห็นสถานะได้อย่างรวดเร็ว

## ฟิลด์ของ Presence (สิ่งที่แสดงขึ้นมา)

รายการ Presence เป็นออบเจ็กต์แบบมีโครงสร้าง โดยมีฟิลด์เช่น:

- `instanceId` (เป็นตัวเลือกแต่แนะนำอย่างยิ่ง): ตัวตนไคลเอนต์ที่เสถียร (โดยทั่วไปคือ `connect.client.instanceId`)
- `host`: ชื่อโฮสต์ที่เป็นมิตรต่อมนุษย์
- `ip`: ที่อยู่ IP แบบ best-effort
- `version`: สตริงเวอร์ชันของไคลเอนต์
- `deviceFamily` / `modelIdentifier`: ข้อมูลบอกใบ้เกี่ยวกับฮาร์ดแวร์
- `mode`: `ui`, `webchat`, `cli`, `backend`, `probe`, `test`, `node`, ...
- `lastInputSeconds`: “จำนวนวินาทีนับจากอินพุตล่าสุดของผู้ใช้” (หากทราบ)
- `reason`: `self`, `connect`, `node-connected`, `periodic`, ...
- `ts`: การประทับเวลาการอัปเดตล่าสุด (มิลลิวินาทีตั้งแต่ epoch)

## ผู้สร้าง (Presence มาจากไหน)

รายการ Presence ถูกสร้างจากหลายแหล่งและ **รวมเข้าด้วยกัน**

### 1) รายการ self ของ Gateway

Gateway จะวางรายการ “self” ไว้เสมอเมื่อเริ่มต้น เพื่อให้ UI แสดงโฮสต์ของ Gateway ได้แม้ก่อนที่ไคลเอนต์ใด ๆ จะเชื่อมต่อ

### 2) การเชื่อมต่อ WebSocket

ไคลเอนต์ WS ทุกตัวจะเริ่มด้วยคำขอ `connect` เมื่อ handshake สำเร็จ
Gateway จะ upsert รายการ presence สำหรับการเชื่อมต่อนั้น

#### ทำไมคำสั่ง CLI แบบครั้งเดียวจึงไม่แสดงขึ้นมา

CLI มักเชื่อมต่อเพื่อรันคำสั่งสั้น ๆ แบบครั้งเดียว เพื่อหลีกเลี่ยงการรบกวนรายการ Instances
`client.mode === "cli"` จึง **ไม่** ถูกเปลี่ยนเป็นรายการ presence

### 3) beacon ของ `system-event`

ไคลเอนต์สามารถส่ง beacon แบบเป็นระยะที่มีข้อมูลสมบูรณ์กว่า ผ่านเมธอด `system-event` แอป mac ใช้สิ่งนี้เพื่อรายงานชื่อโฮสต์, IP และ `lastInputSeconds`

### 4) การเชื่อมต่อของ Node (`role: node`)

เมื่อ Node เชื่อมต่อผ่าน Gateway WebSocket ด้วย `role: node`, Gateway จะ upsert รายการ presence สำหรับ Node นั้น (เป็นโฟลว์เดียวกับไคลเอนต์ WS ประเภทอื่น)

## กฎการรวม + การลบรายการซ้ำ (ทำไม `instanceId` จึงสำคัญ)

รายการ Presence ถูกเก็บไว้ในแผนที่ในหน่วยความจำเพียงชุดเดียว:

- รายการถูกคีย์ด้วย **presence key**
- คีย์ที่ดีที่สุดคือ `instanceId` ที่เสถียร (จาก `connect.client.instanceId`) ซึ่งคงอยู่ข้ามการรีสตาร์ต
- คีย์ไม่แยกตัวพิมพ์เล็ก-ใหญ่

หากไคลเอนต์เชื่อมต่อใหม่โดยไม่มี `instanceId` ที่เสถียร มันอาจแสดงเป็นแถว **ซ้ำกัน**

## TTL และขนาดที่มีขอบเขต

Presence เป็นข้อมูลชั่วคราวโดยตั้งใจ:

- **TTL:** รายการที่เก่ากว่า 5 นาทีจะถูกล้าง
- **จำนวนรายการสูงสุด:** 200 (ลบรายการที่เก่าที่สุดก่อน)

สิ่งนี้ช่วยให้รายการสดใหม่และหลีกเลี่ยงการเติบโตของหน่วยความจำแบบไร้ขีดจำกัด

## ข้อควรระวังสำหรับ remote/tunnel (IP แบบ loopback)

เมื่อไคลเอนต์เชื่อมต่อผ่าน SSH tunnel / local port forward, Gateway อาจเห็นที่อยู่ remote เป็น `127.0.0.1` เพื่อหลีกเลี่ยงการเขียนทับ IP ที่ไคลเอนต์รายงานไว้ซึ่งดีกว่า ระบบจะเพิกเฉยต่อที่อยู่ remote แบบ loopback

## ผู้ใช้ข้อมูล

### แท็บ Instances ของ macOS

แอป macOS จะเรนเดอร์เอาต์พุตของ `system-presence` และใช้ตัวบ่งชี้สถานะขนาดเล็ก (Active/Idle/Stale) ตามอายุของการอัปเดตล่าสุด

## เคล็ดลับการแก้ไขปัญหา

- หากต้องการดูรายการดิบ ให้เรียก `system-presence` กับ Gateway
- หากคุณเห็นรายการซ้ำ:
  - ยืนยันว่าไคลเอนต์ส่ง `client.instanceId` ที่เสถียรใน handshake
  - ยืนยันว่า beacon แบบเป็นระยะใช้ `instanceId` เดียวกัน
  - ตรวจสอบว่ารายการที่ได้มาจากการเชื่อมต่อขาด `instanceId` หรือไม่ (หากใช่ การซ้ำกันถือว่าเป็นเรื่องคาดหมาย)

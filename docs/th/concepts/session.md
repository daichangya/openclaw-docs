OpenClaw จัดระเบียบการสนทนาเป็น **เซสชัน** แต่ละข้อความจะถูกกำหนดเส้นทางไปยัง
เซสชันตามแหล่งที่มา -- DM, แชตกลุ่ม, งาน Cron เป็นต้น

## ข้อความถูกกำหนดเส้นทางอย่างไร

| แหล่งที่มา      | พฤติกรรม                |
| --------------- | ----------------------- |
| ข้อความส่วนตัว  | ใช้เซสชันร่วมกันโดยค่าเริ่มต้น |
| แชตกลุ่ม        | แยกต่อกลุ่ม             |
| ห้อง/ช่อง       | แยกต่อห้อง              |
| งาน Cron        | เซสชันใหม่ทุกรอบการรัน  |
| Webhooks        | แยกต่อ hook             |

## การแยกส่วน DM

โดยค่าเริ่มต้น DM ทั้งหมดจะใช้เซสชันเดียวกันเพื่อความต่อเนื่อง วิธีนี้เหมาะสำหรับ
การตั้งค่าแบบผู้ใช้คนเดียว

<Warning>
หากมีหลายคนที่สามารถส่งข้อความหาเอเจนต์ของคุณได้ ให้เปิดใช้การแยกส่วน DM หากไม่ทำ
ผู้ใช้ทุกคนจะใช้บริบทการสนทนาเดียวกัน -- ข้อความส่วนตัวของ Alice จะ
มองเห็นได้โดย Bob
</Warning>

**วิธีแก้:**

```json5
{
  session: {
    dmScope: "per-channel-peer", // แยกตามช่องทาง + ผู้ส่ง
  },
}
```

ตัวเลือกอื่น:

- `main` (ค่าเริ่มต้น) -- DM ทั้งหมดใช้เซสชันเดียวกัน
- `per-peer` -- แยกตามผู้ส่ง (ข้ามช่องทาง)
- `per-channel-peer` -- แยกตามช่องทาง + ผู้ส่ง (แนะนำ)
- `per-account-channel-peer` -- แยกตามบัญชี + ช่องทาง + ผู้ส่ง

<Tip>
หากบุคคลคนเดียวกันติดต่อคุณจากหลายช่องทาง ให้ใช้
`session.identityLinks` เพื่อเชื่อมตัวตนของพวกเขาเข้าด้วยกัน เพื่อให้ใช้เซสชันเดียวกัน
</Tip>

ตรวจสอบการตั้งค่าของคุณด้วย `openclaw security audit`

## วงจรชีวิตของเซสชัน

เซสชันจะถูกนำกลับมาใช้ซ้ำจนกว่าจะหมดอายุ:

- **รีเซ็ตรายวัน** (ค่าเริ่มต้น) -- เซสชันใหม่เวลา 4:00 น. ตามเวลาท้องถิ่นบนโฮสต์
  ของ gateway
- **รีเซ็ตเมื่อว่าง** (ไม่บังคับ) -- เซสชันใหม่หลังไม่มีการใช้งานช่วงหนึ่ง ตั้งค่า
  `session.reset.idleMinutes`
- **รีเซ็ตด้วยตนเอง** -- พิมพ์ `/new` หรือ `/reset` ในแชต `/new <model>` ยัง
  ใช้สลับโมเดลได้ด้วย

เมื่อกำหนดทั้งการรีเซ็ตรายวันและรีเซ็ตเมื่อว่างไว้ ตัวที่หมดอายุก่อนจะมีผลก่อน

เซสชันที่มี CLI session ซึ่งผู้ให้บริการเป็นเจ้าของและยัง active อยู่ จะไม่ถูกตัดด้วยค่าเริ่มต้น
รายวันแบบ implicit ใช้ `/reset` หรือกำหนด `session.reset` อย่างชัดเจนเมื่อเซสชันเหล่านั้นควรหมดอายุตามตัวจับเวลา

## ตำแหน่งที่เก็บสถานะ

สถานะของเซสชันทั้งหมดเป็นของ **gateway** UI clients จะ query gateway เพื่อ
ดึงข้อมูลเซสชัน

- **Store:** `~/.openclaw/agents/<agentId>/sessions/sessions.json`
- **Transcripts:** `~/.openclaw/agents/<agentId>/sessions/<sessionId>.jsonl`

## การบำรุงรักษาเซสชัน

OpenClaw จำกัดขนาดพื้นที่เก็บเซสชันโดยอัตโนมัติเมื่อเวลาผ่านไป โดยค่าเริ่มต้นจะทำงาน
ในโหมด `warn` (รายงานสิ่งที่จะถูกล้าง) ตั้งค่า `session.maintenance.mode`
เป็น `"enforce"` เพื่อให้ล้างอัตโนมัติ:

```json5
{
  session: {
    maintenance: {
      mode: "enforce",
      pruneAfter: "30d",
      maxEntries: 500,
    },
  },
}
```

ดูตัวอย่างล่วงหน้าด้วย `openclaw sessions cleanup --dry-run`

## การตรวจสอบเซสชัน

- `openclaw status` -- พาธ session store และกิจกรรมล่าสุด
- `openclaw sessions --json` -- เซสชันทั้งหมด (กรองด้วย `--active <minutes>`)
- `/status` ในแชต -- การใช้ context, โมเดล และ toggles
- `/context list` -- สิ่งที่อยู่ใน system prompt

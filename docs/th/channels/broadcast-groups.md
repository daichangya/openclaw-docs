# กลุ่มการกระจายข้อความ

**สถานะ:** Experimental  
**เวอร์ชัน:** เพิ่มใน 2026.1.9

## ภาพรวม

กลุ่มการกระจายข้อความช่วยให้หลายเอเจนต์ประมวลผลและตอบกลับข้อความเดียวกันได้พร้อมกัน ซึ่งช่วยให้คุณสร้างทีมเอเจนต์เฉพาะทางที่ทำงานร่วมกันได้ในกลุ่ม WhatsApp เดียวหรือ DM เดียว โดยทั้งหมดใช้หมายเลขโทรศัพท์เพียงหมายเลขเดียว

ขอบเขตปัจจุบัน: **WhatsApp เท่านั้น** (ช่องทางเว็บ)

กลุ่มการกระจายข้อความจะถูกประเมินหลังจาก allowlist ของช่องทางและกฎการเปิดใช้งานกลุ่ม ในกลุ่ม WhatsApp หมายความว่าการกระจายข้อความจะเกิดขึ้นเมื่อ OpenClaw ปกติจะตอบกลับอยู่แล้ว (ตัวอย่างเช่น เมื่อมีการ mention ขึ้นอยู่กับการตั้งค่ากลุ่มของคุณ)

## กรณีการใช้งาน

### 1. ทีมเอเจนต์เฉพาะทาง

ติดตั้งหลายเอเจนต์ที่มีหน้าที่เฉพาะเจาะจงและชัดเจน:

```
กลุ่ม: "ทีมพัฒนา"
เอเจนต์:
  - CodeReviewer (ตรวจทานโค้ดสั้นๆ)
  - DocumentationBot (สร้างเอกสาร)
  - SecurityAuditor (ตรวจสอบช่องโหว่)
  - TestGenerator (แนะนำกรณีทดสอบ)
```

แต่ละเอเจนต์จะประมวลผลข้อความเดียวกันและให้มุมมองเฉพาะทางของตัวเอง

### 2. การรองรับหลายภาษา

```
กลุ่ม: "ฝ่ายสนับสนุนนานาชาติ"
เอเจนต์:
  - Agent_EN (ตอบเป็นภาษาอังกฤษ)
  - Agent_DE (ตอบเป็นภาษาเยอรมัน)
  - Agent_ES (ตอบเป็นภาษาสเปน)
```

### 3. เวิร์กโฟลว์การประกันคุณภาพ

```
กลุ่ม: "ฝ่ายสนับสนุนลูกค้า"
เอเจนต์:
  - SupportAgent (ให้คำตอบ)
  - QAAgent (ตรวจสอบคุณภาพ ตอบกลับเฉพาะเมื่อพบปัญหา)
```

### 4. ระบบงานอัตโนมัติ

```
กลุ่ม: "การจัดการโครงการ"
เอเจนต์:
  - TaskTracker (อัปเดตฐานข้อมูลงาน)
  - TimeLogger (บันทึกเวลาที่ใช้)
  - ReportGenerator (สร้างสรุป)
```

## การกำหนดค่า

### การตั้งค่าพื้นฐาน

เพิ่มส่วน `broadcast` ระดับบนสุด (ถัดจาก `bindings`) คีย์คือ peer id ของ WhatsApp:

- แชตกลุ่ม: group JID (เช่น `120363403215116621@g.us`)
- DM: หมายเลขโทรศัพท์รูปแบบ E.164 (เช่น `+15551234567`)

```json
{
  "broadcast": {
    "120363403215116621@g.us": ["alfred", "baerbel", "assistant3"]
  }
}
```

**ผลลัพธ์:** เมื่อ OpenClaw ควรตอบกลับในแชตนี้ ระบบจะเรียกใช้เอเจนต์ทั้งสามตัว

### กลยุทธ์การประมวลผล

ควบคุมวิธีที่เอเจนต์ประมวลผลข้อความ:

#### แบบขนาน (ค่าเริ่มต้น)

เอเจนต์ทั้งหมดประมวลผลพร้อมกัน:

```json
{
  "broadcast": {
    "strategy": "parallel",
    "120363403215116621@g.us": ["alfred", "baerbel"]
  }
}
```

#### ตามลำดับ

เอเจนต์ประมวลผลตามลำดับ (ตัวหนึ่งรอให้ตัวก่อนหน้าเสร็จก่อน):

```json
{
  "broadcast": {
    "strategy": "sequential",
    "120363403215116621@g.us": ["alfred", "baerbel"]
  }
}
```

## วิธีการทำงาน

### ลำดับการไหลของข้อความ

1. **ข้อความขาเข้า** มาถึงในกลุ่ม WhatsApp
2. **ตรวจสอบการกระจายข้อความ**: ระบบตรวจสอบว่า peer ID อยู่ใน `broadcast` หรือไม่
3. **ถ้าอยู่ในรายการกระจายข้อความ**:
   - เอเจนต์ทั้งหมดที่ระบุไว้จะประมวลผลข้อความ
   - แต่ละเอเจนต์มี session key ของตัวเองและมี context แยกจากกัน
   - เอเจนต์จะประมวลผลแบบขนาน (ค่าเริ่มต้น) หรือตามลำดับ
4. **ถ้าไม่อยู่ในรายการกระจายข้อความ**:
   - จะใช้การกำหนดเส้นทางปกติ (binding แรกที่ตรงกัน)

หมายเหตุ: กลุ่มการกระจายข้อความไม่ได้ข้าม allowlist ของช่องทางหรือกฎการเปิดใช้งานกลุ่ม (mentions/commands/ฯลฯ) แต่เปลี่ยนเพียง _เอเจนต์ตัวใดจะทำงาน_ เมื่อข้อความนั้นเข้าเกณฑ์สำหรับการประมวลผล

### การแยก Session

แต่ละเอเจนต์ในกลุ่มการกระจายข้อความจะมีสิ่งต่อไปนี้แยกออกจากกันโดยสมบูรณ์:

- **Session keys** (`agent:alfred:whatsapp:group:120363...` เทียบกับ `agent:baerbel:whatsapp:group:120363...`)
- **ประวัติการสนทนา** (เอเจนต์จะไม่เห็นข้อความของเอเจนต์ตัวอื่น)
- **Workspace** (sandbox แยกกัน หากมีการกำหนดค่าไว้)
- **การเข้าถึงเครื่องมือ** (มีรายการ allow/deny ต่างกัน)
- **หน่วยความจำ/context** (มี IDENTITY.md, SOUL.md ฯลฯ แยกกัน)
- **บัฟเฟอร์ context ของกลุ่ม** (ข้อความล่าสุดในกลุ่มที่ใช้เป็น context) ใช้ร่วมกันต่อ peer ดังนั้นเอเจนต์ที่อยู่ในการกระจายข้อความทุกตัวจะเห็น context เดียวกันเมื่อถูกเรียกใช้

สิ่งนี้ทำให้แต่ละเอเจนต์สามารถมี:

- บุคลิกที่ต่างกัน
- การเข้าถึงเครื่องมือที่ต่างกัน (เช่น อ่านอย่างเดียว เทียบกับ อ่าน-เขียน)
- โมเดลที่ต่างกัน (เช่น opus เทียบกับ sonnet)
- Skills ที่ติดตั้งต่างกัน

## แนวทางปฏิบัติที่แนะนำ

### 1. ให้เอเจนต์มีหน้าที่ชัดเจน

ออกแบบแต่ละเอเจนต์ให้มีหน้าที่เดียวที่ชัดเจน:

```json
{
  "broadcast": {
    "DEV_GROUP": ["formatter", "linter", "tester"]
  }
}
```

✅ **ดี:** แต่ละเอเจนต์มีงานเดียว  
❌ **ไม่ดี:** มีเอเจนต์ "dev-helper" แบบทั่วไปเพียงตัวเดียว

### 2. ใช้ชื่อที่สื่อความหมาย

ทำให้ชัดเจนว่าแต่ละเอเจนต์ทำอะไร:

```json
{
  "agents": {
    "security-scanner": { "name": "Security Scanner" },
    "code-formatter": { "name": "Code Formatter" },
    "test-generator": { "name": "Test Generator" }
  }
}
```

### 3. กำหนดการเข้าถึงเครื่องมือให้ต่างกัน

ให้เอเจนต์มีเฉพาะเครื่องมือที่จำเป็นต้องใช้:

```json
{
  "agents": {
    "reviewer": {
      "tools": { "allow": ["read", "exec"] } // Read-only
    },
    "fixer": {
      "tools": { "allow": ["read", "write", "edit", "exec"] } // Read-write
    }
  }
}
```

### 4. ติดตามประสิทธิภาพ

หากมีเอเจนต์จำนวนมาก ให้พิจารณา:

- ใช้ `"strategy": "parallel"` (ค่าเริ่มต้น) เพื่อความเร็ว
- จำกัดกลุ่มการกระจายข้อความไว้ที่ 5-10 เอเจนต์
- ใช้โมเดลที่เร็วกว่าสำหรับเอเจนต์ที่ง่ายกว่า

### 5. จัดการความล้มเหลวอย่างเหมาะสม

เอเจนต์ล้มเหลวแยกจากกัน ความผิดพลาดของเอเจนต์ตัวหนึ่งจะไม่บล็อกตัวอื่น:

```
ข้อความ → [เอเจนต์ A ✓, เอเจนต์ B ✗ ข้อผิดพลาด, เอเจนต์ C ✓]
ผลลัพธ์: เอเจนต์ A และ C ตอบกลับ เอเจนต์ B บันทึกข้อผิดพลาด
```

## ความเข้ากันได้

### ผู้ให้บริการ

ปัจจุบันกลุ่มการกระจายข้อความใช้งานได้กับ:

- ✅ WhatsApp (ติดตั้งใช้งานแล้ว)
- 🚧 Telegram (วางแผนไว้)
- 🚧 Discord (วางแผนไว้)
- 🚧 Slack (วางแผนไว้)

### การกำหนดเส้นทาง

กลุ่มการกระจายข้อความทำงานร่วมกับการกำหนดเส้นทางที่มีอยู่ได้:

```json
{
  "bindings": [
    {
      "match": { "channel": "whatsapp", "peer": { "kind": "group", "id": "GROUP_A" } },
      "agentId": "alfred"
    }
  ],
  "broadcast": {
    "GROUP_B": ["agent1", "agent2"]
  }
}
```

- `GROUP_A`: มีเพียง alfred ที่ตอบกลับ (การกำหนดเส้นทางปกติ)
- `GROUP_B`: ทั้ง agent1 และ agent2 ตอบกลับ (การกระจายข้อความ)

**ลำดับความสำคัญ:** `broadcast` มีความสำคัญเหนือ `bindings`

## การแก้ไขปัญหา

### เอเจนต์ไม่ตอบกลับ

**ตรวจสอบ:**

1. มี agent ID อยู่ใน `agents.list`
2. รูปแบบ Peer ID ถูกต้อง (เช่น `120363403215116621@g.us`)
3. เอเจนต์ไม่ได้อยู่ใน deny list

**ดีบัก:**

```bash
tail -f ~/.openclaw/logs/gateway.log | grep broadcast
```

### มีเพียงเอเจนต์ตัวเดียวที่ตอบกลับ

**สาเหตุ:** Peer ID อาจอยู่ใน `bindings` แต่ไม่ได้อยู่ใน `broadcast`

**วิธีแก้:** เพิ่มลงในการตั้งค่า broadcast หรือนำออกจาก bindings

### ปัญหาด้านประสิทธิภาพ

**หากช้ากับเอเจนต์จำนวนมาก:**

- ลดจำนวนเอเจนต์ต่อกลุ่ม
- ใช้โมเดลที่เบากว่า (sonnet แทน opus)
- ตรวจสอบเวลาเริ่มต้นของ sandbox

## ข้อมูลอ้างอิง API

### สคีมาการตั้งค่า

```typescript
interface OpenClawConfig {
  broadcast?: {
    strategy?: "parallel" | "sequential";
    [peerId: string]: string[];
  };
}
```

### ฟิลด์

- `strategy` (ไม่บังคับ): วิธีประมวลผลเอเจนต์
  - `"parallel"` (ค่าเริ่มต้น): เอเจนต์ทั้งหมดประมวลผลพร้อมกัน
  - `"sequential"`: เอเจนต์ประมวลผลตามลำดับในอาร์เรย์
- `[peerId]`: WhatsApp group JID, หมายเลข E.164 หรือ peer ID อื่น
  - ค่า: อาร์เรย์ของ agent ID ที่ควรประมวลผลข้อความ

## ข้อจำกัด

1. **จำนวนเอเจนต์สูงสุด:** ไม่มีข้อจำกัดแบบตายตัว แต่ 10+ เอเจนต์อาจทำงานช้า
2. **context ที่ใช้ร่วมกัน:** เอเจนต์จะไม่เห็นคำตอบของกันและกัน (เป็นไปตามการออกแบบ)
3. **ลำดับข้อความ:** คำตอบแบบขนานอาจมาถึงในลำดับใดก็ได้
4. **ขีดจำกัดอัตรา:** เอเจนต์ทั้งหมดจะนับรวมในขีดจำกัดอัตราของ WhatsApp

## การปรับปรุงในอนาคต

ฟีเจอร์ที่วางแผนไว้:

- [ ] โหมด context ที่ใช้ร่วมกัน (เอเจนต์เห็นคำตอบของกันและกัน)
- [ ] การประสานงานระหว่างเอเจนต์ (เอเจนต์สามารถส่งสัญญาณถึงกันได้)
- [ ] การเลือกเอเจนต์แบบไดนามิก (เลือกเอเจนต์ตามเนื้อหาข้อความ)
- [ ] ลำดับความสำคัญของเอเจนต์ (บางเอเจนต์ตอบก่อนตัวอื่น)

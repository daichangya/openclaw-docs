# Гігієна transcript (виправлення для provider)

Цей документ описує **виправлення, специфічні для provider**, які застосовуються до transcript перед запуском
(побудовою контексту моделі). Це коригування **в пам’яті**, що використовуються для дотримання строгих
вимог provider. Ці кроки гігієни **не** переписують збережений JSONL transcript
на диску; однак окремий прохід відновлення файлу сесії може переписати некоректні JSONL-файли,
відкидаючи недійсні рядки до завантаження сесії. Коли відновлення відбувається, оригінальний
файл зберігається як резервна копія поруч із файлом сесії.

Область охоплює:

- Санітизацію id викликів інструментів
- Валідацію вхідних даних викликів інструментів
- Відновлення пар результатів інструментів
- Валідацію / впорядкування ходів
- Очищення signatures thought
- Санітизацію payload зображень
- Тегування provenance користувацького вводу (для маршрутизованих між сесіями запитів)

Якщо вам потрібні подробиці про зберігання transcript, див.:

- [/reference/session-management-compaction](/uk/reference/session-management-compaction)

## Де це виконується

Уся гігієна transcript централізована у вбудованому runner:

- Вибір політики: `src/agents/transcript-policy.ts`
- Застосування санітизації/відновлення: `sanitizeSessionHistory` у `src/agents/pi-embedded-runner/replay-history.ts`

Політика використовує `provider`, `modelApi` і `modelId`, щоб вирішити, що застосовувати.

Окремо від гігієни transcript файли сесій відновлюються (за потреби) до завантаження:

- `repairSessionFileIfNeeded` у `src/agents/session-file-repair.ts`
- Викликається з `run/attempt.ts` і `compact.ts` (вбудований runner)

## Глобальне правило: санітизація зображень

Payload зображень завжди санітизуються, щоб запобігти відхиленню на боці provider через
обмеження розміру (масштабування/повторне стиснення завеликих base64-зображень).

Це також допомагає контролювати тиск на токени, спричинений зображеннями, для моделей із підтримкою vision.
Менші максимальні розміри зазвичай зменшують використання токенів; більші розміри зберігають деталізацію.

Реалізація:

- `sanitizeSessionMessagesImages` у `src/agents/pi-embedded-helpers/images.ts`
- `sanitizeContentBlocksImages` у `src/agents/tool-images.ts`
- Максимальний бік зображення налаштовується через `agents.defaults.imageMaxDimensionPx` (типово: `1200`).

---

## Глобальне правило: некоректні виклики інструментів

Блоки викликів інструментів асистента, у яких відсутні і `input`, і `arguments`, відкидаються
до побудови контексту моделі. Це запобігає відхиленню provider через частково
збережені виклики інструментів (наприклад, після помилки rate limit).

Реалізація:

- `sanitizeToolCallInputs` у `src/agents/session-transcript-repair.ts`
- Застосовується в `sanitizeSessionHistory` у `src/agents/pi-embedded-runner/replay-history.ts`

---

## Глобальне правило: provenance міжсесійного вводу

Коли агент надсилає запит в іншу сесію через `sessions_send` (включно
з кроками відповіді/оголошення між агентами), OpenClaw зберігає створений хід користувача з:

- `message.provenance.kind = "inter_session"`

Ці метадані записуються під час додавання transcript і не змінюють роль
(`role: "user"` зберігається для сумісності з provider). Читачі transcript можуть використовувати
це, щоб не трактувати маршрутизовані внутрішні запити як інструкції від кінцевого користувача.

Під час повторної побудови контексту OpenClaw також додає на початок короткий маркер `[Inter-session message]`
до таких користувацьких ходів у пам’яті, щоб модель могла відрізняти їх від
зовнішніх інструкцій кінцевого користувача.

## Матриця provider (поточна поведінка)

**OpenAI / OpenAI Codex**

- Лише санітизація зображень.
- Відкидання осиротілих signatures reasoning (окремих елементів reasoning без наступного content block) для transcript OpenAI Responses/Codex.
- Немає санітизації id викликів інструментів.
- Немає відновлення пар результатів інструментів.
- Немає валідації або перевпорядкування ходів.
- Немає синтетичних результатів інструментів.
- Немає видалення signatures thought.

**Google (Generative AI / Gemini CLI / Antigravity)**

- Санітизація id викликів інструментів: строгий буквено-цифровий формат.
- Відновлення пар результатів інструментів і синтетичні результати інструментів.
- Валідація ходів (чергування ходів у стилі Gemini).
- Виправлення порядку ходів Google (додавання спочатку крихітного bootstrap користувача, якщо історія починається з асистента).
- Antigravity Claude: нормалізація signatures thinking; відкидання блоків thinking без підпису.

**Anthropic / Minimax (Anthropic-compatible)**

- Відновлення пар результатів інструментів і синтетичні результати інструментів.
- Валідація ходів (злиття послідовних ходів користувача для дотримання строгого чергування).

**Mistral (включно з виявленням за model-id)**

- Санітизація id викликів інструментів: strict9 (буквено-цифровий формат довжини 9).

**OpenRouter Gemini**

- Очищення signatures thought: видалення значень `thought_signature`, що не є base64 (base64 зберігаються).

**Усе інше**

- Лише санітизація зображень.

## Історична поведінка (до 2026.1.22)

До релізу 2026.1.22 OpenClaw застосовував кілька шарів гігієни transcript:

- **Розширення transcript-sanitize** запускалося під час кожної побудови контексту й могло:
  - Відновлювати зв’язування використання/результатів інструментів.
  - Санітизувати id викликів інструментів (зокрема в нестрогому режимі, який зберігав `_`/`-`).
- Runner також виконував санітизацію, специфічну для provider, що дублювало роботу.
- Додаткові мутації відбувалися поза політикою provider, зокрема:
  - Видалення тегів `<final>` з тексту асистента до збереження.
  - Відкидання порожніх ходів помилки асистента.
  - Обрізання вмісту асистента після викликів інструментів.

Ця складність спричиняла регресії між provider (особливо для зв’язування `openai-responses`
`call_id|fc_id`). Очищення у версії 2026.1.22 прибрало розширення, централізувало
логіку в runner і зробило OpenAI **без змін** понад санітизацію зображень.

Цей документ описує **виправлення, специфічні для постачальника**, які застосовуються до transcript перед запуском
(під час побудови контексту моделі). Це **в пам’яті** коригування, що використовуються для дотримання суворих
вимог постачальника. Ці кроки гігієни **не** переписують збережений JSONL transcript
на диску; однак окремий прохід відновлення файлу сесії може переписати некоректні JSONL-файли,
видаляючи невалідні рядки перед завантаженням сесії. Коли відновлення відбувається, оригінальний
файл резервно копіюється поруч із файлом сесії.

Обсяг охоплює:

- Контекст prompt лише для runtime, який не потрапляє у видимі користувачу ходи transcript
- Санітизацію id викликів інструментів
- Валідацію входів викликів інструментів
- Відновлення парування результатів інструментів
- Валідацію / упорядкування ходів
- Очищення thought signature
- Санітизацію payload зображень
- Позначення походження користувацького введення (для prompt, маршрутизованих між сесіями)

Якщо вам потрібні деталі щодо зберігання transcript, дивіться:

- [/reference/session-management-compaction](/uk/reference/session-management-compaction)

## Глобальне правило: контекст runtime — це не користувацький transcript

Контекст runtime/system можна додавати до prompt моделі для певного ходу, але це
не вміст, створений кінцевим користувачем. OpenClaw зберігає окреме тіло prompt,
орієнтоване на transcript, для відповідей Gateway, followup у черзі, ACP, CLI і вбудованих
запусків Pi. Збережені видимі користувацькі ходи використовують це тіло transcript замість
prompt, збагаченого runtime-контекстом.

Для застарілих сесій, які вже зберегли обгортки runtime, поверхні історії Gateway
застосовують проєкцію відображення перед поверненням повідомлень у WebChat,
TUI, клієнтам REST або SSE.

## Де це виконується

Уся гігієна transcript централізована у вбудованому раннері:

- Вибір політики: `src/agents/transcript-policy.ts`
- Застосування санітизації/відновлення: `sanitizeSessionHistory` у `src/agents/pi-embedded-runner/replay-history.ts`

Політика використовує `provider`, `modelApi` і `modelId`, щоб визначити, що застосовувати.

Окремо від гігієни transcript, файли сесій відновлюються (за потреби) перед завантаженням:

- `repairSessionFileIfNeeded` у `src/agents/session-file-repair.ts`
- Викликається з `run/attempt.ts` і `compact.ts` (вбудований раннер)

## Глобальне правило: санітизація зображень

Payload зображень завжди санітизуються, щоб запобігти відхиленню на боці постачальника через
обмеження розміру (масштабування/повторне стиснення завеликих base64-зображень).

Це також допомагає контролювати тиск на токени, спричинений зображеннями, для моделей із підтримкою vision.
Менші максимальні розміри зазвичай зменшують використання токенів; більші розміри зберігають деталізацію.

Реалізація:

- `sanitizeSessionMessagesImages` у `src/agents/pi-embedded-helpers/images.ts`
- `sanitizeContentBlocksImages` у `src/agents/tool-images.ts`
- Максимальна сторона зображення налаштовується через `agents.defaults.imageMaxDimensionPx` (типово: `1200`).

---

## Глобальне правило: некоректні виклики інструментів

Блоки викликів інструментів асистента, у яких немає ні `input`, ні `arguments`, відкидаються
до побудови контексту моделі. Це запобігає відхиленню постачальником через частково
збережені виклики інструментів (наприклад, після помилки rate limit).

Реалізація:

- `sanitizeToolCallInputs` у `src/agents/session-transcript-repair.ts`
- Застосовується в `sanitizeSessionHistory` у `src/agents/pi-embedded-runner/replay-history.ts`

---

## Глобальне правило: походження введення між сесіями

Коли агент надсилає prompt в іншу сесію через `sessions_send` (включно з
кроками agent-to-agent reply/announce), OpenClaw зберігає створений користувацький хід із:

- `message.provenance.kind = "inter_session"`

Ці метадані записуються під час додавання до transcript і не змінюють роль
(`role: "user"` зберігається для сумісності з постачальниками). Читачі transcript можуть використовувати
це, щоб не трактувати внутрішньо маршрутизовані prompt як інструкції від кінцевого користувача.

Під час перебудови контексту OpenClaw також додає на початок таких користувацьких ходів короткий маркер `[Inter-session message]`
в пам’яті, щоб модель могла відрізняти їх від
зовнішніх інструкцій кінцевого користувача.

## Матриця постачальників (поточна поведінка)

**OpenAI / OpenAI Codex**

- Лише санітизація зображень.
- Відкидання осиротілих reasoning signature (окремих reasoning-елементів без наступного content block) для transcript OpenAI Responses/Codex, а також відкидання replayable OpenAI reasoning після перемикання маршруту моделі.
- Без санітизації id викликів інструментів.
- Без відновлення парування результатів інструментів.
- Без валідації або перевпорядкування ходів.
- Без синтетичних результатів інструментів.
- Без видалення thought signature.

**Google (Generative AI / Gemini CLI / Antigravity)**

- Санітизація id викликів інструментів: суворо алфавітно-цифрові.
- Відновлення парування результатів інструментів і синтетичні результати інструментів.
- Валідація ходів (чергування ходів у стилі Gemini).
- Виправлення порядку ходів Google (додавання малого bootstrap користувача на початок, якщо історія починається з асистента).
- Antigravity Claude: нормалізація thinking signature; відкидання блоків thinking без signature.

**Anthropic / Minimax (Anthropic-сумісні)**

- Відновлення парування результатів інструментів і синтетичні результати інструментів.
- Валідація ходів (об’єднання послідовних користувацьких ходів для дотримання суворого чергування).

**Mistral (включно з виявленням за id моделі)**

- Санітизація id викликів інструментів: strict9 (алфавітно-цифрові, довжина 9).

**OpenRouter Gemini**

- Очищення thought signature: видалення значень `thought_signature`, які не є base64 (base64 зберігаються).

**Усе інше**

- Лише санітизація зображень.

## Історична поведінка (до 2026.1.22)

До випуску 2026.1.22 OpenClaw застосовував кілька рівнів гігієни transcript:

- Розширення **transcript-sanitize** запускалося під час кожної побудови контексту і могло:
  - Відновлювати парування використання/результату інструментів.
  - Санітизувати id викликів інструментів (включно з несуворим режимом, який зберігав `_`/`-`).
- Раннер також виконував санітизацію, специфічну для постачальника, що дублювало роботу.
- Додаткові мутації відбувалися поза політикою постачальника, зокрема:
  - Видалення тегів `<final>` з тексту асистента перед збереженням.
  - Відкидання порожніх ходів помилок асистента.
  - Обрізання вмісту асистента після викликів інструментів.

Ця складність спричиняла міжпостачальницькі регресії (зокрема парування `openai-responses`
`call_id|fc_id`). Очищення в 2026.1.22 видалило розширення, централізувало
логіку в раннері та зробило OpenAI **без втручання** за межами санітизації зображень.

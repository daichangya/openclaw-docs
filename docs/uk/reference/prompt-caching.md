Кешування запитів означає, що провайдер моделі може повторно використовувати незмінні префікси запиту (зазвичай системні/розробницькі інструкції та інший стабільний контекст) між ходами замість повторної обробки їх щоразу. OpenClaw нормалізує використання провайдерів у `cacheRead` і `cacheWrite`, якщо вихідний API напряму надає ці лічильники.

Поверхні статусу також можуть відновлювати лічильники кешу з найновішого журналу використання транскрипту,
коли в живому знімку сеансу їх немає, тож `/status` може й далі
показувати рядок кешу після часткової втрати метаданих сеансу. Наявні ненульові живі
значення кешу все одно мають пріоритет над резервними значеннями з транскрипту.

Чому це важливо: нижча вартість токенів, швидші відповіді та передбачуваніша продуктивність для довготривалих сеансів. Без кешування повторювані запити оплачують повну вартість запиту на кожному ході, навіть якщо більшість вхідних даних не змінилася.

На цій сторінці описано всі параметри, пов’язані з кешем, які впливають на повторне використання запитів і вартість токенів.

Посилання на провайдерів:

- Кешування запитів Anthropic: [https://platform.claude.com/docs/en/build-with-claude/prompt-caching](https://platform.claude.com/docs/en/build-with-claude/prompt-caching)
- Кешування запитів OpenAI: [https://developers.openai.com/api/docs/guides/prompt-caching](https://developers.openai.com/api/docs/guides/prompt-caching)
- Заголовки API OpenAI та ідентифікатори запитів: [https://developers.openai.com/api/reference/overview](https://developers.openai.com/api/reference/overview)
- Ідентифікатори запитів та помилки Anthropic: [https://platform.claude.com/docs/en/api/errors](https://platform.claude.com/docs/en/api/errors)

## Основні параметри

### `cacheRetention` (глобальне значення за замовчуванням, модель і для кожного агента)

Установіть утримання кешу як глобальне значення за замовчуванням для всіх моделей:

```yaml
agents:
  defaults:
    params:
      cacheRetention: "long" # none | short | long
```

Перевизначення для окремої моделі:

```yaml
agents:
  defaults:
    models:
      "anthropic/claude-opus-4-6":
        params:
          cacheRetention: "short" # none | short | long
```

Перевизначення для окремого агента:

```yaml
agents:
  list:
    - id: "alerts"
      params:
        cacheRetention: "none"
```

Порядок злиття конфігурації:

1. `agents.defaults.params` (глобальне значення за замовчуванням — застосовується до всіх моделей)
2. `agents.defaults.models["provider/model"].params` (перевизначення для окремої моделі)
3. `agents.list[].params` (відповідний ідентифікатор агента; перевизначає за ключем)

### `contextPruning.mode: "cache-ttl"`

Очищає старий контекст результатів інструментів після вікон TTL кешу, щоб запити після простою не кешували повторно завелику історію.

```yaml
agents:
  defaults:
    contextPruning:
      mode: "cache-ttl"
      ttl: "1h"
```

Див. [Очищення сеансу](/uk/concepts/session-pruning), щоб дізнатися повну поведінку.

### Підтримання тепла Heartbeat

Heartbeat може підтримувати вікна кешу теплими та зменшувати повторні записи в кеш після проміжків простою.

```yaml
agents:
  defaults:
    heartbeat:
      every: "55m"
```

Heartbeat для окремого агента підтримується в `agents.list[].heartbeat`.

## Поведінка провайдера

### Anthropic (прямий API)

- `cacheRetention` підтримується.
- Для профілів автентифікації Anthropic API-key OpenClaw встановлює `cacheRetention: "short"` для посилань на моделі Anthropic, якщо це значення не задано.
- Власні відповіді Anthropic Messages надають як `cache_read_input_tokens`, так і `cache_creation_input_tokens`, тож OpenClaw може показувати і `cacheRead`, і `cacheWrite`.
- Для власних запитів Anthropic `cacheRetention: "short"` відповідає типовому 5-хвилинному ефемерному кешу, а `cacheRetention: "long"` підвищує TTL до 1 години лише на прямих хостах `api.anthropic.com`.

### OpenAI (прямий API)

- Кешування запитів автоматичне на підтримуваних сучасних моделях. OpenClaw не потрібно вставляти маркери кешу на рівні блоків.
- OpenClaw використовує `prompt_cache_key`, щоб маршрутизація кешу залишалася стабільною між ходами, і використовує `prompt_cache_retention: "24h"` лише коли на прямих хостах OpenAI вибрано `cacheRetention: "long"`.
- OpenAI-сумісні провайдери Completions отримують `prompt_cache_key` лише тоді, коли їхня конфігурація моделі явно задає `compat.supportsPromptCacheKey: true`; `cacheRetention: "none"` усе одно його пригнічує.
- Відповіді OpenAI надають кешовані токени запиту через `usage.prompt_tokens_details.cached_tokens` (або `input_tokens_details.cached_tokens` у подіях Responses API). OpenClaw відображає це в `cacheRead`.
- OpenAI не надає окремий лічильник токенів запису в кеш, тому `cacheWrite` залишається `0` на шляхах OpenAI, навіть коли провайдер прогріває кеш.
- OpenAI повертає корисні заголовки для трасування та обмежень частоти, як-от `x-request-id`, `openai-processing-ms` і `x-ratelimit-*`, але облік влучань у кеш слід брати з payload використання, а не із заголовків.
- На практиці OpenAI часто поводиться як кеш початкового префікса, а не як рухоме повторне використання повної історії у стилі Anthropic. Стабільний довгий текст префікса може виходити на плато близько `4864` кешованих токенів у поточних живих пробах, тоді як транскрипти з інструментами або в стилі MCP часто виходять на плато близько `4608` кешованих токенів навіть за точного повтору.

### Anthropic Vertex

- Моделі Anthropic на Vertex AI (`anthropic-vertex/*`) підтримують `cacheRetention` так само, як і прямий Anthropic.
- `cacheRetention: "long"` відповідає реальному TTL кешу запитів у 1 годину на кінцевих точках Vertex AI.
- Значення утримання кешу за замовчуванням для `anthropic-vertex` збігається зі значеннями за замовчуванням для прямого Anthropic.
- Запити Vertex маршрутизуються через формування кешу з урахуванням меж, щоб повторне використання кешу відповідало тому, що фактично отримують провайдери.

### Amazon Bedrock

- Посилання на моделі Anthropic Claude (`amazon-bedrock/*anthropic.claude*`) підтримують явне наскрізне передавання `cacheRetention`.
- Для моделей Bedrock, що не належать Anthropic, під час виконання примусово встановлюється `cacheRetention: "none"`.

### Моделі Anthropic в OpenRouter

Для посилань на моделі `openrouter/anthropic/*` OpenClaw вставляє Anthropic
`cache_control` у блоки системного/розробницького запиту, щоб покращити повторне використання
кешу запитів, лише коли запит і далі спрямовується на перевірений маршрут OpenRouter
(`openrouter` на його стандартній кінцевій точці або будь-який провайдер/base URL, що вказує
на `openrouter.ai`).

Якщо ви перенаправите модель на довільний проксі-URL, сумісний з OpenAI, OpenClaw
припинить вставляти ці специфічні для OpenRouter маркери кешу Anthropic.

### Інші провайдери

Якщо провайдер не підтримує цей режим кешу, `cacheRetention` не матиме ефекту.

### Google Gemini прямий API

- Прямий транспорт Gemini (`api: "google-generative-ai"`) повідомляє про влучання в кеш
  через вихідний `cachedContentTokenCount`; OpenClaw відображає це в `cacheRead`.
- Коли на прямій моделі Gemini задано `cacheRetention`, OpenClaw автоматично
  створює, повторно використовує та оновлює ресурси `cachedContents` для системних запитів
  у запусках Google AI Studio. Це означає, що вам більше не потрібно
  вручну попередньо створювати дескриптор cached-content.
- Ви все ще можете передати наявний дескриптор Gemini cached-content
  через `params.cachedContent` (або застарілий `params.cached_content`) у налаштованій
  моделі.
- Це окремо від кешування префікса запитів Anthropic/OpenAI. Для Gemini
  OpenClaw керує власним ресурсом провайдера `cachedContents`, а не
  вставляє маркери кешу в запит.

### Використання Gemini CLI JSON

- Вивід Gemini CLI JSON також може показувати влучання в кеш через `stats.cached`;
  OpenClaw відображає це в `cacheRead`.
- Якщо CLI не надає пряме значення `stats.input`, OpenClaw виводить вхідні токени
  з `stats.input_tokens - stats.cached`.
- Це лише нормалізація використання. Це не означає, що OpenClaw створює
  маркери кешу запитів у стилі Anthropic/OpenAI для Gemini CLI.

## Межа кешу системного запиту

OpenClaw розділяє системний запит на **стабільний префікс** і **мінливий
суфікс**, розділені внутрішньою межею префікса кешу. Вміст над
межею (визначення інструментів, метадані Skills, файли робочого простору та інший
відносно статичний контекст) упорядковується так, щоб залишатися побайтно однаковим між ходами.
Вміст під межею (наприклад, `HEARTBEAT.md`, часові мітки виконання та
інші метадані для кожного ходу) може змінюватися без інвалідизації кешованого
префікса.

Ключові рішення проєктування:

- Стабільні файли контексту проєкту робочого простору впорядковуються перед `HEARTBEAT.md`, тож
  зміни Heartbeat не ламають стабільний префікс.
- Межа застосовується до формування кешу для сімейства Anthropic, сімейства OpenAI, Google і CLI-транспорту, щоб усі підтримувані провайдери отримували користь від однакової стабільності префікса.
- Запити Codex Responses і Anthropic Vertex маршрутизуються через
  формування кешу з урахуванням меж, щоб повторне використання кешу відповідало тому, що провайдери
  фактично отримують.
- Відбитки системного запиту нормалізуються (пробіли, закінчення рядків,
  контекст, доданий хуками, упорядкування можливостей середовища виконання), щоб семантично незмінні
  запити спільно використовували KV/кеш між ходами.

Якщо після зміни конфігурації або робочого простору ви бачите неочікувані сплески `cacheWrite`,
перевірте, чи ця зміна потрапляє вище чи нижче межі кешу. Переміщення
мінливого вмісту нижче межі (або його стабілізація) часто розв’язує
цю проблему.

## Захист стабільності кешу в OpenClaw

OpenClaw також зберігає детермінованість кількох чутливих до кешу форм payload, перш ніж
запит досягне провайдера:

- Каталоги інструментів Bundle MCP сортуються детерміновано перед
  реєстрацією інструментів, тож зміни в порядку `listTools()` не змінюють блок інструментів і
  не ламають префікси кешу запитів.
- У застарілих сеансах зі збереженими блоками зображень **3 найновіші
  завершені ходи** залишаються недоторканими; старіші вже оброблені блоки
  зображень можуть бути замінені маркером, щоб наступні запити з великою кількістю зображень не
  надсилали повторно великі застарілі payload.

## Шаблони налаштування

### Змішаний трафік (рекомендоване значення за замовчуванням)

Зберігайте довгоживучу базову лінію на головному агенті, а кешування вимкніть на агентів-сповіщувачах із ривковим навантаженням:

```yaml
agents:
  defaults:
    model:
      primary: "anthropic/claude-opus-4-6"
    models:
      "anthropic/claude-opus-4-6":
        params:
          cacheRetention: "long"
  list:
    - id: "research"
      default: true
      heartbeat:
        every: "55m"
    - id: "alerts"
      params:
        cacheRetention: "none"
```

### Базова лінія з пріоритетом вартості

- Установіть базове `cacheRetention: "short"`.
- Увімкніть `contextPruning.mode: "cache-ttl"`.
- Тримайте heartbeat нижче вашого TTL лише для агентів, які отримують користь від теплого кешу.

## Діагностика кешу

OpenClaw надає окрему діагностику трасування кешу для вбудованих запусків агентів.

Для звичайної діагностики, орієнтованої на користувача, `/status` та інші підсумки використання можуть використовувати
останній запис використання транскрипту як резервне джерело для `cacheRead` /
`cacheWrite`, коли в записі живого сеансу немає цих лічильників.

## Живі регресійні тести

OpenClaw підтримує один комбінований живий регресійний контроль кешу для повторюваних префіксів, ходів з інструментами, ходів із зображеннями, транскриптів інструментів у стилі MCP і контрольного сценарію Anthropic без кешу.

- `src/agents/live-cache-regression.live.test.ts`
- `src/agents/live-cache-regression-baseline.ts`

Запустіть вузький живий контроль так:

```sh
OPENCLAW_LIVE_TEST=1 OPENCLAW_LIVE_CACHE_TEST=1 pnpm test:live:cache
```

Базовий файл зберігає найновіші спостережувані живі значення, а також специфічні для провайдерів порогові значення регресії, які використовує тест.
Виконавець також використовує нові ідентифікатори сеансу для кожного запуску та простори імен запитів, щоб попередній стан кешу не забруднював поточну регресійну вибірку.

Ці тести навмисно не використовують однакові критерії успіху для всіх провайдерів.

### Живі очікування Anthropic

- Очікуються явні записи прогріву через `cacheWrite`.
- Очікується майже повне повторне використання історії на повторних ходах, оскільки контроль кешу Anthropic просуває точку розриву кешу по всій розмові.
- Поточні живі перевірки й далі використовують високі порогові значення частоти влучань для стабільних, інструментальних і графічних шляхів.

### Живі очікування OpenAI

- Очікується лише `cacheRead`. `cacheWrite` залишається `0`.
- Повторне використання кешу на повторних ходах слід розглядати як специфічне для провайдера плато, а не як рухоме повторне використання повної історії у стилі Anthropic.
- Поточні живі перевірки використовують консервативні нижні пороги, виведені зі спостережуваної живої поведінки на `gpt-5.4-mini`:
  - стабільний префікс: `cacheRead >= 4608`, частота влучань `>= 0.90`
  - транскрипт інструмента: `cacheRead >= 4096`, частота влучань `>= 0.85`
  - транскрипт зображення: `cacheRead >= 3840`, частота влучань `>= 0.82`
  - транскрипт у стилі MCP: `cacheRead >= 4096`, частота влучань `>= 0.85`

Свіжа комбінована жива перевірка від 2026-04-04 показала:

- стабільний префікс: `cacheRead=4864`, частота влучань `0.966`
- транскрипт інструмента: `cacheRead=4608`, частота влучань `0.896`
- транскрипт зображення: `cacheRead=4864`, частота влучань `0.954`
- транскрипт у стилі MCP: `cacheRead=4608`, частота влучань `0.891`

Нещодавній локальний час виконання для комбінованого контролю становив близько `88s`.

Чому перевірки відрізняються:

- Anthropic надає явні точки розриву кешу та рухоме повторне використання історії розмови.
- Кешування запитів OpenAI усе ще чутливе до точного префікса, але ефективний префікс, придатний до повторного використання в живому трафіку Responses, може виходити на плато раніше, ніж увесь запит.
- Через це порівняння Anthropic і OpenAI за єдиним міжпровайдерним порогом у відсотках створює хибні регресії.

### Конфігурація `diagnostics.cacheTrace`

```yaml
diagnostics:
  cacheTrace:
    enabled: true
    filePath: "~/.openclaw/logs/cache-trace.jsonl" # optional
    includeMessages: false # default true
    includePrompt: false # default true
    includeSystem: false # default true
```

Значення за замовчуванням:

- `filePath`: `$OPENCLAW_STATE_DIR/logs/cache-trace.jsonl`
- `includeMessages`: `true`
- `includePrompt`: `true`
- `includeSystem`: `true`

### Перемикачі середовища (разове налагодження)

- `OPENCLAW_CACHE_TRACE=1` вмикає трасування кешу.
- `OPENCLAW_CACHE_TRACE_FILE=/path/to/cache-trace.jsonl` перевизначає шлях виводу.
- `OPENCLAW_CACHE_TRACE_MESSAGES=0|1` перемикає захоплення повного payload повідомлень.
- `OPENCLAW_CACHE_TRACE_PROMPT=0|1` перемикає захоплення тексту запиту.
- `OPENCLAW_CACHE_TRACE_SYSTEM=0|1` перемикає захоплення системного запиту.

### Що перевіряти

- Події трасування кешу мають формат JSONL і включають поетапні знімки, як-от `session:loaded`, `prompt:before`, `stream:context` і `session:after`.
- Вплив кешованих токенів на кожному ході видно у звичайних поверхнях використання через `cacheRead` і `cacheWrite` (наприклад, `/usage full` і підсумки використання сеансу).
- Для Anthropic очікуються і `cacheRead`, і `cacheWrite`, коли кешування активне.
- Для OpenAI очікується `cacheRead` при влучаннях у кеш, а `cacheWrite` має залишатися `0`; OpenAI не публікує окреме поле токенів запису в кеш.
- Якщо вам потрібне трасування запитів, журналюйте ідентифікатори запитів і заголовки обмеження частоти окремо від метрик кешу. Поточний вивід трасування кешу в OpenClaw зосереджений на формі запиту/сеансу та нормалізованому використанні токенів, а не на необроблених заголовках відповіді провайдера.

## Швидке усунення проблем

- Високий `cacheWrite` на більшості ходів: перевірте мінливі вхідні дані системного запиту та переконайтеся, що модель/провайдер підтримує ваші параметри кешу.
- Високий `cacheWrite` в Anthropic: часто означає, що точка розриву кешу припадає на вміст, який змінюється в кожному запиті.
- Низький `cacheRead` в OpenAI: переконайтеся, що стабільний префікс стоїть на початку, повторюваний префікс містить щонайменше 1024 токени, а той самий `prompt_cache_key` повторно використовується для ходів, які мають спільно використовувати кеш.
- Немає ефекту від `cacheRetention`: підтвердьте, що ключ моделі збігається з `agents.defaults.models["provider/model"]`.
- Запити Bedrock Nova/Mistral з параметрами кешу: очікуване примусове встановлення `none` під час виконання.

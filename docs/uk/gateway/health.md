# Перевірки стану (CLI)

Короткий посібник, щоб перевіряти підключення каналу без здогадок.

## Швидкі перевірки

- `openclaw status` — локальне зведення: доступність/режим gateway, підказка щодо оновлення, вік пов’язаної auth каналу, сесії та нещодавня активність.
- `openclaw status --all` — повна локальна діагностика (лише читання, кольори, безпечно вставляти для налагодження).
- `openclaw status --deep` — запитує у запущеного gateway живу перевірку стану (`health` з `probe:true`), включно з перевірками каналів для кожного облікового запису, де це підтримується.
- `openclaw health` — запитує у запущеного gateway його знімок стану (лише WS; без прямих сокетів каналів із CLI).
- `openclaw health --verbose` — примусово запускає живу перевірку стану й виводить подробиці підключення gateway.
- `openclaw health --json` — машинозчитуваний вивід знімка стану.
- Надішліть `/status` окремим повідомленням у WhatsApp/WebChat, щоб отримати відповідь зі станом без виклику агента.
- Журнали: переглядайте `/tmp/openclaw/openclaw-*.log` у реальному часі та фільтруйте за `web-heartbeat`, `web-reconnect`, `web-auto-reply`, `web-inbound`.

## Глибока діагностика

- Облікові дані на диску: `ls -l ~/.openclaw/credentials/whatsapp/<accountId>/creds.json` (`mtime` має бути нещодавнім).
- Сховище сесій: `ls -l ~/.openclaw/agents/<agentId>/sessions/sessions.json` (шлях можна перевизначити в конфігурації). Кількість і нещодавні одержувачі відображаються через `status`.
- Потік повторного зв’язування: `openclaw channels logout && openclaw channels login --verbose`, коли в журналах з’являються коди стану 409–515 або `loggedOut`. (Примітка: потік входу через QR автоматично перезапускається один раз для стану 515 після сполучення.)
- Diagnostics увімкнено за замовчуванням. Gateway записує операційні факти, якщо не встановлено `diagnostics.enabled: false`. Події пам’яті записують кількість байтів RSS/heap, тиск порогів і тиск зростання. Події надмірно великих payload записують, що було відхилено, усічено або розбито на частини, а також розміри й ліміти, де доступно. Вони не записують текст повідомлення, вміст вкладень, тіло Webhook, сире тіло запиту чи відповіді, токени, cookies або секретні значення. Той самий Heartbeat запускає обмежений записувач стабільності, доступний через `openclaw gateway stability` або Gateway RPC `diagnostics.stability`. Критичні завершення Gateway, тайм-аути вимкнення й збої запуску після перезапуску зберігають останній знімок записувача в `~/.openclaw/logs/stability/`, якщо події існують; перевірити найновіший збережений набір можна командою `openclaw gateway stability --bundle latest`.
- Для звітів про помилки запустіть `openclaw gateway diagnostics export` і додайте згенерований zip-файл. Експорт об’єднує підсумок Markdown, найновіший набір стабільності, санітизовані метадані журналів, санітизовані знімки статусу/стану Gateway і форму конфігурації. Його призначено для спільного використання: текст чату, тіла Webhook, виводи інструментів, облікові дані, cookies, ідентифікатори облікових записів/повідомлень і секретні значення опускаються або редагуються. Див. [Експорт diagnostics](/uk/gateway/diagnostics).

## Конфігурація монітора стану

- `gateway.channelHealthCheckMinutes`: як часто gateway перевіряє стан каналу. За замовчуванням: `5`. Установіть `0`, щоб глобально вимкнути перезапуски монітора стану.
- `gateway.channelStaleEventThresholdMinutes`: як довго підключений канал може залишатися бездіяльним, перш ніж монітор стану вважатиме його застарілим і перезапустить. За замовчуванням: `30`. Зберігайте це значення більшим або рівним `gateway.channelHealthCheckMinutes`.
- `gateway.channelMaxRestartsPerHour`: ковзне одногодинне обмеження на кількість перезапусків монітора стану для кожного каналу/облікового запису. За замовчуванням: `10`.
- `channels.<provider>.healthMonitor.enabled`: вимикає перезапуски монітора стану для певного каналу, залишаючи глобальний моніторинг увімкненим.
- `channels.<provider>.accounts.<accountId>.healthMonitor.enabled`: перевизначення для кількох облікових записів, яке має пріоритет над налаштуванням на рівні каналу.
- Ці перевизначення для окремих каналів застосовуються до вбудованих моніторів каналів, які наразі їх підтримують: Discord, Google Chat, iMessage, Microsoft Teams, Signal, Slack, Telegram і WhatsApp.

## Коли щось не працює

- `logged out` або стан 409–515 → виконайте повторне зв’язування через `openclaw channels logout`, а потім `openclaw channels login`.
- Gateway недоступний → запустіть його: `openclaw gateway --port 18789` (використайте `--force`, якщо порт зайнято).
- Немає вхідних повідомлень → переконайтеся, що пов’язаний телефон онлайн, а відправника дозволено (`channels.whatsapp.allowFrom`); для групових чатів переконайтеся, що allowlist і правила згадки збігаються (`channels.whatsapp.groups`, `agents.list[].groupChat.mentionPatterns`).

## Окрема команда "health"

`openclaw health` запитує у запущеного gateway його знімок стану (без прямих сокетів
каналів із CLI). За замовчуванням вона може повертати свіжий кешований знімок gateway; після цього
gateway оновлює цей кеш у фоновому режимі. `openclaw health --verbose` натомість примусово
запускає живу перевірку. Команда повідомляє про пов’язані creds/вік auth, коли вони доступні,
зведення перевірок для кожного каналу, зведення сховища сесій і тривалість перевірки. Вона завершується
з ненульовим кодом, якщо gateway недоступний або перевірка не проходить/перевищує тайм-аут.

Параметри:

- `--json`: машинозчитуваний вивід JSON
- `--timeout <ms>`: перевизначити типовий тайм-аут перевірки 10 с
- `--verbose`: примусово запустити живу перевірку й вивести подробиці підключення gateway
- `--debug`: псевдонім для `--verbose`

Знімок стану містить: `ok` (boolean), `ts` (часова позначка), `durationMs` (час перевірки), стан для кожного каналу, доступність агента та зведення сховища сесій.

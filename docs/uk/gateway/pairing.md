# Pairing, яким керує Gateway (варіант B)

У pairing, яким керує Gateway, **Gateway** є джерелом істини щодо того, яким Node
дозволено приєднуватися. UI (застосунок macOS, майбутні клієнти) — це лише фронтенди,
які схвалюють або відхиляють запити, що очікують розгляду.

**Важливо:** WS Node використовують **pairing пристрою** (роль `node`) під час `connect`.
`node.pair.*` — це окреме сховище pairing і воно **не** керує WS-handshake.
Цей потік використовують лише клієнти, які явно викликають `node.pair.*`.

## Поняття

- **Запит, що очікує розгляду**: Node попросив приєднатися; потрібне схвалення.
- **Спарений Node**: схвалений Node із виданим токеном автентифікації.
- **Транспорт**: кінцева точка Gateway WS пересилає запити, але не вирішує
  питання членства. (Підтримку застарілого мосту TCP видалено.)

## Як працює pairing

1. Node підключається до Gateway WS і запитує pairing.
2. Gateway зберігає **запит, що очікує розгляду**, і надсилає `node.pair.requested`.
3. Ви схвалюєте або відхиляєте запит (через CLI або UI).
4. Після схвалення Gateway видає **новий токен** (під час повторного pairing токени ротуються).
5. Node перепідключається з використанням токена і тепер вважається «спареним».

Запити, що очікують розгляду, автоматично спливають через **5 хвилин**.

## Потік CLI (зручно без headless)

```bash
openclaw nodes pending
openclaw nodes approve <requestId>
openclaw nodes reject <requestId>
openclaw nodes status
openclaw nodes rename --node <id|name|ip> --name "Living Room iPad"
```

`nodes status` показує спарені/підключені Node та їхні можливості.

## Поверхня API (протокол gateway)

Події:

- `node.pair.requested` — надсилається, коли створено новий запит, що очікує розгляду.
- `node.pair.resolved` — надсилається, коли запит схвалено/відхилено/прострочено.

Методи:

- `node.pair.request` — створити або повторно використати запит, що очікує розгляду.
- `node.pair.list` — список Node, що очікують розгляду, + спарених Node (`operator.pairing`).
- `node.pair.approve` — схвалити запит, що очікує розгляду (видає токен).
- `node.pair.reject` — відхилити запит, що очікує розгляду.
- `node.pair.verify` — перевірити `{ nodeId, token }`.

Примітки:

- `node.pair.request` є ідемпотентним для кожного Node: повторні виклики повертають той самий
  запит, що очікує розгляду.
- Повторні запити для того самого Node, що очікує розгляду, також оновлюють збережені метадані Node
  та найновіший allowlisted-знімок оголошених команд для видимості оператору.
- Схвалення **завжди** генерує новий токен; жоден токен ніколи не повертається з
  `node.pair.request`.
- Запити можуть містити `silent: true` як підказку для потоків автоматичного схвалення.
- `node.pair.approve` використовує оголошені команди запиту, що очікує розгляду, щоб застосувати
  додаткові області схвалення:
  - запит без команд: `operator.pairing`
  - запит не-`exec` команд: `operator.pairing` + `operator.write`
  - запит `system.run` / `system.run.prepare` / `system.which`:
    `operator.pairing` + `operator.admin`

Важливо:

- Pairing Node — це потік довіри/ідентичності плюс видача токена.
- Він **не** фіксує поверхню live-команд Node для кожного Node.
- Live-команди Node надходять із того, що Node оголошує під час connect після
  застосування глобальної політики команд Node gateway (`gateway.nodes.allowCommands` /
  `denyCommands`).
- Політика дозволу/запиту для `system.run` на рівні Node зберігається на самому Node в
  `exec.approvals.node.*`, а не в записі pairing.

## Керування командами Node (2026.3.31+)

<Warning>
**Несумісна зміна:** Починаючи з `2026.3.31`, команди Node вимкнені, доки pairing Node не буде схвалено. Лише pairing пристрою більше недостатньо, щоб відкривати оголошені команди Node.
</Warning>

Коли Node підключається вперше, pairing запитується автоматично. Поки запит на pairing не буде схвалено, усі відкладені команди Node від цього Node фільтруються і не виконуються. Щойно довіру встановлено через схвалення pairing, оголошені команди Node стають доступними відповідно до звичайної політики команд.

Це означає:

- Node, які раніше покладалися лише на pairing пристрою для відкриття команд, тепер мають завершити pairing Node.
- Команди, поставлені в чергу до схвалення pairing, відкидаються, а не відкладаються.

## Межі довіри для подій Node (2026.3.31+)

<Warning>
**Несумісна зміна:** Запуски, ініційовані Node, тепер залишаються в межах зменшеної довіреної поверхні.
</Warning>

Підсумки, ініційовані Node, і пов’язані події сесії обмежуються призначеною довіреною поверхнею. Потоки, керовані сповіщеннями або ініційовані Node, які раніше покладалися на ширший доступ до інструментів хоста або сесії, можуть потребувати коригування. Це посилення гарантує, що події Node не можуть підвищити привілеї до доступу до інструментів рівня хоста за межами того, що дозволяє межа довіри Node.

## Автоматичне схвалення (застосунок macOS)

Застосунок macOS за бажанням може спробувати **тихе схвалення**, коли:

- запит позначено як `silent`, і
- застосунок може перевірити SSH-з’єднання з хостом gateway від імені того самого користувача.

Якщо тихе схвалення не вдається, використовується звичайний запит «Схвалити/Відхилити».

## Автоматичне схвалення оновлення метаданих

Коли вже спарений пристрій перепідключається лише зі змінами нечутливих
метаданих (наприклад, відображуване ім’я або підказки щодо платформи клієнта), OpenClaw розглядає
це як `metadata-upgrade`. Тихе автоматичне схвалення є вузьким: воно застосовується лише до довірених локальних перепідключень CLI/допоміжних засобів, які вже довели володіння спільним токеном або паролем через loopback. Клієнти браузера/Control UI та віддалені
клієнти, як і раніше, використовують явний потік повторного схвалення. Оновлення області
(read до write/admin) і зміни відкритого ключа **не** підпадають під автоматичне схвалення `metadata-upgrade`
— для них і далі потрібні явні запити на повторне схвалення.

## Допоміжні засоби QR pairing

`/pair qr` відтворює payload pairing як структуровані медіадані, щоб мобільні та
браузерні клієнти могли сканувати його безпосередньо.

Видалення пристрою також очищає всі застарілі запити pairing, що очікують розгляду, для цього
ідентифікатора пристрою, тому `nodes pending` не показує осиротілі рядки після відкликання.

## Локальність і forwarded headers

Gateway pairing розглядає з’єднання як loopback лише тоді, коли збігаються як дані сирого сокета,
так і будь-які свідчення від проксі вище в ланцюгу. Якщо запит надходить через loopback, але
містить заголовки `X-Forwarded-For` / `X-Forwarded-Host` / `X-Forwarded-Proto`, які
вказують на нелокальне походження, ці forwarded headers спростовують твердження про
локальність loopback. Тоді шлях pairing вимагає явного схвалення замість
тихого трактування запиту як з’єднання з того самого хоста. Еквівалентне правило для
автентифікації оператора див. у [Trusted Proxy Auth](/uk/gateway/trusted-proxy-auth).

## Зберігання (локальне, приватне)

Стан pairing зберігається в каталозі стану Gateway (типово `~/.openclaw`):

- `~/.openclaw/nodes/paired.json`
- `~/.openclaw/nodes/pending.json`

Якщо ви перевизначаєте `OPENCLAW_STATE_DIR`, тека `nodes/` переміщується разом із ним.

Примітки щодо безпеки:

- Токени є секретами; вважайте `paired.json` чутливим файлом.
- Ротація токена вимагає повторного схвалення (або видалення запису Node).

## Поведінка транспорту

- Транспорт є **безстановим**; він не зберігає дані про членство.
- Якщо Gateway офлайн або pairing вимкнено, Node не можуть виконати pairing.
- Якщо Gateway працює у віддаленому режимі, pairing усе одно відбувається відносно сховища віддаленого Gateway.

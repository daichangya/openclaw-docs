OpenClaw зберігає старі контракти сумісності Plugin, підключені через іменовані
адаптери сумісності, перш ніж видаляти їх. Це захищає наявні вбудовані та зовнішні
Plugin, поки розвиваються контракти SDK, маніфесту, налаштування, конфігурації та
середовища виконання агентів.

## Реєстр сумісності

Контракти сумісності Plugin відстежуються в основному реєстрі за адресою
`src/plugins/compat/registry.ts`.

Кожен запис має:

- стабільний код сумісності
- status: `active`, `deprecated`, `removal-pending` або `removed`
- owner: SDK, config, setup, channel, provider, виконання plugin, середовище виконання агента
  або core
- дати запровадження та застарівання, де це застосовно
- вказівки щодо заміни
- docs, diagnostics і tests, які охоплюють стару й нову поведінку

Реєстр є джерелом для планування супроводу та майбутніх перевірок інспектора
Plugin. Якщо змінюється поведінка, видима для Plugin, додайте або оновіть запис
сумісності в тій самій зміні, яка додає адаптер.

## Пакет інспектора Plugin

Інспектор Plugin має розміщуватися поза основним репозиторієм OpenClaw як окремий
пакет/репозиторій, що спирається на версійовані контракти сумісності та
маніфесту.

CLI першого дня має бути таким:

```sh
openclaw-plugin-inspector ./my-plugin
```

Він має виводити:

- валідацію маніфесту/схеми
- версію сумісності контракту, яка перевіряється
- перевірки метаданих встановлення/джерела
- перевірки cold-path import
- попередження про застарівання та сумісність

Використовуйте `--json` для стабільного машинозчитуваного виводу в анотаціях CI. Core OpenClaw
має надавати контракти й фікстури, які може використовувати інспектор, але не
має публікувати бінарник інспектора з основного пакета `openclaw`.

## Політика застарівання

OpenClaw не повинен видаляти задокументований контракт Plugin у тому самому
релізі, у якому з’являється його заміна.

Послідовність міграції така:

1. Додати новий контракт.
2. Зберегти стару поведінку, підключену через іменований адаптер сумісності.
3. Видавати diagnostics або warnings, коли автори Plugin можуть відреагувати.
4. Задокументувати заміну та часову шкалу.
5. Протестувати як старий, так і новий шляхи.
6. Дочекатися завершення оголошеного вікна міграції.
7. Видаляти лише з явним схваленням для релізу з несумісними змінами.

Записи зі статусом deprecated мають містити дату початку попередження, заміну,
посилання на docs і остаточну дату видалення не пізніше ніж через три місяці
після початку попередження. Не додавайте шлях сумісності зі статусом deprecated
з відкритим строком видалення, якщо супровідники явно не вирішили, що це
постійна сумісність, і натомість не позначили її як `active`.

## Поточні області сумісності

Поточні записи сумісності включають:

- застарілі широкі імпорти SDK, такі як `openclaw/plugin-sdk/compat`
- застарілі форми Plugin лише з hooks і `before_agent_start`
- застарілі точки входу Plugin `activate(api)`, поки Plugin мігрують на
  `register(api)`
- застарілі псевдоніми SDK, такі як `openclaw/extension-api`,
  `openclaw/plugin-sdk/channel-runtime`, побудовники status для `openclaw/plugin-sdk/command-auth`,
  `openclaw/plugin-sdk/test-utils` і псевдонім типу `ClawdbotConfig`
- allowlist і поведінка вмикання вбудованих Plugin
- застарілі метадані маніфесту provider/channel для env-var
- застарілі hooks і псевдоніми типів Plugin provider, поки providers переходять на
  явні hooks для catalog, auth, thinking, replay і transport
- застарілі псевдоніми середовища виконання, такі як `api.runtime.taskFlow`,
  `api.runtime.subagent.getSession` і `api.runtime.stt`
- застаріла розділена реєстрація memory-plugin, поки memory plugins переходять на
  `registerMemoryCapability`
- застарілі допоміжні засоби SDK channel для нативних схем повідомлень, контролю
  згадок, форматування вхідних envelope і вкладення можливостей погодження
- підказки активації, які замінюються ownership внесків маніфесту
- резервний механізм середовища виконання `setup-api`, поки дескриптори setup переходять на cold
  метадані `setup.requiresRuntime: false`
- hooks provider `discovery`, поки hooks каталогу provider переходять на
  `catalog.run(...)`
- метадані channel `showConfigured` / `showInSetup`, поки пакети channel переходять
  на `openclaw.channel.exposure`
- застарілі ключі конфігурації політики середовища виконання, поки doctor мігрує операторів на
  `agentRuntime`
- резервний механізм для згенерованих метаданих конфігурації вбудованих channel, поки
  з’являються метадані `channelConfigs` у моделі registry-first
- прапорці env для вимкнення збереженого реєстру Plugin і міграції встановлення, поки
  потоки відновлення мігрують операторів на `openclaw plugins registry --refresh` і
  `openclaw doctor --fix`

Новий код Plugin має надавати перевагу заміні, зазначеній у реєстрі та в
конкретному посібнику з міграції. Наявні Plugin можуть і надалі використовувати
шлях сумісності, доки docs, diagnostics і примітки до релізу не оголосять вікно
видалення.

## Примітки до релізу

Примітки до релізу мають включати майбутні застарівання Plugin із цільовими
датами та посиланнями на docs міграції. Це попередження має з’явитися до того,
як шлях сумісності перейде в `removal-pending` або `removed`.

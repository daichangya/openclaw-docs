**Середовище агента** — це низькорівневий виконавець для одного підготовленого ходу агента OpenClaw. Це не постачальник моделей, не канал і не реєстр інструментів.
Для користувацької ментальної моделі див. [Середовища виконання агента](/uk/concepts/agent-runtimes).

Використовуйте цю поверхню лише для вбудованих або довірених native-плагінів. Контракт
досі є експериментальним, оскільки типи параметрів навмисно віддзеркалюють поточний
вбудований виконавець.

## Коли використовувати harness

Реєструйте середовище агента, коли сімейство моделей має власне native-сеансове
середовище виконання і звичайний транспорт постачальника OpenClaw є хибною абстракцією.

Приклади:

- native-сервер coding-agent, який володіє тредами та Compaction
- локальний CLI або демон, який має стримити native-події plan/reasoning/tool
- середовище виконання моделі, якому потрібен власний resume id на додачу до
  стенограми сеансу OpenClaw

**Не** реєструйте harness лише для додавання нового LLM API. Для звичайних HTTP або
WebSocket API моделей створюйте [плагін постачальника](/uk/plugins/sdk-provider-plugins).

## Чим усе ще володіє ядро

Перш ніж буде вибрано harness, OpenClaw уже розв’язав:

- постачальника і модель
- стан автентифікації середовища виконання
- рівень thinking і бюджет контексту
- файл стенограми/сеансу OpenClaw
- робочий простір, sandbox і політику інструментів
- зворотні виклики відповіді каналу та зворотні виклики стримінгу
- політику fallback моделі та live-перемикання моделей

Такий поділ навмисний. Harness виконує підготовлену спробу; він не вибирає
постачальників, не замінює доставку каналу і не перемикає моделі непомітно.

Підготовлена спроба також містить `params.runtimePlan` — пакет політик, що належить OpenClaw, для рішень середовища виконання, які мають залишатися спільними між PI та native-harness:

- `runtimePlan.tools.normalize(...)` і
  `runtimePlan.tools.logDiagnostics(...)` для політики схеми інструментів з урахуванням постачальника
- `runtimePlan.transcript.resolvePolicy(...)` для політики санітизації стенограми та
  відновлення викликів інструментів
- `runtimePlan.delivery.isSilentPayload(...)` для спільного приглушення доставки `NO_REPLY` і медіа
- `runtimePlan.outcome.classifyRunResult(...)` для класифікації fallback моделі
- `runtimePlan.observability` для розв’язаних метаданих постачальника/моделі/harness

Harness можуть використовувати план для рішень, які мають збігатися з поведінкою PI, але
все одно повинні трактувати його як стан спроби, що належить хосту. Не змінюйте його і не використовуйте для
перемикання постачальників/моделей усередині ходу.

## Зареєструвати harness

**Імпорт:** `openclaw/plugin-sdk/agent-harness`

```typescript
import type { AgentHarness } from "openclaw/plugin-sdk/agent-harness";
import { definePluginEntry } from "openclaw/plugin-sdk/plugin-entry";

const myHarness: AgentHarness = {
  id: "my-harness",
  label: "Моє native-середовище агента",

  supports(ctx) {
    return ctx.provider === "my-provider"
      ? { supported: true, priority: 100 }
      : { supported: false };
  },

  async runAttempt(params) {
    // Запустіть або відновіть ваш native-тред.
    // Використовуйте params.prompt, params.tools, params.images, params.onPartialReply,
    // params.onAgentEvent та інші поля підготовленої спроби.
    return await runMyNativeTurn(params);
  },
};

export default definePluginEntry({
  id: "my-native-agent",
  name: "Мій Native Agent",
  description: "Запускає вибрані моделі через native-демон агента.",
  register(api) {
    api.registerAgentHarness(myHarness);
  },
});
```

## Політика вибору

OpenClaw вибирає harness після розв’язання постачальника/моделі:

1. Записаний id harness наявного сеансу має перевагу, тож зміни config/env не
   перемикають цю стенограму на інше середовище виконання «на гарячу».
2. `OPENCLAW_AGENT_RUNTIME=<id>` примусово вибирає зареєстрований harness з цим id для
   сеансів, які ще не закріплені.
3. `OPENCLAW_AGENT_RUNTIME=pi` примусово вибирає вбудований harness PI.
4. `OPENCLAW_AGENT_RUNTIME=auto` запитує в зареєстрованих harness, чи підтримують вони розв’язану
   пару постачальник/модель.
5. Якщо жоден зареєстрований harness не підходить, OpenClaw використовує PI, якщо fallback PI
   не вимкнено.

Помилки harness плагіна відображаються як помилки виконання. У режимі `auto` fallback на PI
використовується лише тоді, коли жоден зареєстрований harness плагіна не підтримує розв’язану
пару постачальник/модель. Щойно harness плагіна заявив про виконання, OpenClaw не
повторює той самий хід через PI, оскільки це може змінити семантику автентифікації/середовища виконання
або дублювати побічні ефекти.

Вибраний id harness зберігається разом з id сеансу після вбудованого запуску.
Застарілі сеанси, створені до появи закріплення harness, вважаються закріпленими за PI, щойно в них
з’являється історія стенограми. Використовуйте новий/скинутий сеанс, коли перемикаєтеся між PI та
native-harness плагіна. `/status` показує нестандартні id harness, як-от `codex`,
поруч із `Fast`; PI приховано, оскільки це типовий шлях сумісності.
Якщо вибраний harness виглядає несподіваним, увімкніть журналювання налагодження `agents/harness` і
перегляньте структурований запис gateway `agent harness selected`. Він містить
вибраний id harness, причину вибору, політику runtime/fallback і, у режимі
`auto`, результат підтримки для кожного кандидата-плагіна.

Вбудований плагін Codex реєструє `codex` як свій id harness. Ядро трактує це як
звичайний id harness плагіна; специфічні для Codex аліаси мають належати плагіну
або конфігурації оператора, а не спільному селектору середовища виконання.

## Поєднання постачальника й harness

Більшість harness також мають реєструвати постачальника. Постачальник робить посилання на моделі,
статус автентифікації, метадані моделі та вибір `/model` видимими для решти
OpenClaw. Потім harness заявляє цей постачальник у `supports(...)`.

Вбудований плагін Codex дотримується цієї схеми:

- бажані користувацькі посилання на моделі: `openai/gpt-5.5` плюс
  `agentRuntime.id: "codex"`
- посилання сумісності: застарілі посилання `codex/gpt-*` усе ще приймаються, але нові
  конфігурації не повинні використовувати їх як звичайні посилання постачальник/модель
- id harness: `codex`
- auth: синтетична доступність постачальника, оскільки harness Codex володіє
  native-входом/сеансом Codex
- запит app-server: OpenClaw надсилає до Codex чистий id моделі і дозволяє
  harness спілкуватися з native-протоколом app-server

Плагін Codex є адитивним. Звичайні посилання `openai/gpt-*` і далі використовують
звичайний шлях постачальника OpenClaw, якщо ви не примусите harness Codex через
`agentRuntime.id: "codex"`. Старіші посилання `codex/gpt-*` для сумісності все ще вибирають
постачальника і harness Codex.

Для налаштування оператором, прикладів префіксів моделей і конфігурацій лише для Codex див.
[Harness Codex](/uk/plugins/codex-harness).

OpenClaw вимагає Codex app-server версії `0.125.0` або новішої. Плагін Codex перевіряє
initialize-handshake app-server і блокує старіші або неверсіоновані сервери, щоб
OpenClaw працював лише з тією поверхнею протоколу, на якій його було протестовано. Поріг
`0.125.0` включає підтримку native payload hook MCP, що з’явилася в Codex `0.124.0`,
водночас прив’язуючи OpenClaw до новішої протестованої стабільної гілки.

### Проміжний шар результатів інструментів

Вбудовані плагіни можуть підключати нейтральний до середовища виконання проміжний шар результатів інструментів через
`api.registerAgentToolResultMiddleware(...)`, коли їхній маніфест оголошує
цільові id середовищ виконання в `contracts.agentToolResultMiddleware`. Цей довірений
шов призначено для асинхронних перетворень результатів інструментів, які мають виконуватися до того, як PI або Codex подадуть
вивід інструмента назад у модель.

Застарілі вбудовані плагіни все ще можуть використовувати
`api.registerCodexAppServerExtensionFactory(...)` для проміжного шару лише для Codex app-server,
але нові перетворення результатів мають використовувати нейтральний до середовища виконання API.
Хук лише для Pi `api.registerEmbeddedExtensionFactory(...)` було видалено;
перетворення результатів інструментів Pi мають використовувати нейтральний проміжний шар.

### Класифікація термінального результату

Native-harness, які володіють власною проєкцією протоколу, можуть використовувати
`classifyAgentHarnessTerminalOutcome(...)` з
`openclaw/plugin-sdk/agent-harness-runtime`, коли завершений хід не породив
видимого тексту помічника. Допоміжна функція повертає `empty`, `reasoning-only` або
`planning-only`, щоб політика fallback OpenClaw могла вирішити, чи повторювати спробу на
іншій моделі. Вона навмисно залишає некласифікованими помилки prompt, ходи в процесі виконання
та навмисно тихі відповіді, такі як `NO_REPLY`.

### Режим native-harness Codex

Вбудований harness `codex` — це native-режим Codex для вбудованих ходів
агента OpenClaw. Спочатку увімкніть вбудований плагін `codex` і додайте `codex` до
`plugins.allow`, якщо у вашій конфігурації використовується обмежувальний allowlist. Конфігурації native app-server
мають використовувати `openai/gpt-*` з `agentRuntime.id: "codex"`.
Натомість використовуйте `openai-codex/*` для Codex OAuth через PI. Застарілі посилання моделей `codex/*`
залишаються аліасами сумісності для native-harness.

Коли цей режим працює, Codex володіє native id треду, поведінкою resume,
Compaction і виконанням app-server. OpenClaw і далі володіє chat-каналом,
дзеркалом видимої стенограми, політикою інструментів, погодженнями, доставкою медіа та
вибором сеансу. Використовуйте `agentRuntime.id: "codex"` без перевизначення `fallback`,
коли вам потрібно довести, що лише шлях app-server Codex може заявити це виконання.
Явні runtime плагінів уже типово завершуються fail-closed. Установлюйте `fallback: "pi"`
лише тоді, коли ви навмисно хочете, щоб PI обробляв відсутність вибору harness. Помилки Codex
app-server уже завершуються безпосередньо, а не повторюються через PI.

## Вимкнення fallback на PI

Типово OpenClaw запускає вбудованих агентів з `agents.defaults.agentRuntime`,
встановленим у `{ id: "auto", fallback: "pi" }`. У режимі `auto` зареєстровані harness
плагінів можуть заявляти пару постачальник/модель. Якщо жоден не підходить, OpenClaw
повертається до PI.

У режимі `auto` встановлюйте `fallback: "none"`, коли вам потрібно, щоб відсутність вибору harness плагіна
призводила до помилки, а не до використання PI. Явні runtime плагінів, як-от
`runtime: "codex"`, уже типово завершуються fail-closed, якщо лише `fallback: "pi"` не
задано в тій самій конфігурації або області перевизначення середовища. Помилки вибраного harness плагіна
завжди є жорсткими збоями. Це не блокує явний `runtime: "pi"` або
`OPENCLAW_AGENT_RUNTIME=pi`.

Для вбудованих запусків лише з Codex:

```json
{
  "agents": {
    "defaults": {
      "model": "openai/gpt-5.5",
      "agentRuntime": {
        "id": "codex"
      }
    }
  }
}
```

Якщо ви хочете, щоб будь-який зареєстрований harness плагіна заявляв відповідні моделі, але ніколи
не хочете, щоб OpenClaw непомітно повертався до PI, залишайте `runtime: "auto"` і вимикайте
fallback:

```json
{
  "agents": {
    "defaults": {
      "agentRuntime": {
        "id": "auto",
        "fallback": "none"
      }
    }
  }
}
```

Перевизначення для окремих агентів використовують ту саму форму:

```json
{
  "agents": {
    "defaults": {
      "agentRuntime": {
        "id": "auto",
        "fallback": "pi"
      }
    },
    "list": [
      {
        "id": "codex-only",
        "model": "openai/gpt-5.5",
        "agentRuntime": {
          "id": "codex",
          "fallback": "none"
        }
      }
    ]
  }
}
```

`OPENCLAW_AGENT_RUNTIME` і далі перевизначає налаштоване середовище виконання. Використовуйте
`OPENCLAW_AGENT_HARNESS_FALLBACK=none`, щоб вимкнути fallback на PI з
середовища.

```bash
OPENCLAW_AGENT_RUNTIME=codex \
OPENCLAW_AGENT_HARNESS_FALLBACK=none \
openclaw gateway run
```

Коли fallback вимкнено, сеанс завершується рано з помилкою, якщо запитаний harness не
зареєстровано, він не підтримує розв’язану пару постачальник/модель або завершується помилкою до
породження побічних ефектів ходу. Це навмисно для розгортань лише з Codex і
для live-тестів, які мають довести, що шлях app-server Codex справді використовується.

Це налаштування керує лише вбудованим harness агента. Воно не вимикає
маршрутизацію зображень, відео, музики, TTS, PDF або іншого маршрутизування моделей, специфічного для постачальника.

## Native-сеанси та дзеркало стенограми

Harness може зберігати native id сеансу, id треду або daemon-side токен відновлення.
Явно зберігайте цей зв’язок асоційованим із сеансом OpenClaw і продовжуйте
дзеркалити видимий користувачу вивід помічника/інструментів у стенограму OpenClaw.

Стенограма OpenClaw залишається шаром сумісності для:

- видимої в каналі історії сеансу
- пошуку та індексації стенограми
- перемикання назад до вбудованого harness PI на пізнішому ході
- загальної поведінки `/new`, `/reset` і видалення сеансу

Якщо ваш harness зберігає sidecar-зв’язування, реалізуйте `reset(...)`, щоб OpenClaw міг
очистити його, коли пов’язаний сеанс OpenClaw скидається.

## Результати інструментів і медіа

Ядро формує список інструментів OpenClaw і передає його в підготовлену спробу.
Коли harness виконує динамічний виклик інструмента, повертайте результат інструмента
назад через форму результату harness замість того, щоб самостійно надсилати медіа в канал.

Це зберігає текст, зображення, відео, музику, TTS, погодження та виводи
інструментів обміну повідомленнями на тому самому шляху доставки, що й у запусків на основі PI.

## Поточні обмеження

- Публічний шлях імпорту є загальним, але деякі псевдоніми типів спроб/результатів досі
  містять назви `Pi` для сумісності.
- Установлення сторонніх harness є експериментальним. Віддавайте перевагу плагінам постачальників,
  доки вам не знадобиться native-сеансове середовище виконання.
- Перемикання harness підтримується між ходами. Не перемикайте harness посеред ходу після того, як уже почалися native-інструменти, погодження, текст помічника або надсилання повідомлень.

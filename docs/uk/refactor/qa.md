# Рефакторинг QA

Статус: базову міграцію завершено.

## Мета

Перевести QA в OpenClaw із моделі розділених визначень до єдиного джерела істини:

- метадані сценарію
- prompts, що надсилаються моделі
- налаштування та завершення
- логіка тестового каркаса
- перевірки та критерії успіху
- артефакти та підказки для звітів

Бажаний кінцевий стан — універсальний тестовий каркас QA, який завантажує потужні файли визначень сценаріїв замість жорсткого кодування більшості поведінки в TypeScript.

## Поточний стан

Основне джерело істини тепер міститься в `qa/scenarios/index.md` плюс по одному файлу на
сценарій у `qa/scenarios/<theme>/*.md`.

Реалізовано:

- `qa/scenarios/index.md`
  - канонічні метадані пакета QA
  - ідентичність оператора
  - стартова місія
- `qa/scenarios/<theme>/*.md`
  - один markdown-файл на сценарій
  - метадані сценарію
  - прив’язки обробників
  - конфігурація виконання, специфічна для сценарію
- `extensions/qa-lab/src/scenario-catalog.ts`
  - markdown-парсер пакета + валідація zod
- `extensions/qa-lab/src/qa-agent-bootstrap.ts`
  - рендеринг плану з markdown-пакета
- `extensions/qa-lab/src/qa-agent-workspace.ts`
  - початкові згенеровані файли сумісності плюс `QA_SCENARIOS.md`
- `extensions/qa-lab/src/suite.ts`
  - вибирає виконувані сценарії через прив’язки обробників, визначені в markdown
- Протокол шини QA + UI
  - універсальні inline-вкладення для рендерингу зображень/відео/аудіо/файлів

Поверхні, які ще залишаються розділеними:

- `extensions/qa-lab/src/suite.ts`
  - досі містить більшість виконуваної логіки користувацьких обробників
- `extensions/qa-lab/src/report.ts`
  - досі виводить структуру звіту з результатів runtime

Тож поділ джерела істини виправлено, але виконання все ще здебільшого спирається на обробники, а не є повністю декларативним.

## Як виглядає реальна поверхня сценаріїв

Читання поточного suite показує кілька різних класів сценаріїв.

### Проста взаємодія

- базовий сценарій каналу
- базовий сценарій DM
- подальша дія в треді
- перемикання моделі
- завершення після підтвердження
- реакція/редагування/видалення

### Мутація конфігурації та runtime

- вимкнення skill через patch конфігурації
- пробудження після перезапуску через config apply
- перемикання можливостей через перезапуск конфігурації
- перевірка дрейфу inventory у runtime

### Перевірки файлової системи та репозиторію

- звіт про виявлення source/docs
- збирання Lobster Invaders
- пошук артефакта згенерованого зображення

### Оркестрація пам’яті

- виклик пам’яті
- інструменти пам’яті в контексті каналу
- резервний сценарій при збої пам’яті
- ранжування пам’яті сеансу
- ізоляція пам’яті треду
- прохід Dreaming пам’яті

### Інтеграція інструментів і Plugin

- виклик plugin-tools MCP
- видимість skill
- гаряче встановлення skill
- нативна генерація зображень
- повний цикл обробки зображення
- розуміння зображення з вкладення

### Багатоходові та багатосторонні сценарії

- передавання субагенту
- fanout-синтез субагента
- потоки стилю відновлення після перезапуску

Ці категорії важливі, бо вони визначають вимоги до DSL. Простого списку prompt + очікуваний текст недостатньо.

## Напрямок

### Єдине джерело істини

Використовувати `qa/scenarios/index.md` плюс `qa/scenarios/<theme>/*.md` як
авторське джерело істини.

Пакет має залишатися:

- читабельним для людини під час рев’ю
- придатним до машинного розбору
- достатньо багатим, щоб керувати:
  - виконанням suite
  - bootstrap робочого простору QA
  - метаданими UI QA Lab
  - prompts для docs/discovery
  - генерацією звітів

### Бажаний формат авторства

Використовувати markdown як формат верхнього рівня зі структурованим YAML усередині нього.

Рекомендована форма:

- YAML frontmatter
  - id
  - title
  - surface
  - tags
  - refs документації
  - refs коду
  - перевизначення моделей/провайдерів
  - передумови
- текстові розділи
  - objective
  - notes
  - debugging hints
- блоки YAML в огорожах
  - setup
  - steps
  - assertions
  - cleanup

Це дає:

- кращу читабельність PR, ніж гігантський JSON
- багатший контекст, ніж чистий YAML
- строгий парсинг і валідацію zod

Сирий JSON прийнятний лише як проміжна згенерована форма.

````md
# Мета

Перевірити, що згенерований медіаконтент знову прикріплюється на наступному ході.

# Налаштування

```yaml scenario.setup
- action: config.patch
  patch:
    agents:
      defaults:
        imageGenerationModel:
          primary: openai/gpt-image-1
- action: session.create
  key: agent:qa:image-roundtrip
```

# Кроки

```yaml scenario.steps
- action: agent.send
  session: agent:qa:image-roundtrip
  message: |
    Перевірка генерації зображення: згенеруй QA-зображення маяка і підсумуй його одним коротким реченням.
- action: artifact.capture
  kind: generated-image
  promptSnippet: Перевірка генерації зображення
  saveAs: lighthouseImage
- action: agent.send
  session: agent:qa:image-roundtrip
  message: |
    Перевірка огляду зображення після повного циклу: опиши вкладення зі згенерованим маяком одним коротким реченням.
  attachments:
    - fromArtifact: lighthouseImage
```

# Очікування

```yaml scenario.expect
- assert: outbound.textIncludes
  value: lighthouse
- assert: requestLog.matches
  where:
    promptIncludes: Перевірка огляду зображення після повного циклу
  imageInputCountGte: 1
- assert: artifact.exists
  ref: lighthouseImage
```
````

## Змінні та посилання на артефакти

DSL має підтримувати збережені виводи та подальші посилання на них.

Приклади з поточного suite:

- створити тред, а потім повторно використати `threadId`
- створити сеанс, а потім повторно використати `sessionKey`
- згенерувати зображення, а потім прикріпити файл на наступному ході
- згенерувати рядок-маркер пробудження, а потім перевірити, що він з’явиться пізніше

Потрібні можливості:

- `saveAs`
- `${vars.name}`
- `${artifacts.name}`
- типізовані посилання для шляхів, ключів сеансу, ідентифікаторів тредів, маркерів, виводів інструментів

Без підтримки змінних тестовий каркас і далі повертатиме логіку сценаріїв назад у TypeScript.

## Що має залишитися як аварійні механізми

Повністю чистий декларативний виконавець нереалістичний на фазі 1.

Деякі сценарії за своєю природою потребують складної оркестрації:

- прохід Dreaming пам’яті
- пробудження після перезапуску через config apply
- перемикання можливостей через перезапуск конфігурації
- визначення артефакта згенерованого зображення за часовою міткою/шляхом
- оцінювання discovery-report

Поки що для них слід використовувати явні користувацькі обробники.

Рекомендоване правило:

- 85-90% декларативно
- явні кроки `customHandler` для складного залишку
- лише іменовані й задокументовані користувацькі обробники
- жодного анонімного inline-коду у файлі сценарію

Це збереже універсальний рушій чистим і водночас дозволить рухатися вперед.

## Підтримка фальшивого Slack / насичених медіа

Поточна шина QA орієнтована насамперед на текст.

Пов’язані файли:

- `extensions/qa-channel/src/protocol.ts`
- `extensions/qa-lab/src/bus-state.ts`
- `extensions/qa-lab/src/bus-queries.ts`
- `extensions/qa-lab/src/bus-server.ts`
- `extensions/qa-lab/web/src/ui-render.ts`

Сьогодні шина QA підтримує:

- текст
- реакції
- треди

Вона ще не моделює inline-медіавкладення.

### Потрібний транспортний контракт

Додати універсальну модель вкладень шини QA:

```ts
type QaBusAttachment = {
  id: string;
  kind: "image" | "video" | "audio" | "file";
  mimeType: string;
  fileName?: string;
  inline?: boolean;
  url?: string;
  contentBase64?: string;
  width?: number;
  height?: number;
  durationMs?: number;
  altText?: string;
  transcript?: string;
};
```

Потім додати `attachments?: QaBusAttachment[]` до:

- `QaBusMessage`
- `QaBusInboundMessageInput`
- `QaBusOutboundMessageInput`

### Чому спочатку універсально

Не створювати модель медіа лише для Slack.

Натомість:

- одна універсальна транспортна модель QA
- кілька рендерерів поверх неї
  - поточний чат QA Lab
  - майбутній веб-фейк Slack
  - будь-які інші фейкові транспортні подання

Це запобігає дублюванню логіки й дозволяє медіасценаріям залишатися транспортно-агностичними.

### Потрібна робота в UI

Оновити UI QA для рендерингу:

- inline-попереднього перегляду зображення
- inline-аудіопрогравача
- inline-відеопрогравача
- chip вкладеного файла

Поточний UI уже може рендерити треди й реакції, тож рендеринг вкладень має нашаровуватися на ту саму модель картки повідомлення.

### Робота над сценаріями, яку відкриває медіатранспорт

Щойно вкладення підуть через шину QA, можна буде додати багатші сценарії фейкового чату:

- inline-відповідь із зображенням у фейковому Slack
- розуміння аудіовкладення
- розуміння відеовкладення
- змішаний порядок вкладень
- відповідь у треді зі збереженням медіа

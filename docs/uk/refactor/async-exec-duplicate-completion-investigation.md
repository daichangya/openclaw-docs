## Область

- Сесія: `agent:main:telegram:group:-1003774691294:topic:1`
- Симптом: те саме асинхронне завершення exec для сесії/запуску `keen-nexus` було двічі записане в LCM як ходи користувача.
- Мета: визначити, що найімовірніше сталося — дубльоване ін’єктування сесії чи звичайний повтор вихідної доставки.

## Висновок

Найімовірніше, це **дубльоване ін’єктування сесії**, а не простий повтор вихідної доставки.

Найсильніша прогалина з боку gateway — у **шляху завершення node exec**:

1. Завершення exec на боці вузла випромінює `exec.finished` із повним `runId`.
2. Gateway `server-node-events` перетворює це на системну подію та запитує Heartbeat.
3. Запуск Heartbeat ін’єктує злитий блок системних подій у підказку агента.
4. Вбудований runner зберігає цю підказку як новий хід користувача в транскрипті сесії.

Якщо той самий `exec.finished` з будь-якої причини (відтворення, дубль після перепідключення, повторне надсилання зверху, дубльований продуцент) двічі потрапляє в gateway для того самого `runId`, OpenClaw наразі **не має перевірки ідемпотентності за ключем `runId`/`contextKey`** на цьому шляху. Друга копія стане другим повідомленням користувача з тим самим вмістом.

## Точний шлях коду

### 1. Продуцент: подія завершення node exec

- `src/node-host/invoke.ts:340-360`
  - `sendExecFinishedEvent(...)` випромінює `node.event` з подією `exec.finished`.
  - Корисне навантаження містить `sessionKey` і повний `runId`.

### 2. Приймання події gateway

- `src/gateway/server-node-events.ts:574-640`
  - Обробляє `exec.finished`.
  - Формує текст:
    - `Exec finished (node=..., id=<runId>, code ...)`
  - Ставить його в чергу через:
    - `enqueueSystemEvent(text, { sessionKey, contextKey: runId ? \`exec:${runId}\` : "exec", trusted: false })`
  - Негайно запитує пробудження:
    - `requestHeartbeatNow(scopedHeartbeatWakeOptions(sessionKey, { reason: "exec-event" }))`

### 3. Слабке місце дедуплікації системних подій

- `src/infra/system-events.ts:90-115`
  - `enqueueSystemEvent(...)` пригнічує лише **послідовні дублікати тексту**:
    - `if (entry.lastText === cleaned) return false`
  - Він зберігає `contextKey`, але **не** використовує `contextKey` для ідемпотентності.
  - Після зливу черги пригнічення дублікатів скидається.

Це означає, що повторно відтворений `exec.finished` з тим самим `runId` може бути знову прийнятий пізніше, хоча код уже мав стабільного кандидата для ідемпотентності (`exec:<runId>`).

### 4. Обробка пробудження — не основне джерело дублювання

- `src/infra/heartbeat-wake.ts:79-117`
  - Пробудження коалесцюються за `(agentId, sessionKey)`.
  - Дубльовані запити на пробудження для тієї самої цілі згортаються в один запис очікування.

Через це **саме дубльована обробка пробудження** є слабшим поясненням, ніж дубльоване приймання подій.

### 5. Heartbeat споживає подію й перетворює її на вхід підказки

- `src/infra/heartbeat-runner.ts:535-574`
  - Попередня перевірка переглядає очікувані системні події та класифікує запуски exec-event.
- `src/auto-reply/reply/session-system-events.ts:86-90`
  - `drainFormattedSystemEvents(...)` зливає чергу для сесії.
- `src/auto-reply/reply/get-reply-run.ts:400-427`
  - Злитий блок системних подій додається на початок тіла підказки агента.

### 6. Точка ін’єкції транскрипту

- `src/agents/pi-embedded-runner/run/attempt.ts:2000-2017`
  - `activeSession.prompt(effectivePrompt)` надсилає повну підказку до вбудованої сесії PI.
  - Саме в цій точці підказка, похідна від завершення, стає збереженим ходом користувача.

Отже, щойно ту саму системну подію двічі перебудовано в підказку, дубльовані повідомлення користувача в LCM є очікуваними.

## Чому простий повтор вихідної доставки менш імовірний

У runner Heartbeat справді є шлях помилки вихідної доставки:

- `src/infra/heartbeat-runner.ts:1194-1242`
  - Спочатку генерується відповідь.
  - Вихідна доставка відбувається пізніше через `deliverOutboundPayloads(...)`.
  - Збій там повертає `{ status: "failed" }`.

Однак для того самого запису черги системної події цього **недостатньо**, щоб пояснити дубльовані ходи користувача:

- `src/auto-reply/reply/session-system-events.ts:86-90`
  - Черга системних подій уже зливається до вихідної доставки.

Тобто сам по собі повтор надсилання каналу не відтворить точно той самий елемент черги. Це може пояснити відсутню/невдалу зовнішню доставку, але не друге ідентичне повідомлення користувача в сесії.

## Вторинна, менш упевнена можливість

У runner агента є повний цикл повторного запуску:

- `src/auto-reply/reply/agent-runner-execution.ts:741-1473`
  - Певні тимчасові збої можуть повторно запустити весь цикл і повторно надіслати те саме `commandBody`.

Це може продублювати збережену підказку користувача **в межах того самого виконання відповіді**, якщо підказку вже було додано до моменту спрацювання умови повтору.

Я оцінюю це нижче, ніж дубльоване приймання `exec.finished`, тому що:

- спостережуваний розрив був близько 51 секунди, що більше схоже на друге пробудження/хід, ніж на повтор усередині процесу;
- у звіті вже згадуються повторні збої надсилання повідомлень, що більше вказує на окремий пізніший хід, ніж на негайний повтор моделі/часу виконання.

## Гіпотеза першопричини

Гіпотеза з найвищою впевненістю:

- завершення `keen-nexus` пройшло через **шлях подій node exec**;
- той самий `exec.finished` було доставлено в `server-node-events` двічі;
- Gateway прийняв обидва, тому що `enqueueSystemEvent(...)` не виконує дедуплікацію за `contextKey` / `runId`;
- кожна прийнята подія спричинила Heartbeat і була ін’єктована як хід користувача в транскрипт PI.

## Запропоноване невелике точкове виправлення

Якщо потрібне виправлення, найменша зміна з високою цінністю така:

- зробити ідемпотентність exec/системних подій такою, щоб вона враховувала `contextKey` протягом короткого горизонту, принаймні для точних повторів `(sessionKey, contextKey, text)`;
- або додати окрему дедуплікацію в `server-node-events` для `exec.finished` за ключем `(sessionKey, runId, kind події)`.

Це безпосередньо блокує повторні дублікати `exec.finished` до того, як вони перетворяться на ходи сесії.

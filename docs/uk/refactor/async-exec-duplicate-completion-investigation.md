# Розслідування дубльованого завершення Async Exec

## Обсяг

- Сесія: `agent:main:telegram:group:-1003774691294:topic:1`
- Симптом: те саме async exec completion для сесії/запуску `keen-nexus` було двічі записано в LCM як ходи користувача.
- Мета: визначити, що є найімовірнішою причиною — дубльована ін’єкція в сесію чи звичайна повторна спроба вихідної доставки.

## Висновок

Найімовірніше, це **дубльована ін’єкція в сесію**, а не просто повторна спроба вихідної доставки.

Найсильніший пробіл на боці Gateway — у **шляху завершення node exec**:

1. Завершення exec на боці Node надсилає `exec.finished` із повним `runId`.
2. Gateway `server-node-events` перетворює це на системну подію й запитує Heartbeat.
3. Запуск Heartbeat ін’єктує осушений блок системної події в промпт агента.
4. Вбудований runner зберігає цей промпт як новий хід користувача в транскрипті сесії.

Якщо той самий `exec.finished` із тим самим `runId` з будь-якої причини двічі потрапляє до Gateway (повторне відтворення, дубль при перепідключенні, повторне надсилання зверху, дубльований producer), OpenClaw наразі **не має перевірки ідемпотентності за ключем `runId`/`contextKey`** на цьому шляху. Друга копія перетвориться на друге користувацьке повідомлення з тим самим вмістом.

## Точний шлях у коді

### 1. Producer: подія завершення node exec

- `src/node-host/invoke.ts:340-360`
  - `sendExecFinishedEvent(...)` надсилає `node.event` з подією `exec.finished`.
  - Payload містить `sessionKey` і повний `runId`.

### 2. Індексація події в Gateway

- `src/gateway/server-node-events.ts:574-640`
  - Обробляє `exec.finished`.
  - Формує текст:
    - `Exec finished (node=..., id=<runId>, code ...)`
  - Ставить його в чергу через:
    - `enqueueSystemEvent(text, { sessionKey, contextKey: runId ? \`exec:${runId}\` : "exec", trusted: false })`
  - Одразу запитує пробудження:
    - `requestHeartbeatNow(scopedHeartbeatWakeOptions(sessionKey, { reason: "exec-event" }))`

### 3. Слабке місце дедуплікації системних подій

- `src/infra/system-events.ts:90-115`
  - `enqueueSystemEvent(...)` пригнічує лише **послідовні дублікати тексту**:
    - `if (entry.lastText === cleaned) return false`
  - Він зберігає `contextKey`, але **не** використовує `contextKey` для ідемпотентності.
  - Після осушення пригнічення дублікатів скидається.

Це означає, що повторно доставлений `exec.finished` з тим самим `runId` може бути знову прийнятий пізніше, хоча код уже мав стабільного кандидата для ідемпотентності (`exec:<runId>`).

### 4. Обробка пробудження — не основне джерело дублювання

- `src/infra/heartbeat-wake.ts:79-117`
  - Пробудження об’єднуються за `(agentId, sessionKey)`.
  - Дубльовані запити пробудження для тієї самої цілі згортаються в один запис очікування пробудження.

Через це **саме дублювання обробки пробуджень** є слабшим поясненням, ніж дубльована індексація події.

### 5. Heartbeat споживає подію й перетворює її на вхід промпта

- `src/infra/heartbeat-runner.ts:535-574`
  - На етапі preflight переглядає очікувані системні події й класифікує запуски exec-event.
- `src/auto-reply/reply/session-system-events.ts:86-90`
  - `drainFormattedSystemEvents(...)` осушує чергу для сесії.
- `src/auto-reply/reply/get-reply-run.ts:400-427`
  - Осушений блок системної події додається на початок тіла промпта агента.

### 6. Точка ін’єкції в транскрипт

- `src/agents/pi-embedded-runner/run/attempt.ts:2000-2017`
  - `activeSession.prompt(effectivePrompt)` надсилає повний промпт до вбудованої сесії PI.
  - Саме в цій точці промпт, сформований із completion, стає збереженим ходом користувача.

Тож щойно ту саму системну подію двічі вбудовано в промпт, поява дубльованих користувацьких повідомлень LCM є очікуваною.

## Чому проста повторна спроба вихідної доставки менш імовірна

У runner Heartbeat справді існує шлях збоїв вихідної доставки:

- `src/infra/heartbeat-runner.ts:1194-1242`
  - Відповідь спочатку генерується.
  - Вихідна доставка відбувається пізніше через `deliverOutboundPayloads(...)`.
  - Збій там повертає `{ status: "failed" }`.

Однак для того самого запису черги системної події цього **недостатньо**, щоб пояснити дубльовані ходи користувача:

- `src/auto-reply/reply/session-system-events.ts:86-90`
  - Черга системної події вже осушена до вихідної доставки.

Отже, проста повторна спроба надсилання каналом сама по собі не відтворила б ту саму queued-подію. Вона могла б пояснити відсутню/невдалу зовнішню доставку, але не друге ідентичне користувацьке повідомлення в сесії.

## Вторинна, менш упевнена можливість

У runner агента є повний цикл повторного запуску:

- `src/auto-reply/reply/agent-runner-execution.ts:741-1473`
  - Деякі тимчасові збої можуть повторити весь запуск і повторно надіслати те саме `commandBody`.

Це може дублювати збережений користувацький промпт **у межах того самого виконання відповіді**, якщо промпт уже було додано до виникнення умови повтору.

Я ставлю цей варіант нижче, ніж дубльовану індексацію `exec.finished`, тому що:

- спостережуваний розрив становив близько 51 секунди, що більше схоже на друге пробудження/хід, ніж на повтор усередині процесу;
- у звіті вже згадано повторні збої надсилання повідомлень, що радше вказує на окремий пізніший хід, ніж на негайний повтор моделі/runtime.

## Гіпотеза першопричини

Гіпотеза з найвищою впевненістю:

- Завершення `keen-nexus` пройшло через **шлях подій node exec**.
- Той самий `exec.finished` було двічі доставлено в `server-node-events`.
- Gateway прийняв обидві події, тому що `enqueueSystemEvent(...)` не виконує дедуплікацію за `contextKey` / `runId`.
- Кожна прийнята подія запускала Heartbeat і була ін’єктована як хід користувача в транскрипт PI.

## Запропоноване маленьке точкове виправлення

Якщо потрібне виправлення, найменша зміна з високою цінністю така:

- зробити так, щоб ідемпотентність exec/системних подій враховувала `contextKey` протягом короткого горизонту, принаймні для точних повторів `(sessionKey, contextKey, text)`;
- або додати окрему дедуплікацію в `server-node-events` для `exec.finished` за ключем `(sessionKey, runId, event kind)`.

Це безпосередньо блокуватиме повторно доставлені дублікати `exec.finished` до того, як вони перетворяться на ходи сесії.

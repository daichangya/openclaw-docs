## Цілі

- Повторювати спробу для кожного HTTP-запиту, а не для всього багатокрокового потоку.
- Зберігати порядок, повторюючи лише поточний крок.
- Уникати дублювання неідемпотентних операцій.

## Значення за замовчуванням

- Кількість спроб: 3
- Максимальна межа затримки: 30000 мс
- Jitter: 0.1 (10 відсотків)
- Значення provider за замовчуванням:
  - Мінімальна затримка Telegram: 400 мс
  - Мінімальна затримка Discord: 500 мс

## Поведінка

### Provider моделей

- OpenClaw дозволяє SDK provider обробляти звичайні короткі повторні спроби.
- Для SDK на базі Stainless, таких як Anthropic і OpenAI, відповіді, придатні до повторної спроби
  (`408`, `409`, `429` і `5xx`), можуть містити `retry-after-ms` або
  `retry-after`. Якщо це очікування довше за 60 секунд, OpenClaw додає
  `x-should-retry: false`, щоб SDK негайно повернув помилку, а failover моделей міг переключитися на інший auth profile або резервну модель.
- Перевизначте межу через `OPENCLAW_SDK_RETRY_MAX_WAIT_SECONDS=<seconds>`.
  Установіть `0`, `false`, `off`, `none` або `disabled`, щоб дозволити SDK внутрішньо дотримуватися довгих
  затримок `Retry-After`.

### Discord

- Повторює спробу лише для помилок обмеження частоти (HTTP 429).
- Використовує `retry_after` Discord, коли доступно, інакше — експоненційне збільшення затримки.

### Telegram

- Повторює спробу для тимчасових помилок (429, timeout, connect/reset/closed, тимчасово недоступно).
- Використовує `retry_after`, коли доступно, інакше — експоненційне збільшення затримки.
- Помилки розбору Markdown не повторюються; замість цього використовується звичайний текст.

## Конфігурація

Налаштуйте політику повторних спроб для кожного provider у `~/.openclaw/openclaw.json`:

```json5
{
  channels: {
    telegram: {
      retry: {
        attempts: 3,
        minDelayMs: 400,
        maxDelayMs: 30000,
        jitter: 0.1,
      },
    },
    discord: {
      retry: {
        attempts: 3,
        minDelayMs: 500,
        maxDelayMs: 30000,
        jitter: 0.1,
      },
    },
  },
}
```

## Примітки

- Повторні спроби застосовуються до кожного запиту окремо (надсилання повідомлення, завантаження медіа, реакція, опитування, стікер).
- Для складених потоків завершені кроки не повторюються.

Кожна модель має вікно контексту: максимальну кількість токенів, яку вона може обробити. Коли розмова наближається до цього ліміту, OpenClaw **виконує Compaction** старіших повідомлень у підсумок, щоб чат міг продовжуватися.

## Як це працює

1. Старіші ходи розмови підсумовуються в компактний запис.
2. Підсумок зберігається в транскрипті сесії.
3. Недавні повідомлення зберігаються без змін.

Коли OpenClaw розбиває історію на фрагменти Compaction, він зберігає виклики інструментів помічника в парі з відповідними записами `toolResult`. Якщо точка розбиття потрапляє всередину блоку інструмента, OpenClaw зміщує межу так, щоб пара залишалася разом і поточний непідсумований хвіст зберігався.

Повна історія розмови залишається на диску. Compaction змінює лише те, що модель бачить на наступному ході.

## Автоматичний Compaction

Автоматичний Compaction увімкнено за замовчуванням. Він запускається, коли сесія наближається до ліміту контексту, або коли модель повертає помилку переповнення контексту (у такому разі OpenClaw виконує Compaction і повторює спробу).

Ви побачите:

- `🧹 Auto-compaction complete` у докладному режимі.
- `/status`, що показує `🧹 Compactions: <count>`.

<Info>
Перед виконанням Compaction OpenClaw автоматично нагадує агенту зберегти важливі нотатки у файли [пам’яті](/uk/concepts/memory). Це запобігає втраті контексту.
</Info>

<AccordionGroup>
  <Accordion title="Розпізнані сигнатури переповнення">
    OpenClaw виявляє переповнення контексту за такими шаблонами помилок провайдерів:

    - `request_too_large`
    - `context length exceeded`
    - `input exceeds the maximum number of tokens`
    - `input token count exceeds the maximum number of input tokens`
    - `input is too long for the model`
    - `ollama error: context length exceeded`
  </Accordion>
</AccordionGroup>

## Ручний Compaction

Введіть `/compact` у будь-якому чаті, щоб примусово виконати Compaction. Додайте інструкції, щоб спрямувати підсумок:

````
/compact Focus on the API design decisions
````

Коли встановлено `agents.defaults.compaction.keepRecentTokens`, ручний Compaction враховує цю точку відсікання Pi і зберігає недавній хвіст у перебудованому контексті. Без явно заданого бюджету збереження ручний Compaction поводиться як жорстка контрольна точка й продовжується лише з нового підсумку.

## Конфігурація

Налаштовуйте Compaction у розділі `agents.defaults.compaction` вашого `openclaw.json`. Нижче наведено найуживаніші параметри; повний довідник дивіться в [Поглиблене керування сесіями](/reference/session-management-compaction).

### Використання іншої моделі

За замовчуванням Compaction використовує основну модель агента. Установіть `agents.defaults.compaction.model`, щоб доручити підсумовування більш потужній або спеціалізованій моделі. Це перевизначення приймає будь-який рядок `provider/model-id`:

```json
{
  "agents": {
    "defaults": {
      "compaction": {
        "model": "openrouter/anthropic/claude-sonnet-4-6"
      }
    }
  }
}
```

Це також працює з локальними моделями, наприклад із другою моделлю Ollama, виділеною для підсумовування:

```json
{
  "agents": {
    "defaults": {
      "compaction": {
        "model": "ollama/llama3.1:8b"
      }
    }
  }
}
```

Якщо значення не задано, Compaction використовує основну модель агента.

### Збереження ідентифікаторів

Підсумовування Compaction за замовчуванням зберігає непрозорі ідентифікатори (`identifierPolicy: "strict"`). Перевизначте на `identifierPolicy: "off"`, щоб вимкнути це, або на `identifierPolicy: "custom"` разом із `identifierInstructions` для власних вказівок.

### Захист за байтами активного транскрипту

Коли встановлено `agents.defaults.compaction.maxActiveTranscriptBytes`, OpenClaw запускає звичайний локальний Compaction перед виконанням, якщо активний JSONL досягає цього розміру. Це корисно для довготривалих сесій, де керування контекстом на боці провайдера може підтримувати контекст моделі в належному стані, тоді як локальний транскрипт продовжує зростати. Це не розбиває сирі байти JSONL; натомість використовується звичайний конвеєр Compaction для створення семантичного підсумку.

<Warning>
Захист за байтами вимагає `truncateAfterCompaction: true`. Без ротації транскрипту активний файл не зменшиться, і захист залишиться неактивним.
</Warning>

### Наступні транскрипти

Коли ввімкнено `agents.defaults.compaction.truncateAfterCompaction`, OpenClaw не переписує наявний транскрипт на місці. Він створює новий активний наступний транскрипт із підсумку Compaction, збереженого стану та непідсумованого хвоста, а попередній JSONL залишає як архівне джерело контрольної точки.

### Сповіщення про Compaction

За замовчуванням Compaction виконується безшумно. Установіть `notifyUser`, щоб показувати короткі повідомлення про стан, коли Compaction починається та завершується:

```json5
{
  agents: {
    defaults: {
      compaction: {
        notifyUser: true,
      },
    },
  },
}
```

### Скидання пам’яті

Перед Compaction OpenClaw може виконати **тихий скид пам’яті**, щоб зберегти довготривалі нотатки на диск. Докладніше та про конфігурацію див. у [Пам’ять](/uk/concepts/memory).

## Підключувані провайдери Compaction

Plugins можуть реєструвати власного провайдера Compaction через `registerCompactionProvider()` в API плагіна. Коли провайдера зареєстровано й налаштовано, OpenClaw делегує йому підсумовування замість вбудованого конвеєра LLM.

Щоб використовувати зареєстрованого провайдера, установіть його id у конфігурації:

```json
{
  "agents": {
    "defaults": {
      "compaction": {
        "provider": "my-provider"
      }
    }
  }
}
```

Встановлення `provider` автоматично примусово задає `mode: "safeguard"`. Провайдери отримують ті самі інструкції для Compaction і політику збереження ідентифікаторів, що й вбудований шлях, а OpenClaw усе одно зберігає контекст недавніх ходів і суфіксний контекст розділених ходів після виходу провайдера.

<Note>
Якщо провайдер завершується з помилкою або повертає порожній результат, OpenClaw повертається до вбудованого підсумовування LLM.
</Note>

## Compaction vs pruning

|                  | Compaction                    | Pruning                              |
| ---------------- | ----------------------------- | ------------------------------------ |
| **Що робить**    | Підсумовує старішу розмову    | Обрізає старі результати інструментів |
| **Зберігається?**| Так (у транскрипті сесії)     | Ні (лише в пам’яті, для кожного запиту) |
| **Обсяг**        | Уся розмова                   | Лише результати інструментів         |

[Обрізання сесії](/uk/concepts/session-pruning) — це легший доповнювальний механізм, який обрізає вихід інструментів без підсумовування.

## Усунення несправностей

**Compaction виконується надто часто?** Можливо, вікно контексту моделі замале, або результати інструментів занадто великі. Спробуйте ввімкнути [обрізання сесії](/uk/concepts/session-pruning).

**Після Compaction контекст здається застарілим?** Використайте `/compact Focus on <topic>`, щоб спрямувати підсумок, або ввімкніть [скид пам’яті](/uk/concepts/memory), щоб нотатки зберігалися.

**Потрібен чистий старт?** `/new` запускає нову сесію без Compaction.

Для розширеної конфігурації (резервування токенів, збереження ідентифікаторів, власні рушії контексту, server-side Compaction OpenAI) див. [Поглиблене керування сесіями](/uk/reference/session-management-compaction).

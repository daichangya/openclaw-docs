Кожна модель має вікно контексту: максимальну кількість токенів, яку вона може обробити. Коли розмова наближається до цієї межі, OpenClaw **виконує Compaction** старіших повідомлень у стислий підсумок, щоб чат міг продовжуватися.

## Як це працює

1. Старіші ходи розмови підсумовуються в компактний запис.
2. Підсумок зберігається в транскрипті сесії.
3. Недавні повідомлення зберігаються без змін.

Коли OpenClaw ділить історію на фрагменти Compaction, він зберігає виклики інструментів асистента в парі з відповідними записами `toolResult`. Якщо точка поділу припадає всередину блоку інструмента, OpenClaw зсуває межу так, щоб пара залишалася разом, а поточний непідсумований хвіст було збережено.

Повна історія розмови залишається на диску. Compaction змінює лише те, що модель бачить на наступному ході.

## Автоматичний Compaction

Автоматичний Compaction увімкнено за замовчуванням. Він запускається, коли сесія наближається до ліміту контексту, або коли модель повертає помилку переповнення контексту (у такому разі OpenClaw виконує Compaction і повторює спробу).

Ви побачите:

- `🧹 Auto-compaction complete` у докладному режимі.
- `/status`, що показує `🧹 Compactions: <count>`.

<Info>
Перед виконанням Compaction OpenClaw автоматично нагадує агенту зберегти важливі нотатки у файли [memory](/uk/concepts/memory). Це запобігає втраті контексту.
</Info>

<AccordionGroup>
  <Accordion title="Розпізнавані сигнатури переповнення">
    OpenClaw виявляє переповнення контексту за такими шаблонами помилок провайдера:

    - `request_too_large`
    - `context length exceeded`
    - `input exceeds the maximum number of tokens`
    - `input token count exceeds the maximum number of input tokens`
    - `input is too long for the model`
    - `ollama error: context length exceeded`

  </Accordion>
</AccordionGroup>

## Ручний Compaction

Введіть `/compact` у будь-якому чаті, щоб примусово виконати Compaction. Додайте інструкції, щоб спрямувати підсумок:

```
/compact Focus on the API design decisions
```

Коли встановлено `agents.defaults.compaction.keepRecentTokens`, ручний Compaction враховує цю точку відсікання Pi і зберігає недавній хвіст у перебудованому контексті. Без явно заданого бюджету збереження ручний Compaction поводиться як жорстка контрольна точка і продовжує роботу лише з нового підсумку.

## Конфігурація

Налаштовуйте Compaction у `agents.defaults.compaction` у вашому `openclaw.json`. Нижче наведено найпоширеніші параметри; повний довідник дивіться в [Поглиблений огляд керування сесіями](/uk/reference/session-management-compaction).

### Використання іншої моделі

За замовчуванням Compaction використовує основну модель агента. Встановіть `agents.defaults.compaction.model`, щоб делегувати підсумовування більш потужній або спеціалізованій моделі. Це перевизначення приймає будь-який рядок `provider/model-id`:

```json
{
  "agents": {
    "defaults": {
      "compaction": {
        "model": "openrouter/anthropic/claude-sonnet-4-6"
      }
    }
  }
}
```

Це також працює з локальними моделями, наприклад із другою моделлю Ollama, спеціально виділеною для підсумовування:

```json
{
  "agents": {
    "defaults": {
      "compaction": {
        "model": "ollama/llama3.1:8b"
      }
    }
  }
}
```

Якщо значення не задано, Compaction використовує основну модель агента.

### Збереження ідентифікаторів

Підсумовування Compaction за замовчуванням зберігає непрозорі ідентифікатори (`identifierPolicy: "strict"`). Перевизначте через `identifierPolicy: "off"`, щоб вимкнути це, або `identifierPolicy: "custom"` разом із `identifierInstructions` для власних вказівок.

### Захист за байтами активного транскрипту

Коли встановлено `agents.defaults.compaction.maxActiveTranscriptBytes`, OpenClaw запускає звичайний локальний Compaction перед виконанням, якщо активний JSONL досягає цього розміру. Це корисно для довготривалих сесій, де керування контекстом на боці провайдера може підтримувати здоровий контекст моделі, тоді як локальний транскрипт продовжує зростати. Він не розбиває сирі байти JSONL; натомість він просить звичайний конвеєр Compaction створити семантичний підсумок.

<Warning>
Захист за байтами потребує `truncateAfterCompaction: true`. Без ротації транскрипту активний файл не зменшиться, і цей захист залишиться неактивним.
</Warning>

### Наступні транскрипти

Коли ввімкнено `agents.defaults.compaction.truncateAfterCompaction`, OpenClaw не переписує наявний транскрипт на місці. Він створює новий активний наступний транскрипт із підсумку Compaction, збереженого стану та непідсумованого хвоста, а попередній JSONL залишає як архівне джерело контрольної точки.

### Сповіщення про Compaction

За замовчуванням Compaction виконується без повідомлень. Установіть `notifyUser`, щоб показувати короткі повідомлення про стан, коли Compaction починається і завершується:

```json5
{
  agents: {
    defaults: {
      compaction: {
        notifyUser: true,
      },
    },
  },
}
```

### Скидання memory

Перед Compaction OpenClaw може виконати **тихе скидання memory**, щоб зберегти довготривалі нотатки на диск. Докладніше та про конфігурацію дивіться в [Memory](/uk/concepts/memory).

## Підключувані провайдери Compaction

Plugins можуть реєструвати власний провайдер Compaction через `registerCompactionProvider()` в API plugin. Коли провайдера зареєстровано і налаштовано, OpenClaw делегує йому підсумовування замість вбудованого LLM-конвеєра.

Щоб використовувати зареєстрованого провайдера, встановіть його id у конфігурації:

```json
{
  "agents": {
    "defaults": {
      "compaction": {
        "provider": "my-provider"
      }
    }
  }
}
```

Установлення `provider` автоматично примусово задає `mode: "safeguard"`. Провайдери отримують ті самі інструкції Compaction і політику збереження ідентифікаторів, що й вбудований шлях, а OpenClaw і далі зберігає контекст недавніх ходів і суфіксний контекст розділеного ходу після результату провайдера.

<Note>
Якщо провайдер зазнає невдачі або повертає порожній результат, OpenClaw повертається до вбудованого LLM-підсумовування.
</Note>

## Compaction і pruning

|                  | Compaction                      | Pruning                              |
| ---------------- | ------------------------------- | ------------------------------------ |
| **Що це робить** | Підсумовує старішу розмову      | Обрізає старі результати інструментів |
| **Зберігається?**| Так (у транскрипті сесії)       | Ні (лише в пам’яті, для кожного запиту) |
| **Обсяг**        | Уся розмова                     | Лише результати інструментів         |

[Session pruning](/uk/concepts/session-pruning) — це легше доповнення, яке обрізає вивід інструментів без підсумовування.

## Усунення несправностей

**Compaction відбувається надто часто?** Вікно контексту моделі може бути малим, або виводи інструментів можуть бути великими. Спробуйте ввімкнути [session pruning](/uk/concepts/session-pruning).

**Після Compaction контекст здається застарілим?** Використайте `/compact Focus on <topic>`, щоб спрямувати підсумок, або ввімкніть [скидання memory](/uk/concepts/memory), щоб нотатки зберігалися.

**Потрібен чистий старт?** `/new` запускає нову сесію без виконання Compaction.

Для розширеної конфігурації (резерв токенів, збереження ідентифікаторів, власні рушії контексту, серверний Compaction OpenAI) дивіться [Поглиблений огляд керування сесіями](/uk/reference/session-management-compaction).

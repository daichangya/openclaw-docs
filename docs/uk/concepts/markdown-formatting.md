OpenClaw форматує вихідний Markdown, перетворюючи його на спільне проміжне
подання (IR) перед рендерингом вихідних даних для конкретного каналу. IR зберігає
початковий текст без змін, одночасно містячи діапазони стилів/посилань, щоб поділ на частини й рендеринг
залишалися узгодженими в різних каналах.

## Цілі

- **Узгодженість:** один етап парсингу, кілька рендерерів.
- **Безпечний поділ на частини:** розбивати текст до рендерингу, щоб вбудоване форматування
  ніколи не ламалося між частинами.
- **Відповідність каналу:** зіставляти той самий IR зі Slack mrkdwn, Telegram HTML і Signal
  діапазонами стилів без повторного парсингу Markdown.

## Конвеєр

1. **Розбір Markdown -> IR**
   - IR — це звичайний текст плюс діапазони стилів (bold/italic/strike/code/spoiler) і діапазони посилань.
   - Зміщення задаються в кодових одиницях UTF-16, щоб діапазони стилів Signal узгоджувалися з його API.
   - Таблиці розбираються лише тоді, коли канал увімкнув перетворення таблиць.
2. **Поділ IR на частини (спочатку форматування)**
   - Поділ на частини відбувається на тексті IR до рендерингу.
   - Вбудоване форматування не розривається між частинами; діапазони нарізаються окремо для кожної частини.
3. **Рендеринг для кожного каналу**
   - **Slack:** токени mrkdwn (bold/italic/strike/code), посилання як `<url|label>`.
   - **Telegram:** HTML-теги (`<b>`, `<i>`, `<s>`, `<code>`, `<pre><code>`, `<a href>`).
   - **Signal:** звичайний текст + діапазони `text-style`; посилання стають `label (url)`, коли мітка відрізняється.

## Приклад IR

Вхідний Markdown:

```markdown
Hello **world** — see [docs](https://docs.openclaw.ai).
```

IR (схематично):

```json
{
  "text": "Hello world — see docs.",
  "styles": [{ "start": 6, "end": 11, "style": "bold" }],
  "links": [{ "start": 19, "end": 23, "href": "https://docs.openclaw.ai" }]
}
```

## Де це використовується

- Вихідні адаптери Slack, Telegram і Signal виконують рендеринг з IR.
- Інші канали (WhatsApp, iMessage, Microsoft Teams, Discord) і далі використовують звичайний текст або
  власні правила форматування, причому перетворення таблиць Markdown застосовується до
  поділу на частини, коли це ввімкнено.

## Обробка таблиць

Таблиці Markdown підтримуються неузгоджено в різних клієнтах чату. Використовуйте
`markdown.tables`, щоб керувати перетворенням для кожного каналу (і для кожного облікового запису).

- `code`: рендерити таблиці як блоки коду (типово для більшості каналів).
- `bullets`: перетворювати кожен рядок на марковані пункти (типово для Signal + WhatsApp).
- `off`: вимкнути розбір і перетворення таблиць; сирий текст таблиці передається без змін.

Ключі конфігурації:

```yaml
channels:
  discord:
    markdown:
      tables: code
    accounts:
      work:
        markdown:
          tables: off
```

## Правила поділу на частини

- Обмеження розміру частин надходять з адаптерів/конфігурації каналу і застосовуються до тексту IR.
- Огорожі коду зберігаються як єдиний блок із кінцевим символом нового рядка, щоб канали
  рендерили їх правильно.
- Префікси списків і цитат є частиною тексту IR, тому поділ на частини
  не розриває їх посередині префікса.
- Вбудовані стилі (bold/italic/strike/inline-code/spoiler) ніколи не розриваються між
  частинами; рендерер знову відкриває стилі всередині кожної частини.

Якщо вам потрібно більше деталей про поведінку поділу на частини в різних каналах, див.
[Потокова передача + поділ на частини](/uk/concepts/streaming).

## Політика посилань

- **Slack:** `[label](url)` -> `<url|label>`; прості URL залишаються без змін. Автопосилання
  вимкнено під час парсингу, щоб уникнути подвійного лінкування.
- **Telegram:** `[label](url)` -> `<a href="url">label</a>` (режим розбору HTML).
- **Signal:** `[label](url)` -> `label (url)`, якщо мітка не збігається з URL.

## Спойлери

Маркери спойлерів (`||spoiler||`) розбираються лише для Signal, де вони зіставляються з
діапазонами стилю SPOILER. Інші канали сприймають їх як звичайний текст.

## Як додати або оновити форматер каналу

1. **Парсинг один раз:** використовуйте спільний помічник `markdownToIR(...)` з параметрами,
   що відповідають каналу (autolink, стиль заголовків, префікс blockquote).
2. **Рендеринг:** реалізуйте рендерер за допомогою `renderMarkdownWithMarkers(...)` і
   мапи маркерів стилів (або діапазонів стилів Signal).
3. **Поділ на частини:** викличте `chunkMarkdownIR(...)` перед рендерингом; рендерте кожну частину.
4. **Підключення адаптера:** оновіть вихідний адаптер каналу так, щоб він використовував новий механізм поділу на частини
   і рендерер.
5. **Тестування:** додайте або оновіть тести форматування і тест вихідної доставки, якщо
   канал використовує поділ на частини.

## Типові підводні камені

- Токени Slack у кутових дужках (`<@U123>`, `<#C123>`, `<https://...>`) мають
  зберігатися; безпечно екрануйте сирий HTML.
- Telegram HTML вимагає екранування тексту поза тегами, щоб уникнути зламаної розмітки.
- Діапазони стилів Signal залежать від зміщень UTF-16; не використовуйте зміщення за кодовими точками.
- Зберігайте кінцеві символи нового рядка для огороджених блоків коду, щоб закривальні маркери опинялися
  на окремому рядку.

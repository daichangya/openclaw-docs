# Dreaming (експериментально)

Dreaming — це фонова система консолідації пам’яті в `memory-core`.
Вона допомагає OpenClaw переносити сильні короткострокові сигнали в довготривалу пам’ять,
зберігаючи при цьому зрозумілість і можливість перевірки процесу.

Dreaming є **опціональним** і за замовчуванням вимкнений.

## Що записує Dreaming

Dreaming зберігає два види вихідних даних:

- **Стан машини** в `memory/.dreams/` (сховище recall, сигнали фаз, контрольні точки ingestion, блокування).
- **Людинозрозумілий вивід** у `DREAMS.md` (або наявний `dreams.md`) і необов’язкові файли звітів фаз у `memory/dreaming/<phase>/YYYY-MM-DD.md`.

Просування в довготривалу пам’ять, як і раніше, записується лише в `MEMORY.md`.

## Модель фаз

Dreaming використовує три взаємодоповнювальні фази:

| Фаза | Призначення | Довготривалий запис |
| ----- | ----------- | ------------------- |
| Light | Сортує та готує недавній короткостроковий матеріал | Ні |
| Deep  | Оцінює та просуває довготривалих кандидатів | Так (`MEMORY.md`) |
| REM   | Аналізує теми та повторювані ідеї | Ні |

Ці фази є внутрішніми деталями реалізації, а не окремими
користувацькими «режимами».

### Фаза Light

Фаза Light поглинає недавні щоденні сигнали пам’яті та сліди recall, усуває дублікати
й готує рядки кандидатів.

- Читає зі стану короткострокового recall, нещодавніх щоденних файлів пам’яті та відредагованих транскриптів сесій, якщо вони доступні.
- Записує керований блок `## Light Sleep`, коли сховище містить вбудований вивід.
- Фіксує сигнали підкріплення для подальшого ранжування Deep.
- Ніколи не записує в `MEMORY.md`.

### Фаза Deep

Фаза Deep визначає, що стане довготривалою пам’яттю.

- Ранжує кандидатів за допомогою зваженого оцінювання та порогових фільтрів.
- Вимагає проходження `minScore`, `minRecallCount` і `minUniqueQueries`.
- Повторно завантажує фрагменти з актуальних щоденних файлів перед записом, тому застарілі або видалені фрагменти пропускаються.
- Додає просунуті записи до `MEMORY.md`.
- Записує підсумок `## Deep Sleep` у `DREAMS.md` і за потреби записує `memory/dreaming/deep/YYYY-MM-DD.md`.

### Фаза REM

Фаза REM виділяє закономірності та рефлексивні сигнали.

- Будує підсумки тем і рефлексій на основі недавніх короткострокових слідів.
- Записує керований блок `## REM Sleep`, коли сховище містить вбудований вивід.
- Фіксує сигнали підкріплення REM, які використовуються в ранжуванні Deep.
- Ніколи не записує в `MEMORY.md`.

## Поглинання транскриптів сесій

Dreaming може поглинати відредаговані транскрипти сесій у корпус Dreaming. Якщо
транскрипти доступні, вони подаються у фазу Light разом із щоденними
сигналами пам’яті та слідами recall. Особистий і чутливий вміст редагується
перед поглинанням.

## Dream Diary

Dreaming також веде наративний **Dream Diary** у `DREAMS.md`.
Після того як для кожної фази накопичується достатньо матеріалу, `memory-core` виконує фоновий
хід субагента в режимі best-effort (використовуючи стандартну модель runtime) і додає
короткий запис щоденника.

Цей щоденник призначений для читання людиною в інтерфейсі Dreams, а не як джерело
просування. Артефакти щоденника/звіту, згенеровані Dreaming, виключаються з
короткострокового просування. До просування в
`MEMORY.md` допускаються лише обґрунтовані фрагменти пам’яті.

Також є обґрунтована історична доріжка backfill для перегляду та відновлення:

- `memory rem-harness --path ... --grounded` показує попередній перегляд обґрунтованого виводу щоденника з історичних нотаток `YYYY-MM-DD.md`.
- `memory rem-backfill --path ...` записує оборотні обґрунтовані записи щоденника в `DREAMS.md`.
- `memory rem-backfill --path ... --stage-short-term` додає обґрунтованих довготривалих кандидатів до того самого сховища короткострокових доказів, яке вже використовує звичайна фаза Deep.
- `memory rem-backfill --rollback` і `--rollback-short-term` видаляють ці staged-артефакти backfill, не зачіпаючи звичайні записи щоденника або живий короткостроковий recall.

Control UI надає той самий процес backfill/reset для щоденника, щоб ви могли переглянути
результати в сцені Dreams, перш ніж вирішувати, чи
заслуговують обґрунтовані кандидати на просування. Сцена також показує окрему обґрунтовану доріжку, щоб ви могли бачити,
які staged короткострокові записи надійшли з історичного відтворення, які просунуті
елементи були зумовлені grounded-даними, і очищати лише staged-записи, пов’язані виключно з grounded,
не зачіпаючи звичайний живий короткостроковий стан.

## Сигнали ранжування Deep

Ранжування Deep використовує шість зважених базових сигналів плюс підкріплення фаз:

| Сигнал | Вага | Опис |
| ------ | ---- | ---- |
| Частота | 0.24 | Скільки короткострокових сигналів накопичив запис |
| Релевантність | 0.30 | Середня якість пошуку для запису |
| Різноманітність запитів | 0.15 | Окремі контексти запитів/днів, у яких він з’являвся |
| Актуальність | 0.15 | Оцінка свіжості з часовим згасанням |
| Консолідація | 0.10 | Сила повторюваності впродовж кількох днів |
| Концептуальна насиченість | 0.06 | Щільність concept-tag із фрагмента/шляху |

Влучання фаз Light і REM додають невелике підсилення зі згасанням за часом із
`memory/.dreams/phase-signals.json`.

## Розклад

Коли Dreaming увімкнено, `memory-core` автоматично керує одним завданням Cron для повного
циклу Dreaming. Кожен цикл запускає фази по порядку: light -> REM -> deep.

Типова поведінка розкладу:

| Налаштування | Типове значення |
| ------------ | --------------- |
| `dreaming.frequency` | `0 3 * * *` |

## Швидкий старт

Увімкнути Dreaming:

```json
{
  "plugins": {
    "entries": {
      "memory-core": {
        "config": {
          "dreaming": {
            "enabled": true
          }
        }
      }
    }
  }
}
```

Увімкнути Dreaming із власною частотою циклу:

```json
{
  "plugins": {
    "entries": {
      "memory-core": {
        "config": {
          "dreaming": {
            "enabled": true,
            "timezone": "America/Los_Angeles",
            "frequency": "0 */6 * * *"
          }
        }
      }
    }
  }
}
```


## Робочий процес CLI

Використовуйте просування через CLI для попереднього перегляду або ручного застосування:

```bash
openclaw memory promote
openclaw memory promote --apply
openclaw memory promote --limit 5
openclaw memory status --deep
```

Ручна команда `memory promote` за замовчуванням використовує пороги фази Deep, якщо їх не перевизначено
через прапорці CLI.

Пояснити, чому конкретний кандидат буде або не буде просунутий:

```bash
openclaw memory promote-explain "router vlan"
openclaw memory promote-explain "router vlan" --json
```

Попередній перегляд рефлексій REM, істин кандидатів і результату просування Deep без
будь-якого запису:

```bash
openclaw memory rem-harness
openclaw memory rem-harness --json
```

## Ключові типові значення

Усі налаштування розташовані в `plugins.entries.memory-core.config.dreaming`.

| Ключ | Типове значення |
| ---- | --------------- |
| `enabled` | `false` |
| `frequency` | `0 3 * * *` |

Політика фаз, пороги та поведінка сховища є внутрішніми деталями реалізації
(не належать до користувацької конфігурації).

Див. [довідник із конфігурації Memory](/uk/reference/memory-config#dreaming-experimental) для повного списку ключів.

## UI Dreams

Коли Dreaming увімкнено, вкладка Gateway **Dreams** показує:

- поточний стан увімкнення Dreaming
- статус на рівні фаз і наявність керованого циклу
- кількість короткострокових, grounded, сигналів і просунутих сьогодні записів
- час наступного запланованого запуску
- окрему доріжку Scene для staged записів історичного відтворення
- розгортаний переглядач Dream Diary на основі `doctor.memory.dreamDiary`

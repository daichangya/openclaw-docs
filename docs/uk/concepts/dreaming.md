# Dreaming

Dreaming — це фонова система консолідування пам’яті в `memory-core`.
Вона допомагає OpenClaw переносити сильні короткострокові сигнали в довговічну пам’ять,
зберігаючи процес зрозумілим і придатним для перевірки.

Dreaming є **opt-in** і типово вимкнений.

## Що записує Dreaming

Dreaming зберігає два типи виводу:

- **Машинний стан** у `memory/.dreams/` (сховище recall, сигнали фаз, контрольні точки індексації, блокування).
- **Людинозрозумілий вивід** у `DREAMS.md` (або наявному `dreams.md`) і необов’язкові файли звітів фаз у `memory/dreaming/<phase>/YYYY-MM-DD.md`.

Довгострокове підвищення, як і раніше, записує дані лише в `MEMORY.md`.

## Модель фаз

Dreaming використовує три взаємодіючі фази:

| Фаза | Призначення                               | Довговічний запис |
| ----- | ----------------------------------------- | ----------------- |
| Light | Сортування та підготовка недавнього короткострокового матеріалу | Ні                |
| Deep  | Оцінювання та підвищення довговічних кандидатів | Так (`MEMORY.md`) |
| REM   | Осмислення тем і повторюваних ідей        | Ні                |

Ці фази є внутрішніми деталями реалізації, а не окремими користувацькими
"режимами."

### Фаза Light

Фаза Light поглинає недавні щоденні сигнали пам’яті й сліди recall, усуває дублікати
та готує рядки-кандидати.

- Читає дані зі стану короткострокового recall, недавніх щоденних файлів пам’яті та відредагованих транскриптів сесій, якщо вони доступні.
- Записує керований блок `## Light Sleep`, якщо сховище містить inline-вивід.
- Фіксує сигнали підкріплення для подальшого ранжування Deep.
- Ніколи не записує в `MEMORY.md`.

### Фаза Deep

Фаза Deep вирішує, що стане довгостроковою пам’яттю.

- Ранжує кандидатів за допомогою зваженого оцінювання й порогових фільтрів.
- Вимагає проходження `minScore`, `minRecallCount` і `minUniqueQueries`.
- Повторно гідратує фрагменти з актуальних щоденних файлів перед записом, тому застарілі/видалені фрагменти пропускаються.
- Додає підвищені записи до `MEMORY.md`.
- Записує підсумок `## Deep Sleep` у `DREAMS.md` і за потреби — у `memory/dreaming/deep/YYYY-MM-DD.md`.

### Фаза REM

Фаза REM витягує патерни та рефлексивні сигнали.

- Будує підсумки тем і рефлексій на основі недавніх короткострокових слідів.
- Записує керований блок `## REM Sleep`, якщо сховище містить inline-вивід.
- Фіксує сигнали підкріплення REM, які використовуються під час ранжування Deep.
- Ніколи не записує в `MEMORY.md`.

## Індексація транскриптів сесій

Dreaming може поглинати відредаговані транскрипти сесій у корпус Dreaming. Коли
транскрипти доступні, вони подаються у фазу Light разом із щоденними
сигналами пам’яті та слідами recall. Особистий і чутливий вміст редагується
перед індексацією.

## Dream Diary

Dreaming також веде наративний **Dream Diary** у `DREAMS.md`.
Після того як для кожної фази накопичується достатньо матеріалу, `memory-core` виконує фоновий
хід субагента в режимі best-effort (використовуючи типову модель runtime) і додає короткий запис щоденника.

Цей щоденник призначений для читання людьми в UI Dreams, а не як джерело підвищення.
Артефакти щоденника/звіту, згенеровані Dreaming, виключаються з короткострокового
підвищення. Для підвищення в `MEMORY.md` придатні лише фрагменти пам’яті, прив’язані до фактів.

Також існує шлях grounded historical backfill для перевірки та відновлення:

- `memory rem-harness --path ... --grounded` попередньо переглядає grounded-вивід щоденника з історичних нотаток `YYYY-MM-DD.md`.
- `memory rem-backfill --path ...` записує зворотні grounded-записи щоденника до `DREAMS.md`.
- `memory rem-backfill --path ... --stage-short-term` розміщує grounded-довговічних кандидатів у тому самому short-term evidence store, який уже використовує звичайна фаза Deep.
- `memory rem-backfill --rollback` і `--rollback-short-term` видаляють ці staged-артефакти backfill, не зачіпаючи звичайні записи щоденника чи актуальний короткостроковий recall.

Control UI надає той самий процес backfill/reset для щоденника, щоб ви могли перевірити
результати в сцені Dreams, перш ніж вирішувати, чи заслуговують grounded-кандидати
на підвищення. Сцена також показує окрему grounded-смугу, щоб ви могли бачити,
які staged-короткострокові записи походять з історичного повторного відтворення, які підвищені
елементи були спричинені grounded-потоком, і очищати лише staged-елементи, пов’язані тільки з grounded,
не зачіпаючи звичайний актуальний короткостроковий стан.

## Сигнали ранжування Deep

Ранжування Deep використовує шість зважених базових сигналів плюс підкріплення фаз:

| Сигнал              | Вага | Опис                                              |
| ------------------- | ---- | ------------------------------------------------- |
| Частота             | 0.24 | Скільки короткострокових сигналів накопичив запис |
| Релевантність       | 0.30 | Середня якість retrieval для запису               |
| Різноманіття запитів | 0.15 | Різні контексти запитів/днів, у яких він з’являвся |
| Актуальність        | 0.15 | Оцінка свіжості з часовим згасанням               |
| Консолідація        | 0.10 | Сила повторюваності протягом кількох днів         |
| Концептуальна насиченість | 0.06 | Щільність concept-тегів із фрагмента/шляху      |

Влучання фаз Light і REM додають невелике підсилення зі згасанням за часом із
`memory/.dreams/phase-signals.json`.

## Планування

Коли ввімкнено, `memory-core` автоматично керує одним завданням Cron для повного
циклу Dreaming. Кожен цикл запускає фази в такому порядку: light -> REM -> deep.

Типова поведінка частоти:

| Налаштування        | Типове значення |
| ------------------- | --------------- |
| `dreaming.frequency` | `0 3 * * *`    |

## Швидкий старт

Увімкнути Dreaming:

```json
{
  "plugins": {
    "entries": {
      "memory-core": {
        "config": {
          "dreaming": {
            "enabled": true
          }
        }
      }
    }
  }
}
```

Увімкнути Dreaming із власною частотою циклу:

```json
{
  "plugins": {
    "entries": {
      "memory-core": {
        "config": {
          "dreaming": {
            "enabled": true,
            "timezone": "America/Los_Angeles",
            "frequency": "0 */6 * * *"
          }
        }
      }
    }
  }
}
```

```
/dreaming status

```

## Робочий процес CLI

Використовуйте підвищення через CLI для попереднього перегляду або ручного застосування:

```bash
openclaw memory promote
openclaw memory promote --apply
openclaw memory promote --limit 5
openclaw memory status --deep
```

Ручний `memory promote` типово використовує пороги фази Deep, якщо їх не перевизначено
через прапорці CLI.

Пояснити, чому конкретний кандидат буде або не буде підвищено:

```bash
openclaw memory promote-explain "router vlan"
openclaw memory promote-explain "router vlan" --json
```

Попередньо переглянути рефлексії REM, candidate truths і вивід підвищення Deep без
будь-якого запису:

```bash
openclaw memory rem-harness
openclaw memory rem-harness --json
```

## Ключові типові значення

Усі налаштування розміщені в `plugins.entries.memory-core.config.dreaming`.

| Ключ       | Типове значення |
| ---------- | --------------- |
| `enabled`  | `false`         |
| `frequency` | `0 3 * * *`    |

Політика фаз, пороги та поведінка сховища — це внутрішні деталі реалізації
(не користувацька конфігурація).

Див. [Memory configuration reference](/uk/reference/memory-config#dreaming),
щоб ознайомитися з повним списком ключів.

## UI Dreams

Коли ввімкнено, вкладка **Dreams** у Gateway показує:

- поточний стан увімкнення Dreaming
- статус на рівні фаз і наявність керованого циклу
- лічильники short-term, grounded, signal і promoted-today
- час наступного запланованого запуску
- окрему grounded-смугу сцени для staged-записів історичного повторного відтворення
- розгортаний переглядач Dream Diary на основі `doctor.memory.dreamDiary`

## Усунення несправностей

### Dreaming ніколи не запускається (статус показує blocked)

Керований Cron Dreaming працює на Heartbeat типового агента. Якщо Heartbeat не спрацьовує для цього агента, Cron ставить системну подію в чергу, яку ніхто не споживає, і Dreaming мовчки не запускається. І `openclaw memory status`, і `/dreaming status` у такому разі покажуть `blocked` і назвуть агента, чий Heartbeat є блокером.

Дві поширені причини:

- Інший агент оголошує явний блок `heartbeat:`. Коли будь-який запис у `agents.list` має власний блок `heartbeat`, Heartbeat працює лише для цих агентів — типові значення більше не застосовуються до всіх інших, тому типовий агент може замовкнути. Перемістіть налаштування Heartbeat у `agents.defaults.heartbeat` або додайте явний блок `heartbeat` до типового агента. Див. [Scope and precedence](/uk/gateway/heartbeat#scope-and-precedence).
- `heartbeat.every` дорівнює `0`, порожній або не піддається розбору. У Cron немає інтервалу, за яким можна планувати, тому Heartbeat фактично вимкнений. Установіть для `every` додатну тривалість, наприклад `30m`. Див. [Defaults](/uk/gateway/heartbeat#defaults).

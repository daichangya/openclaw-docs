Dreaming — це фонова система консолідування пам’яті в `memory-core`.
Вона допомагає OpenClaw переносити сильні короткострокові сигнали в стійку пам’ять, водночас
зберігаючи процес зрозумілим і придатним для перевірки.

Dreaming є **опціональним** і за замовчуванням вимкнений.

## Що записує Dreaming

Dreaming підтримує два типи виводу:

- **Стан машини** у `memory/.dreams/` (сховище recall, сигнали фаз, контрольні точки інжесту, блокування).
- **Людинозрозумілий вивід** у `DREAMS.md` (або наявному `dreams.md`) і необов’язкові файли звітів фаз у `memory/dreaming/<phase>/YYYY-MM-DD.md`.

Довгострокове підвищення, як і раніше, записується лише в `MEMORY.md`.

## Модель фаз

Dreaming використовує три взаємодіючі фази:

| Фаза | Призначення                               | Стійкий запис     |
| ----- | ----------------------------------------- | ----------------- |
| Легка | Сортує та готує нещодавній короткостроковий матеріал | Ні                |
| Глибока  | Оцінює та підвищує стійких кандидатів      | Так (`MEMORY.md`) |
| REM   | Осмислює теми та повторювані ідеї          | Ні                |

Ці фази є внутрішніми деталями реалізації, а не окремими
користувацькими «режимами».

### Легка фаза

Легка фаза інжестує нещодавні щоденні сигнали пам’яті та сліди recall, прибирає дублікати
й готує рядки-кандидати.

- Читає зі стану короткострокового recall, нещодавніх щоденних файлів пам’яті та відредагованих транскриптів сесій, якщо вони доступні.
- Записує керований блок `## Light Sleep`, коли сховище містить вбудований вивід.
- Фіксує сигнали підкріплення для подальшого глибокого ранжування.
- Ніколи не записує в `MEMORY.md`.

### Глибока фаза

Глибока фаза визначає, що стане довгостроковою пам’яттю.

- Ранжує кандидатів за допомогою зваженого скорингу та порогових перевірок.
- Вимагає проходження `minScore`, `minRecallCount` і `minUniqueQueries`.
- Перед записом повторно завантажує фрагменти з актуальних щоденних файлів, тому застарілі або видалені фрагменти пропускаються.
- Додає підвищені записи до `MEMORY.md`.
- Записує підсумок `## Deep Sleep` у `DREAMS.md` і за потреби записує `memory/dreaming/deep/YYYY-MM-DD.md`.

### REM-фаза

REM-фаза витягує шаблони та рефлексивні сигнали.

- Формує підсумки тем і рефлексій із нещодавніх короткострокових слідів.
- Записує керований блок `## REM Sleep`, коли сховище містить вбудований вивід.
- Фіксує сигнали підкріплення REM, які використовуються для глибокого ранжування.
- Ніколи не записує в `MEMORY.md`.

## Інжест транскриптів сесій

Dreaming може інжестувати відредаговані транскрипти сесій у корпус Dreaming. Коли
транскрипти доступні, вони подаються в легку фазу разом із щоденними
сигналами пам’яті та слідами recall. Особистий і чутливий вміст редагується
перед інжестом.

## Dream Diary

Dreaming також веде наративний **Dream Diary** у `DREAMS.md`.
Після того як для кожної фази назбирається достатньо матеріалу, `memory-core` запускає фоновий
підхід subagent у режимі best-effort (з використанням типової runtime-моделі)
і додає короткий запис щоденника.

Цей щоденник призначений для читання людиною в інтерфейсі Dreams, а не як джерело підвищення.
Артефакти щоденника/звіту, згенеровані Dreaming, виключаються з короткострокового
підвищення. До підвищення в
`MEMORY.md` можуть потрапити лише прив’язані до джерел фрагменти пам’яті.

Також є прив’язана до джерел історична смуга backfill для перевірки та відновлення:

- `memory rem-harness --path ... --grounded` попередньо переглядає прив’язаний до джерел вивід щоденника з історичних нотаток `YYYY-MM-DD.md`.
- `memory rem-backfill --path ...` записує оборотні прив’язані до джерел записи щоденника в `DREAMS.md`.
- `memory rem-backfill --path ... --stage-short-term` поміщає прив’язаних до джерел стійких кандидатів у те саме сховище короткострокових доказів, яке вже використовує звичайна глибока фаза.
- `memory rem-backfill --rollback` і `--rollback-short-term` видаляють ці staged-артефакти backfill, не зачіпаючи звичайні записи щоденника або живий короткостроковий recall.

Інтерфейс Control надає той самий процес backfill/скидання щоденника, щоб ви могли перевірити
результати в сцені Dreams, перш ніж вирішувати, чи заслуговують прив’язані до джерел кандидати
на підвищення. Сцена також показує окрему прив’язану до джерел смугу, щоб ви могли бачити,
які staged-короткострокові записи надійшли з історичного відтворення, які підвищені
елементи були спричинені прив’язкою до джерел, а також очищати лише staged-записи, що належать
лише до прив’язаних до джерел, не зачіпаючи звичайний живий короткостроковий стан.

## Сигнали глибокого ранжування

Глибоке ранжування використовує шість зважених базових сигналів плюс підкріплення фаз:

| Сигнал              | Вага | Опис                                              |
| ------------------- | ---- | ------------------------------------------------- |
| Частота           | 0.24   | Скільки короткострокових сигналів накопичив запис |
| Релевантність           | 0.30   | Середня якість отримання для запису           |
| Різноманітність запитів     | 0.15   | Різні контексти запитів/днів, у яких він з’являвся      |
| Нещодавність             | 0.15   | Оцінка свіжості з часовим згасанням                      |
| Консолідація       | 0.10   | Сила повторюваності протягом кількох днів                     |
| Концептуальна насиченість | 0.06   | Щільність concept-тегів за фрагментом/шляхом             |

Влучання легкої та REM-фаз додають невелике підсилення із часовим згасанням із
`memory/.dreams/phase-signals.json`.

## Розклад

Коли цю функцію ввімкнено, `memory-core` автоматично керує одним Cron-завданням для повного
циклу Dreaming. Кожен цикл запускає фази по порядку: light -> REM -> deep.

Типова поведінка частоти:

| Налаштування              | Типове значення     |
| -------------------- | ----------- |
| `dreaming.frequency` | `0 3 * * *` |

## Швидкий старт

Увімкніть Dreaming:

```json
{
  "plugins": {
    "entries": {
      "memory-core": {
        "config": {
          "dreaming": {
            "enabled": true
          }
        }
      }
    }
  }
}
```

Увімкніть Dreaming із власною частотою циклу:

```json
{
  "plugins": {
    "entries": {
      "memory-core": {
        "config": {
          "dreaming": {
            "enabled": true,
            "timezone": "America/Los_Angeles",
            "frequency": "0 */6 * * *"
          }
        }
      }
    }
  }
}
```

## Робочий процес CLI

Використовуйте підвищення через CLI для попереднього перегляду або ручного застосування:

```bash
openclaw memory promote
openclaw memory promote --apply
openclaw memory promote --limit 5
openclaw memory status --deep
```

Ручний `memory promote` типово використовує пороги глибокої фази, якщо їх не перевизначено
через прапорці CLI.

Поясніть, чому конкретний кандидат буде або не буде підвищений:

```bash
openclaw memory promote-explain "router vlan"
openclaw memory promote-explain "router vlan" --json
```

Попередньо перегляньте REM-рефлексії, candidate truths і вивід глибокого підвищення, нічого
не записуючи:

```bash
openclaw memory rem-harness
openclaw memory rem-harness --json
```

## Ключові типові значення

Усі налаштування розміщені в `plugins.entries.memory-core.config.dreaming`.

| Ключ         | Типове значення     |
| ----------- | ----------- |
| `enabled`   | `false`     |
| `frequency` | `0 3 * * *` |

Політика фаз, пороги та поведінка сховища — це внутрішні деталі реалізації
(не користувацька конфігурація).

Повний список ключів див. у [довіднику з конфігурації Memory](/uk/reference/memory-config#dreaming).

## Інтерфейс Dreams

Коли функцію ввімкнено, вкладка Gateway **Dreams** показує:

- поточний стан увімкнення Dreaming
- стан на рівні фаз і наявність керованого циклу
- кількість короткострокових, прив’язаних до джерел, сигнальних і підвищених за сьогодні записів
- час наступного запланованого запуску
- окрему смугу сцени з прив’язкою до джерел для staged-записів історичного відтворення
- розгортаний читач Dream Diary на основі `doctor.memory.dreamDiary`

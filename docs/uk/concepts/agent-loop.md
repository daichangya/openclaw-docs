# Цикл агента (OpenClaw)

Агентний цикл — це повний «реальний» запуск агента: приймання → збирання контексту → інференс моделі →
виконання інструментів → потокове передавання відповідей → збереження. Це авторитетний шлях, який перетворює повідомлення
на дії та фінальну відповідь, водночас зберігаючи узгоджений стан сесії.

В OpenClaw цикл — це один серіалізований запуск на сесію, який надсилає події життєвого циклу й потоків,
поки модель думає, викликає інструменти та передає вихідні дані потоком. У цьому документі пояснюється, як цей справжній цикл
зв’язаний наскрізно.

## Точки входу

- Gateway RPC: `agent` і `agent.wait`.
- CLI: команда `agent`.

## Як це працює (на високому рівні)

1. RPC `agent` перевіряє параметри, визначає сесію (sessionKey/sessionId), зберігає метадані сесії, одразу повертає `{ runId, acceptedAt }`.
2. `agentCommand` запускає агента:
   - визначає типові значення model + thinking/verbose/trace
   - завантажує знімок Skills
   - викликає `runEmbeddedPiAgent` (середовище виконання pi-agent-core)
   - надсилає **lifecycle end/error**, якщо вбудований цикл цього не надсилає
3. `runEmbeddedPiAgent`:
   - серіалізує запуски через черги на рівні сесії та глобальні черги
   - визначає model + профіль автентифікації та будує сесію Pi
   - підписується на події Pi і передає assistant/tool delta потоком
   - застосовує тайм-аут -> перериває запуск, якщо його перевищено
   - повертає payload та метадані usage
4. `subscribeEmbeddedPiSession` з’єднує події pi-agent-core з потоком OpenClaw `agent`:
   - події tool => `stream: "tool"`
   - delta assistant => `stream: "assistant"`
   - події lifecycle => `stream: "lifecycle"` (`phase: "start" | "end" | "error"`)
5. `agent.wait` використовує `waitForAgentRun`:
   - очікує **lifecycle end/error** для `runId`
   - повертає `{ status: ok|error|timeout, startedAt, endedAt, error? }`

## Постановка в чергу + конкурентність

- Запуски серіалізуються для кожного ключа сесії (доріжка сесії) і, за потреби, проходять через глобальну доріжку.
- Це запобігає гонкам інструментів/сесій і зберігає узгодженість історії сесії.
- Канали обміну повідомленнями можуть вибирати режими черги (collect/steer/followup), які подають дані в цю систему доріжок.
  Див. [Черга команд](/uk/concepts/queue).
- Записи транскриптів також захищені блокуванням запису сесії у файлі сесії. Блокування
  враховує процеси й базується на файлах, тому воно виявляє записувачі, які обходять внутрішньопроцесну чергу або надходять
  з іншого процесу.
- Блокування запису сесії за типових умов не є реентерабельними. Якщо допоміжний код навмисно вкладає отримання
  того самого блокування, зберігаючи одного логічного записувача, він має явно ввімкнути це через
  `allowReentrant: true`.

## Підготовка сесії + робочого простору

- Робочий простір визначається і створюється; запуск у sandbox може перенаправлятися до кореня робочого простору sandbox.
- Skills завантажуються (або повторно використовуються зі знімка) і впроваджуються в env і prompt.
- Bootstrap/context файли визначаються і впроваджуються у звіт системного prompt.
- Отримується блокування запису сесії; `SessionManager` відкривається й готується до потокового передавання. Будь-який
  подальший шлях переписування, Compaction або обрізання транскрипту має отримати це саме блокування перед відкриттям або
  зміною файлу транскрипту.

## Збирання prompt + системний prompt

- Системний prompt будується з базового prompt OpenClaw, prompt Skills, bootstrap-контексту та перевизначень для окремого запуску.
- Застосовуються обмеження для конкретної моделі та резерв токенів для Compaction.
- Див. [Системний prompt](/uk/concepts/system-prompt), щоб зрозуміти, що бачить модель.

## Точки hook (де можна перехопити)

OpenClaw має дві системи hook:

- **Внутрішні hook** (hook Gateway): скрипти на основі подій для команд і подій життєвого циклу.
- **Hook Plugin**: точки розширення всередині життєвого циклу агента/інструмента та конвеєра gateway.

### Внутрішні hook (hook Gateway)

- **`agent:bootstrap`**: виконується під час побудови bootstrap-файлів до остаточного формування системного prompt.
  Використовуйте це, щоб додавати/видаляти bootstrap-файли контексту.
- **Hook команд**: `/new`, `/reset`, `/stop` та інші події команд (див. документ Hooks).

Див. [Hooks](/uk/automation/hooks), щоб ознайомитися з налаштуванням і прикладами.

### Hook Plugin (життєвий цикл агента + gateway)

Вони виконуються всередині циклу агента або конвеєра gateway:

- **`before_model_resolve`**: виконується до сесії (без `messages`), щоб детерміновано перевизначити provider/model до визначення моделі.
- **`before_prompt_build`**: виконується після завантаження сесії (з `messages`), щоб впровадити `prependContext`, `systemPrompt`, `prependSystemContext` або `appendSystemContext` перед надсиланням prompt. Використовуйте `prependContext` для динамічного тексту на кожен хід, а поля системного контексту — для стабільних вказівок, які мають перебувати в просторі системного prompt.
- **`before_agent_start`**: застарілий hook сумісності, який може виконуватися на будь-якій фазі; надавайте перевагу явним hook вище.
- **`before_agent_reply`**: виконується після вбудованих дій і до виклику LLM, дозволяючи Plugin перехопити хід і повернути синтетичну відповідь або повністю приглушити хід.
- **`agent_end`**: перевіряє фінальний список повідомлень і метадані запуску після завершення.
- **`before_compaction` / `after_compaction`**: спостерігають за циклами Compaction або додають до них анотації.
- **`before_tool_call` / `after_tool_call`**: перехоплюють параметри/результати інструментів.
- **`before_install`**: перевіряє результати вбудованого сканування і за потреби блокує встановлення skill або Plugin.
- **`tool_result_persist`**: синхронно перетворює результати інструмента перед записом у транскрипт сесії, яким володіє OpenClaw.
- **`message_received` / `message_sending` / `message_sent`**: hook вхідних і вихідних повідомлень.
- **`session_start` / `session_end`**: межі життєвого циклу сесії.
- **`gateway_start` / `gateway_stop`**: події життєвого циклу gateway.

Правила рішень hook для захисту вихідних повідомлень/інструментів:

- `before_tool_call`: `{ block: true }` є кінцевим рішенням і зупиняє обробники з нижчим пріоритетом.
- `before_tool_call`: `{ block: false }` нічого не робить і не скасовує попереднє блокування.
- `before_install`: `{ block: true }` є кінцевим рішенням і зупиняє обробники з нижчим пріоритетом.
- `before_install`: `{ block: false }` нічого не робить і не скасовує попереднє блокування.
- `message_sending`: `{ cancel: true }` є кінцевим рішенням і зупиняє обробники з нижчим пріоритетом.
- `message_sending`: `{ cancel: false }` нічого не робить і не скасовує попереднє скасування.

Див. [Hook Plugin](/uk/plugins/architecture-internals#provider-runtime-hooks), щоб ознайомитися з API hook і подробицями реєстрації.

Harness можуть адаптувати ці hook по-різному. Harness app-server Codex зберігає
hook Plugin OpenClaw як контракт сумісності для задокументованих дзеркальних
поверхонь, тоді як власні hook Codex залишаються окремим низькорівневим механізмом Codex.

## Потокове передавання + часткові відповіді

- Delta assistant передаються потоком із pi-agent-core і надсилаються як події `assistant`.
- Блокове потокове передавання може надсилати часткові відповіді або на `text_end`, або на `message_end`.
- Потокове передавання reasoning може надсилатися як окремий потік або як блокові відповіді.
- Див. [Потокове передавання](/uk/concepts/streaming), щоб ознайомитися з нарізанням на фрагменти та поведінкою блокових відповідей.

## Виконання інструментів + інструменти обміну повідомленнями

- Події початку/оновлення/завершення інструмента надсилаються в потік `tool`.
- Результати інструментів очищуються з огляду на розмір і payload зображень перед журналюванням/надсиланням.
- Надсилання інструментів обміну повідомленнями відстежуються, щоб пригнічувати дубльовані підтвердження assistant.

## Формування відповіді + пригнічення

- Фінальні payload збираються з:
  - тексту assistant (і необов’язкового reasoning)
  - вбудованих підсумків інструментів (коли verbose увімкнено й дозволено)
  - тексту помилки assistant, коли модель завершується помилкою
- Точний маркер тиші `NO_REPLY` / `no_reply` відфільтровується з вихідних
  payload.
- Дублікати інструментів обміну повідомленнями видаляються з фінального списку payload.
- Якщо не залишається payload, які можна відобразити, і інструмент завершився помилкою, надсилається
  резервна відповідь про помилку інструмента
  (якщо тільки інструмент обміну повідомленнями вже не надіслав користувачеві видиму відповідь).

## Compaction + повторні спроби

- Автоматичний Compaction надсилає події потоку `compaction` і може спричинити повторну спробу.
- Під час повторної спроби буфери в пам’яті та підсумки інструментів скидаються, щоб уникнути дубльованого виводу.
- Див. [Compaction](/uk/concepts/compaction), щоб ознайомитися з конвеєром Compaction.

## Потоки подій (зараз)

- `lifecycle`: надсилається через `subscribeEmbeddedPiSession` (і як резервний варіант через `agentCommand`)
- `assistant`: потокові delta з pi-agent-core
- `tool`: потокові події інструментів з pi-agent-core

## Обробка chat-каналу

- Delta assistant буферизуються в повідомлення chat `delta`.
- `final` chat надсилається на **lifecycle end/error**.

## Тайм-аути

- Типове значення `agent.wait`: 30s (лише очікування). Параметр `timeoutMs` перевизначає його.
- Середовище виконання агента: типове значення `agents.defaults.timeoutSeconds` — 172800s (48 годин); застосовується в таймері переривання `runEmbeddedPiAgent`.
- Тайм-аут бездіяльності LLM: `agents.defaults.llm.idleTimeoutSeconds` перериває запит до моделі, якщо до завершення вікна бездіяльності не надходять фрагменти відповіді. Установіть його явно для повільних локальних моделей або provider reasoning/tool-call; установіть `0`, щоб вимкнути. Якщо його не встановлено, OpenClaw використовує `agents.defaults.timeoutSeconds`, якщо його налаштовано, інакше `120s`. Запуски, ініційовані Cron, без явного тайм-ауту LLM або агента вимикають watchdog бездіяльності й покладаються на зовнішній тайм-аут cron.

## Де все може завершитися раніше

- Тайм-аут агента (переривання)
- AbortSignal (скасування)
- Роз’єднання Gateway або тайм-аут RPC
- Тайм-аут `agent.wait` (лише очікування, не зупиняє агента)

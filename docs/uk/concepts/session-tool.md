OpenClaw надає агентам інструменти для роботи між сесіями, перевірки стану й оркестрації sub-agent.

## Доступні інструменти

| Інструмент        | Що він робить                                                             |
| ----------------- | ------------------------------------------------------------------------- |
| `sessions_list`   | Показує список сесій з необов’язковими фільтрами (kind, label, agent, recency, preview) |
| `sessions_history`| Читає транскрипт конкретної сесії                                         |
| `sessions_send`   | Надсилає повідомлення в іншу сесію й за потреби чекає                     |
| `sessions_spawn`  | Створює ізольовану сесію sub-agent для фонової роботи                     |
| `sessions_yield`  | Завершує поточний хід і чекає на подальші результати sub-agent            |
| `subagents`       | Показує список, керує або завершує запущені sub-agent для цієї сесії      |
| `session_status`  | Показує картку в стилі `/status` і за потреби встановлює перевизначення моделі для сесії |

## Перегляд і читання сесій

`sessions_list` повертає сесії з їхнім ключем, agentId, kind, channel, model,
кількістю токенів і часовими мітками. Можна фільтрувати за kind (`main`, `group`, `cron`, `hook`,
`node`), точним `label`, точним `agentId`, текстом пошуку або давністю
(`activeMinutes`). Якщо вам потрібен тріаж у стилі поштової скриньки, інструмент також може
запитати похідний заголовок в межах видимості, фрагмент попереднього перегляду останнього повідомлення
або обмежені недавні повідомлення в кожному рядку. Похідні заголовки й попередні перегляди
створюються лише для сесій, які викликач уже може бачити відповідно до налаштованої політики
видимості інструментів сесії, тому не пов’язані сесії залишаються прихованими.

`sessions_history` отримує транскрипт розмови для конкретної сесії.
За замовчуванням результати інструментів виключені -- передайте `includeTools: true`, щоб їх побачити.
Повернене подання навмисно є обмеженим і відфільтрованим з міркувань безпеки:

- текст assistant нормалізується перед відновленням контексту:
  - thinking tags прибираються
  - блоки каркаса `<relevant-memories>` / `<relevant_memories>` прибираються
  - блоки XML-корисного навантаження викликів інструментів у звичайному тексті, такі як `<tool_call>...</tool_call>`,
    `<function_call>...</function_call>`, `<tool_calls>...</tool_calls>` і
    `<function_calls>...</function_calls>`, прибираються, включно з обрізаними
    корисними навантаженнями, які не закриваються коректно
  - понижений каркас викликів/результатів інструментів, такий як `[Tool Call: ...]`,
    `[Tool Result ...]` і `[Historical context ...]`, прибирається
  - витеклі токени керування моделлю, такі як `<|assistant|>`, інші ASCII
    токени `<|...|>` і повноширинні варіанти `<｜...｜>`, прибираються
  - некоректний XML викликів інструментів MiniMax, такий як `<invoke ...>` /
    `</minimax:tool_call>`, прибирається
- текст, схожий на облікові дані/токени, редагується перед поверненням
- довгі текстові блоки обрізаються
- дуже великі історії можуть відкидати старіші рядки або замінювати надто великий рядок на
  `[sessions_history omitted: message too large]`
- інструмент повідомляє підсумкові прапорці, такі як `truncated`, `droppedMessages`,
  `contentTruncated`, `contentRedacted` і `bytes`

Обидва інструменти приймають або **ключ сесії** (наприклад, `"main"`), або **ID сесії**
з попереднього виклику списку.

Якщо вам потрібен точний побайтовий транскрипт, перегляньте файл транскрипту на диску,
а не використовуйте `sessions_history` як необроблений дамп.

## Надсилання повідомлень між сесіями

`sessions_send` доставляє повідомлення в іншу сесію й за потреби чекає на
відповідь:

- **Надіслати без очікування:** встановіть `timeoutSeconds: 0`, щоб поставити в чергу й
  одразу повернути результат.
- **Чекати на відповідь:** задайте тайм-аут і отримайте відповідь вбудовано.

Після того як ціль відповість, OpenClaw може запустити **цикл зворотних відповідей**,
де агенти по черзі обмінюються повідомленнями (до 5 ходів). Цільовий агент може відповісти
`REPLY_SKIP`, щоб зупинити це раніше.

## Допоміжні інструменти для стану й оркестрації

`session_status` — це полегшений еквівалент `/status` для поточної
або іншої видимої сесії. Він повідомляє про використання, час, стан моделі/середовища виконання
і пов’язаний контекст фонових завдань, якщо він є. Як і `/status`, він може
дозаповнювати розріджені лічильники токенів/кешу з останнього запису використання в транскрипті, а
`model=default` скидає перевизначення для конкретної сесії.

`sessions_yield` навмисно завершує поточний хід, щоб наступне повідомлення могло бути
очікуваною подією продовження. Використовуйте його після запуску sub-agent, коли
хочете, щоб результати завершення надійшли наступним повідомленням, а не вимагали побудови циклів опитування.

`subagents` — це допоміжний інструмент площини керування для вже запущених OpenClaw
sub-agent. Він підтримує:

- `action: "list"` для перевірки активних/недавніх запусків
- `action: "steer"` для надсилання додаткових вказівок запущеному дочірньому агенту
- `action: "kill"` для зупинки одного дочірнього агента або `all`

## Запуск sub-agent

`sessions_spawn` за замовчуванням створює ізольовану сесію для фонового завдання.
Він завжди неблокувальний -- одразу повертає `runId` і
`childSessionKey`.

Основні параметри:

- `runtime: "subagent"` (типово) або `"acp"` для зовнішніх harness-агентів.
- перевизначення `model` і `thinking` для дочірньої сесії.
- `thread: true`, щоб прив’язати запуск до потоку чату (Discord, Slack тощо).
- `sandbox: "require"`, щоб примусово застосувати sandbox для дочірнього агента.
- `context: "fork"` для нативних sub-agent, коли дочірньому агенту потрібен транскрипт поточного
  запитувача; пропустіть його або використайте `context: "isolated"` для чистого дочірнього агента.

Типові leaf sub-agent не отримують інструменти сесії. Коли
`maxSpawnDepth >= 2`, orchestrator sub-agent рівня глибини 1 додатково отримують
`sessions_spawn`, `subagents`, `sessions_list` і `sessions_history`, щоб вони
могли керувати власними дочірніми агентами. Leaf-запуски все одно не отримують рекурсивних
інструментів оркестрації.

Після завершення крок announce публікує результат у канал запитувача.
Доставка завершення зберігає прив’язану маршрутизацію thread/topic, якщо вона доступна, і якщо
джерело завершення ідентифікує лише канал, OpenClaw усе одно може повторно використати збережений маршрут сесії запитувача
(`lastChannel` / `lastTo`) для прямої доставки.

Специфічну поведінку ACP див. у [ACP Agents](/uk/tools/acp-agents).

## Видимість

Інструменти сесії мають область видимості, щоб обмежувати те, що агент може бачити:

| Рівень  | Область                                    |
| ------- | ------------------------------------------ |
| `self`  | Лише поточна сесія                         |
| `tree`  | Поточна сесія + запущені sub-agent         |
| `agent` | Усі сесії цього агента                     |
| `all`   | Усі сесії (між агентами, якщо налаштовано) |

Типове значення — `tree`. Сесії в sandbox примусово обмежуються до `tree` незалежно від
конфігурації.

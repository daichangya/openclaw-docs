**Середовище виконання агента** — це компонент, який відповідає за один підготовлений цикл моделі: він отримує запит, керує виводом моделі, обробляє нативні виклики інструментів і повертає завершений хід до OpenClaw.

Середовища виконання легко сплутати з постачальниками, оскільки обидва з’являються поруч із конфігурацією моделі. Це різні рівні:

| Рівень        | Приклади                              | Що це означає                                                      |
| ------------- | ------------------------------------- | ------------------------------------------------------------------ |
| Постачальник  | `openai`, `anthropic`, `openai-codex` | Як OpenClaw автентифікується, виявляє моделі та називає посилання на моделі. |
| Модель        | `gpt-5.5`, `claude-opus-4-6`          | Модель, вибрана для ходу агента.                                   |
| Середовище виконання агента | `pi`, `codex`, ACP-backed runtimes    | Низькорівневий цикл, який виконує підготовлений хід.               |
| Канал         | Telegram, Discord, Slack, WhatsApp    | Звідки повідомлення надходять до OpenClaw і куди виходять.         |

Ви також побачите слово **harness** у коді та конфігурації. Harness — це реалізація, яка надає середовище виконання агента. Наприклад, вбудований harness Codex реалізує середовище виконання `codex`. Ключ конфігурації все ще має назву `embeddedHarness` для сумісності, але в документації для користувачів і у виводі статусу зазвичай слід використовувати термін runtime.

## Три речі з назвою Codex

Найбільше непорозумінь виникає через три різні поверхні, які мають спільну назву Codex:

| Поверхня                                            | Назва/конфігурація в OpenClaw       | Що вона робить                                                                                     |
| --------------------------------------------------- | ----------------------------------- | -------------------------------------------------------------------------------------------------- |
| Маршрут OAuth-постачальника Codex                   | `openai-codex/*` model refs         | Використовує OAuth підписки ChatGPT/Codex через звичайний виконувач PI в OpenClaw.                |
| Нативне середовище виконання app-server Codex       | `embeddedHarness.runtime: "codex"`  | Виконує вбудований хід агента через комплектний harness app-server Codex.                          |
| Адаптер ACP для Codex                               | `runtime: "acp"`, `agentId: "codex"` | Запускає Codex через зовнішню площину керування ACP/acpx. Використовуйте лише тоді, коли ACP/acpx запитано явно. |
| Нативний набір команд керування чатом Codex         | `/codex ...`                        | Прив’язує, відновлює, спрямовує, зупиняє та перевіряє потоки app-server Codex із чату.            |
| Маршрут API OpenAI Platform для моделей у стилі GPT/Codex | `openai/*` model refs               | Використовує автентифікацію ключем API OpenAI, якщо тільки перевизначення runtime, наприклад `runtime: "codex"`, не виконує хід. |

Ці поверхні навмисно незалежні. Увімкнення plugin `codex` робить доступними нативні можливості app-server; воно не переписує
`openai-codex/*` на `openai/*`, не змінює наявні сесії та не робить ACP типовим режимом для Codex. Вибір `openai-codex/*` означає «використовувати маршрут OAuth-постачальника Codex», якщо ви окремо не примусите runtime.

Типове налаштування Codex використовує постачальника `openai` із середовищем виконання `codex`:

```json5
{
  agents: {
    defaults: {
      model: "openai/gpt-5.5",
      embeddedHarness: {
        runtime: "codex",
      },
    },
  },
}
```

Це означає, що OpenClaw вибирає посилання на модель OpenAI, а потім просить середовище виконання app-server Codex виконати вбудований хід агента. Це не означає, що канал, каталог постачальників моделей або сховище сесій OpenClaw стає Codex.

Коли вбудований plugin `codex` увімкнено, керування Codex природною мовою слід виконувати через нативну поверхню команд `/codex` (`/codex bind`, `/codex threads`,
`/codex resume`, `/codex steer`, `/codex stop`), а не через ACP. Використовуйте ACP для Codex лише тоді, коли користувач явно просить ACP/acpx або тестує шлях адаптера ACP. Claude Code, Gemini CLI, OpenCode, Cursor та подібні зовнішні harness, як і раніше, використовують ACP.

Ось дерево рішень для агента:

1. Якщо користувач просить **Codex bind/control/thread/resume/steer/stop**, використовуйте
   нативну поверхню команд `/codex`, коли вбудований plugin `codex` увімкнено.
2. Якщо користувач просить **Codex як вбудоване середовище виконання**, використовуйте
   `openai/<model>` з `embeddedHarness.runtime: "codex"`.
3. Якщо користувач просить **Codex OAuth/subscription auth у звичайному виконувачі OpenClaw**,
   використовуйте `openai-codex/<model>` і залиште runtime як PI.
4. Якщо користувач явно вказує **ACP**, **acpx** або **адаптер ACP для Codex**,
   використовуйте ACP з `runtime: "acp"` і `agentId: "codex"`.
5. Якщо запит стосується **Claude Code, Gemini CLI, OpenCode, Cursor, Droid або
   іншого зовнішнього harness**, використовуйте ACP/acpx, а не нативне середовище виконання підлеглого агента.

| Ви маєте на увазі...                     | Використовуйте...                              |
| ---------------------------------------- | ---------------------------------------------- |
| Керування чатом/потоками app-server Codex | `/codex ...` із вбудованого plugin `codex`     |
| Вбудоване середовище виконання агента app-server Codex | `embeddedHarness.runtime: "codex"`             |
| OpenAI Codex OAuth у виконувачі PI       | `openai-codex/*` model refs                    |
| Claude Code або інший зовнішній harness  | ACP/acpx                                       |

Про розділення префіксів сімейства OpenAI див. [OpenAI](/uk/providers/openai) і
[Постачальники моделей](/uk/concepts/model-providers). Про контракт підтримки середовища виконання Codex див. [Codex harness](/uk/plugins/codex-harness#v1-support-contract).

## Відповідальність runtime

Різні середовища виконання відповідають за різний обсяг циклу.

| Поверхня                    | Вбудований OpenClaw PI                   | app-server Codex                                                            |
| --------------------------- | ---------------------------------------- | --------------------------------------------------------------------------- |
| Власник циклу моделі        | OpenClaw через вбудований виконувач PI   | app-server Codex                                                            |
| Канонічний стан потоку      | Транскрипт OpenClaw                      | Потік Codex плюс дзеркало транскрипту OpenClaw                              |
| Динамічні інструменти OpenClaw | Нативний цикл інструментів OpenClaw      | Проксіюються через адаптер Codex                                            |
| Нативні shell- і файлові інструменти | Шлях PI/OpenClaw                         | Нативні інструменти Codex, проксійовані через нативні hooks, де це підтримується |
| Механізм контексту          | Нативне збирання контексту OpenClaw      | Зібраний OpenClaw контекст проєктується в хід Codex                         |
| Compaction                  | OpenClaw або вибраний механізм контексту | Нативний Compaction Codex, зі сповіщеннями OpenClaw і підтримкою дзеркала   |
| Доставка каналом            | OpenClaw                                 | OpenClaw                                                                    |

Цей поділ відповідальності — головне правило проєктування:

- Якщо за поверхню відповідає OpenClaw, OpenClaw може забезпечити звичайну поведінку hooks plugin.
- Якщо за поверхню відповідає нативне runtime, OpenClaw потребує подій runtime або нативних hooks.
- Якщо нативне runtime володіє канонічним станом потоку, OpenClaw має дзеркалити й проєктувати контекст, а не переписувати непідтримувані внутрішні механізми.

## Вибір runtime

OpenClaw вибирає вбудоване середовище виконання після визначення постачальника та моделі:

1. Перевага надається runtime, записаному в сесії. Зміни конфігурації не перемикають «на льоту»
   наявний транскрипт на іншу нативну систему потоків.
2. `OPENCLAW_AGENT_RUNTIME=<id>` примусово встановлює це runtime для нових або скинутих сесій.
3. `agents.defaults.embeddedHarness.runtime` або
   `agents.list[].embeddedHarness.runtime` можуть встановити `auto`, `pi` або зареєстрований
   ідентифікатор runtime, наприклад `codex`.
4. У режимі `auto` зареєстровані plugin runtime можуть заявляти підтримувані пари постачальник/модель.
5. Якщо жодне runtime не приймає хід у режимі `auto` і встановлено
   `fallback: "pi"` (типове значення), OpenClaw використовує PI як сумісний резервний варіант. Встановіть
   `fallback: "none"`, щоб непідібраний вибір у режимі `auto` завершувався помилкою.

Явні plugin runtime за замовчуванням працюють у режимі fail closed. Наприклад,
`runtime: "codex"` означає Codex або чітку помилку вибору, якщо тільки ви не встановите
`fallback: "pi"` у тій самій області перевизначення. Перевизначення runtime не успадковує
ширше налаштування fallback, тому `runtime: "codex"` на рівні агента не буде мовчки
повернено до PI лише тому, що в типових значеннях було `fallback: "pi"`.

Режим `auto` навмисно консервативний. Plugin runtime можуть заявляти
пари постачальник/модель, які вони розуміють, але plugin Codex не заявляє
постачальника `openai-codex` у режимі `auto`. Це зберігає
`openai-codex/*` як явний маршрут PI Codex OAuth і запобігає непомітному
перенесенню конфігурацій із автентифікацією підпискою на нативний harness app-server.

Якщо `openclaw doctor` попереджає, що plugin `codex` увімкнено, але
`openai-codex/*` усе ще маршрутизується через PI, сприймайте це як діагностику, а не як
міграцію. Не змінюйте конфігурацію, якщо вам потрібен саме PI Codex OAuth.
Перемикайтеся на `openai/<model>` плюс `runtime: "codex"` лише тоді, коли вам потрібне нативне
виконання через app-server Codex.

## Контракт сумісності

Коли runtime — не PI, воно має документувати, які поверхні OpenClaw підтримує.
Для документації runtime використовуйте таку форму:

| Питання                               | Чому це важливо                                                                                  |
| ------------------------------------- | ------------------------------------------------------------------------------------------------ |
| Хто володіє циклом моделі?            | Визначає, де відбуваються повторні спроби, продовження інструментів і ухвалення рішення про фінальну відповідь. |
| Хто володіє канонічною історією потоку? | Визначає, чи може OpenClaw редагувати історію, чи лише дзеркалити її.                            |
| Чи працюють динамічні інструменти OpenClaw? | На цьому залежать повідомлення, сесії, cron і інструменти, якими володіє OpenClaw.               |
| Чи працюють hooks динамічних інструментів? | Plugin очікують `before_tool_call`, `after_tool_call` і middleware навколо інструментів, якими володіє OpenClaw. |
| Чи працюють hooks нативних інструментів? | Shell, patch та інструменти, якими володіє runtime, потребують підтримки нативних hooks для політик і спостереження. |
| Чи виконується життєвий цикл механізму контексту? | Plugin пам’яті й контексту залежать від життєвого циклу assemble, ingest, after-turn і compaction. |
| Які дані Compaction доступні?         | Деяким plugin потрібні лише сповіщення, тоді як іншим потрібні метадані про збережене/відкинуте. |
| Що навмисно не підтримується?         | Користувачі не повинні припускати еквівалентність PI там, де нативне runtime володіє більшим станом. |

Контракт підтримки середовища виконання Codex задокументовано в
[Codex harness](/uk/plugins/codex-harness#v1-support-contract).

## Мітки статусу

У виводі статусу можуть відображатися мітки `Execution` і `Runtime`. Сприймайте їх як
діагностику, а не як назви постачальників.

- Посилання на модель, наприклад `openai/gpt-5.5`, показує вибраного постачальника/модель.
- Ідентифікатор runtime, наприклад `codex`, показує, який цикл виконує хід.
- Мітка каналу, наприклад Telegram або Discord, показує, де відбувається розмова.

Якщо сесія все ще показує PI після зміни конфігурації runtime, почніть нову сесію
через `/new` або очистьте поточну через `/reset`. Наявні сесії зберігають своє
записане runtime, щоб один транскрипт не відтворювався через дві несумісні нативні
системи сесій.

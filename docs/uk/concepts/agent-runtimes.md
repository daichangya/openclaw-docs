**Середовище виконання агента** — це компонент, який керує одним підготовленим циклом моделі: він
отримує запит, керує виводом моделі, обробляє нативні виклики інструментів і повертає
завершений хід до OpenClaw.

Середовища виконання легко сплутати з провайдерами, оскільки обидва з’являються поруч із
конфігурацією моделі. Це різні рівні:

| Рівень        | Приклади                              | Що це означає                                                      |
| ------------- | ------------------------------------- | ------------------------------------------------------------------ |
| Провайдер     | `openai`, `anthropic`, `openai-codex` | Як OpenClaw автентифікується, знаходить моделі та називає посилання на моделі. |
| Модель        | `gpt-5.5`, `claude-opus-4-6`          | Модель, вибрана для ходу агента.                                   |
| Середовище виконання агента | `pi`, `codex`, `claude-cli`           | Низькорівневий цикл або бекенд, який виконує підготовлений хід.    |
| Канал         | Telegram, Discord, Slack, WhatsApp    | Звідки повідомлення надходять до OpenClaw і куди виходять.         |

У коді ви також побачите слово **harness**. Harness — це реалізація,
яка надає середовище виконання агента. Наприклад, вбудований Codex harness
реалізує середовище виконання `codex`. У публічній конфігурації використовується `agentRuntime.id`; `openclaw
doctor --fix` переписує старі ключі політики середовища виконання в цю форму.

Є дві родини середовищ виконання:

- **Вбудовані harness-и** працюють усередині підготовленого циклу агента OpenClaw. Наразі це
  вбудоване середовище виконання `pi`, а також зареєстровані harness-и Plugin, такі як
  `codex`.
- **CLI-бекенди** запускають локальний процес CLI, зберігаючи при цьому
  канонічне посилання на модель. Наприклад, `anthropic/claude-opus-4-7` з
  `agentRuntime.id: "claude-cli"` означає «вибрати модель Anthropic, виконати
  через Claude CLI». `claude-cli` не є ідентифікатором вбудованого harness-а і не повинен
  передаватися для вибору AgentHarness.

## Три речі з назвою Codex

Найбільше непорозумінь виникає через три різні поверхні з однаковою назвою Codex:

| Поверхня                                            | Назва/конфігурація в OpenClaw         | Що вона робить                                                                                     |
| --------------------------------------------------- | ------------------------------------- | -------------------------------------------------------------------------------------------------- |
| Маршрут OAuth-провайдера Codex                      | посилання на моделі `openai-codex/*`  | Використовує OAuth підписки ChatGPT/Codex через звичайний виконавець PI в OpenClaw.              |
| Нативне середовище виконання Codex app-server       | `agentRuntime.id: "codex"`            | Запускає вбудований хід агента через комплектний harness Codex app-server.                        |
| ACP-адаптер Codex                                   | `runtime: "acp"`, `agentId: "codex"`  | Запускає Codex через зовнішню площину керування ACP/acpx. Використовуйте лише коли явно запитано ACP/acpx. |
| Нативний набір команд керування чатом Codex         | `/codex ...`                          | Прив’язує, відновлює, спрямовує, зупиняє та перевіряє потоки Codex app-server із чату.            |
| Маршрут OpenAI Platform API для моделей у стилі GPT/Codex | посилання на моделі `openai/*`        | Використовує автентифікацію OpenAI API key, якщо лише перевизначення середовища виконання, наприклад `runtime: "codex"`, не запускає хід. |

Ці поверхні навмисно незалежні. Увімкнення Plugin `codex` робить
доступними нативні можливості app-server; воно не переписує
`openai-codex/*` у `openai/*`, не змінює наявні сесії й не робить
ACP типовим варіантом для Codex. Вибір `openai-codex/*` означає «використовувати маршрут
OAuth-провайдера Codex», якщо ви окремо не примусите інше середовище виконання.

Типове налаштування Codex використовує провайдера `openai` із середовищем виконання `codex`:

```json5
{
  agents: {
    defaults: {
      model: "openai/gpt-5.5",
      agentRuntime: {
        id: "codex",
      },
    },
  },
}
```

Це означає, що OpenClaw вибирає посилання на модель OpenAI, а потім просить середовище виконання
Codex app-server запустити вбудований хід агента. Це не означає, що канал,
каталог провайдерів моделей або сховище сесій OpenClaw стає Codex.

Коли комплектний Plugin `codex` увімкнено, для природномовного керування Codex
слід використовувати нативну поверхню команд `/codex` (`/codex bind`, `/codex threads`,
`/codex resume`, `/codex steer`, `/codex stop`) замість ACP. Використовуйте ACP для
Codex лише коли користувач явно просить ACP/acpx або тестує шлях ACP-адаптера.
Claude Code, Gemini CLI, OpenCode, Cursor та подібні зовнішні
harness-и й далі використовують ACP.

Ось дерево рішень для агента:

1. Якщо користувач просить **Codex bind/control/thread/resume/steer/stop**, використовуйте
   нативну поверхню команд `/codex`, коли комплектний Plugin `codex` увімкнено.
2. Якщо користувач просить **Codex як вбудоване середовище виконання**, використовуйте
   `openai/<model>` з `agentRuntime.id: "codex"`.
3. Якщо користувач просить **Codex OAuth/автентифікацію за підпискою у звичайному виконавці OpenClaw**,
   використовуйте `openai-codex/<model>` і залишайте середовище виконання як PI.
4. Якщо користувач явно каже **ACP**, **acpx** або **ACP-адаптер Codex**, використовуйте
   ACP з `runtime: "acp"` і `agentId: "codex"`.
5. Якщо запит стосується **Claude Code, Gemini CLI, OpenCode, Cursor, Droid або
   іншого зовнішнього harness-а**, використовуйте ACP/acpx, а не нативне середовище виконання субагента.

| Ви маєте на увазі...                   | Використовуйте...                              |
| -------------------------------------- | ---------------------------------------------- |
| Керування чатом/потоками Codex app-server | `/codex ...` із комплектного Plugin `codex` |
| Вбудоване середовище виконання агента Codex app-server | `agentRuntime.id: "codex"`          |
| OpenAI Codex OAuth у виконавці PI      | посилання на моделі `openai-codex/*`           |
| Claude Code або інший зовнішній harness | ACP/acpx                                      |

Щодо поділу префіксів родини OpenAI див. [OpenAI](/uk/providers/openai) і
[Провайдери моделей](/uk/concepts/model-providers). Щодо контракту підтримки середовища виконання Codex
див. [Codex harness](/uk/plugins/codex-harness#v1-support-contract).

## Володіння середовищем виконання

Різні середовища виконання керують різними частинами циклу.

| Поверхня                    | Вбудований PI OpenClaw                    | Codex app-server                                                            |
| --------------------------- | ---------------------------------------- | --------------------------------------------------------------------------- |
| Хто керує циклом моделі     | OpenClaw через вбудований виконавець PI  | Codex app-server                                                            |
| Канонічний стан потоку      | транскрипт OpenClaw                      | потік Codex плюс дзеркало транскрипту OpenClaw                              |
| Динамічні інструменти OpenClaw | нативний цикл інструментів OpenClaw    | Мостом через адаптер Codex                                                  |
| Нативні shell- та файлові інструменти | шлях PI/OpenClaw                 | Нативні інструменти Codex, передані через нативні hooks, де це підтримується |
| Рушій контексту             | нативне збирання контексту OpenClaw      | контекст проєктів OpenClaw, зібраний у хід Codex                            |
| Compaction                  | OpenClaw або вибраний рушій контексту    | нативний Compaction Codex із сповіщеннями OpenClaw і підтримкою дзеркала    |
| Доставка через канал        | OpenClaw                                 | OpenClaw                                                                    |

Цей поділ володіння є головним правилом проєктування:

- Якщо поверхнею керує OpenClaw, OpenClaw може надавати звичайну поведінку hooks Plugin.
- Якщо поверхнею керує нативне середовище виконання, OpenClaw потребує подій середовища виконання або нативних hooks.
- Якщо нативне середовище виконання керує канонічним станом потоку, OpenClaw має дзеркалити й проєктувати контекст, а не переписувати непідтримувані внутрішні механізми.

## Вибір середовища виконання

OpenClaw вибирає вбудоване середовище виконання після визначення провайдера та моделі:

1. Пріоритет має записане для сесії середовище виконання. Зміни конфігурації не перемикають
   існуючий транскрипт «на гарячу» на іншу нативну систему потоків.
2. `OPENCLAW_AGENT_RUNTIME=<id>` примусово встановлює це середовище виконання для нових або скинутих сесій.
3. `agents.defaults.agentRuntime.id` або `agents.list[].agentRuntime.id` можуть встановлювати
   `auto`, `pi`, ідентифікатор зареєстрованого вбудованого harness-а, наприклад `codex`, або
   підтримуваний псевдонім CLI-бекенда, наприклад `claude-cli`.
4. У режимі `auto` зареєстровані середовища виконання Plugin можуть заявляти підтримувані
   пари провайдер/модель.
5. Якщо жодне середовище виконання не бере хід у режимі `auto`, а встановлено
   `fallback: "pi"` (типове значення), OpenClaw використовує PI як сумісний резервний варіант. Установіть
   `fallback: "none"`, щоб вибір у режимі `auto` завершувався помилкою, якщо немає відповідності.

Явно задані середовища виконання Plugin типово працюють у закритому режимі. Наприклад,
`runtime: "codex"` означає Codex або чітку помилку вибору, якщо лише ви не встановите
`fallback: "pi"` у тій самій області перевизначення. Перевизначення середовища виконання не успадковує
ширше налаштування fallback, тому `runtime: "codex"` на рівні агента не буде мовчки
спрямовано назад до PI лише тому, що в значеннях за замовчуванням було `fallback: "pi"`.

Псевдоніми CLI-бекендів відрізняються від ідентифікаторів вбудованих harness-ів. Бажана
форма для Claude CLI така:

```json5
{
  agents: {
    defaults: {
      model: "anthropic/claude-opus-4-7",
      agentRuntime: { id: "claude-cli" },
    },
  },
}
```

Застарілі посилання на кшталт `claude-cli/claude-opus-4-7` і далі підтримуються для
сумісності, але в новій конфігурації слід зберігати канонічну форму провайдера/моделі, а
бекенд виконання вказувати в `agentRuntime.id`.

Режим `auto` навмисно консервативний. Середовища виконання Plugin можуть заявляти
пари провайдер/модель, які вони розуміють, але Plugin Codex не заявляє
провайдера `openai-codex` у режимі `auto`. Це зберігає
`openai-codex/*` як явний маршрут PI Codex OAuth і не дає мовчки
перемістити конфігурації з автентифікацією за підпискою на нативний harness app-server.

Якщо `openclaw doctor` попереджає, що Plugin `codex` увімкнено, але
`openai-codex/*` усе ще маршрутизується через PI, сприймайте це як діагностику, а не
міграцію. Залишайте конфігурацію без змін, якщо вам потрібен саме PI Codex OAuth.
Переходьте на `openai/<model>` плюс `agentRuntime.id: "codex"` лише коли вам потрібне нативне
виконання через Codex app-server.

## Контракт сумісності

Коли середовище виконання — не PI, воно має документувати, які поверхні OpenClaw воно підтримує.
Для документації середовищ виконання використовуйте таку форму:

| Питання                               | Чому це важливо                                                                               |
| ------------------------------------- | --------------------------------------------------------------------------------------------- |
| Хто керує циклом моделі?              | Визначає, де відбуваються повторні спроби, продовження інструментів і рішення щодо фінальної відповіді. |
| Хто керує канонічною історією потоку? | Визначає, чи може OpenClaw редагувати історію, чи лише дзеркалити її.                        |
| Чи працюють динамічні інструменти OpenClaw? | На це спираються повідомлення, сесії, Cron та інструменти, що належать OpenClaw.      |
| Чи працюють hooks динамічних інструментів? | Plugins очікують `before_tool_call`, `after_tool_call` і middleware навколо інструментів, що належать OpenClaw. |
| Чи працюють hooks нативних інструментів? | Інструментам shell, patch та інструментам, що належать середовищу виконання, потрібна підтримка нативних hooks для політик і спостереження. |
| Чи виконується життєвий цикл рушія контексту? | Plugins пам’яті та контексту залежать від життєвого циклу assemble, ingest, after-turn і Compaction. |
| Які дані Compaction відкриті?         | Деяким Plugins потрібні лише сповіщення, а іншим — метадані про збережене/відкинуте.        |
| Що навмисно не підтримується?         | Користувачі не повинні припускати еквівалентність із PI там, де нативне середовище виконання керує більшою частиною стану. |

Контракт підтримки середовища виконання Codex задокументовано в
[Codex harness](/uk/plugins/codex-harness#v1-support-contract).

## Мітки status

У виводі status можуть показуватися мітки `Execution` і `Runtime`. Сприймайте їх як
діагностику, а не як назви провайдерів.

- Посилання на модель, наприклад `openai/gpt-5.5`, повідомляє про вибраного провайдера/модель.
- Ідентифікатор середовища виконання, наприклад `codex`, повідомляє, який цикл виконує цей хід.
- Мітка каналу, наприклад Telegram або Discord, повідомляє, де відбувається розмова.

Якщо після зміни конфігурації середовища виконання сесія все ще показує PI, почніть нову сесію
через `/new` або очистьте поточну через `/reset`. Наявні сесії зберігають своє
записане середовище виконання, щоб транскрипт не відтворювався через дві несумісні нативні
системи сесій.

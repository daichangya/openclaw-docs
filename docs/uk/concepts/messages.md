# Повідомлення

Ця сторінка поєднує в собі те, як OpenClaw обробляє вхідні повідомлення, сесії, постановку в чергу,
потокове передавання та видимість міркувань.

## Потік повідомлень (загальний огляд)

```
Inbound message
  -> routing/bindings -> session key
  -> queue (if a run is active)
  -> agent run (streaming + tools)
  -> outbound replies (channel limits + chunking)
```

Ключові параметри розміщені в конфігурації:

- `messages.*` для префіксів, постановки в чергу та поведінки в групах.
- `agents.defaults.*` для типових значень потокового передавання блоків і розбиття на частини.
- Перевизначення каналів (`channels.whatsapp.*`, `channels.telegram.*` тощо) для лімітів і перемикачів потокового передавання.

Див. [Configuration](/uk/gateway/configuration) для повної схеми.

## Усунення дублювання вхідних повідомлень

Канали можуть повторно доставляти те саме повідомлення після повторного підключення. OpenClaw зберігає
короткочасний кеш, прив’язаний до channel/account/peer/session/message id, щоб дубльовані
доставлення не запускали ще один запуск агента.

## Згладжування вхідних повідомлень

Швидкі послідовні повідомлення від **того самого відправника** можуть об’єднуватися в один
хід агента через `messages.inbound`. Згладжування застосовується в межах channel + conversation
і використовує найновіше повідомлення для прив’язки відповіді/ID.

Конфігурація (глобальне типове значення + перевизначення для окремих каналів):

```json5
{
  messages: {
    inbound: {
      debounceMs: 2000,
      byChannel: {
        whatsapp: 5000,
        slack: 1500,
        discord: 1500,
      },
    },
  },
}
```

Примітки:

- Згладжування застосовується до повідомлень **лише з текстом**; медіа/вкладення скидають буфер негайно.
- Керівні команди обходять згладжування, щоб залишатися окремими — **окрім** випадків, коли канал явно вмикає об’єднання DM від того самого відправника (наприклад, [BlueBubbles `coalesceSameSenderDms`](/uk/channels/bluebubbles#coalescing-split-send-dms-command--url-in-one-composition)), де команди в DM очікують у межах вікна згладжування, щоб розділений корисний вміст міг приєднатися до того самого ходу агента.

## Сесії та пристрої

Сесіями володіє Gateway, а не клієнти.

- Прямі чати зводяться до ключа основної сесії агента.
- Групи/канали отримують власні ключі сесій.
- Сховище сесій і транскрипти розташовані на хості Gateway.

Кілька пристроїв/каналів можуть відповідати одній і тій самій сесії, але історія не повністю
синхронізується назад до кожного клієнта. Рекомендація: використовуйте один основний пристрій для тривалих
розмов, щоб уникнути розбіжного контексту. Control UI і TUI завжди показують
транскрипт сесії, що підтримується Gateway, тому вони є джерелом істини.

Докладніше: [Session management](/uk/concepts/session).

## Вхідні тіла повідомлень і контекст історії

OpenClaw розділяє **тіло підказки** і **тіло команди**:

- `Body`: текст підказки, надісланий агенту. Він може містити обгортки каналу та
  необов’язкові обгортки історії.
- `CommandBody`: сирий текст користувача для розбору директив/команд.
- `RawBody`: застарілий псевдонім для `CommandBody` (збережений для сумісності).

Коли канал надає історію, використовується спільна обгортка:

- `[Chat messages since your last reply - for context]`
- `[Current message - respond to this]`

Для **непрямих чатів** (груп/каналів/кімнат) **тіло поточного повідомлення** має префікс із
міткою відправника (той самий стиль, що використовується для записів історії). Це забезпечує узгодженість
між повідомленнями реального часу та повідомленнями з черги/історії в підказці агента.

Буфери історії є **лише очікувальними**: вони містять групові повідомлення, які _не_
запустили виконання (наприклад, повідомлення, обмежені згадками), і **не включають** повідомлення,
які вже є в транскрипті сесії.

Видалення директив застосовується лише до розділу **поточного повідомлення**, щоб історія
залишалася незмінною. Канали, які обгортають історію, мають встановлювати `CommandBody` (або
`RawBody`) в оригінальний текст повідомлення та зберігати `Body` як об’єднану підказку.
Буфери історії налаштовуються через `messages.groupChat.historyLimit` (глобальне
типове значення) і перевизначення для окремих каналів, як-от `channels.slack.historyLimit` або
`channels.telegram.accounts.<id>.historyLimit` (встановіть `0`, щоб вимкнути).

## Постановка в чергу та подальші повідомлення

Якщо запуск уже активний, вхідні повідомлення можна поставити в чергу, спрямувати в поточний
запуск або зібрати для подальшого ходу.

- Налаштовується через `messages.queue` (і `messages.queue.byChannel`).
- Режими: `interrupt`, `steer`, `followup`, `collect`, а також варіанти backlog.

Докладніше: [Queueing](/uk/concepts/queue).

## Потокове передавання, розбиття на частини та пакетування

Потокове передавання блоків надсилає часткові відповіді в міру того, як модель продукує текстові блоки.
Розбиття на частини враховує текстові обмеження каналу та уникає розриву огородженого коду.

Ключові налаштування:

- `agents.defaults.blockStreamingDefault` (`on|off`, типово off)
- `agents.defaults.blockStreamingBreak` (`text_end|message_end`)
- `agents.defaults.blockStreamingChunk` (`minChars|maxChars|breakPreference`)
- `agents.defaults.blockStreamingCoalesce` (пакетування на основі періодів бездіяльності)
- `agents.defaults.humanDelay` (людиноподібна пауза між відповідями блоками)
- Перевизначення каналів: `*.blockStreaming` і `*.blockStreamingCoalesce` (канали, окрім Telegram, потребують явного `*.blockStreaming: true`)

Докладніше: [Streaming + chunking](/uk/concepts/streaming).

## Видимість міркувань і токени

OpenClaw може показувати або приховувати міркування моделі:

- `/reasoning on|off|stream` керує видимістю.
- Вміст міркувань усе одно враховується у використанні токенів, якщо його продукує модель.
- Telegram підтримує потокове передавання міркувань у чернетку.

Докладніше: [Thinking + reasoning directives](/uk/tools/thinking) і [Token use](/uk/reference/token-use).

## Префікси, треди та відповіді

Форматування вихідних повідомлень централізоване в `messages`:

- `messages.responsePrefix`, `channels.<channel>.responsePrefix` і `channels.<channel>.accounts.<id>.responsePrefix` (каскад вихідних префіксів), а також `channels.whatsapp.messagePrefix` (вхідний префікс WhatsApp)
- Прив’язка відповідей через `replyToMode` і типові значення для окремих каналів

Докладніше: [Configuration](/uk/gateway/configuration-reference#messages) і документацію каналів.

## Тихі відповіді

Точний тихий токен `NO_REPLY` / `no_reply` означає «не доставляти видиму для користувача відповідь».
OpenClaw визначає цю поведінку за типом розмови:

- Прямі розмови типово не дозволяють тишу й переписують голу тиху
  відповідь на короткий видимий запасний варіант.
- Групи/канали типово дозволяють тишу.
- Внутрішня оркестрація типово дозволяє тишу.

Типові значення розміщені в `agents.defaults.silentReply` і
`agents.defaults.silentReplyRewrite`; `surfaces.<id>.silentReply` і
`surfaces.<id>.silentReplyRewrite` можуть перевизначати їх для кожної поверхні окремо.

Коли батьківська сесія має один або більше дочірніх запусків субагентів, що очікують завершення,
голі тихі відповіді відкидаються на всіх поверхнях замість переписування, щоб
батьківський процес залишався тихим, доки подія завершення дочірнього процесу не доставить справжню відповідь.

OpenClaw запускає єдине вбудоване середовище виконання агента.

## Робочий простір (обов’язково)

OpenClaw використовує єдиний каталог робочого простору агента (`agents.defaults.workspace`) як **єдиний** робочий каталог агента (`cwd`) для інструментів і контексту.

Рекомендовано: використайте `openclaw setup`, щоб створити `~/.openclaw/openclaw.json`, якщо його немає, і ініціалізувати файли робочого простору.

Повний макет робочого простору + посібник із резервного копіювання: [Робочий простір агента](/uk/concepts/agent-workspace)

Якщо `agents.defaults.sandbox` увімкнено, сеанси, відмінні від основного, можуть перевизначати це за допомогою
робочих просторів для окремих сеансів у `agents.defaults.sandbox.workspaceRoot` (див.
[Конфігурація Gateway](/uk/gateway/configuration)).

## Файли початкового налаштування (інжектуються)

Усередині `agents.defaults.workspace` OpenClaw очікує такі файли, які може редагувати користувач:

- `AGENTS.md` — інструкції з роботи + «пам’ять»
- `SOUL.md` — персона, межі, тон
- `TOOLS.md` — нотатки щодо інструментів, які підтримує користувач (наприклад, `imsg`, `sag`, conventions)
- `BOOTSTRAP.md` — одноразовий ритуал першого запуску (видаляється після завершення)
- `IDENTITY.md` — ім’я/вайб/емодзі агента
- `USER.md` — профіль користувача + бажана форма звертання

На першому ході нового сеансу OpenClaw безпосередньо інжектує вміст цих файлів у контекст агента.

Порожні файли пропускаються. Великі файли обрізаються і скорочуються з маркером, щоб промпти залишалися компактними (прочитайте файл, щоб побачити повний вміст).

Якщо файл відсутній, OpenClaw інжектує один рядок-маркер «файл відсутній» (а `openclaw setup` створить безпечний шаблон за замовчуванням).

`BOOTSTRAP.md` створюється лише для **абсолютно нового робочого простору** (коли інші bootstrap-файли відсутні). Якщо ви видалите його після завершення ритуалу, його не слід створювати знову під час наступних перезапусків.

Щоб повністю вимкнути створення bootstrap-файлів (для попередньо підготовлених робочих просторів), задайте:

```json5
{ agent: { skipBootstrap: true } }
```

## Вбудовані інструменти

Основні інструменти (`read`/`exec`/`edit`/`write` і пов’язані системні інструменти) завжди доступні,
з урахуванням політики інструментів. `apply_patch` є необов’язковим і контролюється через
`tools.exec.applyPatch`. `TOOLS.md` **не** керує тим, які інструменти існують; це
вказівки щодо того, як _ви_ хочете їх використовувати.

## Skills

OpenClaw завантажує Skills із таких розташувань (найвищий пріоритет першим):

- Робочий простір: `<workspace>/skills`
- Skills агента проєкту: `<workspace>/.agents/skills`
- Особисті Skills агента: `~/.agents/skills`
- Керовані/локальні: `~/.openclaw/skills`
- Вбудовані (постачаються з установкою)
- Додаткові каталоги Skills: `skills.load.extraDirs`

Skills можуть контролюватися через config/env (див. `skills` у [Конфігурація Gateway](/uk/gateway/configuration)).

## Межі середовища виконання

Вбудоване середовище виконання агента побудоване на ядрі агента Pi (моделі, інструменти та
конвеєр промптів). Керування сеансами, виявлення, підключення інструментів і доставка через канали —
це шари, що належать OpenClaw і працюють поверх цього ядра.

## Сеанси

Транскрипти сеансів зберігаються у форматі JSONL за шляхом:

- `~/.openclaw/agents/<agentId>/sessions/<SessionId>.jsonl`

ID сеансу є стабільним і вибирається OpenClaw.
Застарілі каталоги сеансів з інших інструментів не зчитуються.

## Керування під час потокової передачі

Коли режим черги має значення `steer`, вхідні повідомлення інжектуються в поточний запуск.
Поставлені в чергу керувальні повідомлення доставляються **після того, як поточний хід асистента
завершить виконання своїх викликів інструментів**, перед наступним викликом LLM. Керування більше не пропускає
решту викликів інструментів із поточного повідомлення асистента; натомість воно інжектує повідомлення з черги на наступній межі моделі.

Коли режим черги має значення `followup` або `collect`, вхідні повідомлення утримуються до завершення
поточного ходу, після чого починається новий хід агента з даними з черги. Див.
[Черга](/uk/concepts/queue) для поведінки режимів, debounce і cap.

Блокова потокова передача надсилає завершені блоки асистента щойно вони завершуються; вона
**вимкнена за замовчуванням** (`agents.defaults.blockStreamingDefault: "off"`).
Налаштуйте межу через `agents.defaults.blockStreamingBreak` (`text_end` або `message_end`; типово `text_end`).
Керуйте м’яким поділом блоків на частини за допомогою `agents.defaults.blockStreamingChunk` (типово
800–1200 символів; пріоритет — розриви абзаців, потім нові рядки, речення — в останню чергу).
Об’єднуйте потокові частини через `agents.defaults.blockStreamingCoalesce`, щоб зменшити
спам з одного рядка (об’єднання на основі простою перед надсиланням). Для каналів, відмінних від Telegram,
потрібно явно ввімкнути `*.blockStreaming: true`, щоб активувати блокові відповіді.
Розгорнуті зведення інструментів надсилаються на початку запуску інструмента (без debounce); Control UI
передає вивід інструментів через події агента, коли це доступно.
Докладніше: [Потокова передача + поділ на частини](/uk/concepts/streaming).

## Посилання на моделі

Посилання на моделі в конфігурації (наприклад, `agents.defaults.model` і `agents.defaults.models`) аналізуються шляхом поділу за **першим** символом `/`.

- Використовуйте `provider/model` під час налаштування моделей.
- Якщо сам ID моделі містить `/` (у стилі OpenRouter), додавайте префікс провайдера (приклад: `openrouter/moonshotai/kimi-k2`).
- Якщо ви не вказали провайдера, OpenClaw спочатку намагається використати псевдонім, потім — унікальний
  збіг налаштованого провайдера для цього точного ID моделі, і лише після цього повертається до налаштованого провайдера за замовчуванням. Якщо цей провайдер більше не надає
  налаштовану типову модель, OpenClaw переходить до першого налаштованого
  провайдера/моделі замість того, щоб показувати застаріле типове значення від видаленого провайдера.

# `openclaw status`

Діагностика для каналів і сеансів.

```bash
openclaw status
openclaw status --all
openclaw status --deep
openclaw status --usage
```

Примітки:

- `--deep` запускає живі зонди (WhatsApp Web + Telegram + Discord + Slack + Signal).
- `--usage` виводить нормалізовані вікна використання провайдера у форматі `X% left`.
- Виведення стану сеансу розділяє `Execution:` і `Runtime:`. `Execution` — це шлях sandbox (`direct`, `docker/*`), тоді як `Runtime` показує, чи використовує сеанс `OpenClaw Pi Default`, `OpenAI Codex`, бекенд CLI або бекенд ACP, наприклад `codex (acp/acpx)`. Див. [Середовища виконання агентів](/uk/concepts/agent-runtimes) щодо відмінностей між провайдером, моделлю та середовищем виконання.
- Сирі поля MiniMax `usage_percent` / `usagePercent` означають залишок квоти, тому OpenClaw інвертує їх перед відображенням; поля на основі лічильників мають пріоритет, якщо вони присутні. Відповіді `model_remains` надають перевагу запису chat-моделі, за потреби виводять мітку вікна з часових позначок і включають назву моделі до мітки плану.
- Коли поточний знімок сеансу містить мало даних, `/status` може дозаповнити лічильники токенів і кешу з найновішого журналу використання транскрипту. Наявні ненульові живі значення все одно мають пріоритет над резервними значеннями з транскрипту.
- Резервні дані з транскрипту також можуть відновити мітку моделі активного середовища виконання, якщо її немає в записі живого сеансу. Якщо ця модель із транскрипту відрізняється від вибраної моделі, status визначає контекстне вікно за відновленою моделлю середовища виконання, а не за вибраною.
- Для обліку розміру prompt резервні дані з транскрипту надають перевагу більшому загальному значенню, орієнтованому на prompt, якщо метадані сеансу відсутні або менші, щоб сеанси custom-provider не зводилися до відображення `0` токенів.
- Виведення містить сховища сеансів для кожного агента, якщо налаштовано кілька агентів.
- Огляд містить стан встановлення/виконання служби хоста Gateway + Node, якщо доступно.
- Огляд містить канал оновлення + git SHA (для checkout із джерела).
- Інформація про оновлення відображається в розділі Overview; якщо оновлення доступне, status виводить підказку виконати `openclaw update` (див. [Оновлення](/uk/install/updating)).
- Поверхні стану лише для читання (`status`, `status --json`, `status --all`) за можливості розв’язують підтримувані SecretRef для цільових шляхів конфігурації.
- Якщо підтримуваний канал SecretRef налаштовано, але він недоступний у поточному шляху команди, status лишається лише для читання і повідомляє про деградований вивід замість аварійного завершення. Людинозрозумілий вивід показує попередження на кшталт «configured token unavailable in this command path», а вивід JSON містить `secretDiagnostics`.
- Коли локальне для команди розв’язання SecretRef успішне, status надає перевагу розв’язаному знімку й прибирає тимчасові маркери каналу «secret unavailable» з фінального виводу.
- `status --all` містить рядок огляду Secrets і розділ діагностики, який підсумовує діагностику секретів (усічено для читабельності) без зупинки генерації звіту.

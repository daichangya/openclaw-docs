# `openclaw update`

Безпечно оновлюйте OpenClaw і перемикайтеся між каналами stable/beta/dev.

Якщо ви встановили через **npm/pnpm/bun** (глобальне встановлення, без метаданих git),
оновлення відбуваються через потік менеджера пакунків у [Оновлення](/uk/install/updating).

## Використання

```bash
openclaw update
openclaw update status
openclaw update wizard
openclaw update --channel beta
openclaw update --channel dev
openclaw update --tag beta
openclaw update --tag main
openclaw update --dry-run
openclaw update --no-restart
openclaw update --yes
openclaw update --json
openclaw --update
```

## Параметри

- `--no-restart`: пропустити перезапуск сервісу Gateway після успішного оновлення. Оновлення через менеджер пакунків, які перезапускають Gateway, перевіряють, що перезапущений сервіс повідомляє очікувану оновлену версію, перш ніж команда завершиться успішно.
- `--channel <stable|beta|dev>`: установити канал оновлення (git + npm; зберігається в конфігурації).
- `--tag <dist-tag|version|spec>`: перевизначити цільовий пакунок лише для цього оновлення. Для встановлень через пакунок `main` зіставляється з `github:openclaw/openclaw#main`.
- `--dry-run`: попередньо показати заплановані дії оновлення (канал/тег/цільовий об’єкт/потік перезапуску) без запису конфігурації, встановлення, синхронізації plugins або перезапуску.
- `--json`: вивести машинозчитуваний JSON `UpdateRunResult`, зокрема
  `postUpdate.plugins.integrityDrifts`, якщо під час післяоновлювальної синхронізації plugins через npm
  виявлено розбіжність артефактів plugin.
- `--timeout <seconds>`: тайм-аут для кожного кроку (типово 1800 с).
- `--yes`: пропустити запити на підтвердження (наприклад, підтвердження переходу на старішу версію).

<Warning>
Переходи на старішу версію потребують підтвердження, оскільки старіші версії можуть зламати конфігурацію.
</Warning>

## `update status`

Показати активний канал оновлення + git tag/branch/SHA (для вихідних checkout), а також доступність оновлення.

```bash
openclaw update status
openclaw update status --json
openclaw update status --timeout 10
```

Параметри:

- `--json`: вивести машинозчитуваний JSON статусу.
- `--timeout <seconds>`: тайм-аут для перевірок (типово 3 с).

## `update wizard`

Інтерактивний потік для вибору каналу оновлення та підтвердження, чи перезапускати Gateway
після оновлення (типово — перезапускати). Якщо ви виберете `dev` без checkout git,
буде запропоновано його створити.

Параметри:

- `--timeout <seconds>`: тайм-аут для кожного кроку оновлення (типово `1800`)

## Що це робить

Коли ви явно перемикаєте канали (`--channel ...`), OpenClaw також підтримує
узгодженість методу встановлення:

- `dev` → забезпечує checkout git (типово: `~/openclaw`, перевизначення через `OPENCLAW_GIT_DIR`),
  оновлює його та встановлює глобальний CLI із цього checkout.
- `stable` → встановлює з npm за допомогою `latest`.
- `beta` → надає перевагу npm dist-tag `beta`, але повертається до `latest`, коли beta
  відсутній або старіший за поточний стабільний випуск.

Автооновлювач ядра Gateway (коли ввімкнений через конфігурацію) повторно використовує цей самий шлях оновлення.

Для встановлень через менеджер пакунків `openclaw update` визначає цільову версію
пакунка перед викликом менеджера пакунків. Навіть якщо встановлена версія
вже відповідає цільовій, команда оновлює глобальне встановлення пакунка,
після чого запускає синхронізацію plugins, оновлення completion і перезапуск. Це підтримує
узгодженість запакованих sidecar і записів plugins, що належать каналу, зі встановленою збіркою OpenClaw.

## Потік checkout git

### Вибір каналу

- `stable`: checkout останнього не-beta tag, потім build і doctor.
- `beta`: надає перевагу останньому tag `-beta`, але повертається до останнього stable tag, якщо beta відсутній або старіший.
- `dev`: checkout `main`, потім fetch і rebase.

### Кроки оновлення

<Steps>
  <Step title="Перевірити чистий worktree">
    Потрібна відсутність незакомічених змін.
  </Step>
  <Step title="Перемкнути канал">
    Перемикає на вибраний канал (tag або branch).
  </Step>
  <Step title="Отримати зміни з upstream">
    Лише для dev.
  </Step>
  <Step title="Попередній build (лише dev)">
    Запускає lint і build TypeScript у тимчасовому worktree. Якщо вершина не проходить перевірку, відступає назад до 10 комітів, щоб знайти найновішу чисту збірку.
  </Step>
  <Step title="Rebase">
    Виконує rebase на вибраний коміт (лише dev).
  </Step>
  <Step title="Установити залежності">
    Використовує менеджер пакунків репозиторію. Для checkout із pnpm оновлювач за потреби ініціалізує `pnpm` (спочатку через `corepack`, потім через тимчасовий запасний варіант `npm install pnpm@10`) замість запуску `npm run build` усередині workspace pnpm.
  </Step>
  <Step title="Зібрати Control UI">
    Збирає gateway і Control UI.
  </Step>
  <Step title="Запустити doctor">
    `openclaw doctor` запускається як фінальна перевірка безпечного оновлення.
  </Step>
  <Step title="Синхронізувати plugins">
    Синхронізує plugins з активним каналом. Dev використовує вбудовані plugins; stable і beta використовують npm. Оновлює plugins, установлені через npm.
  </Step>
</Steps>

<Warning>
Якщо оновлення точно закріпленого npm plugin розв’язується до артефакту, цілісність якого відрізняється від збереженого запису встановлення, `openclaw update` перериває оновлення артефакту цього plugin замість його встановлення. Перевстановлюйте або оновлюйте plugin явно лише після перевірки, що ви довіряєте новому артефакту.
</Warning>

<Note>
Збої післяоновлювальної синхронізації plugins призводять до помилки результату оновлення й зупиняють подальші дії з перезапуском. Виправте помилку встановлення або оновлення plugin, а потім повторно запустіть `openclaw update`.

Якщо ініціалізація pnpm усе ще не вдається, оновлювач зупиняється раніше з помилкою, специфічною для менеджера пакунків, замість спроби `npm run build` усередині checkout.
</Note>

## Скорочення `--update`

`openclaw --update` переписується на `openclaw update` (зручно для оболонок і скриптів запуску).

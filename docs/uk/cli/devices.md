# `openclaw devices`

Керуйте запитами на сполучення пристроїв і токенами, прив’язаними до пристроїв.

## Команди

### `openclaw devices list`

Показати запити на сполучення, що очікують схвалення, і сполучені пристрої.

```
openclaw devices list
openclaw devices list --json
```

Вивід запитів, що очікують схвалення, показує запитуваний доступ поруч із поточним
схваленим доступом пристрою, якщо пристрій уже сполучений. Це робить
підвищення обсягу доступу/ролі явним, а не створює враження, ніби сполучення було втрачено.

### `openclaw devices remove <deviceId>`

Видалити один запис про сполучений пристрій.

Коли ви автентифіковані за допомогою токена сполученого пристрою, викликачі без прав адміністратора
можуть видалити лише запис **власного** пристрою. Видалення іншого пристрою вимагає
`operator.admin`.

```
openclaw devices remove <deviceId>
openclaw devices remove <deviceId> --json
```

### `openclaw devices clear --yes [--pending]`

Масово очистити сполучені пристрої.

```
openclaw devices clear --yes
openclaw devices clear --yes --pending
openclaw devices clear --yes --pending --json
```

### `openclaw devices approve [requestId] [--latest]`

Схвалити запит на сполучення пристрою, що очікує схвалення, за точним `requestId`. Якщо `requestId`
не вказано або передано `--latest`, OpenClaw лише показує вибраний запит,
що очікує схвалення, і завершує роботу; повторно запустіть схвалення з точним
ідентифікатором запиту після перевірки деталей.

Примітка: якщо пристрій повторює спробу сполучення зі зміненими даними автентифікації (роль/обсяги доступу/публічний
ключ), OpenClaw замінює попередній запис, що очікує схвалення, і видає новий
`requestId`. Запускайте `openclaw devices list` безпосередньо перед схваленням, щоб використати
поточний ідентифікатор.

Якщо пристрій уже сполучений і запитує ширші обсяги доступу або ширшу роль,
OpenClaw зберігає чинне схвалення та створює новий запит на підвищення,
що очікує схвалення. Перегляньте стовпці `Requested` і `Approved` у `openclaw devices list`
або скористайтеся `openclaw devices approve --latest`, щоб попередньо переглянути точне підвищення
перед його схваленням.

Якщо Gateway явно налаштовано з
`gateway.nodes.pairing.autoApproveCidrs`, запити `role: node` від нових пристроїв із
клієнтських IP-адрес, що відповідають умові, можуть бути схвалені до того, як вони з’являться в цьому списку. Ця політика
типово вимкнена й ніколи не застосовується до клієнтів operator/browser або запитів на підвищення.

```
openclaw devices approve
openclaw devices approve <requestId>
openclaw devices approve --latest
```

### `openclaw devices reject <requestId>`

Відхилити запит на сполучення пристрою, що очікує схвалення.

```
openclaw devices reject <requestId>
```

### `openclaw devices rotate --device <id> --role <role> [--scope <scope...>]`

Ротувати токен пристрою для конкретної ролі (за потреби також оновивши обсяги доступу).
Цільова роль уже має існувати в схваленому контракті сполучення цього пристрою;
ротація не може випустити нову несхвалену роль.
Якщо ви не вкажете `--scope`, подальші повторні підключення зі збереженим ротованим токеном повторно використовуватимуть
кешовані схвалені обсяги доступу цього токена. Якщо ви передасте явні значення `--scope`, вони
стануть збереженим набором обсягів доступу для майбутніх повторних підключень із кешованим токеном.
Викликачі без прав адміністратора зі сполученого пристрою можуть ротувати лише токен **власного** пристрою.
Цільовий набір обсягів доступу токена має також залишатися в межах власних операторських
обсягів доступу сеансу викликача; ротація не може випустити або зберегти ширший операторський токен, ніж той,
який викликач уже має.

```
openclaw devices rotate --device <deviceId> --role operator --scope operator.read --scope operator.write
```

Повертає нове корисне навантаження токена у форматі JSON.

### `openclaw devices revoke --device <id> --role <role>`

Відкликати токен пристрою для конкретної ролі.

Викликачі без прав адміністратора зі сполученого пристрою можуть відкликати лише токен **власного** пристрою.
Відкликання токена іншого пристрою вимагає `operator.admin`.
Цільовий набір обсягів доступу токена також має вкладатися у власні
операторські обсяги доступу сеансу викликача; викликачі лише зі сполученням не можуть відкликати операторські токени admin/write.

```
openclaw devices revoke --device <deviceId> --role node
```

Повертає результат відкликання у форматі JSON.

## Загальні параметри

- `--url <url>`: URL WebSocket Gateway (типово `gateway.remote.url`, якщо налаштовано).
- `--token <token>`: токен Gateway (за потреби).
- `--password <password>`: пароль Gateway (автентифікація паролем).
- `--timeout <ms>`: тайм-аут RPC.
- `--json`: вивід JSON (рекомендовано для скриптів).

Примітка: коли ви задаєте `--url`, CLI не використовує конфігурацію або облікові дані з середовища як резервний варіант.
Явно передайте `--token` або `--password`. Відсутність явно вказаних облікових даних є помилкою.

## Примітки

- Ротація токена повертає новий токен (чутливі дані). Поводьтеся з ним як із секретом.
- Для цих команд потрібен обсяг доступу `operator.pairing` (або `operator.admin`).
- `gateway.nodes.pairing.autoApproveCidrs` — це опційна політика Gateway для
  сполучення лише нових пристроїв node; вона не змінює повноваження схвалення в CLI.
- Ротація та відкликання токенів залишаються в межах схваленого набору ролей сполучення та
  схваленої базової лінії обсягів доступу для цього пристрою. Випадковий кешований запис токена не
  надає ціль для керування токеном.
- Для сеансів токенів сполучених пристроїв міжпристроєве керування доступне лише адміністраторам:
  `remove`, `rotate` і `revoke` дозволені лише для власного пристрою, якщо викликач не має
  `operator.admin`.
- Зміна токена також обмежена обсягами доступу викликача: сеанс лише зі сполученням не може
  ротувати або відкликати токен, який наразі має `operator.admin` або
  `operator.write`.
- `devices clear` навмисно захищено параметром `--yes`.
- Якщо обсяг доступу до сполучення недоступний у local loopback (і явний `--url` не передано), list/approve можуть використовувати локальний резервний варіант для сполучення.
- `devices approve` вимагає явного ідентифікатора запиту перед випуском токенів; якщо не вказати `requestId` або передати `--latest`, буде лише показано найновіший запит, що очікує схвалення.

## Контрольний список відновлення при розсинхронізації токенів

Використовуйте це, коли Control UI або інші клієнти продовжують завершуватися з помилкою `AUTH_TOKEN_MISMATCH` або `AUTH_DEVICE_TOKEN_MISMATCH`.

1. Підтвердьте поточне джерело токена gateway:

```bash
openclaw config get gateway.auth.token
```

2. Показати сполучені пристрої та визначити ідентифікатор проблемного пристрою:

```bash
openclaw devices list
```

3. Ротувати операторський токен для проблемного пристрою:

```bash
openclaw devices rotate --device <deviceId> --role operator
```

4. Якщо ротації недостатньо, видаліть застаріле сполучення та схваліть знову:

```bash
openclaw devices remove <deviceId>
openclaw devices list
openclaw devices approve <requestId>
```

5. Повторіть спробу підключення клієнта з поточним спільним токеном/паролем.

Примітки:

- Звичайний пріоритет автентифікації під час повторного підключення: спочатку явний спільний токен/пароль, потім явний `deviceToken`, потім збережений токен пристрою, потім bootstrap token.
- Надійне відновлення після `AUTH_TOKEN_MISMATCH` може тимчасово надсилати одночасно і спільний токен, і збережений токен пристрою для однієї обмеженої повторної спроби.

# `openclaw devices`

Керуйте запитами на сполучення пристроїв і токенами рівня пристрою.

## Команди

### `openclaw devices list`

Показати очікувальні запити на сполучення та сполучені пристрої.

```
openclaw devices list
openclaw devices list --json
```

У виводі очікувальних запитів запитуваний доступ показується поруч із поточним
схваленим доступом пристрою, якщо пристрій уже сполучений. Це робить
підвищення scope/ролі явними, замість того щоб створювати враження, ніби
сполучення було втрачено.

### `openclaw devices remove <deviceId>`

Видалити один запис сполученого пристрою.

Коли ви автентифіковані токеном сполученого пристрою, виклики від неадміністратора
можуть видаляти лише запис **власного** пристрою. Видалення іншого пристрою вимагає
`operator.admin`.

```
openclaw devices remove <deviceId>
openclaw devices remove <deviceId> --json
```

### `openclaw devices clear --yes [--pending]`

Масово очистити сполучені пристрої.

```
openclaw devices clear --yes
openclaw devices clear --yes --pending
openclaw devices clear --yes --pending --json
```

### `openclaw devices approve [requestId] [--latest]`

Схвалити очікувальний запит на сполучення пристрою за точним `requestId`. Якщо `requestId`
не вказано або передано `--latest`, OpenClaw лише виводить вибраний очікувальний
запит і завершує роботу; повторно виконайте схвалення з точним ідентифікатором
запиту після перевірки деталей.

Примітка: якщо пристрій повторно намагається виконати сполучення зі зміненими
даними автентифікації (роль/scopes/public key), OpenClaw замінює попередній
очікувальний запис і видає новий `requestId`. Запускайте `openclaw devices list`
безпосередньо перед схваленням, щоб використати актуальний ідентифікатор.

Якщо пристрій уже сполучений і запитує ширші scopes або ширшу роль,
OpenClaw зберігає наявне схвалення та створює новий очікувальний запит
на підвищення. Перегляньте стовпці `Requested` і `Approved` у `openclaw devices list`
або скористайтеся `openclaw devices approve --latest`, щоб попередньо переглянути
точне підвищення перед його схваленням.

```
openclaw devices approve
openclaw devices approve <requestId>
openclaw devices approve --latest
```

### `openclaw devices reject <requestId>`

Відхилити очікувальний запит на сполучення пристрою.

```
openclaw devices reject <requestId>
```

### `openclaw devices rotate --device <id> --role <role> [--scope <scope...>]`

Виконати ротацію токена пристрою для певної ролі (за потреби оновивши scopes).
Цільова роль уже повинна існувати в контракті схваленого сполучення цього пристрою;
ротація не може випустити нову несхвалену роль.
Якщо ви не вкажете `--scope`, пізніші повторні підключення зі збереженим токеном після ротації повторно використовуватимуть
закешовані схвалені scopes цього токена. Якщо ви передасте явні значення `--scope`, вони
стануть збереженим набором scopes для майбутніх повторних підключень із кешованим токеном.
Виклики від неадміністратора зі сполученого пристрою можуть виконувати ротацію лише
для токена **власного** пристрою.
Також будь-які явні значення `--scope` мають залишатися в межах власних
операторських scopes сесії викликача; ротація не може випустити ширший операторський токен,
ніж викликач уже має.

```
openclaw devices rotate --device <deviceId> --role operator --scope operator.read --scope operator.write
```

Повертає нове корисне навантаження токена у форматі JSON.

### `openclaw devices revoke --device <id> --role <role>`

Відкликати токен пристрою для певної ролі.

Виклики від неадміністратора зі сполученого пристрою можуть відкликати лише
токен **власного** пристрою. Відкликання токена іншого пристрою вимагає
`operator.admin`.

```
openclaw devices revoke --device <deviceId> --role node
```

Повертає результат відкликання у форматі JSON.

## Поширені параметри

- `--url <url>`: URL WebSocket Gateway (типово використовується `gateway.remote.url`, якщо налаштовано).
- `--token <token>`: токен Gateway (за потреби).
- `--password <password>`: пароль Gateway (автентифікація за паролем).
- `--timeout <ms>`: тайм-аут RPC.
- `--json`: вивід JSON (рекомендовано для скриптів).

Примітка: коли ви задаєте `--url`, CLI не використовує резервне отримання облікових даних із конфігурації або середовища.
Явно передайте `--token` або `--password`. Відсутність явно вказаних облікових даних є помилкою.

## Примітки

- Ротація токена повертає новий токен (чутливі дані). Поводьтеся з ним як із секретом.
- Ці команди вимагають scope `operator.pairing` (або `operator.admin`).
- Ротація токена залишається в межах схваленого набору ролей сполучення та базового
  рівня схвалених scopes для цього пристрою. Випадковий запис кешованого токена не надає нову
  ціль для ротації.
- Для сесій токенів сполучених пристроїв міжпристроєве керування доступне лише адміністратору:
  `remove`, `rotate` і `revoke` дозволені лише для себе, якщо викликач не має
  `operator.admin`.
- `devices clear` навмисно захищено параметром `--yes`.
- Якщо scope сполучення недоступний на local loopback (і явний `--url` не передано), list/approve можуть використовувати локальний резервний механізм сполучення.
- `devices approve` вимагає явного ідентифікатора запиту перед випуском токенів; якщо не вказати `requestId` або передати `--latest`, буде лише попередній перегляд найновішого очікувального запиту.

## Контрольний список відновлення при розсинхронізації токенів

Використовуйте це, коли Control UI або інші клієнти постійно завершуються помилками `AUTH_TOKEN_MISMATCH` або `AUTH_DEVICE_TOKEN_MISMATCH`.

1. Підтвердьте поточне джерело токена Gateway:

```bash
openclaw config get gateway.auth.token
```

2. Перелічіть сполучені пристрої та визначте ідентифікатор проблемного пристрою:

```bash
openclaw devices list
```

3. Виконайте ротацію операторського токена для проблемного пристрою:

```bash
openclaw devices rotate --device <deviceId> --role operator
```

4. Якщо ротації недостатньо, видаліть застаріле сполучення та схваліть знову:

```bash
openclaw devices remove <deviceId>
openclaw devices list
openclaw devices approve <requestId>
```

5. Повторіть підключення клієнта з поточним спільним токеном/паролем.

Примітки:

- Звичайний пріоритет автентифікації при повторному підключенні: спочатку явний спільний токен/пароль, потім явний `deviceToken`, потім збережений токен пристрою, потім bootstrap token.
- Надійне відновлення після `AUTH_TOKEN_MISMATCH` може тимчасово надсилати і спільний токен, і збережений токен пристрою разом для однієї обмеженої повторної спроби.

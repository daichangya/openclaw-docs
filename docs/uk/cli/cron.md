# `openclaw cron`

Керуйте Cron-завданнями для планувальника Gateway.

<Tip>
Запустіть `openclaw cron --help`, щоб переглянути повну поверхню команд. Див. [Cron-завдання](/uk/automation/cron-jobs) для концептуального посібника.
</Tip>

## Сесії

`--session` приймає `main`, `isolated`, `current` або `session:<id>`.

<AccordionGroup>
  <Accordion title="Ключі сесій">
    - `main` прив’язується до основної сесії агента.
    - `isolated` створює нову стенограму та ідентифікатор сесії для кожного запуску.
    - `current` прив’язується до активної сесії на момент створення.
    - `session:<id>` закріплює за явним ключем постійної сесії.
  </Accordion>
  <Accordion title="Семантика ізольованої сесії">
    Ізольовані запуски скидають фоновий контекст розмови. Маршрутизація каналу й групи, політика надсилання/черги, підвищення прав, походження та прив’язка середовища виконання ACP скидаються для нового запуску. Безпечні налаштування та явні вибрані користувачем перевизначення моделі або автентифікації можуть переноситися між запусками.
  </Accordion>
</AccordionGroup>

## Доставка

`openclaw cron list` і `openclaw cron show <job-id>` попередньо показують розв’язаний маршрут доставки. Для `channel: "last"` попередній перегляд показує, чи маршрут було розв’язано з основної або поточної сесії, чи він завершиться безпечною відмовою.

<Note>
Ізольовані завдання `cron add` типово використовують доставку `--announce`. Використовуйте `--no-deliver`, щоб залишити вивід внутрішнім. `--deliver` залишається застарілим псевдонімом для `--announce`.
</Note>

### Відповідальність за доставку

Доставка чату для ізольованого Cron ділиться між агентом і виконавцем:

- Агент може надсилати напряму за допомогою інструмента `message`, коли маршрут чату доступний.
- `announce` резервно доставляє лише фінальну відповідь, якщо агент не надіслав її напряму до розв’язаного призначення.
- `webhook` надсилає завершене навантаження на URL.
- `none` вимикає резервну доставку виконавцем.

`--announce` — це резервна доставка виконавцем для фінальної відповіді. `--no-deliver` вимикає цей резервний механізм, але не прибирає інструмент `message` в агента, коли маршрут чату доступний.

Нагадування, створені з активного чату, зберігають ціль доставки живого чату для резервної доставки announce. Внутрішні ключі сесій можуть бути в нижньому регістрі; не використовуйте їх як джерело істини для чутливих до регістру ідентифікаторів провайдерів, таких як ідентифікатори кімнат Matrix.

### Доставка помилок

Сповіщення про помилки розв’язуються в такому порядку:

1. `delivery.failureDestination` у завданні.
2. Глобальний `cron.failureDestination`.
3. Основна announce-ціль завдання (коли явну ціль помилки не задано).

<Note>
Завдання основної сесії можуть використовувати `delivery.failureDestination` лише тоді, коли основний режим доставки — `webhook`. Ізольовані завдання приймають його в усіх режимах.
</Note>

## Планування

### Одноразові завдання

`--at <datetime>` планує одноразовий запуск. Дата й час без зсуву трактуються як UTC, якщо ви також не передасте `--tz <iana>`, який інтерпретує локальний час у заданому часовому поясі.

<Note>
Одноразові завдання типово видаляються після успішного виконання. Використовуйте `--keep-after-run`, щоб зберегти їх.
</Note>

### Повторювані завдання

Повторювані завдання використовують експоненційне збільшення інтервалу повторних спроб після послідовних помилок: 30 с, 1 хв, 5 хв, 15 хв, 60 хв. Після наступного успішного запуску розклад повертається до норми.

### Ручні запуски

`openclaw cron run` повертає керування, щойно ручний запуск поставлено в чергу. Успішні відповіді містять `{ ok: true, enqueued: true, runId }`. Використовуйте `openclaw cron runs --id <job-id>`, щоб відстежити підсумковий результат.

<Note>
`openclaw cron run <job-id>` типово примусово запускає завдання. Використовуйте `--due`, щоб зберегти старішу поведінку «запускати лише якщо настав час».
</Note>

## Моделі

`cron add|edit --model <ref>` вибирає дозволену модель для завдання.

<Warning>
Якщо модель не дозволено, Cron попереджає й повертається до вибору моделі агента або типової моделі для завдання. Налаштовані ланцюжки резервного переходу все ще застосовуються, але просте перевизначення моделі без явного списку резервних варіантів для завдання більше не додає основну модель агента як приховану додаткову ціль повторної спроби.
</Warning>

### Пріоритет моделі для ізольованого Cron

Ізольований Cron визначає активну модель у такому порядку:

1. Перевизначення Gmail-hook.
2. `--model` для завдання.
3. Збережене перевизначення моделі cron-сесії (коли користувач вибрав його).
4. Вибір моделі агента або типової моделі.

### Швидкий режим

Швидкий режим ізольованого Cron дотримується розв’язаного вибору живої моделі. Конфігурація моделі `params.fastMode` застосовується типово, але збережене перевизначення сесії `fastMode` все одно має вищий пріоритет за конфігурацію.

### Повторні спроби під час перемикання живої моделі

Якщо ізольований запуск викидає `LiveSessionModelSwitchError`, Cron зберігає перевизначені провайдера й модель (а також перевизначення профілю автентифікації, якщо воно є) для активного запуску перед повторною спробою. Зовнішній цикл повторних спроб обмежений двома повторними спробами перемикання після початкової спроби, після чого переривається замість нескінченного циклу.

## Вивід запуску та відмови

### Приглушення застарілих підтверджень

Ізольовані ходи Cron приглушують відповіді, що містять лише застаріле підтвердження. Якщо перший результат — лише проміжне оновлення статусу, і жоден дочірній запуск субагента не відповідає за підсумкову відповідь, Cron один раз повторно формулює запит для отримання реального результату перед доставкою.

### Приглушення тихого токена

Якщо ізольований запуск Cron повертає лише тихий токен (`NO_REPLY` або `no_reply`), Cron приглушує і пряме вихідне надсилання, і резервний шлях підсумку через чергу, тому назад у чат нічого не надсилається.

### Структуровані відмови

Ізольовані запуски Cron надають перевагу структурованим метаданим відмови виконання з вбудованого запуску, а потім переходять до відомих маркерів відмови у фінальному виводі, таких як `SYSTEM_RUN_DENIED`, `INVALID_REQUEST` і фрази відмови, прив’язані до схвалення.

`cron list` та історія запусків показують причину відмови замість того, щоб повідомляти заблоковану команду як `ok`.

## Зберігання

Зберігання та очищення налаштовуються в конфігурації:

- `cron.sessionRetention` (типово `24h`) очищує завершені сесії ізольованих запусків.
- `cron.runLog.maxBytes` і `cron.runLog.keepLines` очищують `~/.openclaw/cron/runs/<jobId>.jsonl`.

## Міграція старіших завдань

<Note>
Якщо у вас є Cron-завдання, створені до поточного формату доставки й сховища, запустіть `openclaw doctor --fix`. Doctor нормалізує застарілі поля Cron (`jobId`, `schedule.cron`, поля доставки верхнього рівня, включно із застарілим `threadId`, псевдоніми доставки `provider` у навантаженні) і мігрує прості резервні webhook-завдання `notify: true` до явної доставки webhook, коли налаштовано `cron.webhook`.
</Note>

## Поширені зміни

Оновіть параметри доставки, не змінюючи повідомлення:

```bash
openclaw cron edit <job-id> --announce --channel telegram --to "123456789"
```

Вимкніть доставку для ізольованого завдання:

```bash
openclaw cron edit <job-id> --no-deliver
```

Увімкніть полегшений контекст початкового завантаження для ізольованого завдання:

```bash
openclaw cron edit <job-id> --light-context
```

Оголосіть у конкретний канал:

```bash
openclaw cron edit <job-id> --announce --channel slack --to "channel:C1234567890"
```

Створіть ізольоване завдання з полегшеним контекстом початкового завантаження:

```bash
openclaw cron add \
  --name "Lightweight morning brief" \
  --cron "0 7 * * *" \
  --session isolated \
  --message "Summarize overnight updates." \
  --light-context \
  --no-deliver
```

`--light-context` застосовується лише до ізольованих ходів агента завдань. Для запусків cron полегшений режим зберігає порожній контекст початкового завантаження замість ін’єкції повного набору початкового завантаження робочого простору.

## Поширені команди адміністрування

Ручний запуск і перевірка:

```bash
openclaw cron list
openclaw cron show <job-id>
openclaw cron run <job-id>
openclaw cron run <job-id> --due
openclaw cron runs --id <job-id> --limit 50
```

Записи `cron runs` містять діагностику доставки з наміченою ціллю cron, розв’язаною ціллю, надсиланнями через інструмент message, використанням резервної доставки та станом доставки.

Перепризначення агента й сесії:

```bash
openclaw cron edit <job-id> --agent ops
openclaw cron edit <job-id> --clear-agent
openclaw cron edit <job-id> --session current
openclaw cron edit <job-id> --session "session:daily-brief"
```

Налаштування доставки:

```bash
openclaw cron edit <job-id> --announce --channel slack --to "channel:C1234567890"
openclaw cron edit <job-id> --best-effort-deliver
openclaw cron edit <job-id> --no-best-effort-deliver
openclaw cron edit <job-id> --no-deliver
```

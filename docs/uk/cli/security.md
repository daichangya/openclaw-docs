# `openclaw security`

Інструменти безпеки (аудит + необов’язкові виправлення).

Пов’язане:

- Посібник із безпеки: [Безпека](/uk/gateway/security)

## Аудит

```bash
openclaw security audit
openclaw security audit --deep
openclaw security audit --deep --password <password>
openclaw security audit --deep --token <token>
openclaw security audit --fix
openclaw security audit --json
```

Аудит попереджає, коли кілька відправників DM використовують спільну основну сесію, і рекомендує **безпечний режим DM**: `session.dmScope="per-channel-peer"` (або `per-account-channel-peer` для багатoакаунтних каналів) для спільних вхідних скриньок.
Це призначено для посилення безпеки кооперативних/спільних вхідних скриньок. Один Gateway, спільний для взаємно недовірених/ворожих операторів, не є рекомендованим налаштуванням; розділяйте межі довіри за допомогою окремих gateway (або окремих користувачів ОС/хостів).
Також він виводить `security.trust_model.multi_user_heuristic`, коли конфігурація вказує на ймовірний спільний користувацький вхідний потік (наприклад, відкрита політика DM/груп, налаштовані групові цілі або правила відправників із шаблоном wildcard), і нагадує, що типова модель довіри OpenClaw — це персональний помічник.
Для навмисних багатокористувацьких налаштувань рекомендація аудиту полягає в тому, щоб ізолювати всі сесії, обмежувати доступ до файлової системи межами робочого простору та не зберігати особисті/приватні ідентичності або облікові дані в цьому runtime.
Він також попереджає, коли малі моделі (`<=300B`) використовуються без ізоляції та з увімкненими веб-/браузерними інструментами.
Для webhook-входу він попереджає, коли `hooks.token` повторно використовує токен Gateway, коли `hooks.token` є коротким, коли `hooks.path="/"`, коли `hooks.defaultSessionKey` не задано, коли `hooks.allowedAgentIds` не обмежено, коли увімкнено перевизначення `sessionKey` у запиті, і коли перевизначення увімкнено без `hooks.allowedSessionKeyPrefixes`.
Він також попереджає, коли налаштування Docker для sandbox задано, але режим sandbox вимкнено, коли `gateway.nodes.denyCommands` використовує неефективні шаблоноподібні/невідомі записи (лише точний збіг імен команд node, без фільтрації shell-тексту), коли `gateway.nodes.allowCommands` явно вмикає небезпечні команди node, коли глобальний `tools.profile="minimal"` перевизначається профілями інструментів агента, коли відкриті групи надають інструменти runtime/файлової системи без захисту sandbox/робочого простору, і коли встановлені інструменти Plugin можуть бути доступні за поблажливої політики інструментів.
Він також позначає `gateway.allowRealIpFallback=true` (ризик підробки заголовків, якщо проксі налаштовано неправильно) і `discovery.mdns.mode="full"` (витік метаданих через записи mDNS TXT).
Він також попереджає, коли браузер sandbox використовує мережу Docker `bridge` без `sandbox.browser.cdpSourceRange`.
Він також позначає небезпечні режими мережі Docker для sandbox (включно з `host` і приєднанням до просторів імен `container:*`).
Він також попереджає, коли наявні Docker-контейнери браузера sandbox мають відсутні/застарілі hash-мітки (наприклад, контейнери до міграції без `openclaw.browserConfigEpoch`) і рекомендує `openclaw sandbox recreate --browser --all`.
Він також попереджає, коли записи встановлення Plugin/hook на основі npm не закріплено, їм бракує метаданих цілісності або вони розходяться з поточно встановленими версіями пакетів.
Він попереджає, коли allowlist каналів покладаються на змінні імена/email/теги замість стабільних ID (Discord, Slack, Google Chat, Microsoft Teams, Mattermost, області IRC, де це застосовно).
Він попереджає, коли `gateway.auth.mode="none"` залишає HTTP API Gateway доступними без спільного секрету (`/tools/invoke` плюс будь-яка увімкнена кінцева точка `/v1/*`).
Параметри з префіксами `dangerous`/`dangerously` є явними аварійними перевизначеннями оператора; саме по собі ввімкнення одного з них не є повідомленням про вразливість безпеки.
Повний перелік небезпечних параметрів дивіться в розділі "Підсумок небезпечних або незахищених прапорців" у [Безпека](/uk/gateway/security).

Поведінка SecretRef:

- `security audit` розв’язує підтримувані SecretRef у режимі лише для читання для своїх цільових шляхів.
- Якщо SecretRef недоступний у поточному шляху команди, аудит продовжується та повідомляє `secretDiagnostics` (замість аварійного завершення).
- `--token` і `--password` перевизначають автентифікацію deep-probe лише для цього виклику команди; вони не переписують конфігурацію або зіставлення SecretRef.

## Вивід JSON

Використовуйте `--json` для перевірок CI/політик:

```bash
openclaw security audit --json | jq '.summary'
openclaw security audit --deep --json | jq '.findings[] | select(.severity=="critical") | .checkId'
```

Якщо поєднати `--fix` і `--json`, вивід міститиме як дії виправлення, так і підсумковий звіт:

```bash
openclaw security audit --fix --json | jq '{fix: .fix.ok, summary: .report.summary}'
```

## Що змінює `--fix`

`--fix` застосовує безпечні, детерміновані виправлення:

- перемикає поширене `groupPolicy="open"` на `groupPolicy="allowlist"` (включно з варіантами для акаунтів у підтримуваних каналах)
- коли політика груп WhatsApp перемикається на `allowlist`, ініціалізує `groupAllowFrom` з
  збереженого файлу `allowFrom`, якщо цей список існує, а конфігурація ще не
  визначає `allowFrom`
- встановлює `logging.redactSensitive` з `"off"` на `"tools"`
- посилює права доступу для стану/конфігурації та поширених чутливих файлів
  (`credentials/*.json`, `auth-profiles.json`, `sessions.json`, `session
  *.jsonl`)
- також посилює файли включення конфігурації, на які посилається `openclaw.json`
- використовує `chmod` на POSIX-хостах і скидання `icacls` на Windows

`--fix` **не**:

- ротуватиме токени/паролі/API-ключі
- вимикатиме інструменти (`gateway`, `cron`, `exec` тощо)
- змінюватиме вибір прив’язки/автентифікації/мережевої експозиції gateway
- видалятиме або переписуватиме Plugin/Skills

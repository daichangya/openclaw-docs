Підагенти — це фонові запуски агентів, породжені з наявного запуску агента.
Вони працюють у власній сесії (`agent:<agentId>:subagent:<uuid>`) і,
коли завершуються, **повідомляють** про результат назад у чат-канал
запитувача. Кожен запуск під-агента відстежується як
[фонове завдання](/uk/automation/tasks).

Основні цілі:

- Розпаралелювати роботу типу «дослідження / довге завдання / повільний інструмент» без блокування основного запуску.
- Типово ізолювати під-агентів (розділення сесій + необов’язкова пісочниця).
- Зробити поверхню інструментів складною для неправильного використання: під-агенти **не** отримують інструменти сесії типово.
- Підтримувати налаштовувану глибину вкладеності для шаблонів оркестрації.

<Note>
**Примітка щодо вартості:** кожен під-агент типово має власний контекст і власне використання токенів.
Для важких або повторюваних завдань задайте для під-агентів дешевшу модель,
а основного агента залиште на якіснішій моделі. Налаштовується через
`agents.defaults.subagents.model` або перевизначення для окремого агента. Коли дочірньому агенту
справді потрібен поточний transcript запитувача, агент може запросити
`context: "fork"` лише для цього одного породження.
</Note>

## Slash-команда

Використовуйте `/subagents`, щоб переглядати або керувати запусками під-агентів для **поточної
сесії**:

```text
/subagents list
/subagents kill <id|#|all>
/subagents log <id|#> [limit] [tools]
/subagents info <id|#>
/subagents send <id|#> <message>
/subagents steer <id|#> <message>
/subagents spawn <agentId> <task> [--model <model>] [--thinking <level>]
```

`/subagents info` показує метадані запуску (статус, часові мітки, id сесії,
шлях transcript, очищення). Використовуйте `sessions_history` для обмеженого,
відфільтрованого з погляду безпеки перегляду історії; перевіряйте шлях transcript на диску, коли вам
потрібен сирий повний transcript.

### Керування прив’язкою до гілок

Ці команди працюють у каналах, що підтримують постійні прив’язки до гілок.
Див. [Канали з підтримкою гілок](#thread-supporting-channels) нижче.

```text
/focus <subagent-label|session-key|session-id|session-label>
/unfocus
/agents
/session idle <duration|off>
/session max-age <duration|off>
```

### Поведінка spawn

`/subagents spawn` запускає фоновий під-агент як команду користувача (а не
внутрішню пересилку) і надсилає одне фінальне оновлення про завершення назад у
чат запитувача, коли запуск завершується.

<AccordionGroup>
  <Accordion title="Неблокувальне завершення на основі push">
    - Команда spawn є неблокувальною; вона відразу повертає id запуску.
    - Після завершення під-агент повідомляє зведення/результат назад у чат-канал запитувача.
    - Доставка завершення відбувається через push. Після spawn **не** опитуйте по колу `/subagents list`, `sessions_list` або `sessions_history` лише для того, щоб дочекатися завершення; перевіряйте стан лише за потреби для налагодження або втручання.
    - Після завершення OpenClaw за принципом best-effort закриває відстежувані вкладки/процеси браузера, відкриті цією сесією під-агента, перш ніж продовжиться потік очищення повідомлення.
  </Accordion>
  <Accordion title="Стійкість доставки ручного spawn">
    - OpenClaw спочатку намагається доставити напряму через `agent` зі стабільним ключем ідемпотентності.
    - Якщо пряма доставка не вдається, використовується резервна маршрутизація через чергу.
    - Якщо маршрутизація через чергу теж недоступна, спроба повідомлення повторюється з коротким експоненційним backoff перед остаточною відмовою.
    - Доставка завершення зберігає визначений маршрут запитувача: маршрути завершення, прив’язані до гілки або conversation, мають перевагу, коли доступні; якщо джерело завершення надає лише канал, OpenClaw заповнює відсутні target/account із визначеного маршруту сесії запитувача (`lastChannel` / `lastTo` / `lastAccountId`), щоб пряма доставка все одно спрацювала.
  </Accordion>
  <Accordion title="Метадані передачі завершення">
    Передача завершення до сесії запитувача — це внутрішній контекст,
    згенерований runtime (а не текст, створений користувачем), і він містить:

    - `Result` — текст останньої видимої відповіді `assistant`, або, якщо її немає, санітизований текст останнього `tool`/`toolResult`. Для термінальних невдалих запусків захоплений текст відповіді повторно не використовується.
    - `Status` — `completed successfully` / `failed` / `timed out` / `unknown`.
    - Компактну статистику runtime/токенів.
    - Інструкцію доставки, яка наказує агентові запитувача переписати це звичайним голосом assistant, а не пересилати сирі внутрішні метадані.

  </Accordion>
  <Accordion title="Режими й ACP runtime">
    - `--model` і `--thinking` перевизначають типові значення для цього конкретного запуску.
    - Використовуйте `info`/`log`, щоб переглядати деталі та вивід після завершення.
    - `/subagents spawn` — це одноразовий режим (`mode: "run"`). Для постійних сесій, прив’язаних до гілок, використовуйте `sessions_spawn` із `thread: true` і `mode: "session"`.
    - Для сесій ACP harness (Claude Code, Gemini CLI, OpenCode або явно Codex ACP/acpx) використовуйте `sessions_spawn` із `runtime: "acp"`, коли інструмент оголошує цей runtime. Див. [Модель доставки ACP](/uk/tools/acp-agents#delivery-model), коли налагоджуєте завершення або цикли агент-до-агента. Коли Plugin `codex` увімкнено, керування чатом/гілками Codex має віддавати перевагу `/codex ...` замість ACP, якщо користувач явно не просить ACP/acpx.
    - OpenClaw приховує `runtime: "acp"`, доки ACP не увімкнено, запитувач не є в пісочниці, а backend Plugin на кшталт `acpx` не завантажено. `runtime: "acp"` очікує зовнішній id ACP harness або запис `agents.list[]` із `runtime.type="acp"`; для звичайних конфігурованих агентів OpenClaw із `agents_list` використовуйте типовий runtime під-агента.
  </Accordion>
</AccordionGroup>

## Режими контексту

Нативні під-агенти запускаються ізольовано, якщо викликач явно не просить відгалузити
поточний transcript.

| Режим      | Коли його використовувати                                                                                                              | Поведінка                                                                          |
| ---------- | -------------------------------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------- |
| `isolated` | Нове дослідження, незалежна реалізація, робота з повільними інструментами або будь-що, що можна коротко описати в тексті завдання     | Створює чистий дочірній transcript. Це типовий режим і він зменшує використання токенів. |
| `fork`     | Робота, що залежить від поточної розмови, попередніх результатів інструментів або нюансованих інструкцій, уже наявних у transcript запитувача | Відгалужує transcript запитувача в дочірню сесію до старту дочірнього агента. |

Використовуйте `fork` помірковано. Це для делегування, чутливого до контексту, а не
заміна чіткому формулюванню завдання.

## Інструмент: `sessions_spawn`

Запускає запуск під-агента з `deliver: false` у глобальній смузі `subagent`,
потім виконує крок повідомлення й публікує відповідь-повідомлення в чат-канал
запитувача.

**Типові значення:**

- **Модель:** успадковується від викликача, якщо ви не задасте `agents.defaults.subagents.model` (або `agents.list[].subagents.model` для окремого агента); явне `sessions_spawn.model` усе одно має пріоритет.
- **Thinking:** успадковується від викликача, якщо ви не задасте `agents.defaults.subagents.thinking` (або `agents.list[].subagents.thinking` для окремого агента); явне `sessions_spawn.thinking` усе одно має пріоритет.
- **Тайм-аут запуску:** якщо `sessions_spawn.runTimeoutSeconds` пропущено, OpenClaw використовує `agents.defaults.subagents.runTimeoutSeconds`, коли його задано; інакше використовується `0` (без тайм-ауту).

### Параметри інструмента

<ParamField path="task" type="string" required>
  Опис завдання для під-агента.
</ParamField>
<ParamField path="label" type="string">
  Необов’язкова людиночитана мітка.
</ParamField>
<ParamField path="agentId" type="string">
  Запуск під іншим id агента, якщо це дозволено `subagents.allowAgents`.
</ParamField>
<ParamField path="runtime" type='"subagent" | "acp"' default="subagent">
  `acp` призначений лише для зовнішніх ACP harness (`claude`, `droid`, `gemini`, `opencode` або явно запитаний Codex ACP/acpx) і для записів `agents.list[]`, у яких `runtime.type` дорівнює `acp`.
</ParamField>
<ParamField path="model" type="string">
  Перевизначає модель під-агента. Некоректні значення пропускаються, і під-агент запускається на типовій моделі з попередженням у результаті інструмента.
</ParamField>
<ParamField path="thinking" type="string">
  Перевизначає рівень thinking для запуску під-агента.
</ParamField>
<ParamField path="runTimeoutSeconds" type="number">
  Типово використовується `agents.defaults.subagents.runTimeoutSeconds`, якщо його задано, інакше `0`. Якщо встановлено, запуск під-агента переривається через N секунд.
</ParamField>
<ParamField path="thread" type="boolean" default="false">
  Коли `true`, запитує прив’язку до гілки каналу для цієї сесії під-агента.
</ParamField>
<ParamField path="mode" type='"run" | "session"' default="run">
  Якщо `thread: true`, а `mode` пропущено, типовим стає `session`. `mode: "session"` вимагає `thread: true`.
</ParamField>
<ParamField path="cleanup" type='"delete" | "keep"' default="keep">
  `"delete"` архівує одразу після повідомлення (але все одно зберігає transcript через перейменування).
</ParamField>
<ParamField path="sandbox" type='"inherit" | "require"' default="inherit">
  `require` відхиляє spawn, якщо цільовий дочірній runtime не працює в пісочниці.
</ParamField>
<ParamField path="context" type='"isolated" | "fork"' default="isolated">
  `fork` відгалужує поточний transcript запитувача в дочірню сесію. Лише для нативних під-агентів. Використовуйте `fork` тільки тоді, коли дочірньому агенту потрібен поточний transcript.
</ParamField>

<Warning>
`sessions_spawn` **не** приймає параметри доставки каналу (`target`,
`channel`, `to`, `threadId`, `replyTo`, `transport`). Для доставки використовуйте
`message`/`sessions_send` із породженого запуску.
</Warning>

## Сесії, прив’язані до гілок

Коли для каналу ввімкнено прив’язки до гілок, під-агент може залишатися прив’язаним
до гілки, щоб подальші повідомлення користувача в цій гілці й далі маршрутизувалися
до тієї самої сесії під-агента.

### Канали з підтримкою гілок

**Discord** наразі є єдиним підтримуваним каналом. Він підтримує
постійні сесії під-агентів, прив’язані до гілок (`sessions_spawn` із
`thread: true`), ручне керування гілками (`/focus`, `/unfocus`, `/agents`,
`/session idle`, `/session max-age`) і ключі адаптера
`channels.discord.threadBindings.enabled`,
`channels.discord.threadBindings.idleHours`,
`channels.discord.threadBindings.maxAgeHours` та
`channels.discord.threadBindings.spawnSubagentSessions`.

### Швидкий сценарій

<Steps>
  <Step title="Породження">
    `sessions_spawn` із `thread: true` (і, за потреби, `mode: "session"`).
  </Step>
  <Step title="Прив’язка">
    OpenClaw створює гілку або прив’язує її до цієї цільової сесії в активному каналі.
  </Step>
  <Step title="Маршрутизація подальших повідомлень">
    Відповіді та подальші повідомлення в цій гілці маршрутизуються до прив’язаної сесії.
  </Step>
  <Step title="Перевірка тайм-аутів">
    Використовуйте `/session idle`, щоб переглядати/оновлювати автоматичне відв’язування через неактивність, і
    `/session max-age`, щоб керувати жорстким обмеженням.
  </Step>
  <Step title="Від’єднання">
    Використовуйте `/unfocus`, щоб від’єднати вручну.
  </Step>
</Steps>

### Ручне керування

| Команда            | Дія                                                                     |
| ------------------ | ----------------------------------------------------------------------- |
| `/focus <target>`  | Прив’язує поточну гілку (або створює її) до цілі під-агента/сесії       |
| `/unfocus`         | Видаляє прив’язку для поточної прив’язаної гілки                        |
| `/agents`          | Показує список активних запусків і стан прив’язки (`thread:<id>` або `unbound`) |
| `/session idle`    | Переглядає/оновлює автовідв’язування через неактивність (лише для прив’язаних сфокусованих гілок) |
| `/session max-age` | Переглядає/оновлює жорстке обмеження (лише для прив’язаних сфокусованих гілок) |

### Перемикачі конфігурації

- **Глобальний типовый параметр:** `session.threadBindings.enabled`, `session.threadBindings.idleHours`, `session.threadBindings.maxAgeHours`.
- **Перевизначення каналу та ключі автоприв’язки spawn** залежать від конкретного адаптера. Див. [Канали з підтримкою гілок](#thread-supporting-channels) вище.

Див. [Довідка з конфігурації](/uk/gateway/configuration-reference) і
[Slash-команди](/uk/tools/slash-commands) для поточних відомостей щодо адаптерів.

### Список дозволених значень

<ParamField path="agents.list[].subagents.allowAgents" type="string[]">
  Список id агентів, на які можна націлюватися через `agentId` (`["*"]` дозволяє будь-який). Типово: лише агент запитувача.
</ParamField>
<ParamField path="agents.defaults.subagents.allowAgents" type="string[]">
  Типовий список дозволених цільових агентів, що використовується, коли агент запитувача не задає власний `subagents.allowAgents`.
</ParamField>
<ParamField path="agents.defaults.subagents.requireAgentId" type="boolean" default="false">
  Блокує виклики `sessions_spawn`, у яких пропущено `agentId` (примушує до явного вибору профілю). Перевизначення для окремого агента: `agents.list[].subagents.requireAgentId`.
</ParamField>

Якщо сесія запитувача працює в пісочниці, `sessions_spawn` відхиляє цілі,
які працювали б без пісочниці.

### Виявлення

Використовуйте `agents_list`, щоб побачити, які id агентів зараз дозволені для
`sessions_spawn`. Відповідь містить ефективну
модель кожного наведеного агента та метадані вбудованого runtime, щоб викликачі могли розрізняти PI, app-server Codex та інші налаштовані нативні runtime.

### Автоархівація

- Сесії під-агентів автоматично архівуються після `agents.defaults.subagents.archiveAfterMinutes` (типово `60`).
- Архівація використовує `sessions.delete` і перейменовує transcript на `*.deleted.<timestamp>` (у тій самій папці).
- `cleanup: "delete"` архівує одразу після повідомлення (але все одно зберігає transcript через перейменування).
- Автоархівація працює за принципом best-effort; відкладені таймери втрачаються, якщо gateway перезапускається.
- `runTimeoutSeconds` **не** виконує автоархівацію; він лише зупиняє запуск. Сесія лишається до автоархівації.
- Автоархівація однаково застосовується до сесій рівня 1 і рівня 2.
- Очищення браузера відокремлене від очищення архіву: відстежувані вкладки/процеси браузера за принципом best-effort закриваються, коли запуск завершується, навіть якщо запис transcript/сесії зберігається.

## Вкладені підагенти

Типово підагенти не можуть породжувати власних підагентів
(`maxSpawnDepth: 1`). Установіть `maxSpawnDepth: 2`, щоб увімкнути один рівень
вкладеності — **шаблон оркестратора**: main → під-агент-оркестратор →
робочі під-підагенти.

```json5
{
  agents: {
    defaults: {
      subagents: {
        maxSpawnDepth: 2, // дозволяє підагентам породжувати дочірні (типово: 1)
        maxChildrenPerAgent: 5, // макс. кількість активних дочірніх для сесії агента (типово: 5)
        maxConcurrent: 8, // глобальне обмеження паралельності смуги (типово: 8)
        runTimeoutSeconds: 900, // типовий тайм-аут для sessions_spawn, якщо не вказано (0 = без тайм-ауту)
      },
    },
  },
}
```

### Рівні глибини

| Глибина | Форма ключа сесії                            | Роль                                          | Чи може породжувати?        |
| ------- | -------------------------------------------- | --------------------------------------------- | --------------------------- |
| 0       | `agent:<id>:main`                            | Основний агент                                | Завжди                      |
| 1       | `agent:<id>:subagent:<uuid>`                 | Підагент (оркестратор, якщо дозволено глибину 2) | Лише якщо `maxSpawnDepth >= 2` |
| 2       | `agent:<id>:subagent:<uuid>:subagent:<uuid>` | Під-підагент (кінцевий робочий агент)         | Ніколи                      |

### Ланцюжок повідомлення

Результати піднімаються вгору ланцюжком:

1. Робочий агент глибини 2 завершується → повідомляє своєму батьку (оркестратору глибини 1).
2. Оркестратор глибини 1 отримує повідомлення, синтезує результати, завершується → повідомляє main.
3. Основний агент отримує повідомлення й доставляє його користувачеві.

Кожен рівень бачить лише повідомлення від своїх прямих дочірніх агентів.

<Note>
**Операційна рекомендація:** запускайте дочірню роботу один раз і чекайте
подій завершення, а не будуйте цикли опитування навколо `sessions_list`,
`sessions_history`, `/subagents list` або команд `exec` зі sleep.
`sessions_list` і `/subagents list` тримають зв’язки між дочірніми сесіями
зосередженими на live-роботі — live-дочірні лишаються прикріпленими, завершені дочірні
лишаються видимими протягом короткого недавнього вікна, а застарілі посилання лише зі сховища
ігноруються після завершення їхнього вікна свіжості. Це не дає старим метаданим `spawnedBy` /
`parentSessionKey` відроджувати примарних дочірніх агентів після
перезапуску. Якщо подія завершення дочірнього агента приходить уже після того, як ви надіслали
фінальну відповідь, правильна подальша дія — точний тихий токен
`NO_REPLY` / `no_reply`.
</Note>

### Політика інструментів за глибиною

- Роль і область керування записуються в метадані сесії під час spawn. Це не дає плоским або відновленим ключам сесій випадково знову отримати привілеї оркестратора.
- **Глибина 1 (оркестратор, коли `maxSpawnDepth >= 2`):** отримує `sessions_spawn`, `subagents`, `sessions_list`, `sessions_history`, щоб керувати своїми дочірніми агентами. Інші інструменти сесії/системи лишаються забороненими.
- **Глибина 1 (лист, коли `maxSpawnDepth == 1`):** без інструментів сесії (поточна типова поведінка).
- **Глибина 2 (кінцевий робочий агент):** без інструментів сесії — `sessions_spawn` на глибині 2 завжди заборонений. Не може породжувати дальших дочірніх агентів.

### Обмеження породження для окремого агента

Кожна сесія агента (на будь-якій глибині) може одночасно мати щонайбільше `maxChildrenPerAgent`
(типово `5`) активних дочірніх агентів. Це запобігає неконтрольованому розгалуженню
від одного оркестратора.

### Каскадна зупинка

Зупинка оркестратора глибини 1 автоматично зупиняє всіх його дочірніх агентів
глибини 2:

- `/stop` в основному чаті зупиняє всіх агентів глибини 1 і каскадно зупиняє їхніх дочірніх агентів глибини 2.
- `/subagents kill <id>` зупиняє конкретного під-агента й каскадно зупиняє його дочірніх агентів.
- `/subagents kill all` зупиняє всіх під-агентів для запитувача й виконує каскадну зупинку.

## Автентифікація

Автентифікація під-агента визначається за **id агента**, а не за типом сесії:

- Ключ сесії під-агента — `agent:<agentId>:subagent:<uuid>`.
- Сховище автентифікації завантажується з `agentDir` цього агента.
- Профілі автентифікації основного агента об’єднуються як **резервний варіант**; профілі агента мають пріоритет над профілями основного агента у разі конфліктів.

Об’єднання є адитивним, тож профілі основного агента завжди доступні як
резервні. Повністю ізольована автентифікація для кожного агента поки що не підтримується.

## Повідомлення

Підагенти звітують назад через крок повідомлення:

- Крок повідомлення виконується всередині сесії під-агента (а не сесії запитувача).
- Якщо під-агент відповідає рівно `ANNOUNCE_SKIP`, нічого не публікується.
- Якщо останній текст assistant є точним тихим токеном `NO_REPLY` / `no_reply`, вивід повідомлення пригнічується, навіть якщо раніше був видимий прогрес.

Доставка залежить від глибини запитувача:

- Сесії запитувача верхнього рівня використовують подальший виклик `agent` із зовнішньою доставкою (`deliver=true`).
- Вкладені сесії під-агента запитувача отримують внутрішнє подальше впровадження (`deliver=false`), щоб оркестратор міг синтезувати результати дочірніх агентів у межах сесії.
- Якщо вкладена сесія під-агента запитувача зникла, OpenClaw резервно переходить до запитувача цієї сесії, якщо це можливо.

Для сесій запитувача верхнього рівня пряма доставка в режимі завершення спочатку
визначає будь-який прив’язаний маршрут conversation/thread і перевизначення hook, а потім заповнює
відсутні поля channel-target із збереженого маршруту сесії запитувача.
Це утримує завершення в правильному чаті/темі, навіть коли джерело завершення
ідентифікує лише канал.

Агрегація завершень дочірніх агентів обмежується поточним запуском запитувача під час
побудови вкладених даних завершення, що не дає застарілим виводам дочірніх агентів із попередніх запусків
потрапляти в поточне повідомлення. Відповіді повідомлень зберігають
маршрутизацію гілки/теми, коли вона доступна в адаптерах каналів.

### Контекст повідомлення

Контекст повідомлення нормалізується до стабільного внутрішнього блоку події:

| Поле           | Джерело                                                                                                      |
| -------------- | ------------------------------------------------------------------------------------------------------------ |
| Джерело        | `subagent` або `cron`                                                                                        |
| Id сесії       | Ключ/id дочірньої сесії                                                                                      |
| Тип            | Тип повідомлення + мітка завдання                                                                            |
| Статус         | Виводиться з результату runtime (`success`, `error`, `timeout` або `unknown`) — **не** виводиться з тексту моделі |
| Вміст результату | Останній видимий текст assistant, або, якщо його немає, санітизований текст останнього `tool`/`toolResult` |
| Подальша дія   | Інструкція, що описує, коли відповідати, а коли зберігати тишу                                              |

Термінальні невдалі запуски повідомляють про статус помилки без повторного відтворення захопленого
тексту відповіді. У разі тайм-ауту, якщо дочірній агент встиг лише до викликів інструментів, повідомлення
може згорнути цю історію до короткого підсумку часткового прогресу
замість відтворення сирого виводу інструментів.

### Рядок статистики

Payload повідомлень містять рядок статистики наприкінці (навіть коли вони обгорнуті):

- Runtime (наприклад `runtime 5m12s`).
- Використання токенів (input/output/total).
- Оцінювану вартість, якщо для моделі налаштовано ціни (`models.providers.*.models[].cost`).
- `sessionKey`, `sessionId` і шлях transcript, щоб основний агент міг отримати історію через `sessions_history` або перевірити файл на диску.

Внутрішні метадані призначені лише для оркестрації; відповіді, орієнтовані на користувача,
слід переписувати звичайним голосом assistant.

### Чому варто віддавати перевагу `sessions_history`

`sessions_history` — безпечніший шлях оркестрації:

- Спочатку нормалізується відтворення assistant: теги thinking прибираються; службові блоки `<relevant-memories>` / `<relevant_memories>` прибираються; блоки XML payload викликів інструментів у звичайному тексті (`<tool_call>`, `<function_call>`, `<tool_calls>`, `<function_calls>`) прибираються, зокрема обрізані payload, які так і не закрилися належним чином; прибираються знижені до службових каркасів викликів/результатів інструментів і маркери історичного контексту; прибираються витеклі службові токени керування моделлю (`<|assistant|>`, інші ASCII `<|...|>`, full-width `<｜...｜>`); прибирається некоректний XML викликів інструментів MiniMax.
- Текст, схожий на облікові дані/токени, редагується.
- Довгі блоки можуть обрізатися.
- Дуже великі історії можуть відкидати старіші рядки або заміняти надто великий рядок на `[sessions_history omitted: message too large]`.
- Перевірка сирого transcript на диску — це резервний варіант, коли вам потрібен повний transcript байт-у-байт.

## Політика інструментів

Підагенти використовують той самий профіль і той самий конвеєр політики інструментів, що й батьківський або
цільовий агент, на першому етапі. Після цього OpenClaw застосовує рівень обмежень
під-агента.

Якщо обмежувальний `tools.profile` не задано, підагенти отримують **усі інструменти, крім
інструментів сесії** і системних інструментів:

- `sessions_list`
- `sessions_history`
- `sessions_send`
- `sessions_spawn`

`sessions_history` і тут лишається обмеженим, санітизованим переглядом історії —
це не сирий дамп transcript.

Коли `maxSpawnDepth >= 2`, підагенти-оркестратори глибини 1 додатково
отримують `sessions_spawn`, `subagents`, `sessions_list` і
`sessions_history`, щоб керувати своїми дочірніми агентами.

### Перевизначення через конфігурацію

```json5
{
  agents: {
    defaults: {
      subagents: {
        maxConcurrent: 1,
      },
    },
  },
  tools: {
    subagents: {
      tools: {
        // deny wins
        deny: ["gateway", "cron"],
        // if allow is set, it becomes allow-only (deny still wins)
        // allow: ["read", "exec", "process"]
      },
    },
  },
}
```

`tools.subagents.tools.allow` — це фінальний фільтр дозволених значень. Він може звузити
вже визначений набір інструментів, але не може **повернути назад** інструмент,
видалений `tools.profile`. Наприклад, `tools.profile: "coding"` містить
`web_search`/`web_fetch`, але не інструмент `browser`. Щоб
підагенти з профілем coding могли використовувати автоматизацію браузера, додайте browser на
етапі профілю:

```json5
{
  tools: {
    profile: "coding",
    alsoAllow: ["browser"],
  },
}
```

Використовуйте `agents.list[].tools.alsoAllow: ["browser"]` для окремого агента, якщо лише один
агент має отримати автоматизацію браузера.

## Паралельність

Підагенти використовують окрему смугу черги в процесі:

- **Назва смуги:** `subagent`
- **Паралельність:** `agents.defaults.subagents.maxConcurrent` (типово `8`)

## Живучість і відновлення

OpenClaw не вважає відсутність `endedAt` постійним доказом того, що
під-агент усе ще живий. Незавершені запуски, старші за вікно застарілого запуску,
перестають рахуватися як активні/очікувані в `/subagents list`, підсумках статусу,
контролі завершення нащадків і перевірках паралельності для окремої сесії.

Після перезапуску gateway застарілі відновлені незавершені запуски відкидаються, якщо
їхня дочірня сесія не позначена як `abortedLastRun: true`. Такі
дочірні сесії, перервані перезапуском, лишаються відновлюваними через потік відновлення сиріт під-агентів, який надсилає синтетичне повідомлення відновлення, перш ніж
очистити маркер переривання.

<Note>
Якщо spawn під-агента завершується помилкою Gateway `PAIRING_REQUIRED` /
`scope-upgrade`, перевірте викликач RPC, перш ніж редагувати стан сполучення.
Внутрішня координація `sessions_spawn` має підключатися як
`client.id: "gateway-client"` з `client.mode: "backend"` через пряму
автентифікацію loopback зі спільним токеном/паролем; цей шлях не залежить від
базового рівня області paired-device у CLI. Віддалені викликачі, явний
`deviceIdentity`, явні шляхи токена пристрою та клієнти браузера/node
і далі потребують звичайного схвалення пристрою для підвищення області.
</Note>

## Зупинка

- Надсилання `/stop` у чаті запитувача перериває сесію запитувача і зупиняє всі активні запуски під-агентів, породжені з неї, з каскадним поширенням на вкладених дочірніх агентів.
- `/subagents kill <id>` зупиняє конкретного під-агента і каскадно зупиняє його дочірніх агентів.

## Обмеження

- Повідомлення під-агента працює за принципом **best-effort**. Якщо gateway перезапускається, очікувана робота «повідомити назад» втрачається.
- Підагенти все ще ділять ті самі ресурси процесу gateway; розглядайте `maxConcurrent` як запобіжний клапан.
- `sessions_spawn` завжди неблокувальний: він одразу повертає `{ status: "accepted", runId, childSessionKey }`.
- Контекст під-агента впроваджує лише `AGENTS.md` + `TOOLS.md` (без `SOUL.md`, `IDENTITY.md`, `USER.md`, `HEARTBEAT.md` або `BOOTSTRAP.md`).
- Максимальна глибина вкладеності — 5 (діапазон `maxSpawnDepth`: 1–5). Для більшості випадків використання рекомендовано глибину 2.
- `maxChildrenPerAgent` обмежує кількість активних дочірніх агентів на сесію (типово `5`, діапазон `1–20`).

Схвалення exec — це **запобіжник companion app / node host**, який дозволяє
агенту в пісочниці запускати команди на реальному хості (`gateway` або `node`). Це
блокування безпеки: команди дозволяються лише тоді, коли політика + список дозволів +
(необов’язкове) схвалення користувача всі погоджуються. Схвалення exec нашаровуються **поверх**
політики інструментів і elevated gating (якщо лише elevated не встановлено в `full`, що
пропускає схвалення).

<Note>
Ефективна політика — це **суворіша** з `tools.exec.*` і типових значень
схвалень; якщо поле схвалень пропущено, використовується значення `tools.exec`.
Host exec також використовує локальний стан схвалень на цій машині — локальне
`ask: "always"` у `~/.openclaw/exec-approvals.json` і далі показуватиме
запит, навіть якщо типові значення сесії або config запитують `ask: "on-miss"`.
</Note>

## Перегляд ефективної політики

| Команда                                                          | Що вона показує                                                                    |
| ---------------------------------------------------------------- | ---------------------------------------------------------------------------------- |
| `openclaw approvals get` / `--gateway` / `--node <id\|name\|ip>` | Запитану політику, джерела політики хоста та ефективний результат.                 |
| `openclaw exec-policy show`                                      | Об’єднане подання для локальної машини.                                            |
| `openclaw exec-policy set` / `preset`                            | Синхронізує локальну запитану політику з локальним файлом схвалень хоста за один крок. |

Коли локальна область дії запитує `host=node`, `exec-policy show` повідомляє цю
область дії як таку, що керується вузлом під час runtime, замість того щоб удавати, що локальний
файл схвалень є джерелом істини.

Якщо UI companion app **недоступний**, будь-який запит, який зазвичай мав би
показати запит, розв’язується через **ask fallback** (типово: `deny`).

<Tip>
Рідні клієнти схвалення в чаті можуть додавати зручні елементи, специфічні для каналу, до
повідомлення про очікуване схвалення. Наприклад, Matrix додає скорочення реакціями
(`✅` дозволити один раз, `❌` відхилити, `♾️` дозволити завжди), водночас
залишаючи в повідомленні команди `/approve ...` як резервний варіант.
</Tip>

## Де це застосовується

Схвалення exec примусово застосовуються локально на хості виконання:

- **Хост Gateway** → процес `openclaw` на машині gateway.
- **Node host** → виконавець вузла (macOS companion app або безголовий вузол-хост).

### Модель довіри

- Виклики, автентифіковані через gateway, вважаються довіреними операторами для цього Gateway.
- Сполучені вузли поширюють цю можливість довіреного оператора на node host.
- Схвалення exec зменшують ризик випадкового виконання, але **не** є межею автентифікації для кожного користувача.
- Схвалені запуски на node host прив’язують канонічний контекст виконання: канонічний cwd, точний argv, прив’язку env, коли вона є, і зафіксований шлях до виконуваного файла, коли це застосовно.
- Для shell-скриптів і прямих викликів файлів через interpreter/runtime OpenClaw також намагається прив’язати один конкретний локальний файловий операнд. Якщо цей прив’язаний файл змінюється після схвалення, але до виконання, запуск відхиляється замість виконання зміненого вмісту.
- Прив’язка файлу навмисно є best-effort, **а не** повною семантичною моделлю для кожного шляху завантаження interpreter/runtime. Якщо режим схвалення не може однозначно визначити рівно один конкретний локальний файл для прив’язки, він відмовляється створювати запуск із підтримкою схвалення замість того, щоб удавати повне покриття.

### Розділення macOS

- **Сервіс node host** пересилає `system.run` до **macOS app** через локальний IPC.
- **macOS app** застосовує схвалення й виконує команду в контексті UI.

## Налаштування та зберігання

Схвалення зберігаються в локальному JSON-файлі на хості виконання:

```text
~/.openclaw/exec-approvals.json
```

Приклад schema:

```json
{
  "version": 1,
  "socket": {
    "path": "~/.openclaw/exec-approvals.sock",
    "token": "base64url-token"
  },
  "defaults": {
    "security": "deny",
    "ask": "on-miss",
    "askFallback": "deny",
    "autoAllowSkills": false
  },
  "agents": {
    "main": {
      "security": "allowlist",
      "ask": "on-miss",
      "askFallback": "deny",
      "autoAllowSkills": true,
      "allowlist": [
        {
          "id": "B0C8C0B3-2C2D-4F8A-9A3C-5A4B3C2D1E0F",
          "pattern": "~/Projects/**/bin/rg",
          "lastUsedAt": 1737150000000,
          "lastUsedCommand": "rg -n TODO",
          "lastResolvedPath": "/Users/user/Projects/.../bin/rg"
        }
      ]
    }
  }
}
```

## Параметри політики

### `exec.security`

<ParamField path="security" type='"deny" | "allowlist" | "full"'>
  - `deny` — блокувати всі запити host exec.
  - `allowlist` — дозволяти лише команди зі списку дозволів.
  - `full` — дозволяти все (еквівалент elevated).
</ParamField>

### `exec.ask`

<ParamField path="ask" type='"off" | "on-miss" | "always"'>
  - `off` — ніколи не показувати запит.
  - `on-miss` — показувати запит лише тоді, коли список дозволів не збігається.
  - `always` — показувати запит для кожної команди. Тривала довіра `allow-always` **не** пригнічує запити, коли ефективний режим ask — `always`.
</ParamField>

### `askFallback`

<ParamField path="askFallback" type='"deny" | "allowlist" | "full"'>
  Розв’язання, коли запит обов’язковий, але жодний UI недоступний.

- `deny` — блокувати.
- `allowlist` — дозволяти лише за збігу зі списком дозволів.
- `full` — дозволяти.
  </ParamField>

### `tools.exec.strictInlineEval`

<ParamField path="strictInlineEval" type="boolean">
  Коли `true`, OpenClaw трактує inline-форми code-eval як такі, що потребують лише схвалення,
  навіть якщо сам бінарний файл interpreter є в allowlist. Це додатковий захист
  для завантажувачів interpreter, які не зіставляються чисто з одним стабільним
  файловим операндом.
</ParamField>

Приклади, які перехоплює strict mode:

- `python -c`
- `node -e`, `node --eval`, `node -p`
- `ruby -e`
- `perl -e`, `perl -E`
- `php -r`
- `lua -e`
- `osascript -e`

У strict mode ці команди все одно потребують явного схвалення, і
`allow-always` не зберігає для них нові записи allowlist
автоматично.

## Режим YOLO (без схвалення)

Якщо ви хочете, щоб host exec виконувався без запитів на схвалення, потрібно відкрити
**обидва** шари політики — запитану політику exec у config OpenClaw
(`tools.exec.*`) **і** локальну політику схвалень хоста в
`~/.openclaw/exec-approvals.json`.

YOLO — це типова поведінка хоста, якщо ви явно не зробите її суворішою:

| Шар                   | Налаштування YOLO            |
| --------------------- | ---------------------------- |
| `tools.exec.security` | `full` на `gateway`/`node`   |
| `tools.exec.ask`      | `off`                        |
| Хост `askFallback`    | `full`                       |

<Warning>
**Важливі відмінності:**

- `tools.exec.host=auto` вибирає, **де** виконується exec: у пісочниці, якщо доступно, інакше на gateway.
- YOLO вибирає, **як** схвалюється host exec: `security=full` плюс `ask=off`.
- У режимі YOLO OpenClaw **не** додає окремий евристичний бар’єр схвалення для обфускації команд або шар відхилення script-preflight поверх налаштованої політики host exec.
- `auto` не робить маршрутизацію до gateway безкоштовним перевизначенням із сесії в пісочниці. Запит `host=node` на рівні окремого виклику дозволений з `auto`; `host=gateway` дозволений з `auto` лише тоді, коли немає активного runtime пісочниці. Для стабільного типового значення не `auto` установіть `tools.exec.host` або явно використовуйте `/exec host=...`.
  </Warning>

Провайдери на основі CLI, які надають власний неінтерактивний режим дозволів,
можуть дотримуватися цієї політики. Claude CLI додає
`--permission-mode bypassPermissions`, коли запитана політика exec OpenClaw
є YOLO. Перевизначте цю backend-поведінку явними аргументами Claude
в `agents.defaults.cliBackends.claude-cli.args` / `resumeArgs` —
наприклад `--permission-mode default`, `acceptEdits` або
`bypassPermissions`.

Якщо вам потрібне більш консервативне налаштування, зробіть суворішим будь-який шар до
`allowlist` / `on-miss` або `deny`.

### Постійне налаштування gateway-host «ніколи не показувати запит»

<Steps>
  <Step title="Установіть запитану політику config">
    ```bash
    openclaw config set tools.exec.host gateway
    openclaw config set tools.exec.security full
    openclaw config set tools.exec.ask off
    openclaw gateway restart
    ```
  </Step>
  <Step title="Узгодьте з файлом схвалень хоста">
    ```bash
    openclaw approvals set --stdin <<'EOF'
    {
      version: 1,
      defaults: {
        security: "full",
        ask: "off",
        askFallback: "full"
      }
    }
    EOF
    ```
  </Step>
</Steps>

### Локальне скорочення

```bash
openclaw exec-policy preset yolo
```

Це локальне скорочення оновлює обидва:

- Локальні `tools.exec.host/security/ask`.
- Типові значення локального `~/.openclaw/exec-approvals.json`.

Воно навмисно працює лише локально. Щоб змінити схвалення gateway-host або node-host
віддалено, використовуйте `openclaw approvals set --gateway` або
`openclaw approvals set --node <id|name|ip>`.

### Node host

Для node host застосуйте той самий файл схвалень на цьому вузлі:

```bash
openclaw approvals set --node <id|name|ip> --stdin <<'EOF'
{
  version: 1,
  defaults: {
    security: "full",
    ask: "off",
    askFallback: "full"
  }
}
EOF
```

<Note>
**Обмеження лише для локального режиму:**

- `openclaw exec-policy` не синхронізує схвалення вузла.
- `openclaw exec-policy set --host node` відхиляється.
- Схвалення node exec отримуються від вузла під час runtime, тому оновлення для вузлів потрібно виконувати через `openclaw approvals --node ...`.
  </Note>

### Скорочення лише для сесії

- `/exec security=full ask=off` змінює лише поточну сесію.
- `/elevated full` — це аварійне скорочення, яке також пропускає схвалення exec для цієї сесії.

Якщо файл схвалень хоста лишається суворішим за config, усе одно перемагає суворіша
політика хоста.

## Allowlist (для кожного агента)

Allowlists є **для кожного агента окремо**. Якщо існує кілька агентів, перемкніть агента,
якого ви редагуєте, у macOS app. Patterns — це glob-збіги.

Patterns можуть бути glob для шляху до розв’язаного бінарного файла або glob для голого імені команди.
Голі імена збігаються лише для команд, викликаних через `PATH`, тож `rg` може збігатися з
`/opt/homebrew/bin/rg`, коли команда — `rg`, але **не** `./rg` і не
`/tmp/rg`. Використовуйте glob шляху, коли хочете довіряти одному конкретному
розташуванню бінарного файла.

Застарілі записи `agents.default` мігрують до `agents.main` під час завантаження.
Shell-ланцюжки, такі як `echo ok && pwd`, усе одно вимагають, щоб кожен сегмент верхнього рівня
відповідав правилам allowlist.

Приклади:

- `rg`
- `~/Projects/**/bin/peekaboo`
- `~/.local/bin/*`
- `/opt/homebrew/bin/rg`

Кожен запис allowlist відстежує:

| Поле               | Значення                             |
| ------------------ | ------------------------------------ |
| `id`               | Стабільний UUID для identity в UI    |
| `lastUsedAt`       | Мітка часу останнього використання    |
| `lastUsedCommand`  | Остання команда, що збіглася          |
| `lastResolvedPath` | Останній розв’язаний шлях до бінарника |

## Автодозвіл для CLI Skills

Коли ввімкнено **Auto-allow skill CLIs**, виконувані файли, на які посилаються
відомі Skills, вважаються allowlisted на вузлах (macOS node або безголовий
node host). Для цього використовується `skills.bins` через Gateway RPC, щоб отримати
список skill bin. Вимкніть це, якщо хочете суворі ручні allowlists.

<Warning>
- Це **неявний список дозволів для зручності**, окремий від ручних записів path allowlist.
- Він призначений для довірених операторських середовищ, де Gateway і вузол перебувають в одній межі довіри.
- Якщо вам потрібна сувора явна довіра, залиште `autoAllowSkills: false` і використовуйте лише ручні записи path allowlist.
</Warning>

## Безпечні bin і пересилання схвалень

Про safe bins (швидкий шлях лише через stdin), деталі прив’язки interpreter і
пересилання запитів на схвалення в Slack/Discord/Telegram (або запуск їх як
рідних клієнтів схвалення) див. у
[Схвалення exec — розширено](/uk/tools/exec-approvals-advanced).

## Редагування в Control UI

Використовуйте картку **Control UI → Nodes → Exec approvals**, щоб редагувати типові значення,
перевизначення для окремих агентів і allowlists. Виберіть область дії (Defaults або агент),
налаштуйте політику, додайте/видаліть patterns allowlist, а потім натисніть **Save**. UI
показує метадані останнього використання для кожного pattern, щоб ви могли підтримувати список охайним.

Селектор цілі вибирає **Gateway** (локальні схвалення) або **Node**.
Вузли мають оголошувати `system.execApprovals.get/set` (macOS app або
безголовий node host). Якщо вузол ще не оголошує схвалення exec,
відредагуйте його локальний `~/.openclaw/exec-approvals.json` напряму.

CLI: `openclaw approvals` підтримує редагування gateway або node — див.
[CLI схвалень](/uk/cli/approvals).

## Потік схвалення

Коли потрібен запит на схвалення, gateway транслює
`exec.approval.requested` клієнтам-операторам. Control UI та macOS
app розв’язують його через `exec.approval.resolve`, після чого gateway пересилає
схвалений запит на node host.

Для `host=node` запити на схвалення містять канонічне корисне навантаження
`systemRunPlan`. Gateway використовує цей план як авторитетний
контекст command/cwd/session під час пересилання схвалених запитів `system.run`.

Це важливо для затримки асинхронного схвалення:

- Шлях node exec заздалегідь готує один канонічний план.
- Запис схвалення зберігає цей план і його метадані прив’язки.
- Після схвалення фінальний пересланий виклик `system.run` повторно використовує збережений план замість довіри до пізніших редагувань викликача.
- Якщо викликач змінює `command`, `rawCommand`, `cwd`, `agentId` або `sessionKey` після створення запиту на схвалення, gateway відхиляє пересланий запуск як невідповідність схваленню.

## Системні події

Життєвий цикл exec відображається як системні повідомлення:

- `Exec running` (лише якщо команда перевищує поріг сповіщення про виконання).
- `Exec finished`.
- `Exec denied`.

Вони публікуються в сесії агента після того, як вузол повідомляє про подію.
Схвалення exec на gateway-host створюють такі самі події життєвого циклу, коли
команда завершується (і, за потреби, коли виконується довше за поріг).
Exec із керуванням через схвалення повторно використовують id схвалення як `runId` у цих
повідомленнях для зручної кореляції.

## Поведінка при відхиленому схваленні

Коли асинхронне схвалення exec відхиляється, OpenClaw не дозволяє агенту
повторно використовувати вивід із будь-якого попереднього запуску тієї самої команди в цій сесії.
Причина відхилення передається з явною вказівкою, що вивід команди
недоступний, що не дає агенту стверджувати, що є новий вивід, або
повторювати відхилену команду зі застарілими результатами попереднього успішного
запуску.

## Наслідки

- **`full`** — потужний режим; якщо можливо, надавайте перевагу allowlists.
- **`ask`** тримає вас у циклі, водночас дозволяючи швидкі схвалення.
- Allowlists для окремих агентів не дають схваленням одного агента перетікати до інших.
- Схвалення застосовуються лише до запитів host exec від **авторизованих відправників**. Неавторизовані відправники не можуть видавати `/exec`.
- `/exec security=full` — це зручність на рівні сесії для авторизованих операторів, і за задумом вона пропускає схвалення. Щоб жорстко заблокувати host exec, установіть безпеку схвалень у `deny` або забороніть інструмент `exec` через політику інструментів.
